# Module Roadmap

This page tracks every module in the system — from actively developed modules through to ideas on the horizon. It serves as the single source of truth for what exists, what's coming, and what's being considered.

<Aside type="tip">
Each module listed here includes enough context for a Claude Code session to pick it up using the [Create a New Module](/module-starter/building-modules/) prompt without needing extra explanation.
</Aside>

---

## The Vision

The Module System is WordPress for React SaaS — a registry-based architecture where self-contained modules can be installed, updated, and eventually managed directly from an admin dashboard.

The system builds **single-tenant SaaS applications designed for individuals and solo/small businesses**. These are focused, purpose-built tools — not enterprise platforms with complex org hierarchies and multi-tenancy. Think personal productivity apps, client-facing tools for freelancers, niche vertical SaaS for small business owners.

<Aside type="note" title="Single-tenant product, tenant-grade isolation">
The *product* model and the *security* architecture are deliberately distinct — this is the reconciliation of "single-tenant" with an isolation layer that keys on a tenant:

- **The product is single-tenant.** Each app serves one owner (an individual or a solo/small business). There are **no team or organisation features and no per-seat billing** — those are permanently out of scope. The furthest this stretches is an opt-in "shared space" (e.g. a family), never enterprise team tenancy.
- **The security architecture keys all isolation on a per-owner column, not on the product being multi-tenant.** Cross-user isolation is delegated to **Neon** — Row Level Security policies, and optionally database-per-tenant — scoping on the security registry's `ownerColumn`, which modules declare as `tenantId`. In the common **solo-owner** deployment each user maps 1:1 to their own tenant and the deployment's own database is the wall; tenant-level facts (security tier, Stripe customer, deletion state) belong on the **tenant record** rather than the user. The exact tenant-record shape — a distinct record vs. reusing the user's id — is being confirmed pre-production (see the [Starter Template Roadmap](/starter-template/roadmap/)).

Stated plainly: "cross-tenant" isolation tests are really **user-boundary** tests; and admin impersonation, *if it is ever built*, is tenant-scoped operator support tooling (see [User Impersonation](#user-impersonation) below), never a team-management surface. The exact default isolation key and the database-per-tenant control-plane are finalised in the security **isolation phase** — the authoritative reference is the [Security Feature](/starter-template/security-feature/).
</Aside>

**Who builds with the system** is a separate question with two paths:

- **Developers** use the CLI workflow (`npx module-name init`) and the [Module Starter Template](/module-starter/) to create, customise, and publish modules.
- **Non-technical admins** will eventually be able to discover, install, configure, and update modules directly from the admin dashboard — just as WordPress admins browse and install plugins without touching a terminal.

The CLI is the developer path; the admin dashboard is the non-technical path to the same result.

## How to Read This Roadmap

Modules are grouped by status:

- **✅ Released** — Available and documented. Install with `npx`.
- **🔄 In Progress** — Actively being developed.
- **📋 Planned** — Committed to building. Design work may have started.
- **💡 Under Consideration** — Strong candidates. Not yet committed.
- **💭 Ideas** — Early-stage thinking. May or may not happen.

Each module entry includes:

| Field | Description |
|-------|-------------|
| **Module ID** | The kebab-case identifier used in `module.ts` |
| **Priority** | High / Medium / Low |
| **Dependencies** | Other modules required first |
| **Complexity** | Small (1–2 days) / Medium (3–5 days) / Large (1–2 weeks) |
| **Status notes** | Current state and what needs to happen next |

### Style

| Field | Value |
|-------|-------|
| **Module ID** | `style` |
| **Priority** | Medium |
| **Complexity** | Medium |
| **Status** | Direction confirmed — standalone theming module |
| **Description** | Admin-configurable theme and style management. Provides an admin panel section where admins can customise the app's visual appearance: shadcn component configuration (`components.json` — border radius, base colour, styling mode), colour palette overrides, and CSS variable customisation applied via `globals.css`. Standalone module that the Block Builder and any other module can consume. |
| **Registries** | adminPagesRegistry, routeRegistry, serviceRegistry, actionRegistry |
| **Dependencies** | None (standalone) |

**Architecture:**

- **Admin UI** — settings page for configuring theme: colour palette, border radius, font, shadcn style options
- **Backend** — stores style configuration in database (like app settings), fires `admin.styleUpdated` action hook on save
- **Frontend** — applies CSS variable overrides to `globals.css` at runtime, fires `theme.changed` action hook so other modules can react
- **Action hooks:**
  - `admin.styleUpdated` (backend) — modules can react to style config changes (e.g., cache invalidation, rebuild)
  - `theme.changed` (frontend) — modules can react to live theme updates (e.g., re-render themed components, sync cached styles)

<Aside type="note">
The starter template's existing light/dark mode toggle is a separate user preference and is unaffected by this module. The style module manages the broader design token system — the colours, radius, typography, and component styling that define how the app looks in both light and dark modes.
</Aside>

## Cross-Module Architecture (Stage 3–4)

The entries above are the module *inventory*. This section tracks the **cross-cutting conventions and tooling** that let those modules compose safely as the system grows. Several items are shared with the [Module Starter Roadmap](/module-starter/roadmap/) (the scaffold that emits these conventions) and the [Starter Template Roadmap](/starter-template/roadmap/) (the foundation that enforces them) — the framing below is module-facing and cross-linked, to extend those entries rather than duplicate them.

### Stage 3 — conventions

- **Event-first inter-module communication.** Prefer emitting and subscribing to events (`actionRegistry` / `filterRegistry` hooks) over reaching into another module's service. `serviceRegistry.get()` stays supported but is documented as a **fallback** for the cases events can't express — the Medusa pattern of subscribing to events and resolving a service only when you must. Keeps modules loosely coupled and independently installable.
- **Cross-module links + query layer (Medusa-style).** Because modules deliberately avoid `@relation` across schemas, composing data across modules needs an explicit **link + query** layer — a first-class way to declare a relationship between two modules' models and read across them (Medusa's Module Links + Query). Today cross-module reads go through `serviceRegistry.get()` ad hoc; a declared link layer makes those joins explicit, typed, and safe to evolve as either module changes.
- **Odoo-style module manifest.** Extend `module.ts` with a declarative `depends` list (modules loaded before this one) and an `auto_install`-style **"link module"** flag — a module that installs automatically once all of its dependencies are present, used for optional cross-module integrations between two otherwise-independent modules (Odoo's "link module" pattern). Pairs with per-module **versioned migrations directories**. The resolution algorithm (topological sort + cycle detection) and the migration-tracking system live on the [Module Starter Roadmap](/module-starter/roadmap/).
- **Boundary-enforcement lint.** A lint rule set that keeps the registry pattern honest: no cross-feature deep imports (import only a feature's public `index.ts`), order-collision warnings when two modules register hooks at the same `order`, and "registries only — never patch the shell". Encodes the [LLM Rules](/getting-started/llm-rules/) golden rule as CI.
- **Declarative Query Builder (shared via registry).** Grow Structure's 13-operator [JSON field filtering](/modules/structure/configuration/) (the `DataFilterOperator` engine over the `data` column) into a declarative, JetEngine-inspired **Query Builder** contributed through a registry, so any module can offer typed, admin-authored filters over its own JSON data instead of each reinventing one.

### Stage 4 — longer-term platform

Several of these already have module or starter-template homes — **extend the linked entry, don't duplicate it**:

- **Feature flags** — per-app / per-tenant toggles for gating unfinished or tiered capability.
- **Admin impersonation** — operator support tooling only, and strictly **tenant-scoped** (see [the tenancy framing](#the-vision) and the [User Impersonation](#user-impersonation) entry).
- **GDPR / data-export self-service** — see the **Data Export / Privacy** entry above; the foundation's [deletion cascade](/starter-template/security-feature/#deletion-cascade) already provides the erasure half.
- **In-app module install / update dashboard** — see [Admin Module Installation](#admin-module-installation-vision) below and the starter's [Admin Module Management](/starter-template/roadmap/).
- **Module marketplace / registry index** — see the [Module Starter Roadmap](/module-starter/roadmap/).
- **Internationalisation (i18n)** — see the [i18n](#internationalisation-i18n) entry.

## Admin Module Installation (Vision)

The long-term goal is a WordPress-style admin experience:

1. **Module Catalogue** — Admin panel page showing available modules with descriptions, screenshots, and compatibility info
2. **One-click Install** — Admin clicks "Install", module runs its init process in the background
3. **Configuration** — Post-install configuration wizard within the admin panel
4. **Updates** — Notification when module updates are available, with changelog preview and one-click update
5. **Dependencies** — Automatic dependency resolution (installing module A auto-installs required module B)
6. **Uninstall** — Clean removal with confirmation of what will be removed

<Aside type="caution">
Admin-installable modules require the module system to handle installation without CLI access. This means the init process (file generation, schema migration, registration) needs to run server-side. This is a significant architectural milestone that builds on top of the existing CLI workflow.
</Aside>
