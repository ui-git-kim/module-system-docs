### Shared CLAUDE.md scaffold section documents a removed import path

- **Description:** The module's own Module Overview in `CLAUDE.md` was refreshed for v1.4.2 (2026-07-15), but the **scaffold-owned shared section** below it still documents settings components at `@/features/user/components/settings/SettingsComponents` â€” a path that no longer exists in host v1.52+ (`@/components/settings-blocks` replaced it). The Module Overview carries a corrective note, but the stale text propagates to every module via scaffold updates.
- **Where:** The shared "Starter Template Components" section of `CLAUDE.md` â€” scaffold-owned, fix belongs in the module starter's CLAUDE.md template.
- **Severity:** Low (the module-specific section corrects it inline)
- **Workaround:** Trust the Module Overview note; for this module's templates, `settings-blocks` is correct.

### Prisma models use generic PascalCase names

- **Description:** `model Subscription` / `model BillingSettings` / `enum SubscriptionStatus` carry no module prefix. Convention for module schemas is snake_case, module-prefixed (`billing_subscription`) to avoid collisions in the host's shared `schema.prisma`. Renaming is a breaking migration for installed apps â€” it should ride the scaffold's `migrations` array (expand â†’ backfill â†’ contract) with `@@map` to preserve data.
- **Where:** `src/templates/schema.ts`; `schema.modelNames` in `src/module.ts`; every `prisma.subscription.*` / `prisma.billingSettings.*` call in the backend templates.
- **Severity:** Low (advisory â€” no collision reported; cost of fixing is high)
- **Workaround:** None required; avoid installing another module that defines a `Subscription` model.

### Webhook event dedupe is per-process

- **Description:** Duplicate Stripe deliveries are deduplicated via an in-memory set of the last 1,000 event IDs. Multiple backend instances (or a restart mid-delivery) can process the same event twice. The sync operations are upserts and status transitions, so double-processing is mostly harmless, but it is not guaranteed side-effect-free forever.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (`processedEvents`).
- **Severity:** Low (single-instance deployments are the norm for this product)
- **Workaround:** Run a single backend instance, or back the dedupe with a durable store (see Planned).

## Planned Improvements

### Short-term (next version)

**Complete the end-to-end host verification**
- Re-run the full pass in a current starter app (host â‰Ą v1.54.0) at v1.4.3: install, `prisma migrate dev`, type-check both sides (confirms the v1.4.1/v1.4.2 fixes and the v1.4.3 `allowNoOrigin` registration), boot, complete a test-card checkout with `stripe listen`, verify webhook sync and the dunning/recovery transitions, exercise `/admin/billing` (Stripe config save, webhook connect refusal on localhost, entitlement config save).
- *Context:* the v1.4.0 pass on `veradai` surfaced the two upstream issues, both fixed in starter v1.54.0; this item closes the loop. Record the result on this page.

### Medium-term (upcoming)

**Action hooks on subscription transitions**
- Fire `actionRegistry` hooks from the webhook sync â€” e.g. `billing.subscription.updated { tenantId, previousStatus, status, planId }`, `billing.payment.failed { tenantId }`, `billing.subscription.canceled { tenantId }` â€” so other modules can react to plan changes without polling `getSubscription`.
- *Context:* the Integration page currently documents "no action hooks"; update it when this lands. Follow the cog-ingest hook-naming pattern (`module.noun.verb`), fire **after** the DB upsert commits, and use `addActionAsync` semantics.

**Migrate webhook verification to the host security helpers**
- Optionally replace the handler's Stripe-SDK verification (`stripe.webhooks.constructEvent`) with the host's `verifyTimestampedWebhookSignature({ payload: req.rawBody, header, secret })` from `@/features/security` (starter â‰Ą v1.54.0) â€” replay window and rotation candidates handled host-side, one less behaviour to own.
- *Context:* deliberately **not** bundled into v1.4.3: `constructEvent` is Stripe's official implementation and already enforces a 300-second replay tolerance and checks all rotation signatures in the header, so the practical gain is small â€” and the swap touches the payment-critical path, so it should be made against a live â‰Ą v1.54.0 host (verify the backend import path and parse the event JSON explicitly after verification). Minor release when it lands.

**Durable webhook event dedupe**
- Replace the in-memory `processedEvents` set with a small `billing_webhook_event` table (event ID + processed timestamp, pruned by age), making processing idempotent across instances and restarts.
- *Context:* clears the per-process dedupe known issue; keep the "mark handled only after success" ordering so Stripe retries still work.

**Full-erasure owner flow**
- `billingService.deleteStripeCustomerForTenant(tenantId)` exists but nothing calls it. Add a deliberate, confirmed owner action (likely on `/admin/billing` in a danger section) that deletes the Stripe Customer â€” removing invoices from Stripe â€” for "delete everything" requests, distinct from the default deletion path that retains tax records.
- *Context:* two-mode deletion is the agreed design: default = cancel + retain records ('preserved'); full erasure = explicit second action.

**`onUninstall` lifecycle hook**
- Offer a data export and optionally cancel the live Stripe subscription when the module is uninstalled from an app, so an uninstall can't silently leave a paying subscription running.
- *Context:* scaffold `lifecycle.onUninstall` is available; mirror the deletion handler's cancel logic.

**Adopt the `migrations` array for any breaking schema change**
- If the model-prefix rename (or the next re-key) goes ahead, ship it via the scaffold v1.18 per-module migration runner (`npx billing migrate-db`) with expand â†’ backfill â†’ contract steps rather than asking apps to hand-migrate.

### Long-term / Ideas

**Multi-member tenancy readiness**
- Billing is already keyed on `tenantId`, so when the host resolves shared tenants (an invited member/VA on the same account), members inherit the owner's plan with no schema change. Remaining work is UX: the member read-only state exists on `/settings/billing`; extend the same "managed by owner" treatment anywhere else billing surfaces.
- *Context:* do **not** build tenancy into billing itself â€” the host owns membership resolution.

**Dashboard widget**
- A small plan/trial-status widget via `dashboardRegistry` (plan name, status badge, trial days remaining, past-due warning). Deliberately not shipped yet; the hooks make it trivial for apps to build their own.

**Usage-based pricing, if it ever returns**
- Metered billing was retired in v1.3.0 and remains out of scope. If usage pricing is ever revisited, build on Stripe's current Billing Meters APIs (the legacy usage-record APIs the old implementation used are deprecated) â€” treat the old code as unrecoverable prior art.
