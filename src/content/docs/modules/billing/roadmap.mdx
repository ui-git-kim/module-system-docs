## Known Issues

<Aside type="caution" title="2026-07-23 meta-audit">
A four-dimension audit (standards / production / performance / security) of the generated code ran on 2026-07-23 against module v1.4.3 and host v1.54.0. It confirmed **21 findings** via adversarial verification â€” after merging one duplicate pair (the two admin Save buttons share a root cause), **20 distinct issues: 1 critical, 6 moderate, 13 minor**. The critical + moderate items are listed below; the minor items are grouped under [Planned Improvements â†’ Meta-audit polish](#meta-audit-polish-2026-07-23). None are shipping blockers **except** the cross-tenant subscription binding issue, which should be fixed before any real-money deployment.
</Aside>

### Cross-tenant subscription binding via client-controlled `client_reference_id`

- **Description:** `StripePricingTable` renders `client-reference-id={userId}` as a plain DOM attribute on the native `<stripe-pricing-table>` element â€” any authenticated user can edit it in devtools before checking out. `syncSubscriptionFromWebhook` then trusts `session.client_reference_id` as the **first-priority** tenant key (`let resolvedTenantId = tenantId;`; the existing-row and customer-metadata fallbacks only run when it is absent) and upserts `where: { tenantId: resolvedTenantId }` with the paying session's `stripeCustomerId`. A user who knows a victim's `tenantId` (in the solo product `tenantId === userId`) can complete a checkout (free if a trial plan exists) whose signed `checkout.session.completed` event carries the victim's id, overwriting the **victim's** `Subscription` row with the attacker's Stripe customer. Downstream `/portal`, `/payment-method`, `/invoices` and account-deletion all resolve the customer from that row, so this is a bidirectional billing-data exposure and a subscription-hijack write primitive. The Stripe HMAC does not defend against it â€” the hostile value rides inside a legitimately-signed event.
- **Where:** `src/templates/frontend/index.ts` (`StripePricingTable`, `client-reference-id={userId}`); `src/templates/backend/index.ts` `syncSubscriptionFromWebhook` (tenant-resolution order) and the `checkout.session.completed` handler (passes `client_reference_id` through unvalidated).
- **Severity:** **Critical** (cross-tenant isolation break / IDOR-class). Likelihood is bounded â€” needs the victim's cuid-form id and a completed checkout, so it is targeted, not mass â€” but the consequence (destroying/hijacking any tenant's subscription) makes it fix-before-ship.
- **Fix:** Invert the trust order in `syncSubscriptionFromWebhook` â€” resolve from an existing row for `stripeCustomerId`, then `customer.metadata.tenantId`, and only fall back to `client_reference_id` last; refuse to rebind a tenant whose existing row points at a different `stripeCustomerId`. Robust remedy: stop passing the raw `userId` â€” issue a short-lived HMAC-signed reference token from a member endpoint (signed with `SECURITY_ENCRYPTION_KEY`), use it as `client-reference-id`, and verify it in the webhook before use (a first-time subscriber has no existing row and the pricing-table customer carries no `tenantId` metadata, so the reference cannot simply be dropped).

### Read paths have no Stripe error handling or stale-on-error fallback

- **Description:** `getPlanDetails()` (awaited unguarded by `getSubscription()`) and `getPlanLimits()` call Stripe with no try/catch. During any Stripe outage or transient 429/5xx, `GET /subscription` and `GET /entitlements` â€” member reads whose core data already lives in the DB â€” fail completely instead of degrading, and the 5-minute `planCache` is discarded the moment it expires rather than served stale-on-error. Worse, the raw `StripeError` carries a `statusCode`, and the host error middleware does `err.statusCode || 500`, so a Stripe 401 (rotated/bad secret key) surfaces to the browser as HTTP 401 and a 404/429 as 404/429 â€” Stripe failures are indistinguishable from client validation failures. (The host's api interceptor does not force-logout on 401, and production messages are genericised, so no internals leak â€” but the misleading status remains.)
- **Where:** `src/templates/backend/index.ts` â€” `getPlanDetails` (~line 527), `getSubscription` (~line 427), `getPlanLimits` (~line 811).
- **Severity:** Moderate.
- **Fix:** Wrap the Stripe calls in try/catch; on failure serve the cached plan even past TTL (add a cache to `getPlanLimits` â€” see the entitlements caching item), else translate the Stripe error into a 502/503 `AppError` rather than propagating Stripe's own `statusCode`. `getSubscription` should still return DB-backed status/dates when plan enrichment is unavailable.

### Out-of-order webhook delivery can overwrite newer subscription state

- **Description:** `syncSubscriptionFromWebhook` unconditionally upserts the event's subscription snapshot. Stripe does not guarantee delivery order (and the handler returns 5xx without recording the event id on failure, so retries arrive later). A delayed `customer.subscription.created` (TRIALING, earlier period-end) processed after `customer.subscription.updated` (ACTIVE, later period-end) reverts `status`/`currentPeriodEnd`/`cancelAtPeriodEnd` to stale values until the next event re-syncs. This drives entitlement resolution and the dunning/settings UI. Distinct from the per-process dedupe item (that covers exact-duplicate ids, not ordering).
- **Where:** `src/templates/backend/index.ts` â€” `syncSubscriptionFromWebhook` upsert (~line 609) and the webhook handler (passes `event.data.object` directly).
- **Severity:** Moderate (self-heals on the next event; no permanent corruption).
- **Fix:** Thread the Stripe event `created` timestamp into the sync and make the update monotonic â€” persist a `lastEventAt` marker and guard the update (`updateMany` on `tenantId` AND `lastEventAt < incoming`), or at minimum ignore `customer.subscription.created` when a newer row already exists.

### Admin save buttons can wipe stored config on a slow/errored load

- **Description:** On `BillingAdminPage`, `entries` and `stripeCfg` initialise empty and populate only from their queries' data, but both Save buttons ("Save Stripe config", "Save entitlements") are gated solely on the mutation's `isPending` â€” never on the backing query's `isSuccess` â€” and none of the four load queries render an error state. Because both PUT endpoints full-replace, a Save clicked while `configQuery`/`stripeConfigQuery` is still loading or has transiently errored (the form then looks identical to "nothing configured yet") silently wipes the tenant's entitlement options or Stripe config â€” the latter breaks the pricing table served to every member.
- **Where:** `src/templates/frontend/index.ts` â€” `BillingAdminPage` handlers/buttons (~lines 1862, 1930); backend full-replace upsert in `setEntitlementConfig`/stripe-config PUT.
- **Severity:** Moderate (requires a transient load failure/race plus an admin clicking Save on a misleadingly-empty form; recoverable by re-entry).
- **Fix:** Gate each Save on its source query â€” `disabled={mutation.isPending || !sourceQuery.isSuccess}` â€” and render explicit error/retry states for `configQuery`/`stripeConfigQuery` instead of empty editable forms.

### Post-checkout success alert races the webhook

- **Description:** After the hosted Pricing Table redirects to `/settings/billing?success=true`, the page shows a green "Payment successful! Your subscription is now active." but nothing invalidates or polls the subscription query. The `checkout.session.completed` webhook that writes the row races the redirect and often has not landed, so the "now active" alert can render directly above a "No active plan" card until a manual refresh.
- **Where:** `src/templates/frontend/index.ts` â€” `BillingSettingsPage` success branch (~line 1417); the only `success` effect is a 5s param-clear.
- **Severity:** Moderate (every checkout is exposed; self-heals on refresh; no data impact).
- **Fix:** On `checkoutSuccess`, invalidate the `billing` query keys and briefly poll (`refetchInterval` until status is active/trialing, with a cap), and/or soften the copy to "Payment received â€” your subscription will activate shortly" until the query confirms.

### Entitlements resolution hits Stripe live on every check (no product cache)

- **Description:** The module has a 5-minute `planCache`, but only `getPlanDetails()` consults it. `getPlanLimits()` â€” which backs `GET /entitlements`, `POST /entitlements/check`, and the backend `checkEntitlement()` other modules call per-request â€” calls `products.retrieve()` unconditionally, so every entitlement check is a live Stripe round-trip (~200â€“500 ms) with rate-limit exposure scaling linearly with active users and with any module gating actions on `checkEntitlement`.
- **Where:** `src/templates/backend/index.ts` â€” `getPlanLimits` (~line 811); existing `planCache` (~line 373) consulted only by `getPlanDetails`.
- **Severity:** Moderate (performance; no correctness impact).
- **Fix:** Add a small product-level cache (name + `max_*`/`limit_*` metadata) keyed by `productId` with the same 5-minute TTL and have `getPlanLimits()` read through it. (Pairs with the stale-on-error item above.)

### Stripe SDK dependency is unpinned (`@latest` at install)

- **Description:** `backend.dependencies` declares the SDK as a bare `'stripe'`; the scaffold installer appends `@latest`, so `npx billing init` installs whatever major is current. The `apiVersion` cast neutralises the most common cross-major type break, but a future SDK major can still drift method signatures the generated service/webhook rely on, and installs are non-reproducible across dates.
- **Where:** `src/module.ts` (`backend.dependencies: ['stripe']`, ~line 145).
- **Severity:** Moderate (install/build-time drift; not an end-user issue).
- **Fix:** Pin a compatible major range (e.g. `'stripe@^18.0.0'`) â€” the scaffold leaves entries containing `@` untouched.

What remains: re-run the webhook sync leg end-to-end on a host â‰Ą v1.54.0 (it was unverifiable before â€” the handler rejects unverifiable deliveries by design), and the v1.4.1/v1.4.2 type-cleanliness fixes landed after that run and haven't been re-verified in the host.
- **Where:** The whole generated output; re-run in a current starter app (host â‰Ą v1.54.0, after `template:update`).
- **Severity:** Medium

### Shared CLAUDE.md scaffold section documents a removed import path

- **Description:** The module's own Module Overview in `CLAUDE.md` was refreshed for v1.4.2 (2026-07-15), but the **scaffold-owned shared section** below it still documents settings components at `@/features/user/components/settings/SettingsComponents` â€” a path that no longer exists in host v1.52+ (`@/components/settings-blocks` replaced it). The Module Overview carries a corrective note, but the stale text propagates to every module via scaffold updates.
- **Where:** The shared "Starter Template Components" section of `CLAUDE.md` â€” scaffold-owned, fix belongs in the module starter's CLAUDE.md template.
- **Severity:** Low (the module-specific section corrects it inline)
- **Workaround:** Trust the Module Overview note; for this module's templates, `settings-blocks` is correct.

### Prisma models use generic PascalCase names

- **Description:** `model Subscription` / `model BillingSettings` / `enum SubscriptionStatus` carry no module prefix. Convention for module schemas is snake_case, module-prefixed (`billing_subscription`) to avoid collisions in the host's shared `schema.prisma`. Renaming is a breaking migration for installed apps â€” it should ride the scaffold's `migrations` array (expand â†’ backfill â†’ contract) with `@@map` to preserve data.
- **Where:** `src/templates/schema.ts`; `schema.modelNames` in `src/module.ts`; every `prisma.subscription.*` / `prisma.billingSettings.*` call in the backend templates.
- **Severity:** Low (advisory â€” no collision reported; cost of fixing is high)
- **Workaround:** None required; avoid installing another module that defines a `Subscription` model.

### Webhook event dedupe is per-process

- **Description:** Duplicate Stripe deliveries are deduplicated via an in-memory set of the last 1,000 event IDs. Multiple backend instances (or a restart mid-delivery) can process the same event twice. The sync operations are upserts and status transitions, so double-processing is mostly harmless, but it is not guaranteed side-effect-free forever.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (`processedEvents`).
- **Severity:** Low (single-instance deployments are the norm for this product)
- **Workaround:** Run a single backend instance, or back the dedupe with a durable store (see Planned).

## Planned Improvements

### Short-term (next version)

**Complete the end-to-end host verification**
- Re-run the full pass in a current starter app (host â‰Ą v1.54.0) at v1.4.3: install, `prisma migrate dev`, type-check both sides (confirms the v1.4.1/v1.4.2 fixes and the v1.4.3 `allowNoOrigin` registration), boot, complete a test-card checkout with `stripe listen`, verify webhook sync and the dunning/recovery transitions, exercise `/admin/billing` (Stripe config save, webhook connect refusal on localhost, entitlement config save).
- *Context:* the v1.4.0 pass on `veradai` surfaced the two upstream issues, both fixed in starter v1.54.0; this item closes the loop. Record the result on this page.

### Medium-term (upcoming)

**Action hooks on subscription transitions**
- Fire `actionRegistry` hooks from the webhook sync â€” e.g. `billing.subscription.updated { tenantId, previousStatus, status, planId }`, `billing.payment.failed { tenantId }`, `billing.subscription.canceled { tenantId }` â€” so other modules can react to plan changes without polling `getSubscription`.
- *Context:* the Integration page currently documents "no action hooks"; update it when this lands. Follow the cog-ingest hook-naming pattern (`module.noun.verb`), fire **after** the DB upsert commits, and use `addActionAsync` semantics.

**Migrate webhook verification to the host security helpers**
- Optionally replace the handler's Stripe-SDK verification (`stripe.webhooks.constructEvent`) with the host's `verifyTimestampedWebhookSignature({ payload: req.rawBody, header, secret })` from `@/features/security` (starter â‰Ą v1.54.0) â€” replay window and rotation candidates handled host-side, one less behaviour to own.
- *Context:* deliberately **not** bundled into v1.4.3: `constructEvent` is Stripe's official implementation and already enforces a 300-second replay tolerance and checks all rotation signatures in the header, so the practical gain is small â€” and the swap touches the payment-critical path, so it should be made against a live â‰Ą v1.54.0 host (verify the backend import path and parse the event JSON explicitly after verification). Minor release when it lands.

**Durable webhook event dedupe**
- Replace the in-memory `processedEvents` set with a small `billing_webhook_event` table (event ID + processed timestamp, pruned by age), making processing idempotent across instances and restarts.
- *Context:* clears the per-process dedupe known issue; keep the "mark handled only after success" ordering so Stripe retries still work.

**Full-erasure owner flow**
- `billingService.deleteStripeCustomerForTenant(tenantId)` exists but nothing calls it. Add a deliberate, confirmed owner action (likely on `/admin/billing` in a danger section) that deletes the Stripe Customer â€” removing invoices from Stripe â€” for "delete everything" requests, distinct from the default deletion path that retains tax records.
- *Context:* two-mode deletion is the agreed design: default = cancel + retain records ('preserved'); full erasure = explicit second action.

**`onUninstall` lifecycle hook**
- Offer a data export and optionally cancel the live Stripe subscription when the module is uninstalled from an app, so an uninstall can't silently leave a paying subscription running.
- *Context:* scaffold `lifecycle.onUninstall` is available; mirror the deletion handler's cancel logic.

**Adopt the `migrations` array for any breaking schema change**
- If the model-prefix rename (or the next re-key) goes ahead, ship it via the scaffold v1.18 per-module migration runner (`npx billing migrate-db`) with expand â†’ backfill â†’ contract steps rather than asking apps to hand-migrate.

### Meta-audit polish (2026-07-23)

The minor findings from the 2026-07-23 meta-audit â€” correctness/robustness/consistency gaps with bounded, self-healing, or edge-only impact. Grouped by file area for batching.

**Backend service/handlers** (`src/templates/backend/index.ts`)
- `handlePaymentFailed()` flips every row for a customer to `PAST_DUE` with no status guard, whereas `handlePaymentSucceeded()` guards on `status: 'PAST_DUE'`. A late/stray `invoice.payment_failed` after cancellation can resurrect a `CANCELED` row to `PAST_DUE` and show a spurious dunning banner. Mirror the counterpart's guard (`status: { in: ['ACTIVE','TRIALING','INCOMPLETE'] }`).
- `getPlanLimits()` awaits `subscription.findUnique` then `getEntitlementConfig()` serially though they are independent â€” batch with `Promise.all` (and add a `select`).
- `connectWebhook()` creates the Stripe endpoint then does an **unguarded** `billingSettings.upsert`; a DB failure orphans the endpoint whose signing secret Stripe returns only once. Wrap the upsert and delete the endpoint on failure (mirroring the encrypt-failure rollback). Self-heals on retry, hence minor.
- `GET /invoices` caps `limit` at 100 but never floors it â€” a negative `?limit` passes `|| 10` and reaches Stripe as `limit: -N`, throwing. Floor with `Math.max(1, â€¦)`.
- `getStripe()`/`apiVersion '2024-12-18.acacia'` is duplicated verbatim in the service and the webhook handler â€” single-source it so a version bump touches one place.

**Frontend settings page** (`src/templates/frontend/index.ts`, `BillingSettingsPage`)
- `handleOpenPortal` has no try/catch, no in-flight state, and no button-disable (unlike `ManageSubscriptionButton`, which does it correctly a few hundred lines away) â€” a failed portal open is a silent unhandled rejection, and double-clicks fire multiple sessions.
- Payment-method and invoice queries conflate loading/error with empty â€” a subscriber sees "No payment method on file." / "No invoices yet" while the query is in flight or errored. Add skeleton + distinct error states.
- Invoice links use `window.open(url, '_blank')` without `'noopener,noreferrer'` (reverse-tabnabbing hardening; Stripe-hosted URLs so low risk).
- The 5s success-param cleanup calls `setSearchParams({})`, wiping **all** query params and pushing a history entry â€” delete only `success`/`canceled` and pass `{ replace: true }`.
- `planStatusLabel()` capitalises the raw Stripe status, rendering American "Canceled" â€” map to Australian/friendly labels as `SubscriptionStatus` already does.

**Frontend admin page & hooks** (`src/templates/frontend/index.ts`)
- `StripePricingTable` appends `pricing-table.js` with no `onerror` and never reads the config query's `isError` â€” a blocked script leaves the purchase section blank and a config-fetch error shows a misleading "not configured" message. Add an `onerror` fallback and branch on `isError`.
- `BillingAdminPage` read-only status displays collapse `undefined` (loading/error) to definitive wrong values ("Not connected", "No active plan") â€” show "Checkingâ€¦" / an error line first.
- `useEntitlements.checkLimit()` gates its local-cache branch on `!isLoading`, so on an **errored** query (where `isLoading` is false but data is undefined) it silently computes against free-tier defaults instead of reaching the server fallback. Branch on `query.data` presence instead. Fails restrictive and self-heals on retry, hence minor.

### Long-term / Ideas

**Multi-member tenancy readiness**
- Billing is already keyed on `tenantId`, so when the host resolves shared tenants (an invited member/VA on the same account), members inherit the owner's plan with no schema change. Remaining work is UX: the member read-only state exists on `/settings/billing`; extend the same "managed by owner" treatment anywhere else billing surfaces.
- *Context:* do **not** build tenancy into billing itself â€” the host owns membership resolution.

**Dashboard widget**
- A small plan/trial-status widget via `dashboardRegistry` (plan name, status badge, trial days remaining, past-due warning). Deliberately not shipped yet; the hooks make it trivial for apps to build their own.

**Usage-based pricing, if it ever returns**
- Metered billing was retired in v1.3.0 and remains out of scope. If usage pricing is ever revisited, build on Stripe's current Billing Meters APIs (the legacy usage-record APIs the old implementation used are deprecated) â€” treat the old code as unrecoverable prior art.
