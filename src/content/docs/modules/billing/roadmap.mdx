## Known Issues

<Aside type="tip" title="2026-07-23 meta-audit — all 20 findings fixed in v1.5.0">
A four-dimension audit (standards / production / performance / security) of the generated code ran on 2026-07-23 against module v1.4.3 and host v1.54.0. It confirmed **21 findings** via adversarial verification — **20 distinct issues** after merging one duplicate pair (1 critical, 6 moderate, 13 minor). All were fixed in **v1.5.0** (see [Completed](#completed)), including the critical cross-tenant subscription-binding hole (`client-reference-id` is now HMAC-signed and verified in the webhook). The code fixes pass the module's wrapper checks and a full generated-code TS/TSX parse, and — as of v2.1.2 — a full **host type-check** in `veradai` (see the item below); the live webhook runtime leg is still pending.
</Aside>

### End-to-end host verification — type-check leg DONE (v2.1.2); runtime leg pending

- **Type-check ✅ (2026-07-24):** the full generated output — backend **and** frontend — compiles cleanly in `veradai` (host ≥ v1.54.0, React 19, Stripe SDK v22, Prisma client generated from the v2.0.0 schema). This closes a gap open since v1.4.0. It caught exactly one real bug the wrapper checks structurally cannot (Stripe v22 typing `accounts.retrieve` as requiring an id), fixed in **v2.1.2**. The admin page has also been eyeballed live.
- **Runtime ⏳ still pending:** the live webhook-delivery + test-payment loop is unverified. The connect flow refuses localhost and the three-tick verification needs a public endpoint, so this requires an actual deployment. What remains to exercise once deployed: connect the webhook, run the three-tick verification green, complete a test-card checkout with `stripe listen`/a live endpoint, and confirm the subscription syncs (and that a devtools-edited `client-reference-id` is rejected), plus the dunning/recovery transitions.
- **Where:** re-run in a deployed starter app (host ≥ v1.54.0) after `template:update` + `update-schema` + `prisma migrate dev`.
- **Severity:** Medium
- **Workaround:** None needed on ≥ v1.54.0 hosts. Verification steps: type-check both sides; boot; complete a test-card checkout with `stripe listen`; confirm the subscription syncs (and that a devtools-edited `client-reference-id` is now rejected, not honoured); exercise the dunning/recovery transitions and `/admin/billing`.

### Shared CLAUDE.md scaffold section documents a removed import path

- **Description:** The module's own Module Overview in `CLAUDE.md` was refreshed for v1.4.2 (2026-07-15), but the **scaffold-owned shared section** below it still documents settings components at `@/features/user/components/settings/SettingsComponents` — a path that no longer exists in host v1.52+ (`@/components/settings-blocks` replaced it). The Module Overview carries a corrective note, but the stale text propagates to every module via scaffold updates.
- **Where:** The shared "Starter Template Components" section of `CLAUDE.md` — scaffold-owned, fix belongs in the module starter's CLAUDE.md template.
- **Severity:** Low (the module-specific section corrects it inline)
- **Workaround:** Trust the Module Overview note; for this module's templates, `settings-blocks` is correct.

### Prisma models use generic PascalCase names

- **Description:** `model Subscription` / `model BillingSettings` / `enum SubscriptionStatus` carry no module prefix. Convention for module schemas is snake_case, module-prefixed (`billing_subscription`) to avoid collisions in the host's shared `schema.prisma`. Renaming is a breaking migration for installed apps — it should ride the scaffold's `migrations` array (expand → backfill → contract) with `@@map` to preserve data.
- **Where:** `src/templates/schema.ts`; `schema.modelNames` in `src/module.ts`; every `prisma.subscription.*` / `prisma.billingSettings.*` call in the backend templates.
- **Severity:** Low (advisory — no collision reported; cost of fixing is high)
- **Workaround:** None required; avoid installing another module that defines a `Subscription` model.

### Webhook event dedupe is per-process

- **Description:** Duplicate Stripe deliveries are deduplicated via an in-memory set of the last 1,000 event IDs. Multiple backend instances (or a restart mid-delivery) can process the same event twice. The sync operations are upserts and status transitions, so double-processing is mostly harmless, but it is not guaranteed side-effect-free forever.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (`processedEvents`).
- **Severity:** Low (single-instance deployments are the norm for this product)
- **Workaround:** Run a single backend instance, or back the dedupe with a durable store (see Planned).

## Planned Improvements

### Short-term (next version)

**Phase 3 — promotions (v2.1.0)**
- Promotions: start with *select an existing Stripe coupon* + an `allow_promotion_codes` toggle, which delivers most of the value for a fraction of the surface. The full builder (reference-B style) is a second versioning UI on near-immutable Coupons. Two hard rules when it lands: "Free Trial" must write `trial_period_days`, **not** a 100%-off coupon (which still collects a payment method and yields `active`, not `trialing`, breaking `TrialCountdown`/`TrialExpiringAlert`); and on the pinned `2024-12-18.acacia` API version promotion codes use the flat `coupon` param, not the 2025-09-30 polymorphic shape.

**Permanently out of scope** (these are per-seat/metered billing wearing a different hat — principle 4): installment payments, per-seat quantity, adjustable quantity in the Portal, product variants, and proforma-invoice line-item builders. Stripe's native Entitlements API is also *not* adopted — it is boolean feature flags with no numeric field, so `max_projects: 3` has no representation there.

**Complete the end-to-end host verification**
- Re-run the full pass in a current starter app (host ≥ v1.54.0) at v1.4.3: install, `prisma migrate dev`, type-check both sides (confirms the v1.4.1/v1.4.2 fixes and the v1.4.3 `allowNoOrigin` registration), boot, complete a test-card checkout with `stripe listen`, verify webhook sync and the dunning/recovery transitions, exercise `/admin/billing` (Stripe config save, webhook connect refusal on localhost, entitlement config save).
- *Context:* the v1.4.0 pass on `veradai` surfaced the two upstream issues, both fixed in starter v1.54.0; this item closes the loop. Record the result on this page.

### Medium-term (upcoming)

**Migrate webhook verification to the host security helpers**
- Optionally replace the handler's Stripe-SDK verification (`stripe.webhooks.constructEvent`) with the host's `verifyTimestampedWebhookSignature({ payload: req.rawBody, header, secret })` from `@/features/security` (starter ≥ v1.54.0) — replay window and rotation candidates handled host-side, one less behaviour to own.
- *Context:* deliberately **not** bundled into v1.4.3: `constructEvent` is Stripe's official implementation and already enforces a 300-second replay tolerance and checks all rotation signatures in the header, so the practical gain is small — and the swap touches the payment-critical path, so it should be made against a live ≥ v1.54.0 host (verify the backend import path and parse the event JSON explicitly after verification). Minor release when it lands.

**Durable webhook event dedupe**
- Replace the in-memory `processedEvents` set with a small `billing_webhook_event` table (event ID + processed timestamp, pruned by age), making processing idempotent across instances and restarts.
- *Context:* clears the per-process dedupe known issue; keep the "mark handled only after success" ordering so Stripe retries still work.

**Full-erasure owner flow**
- `billingService.deleteStripeCustomerForTenant(tenantId)` exists but nothing calls it. Add a deliberate, confirmed owner action (likely on `/admin/billing` in a danger section) that deletes the Stripe Customer — removing invoices from Stripe — for "delete everything" requests, distinct from the default deletion path that retains tax records.
- *Context:* two-mode deletion is the agreed design: default = cancel + retain records ('preserved'); full erasure = explicit second action.

**`onUninstall` lifecycle hook**
- Offer a data export and optionally cancel the live Stripe subscription when the module is uninstalled from an app, so an uninstall can't silently leave a paying subscription running.
- *Context:* scaffold `lifecycle.onUninstall` is available; mirror the deletion handler's cancel logic.

**Adopt the `migrations` array for any breaking schema change**
- If the model-prefix rename (or the next re-key) goes ahead, ship it via the scaffold v1.18 per-module migration runner (`npx billing migrate-db`) with expand → backfill → contract steps rather than asking apps to hand-migrate.

### Long-term / Ideas

**Multi-member tenancy readiness**
- Billing is already keyed on `tenantId`, so when the host resolves shared tenants (an invited member/VA on the same account), members inherit the owner's plan with no schema change. Remaining work is UX: the member read-only state exists on `/settings/billing`; extend the same "managed by owner" treatment anywhere else billing surfaces.
- *Context:* do **not** build tenancy into billing itself — the host owns membership resolution.

**Dashboard widget**
- A small plan/trial-status widget via `dashboardRegistry` (plan name, status badge, trial days remaining, past-due warning). Deliberately not shipped yet; the hooks make it trivial for apps to build their own.

**Usage-based pricing, if it ever returns**
- Metered billing was retired in v1.3.0 and remains out of scope. If usage pricing is ever revisited, build on Stripe's current Billing Meters APIs (the legacy usage-record APIs the old implementation used are deprecated) — treat the old code as unrecoverable prior art.
