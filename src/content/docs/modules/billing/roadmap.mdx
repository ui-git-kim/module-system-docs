## Known Issues

### Webhook signature verification depends on host raw-body support

- **Description:** Stripe signature verification needs the raw request body. The handler reads `req.rawBody`, which the host must populate via an `express.json({ verify })` callback; the fallback to `req.body` fails verification safely (rejects) rather than accepting unsigned payloads. On a host without the verify callback, **all webhook deliveries fail signature verification** and subscription state never syncs.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (reads `rawBody`), `webhook.registration.ts` (documents the requirement); the fix belongs in the host's `app.ts` body-parser setup.
- **Severity:** High — **current starter versions do not provide the verify callback** (the upstream issue is still open, confirmed against host v1.52.1/v1.53.0 during the v1.4.0 end-to-end test), so webhook sync fails on an unmodified host until the workaround is applied or the starter fix ships.
- **Workaround:** In the host: `express.json({ verify: (req, _res, buf) => { (req as any).rawBody = buf; } })`. Logged as an open upstream starter issue — "No way to verify inbound third-party webhook signatures (e.g. Stripe HMAC-SHA256)" on the [Starter Template → Upstream Issues](/starter-template/upstream-issues/) page. Billing is forward-compatible: either proposed host fix enables verification with no module change.

### Webhook-sync end-to-end verification blocked; v1.4.1/v1.4.2 not re-run

- **Description:** A v1.4.0 end-to-end test **was** run on the `veradai` app (host v1.52.1 → v1.53.0) — it surfaced the two upstream starter issues logged on 2026-07-14 (raw-body/webhook verification, and `template:update` not propagating sub-package `type-check` scripts). What remains: the **webhook sync leg cannot pass end-to-end** until the host raw-body issue above is fixed (the handler rejects unverifiable deliveries by design), and the v1.4.1/v1.4.2 type-cleanliness fixes landed after that run and haven't been re-verified in the host.
- **Where:** The whole generated output; re-run in a current starter app once the raw-body fix ships (or with the manual `verify` callback applied).
- **Severity:** Medium
- **Workaround:** Apply the raw-body workaround from the issue above to test webhook sync locally; the rest of the module (checkout, portal, entitlements, admin page) is verifiable today.

### Module repo `CLAUDE.md` is stale

- **Description:** The module's own `CLAUDE.md` still describes v1.2.0: seat-based/metered billing endpoints (`/seats`, `/usage`), billing-model detection, `VITE_STRIPE_*` env vars, and the removed `@/features/user/components/settings/SettingsComponents` import path. Anyone (or any agent) reading it gets pre-redesign facts.
- **Where:** `CLAUDE.md` (module repo root, Module Overview section).
- **Severity:** Medium (misleads future development sessions; generated code is unaffected)
- **Workaround:** Treat `src/module.ts` and `src/templates/**` as the source of truth; this docs site is current.

### Prisma models use generic PascalCase names

- **Description:** `model Subscription` / `model BillingSettings` / `enum SubscriptionStatus` carry no module prefix. Convention for module schemas is snake_case, module-prefixed (`billing_subscription`) to avoid collisions in the host's shared `schema.prisma`. Renaming is a breaking migration for installed apps — it should ride the scaffold's `migrations` array (expand → backfill → contract) with `@@map` to preserve data.
- **Where:** `src/templates/schema.ts`; `schema.modelNames` in `src/module.ts`; every `prisma.subscription.*` / `prisma.billingSettings.*` call in the backend templates.
- **Severity:** Low (advisory — no collision reported; cost of fixing is high)
- **Workaround:** None required; avoid installing another module that defines a `Subscription` model.

### Webhook event dedupe is per-process

- **Description:** Duplicate Stripe deliveries are deduplicated via an in-memory set of the last 1,000 event IDs. Multiple backend instances (or a restart mid-delivery) can process the same event twice. The sync operations are upserts and status transitions, so double-processing is mostly harmless, but it is not guaranteed side-effect-free forever.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (`processedEvents`).
- **Severity:** Low (single-instance deployments are the norm for this product)
- **Workaround:** Run a single backend instance, or back the dedupe with a durable store (see Planned).

## Planned Improvements

### Short-term (next version)

**Refresh the module repo `CLAUDE.md`**
- Rewrite the Module Overview section to match v1.4.x: tenant-keyed `Subscription` + `BillingSettings`, no seats/metered, admin page config, current endpoint table, env vars down to `STRIPE_SECRET_KEY` (+ optional local `STRIPE_WEBHOOK_SECRET`), settings-blocks import path.
- *Context:* clears the "stale CLAUDE.md" known issue; pure documentation, no version bump needed for docs but the CLAUDE.md lives in the module repo so bump patch if templates are touched at the same time.

**Complete the end-to-end host verification**
- Re-run the full pass in a current starter app at v1.4.2: install, `prisma migrate dev`, type-check both sides (confirms the v1.4.1/v1.4.2 fixes), boot, complete a test-card checkout with `stripe listen`, verify webhook sync and the dunning/recovery transitions (needs the raw-body `verify` callback in the host — see Known Issues), exercise `/admin/billing` (Stripe config save, webhook connect refusal on localhost, entitlement config save).
- *Context:* the v1.4.0 pass on `veradai` surfaced the two upstream issues; this item closes the loop once the host raw-body fix (or workaround) is in place. Record the result on this page.

### Medium-term (upcoming)

**Action hooks on subscription transitions**
- Fire `actionRegistry` hooks from the webhook sync — e.g. `billing.subscription.updated { tenantId, previousStatus, status, planId }`, `billing.payment.failed { tenantId }`, `billing.subscription.canceled { tenantId }` — so other modules can react to plan changes without polling `getSubscription`.
- *Context:* the Integration page currently documents "no action hooks"; update it when this lands. Follow the cog-ingest hook-naming pattern (`module.noun.verb`), fire **after** the DB upsert commits, and use `addActionAsync` semantics.

**Durable webhook event dedupe**
- Replace the in-memory `processedEvents` set with a small `billing_webhook_event` table (event ID + processed timestamp, pruned by age), making processing idempotent across instances and restarts.
- *Context:* clears the per-process dedupe known issue; keep the "mark handled only after success" ordering so Stripe retries still work.

**Full-erasure owner flow**
- `billingService.deleteStripeCustomerForTenant(tenantId)` exists but nothing calls it. Add a deliberate, confirmed owner action (likely on `/admin/billing` in a danger section) that deletes the Stripe Customer — removing invoices from Stripe — for "delete everything" requests, distinct from the default deletion path that retains tax records.
- *Context:* two-mode deletion is the agreed design: default = cancel + retain records ('preserved'); full erasure = explicit second action.

**`onUninstall` lifecycle hook**
- Offer a data export and optionally cancel the live Stripe subscription when the module is uninstalled from an app, so an uninstall can't silently leave a paying subscription running.
- *Context:* scaffold `lifecycle.onUninstall` is available; mirror the deletion handler's cancel logic.

**Adopt the `migrations` array for any breaking schema change**
- If the model-prefix rename (or the next re-key) goes ahead, ship it via the scaffold v1.18 per-module migration runner (`npx billing migrate-db`) with expand → backfill → contract steps rather than asking apps to hand-migrate.

### Long-term / Ideas

**Multi-member tenancy readiness**
- Billing is already keyed on `tenantId`, so when the host resolves shared tenants (an invited member/VA on the same account), members inherit the owner's plan with no schema change. Remaining work is UX: the member read-only state exists on `/settings/billing`; extend the same "managed by owner" treatment anywhere else billing surfaces.
- *Context:* do **not** build tenancy into billing itself — the host owns membership resolution.

**Dashboard widget**
- A small plan/trial-status widget via `dashboardRegistry` (plan name, status badge, trial days remaining, past-due warning). Deliberately not shipped yet; the hooks make it trivial for apps to build their own.

**Usage-based pricing, if it ever returns**
- Metered billing was retired in v1.3.0 and remains out of scope. If usage pricing is ever revisited, build on Stripe's current Billing Meters APIs (the legacy usage-record APIs the old implementation used are deprecated) — treat the old code as unrecoverable prior art.
