## Known Issues

<Aside type="tip" title="2026-07-23 meta-audit тАФ all 20 findings fixed in v1.5.0">
A four-dimension audit (standards / production / performance / security) of the generated code ran on 2026-07-23 against module v1.4.3 and host v1.54.0. It confirmed **21 findings** via adversarial verification тАФ **20 distinct issues** after merging one duplicate pair (1 critical, 6 moderate, 13 minor). All were fixed in **v1.5.0** (see [Completed](#completed)), including the critical cross-tenant subscription-binding hole (`client-reference-id` is now HMAC-signed and verified in the webhook). The code fixes pass the module's wrapper checks and a full generated-code TS/TSX parse; **end-to-end host verification is still pending** тАФ see the item below.
</Aside>

### End-to-end host verification pending (now covers the v1.5.0 changes)

- **Description:** The generated code has not been type-checked or run in a host since **v1.4.0** (the `veradai` pass on host v1.52.1 тЖТ v1.53.0). The v1.4.1тАУv1.5.0 changes are verified only by the module's wrapper `type-check`, `check:templates`, `parity:check`, and a full generated-code TS/TSX **parse** тАФ not by a host build or runtime. **v1.5.0 specifically needs host verification** because it adds a schema column (`Subscription.lastEventAt`, so `prisma generate` must run before the new sync/`select` code type-checks) and reworks the checkout-attribution + webhook trust path.
- **Where:** The whole generated output; re-run in a current starter app (host тЙе v1.54.0) after `template:update` + `update-schema` + `prisma migrate dev`.
- **Severity:** Medium
- **Workaround:** None needed on тЙе v1.54.0 hosts. Verification steps: type-check both sides; boot; complete a test-card checkout with `stripe listen`; confirm the subscription syncs (and that a devtools-edited `client-reference-id` is now rejected, not honoured); exercise the dunning/recovery transitions and `/admin/billing`.

### Shared CLAUDE.md scaffold section documents a removed import path

- **Description:** The module's own Module Overview in `CLAUDE.md` was refreshed for v1.4.2 (2026-07-15), but the **scaffold-owned shared section** below it still documents settings components at `@/features/user/components/settings/SettingsComponents` тАФ a path that no longer exists in host v1.52+ (`@/components/settings-blocks` replaced it). The Module Overview carries a corrective note, but the stale text propagates to every module via scaffold updates.
- **Where:** The shared "Starter Template Components" section of `CLAUDE.md` тАФ scaffold-owned, fix belongs in the module starter's CLAUDE.md template.
- **Severity:** Low (the module-specific section corrects it inline)
- **Workaround:** Trust the Module Overview note; for this module's templates, `settings-blocks` is correct.

### Prisma models use generic PascalCase names

- **Description:** `model Subscription` / `model BillingSettings` / `enum SubscriptionStatus` carry no module prefix. Convention for module schemas is snake_case, module-prefixed (`billing_subscription`) to avoid collisions in the host's shared `schema.prisma`. Renaming is a breaking migration for installed apps тАФ it should ride the scaffold's `migrations` array (expand тЖТ backfill тЖТ contract) with `@@map` to preserve data.
- **Where:** `src/templates/schema.ts`; `schema.modelNames` in `src/module.ts`; every `prisma.subscription.*` / `prisma.billingSettings.*` call in the backend templates.
- **Severity:** Low (advisory тАФ no collision reported; cost of fixing is high)
- **Workaround:** None required; avoid installing another module that defines a `Subscription` model.

### Webhook event dedupe is per-process

- **Description:** Duplicate Stripe deliveries are deduplicated via an in-memory set of the last 1,000 event IDs. Multiple backend instances (or a restart mid-delivery) can process the same event twice. The sync operations are upserts and status transitions, so double-processing is mostly harmless, but it is not guaranteed side-effect-free forever.
- **Where:** `backend/src/features/billing/webhooks/stripe.webhook.ts` (`processedEvents`).
- **Severity:** Low (single-instance deployments are the norm for this product)
- **Workaround:** Run a single backend instance, or back the dedupe with a durable store (see Planned).

## Planned Improvements

### Short-term (next version)

**Phase 2 тАФ plan authoring on `/admin/billing` (v2.0.0, breaking)**
- Add a `BillingPlan` model holding **authoring intent only** (drafts, compare-at prices, mode-scoped product ids); Stripe stays the render source, read back via `products.list`/`prices.list`, so there is deliberately **no local price table and no drift workflow**. Price identity is the `lookup_key` (`transfer_lookup_key` on version), since Stripe already stores every price version forever and `Subscription.stripePriceId` already says which one each subscriber sits on.
- Provision **eagerly on Save**, never lazily at checkout тАФ the deliberate rejection of how Gravity Forms and Paid Memberships Pro both do it, whose provisioning bugs all surface at *checkout time*, in front of a customer, rather than on the admin page where they can be retried.
- Price edits are a **server-enforced `409 price_change_requires_confirmation`** carrying `affectedSubscribers`, not a UI nicety: create new price тЖТ move the lookup key тЖТ set `default_price` тЖТ archive old. Archiving never disturbs existing subscribers, so no admin edit can ever reprice a live customer.
- Publish writes plan limits straight into product metadata, so `BillingSettings.entitlements` demotes to the **free/no-subscription fallback only** (retitle that editor accordingly).
- Replace the Stripe Pricing Table with our own plan cards driving the existing `POST /checkout`; remove `stripePricingTableId` and `stripePublishableKey`, which become dead. Push the authored catalogue into the Customer Portal via `billing_portal/configurations` (never `is_default`), so plan changes for existing subscribers stay native.
- **Why the Pricing Table must go:** there is no `/v1/pricing_tables` resource тАФ it is Dashboard-only and capped at 4 products, so an API-created product can never appear in one. Confirmed 2026-07-23.

**Phase 3 тАФ mode safety and promotions (v2.1.0)**
- `copy-to-live` replay plus the blocking mode-mismatch banner (test and live objects never cross; this also fixes the latent key-swap hazard that already exists for `stripePricingTableId` / `stripeWebhookEndpointId`).
- Promotions: start with *select an existing Stripe coupon* + an `allow_promotion_codes` toggle, which delivers most of the value for a fraction of the surface. The full builder (reference-B style) is a second versioning UI on near-immutable Coupons. Two hard rules when it lands: "Free Trial" must write `trial_period_days`, **not** a 100%-off coupon (which still collects a payment method and yields `active`, not `trialing`, breaking `TrialCountdown`/`TrialExpiringAlert`); and on the pinned `2024-12-18.acacia` API version promotion codes use the flat `coupon` param, not the 2025-09-30 polymorphic shape.

**Permanently out of scope** (these are per-seat/metered billing wearing a different hat тАФ principle 4): installment payments, per-seat quantity, adjustable quantity in the Portal, product variants, and proforma-invoice line-item builders. Stripe's native Entitlements API is also *not* adopted тАФ it is boolean feature flags with no numeric field, so `max_projects: 3` has no representation there.

**Complete the end-to-end host verification**
- Re-run the full pass in a current starter app (host тЙе v1.54.0) at v1.4.3: install, `prisma migrate dev`, type-check both sides (confirms the v1.4.1/v1.4.2 fixes and the v1.4.3 `allowNoOrigin` registration), boot, complete a test-card checkout with `stripe listen`, verify webhook sync and the dunning/recovery transitions, exercise `/admin/billing` (Stripe config save, webhook connect refusal on localhost, entitlement config save).
- *Context:* the v1.4.0 pass on `veradai` surfaced the two upstream issues, both fixed in starter v1.54.0; this item closes the loop. Record the result on this page.

### Medium-term (upcoming)

**Action hooks on subscription transitions**
- Fire `actionRegistry` hooks from the webhook sync тАФ e.g. `billing.subscription.updated { tenantId, previousStatus, status, planId }`, `billing.payment.failed { tenantId }`, `billing.subscription.canceled { tenantId }` тАФ so other modules can react to plan changes without polling `getSubscription`.
- *Context:* the Integration page currently documents "no action hooks"; update it when this lands. Follow the cog-ingest hook-naming pattern (`module.noun.verb`), fire **after** the DB upsert commits, and use `addActionAsync` semantics.

**Migrate webhook verification to the host security helpers**
- Optionally replace the handler's Stripe-SDK verification (`stripe.webhooks.constructEvent`) with the host's `verifyTimestampedWebhookSignature({ payload: req.rawBody, header, secret })` from `@/features/security` (starter тЙе v1.54.0) тАФ replay window and rotation candidates handled host-side, one less behaviour to own.
- *Context:* deliberately **not** bundled into v1.4.3: `constructEvent` is Stripe's official implementation and already enforces a 300-second replay tolerance and checks all rotation signatures in the header, so the practical gain is small тАФ and the swap touches the payment-critical path, so it should be made against a live тЙе v1.54.0 host (verify the backend import path and parse the event JSON explicitly after verification). Minor release when it lands.

**Durable webhook event dedupe**
- Replace the in-memory `processedEvents` set with a small `billing_webhook_event` table (event ID + processed timestamp, pruned by age), making processing idempotent across instances and restarts.
- *Context:* clears the per-process dedupe known issue; keep the "mark handled only after success" ordering so Stripe retries still work.

**Full-erasure owner flow**
- `billingService.deleteStripeCustomerForTenant(tenantId)` exists but nothing calls it. Add a deliberate, confirmed owner action (likely on `/admin/billing` in a danger section) that deletes the Stripe Customer тАФ removing invoices from Stripe тАФ for "delete everything" requests, distinct from the default deletion path that retains tax records.
- *Context:* two-mode deletion is the agreed design: default = cancel + retain records ('preserved'); full erasure = explicit second action.

**`onUninstall` lifecycle hook**
- Offer a data export and optionally cancel the live Stripe subscription when the module is uninstalled from an app, so an uninstall can't silently leave a paying subscription running.
- *Context:* scaffold `lifecycle.onUninstall` is available; mirror the deletion handler's cancel logic.

**Adopt the `migrations` array for any breaking schema change**
- If the model-prefix rename (or the next re-key) goes ahead, ship it via the scaffold v1.18 per-module migration runner (`npx billing migrate-db`) with expand тЖТ backfill тЖТ contract steps rather than asking apps to hand-migrate.

### Long-term / Ideas

**Multi-member tenancy readiness**
- Billing is already keyed on `tenantId`, so when the host resolves shared tenants (an invited member/VA on the same account), members inherit the owner's plan with no schema change. Remaining work is UX: the member read-only state exists on `/settings/billing`; extend the same "managed by owner" treatment anywhere else billing surfaces.
- *Context:* do **not** build tenancy into billing itself тАФ the host owns membership resolution.

**Dashboard widget**
- A small plan/trial-status widget via `dashboardRegistry` (plan name, status badge, trial days remaining, past-due warning). Deliberately not shipped yet; the hooks make it trivial for apps to build their own.

**Usage-based pricing, if it ever returns**
- Metered billing was retired in v1.3.0 and remains out of scope. If usage pricing is ever revisited, build on Stripe's current Billing Meters APIs (the legacy usage-record APIs the old implementation used are deprecated) тАФ treat the old code as unrecoverable prior art.
