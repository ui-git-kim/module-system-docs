## Standards alignment audit — 2026-07-14

A full standards audit of the generated templates was run against module-starter scaffold **v1.18.0** (contract v1.2.0) and host starter **v1.52.1**. The module is already modern in its infrastructure: it ships the runtime module registration (`module.registration.ts`, starter v1.48+) correctly, the scaffold updater and `docs/` submodule are in place, `shadcnComponents` is complete, all templates carry the `FILE`/`PURPOSE` banner, backend imports use `.js` extensions, the schema is self-contained (`userId String`, no `@relation`), and the frontend uses the shared `@/lib/api` client. The gaps below were found during that audit.

<Aside type="caution">
The `module.registration.ts` runtime registration was verified byte-for-byte against the canonical scaffold template and is **correct** — no action needed there.
</Aside>

## Known Issues

Issues discovered but not yet resolved.

| Issue | Severity | Where | Notes |
|-------|----------|-------|-------|
| No `security.registration.ts` / deletion handler | High | The module stores per-user financial data (`Subscription` keyed by `userId`, plus Stripe customer/subscription ids) but registers no security manifest or account-deletion handler | On host account deletion / GDPR erasure the `Subscription` row (userId → Stripe customer mapping) is orphaned. Needs a `SECURITY_REGISTRATION` template deleting `Subscription` rows by `userId`, wired into `module.ts` + the backend `INDEX_TEMPLATE` |
| Frontend uses manual `useState`/`useEffect` fetching, not TanStack Query | High | `useSubscription`, `useEntitlements`, `SeatManagement`, `UsageDisplay`, `BillingSettingsPage` all fetch into component-local state | Current rule mandates TanStack Query with namespaced keys (`['billing', …]`). No request de-dup, shared cache, or invalidation today; multiple consumers fire duplicate requests |
| `console.*` used in backend templates | Medium | 11 `console.log`/`error`/`warn` calls in `src/templates/backend/index.ts`; zero `createModuleLogger` | Rule: no `console.*` in backend — use `createModuleLogger('billing')` from `../../lib/logger.js`, log-object-first, `audit: true` on security ops |
| Forbidden Tailwind utilities in generated components | Medium | ~13 clusters across `SubscriptionStatus`, `TrialCountdown`, `DunningBanner`, `TrialExpiringAlert`, `UsageDisplay`, `PromoCodeInput`, `SeatManagement`, `BillingSettingsPage` | Colour literals (`text-green-600`, `bg-red-50`, amber/blue) break theming; typography scale (`text-sm`) and `font-*` weights are design-system decisions. Migrate to semantic tokens (`text-muted-foreground`, `text-destructive`, `bg-muted`) and drop typography/weight classes |
| Stripe webhook returns HTTP 200 on internal processing failure | Medium | `stripeWebhookHandler` catch block returns `{ received: true }` with 200 even when a DB write threw | Stripe treats any 2xx as acknowledged, so transient failures (DB down, timeout) are never retried → subscription-state drift. Return 5xx for transient/processing errors |
| `status as any` casts in the webhook upsert | Medium | `syncSubscriptionFromWebhook` casts the mapped status `as any` in both create and update branches | A wrong status string compiles silently against the Prisma enum. Type `statusMap` to the generated `SubscriptionStatus` enum (from `@prisma/client`) instead of `as any` |
| Model/enum use generic PascalCase names | Medium | `model Subscription` / `enum SubscriptionStatus` have no module prefix; convention is snake_case, module-prefixed (`billing_subscription`) | Merged into the host's shared `schema.prisma`, an unprefixed `Subscription` risks collision. Rename to `billing_subscription` with `@@map("Subscription")` to preserve existing data + update `modelNames`. Breaking migration — weigh against installed apps |
| `SeatManagement` conflicts with the product vision | Medium | Module ships seat-based billing UI + detection, but the [Module Roadmap](/modules/roadmap/) states the system is single-tenant with **no per-seat billing** ("permanently out of scope") | Design-alignment question: keep seat support as an opt-in for non-standard hosts, or retire it. Needs a product decision before further seat work |
| `navigation.registration.ts` is an orphan placeholder | Low | Generated from `NAVIGATION_REGISTRATION` (comment-only, no registration call) and never imported by the frontend `INDEX_TEMPLATE` | Billing is surfaced via the settings-page replacement; delete the entry + export |
| 13 unused "placeholder" template exports | Low | `PLACEHOLDER EXPORTS` block in `frontend/index.ts` — none referenced by `module.ts` | Dead code; several would emit banner-less files if ever wired. Delete the block |
| `serviceRegistry` version hardcoded to `'1.0.0'` | Low | `SERVICE_REGISTRATION` passes a static version while the module is at 1.2.1 | Use `version: '{{moduleVersion}}'` (the pattern `MODULE_REGISTRATION` already uses) so it stays in sync |
| Redundant `@@index` on unique columns | Low | `@@index([userId])` / `@@index([stripeCustomerId])` duplicate the single-column `@unique` backing indexes | Remove both; `@unique` already indexes them |
| American spelling in user-facing text | Low | A few strings/comments in `frontend/index.ts` + one backend comment | Australian spelling required in prose/user-facing text |
| Stale version + install docs | Low | `CLAUDE.md` Module Overview says v1.2.0 (actual 1.2.1); this docs page's Overview describes a manual `cp -r` install rather than the CLI `npx billing init` flow | Refresh both |

## Planned Improvements

### Short-term (Next Version)

**Scaffold currency — v1.17.0 → v1.18.0**
- Merge the scaffold update (clean merge; shared history) to pick up contract **v1.2.0**, the per-module DB migration runner (`src/scaffold/migrations.ts`), and the `migrate-db` CLI command.
- *Context:* `npm run scaffold:update` — infrastructure-only merge; `src/templates/**` and `src/module.ts` stay module-owned.

**Security registration + deletion handler**
- Ship `security.registration.ts` declaring the `Subscription` data manifest and an idempotent account-deletion handler (`prisma.subscription.deleteMany({ where: { userId } })`).
- *Context:* mirror the scaffold's `SECURITY_REGISTRATION` template; wire into `module.ts` `backend.templates` and the backend `INDEX_TEMPLATE` side-effect imports.

**TanStack Query migration**
- Move reads to `useQuery` and writes to `useMutation` with namespaced keys via `createModuleQueryKeys('billing')`; invalidate on success instead of manual `refetch()`.

**Structure-only styling pass**
- Replace colour literals with semantic tokens and remove typography-scale / font-weight utilities across the eight affected components.

### Medium-term (Upcoming)

**Opt-in `migrations` field**
- Adopt the v1.18.0 `migrations` array (STEP 4b) when a schema change needs expand → backfill → contract (e.g. the `billing_subscription` rename). Not needed for a purely additive change (`prisma migrate dev` handles those).

**Lifecycle hooks**
- Consider `onUninstall` to offer a data export and optionally cancel live Stripe subscriptions / clean up external resources.

**Webhook robustness**
- Distinguish permanent vs transient webhook failures so Stripe retries transient errors; add structured audit logging on subscription-state transitions.
