## v1.6.0 — 2026-07-23

Guided owner setup and webhook verification — the first phase of moving plan setup onto `/admin/billing`. **Requires a schema migration** (two new nullable `BillingSettings` columns): run `npx billing update-schema` then `prisma migrate dev`. No breaking changes; the Stripe Pricing Table path is untouched.

<Aside type="tip" title="Existing installs: press Rebuild once">
Webhook endpoints created by earlier versions are subscribed to the original seven events, and **Stripe never backfills** an existing endpoint's event list. The admin page now diffs your live endpoint against the required list and offers a **Rebuild** action — v1.6.0 also adds `invoice.payment_succeeded`, which the handler has always processed but was never subscribed to.
</Aside>

**Added**
- **Guided setup checklist** on `/admin/billing` — connect Stripe → connect the webhook → verify → set up plans. Every step's state is derived from **live probes on each read, never a stored cursor**, so it can't wedge and it re-opens when a step regresses (endpoint deleted in the Dashboard, key swapped, events missing).
- **Account and mode banner** — shows which Stripe account the secret key actually points at, with a TEST/LIVE badge. The only affordance that catches "I'm configuring the wrong account".
- **Webhook verification** (`POST /admin/verify/start`, test mode only) — proves the pipeline with three independent assertions: the probe was created (your key works), **a signed delivery physically arrived and verified** (raw body, CORS/no-Origin and the signing secret are all correct), and the delivery was the probe's own subscription event. Previously a missing `req.rawBody`, a CORS rejection, a stale secret and a disabled endpoint all presented identically as silence. The probe uses a throwaway customer carrying **no tenant metadata**, so `syncSubscriptionFromWebhook` provably can't resolve a tenant for it and never touches your own subscription row.
- **Webhook health** — endpoint enabled/disabled, URL, API version, a missing-events diff against `REQUIRED_WEBHOOK_EVENTS`, "last delivery received", and a **Rebuild** action.
- **Read-only Stripe catalogue** on the admin page, with warnings the Stripe Dashboard itself never surfaces: no default price, no active price, `tax_behavior` unset (which silently degrades the automatic tax this module enables), `max_*`/`limit_*` metadata that fails to parse and is therefore silently ignored, and more than 10 active products (the Customer Portal plan-change ceiling).
- `BillingSettings.lastWebhookEventAt` / `lastWebhookEventType`, stamped on every signature-verified delivery **before** dispatch — that ordering is what separates "couldn't reach us" from "arrived but failed".

**Changed**
- `enabled_events` is now a single exported `REQUIRED_WEBHOOK_EVENTS` constant (gains `invoice.payment_succeeded`), so the health check can diff against it.
- The Stripe client sets `maxNetworkRetries: 2` to absorb transient 429s.
- Owner-initiated **writes** surface Stripe's own message as a 400 (`stripeToWriteError`) instead of the blanket "temporarily unavailable" used for member reads — an invalid URL or key is owner-actionable.
- Exported `invalidateBillingCaches()` so later phases can drop cached Stripe reads after a write.

**Design principle amended**
- Principle 2 was "use Stripe's native UI everywhere". It is now **"Stripe owns payment data, not the catalogue"**: card details and PCI scope stay with Stripe (Checkout, Customer Portal), while plan definitions and pricing may live in our admin. Corollary — use Stripe's native UI wherever Stripe provides a **provisionable** one. The Customer Portal qualifies; the **Pricing Table does not** (Dashboard-only, no API, capped at 4 products), which is why plan authoring will replace it rather than feed it.

## v1.5.0 — 2026-07-23

The 2026-07-23 meta-audit fixes — 20 findings across security, production readiness, performance, and standards. **Requires a schema migration** (new nullable `Subscription.lastEventAt` column): run `npx billing update-schema` then `prisma migrate dev` after updating.

<Aside type="caution" title="Upgrade note — checkout attribution now signed">
The embedded Pricing Table's `client-reference-id` is now a **signed** token (served by `GET /config`), not the raw user id, and the webhook only trusts a verified reference. Signing uses `SECURITY_ENCRYPTION_KEY` — already required for the admin webhook-connect flow, so production deployments have it. Without the key, first-time **pricing-table** checkouts fall back to Stripe customer-metadata resolution (the backend `/checkout` path is unaffected). See the roadmap Completed table for the full finding list.
</Aside>

**Security**
- **Fixed a cross-tenant subscription-binding hole (critical).** `client-reference-id` was the raw, user-editable user id and was trusted first when resolving the tenant in the webhook, so a user who knew another tenant's id could bind their checkout to that tenant. The reference is now HMAC-signed and verified; tenant resolution trusts a valid signed reference, then an existing row for the Stripe customer, then customer metadata — a raw/forged value never resolves a tenant. A tenant row is never rebound to a different Stripe customer without a signed reference.
- `GET /invoices` now floors the `limit` param (a negative value no longer reaches Stripe); invoice links open with `noopener,noreferrer`.

**Fixed**
- **Read resilience** — `getSubscription`/`getPlanDetails`/`getPlanLimits` no longer fail member reads on a Stripe outage: DB-backed status/dates are returned without plan enrichment, cached plans are served stale-on-error, and a Stripe error is translated to a 503 rather than leaking Stripe's own status (401/404/429) to the browser.
- **Out-of-order webhooks** — the sync is now monotonic (a new `lastEventAt` marker drops deliveries older than the last applied), so a delayed event can't revert newer subscription state.
- `handlePaymentFailed` only transitions active-ish rows, so a stray post-cancel `invoice.payment_failed` can't resurrect a `CANCELED` subscription to `PAST_DUE`.
- `connectWebhook` deletes the just-created Stripe endpoint if persisting its secret fails, so a DB error no longer orphans an unusable endpoint.
- **Admin page** — the two Save buttons are disabled until their source query has loaded, and load failures render an explicit error instead of an empty form, so a slow/errored load can no longer be saved over the tenant's Stripe config or entitlement options. Read-only status displays show "Checking…"/"Unavailable" instead of asserting "Not connected"/"No active plan" while loading.
- **Settings page** — the post-checkout success alert refreshes and briefly polls the subscription (and softens its copy) instead of claiming "now active" before the webhook lands; the portal button handles errors and disables while opening; payment-method and invoice loading/error states are distinct from empty; the success-param cleanup keeps other query params and replaces history.
- `StripePricingTable` shows a message when Stripe's script is blocked and distinguishes a config-fetch error from an unconfigured tenant.
- `useEntitlements.checkLimit` consults the server on an errored query instead of silently falling back to free-tier limits.

**Performance**
- Entitlement checks read the plan's Stripe product through a 5-minute cache instead of a live Stripe call on every `GET /entitlements` / `checkEntitlement`; the two independent reads in `getPlanLimits` now run concurrently.

**Standards / config**
- The Stripe SDK dependency is pinned (`stripe@^22.3.1`) instead of installing `@latest`; the `getStripe()` client + pinned `apiVersion` are single-sourced (webhook handler imports from the service); user-facing status labels use Australian spelling.

## v1.4.3 — 2026-07-16

Host requirement: **starter ≥ v1.54.0** (the release that captures `req.rawBody` app-wide and adds the `allowNoOrigin` route-registry flag — the generated webhook registration won't type-check on older hosts).

**Fixed**
- **Production webhook delivery** — `webhook.registration.ts` now passes `allowNoOrigin: true` to `registerPublic`. Stripe deliveries are server-to-server (no `Origin` header) and the starter's production CORS rejected them by default; the HMAC signature — not the Origin — authenticates the caller. Local development (`stripe listen`) was unaffected.

**Not changed**
- The handler keeps Stripe's own `constructEvent` verification (which already enforces a replay tolerance and checks rotation signatures). Migrating to the host's `verifyTimestampedWebhookSignature` helper remains an optional roadmap item.

**Fixed**

## v1.4.1 — 2026-07-14

**Fixed**
- Generated code is type-clean against the current host stack:
  - React 19 — the `stripe-pricing-table` JSX element is declared via the `react` module's `JSX` namespace (global `namespace JSX` no longer applies).
  - Current `stripe` SDK — the pinned API version is cast to `Stripe.LatestApiVersion` so the literal type check passes regardless of the SDK release installed by the host.
  - Current Prisma — subscription status writes use a typed local literal union that assigns cleanly into generated inputs.

## v1.4.0 — 2026-07-14

**Added**
- **Admin-automated webhook setup** — a "Set up webhook" button on `/admin/billing` creates the Stripe webhook endpoint (subscribed to exactly the handled events), captures the signing secret, and stores it **encrypted** (AES-256-GCM via the host's `encryptField`; requires `SECURITY_ENCRYPTION_KEY`). No `.env` edit, no restart. Refuses `localhost` with guidance to use the Stripe CLI locally.
- Webhook secret resolution is **DB-first, env-fallback** — `STRIPE_WEBHOOK_SECRET` is now optional and intended for local development only.
- **Non-secret Stripe config moved to the admin page** — publishable key, pricing table ID, and product ID are owner-managed in `BillingSettings`; the publishable key and pricing table ID are served to members via `GET /api/v1/billing/config`, while the product ID is only exposed on the owner-only `GET /api/v1/billing/admin/stripe-config`. The `VITE_STRIPE_*` frontend env vars are gone.
- Webhook connection status endpoint (`GET /admin/connection-status`) showing whether a secret is available and from where (`db` / `env` / `none`).

**Fixed**
- **Stripe API version drift** — the module pins `2024-12-18.acacia`, but webhook payloads arrive in the account's API version. The sync now reads `current_period_end` from the subscription **item** first (where `2025-03-31.basil` moved it) with a subscription-level fallback.

## v1.3.0 — 2026-07-14

The tenant-keyed redesign. One release, but it touched nearly every template.

**Changed (breaking)**
- **Schema re-keyed on `tenantId`** — `Subscription` is now `tenantId String @unique` (the host isolation key) with `userId` kept as the optional actor. A subscription belongs to the account, not a login.
- **Seat-based and metered billing retired** — `SeatManagement`, `UsageDisplay`, the `/seats` and `/usage` endpoints, and billing-model detection removed. Single-tenant product: one owner, one subscription, permanently out of scope.
- **Settings components import path fixed** — the settings page now imports from `@/components/settings-blocks` (host v1.52.1); the old `SettingsComponents` path no longer exists and the module previously failed to compile in a current host.

**Added**
- **`BillingSettings` model** and the **`/admin/billing` page** (via `adminPagesRegistry`) — owner-configurable entitlement options merged with Stripe product metadata at runtime.
- **Security registration** — data-ownership manifest plus an account-deletion handler (order 500) that **cancels the live Stripe subscription** and hard-deletes local rows, while retaining the Stripe Customer + invoices for tax records. A separate `deleteStripeCustomerForTenant` service method covers full erasure.
- Admin-only route split — checkout, portal, and all config writes now require `requireAdmin`; reads stay member-level for feature gating. Non-owner members get a read-only plan view on `/settings/billing`.
- `invoice.paid` recovery handling — `PAST_DUE` promotes back to `ACTIVE` on successful payment (never overriding trial/cancelled states).

**Changed**
- **TanStack Query migration** — `useSubscription`, `useEntitlements`, and both pages now use `useQuery`/`useMutation` with `createModuleQueryKeys('billing')` keys and invalidation on success.
- **Structured logging** — all backend `console.*` replaced with `createModuleLogger` (`'billing'`, `'billing-webhook'`, `'billing-security'`), `audit: true` on financial state changes.
- **Structure-only styling pass** — colour literals and typography-scale/font-weight utilities replaced with semantic tokens across all components.
- Webhook handler returns **5xx on processing failures** so Stripe retries transient errors, with in-memory event-ID deduplication; the typed status map removed the `as any` casts.
- `service.registration.ts` stamps `version: '{{moduleVersion}}'` instead of a hardcoded string; redundant `@@index` entries removed; Australian spelling in user-facing text.
- Scaffold merged v1.17.0 → v1.18.1 (module contract v1.2.0, per-module DB migration runner, `migrate-db` command).

## v1.2.0 and earlier — baseline

Initial feature set as installed by `npx billing init`:

- Stripe subscription management with trial support
- Stripe Pricing Table integration for plan selection and checkout (automatic tax enabled)
- Stripe Customer Portal for payment methods, invoices, and cancellation
- Webhook handler syncing Stripe events to the local `Subscription` table
- `useSubscription` and `useEntitlements` hooks
- Entitlements from Stripe product metadata (`max_*` / `limit_*`) with the `entitlementsRegistry`
- Trial countdown, trial-expiring alert, dunning banner, promo code input
- Settings page replacement at `/settings/billing`
- Seat management and metered usage display *(retired in v1.3.0)*
