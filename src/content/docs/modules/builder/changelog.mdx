## v1.25.1 — July 2026

**Current release.**

- **Variant badge on field templates.** A field template can carry a variant (a Radix-only `asChild`, say), but the **Add Field → Templates** list showed only its label and type, so you couldn't tell which variant you were about to add. Template entries now show a `radix` / `base` badge — **no badge means it applies to both**. Shown only for shadcn components, since a `primitive`/`plate`/`custom` component has no variant split.
  - This is the fast path for marking the catalogue: save `asChild` as a template once (Radix only), then add it to each of the 16 components that need it with the variant already set.
  - Adding a variant-marked template to a component with **no** variants strips the marker — otherwise the field would be silently hidden from every Base project for a component that has no such split.

## v1.25.0 — July 2026

**You can now mark which variant a shadcn field belongs to.**

- **Variant selector on native component fields.** The admin **Edit Field** dialog gains a **Variant** select — *Both* / *Radix only* / *Base only* — writing `PropertyDefinition.variant`, alongside the other code-shape controls (Code Placement / Code Alias / Code Format). The v1.24.0 marker existed but nothing could set it, so the catalogue stayed Radix-flavoured.
- **Only for shadcn components.** radix/base are variants *within* the shadcn library, so the selector appears only when the component's **Library** is `shadcn`. A `primitive` / `plate` / `custom` component has no such split — Typography, Div and Img are plain HTML either way, so their fields stay unmarked and always apply.
- **Visible at a glance.** Each field row shows a `radix only` / `base only` marker, and the Fields header summarises the split (e.g. *"2 both · 2 radix · 1 base"*), so a component's radix and base fields can be seen without opening each one.
- No persistence work was needed: `properties` is a Json column and `normalizeComponentProperties` spreads unknown keys, so `variant` rides straight through save → DB → `transformProperty` → `filterConfigByVariant`.

## v1.24.1 — July 2026

- **Naming: radix/base is a *variant within* the shadcn library, not a library of its own.** `library` is the **source** (`shadcn | primitive | plate | custom`); under `shadcn`, each component is published wrapping either Radix or Base UI. v1.24.0 named the marker `PropertyDefinition.library`, which put a property's `library` right next to its own component's `library` meaning something entirely different. Renamed: `UiLibrary` → **`ShadcnVariant`**, `PropertyDefinition.library` → **`variant`**, `CodeGenOptions.library` → `variant`, `filterConfigByLibrary` → `filterConfigByVariant`, `useActiveLibrary` → `useActiveVariant` (and its generated file). Behaviour is unchanged.
  - A **`primitive`** component (Typography, Div, Img) has **no variant** — it is plain HTML either way, so its properties stay unmarked and apply to both.
  - Deliberately untouched, being different axes: `ComponentLibrary` / `ComponentDefinition.library`, and `builder_component_set.library` (which holds radix|base and predates the distinction — it *is* the variant).

## v1.24.0 — July 2026

**Radix and Base UI are now distinct at the property level.**

- **Fix (code generation): base projects were emitting Radix props.** shadcn's components wrap either Radix or Base UI, and the wrapper passes props **straight through** to the primitive — so the two expose different APIs. Measured across the registry: **43 of 61** components wrap a different primitive (Radix `asChild` vs Base `render`; Accordion's Radix `type`/`collapsible` vs Base `openMultiple`), and only **17** are byte-identical. The catalogue is Radix-flavoured (16 components carry `asChild`) while `builder_component_set.library` has been per-project radix|base since v1.16.0 — so a base project generated `<Accordion type="single">`, which Base UI does not understand. This predates the importer.
- **`PropertyDefinition.library`.** A property can be marked `'radix'` or `'base'`; **undefined means it exists in both** (`className`, and everything on the 17 identical components). `properties` is a Json column, so there is **no schema migration**.
- **One filter point.** `generateNodeCode` filters the component config once through `filterConfigByLibrary`, and `resolveTag` / `resolveInnerContent` / `collectProps` all read the filtered set — no signature changes, nothing stops being pure, and the recursion already forwards options. Code generation, the live preview, the property panel and the JSX importer all apply it; the importer imports the **same** helper rather than reimplementing it, so import and code generation can never disagree about which properties exist.
- **`useActiveLibrary`.** Resolves the active library project-first: the project's component set → the app-wide global set → the `shadcn:builder-style` config. The component set wins because it is what the preview actually **compiles**, and generated code must match what is on screen (the style module's project-node field is *intent*, so it is deliberately not an input). While those queries load it returns undefined, which means "do not filter" — a slow query degrades to previous behaviour rather than briefly emitting the wrong props.
- **Import names the library.** An unmapped attribute now reads *"type is not a property of Accordion (base) — an admin can add it"*, and `ImportIssue` carries the library ready for admin promotion.

## v1.23.2 — July 2026

- **Fix (import): free text was dropped even with a text primitive registered.** The importer treated only a property explicitly marked `codePlacement: 'inner'` as a component's text slot. `resolveInnerContent` also falls back to the first `string`/`textarea` property for **dynamic-tag** (`$htmlTag`) and tagless components — and that fallback is precisely why Typography renders `<p>text</p>` without its text field being flagged. Typography was therefore rejected as a text carrier, and every `<AccordionContent>` body reported *"no text primitive is registered"*. The importer now mirrors `resolveInnerContent` exactly, **including the asymmetry that a static tag gets no fallback** — so text inside `<AccordionItem>` still becomes a node rather than landing in its plain-string `value` property.

## v1.23.1 — July 2026

- **Import maps icons.** An unresolved PascalCase tag now resolves to the `$icon` primitive when the snippet imports it from a known icon module (`lucide-react`, Tabler, Phosphor, Remix, Hugeicons, react-icons) — `<Heart className="size-4" />` becomes an Icon node with `name: "Heart"`. `resolveTag` special-cases `$icon` (the tag *is* the icon name, read from the node's `name` rather than a select), so the dynamic-tag lookup never matched it and icons were previously reported as unknown components. The **import source is the evidence**: a tag imported from anywhere else stays an `unknown-component` report line rather than being guessed into an icon, since a wrong icon is a silent mis-map while an unknown component is a visible, fixable one.

## v1.23.0 — July 2026

**Import a shadcn component straight into the canvas.**

- **Import block implemented.** The **Import block** toolbar action now opens a dialog: paste a component (or just its JSX) and it becomes an editable block tree — the preview and the normal **Save** flow then behave exactly as they do for a hand-built block. The action and its `builder:import-request` event already existed; the handler was an empty stub.
- **`shared/jsx-walker.ts` — the inverse of `tree-walker`.** The mapping is config-driven from the same registry fields code generation uses, so the two stay in step: `jsxTag` resolves the component, `codeAlias` reverses the attribute name (`type="single"` → `accordionType`), a `codePlacement: 'inner'` property takes a text child, and `codeFormat: 'array'` / `invertValue` / boolean shorthand reverse deterministically. A dynamic-tag component (`$htmlTag`) claims a matching HTML tag via its select options.
- **Free text becomes a node.** Text inside a container that declares no inner property (e.g. `AccordionContent`) is written into the registry's dynamic-tag text primitive (Typography) as a child, rather than dropped.
- **Nothing vanishes silently.** Unknown tags, attributes with no matching property, and runtime expressions (`{expr}`, `.map()`, spreads) cannot become nodes — they are collected and shown in a **mapping report** before you commit the import. Unmapped attributes carry their literal value and inferred type, ready to be promoted onto the native component.
- **Replace or insert.** The dialog chooses between replacing the canvas contents (the root keeps its own settings) and inserting into the selected container.
- `@babel/parser` is a new frontend dependency, lazy-loaded only when the dialog opens.

## v1.22.2 — July 2026

- **Fix (data loss): component import no longer blanks children, templates or palette order.** The component upsert built a single fully-defaulted payload and passed it as **both** halves, so any catalogue omitting `allowedChildren` / `defaultChildren` / `codeStructure` / `sortOrder` wrote the defaults over the existing values. The shipped `examples/shadcn-components.json` carries none of the four for any of its 286 components, so importing it silently wiped admin-authored child rules, default templates, code structure and palette order on every component that had them. Create keeps the defaults; update now passes values through so Prisma skips absent fields, while an explicit `[]` / `null` still clears deliberately.

## v1.22.1 — July 2026

- **Registers with the host module registry.** A generated `module.registration.ts` registers `{ id, name, version, depends }` with the starter's `moduleRegistry` (v1.48.0+), so the builder appears on `/admin/modules` with an `InstalledModule` row. Guarded dynamic import — a silent no-op on older starters.
- **`requiredModules` → `depends`.** Now `depends: { structure: '*' }`, per module contract v1.2.0.
- Scaffold updated to v1.18.0 (contract v1.2.0, `migrate-db` runner); orphaned `frontend/routes.ts` and empty `frontend/types.ts` removed.

## v1.22.0 — July 2026

- **Per-component source update / promote.** The admin **Native Components** table gains a **Check for source updates** button and a per-row source action. *Check* diffs every synced component's pristine source against the latest shadcn registry version and flags the ones that changed (with a count badge); the row's source button opens a side-by-side **diff** (current vs latest) with a **Promote** that overwrites the pristine file with the latest. This is the *update* half of the lifecycle — import only **adds** new components, so this is how you pull upstream changes into an existing one. New endpoints `POST /components/source-updates/check`, `GET /components/:type/source-diff`, `POST /components/:type/source-promote` (admin); service methods `checkComponentSourceUpdates` / `getComponentSourceDiff` / `promoteComponentSource`.
- **Fix (route shadowing):** `GET /components/global-set` was registered *after* `GET /components/:type` (which 404s unknown types), so the global-set route never fired and the frontend silently fell back. It is now registered before the param route.

## v1.21.0 — July 2026

**Imported components render in the preview with no manual sync.**

- **Import incrementally adds new components to the preview.** The preview's static component map globs the `ui/pristine/` folder, so on import the builder now writes any component **missing** from that folder into it (`POST /components/sync-pristine`, `skipExisting`) — existing files are never overwritten, so updates flow through a check/promote cycle instead of being clobbered. In production (read-only filesystem) the DB [global set](/modules/builder/integration/#global-component-set) covers this. The Import results panel reports how many **new** components were added.
- **Preview merges pristine + snapshot instead of replacing.** The compiled project/global snapshot is now **overlaid on** the pristine map (`{ ...uiComponentMap, ...compiled }`) rather than replacing it, and the global set likewise layers under a project snapshot. So a component present only in pristine — e.g. one just imported — renders even when a snapshot is active, and no per-project re-sync is needed. The import also invalidates the cached queries so the builder refetches fresh sources.

The post-import result is itemised: components saved (created / updated), field templates, preview sources synced, and any failures listed separately — **import** failures vs **source-fetch** failures (the latter render as `(preview unavailable)` placeholders until synced).

## v1.19.1 — July 2026

- **Import tolerates real-world component JSON.** The shadcn-components.json importer now accepts `import` as an alias for `importGroup` (the field the export tooling actually emits). Previously only `importGroup` was read, so every component's `importPath` came out `null` — which silently broke both the generated imports and the preview source-sync (`paletteShadcnComponentNames` derives the shadcn slug from `importPath`), so imported components like `attachment` showed as `(preview unavailable)`. A `select` property's `options` may now also be a plain string array (`["single", "multiple"]`), expanded to `{ label, value }` on import.

## v1.19.0 — July 2026

- **Global component set — imported components render everywhere, no project required.** Blocks are edited and saved as global or admin/system blocks too, so the preview can't depend on a per-project snapshot. A single app-wide **global component set** (a `builder_component_set` row under a reserved `projectId = "__global__"` — no schema migration) now provides fallback sources, and **component import automatically syncs it**: after the definitions upsert, the Import tab pulls the whole palette's shadcn sources into the global set and reports `Synced N component sources`. The preview resolves **project snapshot → global set → static pristine map**, so an imported shadcn component (e.g. `attachment`) renders for global/system/no-project blocks without pinning a project. New service methods `getGlobalComponentSet` / `syncGlobalComponentSet` / `paletteShadcnComponentNames`; routes `GET /components/global-set` (any authed user) and `POST /components/global-set/sync` (admin); hook `useGlobalComponentSet`. See [JSON Import & Export → Preview rendering](/modules/builder/json-reference/#preview-rendering).

## v1.18.3 — July 2026

- **Snapshot sync covers the whole palette:** the project **Component Sources → Sync** now pulls the standard preview set **plus every shadcn component the palette references** (derived from each `native_component.importPath` matching `…/ui/<slug>`), instead of only the fixed ~40 defaults. Imported shadcn components — e.g. `attachment` — now land in the project snapshot and render for real in the preview instead of showing as placeholders. Slugs that aren't fetchable from the registry are reported in the sync `errors` and skipped.

## v1.18.2 — July 2026

- **Preview resilience:** an unresolvable component in a block no longer blanks the whole preview. Previously, a component whose `jsxTag` was not in the active component set — a custom/native component with no synced source, or a shadcn component not yet synced — threw `X is not defined` and failed the entire preview compilation. The iframe preview now renders each such component as an isolated dashed **"(preview unavailable)"** placeholder (its children still render), so the rest of the block previews normally. Generated code is unchanged. New [JSON Import & Export → Preview rendering](/modules/builder/json-reference/#preview-rendering) documents the component-resolution model.

## v1.18.1 — July 2026

- **Category taxonomy reconciled:** `native_component.category` now lists `layout` (structural layout components — sidebars, panels, section wrappers) alongside `composite` (a block composed of more than one block) as valid values in the schema. The `features` summary reflects the palette groups that actually render — containers, elements, layouts, primitives. `layout` is wired (type, admin filter, "Layouts" palette group); surfacing `composite`, fleshing out layouts, and adding a "Blocks" palette group (drag saved blocks into your build) are tracked on the [roadmap](/modules/builder/roadmap/).
- **Documentation accuracy pass:** audited every builder doc page against the code and corrected the service-method table (dropped frontend-only methods from the backend list, fixed `importNativeComponents`/`findUserByHandle`/`promoteProjectComponentUpdate` return types, documented `getBlocks` filters), the action-hook count (10 — including `builder.component.saved`/`deleted`), the dependency list, the `builder_component_set` table, the `react-resizable-panels` v4 pin, and added coverage for the Security Registry, Component Sources tab, Layers panel, and the `check:templates` / `${` escaping guard. Refreshed the roadmap's Completed list (v1.14.0–v1.18.0).

## v1.18.0 — July 2026

- **Base-class manifest (unblocks the style module's per-instance class designer):** each project's [component snapshot](/modules/builder/integration/#base-class-manifest) now carries a derived `baseClasses` manifest — the baked-in Tailwind classes every component ships, keyed by `data-slot` — extracted from the vendored sources it already syncs. The style module consumes it to `twMerge(base, userClasses)` and warn on variant-group mismatches (e.g. `max-w-2xl` won't override the shipped `sm:max-w-sm`). Flat `cn("…")` slots yield a `base` string; `cva()` components (Button, Badge, Toggle, Tabs, …) also carry their `variants` + `defaultVariants` tables. Extraction is a dependency-free scan that runs on every sync/promote (and lazily backfills older snapshots); dynamic/template-literal classes are skipped rather than mis-parsed.
- **Schema:** new nullable `baseClasses Json?` column on `builder_component_set` (additive, no backfill migration needed — it is derived and regenerated on the next sync/promote).

## v1.17.1 — July 2026

- **Fix (preview theme):** the project's theme now reliably drives the canvas preview. The `/builder/canvas` iframe boots the whole app, so the host app's `global.css` `:root` also lives in it; the injected project theme is now re-appended as the **last** stylesheet in `<head>` so it wins the cascade (equal-specificity `:root` → source order decides) instead of the preview falling back to the app's default colours. `buildPreviewCss` also now normalises `theme_config.cssVars` — accepting keys with or without the `--` prefix, and either the `{ light, dark }` shape or a flat token map — so the project's variables are always emitted as valid custom properties.

## v1.17.0 — July 2026

- **Snapshot runtime renderer:** the preview now renders against the project's pinned [component snapshot](/modules/builder/integration/#per-project-component-snapshots) — completing the v1.16.0 feature. `IframeCanvas` forwards the snapshot to the iframe (`foundry:component-set`); `CanvasPage` compiles each source with Sucrase (`imports` transform + automatic JSX runtime) and executes it through a `require()` shim that resolves relative imports to other snapshot files (or a synthesised `./utils` `cn`) and non-relative imports from a curated `externalScope` (`radix-ui`, the `@base-ui/react/*` subpaths, `class-variance-authority`, `clsx`, `tailwind-merge`, `lucide-react`, `cmdk`, `vaul`, `react-day-picker`, `react-resizable-panels`, `input-otp`, …). Compilation is lazy, memoised and cycle-guarded, and a source that fails to compile is skipped in isolation. **The static pristine map remains the fallback when no snapshot is pinned**, so projects on radix and base render side-by-side against their real library + style.
- **Dependencies:** the frontend now declares `radix-ui`, `cmdk`, `vaul`, `react-day-picker` and `input-otp` (the packages synced components import) so the runtime `externalScope` resolves at build.
- **Fix:** icon-import injection no longer corrupts components whose last import is multi-line (e.g. `command`) — the injected import is now inserted ahead of the first import instead of splitting the last one.

## v1.16.0 — July 2026

- **Import overhaul:** the component import now reports **created vs updated** (it upserts by type — existing components are updated, new ones added). The Import tab gained an **Export catalogue** button, and the redundant Export/Import buttons were removed from the Native Components table toolbar (both live on the Import tab now).
- **Per-project component snapshots:** new `builder_component_set` table + backend/service/routes and a `useProjectComponentSet` hook — each project vendors its own timestamped snapshot of the shadcn component sources for its library + style. A **Component Sources** panel in the project's Blocks tab drives **Sync / Check for updates (diff) / Re-sync / Update files (promote)**, scoped by project ownership. *(Rendering the preview against the snapshot — the runtime component compiler — lands in a follow-up, once the style module writes the project's config field.)*

## v1.13.0 — July 2026

Full standards / production / performance / security audit remediation. Scaffold infrastructure updated v1.9.1 → v1.15.0 (adds the `check:templates` escaping guard, security integration, and template perf fixes).

### Security

- **Authorisation:** every app-wide builder mutation (native component CRUD/import, config, shared fields, builder style, icon library, pristine sync) now requires the admin role — previously any authenticated user could rewrite shared state.
- **Ownership:** block create/update/duplicate verify the target project belongs to the caller, and a client-supplied `isSystem` flag is stripped for non-admins — closing cross-user registry injection and privilege escalation.
- **Generated output:** block-tree strings interpolated into generated JSX and registry output are now escaped (the readable `"..."` form is kept for safe values); dynamic tag and icon names are validated as identifiers.
- **Preview sandbox:** the canvas preview rejects `postMessage` from any origin/source other than its same-origin parent — closing arbitrary-code execution via an external framer — targets outbound messages at the exact origin, and adds a `sandbox` attribute to the preview iframe.
- **Public registry:** constant-time token comparison, enumeration-safe `404`s, `no-store` caching on private responses, and a path-traversal guard on pristine component names.
- **Account deletion:** a `securityRegistry` manifest and deletion handler remove a deleted user's blocks, categories and collections.

### Fixes

- Resolved the PropertyPanel Rules-of-Hooks crash, the `DndProvider` and `ProjectBlocksWidget` strict-build TypeScript errors, the field-template editor dropping input focus on every keystroke, and drag-and-drop into a node's own subtree silently deleting it.
- The public registry now serves each block's already-generated code (it previously emitted empty components) and derives dependencies from the component catalogue; advertised global-block URLs resolve instead of returning `404`.
- A fresh install seeds a starter component palette and declares the `pickers` module dependency and the `REGISTRY_DOMAIN` environment variable, so `npx builder init` produces working, compilable code.

### Performance & cleanup

- Memoised the native-components context value and added a `staleTime` to the near-static catalogue query; added timeouts to the backend shadcn proxy fetches and input validation to backend mutation routes; removed dead code and stray debug logs.
