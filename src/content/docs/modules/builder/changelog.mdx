## v1.31.7 — July 2026

**Current release.**

**Launch-readiness fixes** (from the v1.31.6 production-beta audit — **zero launch blockers**; these are three should-fix-before-GA items):

- **`registry_handle` added to `module.ts` `schema.modelNames`.** The normal update + migrate path was already safe, but *uninstall* left the table orphaned and a *`--force`* reinstall produced a duplicate `model registry_handle` (Prisma P1012). The manifest now matches the schema.
- **Private-registry listing route hardened.** `GET /r/:projectSlug` now sets the same `Cache-Control` its per-block `.json` sibling already sets — `private, no-store` for private projects — so a URL-keyed shared/CDN cache can't store a token-gated private listing and replay it to anonymous clients.
- **Iframe preview handshake watchdog.** The canvas gated the whole preview on a single `foundry:ready` message; a failed lazy-chunk load, a crash before mount, or a lost message left the loading spinner up forever (only a manual page reload recovered). Now a 15s watchdog surfaces a recoverable error overlay with a **Reload preview** button, an `<iframe onError>` catches resource failures, and a late-but-successful `foundry:ready` still wins so a slow-but-working load self-corrects.

<Aside type="caution">
The audit's #1 finding — the public project-registry resolver returns 404 for every request under default config (it filters `nodeType: 'project'` while the seeded node-type slug is `projects`) — is **not** fixed here. It is the first task of the dedicated project-registry work, not a drive-by. See the [roadmap](/modules/builder/roadmap/).
</Aside>

## v1.31.5 – v1.31.6 — July 2026

**v1.31.0 meta-audit remediation, Batch E — R7/R8 standards** (two tranches, adversarially reviewed). The R8/R7 surface was far larger than the audit note implied — **79 verified sites**, not the ~10 estimated.

- **v1.31.5 — part 1 (mechanical/low-risk).** Every `Loader2 … animate-spin` → `Spinner`; both `window.confirm` → `AlertDialog`; the source-update amber colour literals → semantic tokens / `Badge` variants; 3 `api.*<any>` calls → named interfaces mirroring the backend returns; duplicate `ComponentLibrary`/`ComponentCategory` unions → imports from the canonical types module; `module.ts` route metadata synced with the routes registration; `spinner`/`empty`/`combobox`/`input-group` declared in `shadcnComponents`; 8 unused imports/props/state removed. Plus a **latent regression fixed**: the host's react-resizable-panels **v4** removed `autoSaveId`/`order`, so the builder's panel-size persistence had silently stopped and every panel prop failed to type-check — reimplemented with v4's `defaultLayout` + `onLayoutChanged` → localStorage. (Adversarial review caught a 2px loading-icon shrink in the first cut — the host `Button`'s `[&_svg:not([class*='size-'])]:size-4` rule was forcing the old spinners to 16px — fixed before ship.)
- **v1.31.6 — part 2 (R8 primitive restructures).** Raw `<button>` → `Button` across remove/reorder/toggle/search-clear controls and a clickable `Badge` (`asChild`); hand-rolled empty states → page-blocks `EmptyState` / `ui` `Empty`; hand-rolled error boxes → `ErrorAlert`; a duplicate `ShadcnItem` type folded into the service's `ShadcnRegistryItem`.

The interaction-changing conversions (search dropdowns → `Combobox`/`Command`, the sync `Table`, `ToggleGroup`/`Tabs`/`Switch`/`PageHeader`/`InputGroup`, and the ~190-occurrence R7 typography sweep) are deferred to a visual-review pass — see the [roadmap](/modules/builder/roadmap/#from-the-v1310-meta-audit).

## v1.31.1 – v1.31.4 — July 2026

**v1.31.0 meta-audit remediation, batches A–D** (adversarially reviewed per batch):

- **v1.31.1 — backend robustness/security (Batch A).** `duplicateBlock` copies the `variant` stamp, retries `-copy-2`… on a name collision (409 when exhausted, 404 for a missing block — both were 500s). Component-set sync refuses to overwrite a working snapshot with `files: {}` when every shadcn fetch failed (routes answer 502 with the errors). Block names are slug-validated at the API and defensively sanitised (with the project namespace) in the public registry JSON — a legacy row can no longer emit a path-traversing `files[].target`.
- **v1.31.2 — backend auth/cleanup + indexed registry handles (Batch B).** `POST /categories` and `POST /collections` now run the same `userOwnsProject` guard as the blocks routes. The `structure.node.deleted` cascade also removes the project's `builder_component_set` snapshot, in a transaction, with an `audit: true` log. The node-metadata enrichment skips the block count for non-project node types. **New `registry_handle` table**: the public `/r` resolver and `/check-handle` resolve via an indexed lookup instead of a per-request JSON scan of `user.preferences`; `PUT /handle` claims atomically with DB-level uniqueness and a reserved-name denylist; legacy preferences-only handles backfill lazily; rows orphaned by account deletion self-heal, and the security deletion cascade covers the table. **Hosts: run `npx builder update` + `npx prisma migrate dev`.**
- **v1.31.3 — frontend error handling (Batch C).** BlocksLibraryPage and ProjectBlocksTab render a real error state (with retry) instead of "No blocks yet" when a fetch fails; the hooks' `error` is fetch-only so a failed delete or a categories-only failure can't hide loaded data (deletes toast instead). Field-template save/create/delete get try/catch + error toasts; "Save Field" is disabled on an empty or duplicate key.
- **v1.31.4 — JSX import correctness (Batch D).** Mixed content keeps its source order (`<B/>text` no longer imports as `text<B/>`); fragment-root text/expression children are reported as `ImportIssue`s instead of vanishing, and a nested fragment at the root now splices through instead of importing as nothing; deleting an ancestor of the selected node resets the selection (fixed in the live `DndProvider` delete path and `useBuilder`).

## v1.31.0 — July 2026

**Layer taxonomy cleanup — the composition model now reads Component → Block → Collection → Layout.**

- **`block_collection.type` default renamed `component` → `collection`** (`collection | layout | page`). A collection is *more than one block*; typing it `component` was confusing. The field is a plain string and nothing branches on the value yet (collections aren't built), so this is a default + vocabulary change, not a behaviour change. Existing rows keep `component` until migrated — harmless, since no code reads the exact value.
- **`composite` removed from `native_component.category`.** With **Collection** as its own layer, "a component made of more than one block" *is* a Collection — the `composite` category was redundant, was unused in real catalogues, and was already absent from the frontend `ComponentCategory` union and admin filters. Categories are now `container | element | child | primitive | layout`.
- **The Block ↔ Collection boundary is count:** one composed unit is a Block (an "accordion in a card" is one Block), two or more is a Collection.
- **Documented the two meanings of "layout":** `native_component.category = 'layout'` is a structural container *component* (sidebar, scroll area); `block_collection.type = 'layout'` is a whole Layout *collection* (the app shell). Same word, two layers — like the three meanings of `library`, kept distinct on purpose.

<Aside type="note">
Foundation work ahead of building the **Collection** and **Layout** layers first-class, and ahead of the private project registry (which will bake `type` names into public install commands). See the [roadmap Direction](/modules/builder/roadmap/#direction).
</Aside>

## v1.30.0 — July 2026

**Synced snapshots render lucide icons through the starter's `IconRenderer` — and now type-check in the host.**

- **`resolveIconPlaceholders` emits `<IconRenderer icon="…" />` for lucide.** shadcn's registry serves icons as a library-agnostic `<IconPlaceholder lucide="ChevronDownIcon" … />`, resolved on sync. The lucide path now rewrites to `<IconRenderer icon="ChevronDown" … />` plus a single `import { IconRenderer } from "@/components/icon-picker"`, instead of a bare `<ChevronDownIcon />` per icon. IconRenderer is the starter's canonical lucide renderer, so this is one stable import per file rather than a growing list, and — the reason it matters — the snapshot **type-checks in the host**. The previous form bottomed out on a global `IconPlaceholder` with no ambient type, which the host's `tsc` reported as *Cannot find name 'IconPlaceholder'* on every pristine snapshot (45 errors in a representative project). The name is stripped of its trailing `Icon` because IconRenderer looks lucide up by the `icons`-map key (`ChevronDown`), the same strip the runtime fallback already did.

- **Scope and compatibility.** IconRenderer is lucide-only, so tabler / phosphor / remixicon keep their per-icon imports and hugeicons keeps its `HugeiconsIcon` wrapper — all already type-clean. The global `IconPlaceholder` fallback stays in place for snapshots synced before this change, so nothing breaks; the new form only appears on the next sync.

<Aside type="note" title="Takes effect on re-sync">
Existing pristine snapshots keep their old `IconPlaceholder` until re-synced. Run **Component Sources → sync** (or re-import) to rewrite them and clear the host `tsc` errors.
</Aside>

## v1.29.0 — July 2026

**A field template can finally be marked with a variant.**

- **Variant select on the Field Templates tab.** `FieldTemplateEditor` gained the same *Both / Radix only / Base only* control the component field editor has had since v1.25.0. Without it, a template created on that tab could never carry a variant — which broke the intended fast path for marking the catalogue ("save `asChild` as a template once, Radix only, then add it to each component with the variant already set"). The marker persisted through save → DB → template list perfectly well; there was simply nowhere to set it. Confirmed against a real export, where the `asChild` template had `variant` unset.

- **Variant badge on the template list.** A template's variant was invisible on that tab, so a marked template looked unmarked. Rows now show a `radix` / `base` badge — **no badge means it applies to both**, the same convention as the **Add Field → Templates** list.

## v1.28.0 — July 2026

**Blocks now record which shadcn variant they were built for.**

- **`block.variant`** (`'radix' | 'base'`, nullable). Null means **variant-agnostic** — the block touches no divergent prop and works under both. Named `variant` to match `PropertyDefinition.variant`, and deliberately **not** `library`: `native_component.library` already means `shadcn | primitive | plate | custom`, so a third `library` column with a different value domain would be a silent footgun.

- **Stamped on create, never re-derived on update.** `SaveBlockDialog` reads `useActiveVariant(projectId)` and stamps the result when a block is first saved. Re-saving deliberately leaves it alone: re-deriving from the active project would silently reclassify a Radix-built block as `base` the first time it is opened in a Base project, while its tree still holds Radix-only props. It changes only when set explicitly. Unrecognised values collapse to null rather than being stored, so a bad client can't write a value the Library would then compare against and hide everything.

- **A mark, not a filter.** The Blocks Library exposes `activeVariant` and marks blocks built for the other one; it does **not** hide them. A `radix`-stamped block still renders and is still usable in a Base project — most blocks touch no divergent prop at all, and silently emptying half the library with no explanation is worse than showing it. The filter is **client-side** by necessity: `useBlocksLibrary`'s system and global query keys are deliberately shared with `useProjectBlocks`, so a server-side `?variant=` param would poison that shared cache.

  Scope note: the Library lists system + global blocks and has **no project context**, so `activeVariant` resolves the global set / builder-style config — never a project's. `useActiveVariant(projectId)` is not expressible there.

<Aside type="caution" title="Schema change">
This adds a column, so an installed app needs `npx builder update` followed by `npx prisma migrate dev`. Until it migrates, saving a block will fail at runtime — the module's TypeScript cannot catch this, because the generated Prisma client is what knows the column set.
</Aside>

## v1.27.1 — July 2026

**Corrections found while scoping the catalogue-marking pass — two silent bugs and a prop name that no longer exists.**

- **`openMultiple` is not a real prop — it is `multiple`.** Base UI renamed `openMultiple` + `toggleMultiple` → `multiple` in v1.0.0-beta.4 (Oct 2025), and the name was stale in five places, including the **Base only** option in the field editor's Variant select — the one line you read *while* marking a field. Verified against the installed `@base-ui/react` (1.6.0), whose `AccordionRoot` and `ToggleGroup` both declare `multiple?: boolean` and neither of the old names. The rename also collapses two planned keys into one: accordion and toggle-group now use the **same** Base prop, so the mapping is uniform.

  Two related facts worth recording, both confirmed against the installed package rather than the docs: Base's accordion has **no `collapsible` at all** — it is a deletion, not a rename, and Base cannot express "single-open but not collapsible". And `AccordionValue<Value> = Value[]`, so Base's `defaultValue` is unconditionally an array while Radix declares it twice (`string` for `type="single"`, `string[]` for `"multiple"`) — which is exactly why the `defaultValue` + `defaultValueBase` pair from v1.26.1 is the right shape.

- **Fix: re-importing a component catalogue blanked `allowedChildren` on every component that omitted it.** `transformComponentsJson` passed `comp.children ?? []`, which defeats the deliberate merge guard in `componentUpdateData` — that guard relies on `undefined` so Prisma skips the column, and its own comment warns that absent-field defaults "blank those columns on every existing row, silently wiping admin-authored child rules". The `?? []` belongs on CREATE (where it still is) and is wrong on UPDATE. Since a catalogue only declares `children` on the components that actually restrict their children, every *other* component had its admin-authored child rules silently wiped on each re-import. The neighbouring `defaultChildren` already passed the raw value; this brings `allowedChildren` back in line.

- **Fix: the importer's inner-text mirror had drifted from the tree-walker on `$icon`.** `jsx-walker`'s `innerProperty` is a hand-copied mirror of `resolveInnerContent`, and its docblock says the rules "must match resolveInnerContent exactly or import and code generation disagree". They had stopped matching: `dynamicTagKey` slices any `$` prefix, so `$icon` produced a truthy tag key and reached the string/textarea fallback — but `resolveTag` classifies `$icon` as source `icon`, which takes the standard branch and never falls back. So the importer would assign free text to an Icon property that code generation would never emit. The mirror now excludes `$icon` the same way.

## v1.27.0 — July 2026

**The field editor now teaches the Code Alias pattern instead of assuming you know it.**

- **Duplicate-key warning.** Type a key another field already uses and the editor says so, explaining that the key is where the value is stored on a node, so two fields can't share one. For a shadcn component it names the fix: give this one its own key (`defaultValueBase`) and set **Code Alias** to the original so it still renders as that attribute. Advisory, not a block.
- **Variant hint.** Once a variant is chosen, a line covers the case that actually bites — *same prop name, different shape in the other variant*: Accordion's `defaultValue` is a string in Radix and an array in Base, so that pair is `defaultValue` + `defaultValueBase`, both aliased to `defaultValue`.

## v1.26.1 — July 2026

- **Fix: `codeFormat: 'array'` was ignored on every non-number field.** It only applied when `type === 'number'`; anything else fell through to a plain attribute, so the setting silently did nothing — even though the admin UI offers it as *"Array — wraps value in `{[value]}`"* with no type restriction. This blocked a case the variant work surfaces: **Radix's `Accordion` takes `defaultValue` as a bare string, Base's takes a `string[]`**, and the Base field couldn't be expressed at all (Array produced `defaultValue="shipping"` rather than `defaultValue={["shipping"]}`). The value is JSON-quoted inside the array — a raw interpolation would emit `{[shipping]}`, an undefined identifier. The importer already reversed the array form, so only generation was wrong; the round trip now closes.

  Modelling note: give each variant its **own key** and alias both to the same attribute — `defaultValue` (radix, string) and `defaultValueBase` (base, string + Array) both with `codeAlias: 'defaultValue'`. That mirrors the existing `accordionType` → `type` convention, keeps the two shapes stored separately, and avoids two properties sharing one key.

## v1.26.0 — July 2026

**Radix | Base tabs on a shadcn component's fields.**

- **The tab answers "what does a Base project see?"** Each sub-tab lists what a project of that variant really gets: the variant's own fields **plus the shared ones** — the same set `filterConfigByVariant` hands to code generation. A shared field like `className` appears under **both** tabs and is marked `both`; it is one field, edited from either.
- **Reorder moves by visible position.** A flat index swap would trade places with a hidden field of the other variant and appear to do nothing. Moves are now relative to the neighbouring *visible* field and applied to the underlying array, leaving the hidden variant's fields in their existing relative order.
- **Adding from a tab defaults to that variant** — a blank field added on the Radix tab starts as *Radix only*. A **template keeps its own variant** instead: it already knows what it is, so adding a shared template from the Radix tab must not silently make it radix-only.
- A `primitive` / `plate` / `custom` component has no variants, so it keeps a single plain list — no tabs, no `both` noise.

## v1.25.1 — July 2026

- **Variant badge on field templates.** A field template can carry a variant (a Radix-only `asChild`, say), but the **Add Field → Templates** list showed only its label and type, so you couldn't tell which variant you were about to add. Template entries now show a `radix` / `base` badge — **no badge means it applies to both**. Shown only for shadcn components, since a `primitive`/`plate`/`custom` component has no variant split.
  - This is the fast path for marking the catalogue: save `asChild` as a template once (Radix only), then add it to each of the 16 components that need it with the variant already set.
  - Adding a variant-marked template to a component with **no** variants strips the marker — otherwise the field would be silently hidden from every Base project for a component that has no such split.

## v1.25.0 — July 2026

**You can now mark which variant a shadcn field belongs to.**

- **Variant selector on native component fields.** The admin **Edit Field** dialog gains a **Variant** select — *Both* / *Radix only* / *Base only* — writing `PropertyDefinition.variant`, alongside the other code-shape controls (Code Placement / Code Alias / Code Format). The v1.24.0 marker existed but nothing could set it, so the catalogue stayed Radix-flavoured.
- **Only for shadcn components.** radix/base are variants *within* the shadcn library, so the selector appears only when the component's **Library** is `shadcn`. A `primitive` / `plate` / `custom` component has no such split — Typography, Div and Img are plain HTML either way, so their fields stay unmarked and always apply.
- **Visible at a glance.** Each field row shows a `radix only` / `base only` marker, and the Fields header summarises the split (e.g. *"2 both · 2 radix · 1 base"*), so a component's radix and base fields can be seen without opening each one.
- No persistence work was needed: `properties` is a Json column and `normalizeComponentProperties` spreads unknown keys, so `variant` rides straight through save → DB → `transformProperty` → `filterConfigByVariant`.

## v1.24.1 — July 2026

- **Naming: radix/base is a *variant within* the shadcn library, not a library of its own.** `library` is the **source** (`shadcn | primitive | plate | custom`); under `shadcn`, each component is published wrapping either Radix or Base UI. v1.24.0 named the marker `PropertyDefinition.library`, which put a property's `library` right next to its own component's `library` meaning something entirely different. Renamed: `UiLibrary` → **`ShadcnVariant`**, `PropertyDefinition.library` → **`variant`**, `CodeGenOptions.library` → `variant`, `filterConfigByLibrary` → `filterConfigByVariant`, `useActiveLibrary` → `useActiveVariant` (and its generated file). Behaviour is unchanged.
  - A **`primitive`** component (Typography, Div, Img) has **no variant** — it is plain HTML either way, so its properties stay unmarked and apply to both.
  - Deliberately untouched, being different axes: `ComponentLibrary` / `ComponentDefinition.library`, and `builder_component_set.library` (which holds radix|base and predates the distinction — it *is* the variant).

## v1.24.0 — July 2026

**Radix and Base UI are now distinct at the property level.**

- **Fix (code generation): base projects were emitting Radix props.** shadcn's components wrap either Radix or Base UI, and the wrapper passes props **straight through** to the primitive — so the two expose different APIs. Measured across the registry: **43 of 61** components wrap a different primitive (Radix `asChild` vs Base `render`; Accordion's Radix `type`/`collapsible` vs Base `openMultiple`), and only **17** are byte-identical. The catalogue is Radix-flavoured (16 components carry `asChild`) while `builder_component_set.library` has been per-project radix|base since v1.16.0 — so a base project generated `<Accordion type="single">`, which Base UI does not understand. This predates the importer.
- **`PropertyDefinition.library`.** A property can be marked `'radix'` or `'base'`; **undefined means it exists in both** (`className`, and everything on the 17 identical components). `properties` is a Json column, so there is **no schema migration**.
- **One filter point.** `generateNodeCode` filters the component config once through `filterConfigByLibrary`, and `resolveTag` / `resolveInnerContent` / `collectProps` all read the filtered set — no signature changes, nothing stops being pure, and the recursion already forwards options. Code generation, the live preview, the property panel and the JSX importer all apply it; the importer imports the **same** helper rather than reimplementing it, so import and code generation can never disagree about which properties exist.
- **`useActiveLibrary`.** Resolves the active library project-first: the project's component set → the app-wide global set → the `shadcn:builder-style` config. The component set wins because it is what the preview actually **compiles**, and generated code must match what is on screen (the style module's project-node field is *intent*, so it is deliberately not an input). While those queries load it returns undefined, which means "do not filter" — a slow query degrades to previous behaviour rather than briefly emitting the wrong props.
- **Import names the library.** An unmapped attribute now reads *"type is not a property of Accordion (base) — an admin can add it"*, and `ImportIssue` carries the library ready for admin promotion.

## v1.23.2 — July 2026

- **Fix (import): free text was dropped even with a text primitive registered.** The importer treated only a property explicitly marked `codePlacement: 'inner'` as a component's text slot. `resolveInnerContent` also falls back to the first `string`/`textarea` property for **dynamic-tag** (`$htmlTag`) and tagless components — and that fallback is precisely why Typography renders `<p>text</p>` without its text field being flagged. Typography was therefore rejected as a text carrier, and every `<AccordionContent>` body reported *"no text primitive is registered"*. The importer now mirrors `resolveInnerContent` exactly, **including the asymmetry that a static tag gets no fallback** — so text inside `<AccordionItem>` still becomes a node rather than landing in its plain-string `value` property.

## v1.23.1 — July 2026

- **Import maps icons.** An unresolved PascalCase tag now resolves to the `$icon` primitive when the snippet imports it from a known icon module (`lucide-react`, Tabler, Phosphor, Remix, Hugeicons, react-icons) — `<Heart className="size-4" />` becomes an Icon node with `name: "Heart"`. `resolveTag` special-cases `$icon` (the tag *is* the icon name, read from the node's `name` rather than a select), so the dynamic-tag lookup never matched it and icons were previously reported as unknown components. The **import source is the evidence**: a tag imported from anywhere else stays an `unknown-component` report line rather than being guessed into an icon, since a wrong icon is a silent mis-map while an unknown component is a visible, fixable one.

## v1.23.0 — July 2026

**Import a shadcn component straight into the canvas.**

- **Import block implemented.** The **Import block** toolbar action now opens a dialog: paste a component (or just its JSX) and it becomes an editable block tree — the preview and the normal **Save** flow then behave exactly as they do for a hand-built block. The action and its `builder:import-request` event already existed; the handler was an empty stub.
- **`shared/jsx-walker.ts` — the inverse of `tree-walker`.** The mapping is config-driven from the same registry fields code generation uses, so the two stay in step: `jsxTag` resolves the component, `codeAlias` reverses the attribute name (`type="single"` → `accordionType`), a `codePlacement: 'inner'` property takes a text child, and `codeFormat: 'array'` / `invertValue` / boolean shorthand reverse deterministically. A dynamic-tag component (`$htmlTag`) claims a matching HTML tag via its select options.
- **Free text becomes a node.** Text inside a container that declares no inner property (e.g. `AccordionContent`) is written into the registry's dynamic-tag text primitive (Typography) as a child, rather than dropped.
- **Nothing vanishes silently.** Unknown tags, attributes with no matching property, and runtime expressions (`{expr}`, `.map()`, spreads) cannot become nodes — they are collected and shown in a **mapping report** before you commit the import. Unmapped attributes carry their literal value and inferred type, ready to be promoted onto the native component.
- **Replace or insert.** The dialog chooses between replacing the canvas contents (the root keeps its own settings) and inserting into the selected container.
- `@babel/parser` is a new frontend dependency, lazy-loaded only when the dialog opens.

## v1.22.2 — July 2026

- **Fix (data loss): component import no longer blanks children, templates or palette order.** The component upsert built a single fully-defaulted payload and passed it as **both** halves, so any catalogue omitting `allowedChildren` / `defaultChildren` / `codeStructure` / `sortOrder` wrote the defaults over the existing values. The shipped `examples/shadcn-components.json` carries none of the four for any of its 286 components, so importing it silently wiped admin-authored child rules, default templates, code structure and palette order on every component that had them. Create keeps the defaults; update now passes values through so Prisma skips absent fields, while an explicit `[]` / `null` still clears deliberately.

## v1.22.1 — July 2026

- **Registers with the host module registry.** A generated `module.registration.ts` registers `{ id, name, version, depends }` with the starter's `moduleRegistry` (v1.48.0+), so the builder appears on `/admin/modules` with an `InstalledModule` row. Guarded dynamic import — a silent no-op on older starters.
- **`requiredModules` → `depends`.** Now `depends: { structure: '*' }`, per module contract v1.2.0.
- Scaffold updated to v1.18.0 (contract v1.2.0, `migrate-db` runner); orphaned `frontend/routes.ts` and empty `frontend/types.ts` removed.

## v1.22.0 — July 2026

- **Per-component source update / promote.** The admin **Native Components** table gains a **Check for source updates** button and a per-row source action. *Check* diffs every synced component's pristine source against the latest shadcn registry version and flags the ones that changed (with a count badge); the row's source button opens a side-by-side **diff** (current vs latest) with a **Promote** that overwrites the pristine file with the latest. This is the *update* half of the lifecycle — import only **adds** new components, so this is how you pull upstream changes into an existing one. New endpoints `POST /components/source-updates/check`, `GET /components/:type/source-diff`, `POST /components/:type/source-promote` (admin); service methods `checkComponentSourceUpdates` / `getComponentSourceDiff` / `promoteComponentSource`.
- **Fix (route shadowing):** `GET /components/global-set` was registered *after* `GET /components/:type` (which 404s unknown types), so the global-set route never fired and the frontend silently fell back. It is now registered before the param route.

## v1.21.0 — July 2026

**Imported components render in the preview with no manual sync.**

- **Import incrementally adds new components to the preview.** The preview's static component map globs the `ui/pristine/` folder, so on import the builder now writes any component **missing** from that folder into it (`POST /components/sync-pristine`, `skipExisting`) — existing files are never overwritten, so updates flow through a check/promote cycle instead of being clobbered. In production (read-only filesystem) the DB [global set](/modules/builder/integration/#global-component-set) covers this. The Import results panel reports how many **new** components were added.
- **Preview merges pristine + snapshot instead of replacing.** The compiled project/global snapshot is now **overlaid on** the pristine map (`{ ...uiComponentMap, ...compiled }`) rather than replacing it, and the global set likewise layers under a project snapshot. So a component present only in pristine — e.g. one just imported — renders even when a snapshot is active, and no per-project re-sync is needed. The import also invalidates the cached queries so the builder refetches fresh sources.

## v1.18.0 — July 2026

## v1.13.0 — July 2026

Full standards / production / performance / security audit remediation. Scaffold infrastructure updated v1.9.1 → v1.15.0 (adds the `check:templates` escaping guard, security integration, and template perf fixes).

### Security

- **Authorisation:** every app-wide builder mutation (native component CRUD/import, config, shared fields, builder style, icon library, pristine sync) now requires the admin role — previously any authenticated user could rewrite shared state.
- **Ownership:** block create/update/duplicate verify the target project belongs to the caller, and a client-supplied `isSystem` flag is stripped for non-admins — closing cross-user registry injection and privilege escalation.
- **Generated output:** block-tree strings interpolated into generated JSX and registry output are now escaped (the readable `"..."` form is kept for safe values); dynamic tag and icon names are validated as identifiers.
- **Preview sandbox:** the canvas preview rejects `postMessage` from any origin/source other than its same-origin parent — closing arbitrary-code execution via an external framer — targets outbound messages at the exact origin, and adds a `sandbox` attribute to the preview iframe.
- **Public registry:** constant-time token comparison, enumeration-safe `404`s, `no-store` caching on private responses, and a path-traversal guard on pristine component names.
- **Account deletion:** a `securityRegistry` manifest and deletion handler remove a deleted user's blocks, categories and collections.

### Fixes

- Resolved the PropertyPanel Rules-of-Hooks crash, the `DndProvider` and `ProjectBlocksWidget` strict-build TypeScript errors, the field-template editor dropping input focus on every keystroke, and drag-and-drop into a node's own subtree silently deleting it.
- The public registry now serves each block's already-generated code (it previously emitted empty components) and derives dependencies from the component catalogue; advertised global-block URLs resolve instead of returning `404`.
- A fresh install seeds a starter component palette and declares the `pickers` module dependency and the `REGISTRY_DOMAIN` environment variable, so `npx builder init` produces working, compilable code.

### Performance & cleanup

- Memoised the native-components context value and added a `staleTime` to the near-static catalogue query; added timeouts to the backend shadcn proxy fetches and input validation to backend mutation routes; removed dead code and stray debug logs.
