The Block Builder's component palette is a database table (`native_component`) of component definitions. You can **bulk import** those definitions from JSON — to seed a project, share a palette between environments, or version-control your components.

Import happens on the **Admin → Block Builder → Import** tab (paste JSON or upload a file), or via the API. The Import tab **auto-detects** which of the two supported formats you pasted, shows a summary, and reports created/updated counts.

<Aside type="note">
All import endpoints are **admin-only** and mounted under `/api/v1/builder`. Import is an **upsert keyed by `type`** — a component whose `type` already exists is **updated**, a new `type` is **created**. The result is `{ imported, created, updated, errors }`, where `imported = created + updated` and `errors` holds a per-row message for any component that failed (the rest still import).
</Aside>

<Aside type="caution">
**Duplicate palette items?** Because import upserts by `type`, entries with the **same `type`** collapse into one. But two entries with **different `type`s and the same `label`** appear as **two** palette items — the usual cause of an apparently duplicated component (e.g. `alert-dialog` vs `alertdialog`, or a stray second entry with the same label). De-duplicate by both `type` and `label` before importing.
</Aside>

## Two supported formats

| Format | Top-level shape | `components` is | Endpoint |
|--------|-----------------|-----------------|----------|
| **Builder export** | `{ components: [...], sharedFields?: {...} }` (a bare array also works) | an **array** of full component objects | `POST /components/import` |
| **shadcn-components.json** | `{ imports: {...}, components: {...} }` | an **object keyed by `type`** | `POST /components/import-json` |

The **Builder export** format is the one the **Export** button produces (round-trips cleanly — server-managed `id`s are stripped on re-import), and is the recommended format for hand-authoring. The **shadcn-components.json** format is a convenience shape that shares one import path across many components via named `imports` groups.

## Builder export format

The **Export** button produces this shape, wrapped with a `version` and `exportedAt` timestamp (both optional on import and ignored if absent):

```json
{
  "version": "1.0",
  "exportedAt": "2026-07-09T10:30:00.000Z",
  "components": [
    {
      "type": "button",
      "category": "element",
      "label": "Button",
      "icon": "MousePointerClick",
      "jsxTag": "Button",
      "importPath": "@/components/ui/button",
      "library": "shadcn",
      "sortOrder": 10,
      "properties": [
        { "key": "label", "label": "Label", "type": "string", "defaultValue": "Button", "codePlacement": "inner" },
        {
          "key": "variant",
          "label": "Variant",
          "type": "select",
          "defaultValue": "default",
          "options": [
            { "label": "Default", "value": "default" },
            { "label": "Secondary", "value": "secondary" },
            { "label": "Outline", "value": "outline" },
            { "label": "Destructive", "value": "destructive" }
          ]
        },
        { "key": "size", "label": "Size", "type": "select", "defaultValue": "default", "options": [
          { "label": "Default", "value": "default" },
          { "label": "Small", "value": "sm" },
          { "label": "Icon", "value": "icon" }
        ] },
        { "key": "className", "label": "Class Name", "type": "string" }
      ]
    },
    {
      "type": "input",
      "category": "element",
      "label": "Input",
      "icon": "TextCursorInput",
      "jsxTag": "Input",
      "importPath": "@/components/ui/input",
      "selfClosing": true,
      "sortOrder": 20,
      "properties": [
        { "key": "placeholder", "label": "Placeholder", "type": "string" },
        { "key": "type", "label": "Type", "type": "string", "defaultValue": "text" }
      ]
    },
    {
      "type": "card",
      "category": "container",
      "label": "Card",
      "icon": "SquareStack",
      "jsxTag": "Card",
      "importPath": "@/components/ui/card",
      "isContainer": true,
      "allowedChildren": ["card-header", "card-content", "card-footer"],
      "sortOrder": 60,
      "properties": [
        { "key": "className", "label": "Class Name", "type": "string" }
      ]
    }
  ],
  "sharedFields": {
    "className": { "key": "className", "label": "Class Name", "type": "string" }
  }
}
```

<Aside type="tip">
The outer object is optional — you can import a **bare array** of component objects (`[ { "type": "button", ... } ]`). Wrap it as `{ "components": [...], "sharedFields": {...} }` when you also want to import reusable field templates (`sharedFields`).
</Aside>

### Component fields

| Field | Type | Required | Default | Description |
|-------|------|:--------:|---------|-------------|
| `type` | string | ✅ | — | Unique component key (e.g. `button`). **Import upserts by this value.** |
| `category` | string | ✅ | — | `container` \| `element` \| `layout` \| `composite` \| `child` \| `primitive` (see [note below](#category-values)) |
| `label` | string | ✅ | — | Palette display name |
| `jsxTag` | string | ✅ | — | React component name or HTML tag emitted in generated code (e.g. `Button`, `div`) |
| `library` | string | | `shadcn` | `shadcn` \| `primitive` \| `plate` \| `custom` |
| `description` | string | | — | Optional palette description |
| `icon` | string | | — | Lucide icon name shown in the palette |
| `importPath` | string | | — | Import source for generated code (e.g. `@/components/ui/button`); omit for HTML primitives |
| `parentType` | string | | — | Parent component `type` — set on child components (e.g. `card-header` → `card`) |
| `selfClosing` | boolean | | `false` | Emit as `<Tag />` with no children |
| `isContainer` | boolean | | `false` | Can hold child nodes on the canvas |
| `acceptsAny` | boolean | | `false` | Accepts any child type (otherwise limited to `allowedChildren`) |
| `allowedChildren` | string[] | | `[]` | Whitelist of child component `type`s |
| `defaultChildren` | array | | — | Child nodes auto-inserted when the component is dropped on the canvas |
| `properties` | array | | `[]` | Editable property fields (see below) |
| `codeStructure` | object | | — | Code-generation hints (e.g. a conditional wrapper) |
| `sortOrder` | number | | array index | Palette ordering within its category |

### Property fields

Each entry in a component's `properties` array is an editable field shown in the property panel and (optionally) emitted into generated code.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| `key` | string | ✅ | Prop name — becomes a JSX attribute, inner content, or is hidden (see `codePlacement`) |
| `label` | string | ✅ | Field label in the property panel |
| `type` | string | ✅ | One of the [property types](#property-types) below |
| `defaultValue` | string | | Default value; also used as the emitted value when the user hasn't set one |
| `placeholder` | string | | Input placeholder |
| `description` | string | | Helper text under the field |
| `options` | `{ label, value }[]` or `string[]` | | Choices for a `select` field; plain strings are expanded to `{ label: s, value: s }` on import |
| `codePlacement` | string | | `attribute` (default — a JSX attribute) \| `inner` (content between the tags) \| `hidden` (not emitted) |
| `codeAlias` | string | | JSX attribute-name override (e.g. `htmlId` → `id`) |
| `codeFormat` | string | | `default` \| `array` (wraps the value as `{[value]}` — works for any type, not just numbers) |
| `invertValue` | boolean | | Emit a boolean value inverted |
| `variant` | string | | `radix` \| `base` — the [shadcn variant](#radix--base-variants) this prop belongs to. **Omit when it applies to both** |
| `isRequired` / `isUnique` | boolean | | Validation hints |
| `richText` | boolean | | Enable rich-text editing for the field |
| `classNameKey` | string | | Companion className prop key (used by `icon` fields) |

### Radix / Base variants

shadcn publishes each component wrapping either **Radix** or **Base UI**, and the wrapper passes props straight through — so the two accept different props. Of shadcn's 61 components, **43 wrap a different primitive** and only 17 are identical.

`variant` marks which one a property belongs to. Leave it out and the property applies to both — that's the right answer for `className`, for every `library: "primitive"` component, and for the 17 identical ones.

```json
{
  "type": "accordion",
  "library": "shadcn",
  "jsxTag": "Accordion",
  "properties": [
    { "key": "accordionType", "label": "Type", "type": "select", "codeAlias": "type", "variant": "radix" },
    { "key": "collapsible", "label": "Collapsible", "type": "boolean", "variant": "radix" },
    { "key": "multiple", "label": "Multiple", "type": "boolean", "variant": "base" },
    { "key": "className", "label": "Class Name", "type": "className" }
  ]
}
```

Code generation, the live preview, the property panel and the JSX importer all drop properties belonging to the other variant. The active variant resolves from the project's component set, falling back to the global set and then the `shadcn:builder-style` config.

<Aside type="tip">
**Same prop, different shape?** Give each variant its own `key` and alias both to the shared attribute. Radix's Accordion takes `defaultValue` as a string; Base's takes an array — so that pair is `defaultValue` (`variant: "radix"`) and `defaultValueBase` (`variant: "base"`, `codeFormat: "array"`), both with `codeAlias: "defaultValue"`. A `key` is where the value is stored on a node, so two properties must never share one.
</Aside>

<Aside type="caution">
Because code generation falls back to a property's `defaultValue`, a field with `codePlacement: "inner"` and a `defaultValue` (e.g. Button's `label` default `"Button"`) will emit that text as the element's children even before the user edits it — e.g. `<Button>Button</Button>`.
</Aside>

### Property types

The `type` field controls how a property is edited in the property panel:

| `type` | Editor | Notes |
|--------|--------|-------|
| `string` | Single-line text input | |
| `textarea` | Multi-line text input | |
| `number` | Number input | |
| `boolean` | Switch | Pair with `invertValue` to emit the value inverted |
| `select` | Dropdown | Requires `options` (string array or `{ label, value }[]`) |
| `icon` | Lucide icon picker | Use `classNameKey` to name a companion className prop for the icon |
| `id` | Text input | Typically paired with `codeAlias: "id"` to emit an `id` attribute |
| `className` | Class-name input | The per-instance Tailwind classes (consumed by the style module) |

### sharedFields

`sharedFields` is an optional object map of reusable property templates (`{ [name]: propertyDefinition }`) using the same shape as a property field. They let you define a field once and attach it to many components from the admin UI. Only present in the **Builder export** format.

### Complete component (every field)

A single component with all fields populated, for reference:

```json
{
  "type": "table-cell",
  "library": "shadcn",
  "category": "child",
  "label": "Table Cell",
  "description": "A single cell in a table row",
  "icon": "Square",
  "jsxTag": "TableCell",
  "importPath": "@/components/ui/table",
  "parentType": "table-row",
  "selfClosing": false,
  "isContainer": true,
  "acceptsAny": true,
  "allowedChildren": [],
  "defaultChildren": [
    { "type": "label", "props": { "text": "Cell" } }
  ],
  "properties": [
    {
      "key": "colSpan",
      "label": "Column Span",
      "type": "number",
      "defaultValue": "1",
      "placeholder": "1",
      "description": "How many columns this cell spans",
      "codePlacement": "attribute",
      "codeFormat": "default",
      "isRequired": false
    },
    { "key": "className", "label": "Class Name", "type": "string" }
  ],
  "sortOrder": 70
}
```

`defaultChildren` entries are either a bare `type` string (shorthand) or an object `{ "type": string, "props"?: {...}, "children"?: [...] }` — so `["label"]` and `[{ "type": "label", "props": { "text": "Cell" } }]` are both valid, and children can nest. The `props` keys are that component's own property **keys** (the `label` component stores its text under `text`, not `label`), and unique properties are generated rather than set — see [the worked example](#worked-example). `codeStructure` is an advanced object of code-generation hints (e.g. a conditional wrapper) and can be omitted.

## shadcn-components.json format

This shape keys `components` by `type` (an object, not an array) and shares import paths through named `imports` groups. Each component references a group via `import` (or `importGroup` — both are accepted), and uses `parent`/`children` instead of `parentType`/`allowedChildren`.

<a id="worked-example"></a>
The Accordion family below is the useful worked example, because it exercises everything at once: **`children`** (what may be nested), **`defaultChildren`** (the default template dropped on the canvas), and **variant-marked properties** for both Radix and Base.

```json
{
  "imports": {
    "ui-accordion": {
      "path": "@/components/ui/accordion",
      "components": ["Accordion", "AccordionItem", "AccordionTrigger", "AccordionContent"]
    },
    "ui-label": { "path": "@/components/ui/label", "components": ["Label"] }
  },
  "components": {
    "accordion": {
      "category": "container",
      "label": "Accordion",
      "description": "Vertically stacked set of interactive headings that reveal content.",
      "icon": "ListCollapse",
      "jsxTag": "Accordion",
      "import": "ui-accordion",
      "isContainer": true,

      "children": ["accordion-item"],

      "defaultChildren": [
        {
          "type": "accordion-item",
          "children": [
            { "type": "accordion-trigger", "props": { "text": "Is it accessible?" } },
            {
              "type": "accordion-content",
              "children": [
                { "type": "label", "props": { "text": "Yes. It adheres to the WAI-ARIA design pattern." } }
              ]
            }
          ]
        }
      ],

      "properties": [
        { "key": "accordionType", "label": "Type", "type": "select", "options": ["single", "multiple"], "defaultValue": "single", "codeAlias": "type", "isRequired": true, "variant": "radix" },
        { "key": "collapsible", "label": "Collapsible", "type": "boolean", "defaultValue": true, "variant": "radix" },
        { "key": "defaultValue", "label": "Default Open Item", "type": "string", "variant": "radix" },

        { "key": "multiple", "label": "Multiple", "type": "boolean", "defaultValue": false, "variant": "base" },
        { "key": "defaultValueBase", "label": "Default Open Items", "type": "string", "codeAlias": "defaultValue", "codeFormat": "array", "variant": "base" },

        { "key": "className", "label": "Class Name", "type": "className" }
      ]
    },

    "accordion-item": {
      "category": "child",
      "parent": "accordion",
      "label": "Accordion Item",
      "jsxTag": "AccordionItem",
      "import": "ui-accordion",
      "isContainer": true,
      "children": ["accordion-trigger", "accordion-content"],
      "properties": [
        { "key": "value", "label": "Value", "type": "string", "isRequired": true, "isUnique": true },
        { "key": "className", "label": "Class Name", "type": "className" }
      ]
    },

    "accordion-trigger": {
      "category": "child",
      "parent": "accordion",
      "label": "Accordion Trigger",
      "jsxTag": "AccordionTrigger",
      "import": "ui-accordion",
      "isContainer": true,
      "properties": [
        { "key": "text", "label": "Text", "type": "string", "defaultValue": "Accordion trigger", "codePlacement": "inner" },
        { "key": "className", "label": "Class Name", "type": "className" }
      ]
    },

    "accordion-content": {
      "category": "child",
      "parent": "accordion",
      "label": "Accordion Content",
      "jsxTag": "AccordionContent",
      "import": "ui-accordion",
      "isContainer": true,
      "acceptsAny": true,
      "properties": [
        { "key": "className", "label": "Class Name", "type": "className" }
      ]
    },

    "label": {
      "category": "element",
      "label": "Label",
      "description": "Accessible label for a form control.",
      "icon": "Tag",
      "jsxTag": "Label",
      "import": "ui-label",
      "properties": [
        { "key": "text", "label": "Text", "type": "string", "defaultValue": "Label", "codePlacement": "inner" },
        { "key": "htmlFor", "label": "For (input id)", "type": "string" },
        { "key": "className", "label": "Class Name", "type": "className" }
      ]
    }
  }
}
```

### What each part does

**`children` vs `defaultChildren`.** `children` is the *whitelist* — which types may be nested (it maps to `allowedChildren`). `defaultChildren` is the *template* — the tree actually created when the component is dropped on the canvas. They are independent: `accordion` permits only `accordion-item`, and its template supplies one, pre-filled.

**Nesting and `props`.** A `defaultChildren` entry is either a bare type string (`"accordion-item"`) or an object `{ "type", "props"?, "children"? }`. `props` are spread onto the new node, and `children` nests to build a whole subtree in one go. When an entry omits `children`, that type's *own* `defaultChildren` are used instead — so templates compose rather than repeat.

**Text lives where the config says it lives.** `accordion-trigger` carries a `text` property marked `codePlacement: "inner"`, so its label is a **prop**. `accordion-content` declares **no** inner property at all — it is `acceptsAny`, so its body has to be a real child *node* (a `label` here). Copying a `text` prop onto `accordion-content` would silently do nothing.

<Aside type="caution" title="Don't hard-code a unique value">
`accordion-item.value` is `isUnique`, so the builder generates it (`item-a1b2`) on every node it creates — **after** `props` are applied. Setting `"props": { "value": "item-1" }` in a template looks right and is silently overwritten. Leave unique properties out of `defaultChildren`.
</Aside>

### The Radix and Base fields

Both variants' fields live on the **same** component, each tagged with `variant`. `filterConfigByVariant` hands code generation only the fields matching the project's variant, so a Base project never emits `type`/`collapsible` and a Radix project never emits `multiple`. An **untagged** property (`className`) applies to both and shows under either tab.

| Property | `variant` | Emits |
|---|---|---|
| `accordionType` | `radix` | `type="single"` — aliased, because `type` is also the key of unrelated props elsewhere |
| `collapsible` | `radix` | `collapsible` — **no Base equivalent exists**; Base cannot express "single-open but not collapsible" |
| `defaultValue` | `radix` | `defaultValue="item-1"` — a bare string |
| `multiple` | `base` | `multiple` — the Base spelling of Radix's `type="multiple"` |
| `defaultValueBase` | `base` | `defaultValue={["item-1"]}` — same attribute, array shape |
| `className` | *(none)* | Both |

The `defaultValue` / `defaultValueBase` pair is the **same prop with a different shape** in each variant — a string in Radix, an array in Base. Two properties can never share a `key` (the key is where the value is stored on a node), so each gets its own and both alias to `defaultValue` via `codeAlias`. `codeFormat: "array"` produces the `{[…]}` wrapper.

<Aside type="note">
Radix's `defaultValue` is itself conditional on its sibling `type` — a string when `type="single"`, an array when `type="multiple"`. `variant` filters on `radix` / `base` only and cannot see a sibling's value, so `type="multiple"` with a string `defaultValue` is a known wrong emit. Base has no such split: its `defaultValue` is always an array.
</Aside>

### Field mapping

The `import-json` endpoint transforms this shape into the builder format before upserting, so the field semantics match the [component fields](#component-fields) above with these differences:

| shadcn-components.json | Maps to | Notes |
|------------------------|---------|-------|
| `imports[group]` → `{ path, components }` | — | Named import groups referenced by each component's `import` |
| `import` (alias: `importGroup`) | `importPath` | Resolved to `imports[import].path`. **Both field names are accepted** — but if it's missing/misspelled, `importPath` is `null`, which breaks generated imports *and* the preview source sync (the component shows as a placeholder) |
| `parent` | `parentType` | Parent component `type` |
| `children` | `allowedChildren` | Whitelisted child `type`s |
| `jsxTag` (optional) | `jsxTag` | Defaults to the component's `type` when omitted |
| _(all components)_ | `library: "shadcn"` | Forced to `shadcn` |
| _(object key order)_ | `sortOrder` | Assigned sequentially from key order |

The per-component fields `category`, `label`, `description`, `icon`, `isContainer`, `acceptsAny`, `selfClosing`, `properties`, `defaultChildren`, and `codeStructure` carry over unchanged.

<Aside type="note">
**`options` shorthand.** A `select` property's `options` may be a plain string array (`["single", "multiple"]`) or an array of `{ label, value }` objects. Strings are expanded to `{ label: s, value: s }` on import, so both forms work.
</Aside>

<a id="category-values"></a>
<Aside type="note">
Valid `category` values are `container`, `element`, `layout`, `composite`, `child`, and `primitive`. The palette currently surfaces **Containers, Elements, Layouts, and Primitives** as groups (child components appear nested under their parent). `layout` and `composite` are still being fleshed out — see the [roadmap](/modules/builder/roadmap/).
</Aside>

## Preview rendering

Importing a component **definition** (its palette entry) does not, by itself, ship the component's **source code**. But the **import syncs the sources** too, so imported shadcn components render in the preview straight away, no manual sync. The preview compiles generated JSX and resolves each component's `jsxTag` against a **merged component map**: the built-in **pristine set** (globbed from the `ui/pristine/` folder) is the **base layer**, and a pinned project's — or the app-wide global — **snapshot** is compiled and **overlaid** on top (for that snapshot's exact library + style). Merging rather than replacing means a component present only in pristine still renders even when a snapshot is active.

On import, the builder **incrementally adds any new components to the pristine folder** (only those not already there — existing files are left for the check/promote cycle), so `attachment` and any other imported shadcn component render everywhere, including for global/system/no-project blocks. A component whose `importPath` isn't a fetchable shadcn registry entry (a genuinely custom/app-local component) shows as a dashed **"(preview unavailable)"** placeholder: the sandbox has no source to execute, though the **generated code is unaffected** — it still emits `<Attachment>` with your `importPath`, so the exported component works in a real app that ships that import.

<Aside type="note">
**How the sync happens.** On import the builder derives the shadcn slugs from every component's `importPath` (`…/ui/<slug>`) and writes any missing ones into the pristine folder (`POST /components/sync-pristine`, admin) — Vite's glob picks them up. In production (read-only filesystem) it also populates the DB **global set** (`POST /components/global-set/sync`) instead. Fetches are cached; slugs that aren't fetchable are reported in the import result and skipped. Before v1.18.2 an unresolvable component blanked the **entire** preview with `X is not defined`; it is now isolated to a placeholder so the rest of the block still previews.
</Aside>

## Importing

Import from the admin UI — **Admin → Block Builder → Import**:

1. Paste your JSON or upload a `.json` file. Either format is detected automatically and a summary (component count, and field-template count for Builder exports) is shown.
2. Click **Import**. The builder upserts the definitions, then syncs the components' sources into the [global set](/modules/builder/integration/#global-component-set) so they render in the preview. The result reports **created** vs **updated** counts, how many sources were synced, and any per-row errors.

The **Export** button on the same tab downloads the current catalogue (plus `sharedFields`) in the Builder export format.

## Import behaviour

- **Upsert by `type`.** A component whose `type` already exists is **updated**; a new `type` is **created**. The result is `{ imported, created, updated, errors }` (`imported = created + updated`).
- **Lenient.** Unknown/extra fields are ignored rather than rejected, so a slightly different export shape (e.g. a `version` wrapper) still imports. Missing components are left untouched — import never deletes.
- **Per-row errors.** A single malformed component is reported in `errors` and skipped; the rest still import.
- **Accepted field variants.** The `import-json` format takes `import` **or** `importGroup` for the import-group reference; a `select`'s `options` may be a string array **or** `{ label, value }[]`.
- **Automatic source sync.** After the definitions upsert, any component **not already** in the preview's pristine set is added to it (and, in production, the DB [global set](/modules/builder/integration/#global-component-set)), so imported components render in the preview — including for global/system/no-project blocks — without a manual **Component Sources** sync. Existing components are left untouched (updates go through the check/promote cycle). Slugs that aren't fetchable from the shadcn registry are reported and skipped.
