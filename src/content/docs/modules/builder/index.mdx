The **Block Builder** module provides a visual component builder for useFoundryUI. You drag components onto a canvas, configure their properties, preview the result live, and save blocks that generate shadcn-compatible registry JSON вҖ” ready to install into any project with `npx shadcn add`.

<Aside type="note">
This module requires the [Structure](/modules/structure/) module for project integration (organising blocks into projects, per-project registry URLs).
</Aside>

## Quick Install

```bash
npx builder init
```

The CLI walks you through installation вҖ” it creates database tables, generates frontend and backend code, installs shadcn components, seeds default configuration, and syncs pristine shadcn component source files for the live preview.

## Architecture

The builder follows a **config-driven** approach. Block types are defined in a database-driven component registry (`native_component` table), not hardcoded. This means you can add, remove, or modify available components without touching code.

### Component Categories

| Category | Can Have Children? | Examples |
|----------|:------------------:|----------|
| **Containers** | Yes | Card, Dialog, Accordion, ScrollArea |
| **Elements** | No | Button, Input, Badge, Avatar |
| **Layouts** | Yes | Flexbox/grid wrappers, sections (rendered as the palette's "Layouts" group) |
| **Child** | Varies | CardHeader, AccordionItem вҖ” parent-scoped, shown nested rather than as a top-level palette group |
| **Primitives** | Varies | Div, Icon, Span |

The runtime category values are `container`, `element`, `layout`, `child`, and `primitive` (see `ComponentCategory`). The palette surfaces Containers, Elements, Layouts, and Primitives as top-level groups; child components appear nested under their parent.

### Extension Points

The builder provides registries and filter hooks for companion modules:

- **Property Panel Tab Registry** вҖ” add tabs to the right-side property panel (e.g. Style, Logic)
- **Code Generation Registry** вҖ” hook into import generation, component body injection, and prop transforms
- **`builder.canvas.globalCss`** filter вҖ” override the iframe preview CSS with project-specific themes (HSL or OKLCH)
- **`builder.registry.componentsJson`** filter вҖ” enrich the `components.json` snippet with additional config (e.g. `$schema`, `style`, `tailwind`)
- **8 CRUD action hooks** вҖ” react to block, category, and collection lifecycle events

See the [Integration](/modules/builder/integration/) page for full details.

### Registry Output

The builder generates shadcn-compatible registry JSON served via per-project public URLs:

```
https://handle.usefoundryui.app/r/project-slug/block-name.json
```

Consumer apps install blocks using the standard shadcn CLI:

```bash
npx shadcn add https://handle.usefoundryui.app/r/my-project/hero-section.json
```

### Database Tables

| Table | Purpose |
|-------|---------|
| `native_component` | Component palette definitions (public, no userId) |
| `block` | User-saved block trees with generated code |
| `block_category` | User-created folders for organising blocks |
| `block_collection` | Multi-block compositions (layouts, page sets) |
| `builder_config` | Module-level config (shared fields, shadcn style, icon library, sync status) |
| `builder_component_set` | Per-project pinned snapshot of the shadcn component sources (`library` + `style`) plus the derived per-`data-slot` `baseClasses` manifest; unique per project, drives the live preview and feeds the style module |

### Philosophy

"Design down to dev, not dev up to design" вҖ” users see what they're building before code is generated.

- **Structure module** вҖ” must be installed first (`npx structure init`)
- **Node.js 18+** and a running starter template app
- **Neon database** (or compatible PostgreSQL) with Prisma configured
