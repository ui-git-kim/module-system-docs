## Direction

**What this is.** A tool for **designers** to build good-looking shadcn components, blocks, and layouts that AI and developers then **install and use** — each project shipping its own registry. AI is notoriously poor at visual design; this fills that gap by letting a human design the pieces and hand them over as installable, on-brand building blocks.

It is **not** an AI app. There is no in-canvas AI generation and no MCP server. The only AI-facing surface is the *output*: a registry an AI or developer can install, and [useFoundry](https://usefoundryui.app) docs an AI can read to understand what it is installing.

**The layers.** The builder composes at four altitudes, coarsest to finest — three of them already have schema:

| Layer | What it is | Backed by |
|-------|------------|-----------|
| **Layout** | the app shell — sidebar, main panel, section/grid wrappers; its own files, holding multiple blocks | `block_collection` (`type: layout`) |
| **Collection** | **more than one block** (or a block + a loose component) assembled into one feature — e.g. a **chat**: message input + messages list + … | `block_collection` (`type: collection`); member blocks join via `block.collectionId`, one marked `isEntry` |
| **Block** | one block made of multiple components — today's core unit | `block` |
| **Component** | a single shadcn component | `native_component` |

(`block_collection.type` also carries `page` for a full-page collection.) The **Block ↔ Collection boundary is simply count**: one composed unit is a Block (an "accordion in a card" is one Block, however many components it nests); two or more blocks — or a block plus a loose component — is a Collection. There is no separate "composite" category: a component-made-of-blocks *is* a Collection, so the old `composite` value was redundant and was dropped (it was unused in real data anyway). Today the builder works fluently at the **Block** and **Component** layers; **Collection** and **Layout** are the next layers to make first-class.

<Aside type="note" title="Two meanings of &quot;layout&quot;">
`native_component.category = 'layout'` is a **component-level** category — structural container components (sidebar, scroll area, resizable group), the "Layouts" palette group. `block_collection.type = 'layout'` is a whole **Layout collection** (the app shell). Same word, two layers — kept distinct on purpose.
</Aside>

**Near-term sequence.**
1. Get **blocks** working well — catalogue marking, variant parity (`asChild` → `render`), and the palette groups below.
2. Make **Collections** and **Layouts** first-class layers (they have schema; they need canvas, palette, and code-gen support).
3. Then the **[private project registry](#private-project-registry--cli-install)** — each project installable via the shadcn CLI with a per-project auth token. This is the flagship next milestone, gated on the layers above being solid.

## Known Issues

Issues discovered but not yet resolved.

| Issue | Severity | Where | Workaround |
|-------|----------|-------|------------|
| Layout category & the Collection/Layout layers underdeveloped | Medium | The `layout` component category is wired (type, admin filter, palette "Layouts" group) but ships no components yet; the **Collection** and **Layout** layers have schema (`block_collection`) but no canvas/palette experience | Use blocks/containers; see Direction and Planned Improvements |
| No "Blocks" palette group | Medium | Component palette can't drag a saved block into the block you're building — there is no palette group sourced from the `block` table | Compose manually; tracked in Planned Improvements |
| Dynamic-tag text emitted twice when the field isn't marked Inner | Medium | `resolveInnerContent`'s string/textarea fallback (for `$`-dynamic and tagless components) picks a property as inner content, but `collectProps` only skips `codePlacement: 'inner'` — so an **unmarked** field is emitted as *both* an attribute and the content: `<p text="Hello">Hello</p>` instead of `<p>Hello</p>` | Set the field's **Code Placement = Inner** — fixes it immediately and makes the text carrier explicit rather than relying on the fallback. A code fix would have `collectProps` skip whatever `resolveInnerContent` selected; it already receives `config` + `resolved`, so no signature change is needed |
| Tree panel is a placeholder | Low | `LayersPanel` (`TreePanel.tsx`) renders a selectable outline but no expand/collapse or drag reorder | Use the canvas panel for tree navigation |
| No undo/redo support | Medium | Builder canvas — no history stack for block tree changes | Save frequently; use browser back if needed |
| Icon picker limited to Lucide | Low | Since v1.15.0 the builder's `IconPicker` is a thin wrapper over the starter's `@/components/icon-picker`; the Lucide-only limit now lives in the starter, not the builder | Enter non-Lucide icon names as strings for code output |
| No styleRegistry contribution | Low | Module does not register with `styleRegistry` despite controlling iframe preview CSS and pristine shadcn component selection | No runtime impact — CSS is injected directly via iframe postMessage and scaffold shadcn sync |

## Planned Improvements

### Short-term (Next Version)

**Component Palette Categories — Layouts & Blocks**
- **Layouts** — flesh out structural layout *components* (sidebar, main panel, section/grid wrappers). The `layout` category and its palette "Layouts" group already exist but ship no components yet. This is the component-level groundwork for the **Layout** layer (see [Direction](#direction)).
- **Saved blocks in the palette** — add a "Blocks" palette group so users can drag their own saved blocks into the block they are building. Not yet set up; would source from the `block` table rather than `native_component`.
- *Context:* the `native_component.category` column already accepts `layout`; wiring means extending the frontend `ComponentCategory` union (`components.ts` + `pages.ts`), the `CATEGORIES`/`CATEGORY_FILTERS` arrays, and the palette sections, plus decisions on group order, icons, and (for Blocks) drag-to-insert behaviour. *(The `composite` category was removed in the taxonomy cleanup — a component made of blocks is a Collection; see Direction.)*

**Collections & Layouts as first-class layers**
- The **Collection** layer (multiple blocks assembled into one feature — e.g. a chat: message input + messages list) and the **Layout** layer (the app shell) are the next altitudes above single blocks. See the [layers table](#direction).
- Schema already exists: `block_collection` carries `type: collection | layout | page`, member blocks join via `block.collectionId`, and one is marked `isEntry`. What's missing is the *canvas and palette experience* — assembling blocks into a collection, editing a collection as a unit, and dragging a saved collection/layout in like any other palette item. An open design detail: whether a collection holds only blocks, or also loose components directly (Kim's model allows both).
- *Context:* `Multi-Block Canvas` (Long-term, below) is the canvas half of this. The data model is largely in place; the work is UI + code generation that emits a collection's blocks into their own files, and a layout's shell across multiple files.

**Radix/Base — the remaining legs** *(the property dimension landed in v1.24.0)*
- **Mark the catalogue.** No property carries a `variant` yet, so nothing filters. The surface is far smaller than it looks — the only unambiguous marking rows are **accordion's** and **toggle-group's** Radix `type` (mark `radix`, add a Base `multiple` field alongside), and **accordion's `collapsible`** (mark `radix`; Base has **no** equivalent, so "single-open but not collapsible" is a genuine capability loss, not a rename).

  The `asChild` rows are **not** a marking job. `asChild` is a boolean whose child already lives in the block tree; Base spells the same intent `render={<Child />}`, hoisting that child into the prop and self-closing the tag. That is a structural code-generation transform, not a property — and marking `asChild` as `radix` would make it *impossible*, because `filterConfigByVariant` strips the prop before generation ever sees it. The two are mutually exclusive; see **Base's `render`** below.

  The API facts here were verified against the **installed primitives** (`radix-ui` 1.6.2 / `@base-ui/react` 1.6.0) and the shadcn registry, so they hold independently of any catalogue. Which components and keys your catalogue actually carries is a separate question — settle it from an **Admin → Export**, not from `examples/shadcn-components.json` (see the caution below).

  Traps found while scoping, worth keeping recorded:
  - **Mark by `(component, key)` pairs, never by key alone.** `collapsible` appears on two components with unrelated meanings — accordion's boolean (Radix-only) and **sidebar's** `offcanvas | icon | none` select, which is shadcn-authored and declared by *both* wrappers. A blanket key match would strip sidebar's from every Base project.
  - **`inset` is safe** across the menu families. Both the radix and base wrappers declare it, so it is shadcn-authored and variant-agnostic. Same for `checked`, and for the uncontrolled-state props (`defaultOpen`, `defaultChecked`, `defaultPressed`, `defaultValue`) — all present in both primitives.
  - **Several components have a property whose key is literally `variant`** (button, badge, alert, sidebar…). That is the shadcn *styling* variant and sits directly beside the Variant *marker* in the field editor. Marking `button.variant` as `radix` would be catastrophic and is an easy mis-click; consider relabelling the marker **Applies to**.
  - The `61 components / 17 identical / 43 divergent` figure is **unverified** and inconsistent with the current docs index (73 entries). Re-derive it from a registry enumeration before relying on it for scoping.

<Aside type="caution" title="examples/shadcn-components.json is not the catalogue">
It is a one-off sample import payload, added once (July 2026) and never synced since. No code reads it, it is not in `module.ts`, and it is not published to npm. It is useful for **format** questions — never for **content** ones. Counts drawn from it describe that file, not your database; the live `native_component` table (via **Admin → Export** or `GET /components`) is the only source of truth.
</Aside>
- ~~**Admin field editor**~~ — *shipped in v1.25.0*: the Edit Field dialog has a **Variant** select (Both / Radix only / Base only) writing `PropertyDefinition.variant`, shown only for `library: 'shadcn'` components.
- ~~**`block.variant`**~~ — *shipped in v1.28.0*: nullable `variant` column (null = agnostic), stamped by `SaveBlockDialog` from `useActiveVariant` on **create only**, and surfaced in the Blocks Library as a **mark rather than a hard filter**. See Completed.

- **Base's `render` (the other half of `asChild`).** The 16 `asChild` rows are not a marking job — `asChild` is a boolean whose child already lives in the block tree, and Base spells the same intent `render={<Child />}`, hoisting that child into the prop and self-closing the tag. So it is a **structural code-generation transform** keyed on the block's variant, not a property. Marking `asChild` as `radix` would make the transform impossible, because `filterConfigByVariant` strips the prop before `generateNodeCode` ever sees it — the two are mutually exclusive. Until this lands, a Base project cannot express a shadcn-styled trigger. The transform must be mirrored in `jsx-walker` or import stops round-tripping.
- **Deferred:** the public registry endpoint (`GET /r/:projectSlug`) does not filter by library — a base project's namespace can still advertise radix-only blocks. That's a live behaviour change to public output, so it wants its own decision.

**Import — promote unknown properties from the mapping report**
- The import report (v1.23.0) already lists every unmapped attribute with its literal value and inferred type — e.g. `defaultValue="shipping"` is not a property of `accordion`. An **admin** should be able to add it to the `native_component` straight from the report, then re-run the mapping so the value lands on the node.
- Key generation must be collision-aware: `NodeInstance` reserves `id`, `type` and `children`, which is exactly why the accordion's property is keyed `accordionType` with `codeAlias: 'type'`. An incoming `type=` must follow that convention.
- **Non-admins** get a *suggest to admin* action instead — queued for review and applied on approval. `native_component` is app-wide shared state (no `userId`), so changing it stays admin-gated; the queue is the escalation path.
- *Context:* `ImportIssue` already carries `componentType`, `attribute`, `value` and `valueType` for this purpose; the admin save path is `saveNativeComponent` (admin-only, and partial-safe since v1.22.2).

**Undo/Redo Stack**
- Add undo/redo support for block tree changes
- Keyboard shortcuts: Ctrl+Z / Ctrl+Shift+Z
- History panel showing recent operations
- *Context:* The `useBuilder` hook manages block state — add a history array alongside `block` state. Each `setBlock` call pushes to history. Limit to ~50 entries.

**Tree Panel Implementation**
- Full hierarchical tree view with expand/collapse
- Drag-and-drop reorder within the tree
- Right-click context menu (delete, duplicate, wrap)
- Visual indicators for containers vs. elements
- *Context:* `TreePanel.tsx` exists as a placeholder. Use a recursive tree component with `@dnd-kit/sortable` for reorder.

**Broader Structure Hook Integration**
- Listen to `structure.node.updated` to handle project renames (update display references)
- Listen to `structure.node.archived` to cascade archive state to blocks
- *Context:* Currently only listens to `structure.node.deleted` for cleanup and `structure.node.response` for metadata enrichment. The structure module fires 16+ action hooks that the builder could react to.

### Medium-term (Upcoming)

<a id="private-project-registry--cli-install"></a>
**Private project registry (CLI install)** — *flagship next milestone*
- Each project gets its **own private registry**, hosted here on the builder, that a developer installs from with the standard shadcn CLI — e.g. `npx shadcn add @project-slug/some-block`.
- **Auth token** gates it: the registry is private to the project, so installing requires a per-project token (in the consumer's `components.json` registry config / `.env`), not a public URL. This is the difference from today's public `GET /r/:projectSlug` output.
- This is what turns the builder from "design tool" into "design → install pipeline": a designer builds on-brand blocks/collections/layouts, and AI or a developer pulls them straight into an app with one CLI command.
- **Sequencing:** gated on the **Block**, **Collection**, and **Layout** layers being solid first (see [Direction](#direction)) — there's little point shipping the install pipeline until the things being installed are good. Kim: *"the registry plumbing is the next stage once we get the blocks working well enough."*
- *Context:* the registry JSON and per-project serving already exist (`registry.service.ts`, per-project `/r/` URLs). The new work is **token issuance + auth on the registry endpoint**, the consumer-facing namespace + `components.json` snippet, and making the served items include collections/layouts, not just blocks. Also settle the deferred *base-project-advertises-radix-blocks* filtering question (see Known Issues / Radix-Base).
- **Docs for AI consumers:** since AI/devs install these, the [useFoundry](https://usefoundryui.app) docs need to be legible to an AI agent — what a project's registry contains, how to authenticate, and how to install. That's a docs deliverable alongside the endpoint.

**Component Import/Export**
- Export blocks as standalone shadcn registry items
- Import from external shadcn registries
- Version tracking for imported components
- Conflict resolution for name collisions
- *Context:* The registry JSON format is already defined in `registry.service.ts`. Needs a frontend UI for browsing external registries and an import resolution flow.

**Builder Logic Module**
- Separate module (`builder-logic`) for adding interactivity
- Event bindings (onClick, onChange, etc.)
- State management integration
- Data source connections
- Conditional rendering rules
- *Context:* This will be a new module that depends on `builder`. It should extend the property panel with a "Logic" tab (order 20) and the code generation registry with event handler imports and prop transforms. Store event bindings alongside block data.

**Enhanced Code Generation**
- Support for custom component libraries
- Code formatting with Prettier
- Multiple output formats (React, Vue in future)
- *Context:* Import resolution is already catalogue-driven — `registry.service.ts` (`extractRegistryDependencies`) derives dependencies from every `native_component.importPath`, so there is no hardcoded component map to expand. Remaining work is alternate libraries, output targets, and formatting.

**Responsive preview — device & breakpoint toggles**
- Add device toggles (desktop / tablet / mobile) plus a light/dark + theme selector to a dedicated **preview mode**, so responsive checking is a preview concern rather than an editing mode. *(Kim: "very very good" for device/breakpoint.)*
- *Context:* the iframe/snapshot preview already renders the project's real components with project theming (v1.17.0); this constrains the viewport and adds a device switcher over that surface. Per-breakpoint *editing* (overriding classes per breakpoint) belongs to the Style module — this is preview-only.

**Swap-variant ("shuffle")**
- One click swaps a block (or collection) on the canvas for a different one of the same category/kind — a fast way to explore alternatives without rebuilding. Borrowed from shadcn/studio's builder; maps onto the variant concept already in play.
- **Connects to a concept in the colour module** — the two should share one mental model (cycle through variants of the same thing).
- *Context:* depends on the Collection/Block layers and a category-scoped "siblings" query; downstream of the layers work above.

**Standards & audit debt** *(from the v1.31.0 meta audit)*
- **R8 modernization pass — remaining pieces.** The bulk of this shipped in v1.31.5–v1.31.6 (`Loader2`→`Spinner`, local empty states→`EmptyState`, error boxes→`ErrorAlert`, `window.confirm`→`AlertDialog`, raw `<button>`→`Button`). Still deferred to a visual-review pass because they change layout/interaction: the parent-type + default-children search boxes → `Combobox`/`Command`; the `ShadcnSyncPanel` CSS-grid table → `Table`; the category-filter and `SaveBlockDialog` scope pills → `ToggleGroup`; `PreviewPanel`'s Preview/Code buttons → `Tabs`; the global-publish toggle → `Switch`; the blocks-library header → `PageHeader`; `HandleField`'s raw `<input>` and the search fields → `Input`/`InputGroup`.
- **Minor debt** — a misleading `updateBlock` comment; `useProjectComponentSet` mutations cast `projectId as string`; number/`node-id` raw interpolation in the tree-walker (defense-in-depth — values are internally generated). (The `api.*<any>` typing, the local `ComponentLibrary` union, the missing `{ audit: true }` cascade log, and the `protectedRoutes` metadata all shipped in v1.31.2/v1.31.5.)
- *Context:* none of these change generated user-block output; they are convention/robustness polish on the builder's own templates. Batch alongside whatever file area is already being touched.

**Multi-Library Icon Picker**
- Support all configured icon libraries (Tabler, Phosphor, Remix, Hugeicons) in the picker UI
- Load icon sets dynamically based on `shadcn:icon-library` config
- *Context:* Since v1.15.0 the builder's `IconPicker` is a thin wrapper over the starter's shared `@/components/icon-picker` (Lucide-only). Multi-library support is therefore primarily a **starter-template** enhancement to the shared picker; the icon-library config already drives code output regardless.

### Long-term / Ideas

**Style Module — shipped** (see [Style module](/modules/style/))
- Visual styling now lives in a separate **Style** module: a per-instance Tailwind class designer, theme configuration, and shadcn style management. It consumes the builder's per-project component snapshot and the derived `baseClasses` manifest (builder v1.18.0).
- Genuinely-open scope (style presets, responsive per-breakpoint overrides) is tracked on the [Style module's own roadmap](/modules/style/roadmap/).

**Multi-Block Canvas**
- Work with multiple blocks simultaneously on a split canvas
- Cross-block drag-and-drop
- Collection-level editing with entry point management
- Shared component references across blocks

**Consumable by AI & developers — *not* an in-app AI builder**
- Deliberately **not** an AI builder: no natural-language-to-tree generation and **no MCP server**. The premise is the inverse — humans design good components/blocks/layouts *because* AI is poor at visual design, and AI/devs then **consume** them.
- The AI-facing work is on the **output** side: the [private project registry](#private-project-registry--cli-install) an agent can install from, and [useFoundry](https://usefoundryui.app) docs an agent can read to know what a project offers and how to pull it.
- Deterministic helpers (accessibility linting, property suggestions in the panel) remain open ideas — those are not "AI building" and don't change the positioning.

**Collaborative Editing**
- Real-time multi-user editing with cursor presence
- Conflict resolution for simultaneous edits
- Change attribution and activity feed

**Per-block / per-node shadcn style selection**
- *Per-**project** library + style selection and side-by-side radix/base rendering already shipped in v1.16.0–v1.17.0 (per-project component snapshots — see Completed).*
- Remaining: allow a style override at the block or node level (not just per project), and retire the global `shadcn:builder-style` admin config in favour of the per-project snapshot as the single source of truth.

### Code Health (from the v1.12.1 audit)

Lower-severity cleanups surfaced by the health audit, grouped for a maintenance pass:

- **Dead code:** remove the unexported `ADMIN_PAGE_COMPONENT_TEMPLATE`; drop unused imports/vars and DEV `console.*` in the generated CanvasPage. *(The orphaned `frontend/routes.ts` and empty `frontend/types.ts` were removed in v1.22.1.)*
- **Conventions:** fix import ordering (external → `@/` → relative) across generated files; replace forbidden Tailwind classes with semantic tokens; refresh `package.json` description/keywords off the scaffold placeholder.
- **Config declarations:** declare `envVariables` in `module.ts` for `VITE_REGISTRY_DOMAIN`; export `CONFIG_KEYS`; remove dead `native_component` indexes; drop published sourcemaps from the npm package.
- **Performance passes:** `React.memo` on canvas recursion components, memoise context values, add `staleTime` to the near-static native-components query, lazy-load heavy registration imports, and avoid bundling the full lucide icon set.
- **Registry hardening (defence in depth):** reserved-handle denylist, timing-safe token comparison, enumeration-safe 404s, batched component-import upserts, and request timeouts on shadcn proxy fetches.
