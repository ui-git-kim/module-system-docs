## Known Issues

Issues discovered but not yet resolved.

| Issue | Severity | Where | Workaround |
|-------|----------|-------|------------|
| Layout & composite categories underdeveloped | Medium | The `layout` category is wired (type, admin filter, palette "Layouts" group) but ships no components; `composite` is a valid `category` value but has no palette group or admin filter yet | Use containers/elements; see Planned Improvements below |
| No "Blocks" palette group | Medium | Component palette can't drag a saved block into the block you're building — there is no palette group sourced from the `block` table | Compose manually; tracked in Planned Improvements |
| Dynamic-tag text emitted twice when the field isn't marked Inner | Medium | `resolveInnerContent`'s string/textarea fallback (for `$`-dynamic and tagless components) picks a property as inner content, but `collectProps` only skips `codePlacement: 'inner'` — so an **unmarked** field is emitted as *both* an attribute and the content: `<p text="Hello">Hello</p>` instead of `<p>Hello</p>` | Set the field's **Code Placement = Inner** — fixes it immediately and makes the text carrier explicit rather than relying on the fallback. A code fix would have `collectProps` skip whatever `resolveInnerContent` selected; it already receives `config` + `resolved`, so no signature change is needed |
| Tree panel is a placeholder | Low | `LayersPanel` (`TreePanel.tsx`) renders a selectable outline but no expand/collapse or drag reorder | Use the canvas panel for tree navigation |
| No undo/redo support | Medium | Builder canvas — no history stack for block tree changes | Save frequently; use browser back if needed |
| Icon picker limited to Lucide | Low | Since v1.15.0 the builder's `IconPicker` is a thin wrapper over the starter's `@/components/icon-picker`; the Lucide-only limit now lives in the starter, not the builder | Enter non-Lucide icon names as strings for code output |
| No styleRegistry contribution | Low | Module does not register with `styleRegistry` despite controlling iframe preview CSS and pristine shadcn component selection | No runtime impact — CSS is injected directly via iframe postMessage and scaffold shadcn sync |

### From the v1.12.1 health audit

A full standards/production/performance/security audit of the generated templates was run at v1.12.1. **All 16 critical findings and most moderate findings were remediated in [v1.13.0](/modules/builder/changelog/)** — route authorisation, block ownership and `isSystem` authority, generated-code escaping, preview origin/sandbox hardening, the PropertyPanel/DnD/widget crashes, the empty-component registry generator, fresh-install seeding and the `pickers`/env declarations, the `securityRegistry` deletion handler, backend validation, and fetch timeouts.

The items still open are tracked below.

| Issue | Severity | Where | Notes |
|-------|----------|-------|-------|
| Registry handle needs server-side uniqueness/reservation | High | Handle uniqueness and format are only pre-checked client-side before being persisted via the generic preferences path | Backend resolver still trusts `preferences.handle`; needs a reserved-name denylist and a server-side uniqueness check |
| Public `/r` route registered with empty `versions: []` | Medium | `route.registration.ts` — undocumented mounting for the unversioned public routes | Left unchanged (working endpoint); verify against `mountAll` behaviour before altering |
| Forbidden Tailwind classes in the builder's chrome UI | Low | ~190 occurrences (typography-scale, font-weight, a few colour literals, `!important`, opacity modifiers) in the builder's own panels/pages/toolbar (`components.ts`, `pages.ts`). The **generated user-block render path** (`renderer`, `tree-walker`) is clean (v1.22.1 audit) | Cosmetic and largely **functional** (dense dev-tool UI; `!important` is required to override shadcn `Sheet` widths; opacity modifiers drive hover/selection/drop-target feedback). Not mechanically strippable without degrading the UI — needs a careful per-component pass with visual verification |
| Block-list endpoint over-fetches | Medium | `getBlocks` selects full `data` trees + `generatedCode` for list views | Add a `select` projection once list vs. editor needs are separated |
| Schema uniqueness ineffective on nullable columns | Medium | Compound unique constraints include nullable columns (NULLs are distinct in Postgres) | Needs a partial index and a migration for existing installs |
| Duplicate drag id across palette and child-add | Medium | `PaletteItem` and `ChildElementItem` both register `palette-\${type}` | Needs a distinct id plus a matching change to the drag-end/overlay parsing |

<Aside type="caution">
The full audit report (151 findings) is tracked privately. This roadmap records the remediation surface, not exploitation detail.
</Aside>

<Aside type="tip">
The 2026-07-14 standards-alignment audit (against scaffold v1.18.0 / contract v1.2.0) is **complete** — host `moduleRegistry` integration, the `requiredModules → depends` migration, the scaffold update, and dead-code removal all shipped in v1.22.1 (see Completed). Module migrations (`migrate-db`, scaffold v1.18.0) are **available but unused**: the schema has only ever grown additively, so no expand → backfill → contract migration is needed yet.
</Aside>

## Planned Improvements

### Short-term (Next Version)

**Component Palette Categories — Layouts, Composites & Blocks**
- **Layouts** — flesh out structural layout components (sidebar, main panel, section/grid wrappers). The `layout` category and its palette "Layouts" group already exist but ship no components yet.
- **Composites** — surface the `composite` category (a component made of more than one block — a composite block) with its own palette group and admin filter. `composite` is a valid `native_component.category` value today but has no frontend wiring (it is absent from the `ComponentCategory` union, `CATEGORY_FILTERS`, and the palette `getByCategory` sections), so composite components currently render nowhere.
- **Saved blocks in the palette** — add a "Blocks" palette group so users can drag their own saved blocks into the block they are building. Not yet set up; would source from the `block` table rather than `native_component`.
- *Context:* the `native_component.category` column already accepts `composite`/`layout`; wiring means extending the frontend `ComponentCategory` union (`components.ts` + `pages.ts`), the `CATEGORIES`/`CATEGORY_FILTERS` arrays, and the palette sections, plus decisions on group order, icons, and (for Blocks) drag-to-insert behaviour.

**Radix/Base — the remaining legs** *(the property dimension landed in v1.24.0)*
- **Mark the catalogue.** The data itself is still Radix-flavoured — zero properties carry a `variant`, so nothing filters yet. Scoped against the real catalogue and the installed primitives (`radix-ui` 1.6.2 / `@base-ui/react` 1.6.0), the surface is far smaller than it looks: of 521 property instances, only **three** are unambiguous marking rows — `accordion.accordionType` and `toggle-group.toggleGroupType` (both Radix `type`, mark `radix`, add a Base `multiple` field), and `accordion.collapsible` (mark `radix`; Base has **no** equivalent, so "single-open but not collapsible" is a genuine capability loss).

  The 16 `asChild` rows are **not** a marking job. `asChild` is a boolean whose child already lives in the block tree; Base spells the same intent `render={<Child />}`, hoisting that child into the prop and self-closing the tag. That is a structural code-generation transform, not a property — and marking `asChild` as `radix` would make it *impossible*, because `filterConfigByVariant` strips the prop before generation ever sees it. The two are mutually exclusive; see **`block.variant`** below.

  Traps found while scoping, worth keeping recorded:
  - **Mark by `(component, key)` pairs, never by key alone.** `collapsible` exists on two components with unrelated meanings — accordion's boolean (Radix-only) and **sidebar's** `offcanvas | icon | none` select, which is shadcn-authored and declared by *both* wrappers. A blanket key match would strip sidebar's from every Base project.
  - **`inset` (9 menu components) is safe.** Both the radix and base wrappers declare it, so it is shadcn-authored and variant-agnostic. Same for `checked`, and for the uncontrolled-state props (`defaultOpen`, `defaultChecked`, `defaultPressed`, `defaultValue`) — all present in both primitives.
  - **15 components have a property whose key is literally `variant`** (button, badge, alert, sidebar…). That is the shadcn *styling* variant and sits directly beside the Variant *marker* in the field editor. Marking `button.variant` as `radix` would be catastrophic and is an easy mis-click; consider relabelling the marker **Applies to**.
  - The `61 components / 17 identical / 43 divergent` figure is **unverified** and inconsistent with the current docs index (73 entries). Re-derive it from a registry enumeration before relying on it for scoping.
- ~~**Admin field editor**~~ — *shipped in v1.25.0*: the Edit Field dialog has a **Variant** select (Both / Radix only / Base only) writing `PropertyDefinition.variant`, shown only for `library: 'shadcn'` components.
- **`block.variant`** — blocks need the same axis (null = agnostic), plus a filter in the Blocks Library. Named `variant` to match `PropertyDefinition.variant`, and deliberately **not** `library`: `native_component.library` already means `shadcn | primitive | plate | custom`, so a third `library` column with a different value domain would be a silent footgun. Nothing stamps the value until `SaveBlockDialog` reads `useActiveVariant`. Filter **client-side** in `useBlocksLibrary`'s existing `filteredBlocks` memo — the system/global query keys are deliberately shared with `useProjectBlocks`, so a server-side param would poison or fragment that cache. Two constraints found when scoping it: `useBlocksLibrary()` takes no arguments and `BlocksLibraryPage` has no project context, so the filter resolves the **global** variant only; and `createBlock`/`updateBlock` are explicit column allowlists, so `variant` must be added to both or it is silently dropped.
- **Deferred:** the public registry endpoint (`GET /r/:projectSlug`) does not filter by library — a base project's namespace can still advertise radix-only blocks. That's a live behaviour change to public output, so it wants its own decision.

**Import — promote unknown properties from the mapping report**
- The import report (v1.23.0) already lists every unmapped attribute with its literal value and inferred type — e.g. `defaultValue="shipping"` is not a property of `accordion`. An **admin** should be able to add it to the `native_component` straight from the report, then re-run the mapping so the value lands on the node.
- Key generation must be collision-aware: `NodeInstance` reserves `id`, `type` and `children`, which is exactly why the accordion's property is keyed `accordionType` with `codeAlias: 'type'`. An incoming `type=` must follow that convention.
- **Non-admins** get a *suggest to admin* action instead — queued for review and applied on approval. `native_component` is app-wide shared state (no `userId`), so changing it stays admin-gated; the queue is the escalation path.
- *Context:* `ImportIssue` already carries `componentType`, `attribute`, `value` and `valueType` for this purpose; the admin save path is `saveNativeComponent` (admin-only, and partial-safe since v1.22.2).

**Undo/Redo Stack**
- Add undo/redo support for block tree changes
- Keyboard shortcuts: Ctrl+Z / Ctrl+Shift+Z
- History panel showing recent operations
- *Context:* The `useBuilder` hook manages block state — add a history array alongside `block` state. Each `setBlock` call pushes to history. Limit to ~50 entries.

**Tree Panel Implementation**
- Full hierarchical tree view with expand/collapse
- Drag-and-drop reorder within the tree
- Right-click context menu (delete, duplicate, wrap)
- Visual indicators for containers vs. elements
- *Context:* `TreePanel.tsx` exists as a placeholder. Use a recursive tree component with `@dnd-kit/sortable` for reorder.

**Broader Structure Hook Integration**
- Listen to `structure.node.updated` to handle project renames (update display references)
- Listen to `structure.node.archived` to cascade archive state to blocks
- *Context:* Currently only listens to `structure.node.deleted` for cleanup and `structure.node.response` for metadata enrichment. The structure module fires 16+ action hooks that the builder could react to.

### Medium-term (Upcoming)

**Component Import/Export**
- Export blocks as standalone shadcn registry items
- Import from external shadcn registries
- Version tracking for imported components
- Conflict resolution for name collisions
- *Context:* The registry JSON format is already defined in `registry.service.ts`. Needs a frontend UI for browsing external registries and an import resolution flow.

**Builder Logic Module**
- Separate module (`builder-logic`) for adding interactivity
- Event bindings (onClick, onChange, etc.)
- State management integration
- Data source connections
- Conditional rendering rules
- *Context:* This will be a new module that depends on `builder`. It should extend the property panel with a "Logic" tab (order 20) and the code generation registry with event handler imports and prop transforms. Store event bindings alongside block data.

**Enhanced Code Generation**
- Support for custom component libraries
- Code formatting with Prettier
- Multiple output formats (React, Vue in future)
- *Context:* Import resolution is already catalogue-driven — `registry.service.ts` (`extractRegistryDependencies`) derives dependencies from every `native_component.importPath`, so there is no hardcoded component map to expand. Remaining work is alternate libraries, output targets, and formatting.

**Multi-Library Icon Picker**
- Support all configured icon libraries (Tabler, Phosphor, Remix, Hugeicons) in the picker UI
- Load icon sets dynamically based on `shadcn:icon-library` config
- *Context:* Since v1.15.0 the builder's `IconPicker` is a thin wrapper over the starter's shared `@/components/icon-picker` (Lucide-only). Multi-library support is therefore primarily a **starter-template** enhancement to the shared picker; the icon-library config already drives code output regardless.

### Long-term / Ideas

**Style Module — shipped** (see [Style module](/modules/style/))
- Visual styling now lives in a separate **Style** module: a per-instance Tailwind class designer, theme configuration, and shadcn style management. It consumes the builder's per-project component snapshot and the derived `baseClasses` manifest (builder v1.18.0).
- Genuinely-open scope (style presets, responsive per-breakpoint overrides) is tracked on the [Style module's own roadmap](/modules/style/roadmap/).

**Multi-Block Canvas**
- Work with multiple blocks simultaneously on a split canvas
- Cross-block drag-and-drop
- Collection-level editing with entry point management
- Shared component references across blocks

**AI-Assisted Building**
- Natural language to component tree conversion
- Property suggestions based on context and common patterns
- Layout recommendations based on content type
- Accessibility improvement suggestions

**Collaborative Editing**
- Real-time multi-user editing with cursor presence
- Conflict resolution for simultaneous edits
- Change attribution and activity feed

**Per-block / per-node shadcn style selection**
- *Per-**project** library + style selection and side-by-side radix/base rendering already shipped in v1.16.0–v1.17.0 (per-project component snapshots — see Completed).*
- Remaining: allow a style override at the block or node level (not just per project), and retire the global `shadcn:builder-style` admin config in favour of the per-project snapshot as the single source of truth.

### Code Health (from the v1.12.1 audit)

Lower-severity cleanups surfaced by the health audit, grouped for a maintenance pass:

- **Dead code:** remove the unexported `ADMIN_PAGE_COMPONENT_TEMPLATE`; drop unused imports/vars and DEV `console.*` in the generated CanvasPage. *(The orphaned `frontend/routes.ts` and empty `frontend/types.ts` were removed in v1.22.1.)*
- **Conventions:** fix import ordering (external → `@/` → relative) across generated files; replace forbidden Tailwind classes with semantic tokens; refresh `package.json` description/keywords off the scaffold placeholder.
- **Config declarations:** declare `envVariables` in `module.ts` for `VITE_REGISTRY_DOMAIN`; export `CONFIG_KEYS`; remove dead `native_component` indexes; drop published sourcemaps from the npm package.
- **Performance passes:** `React.memo` on canvas recursion components, memoise context values, add `staleTime` to the near-static native-components query, lazy-load heavy registration imports, and avoid bundling the full lucide icon set.
- **Registry hardening (defence in depth):** reserved-handle denylist, timing-safe token comparison, enumeration-safe 404s, batched component-import upserts, and request timeouts on shadcn proxy fetches.
