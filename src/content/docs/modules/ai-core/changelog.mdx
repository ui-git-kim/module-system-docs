This page documents notable changes to the AI Core module. The module uses semantic versioning: **major.minor.patch**.


- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, template fixes

<Aside type="tip">
Check your installed version with `npx ai-core info`.
</Aside>

## v1.2.0 — 2026-07-23

**Vision input: media attachments on segments.** A consumer segment may now carry `media` — `{ mediaType, data }` entries where data is base64 bytes or an http(s) URL. Attachments are handed to the provider adapter as file parts on the user message; they never enter the prompt text, the logs or the ledger. When the resolved model's fetched capabilities say it does not accept image input, the run refuses loudly (`model-lacks-image-input`); an unknown capability proceeds and a provider rejection lands in the ledger. Raised by cog-ingest's vision prompt variants ahead of its integration.

## v1.1.1–v1.1.3 — 2026-07-23

**Install-verification fixes from the first real install.** `npx ai-core init` into a real app succeeded end to end — schema appended, 35 files generated, routes auto-registered, and all six seed migrations applied on the migration runner's first real use. Three defect classes surfaced and were fixed:

- **v1.1.1** — Prisma types now import from the starter's generated client (`backend/src/generated/prisma/client.js`) instead of `@prisma/client`, matching the starter's own features under Prisma 7's custom output; `upsertCheck`'s config writes gained the `toJson` boundary cast for Prisma 7's stricter `InputJsonValue`.
- **v1.1.2–v1.1.3** — `ErrorAlert`'s real prop is `error` (15 sites); the usage query key serialises its filter object (the query-keys helper takes scalar parts); the frontend barrel exports an empty `aiCoreRoutes` so the scaffold's `routes.tsx` wiring resolves — that import is also what executes the admin-page registration side effect.

After the fixes: **zero ai-core type errors in the host app's backend and frontend**. The install also exposed a scaffold bug (scoped package specs mangled to `@scope/name@^4@latest`), fixed upstream in module-starter scaffold v1.19.2. Runtime verification (app boot, admin page render) is the remaining step.

## v1.1.0 — 2026-07-23

**The redesign, implemented**

The full rebuild of the module templates to the published design. The legacy v2 surface — purpose keys, content processors, image and video generation — was **deleted wholesale**; it was never released and nothing downstream imported it.

- **`ai_` schema** — twelve models delivered via the module migrations runner (migrations, never `prisma db push`), plus six seed migrations: providers, prompt frameworks, prompt segments, disciplines, guardrail checks, and model records. Every battery is a row — seeded at install, editable, resettable
- **Provider adapters on Vercel AI SDK v7** — Anthropic, OpenAI, Google and Voyage, confined to a single adapter file behind an interface AI Core defines. Black Forest Labs and Replicate provider rows are seeded **disabled** (terms unconfirmed); their generation adapters throw typed not-implemented errors
- **Model records and capability refresh** — capabilities refreshed from the providers' REST model listings and flagged stale on failure; the knowledge cutoff is admin-entered
- **Prompt segments, frameworks and assembly** — the computed date anchor rebuilt every call, EXT control markers through external content, and the framing segment ordered automatically ahead of it
- **Disciplines and guardrail checks** — the authority hierarchy enforced at **write time** (admin CRUD refuses violations transactionally; a `hard` rule without an enabled check cannot be enabled) and re-validated at **resolution time** (`aiCore.run` refuses an invalid set)
- **Operations resolver** — the sensitivity gate that refuses rather than falls back, dial merge, and explicit parameter transforms recorded in the ledger
- **`aiCore.run` and `runEmbed`** — the consumer surface, with structured feedback on every return; `runEmbed` enforces the Structure dimension gate
- **Flows** — suspend/resume with persisted flow state and `initialState`
- **Usage ledger and billing callback** — audit by reference (IDs, versions, hashes — never content) and `getCurrentUsage` for billing
- **Review queue** — abstentions and low-confidence outcomes land in a queue battery
- **Admin page** — one page, tabbed (Models, Operations, Prompts, Disciplines, Presets, Usage) at `/admin/ai-core`, order 30, platform section
- **Security registration** — the data manifest and deletion handlers for the tenant-owned tables

**Verification:** the generated code was extracted and type-checked under strict harnesses — the backend against the real published SDK packages, the frontend against React 18 and TanStack Query v5. **The module has not yet been installed into a real app**; install verification is the next gate, tracked on the [Roadmap](/modules/ai-core/roadmap/). Retrieval, the semantic cache and groundedness scorer implementations remain unbuilt (medium-term).

## v1.0.1 — 2026-07-22

**July 2026 alignment**

An infrastructure alignment carried out alongside publication of the redesign. The patch bump records that `src/module.ts` changed (version reconciliation and a dead import removed); the template surface itself is unchanged.

- **Scaffold adopted at v1.19.1** (module contract v1.2.0): lifecycle hooks, the migrations runner, uninstall/enable/disable support, the contract npm scripts, and the docs submodule
- **Module version reconciled to the 1.x line.** The version had been advanced to an abandoned, never-shipped 2.0.0; it has been brought back to 1.x, matching the redesign's stated version line — nothing downstream imports the current surface, so it can be replaced cleanly
- **CLAUDE.md adopted the SCAFFOLD marker contract**

**An honest record of the abandoned "v2".** Between the January release and the redesign, a second surface was built in-repo — AI SDK 6 transport, purpose keys, image and video generation, embeddings, model discovery — but it was **never released and never installed anywhere**. It is superseded by the published redesign and was replaced wholesale in v1.1.0 (above). The redesign's shape — models as records, operations as the public surface, disciplines and guardrails — is documented on the [overview](/modules/ai-core/) page, and the path from here to a shippable release is on the [Roadmap](/modules/ai-core/roadmap/). Scaffold and contract mechanics are covered in the [developer workflow](/modules/ai-core/developer-workflow/).

## v1.0.0 — 2026-01-23

**Initial release**

The first installable AI Core module — shared AI infrastructure consumed by other AI-powered modules.

- **Providers** — SDK initialisation for Anthropic and OpenAI, lazily initialised on first use
- **Model capability discovery** — dynamic capability fetching with caching and conservative fallbacks when the provider API is unavailable
- **Content processors** — image (vision), URL content extraction, and PDF processing
- **Prompt registry and assembler** — a composable prompt system; consumers register their own prompts and assembly injects date context automatically
- **Utilities** — token estimation and typed AI error handling

This was the released surface. The repo's templates were since overwritten by the never-released v2 work, and both were replaced wholesale by the v1.1.0 implementation of the published redesign (above).
