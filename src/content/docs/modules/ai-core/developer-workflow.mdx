This page covers the full developer workflow for the AI Core module — local setup, the edit–build–test loop, versioning, scaffold updates, and the documentation workflow — followed by the conventions the implementation must follow and how usage, tokens and the ledger are handled.

<Aside type="note" title="Implemented in v1.3.1 — first live call pending">
This page describes AI Core as implemented in module v1.3.1 (July 2026). The module installs into a real app, type-checks clean there, and its admin page renders and serves records at runtime. What is **not** yet verified is a first live provider call. Items still unbuilt (retrieval, semantic cache, groundedness scorers) are marked inline — see the [Roadmap](/modules/ai-core/roadmap/) for status.

## Setup

### Clone and Install

The repo lives at `ReactApps/Modules/ai-core`:

```bash
git clone <ai-core-repo-url>
cd ReactApps/Modules/ai-core
npm install
```

### Build

```bash
npm run build     # One-time build
npm run dev       # Watch mode (rebuilds on file changes)
```

### Link for Local Testing

```bash
npm link          # Makes the 'ai-core' command available globally
```

<Aside type="note">
AI Core is zero-config by design: it will work with nothing but an API key. The test app's backend needs `ANTHROPIC_API_KEY` in its `.env`; other provider keys are optional and a provider initialises lazily when its key is present. See [Configuration](/modules/ai-core/configuration/).
</Aside>

## Development Loop

The standard workflow when making changes:

```
Edit templates → Build → Check → Test in app → Version bump → Commit → Push
```

### 1. Edit Templates

All your work happens in these locations:

| Location | What You Edit |
|----------|--------------|
| `src/templates/frontend/` | The admin page and its tabs, hooks, services, types, registrations |
| `src/templates/backend/` | Provider adapters, records, resolver, run, assembly, disciplines, flows, ledger, routes, registrations |
| `src/templates/schema.ts` | The Prisma schema template — the twelve `ai_`-prefixed tables |
| `src/templates/migrations/` | The seed migrations — the shipped batteries as rows |
| `src/module.ts` | Module configuration (routes, features, env variables) |

<Aside type="caution">
Never edit files in `src/scaffold/` or `src/cli.ts`. These are shared infrastructure — report bugs upstream to the module starter template.
</Aside>

### 2. Build and Check

```bash
npm run build              # Compile to dist/
npm run type-check         # TypeScript check, no emit
npm run check:templates    # Catch unescaped ${...} in template strings
npm run parity:check       # Catch drift between paired .ps1/.sh scaffold scripts
```

`check:templates` catches the number-one template footgun: every template is itself a backtick string, so a literal `${` in generated code must be escaped as `\${` — an unescaped one interpolates silently at build time and ships wrong output. `parity:check` guards the paired PowerShell/Bash scaffold scripts, which must stay behaviourally identical because downstream modules run whichever their machine supports.

<Aside type="note" title="A designed check, not yet built">
The design extends the `check:templates` pattern to the seeded disciplines and checks: a CI lint that reports orphaned checks and unreferenced rules in the seed data before it ships. At runtime this is already enforced — a `hard` rule with a null `check` cannot be enabled, and resolution refuses an invalid discipline set (see [Usage](/modules/ai-core/usage/)) — but the seed-integrity lint itself is still to build. A hard rule with no enforcement is exactly the failure the module exists to prevent — **prose asks; code holds.**
</Aside>

### 3. Test in a Starter App

In a separate terminal, navigate to a starter template app:

```bash
# Fresh install
npx ai-core init

# Update existing installation (after template changes)
npx ai-core update

# Preview what would change (dry run)
npx ai-core update --dry-run

# Force overwrite all files
npx ai-core init --force

# Check installed version and module info
npx ai-core info

# Remove the module
npx ai-core uninstall
```

### 4. Version Bump

After verifying your changes work, bump the version:

```bash
npm run version:patch    # Bug fix, template fix
npm run version:minor    # New feature, new template
npm run version:major    # Breaking change
```

This updates `package.json` and `src/module.ts` and creates a commit.

### 5. Commit and Push

```bash
git add src/templates/ src/module.ts
git commit -m "feat: description"
git push
```

<Aside type="caution" title="Commit directly to main">
This is an upstream module repo. Downstream apps pull changes via `npm update`. Commit directly to `main` — feature branches delay changes reaching downstream.
</Aside>

## Version Management

The module sits on **version line 1.x**. The redesign shipped as **v1.1.0** (July 2026), replacing the legacy surface wholesale — nothing downstream imported it, which is what allowed the clean replacement.

### When to Bump

**Always bump after ANY change to templates or module.ts.** This is how the update system detects changes.

| Changed File | Bump Required? |
|-------------|----------------|
| `src/templates/**/*.ts` | ✅ Yes |
| `src/module.ts` | ✅ Yes |
| `CLAUDE.md`, `README.md` | ❌ No |
| `docs/**` | ❌ No (push the docs submodule instead) |

### Version Commands

| Command | When to Use |
|---------|------------|
| `npm run version:patch` | Bug fixes, template fixes, minor adjustments (no changelog needed) |
| `npm run version:minor` | New features, new templates, enhancements (changelog required) |
| `npm run version:major` | Breaking changes requiring app migration (changelog required) |

Record notable changes in the [Changelog](/modules/ai-core/changelog/).

### Tagged Releases

For publishing with git tags:

```bash
npm run release:patch    # Bumps, commits, and creates a git tag
npm run release:minor
npm run release:major
```

After a tagged release, push tags: `git push --tags`.

### Why Bumping Matters

1. **Version headers** — generated files include the module version in their headers. Without a bump, regenerated files show the old version.
2. **Update detection** — `npx ai-core update` compares the installed version against the module version. No bump means no update detected.
3. **Hash comparison** — the update system compares file content hashes, but only after the module version has changed.

## Scaffold Updates

The module's infrastructure (CLI, init logic, update system) comes from the **module starter template**. AI Core adopted the shared scaffold at **v1.19.1 (July 2026)**, bringing the repo onto the current module contract before the v1.1.0 implementation — the design's first instruction (*"before anything else: run `scaffold:update`"*), carried out.

### Check for Updates

```bash
npm run scaffold:check
```

This shows what's available without changing anything — your current scaffold version and whether updates exist upstream.

### Apply Updates

```bash
npm run scaffold:update
```

This fetches from the scaffold upstream remote and merges changes into your repo.

### What Gets Updated

| Updated | Not Updated |
|---------|-------------|
| `src/scaffold/**` (init, update, types, utils) | `src/templates/**` (your templates) |
| `src/cli.ts` | `src/module.ts` (your config) |
| `scripts/**` | `CLAUDE.md` (module-specific sections) |
| CLAUDE.md scaffold sections (between markers) | Your customisations |
| `tsup.config.ts`, `tsconfig.json` | `package.json` (name, description, bin) |

### Handling Merge Conflicts

Scaffold updates use `git merge`, so conflicts are possible if you've modified scaffold files (which you shouldn't have). If you get conflicts:

1. Check if the conflicting file is in `src/scaffold/` — if so, accept the upstream version
2. If the conflict is in a shared config file, review both sides and merge manually
3. Run `npm run build` after resolving to verify

## Module Conventions

These are the conventions the implementation follows. They are part of the design, and as of v1.1.0 the shipped templates follow them.

### The Provider SDK Is an Implementation Detail

The implementation uses the **Vercel AI SDK (v7) as the transport layer** — it has the widest provider coverage for this stack, and it is a thin adapter rather than an orchestration framework. But it is built so that replacing it is a contained job: a thinner, more architecturally-aligned option exists (TanStack AI) and is currently too young to build a legal tool on; it may not be in a year. Nothing in the design may make that migration impossible. Concretely:

- **The SDK appears in one layer.** Provider adapters only — a single adapter file in the generated backend. It must not appear in operations, assembly, the admin, or any consumer.
- **None of its opinionated pieces are used.** Agent loops, chat-UI hooks, prompt helpers — all of it duplicates or contradicts what this module owns. Only the generation, embedding and rerank primitives are used.
- **AI Core defines its own provider interface** and has the SDK satisfy it, rather than exposing the SDK's interface and adapting inward.
- **AI Core owns parameter validation above the SDK.** This one is not portability — it is the module's transparency rule. The SDK silently clamps out-of-range values on at least one provider and emits a suppressible console warning. **A silently corrected parameter is a lie about what ran.** AI Core validates against the model record's own rules and either rejects or transforms **explicitly, visibly and in the ledger**.

> **The exit test:** swapping the SDK should touch the adapter layer and nothing else.

The same discipline applies outward: consumers see AI Core's surface and never `@anthropic-ai/sdk`. See [Integration](/modules/ai-core/integration/) for the consumer contract.

### Module Contract

| Convention | Rule |
|-----------|------|
| Version line | 1.x — nothing downstream imports the current surface; it can be replaced cleanly |
| Dependencies | **none** — AI Core is the foundation; nothing may make it depend on a consumer |
| Table prefix | `ai_` |
| Database changes | migrations, **never** `prisma db push` |
| Security | `security.registration.ts` — deletion handlers, encrypted fields, the manifest. AI Core has user-linked data; it must not be the hole in RLS |
| Registries | `serviceRegistry` (the consumer API), `adminPagesRegistry`, `settingsPagesRegistry`, `preferencesRegistry`, `securityRegistry` |
| `ai.config.ts` | install-time architecture choices only — **not** model assignments; those are records |

The last row matters more than it looks. Model assignments live in admin-managed records — seeded at install, editable, resettable — never in config or code. **Ships-with is not hardcoded.** See [Configuration](/modules/ai-core/configuration/) for what is a record and what is config.

### Code Conventions

- **Australian spelling in prose** and user-facing text; **American spelling in code**
- **Three-field file headers** on every file (`FILE` / `PURPOSE` / `DESCRIPTION`)
- **`.js` extensions** on all backend relative imports (ESM requirement)
- **`createModuleLogger`** for all backend logging
- **TanStack Query** for server state, with module-namespaced keys:

```typescript
// ✅ Correct — namespaced under the module id
const modelsQuery = useQuery({ queryKey: ['ai-core', 'models'] });

// ❌ Wrong — collides with other modules
const modelsQuery = useQuery({ queryKey: ['models'] });
```

## Usage, Tokens & the Ledger

> **The point is the framework, not thrift.**

Cost control is not a design value here and must not be built as one. The job is to record enough that the system can be optimised later — with measurement and learning rather than guesswork.

### The Ledger

Every operation writes a ledger entry: what ran, which model, tokens in and out, latency, outcome. Enough to answer *"which model actually does this job well"* with data instead of an opinion. See [Usage](/modules/ai-core/usage/) for the operation surface the ledger records.

**Limits come from measurement.** When a limit is eventually needed, it is derived from what the ledger has seen — not a number pulled from the air.

### Re-Embedding Is AI Core's Problem

**Planned — medium-term; not in v1.1.0.** AI Core owns the embedding model, so it owns the moment the model changes: the existing vectors are invalid, and only AI Core knows it. It reports the scale, then migrates in batches with progress. This is the transparency rule, not thrift — silently leaving stale vectors in place is the same failure as silent truncation. The failure the module prevents is never merely that the AI was wrong; it is that **the AI was wrong and looked right**.

### The Semantic Cache

**Planned — medium-term; not in v1.1.0.** A semantic cache belongs in AI Core: it sees all provider traffic, so every consumer benefits without knowing the cache exists. Two hard rules:

- **Cache keys are scoped to the owner, hard.** A cache keyed on similarity alone will serve one person's answer to another — a near-match is not the same question, and this is a real, reproduced leak, not a theoretical one. Tenancy here is simple (a tenant is a user), which makes the rule simple too: **the owner is part of the key, always, independent of similarity.**
- **Sensitive content respects the provider-terms flag.** Content flagged sensitive is not cached, or is cached only where terms permit.

### Audit by Reference

Legal work needs a past decision to be explicable. Per operation, the ledger records:

| Recorded | As |
|----------|----|
| Exact model version | pinned version id, never an alias |
| Prompt and discipline versions | version ids |
| Retrieval set | record IDs with their trust levels |
| Citations produced | with their verification status |
| Parameters actually sent | including any transformation AI Core applied |
| Groundedness score | the number, with its scorer |
| Provider terms in force | for that call |
| Degradation | any fallback or degradation flag |

**All of it by reference — IDs, versions and hashes, never the content.** The audit trail must be defensible without itself becoming a second copy of the sensitive data. The rows tied to retrieval and groundedness scorers populate when those medium-term pieces land — in v1.1.0 the confidence recorded is the schema-declared confidence, where an operation's schema provides one.

### Billing Is a Callback

Billing has no metering and will not get any. AI Core owns its counter and hands billing `getCurrentUsage` — billing's `requireLimit` already takes that shape. AI Core does not depend on billing; billing does not depend on AI Core.

## Documentation Workflow

The docs live in the shared **module-system-docs** git submodule at `docs/` — the same site used by the starter template, the module starter, and every module. They deploy automatically when pushed.

### Making Documentation Changes

1. Edit `.mdx` files in `docs/src/content/docs/modules/ai-core/`
2. Commit and push **inside the submodule**:
   ```bash
   cd docs
   git add .
   git commit -m "Docs: description of change"
   git push
   cd ..
   ```
3. Bump the gitlink in the parent repo:
   ```bash
   git add docs
   git commit -m "docs: update docs submodule"
   git push
   ```

Docs changes never require a module version bump — the submodule push is the release.

### Adding a New Docs Page

1. Create the `.mdx` file in `docs/src/content/docs/modules/ai-core/`
2. Add the sidebar entry in `docs/astro.config.mjs` under the AI Core section
3. Follow the push workflow above

### Sidebar Configuration

The sidebar is configured in `docs/astro.config.mjs`. Each module has its own section under the "Modules" label, with entries for each page — see the [AI Core overview](/modules/ai-core/) for the full page set.
