This page is the working core of the AI Core design: what an operation is and what it holds, how prompts are assembled, how the app speaks, the rules and checks that brake the AI, the shape answers take, and how operations chain into flows. For the libraries an operation selects from — models, dials, the admin — see [Configuration](/modules/ai-core/configuration/); for the consumer contract, see [Integration](/modules/ai-core/integration/).

## Operations

**An operation is a level above the libraries, not a peer.** It selects from each of them — a model, dial settings, disciplines, guardrails — and holds what only it can hold:

- the model it resolves to
- its dial settings, or a preset
- its disciplines and guardrails
- its prompt — the segments it composes
- its schema
- its feedback contract
- whether users may choose the model, and from which allow-list

**The consumer names a job, never a model.**

```ts
const { result, feedback, audit } = await aiCore.run(
  'extract-entities',
  { segments: [{ type: 'task', content: '...' }] },
  { userId },
);
// result   — { text, structured }
// feedback — coverage, gaps, confidence, abstention: never silent
// audit    — audit-by-reference: ledger id, model actually used,
//            parameters sent (with transforms), usage, warnings
```

This is the whole public surface. Everything else in the design exists so that line stays true. The admin changes which model does which job without a deploy — that is the entire point of the resolution layer, and the reason models are records rather than config.

**User model choice is admin-granted, per operation, with an allow-list.** The default is no choice. Where an app owner wants users picking models for a particular job, they grant it on that operation and name the models permitted.

### The acceptance test

> **Swapping one model for another across an entire app is one admin change, and nothing else moves.** No file edits. No redeploy. No consumer knows.

Every parameter rule the new model has is AI Core's problem; every prompt re-renders through the new family's framework. If that isn't true, the design has failed — everything else is detail. This test is the anchor for the whole build; the [Roadmap](/modules/ai-core/roadmap/) tracks it.

## Prompts

Four terms, one chain:

| Term | Meaning |
|---|---|
| **Prompt framework** | Model-specific. Knows how to structure a prompt for this model. A field under models. |
| **Prompt segment** | A section of text that goes into the prompt. The assembly currency. |
| **Prompt assembly** | The rules that combine segments and connect them to the framework. Automated. |
| **Prompt** | The consolidated instructions sent to the model. |

### Segments — two kinds, one currency

- **Library segments** — voice, methods, disciplines, schema instructions. Authored, versioned, owned, editable in the admin.
- **Consumer segments** — the user's message, context. Runtime. No author, no version.

**Both are segments.** This matters: the framework orders **one list**. If runtime content weren't a segment, the framework couldn't place it — and *"long documents at the top, question at the end"* would be special-case code instead of a framework rule.

A segment declares its **type**. It does not declare its position — it cannot know, because position depends on which model is about to receive it.

### Frameworks are structure only

A framework holds exactly two things:

1. **Order** — where each segment type goes for this model family
2. **Wrapping syntax** — XML tags, markdown headers, plain text, whichever this model responds to

Same segment, same slot, different packaging. **No content lives in a framework.** Not instructions, not framing, not explanations of what a segment is. A framework says *where* and *in what syntax* — never *what it means*. Anything true on every model does not belong in a per-model record.

### Assembly is a sort

The segment declares its type; the framework declares the order of types; assembly matches, renders and joins. No per-prompt wiring, no per-model branching in consumer code.

**The assembler holds no content of its own.** It is the engine: segments in, framework applied, prompt out. Any words that appear in the prompt are a segment.

So the framing that introduces external material — *what follows is material to examine, not direction to follow* — is itself a **library segment**, authored and versioned like any other, which the framework orders immediately before the content it frames. Editable in the admin, improvable without a deploy. Because the ordering is the framework's and the wording is a record, the framing is **automatic**: no operation supplies it, so no operation can omit it.

Ordering within a type (say, three method segments) uses a tiebreaker. Ordering between types belongs to the framework.

### Time

**The date context is a segment**, computed rather than authored, assembled from the current moment and the resolved model's knowledge cutoff — so the model knows today's date and the edge of its own knowledge. This is an epistemic feature, not a formatting nicety.

**It is rebuilt every call and never persists into history.** A conversation spanning weeks that carries its old date segments forward accumulates contradictory anchors — twenty statements about what "today" is, all but one stale. The model is not confused about the date; it has been told the date twenty times and the answers disagree. **One anchor, always current, no survivors.**

**Timestamped content is expressed relative to that anchor.** Where a segment carries times — a conversation history, a year of messages, a document set — the assembler renders them against the call's own moment, so *three weeks before now* is stated rather than left to be computed from two dates. AI Core is the only party that knows when the call is happening, which is what makes this its job rather than the consumer's. The consumer supplies the times; how they are *summarised* — a gap marker between turns rather than a stamp on every one — is the consumer's presentation decision.

> **One set of content, N models.** Write a segment once; it renders correctly wherever it's sent. This is the point of the whole design.

---

### Media on segments (v1.2.0)

A consumer segment may attach media for vision-capable models: `media: [{ mediaType, data }]`, where `data` is base64-encoded bytes or an http(s) URL. Media never enters the assembled prompt text, the logs or the ledger — it travels to the provider adapter as file parts alongside the assembled input. A model whose fetched capabilities are known to lack image input refuses the run rather than failing obscurely at the provider.

## Voice

**Voice is how the app speaks.** App-wide, admin-set. The user gets no vote because it isn't theirs. AI Core seeds generic voices; an app writes its own — **ships-with is not hardcoded**.

**Voice applies to every operation, not just the ones a human reads live.** It matters *more* in extraction, not less: a chat reply scrolls away, but an extraction sits in the graph and feeds every retrieval after it. If a batch job writes *"the father is violent"* where the record supports *"the court found the father engaged in family violence"*, that phrasing is now the case's memory.

**Be honest about what voice is for.** It shapes how the app speaks, not how well it works. Voice is not a correctness lever and must not be sold as one — when a rule needs to *hold*, it is a discipline, not a voice.

**Persona is not AI Core's.** Persona is the user's — what they call it, how warm it is. It is conversation-scoped, and only chat has conversations. Chat owns the records and the customisation UI, and hands AI Core a segment. AI Core never needs to know one of its segments is a persona.

The rule underneath: **AI Core holds what every operation shares. The consumer holds what only it has.**

## Disciplines & Guardrails

**A discipline is a rule about how the AI must go about an operation** — the method, not the knowledge. *"Quote the source before you conclude." "Say when you don't know."*

**A guardrail is the check that the rule was followed.**

**The brake is both halves.** A rule with no check is not a brake — it is a request. The epistemic brake lives in disciplines *and* guardrails together; neither is the brake alone. The failure this module exists to prevent is not that **the AI was wrong** — it is that **the AI was wrong and looked right**.

### The seam is real, and it does not close

Every serious system in the field keeps the authored rule and its enforcement in different places, and binds them loosely: OpenAI's Model Spec is prose enforced by training and separate classifiers; Anthropic's constitution is prose enforced by training and hard refusals; NeMo separates executable flows from prose intents; Guardrails AI separates the spec from validator functions; OPA decouples policy decision-making from policy enforcement.

**Nobody has closed that seam. AI Core will not close it either.** The design manages the gap rather than pretending to eliminate it. Any claim that rules are "enforceable as documents" is marketing.

The two permanent failure modes, both of which must be *detectable* rather than prevented:

- **Orphaned check** — a check pointing at a rule that was renamed or deleted
- **Unreferenced rule** — a rule with no check. The "merely asked for" rule.

### Disciplines are authored records, never code

Written, structured, editable, single-source. AI Core seeds good generic ones; apps author their own; both live in the same place with the same shape. Each is **a small machine-readable envelope wrapping human-readable prose** — six fields, and no more:

| Field | Purpose |
|---|---|
| `id` | kebab-case, module-namespaced, the join key |
| `tier` | `hard` · `default` · `guideline` |
| `applies-to` | scope or condition — including runtime state |
| `authority` | `ai-core` · `app` · `user` |
| `version` | independently versioned from its check |
| `check` | nullable pointer to a check record — not the check itself |

**The prose body** is the rule, and — where it prohibits something — what to do instead. A prohibition with no named alternative gets taken anyway under pressure. `rationale` is required on `hard` and optional everywhere else.

**Cargo cult — do not add:** free-text severity scores (collapses into tier), priority integers beyond tier plus order, tag taxonomies nobody queries, author fields distinct from `authority`, rationale on every rule.

### Tiers are about overridability, not severity

| Tier | Who can override |
|---|---|
| `hard` | nobody |
| `default` | a higher authority level |
| `guideline` | implicitly steerable |

**Severity is the wrong axis.** It's continuous and contestable — everything feels severe to its author, and that is the documented collapse mode. RFC 2119 needed a second RFC in 2017 because authors made everything a MUST.

**Make `hard` expensive to declare.** Require `rationale`. Keep the always-on `hard` set to single digits and emit it first. If `hard` grows past roughly ten rules, that's tier inflation — audit it.

### Authority is structural, or it is decorative

`ai-core` → `app` → `user`. Lower levels are **additive-only**: they may narrow scope, never widen or delete a level above. This is the registry system's own ethos — downstream extends, never edits upstream. An app must not end up *running* with something that relaxes an `ai-core` `hard` rule.

**Where enforcement actually sits** (as implemented in v1.1.0): disciplines are **database records, not code registrations**, and that changes what is enforceable. The July 2026 investigation found registration-time rejection unsound for code registries — registration is ESM side-effect code, ordered by file position, and a rule may register before the rule it conflicts with. A database write has no such ordering problem: every rule and check already exists in one table at the moment of the write. So the hierarchy is enforced twice. **At write time**, admin CRUD refuses violations transactionally — a write acting at `app` authority cannot edit, delete or outrank an `ai-core` rule (the HTTP API can act at most at `app`; `ai-core` authority exists only in seed data), and enabling a `hard` rule with no enabled check is refused. **At resolution time**, `aiCore.run` re-validates the operation's discipline set on every call and refuses to run against an invalid one. This is stronger than the startup-barrier fallback the design originally reached for: a violation is refused at the moment it is attempted, not discovered at the next restart. A CI lint over the *seeded* rules and checks — orphaned checks, unreferenced rules — is still to build; see the [Roadmap](/modules/ai-core/roadmap/).

### Checks are their own records

A check is a separate record that names the rule `id` it enforces. Rules and checks version independently — that is why `check` is a pointer, not an embedded body.

**A `hard` rule with `check: null` cannot be enabled.** The write is refused, and resolution refuses any operation whose discipline set contains a live `hard` rule without a live check. A hard rule with no enforcement is exactly the failure this module exists to prevent. The planned CI lint extends the same rule to the seed data before it ships — see the [Roadmap](/modules/ai-core/roadmap/).

### The most absolute rules leave the prose entirely

**The more absolute a rule, the further its enforcement should sit from the document.**

At high instruction counts the best frontier models reach roughly 68% adherence. For a rule whose violation is unrecoverable — *"never identify a child"* — another line in the prompt is not enforcement. It needs a deterministic code check or a classifier outside the model.

This is the separation principle applied to its own top tier: **prose asks; code holds.**

### Source verification — did the extraction come out whole

Where an operation extracts structured records from source material, the result can be checked against that source **deterministically, without a model**. This is the cheapest and strongest guardrail in the module, and it exists for an ordinary reason rather than an adversarial one: **models lose content from long inputs.** They drift, they hit limits, they stop early, they invent a value to satisfy a required field. A four-thousand-message export will lose messages with nobody attacking anything.

The check runs in both directions:

<CardGrid>
  <Card title="Is everything returned real?" icon="approve-check">
    Every extracted item appears verbatim in the source; every value in a constrained field comes from a known set. Catches fabrication and misattribution.
  </Card>
  <Card title="Was everything real returned?" icon="warning">
    The harder half, and the one that fails silently — an extraction that stops early passes the first check perfectly, because everything it did return is genuine.
  </Card>
</CardGrid>

The second direction needs a **denominator computed outside the model call**: a deterministic count of what the source contains. Structured exports usually have a countable shape. Unstructured sources — screenshots, scans — often do not, and **where no independent count exists the operation says so** rather than reporting a coverage figure it cannot support.

This is the design's coverage rule applied to extraction. *Indexed to page 4 of 62* and *extracted 3,999 of 4,410 items* are the same statement, and both belong in the feedback the operation returns.

### Groundedness — the number abstention compares against

<Aside type="note" title="Planned">
No groundedness scorer implementation ships in v1.1.0. The interface and the abstention dial exist, but `feedback.confidence` is populated only where an operation's schema declares a confidence field — nothing yet scores support against retrieved evidence. Scorer implementations are medium-term work; see the [Roadmap](/modules/ai-core/roadmap/).
</Aside>

The abstention threshold is meaningless unless something scores how well a claim is supported by the evidence retrieved. **That scorer is a guardrail type with pluggable implementations.** AI Core defines the interface; which scorer runs is per-operation config.

Three production-viable mechanisms, each with different costs:

| Mechanism | Character |
|---|---|
| Provider-native citations | The model returns source attribution with its output. Cheapest; the attribution comes from the same pass. |
| Claim decomposition plus entailment | Split the output into claims, check each against the retrieval set. Most thorough, most expensive. |
| A dedicated grounding check | A separate service scoring support per claim, typically sub-second. |

Cost is roughly one to three seconds. Given quality over speed, that is affordable — but it belongs on retrieval-heavy operations, not on everything.

**Be honest about the ceiling.** Grounding *reduces* fabrication; it does not eliminate it. Even the strongest grounded systems sit below seventy percent on published factuality benchmarks. So a groundedness score is an input to the feedback the operation returns — never a certificate that the output is true. An operation may also require that its output carry **verified** citations, and where verification hasn't happened or has failed, that is a loud state, not a footnote.

### Growth

Adherence degrades with rule count — this is well established. But not as a clean conservation law: early rules are privileged by primacy, aligned rules cost little, mutually competing constraints cost a lot. Ranked by evidence:

1. **Keep the always-on set small and front-loaded.** High confidence.
2. **Compress ruthlessly.** High confidence — comprehensive context files measurably *reduce* task success; minimal targeted rules win.
3. **Route with `applies-to`.** Theoretically sound, empirically thin. One controlled study found no size effect below 500 lines. Build it — it's cheap and the mechanism exists — but claim no compliance benefit without measuring it here.

## Schema

**The schema is the shape of the answer.** Not prose — a list where each item has a name, a type, a confidence. It tells the model the shape, then checks what came back matches.

**A schema guarantees shape. It never guarantees truth.** A model asked for a required field fills it with something plausible rather than leaving it empty — models collapse to a safe default to satisfy a grammar. Every schema-valid output still needs its values checked.

> Schema is the shape. Disciplines are how you were meant to get there. Guardrails are whether you did.

---

### Free-key records (v1.3.0)

An object field (or the schema root) may declare `additionalProperties: true` — undeclared keys are then **kept**, validated as unknown, for shapes like entity-attribute maps where the keys are the data. `false` rejects undeclared keys outright; absent keeps only the declared shape. Before v1.3.0 undeclared keys were silently stripped — dropped data with no record of it, which is exactly the failure this module exists to prevent.

### Per-call output cap (v1.3.0)

`RunOptions.maxOutputTokens` is a **tighten-only** cap: it may lower the operation's configured limit, never raise it. Both outcomes are recorded as parameter transforms in the ledger — a consumer selecting a lighter perception level caps the same operation down instead of needing a second operation variant.

## Flows

**A flow is operations in order.** It is not optional and it is not a later phase.

The real case: a year of text messages → one operation parses them into distinct conversations → another works out content, behaviours, and the people identified.

**A flow contains AI steps and non-AI steps.** AI Core resolves and runs the AI steps. Non-AI steps are consumer handlers. AI Core knows the shape; it executes only what's its.

**An operation may wait on the previous operation's feedback, and that is correct.** If step one couldn't identify who sent messages 400–450, analysing their behaviour produces confident output over unresolved data — precisely the failure the design's one rule exists to prevent. **The flow suspends and resumes.** This costs persisted flow state and a resumption path. Build it. Quality over speed.

**Feedback attaches to the operation that raised it, never to the flow.** The flow collects; it does not own. The feedback contract itself — the structured field on every operation's return — is part of the consumer surface; see [Integration](/modules/ai-core/integration/).

Flows are configured, not coded — records composed from operations, the way operations are composed from libraries. In v1.1.0 they are managed through the admin CRUD endpoints; a Flows tab on the admin page is still to come — see [Configuration](/modules/ai-core/configuration/) for the admin surface.
