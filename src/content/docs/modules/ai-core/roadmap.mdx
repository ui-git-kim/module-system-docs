## Known Issues

The six issues recorded here from the July 2026 alignment audit all lived in the **legacy template surface**, and all were **resolved by deletion** — v1.1.0 replaced that surface wholesale (see the [Changelog](/modules/ai-core/changelog/)). These are the current known issues.

### First Live Provider Call Not Yet Made

**Severity:** Medium
**Where:** the host app (install 2026-07-23; admin page verified 2026-07-24)

Install, type-check and the admin page are all verified in a real app: the schema and all six seed migrations applied (the migration runner's first real use), the page renders at `/admin/ai-core`, and the Models tab serves the seeded provider and model records — with terms shown as *not recorded*, capabilities as *never fetched*, and every seeded row marked *shipped default*. What has **not** happened yet is a real `aiCore.run()` against a live provider: prompt assembly, the adapter call, schema validation, feedback and a ledger write have never executed end to end outside the type harnesses.

**Workaround:** record terms on a provider (required if the app installed with the `sensitive` floor), create an operation, and run it; anything found feeds the next patch.


### Knowledge Cutoff Must Be Admin-Entered

**Severity:** Low
**Where:** model records / capability refresh

Capability refresh reads the providers' REST model listings, but does not derive a model's knowledge cutoff from them. The cutoff feeds the date-context segment — the model's anchor for the edge of its own knowledge — so an unset or stale value weakens that anchor. It must be entered on the model record in the admin.

### No CI Lint Yet for Seed Discipline/Check Integrity

**Severity:** Low
**Where:** `src/templates/migrations/`

Write-time and resolution-time enforcement are live (see [Usage](/modules/ai-core/usage/)), but the designed CI lint — the one that reports orphaned checks and unreferenced rules in the *seed data* before it ships — has not been built.

### ~~Frontend Prop Shapes Verified Against Stubs~~ (resolved in v1.1.2)

The first real install confirmed exactly this class: `ErrorAlert`'s real prop is `error`, and the query-keys helper takes scalar parts. Both fixed in v1.1.2–v1.1.3; the host app's frontend now type-checks the feature clean against the real starter components.


## Planned Improvements

### Short-Term

#### Install Verification

The one blocking gate: install v1.1.0 into a real starter app with `npx ai-core init`, run the migrations and seeds, exercise the admin page and `aiCore.run`, and confirm the frontend compiles against the real starter components. Everything under Known Issues that stubs or harnesses could not confirm resolves here.

### Medium-Term

#### Retrieval (design §16)

If AI is doing the retrieving, AI Core owns the mechanism — semantic retrieval as an operation like any other: it uses a model, it can be wrong, it must report what it found **and what it didn't**. AI Core retrieves; Structure stores. In v1.1.0 the context slot is consumer-filled only, and the grounding dials that would configure retrieval are recorded but drive nothing. See [Integration](/modules/ai-core/integration/).

#### Groundedness Scorers

A pluggable guardrail type, configured per operation: provider-native citations, claim decomposition plus entailment, or a dedicated grounding check. The score is what the abstention threshold compares against, and it is an input to the feedback an operation returns — never a certificate that the output is true. It belongs on retrieval-heavy operations, not on everything. In v1.1.0 the interface and the abstention dial exist, but no scorer runs — `feedback.confidence` is populated only by schema-declared confidence fields.

#### Semantic Cache

AI Core sees all provider traffic, so every consumer benefits without knowing the cache exists. Cache keys are scoped to the owner, hard — a cache keyed on similarity alone will serve one person's answer to another, and that is a real, reproduced leak. Sensitive content is not cached, or cached only where provider terms permit.

#### Re-embedding Migration

AI Core owns the embedding model, so it owns the moment the model changes: the existing vectors are invalid, and only AI Core knows it. Report the scale, then migrate in batches with progress. Silently leaving stale vectors in place is the same failure as silent truncation.

#### Review-Queue UI Polish

The review-queue table and its feed from abstentions and low-confidence outcomes shipped in v1.1.0. What remains is surface polish: the consumer-facing component a module can drop in, and the admin gaps alongside it — most visibly a Flows tab (flows have full admin CRUD endpoints but no tab of their own).

#### User Model-Choice UI

The API capability shipped in v1.1.0 — model choice is admin-granted per operation with an allow-list, default off, and a user-selected model outside the allow-list is refused. There is no user-facing UI for making the choice yet.

### Long-Term / Open Questions

The design's own open questions (§21), carried here so they stay visible.

#### Discipline Authority Enforcement

The design's original question — can a registry reject a registration that conflicts with another registry's contents? — was answered in July 2026: registration-time rejection is unsound for **code registries** in the starter's ESM side-effect model, where a rule may register before the rule it conflicts with. The implementation landed somewhere stronger than the startup-barrier fallback the design reached for: disciplines are **database records**, which have no such ordering problem, so v1.1.0 enforces the hierarchy at **write time** — admin CRUD refuses violations transactionally — and re-validates at **resolution time**, where `aiCore.run` refuses an invalid set (see [Usage](/modules/ai-core/usage/)). What remains open is the CI lint for seed discipline/check integrity, recorded under Known Issues. The principle underneath held throughout: **prose asks; code holds.**

#### Which Groundedness Scorer per Operation

The mechanism is settled and pluggable; which implementation runs on which operation is a per-operation decision that needs measurement, not a design argument.

#### Image and Video Provider Data-Handling Terms

Black Forest Labs and Replicate have materially weaker data-handling terms than the model providers. In v1.1.0 their provider rows are seeded **disabled** and their generation adapters are unbuilt (a call throws a typed not-implemented error) — nothing can be routed to them at all yet. Confirm their terms before they are ever enabled; the likely answer for sensitive content is that it never is, enforced by the sensitivity gate in resolution.

#### Chat Module or Generic Chat

If chat goes generic, persona has nowhere natural to live. Persona is the user's, conversation-scoped, and only chat has conversations — chat owns the records and hands AI Core a segment.

#### Naming Collisions

`prompt`, `token` and `model` collide with the module system's own terms. A naming pass is needed before the admin surface freezes.

## Completed

Items that have been done.

### v1.3.0 — Consumer-Integration Surface (July 2026)

Closed the six raise-now gaps from cog-ingest's consumer-side readiness audit: `additionalProperties: true` free-key records in both zod builders (undeclared keys were being silently stripped — silent data loss, the exact failure this module exists to prevent); `ensureOperation` / `ensureSegment` idempotent seeding where consumers state model *requirements* rather than model names and admin edits always win; `RunOptions.maxOutputTokens` as a tighten-only cap recorded in the ledger; automatic framing for external content, with a missing framing segment reported in feedback rather than proceeding unframed; and corrected `run()` examples throughout the docs.

### v1.2.0 — Vision Input (July 2026)

Media attachments on consumer segments (`media: [{ mediaType, data }]`), handed to the provider adapter as file parts alongside the assembled input — never into the prompt text, the logs or the ledger. A model whose fetched capabilities are known to lack image input refuses the run rather than failing obscurely at the provider.

### v1.1.1–v1.1.3 — First Real Install (July 2026)

`npx ai-core init` into a real app, end to end: schema appended, feature generated, registrations wired, all six seed migrations applied (the module migration runner's first real use). Three defect classes the type harnesses could not have caught were found and fixed — Prisma 7 custom-output client imports, real starter component prop shapes, and the barrel's routes export — after which the host app type-checks the feature clean. A scaffold bug (scoped package specs mangled to `@scope/name@^4@latest`) was fixed upstream in module-starter scaffold v1.19.2.

### v1.1.0 — The Redesign Implemented (July 2026)

The whole former Short-Term column shipped as module **v1.1.0** (2026-07-23):

- **The `ai_` schema** — twelve models via the migrations runner, plus six seed migrations (providers, frameworks, segments, disciplines, checks, models). Every battery is a row: seeded, editable, resettable
- **Provider adapters on Vercel AI SDK v7** — Anthropic, OpenAI, Google, Voyage — confined to a single adapter file behind an interface AI Core defines
- **The operations resolver and `aiCore.run`/`runEmbed`** — the sensitivity gate, explicit parameter transforms, the Structure dimension gate, structured feedback on every return
- **Prompt segments, frameworks and assembly** — the computed date anchor and EXT-marked external content
- **Disciplines and guardrail checks** — authority enforced at write time and re-validated at resolution time
- **Flows with suspend/resume** and `initialState`; **the usage ledger** (references only) with the `getCurrentUsage` billing callback; **the review queue**
- **The tabbed admin page** at `/admin/ai-core` (order 30, platform section) and **the security registration** with deletion handlers
- **The legacy v2 surface deleted wholesale** — resolving all six known issues from the July 2026 audit

**Verification:** strict type harnesses over the extracted generated code — the backend against the real published SDK packages, the frontend against React 18 and TanStack Query v5. **Install verification has not yet been performed**; installing into a real starter app is the next gate, recorded under Known Issues. See the [Changelog](/modules/ai-core/changelog/) for the release detail.
