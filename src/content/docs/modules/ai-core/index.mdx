The **AI Core** module is the set of mechanisms an app needs in order to have AI in it — models as admin-managed records, operations called by name, prompts written once and rendered per model, and a single rule that every surface answers to: open, transparent communication instead of guessing.

<Aside type="note" title="Implemented in v1.5.0 — first live call pending">
This page describes AI Core as implemented in module v1.5.0 (July 2026). The module installs into a real app, type-checks clean there, and its admin page renders and serves records at runtime. What is **not** yet verified is a first live provider call. Items still unbuilt (retrieval, semantic cache, groundedness scorers) are marked inline — see the [Roadmap](/modules/ai-core/roadmap/) for status.
</Aside>

## What AI Core Is

The starter template makes an app an app — auth, admin, settings, registries. **AI Core makes it an AI app.** That is the bar: not "does it work" but that installing it should leave an app AI-capable the way installing the starter leaves it app-capable. Every AI-using module is a consumer of it.

It ships **mechanism, not opinions** — but it ships **batteries**. Every built-in is a record: seeded at install, editable, resettable. Like a theme shipping a default palette — you can change every colour; you just don't start from white.

> **Ships-with is not hardcoded.**

Nothing in AI Core knows what any app does. Consumers name jobs; AI Core runs them.

## The One Rule

> **Open, transparent communication — instead of guessing, or making things up.**

AI can be wrong. It can have limits. It can fail to find what it needed. What it may never do is move on silently, letting that quietly shape everything downstream.

The design value, stated plainly: **the user should not have to be as discerning as the person who built this.** *"AI can make mistakes"* in small text at the bottom of the screen is not enough — especially in legal work. The burden of noticing sits with the AI, at the point of uncertainty. Not with the user, in a footer.

This is the reason the module exists, not a feature of it. Every surface below is the same rule wearing a different hat:

| Surface | What it communicates |
|---|---|
| Embeddings | coverage — *"indexed to page 4 of 62"*, never a silent cut |
| Claims | the source relied on — or that there wasn't one |
| Abstention | the gap, named — never silence, never a plausible fill |
| Truncation | what was left out |
| Retrieval | what it found **and what it didn't** |
| Feedback | what it couldn't do, structured, on the way back |

The failure this module prevents is not *"the AI was wrong."* It is **"the AI was wrong and looked right."**

Truncating a 60-page affidavit is sometimes correct engineering. Truncating it silently never is. **The system may do less than it was asked. It may not let anyone believe otherwise.**

There are no exceptions to this rule.

## The Shape

Three levels. Everything else is detail.

```
LIBRARIES        models · prompt segments · disciplines · guardrails
                 records, seeded by AI Core, edited by the admin
                          ↓ selected by
OPERATION        the named job — selects a model, configures the dials,
                 selects disciplines and guardrails, holds the prompt and
                 schema, returns the result and the feedback
                          ↓ sequenced by
FLOW             operations in order
```

**A consumer calls an operation by name. It never names a model.**

```ts
const { result, feedback, audit } = await aiCore.run(
  'extract-entities',
  { segments: [{ type: 'task', content: '...' }] },
  { userId },
);
```

This is the whole public surface. Everything above exists so that line stays true. The [Usage](/modules/ai-core/usage/) page walks through the libraries, operations and flows in detail; [Integration](/modules/ai-core/integration/) covers the consumer contract.

## The Acceptance Test

**Swapping one model for another across an entire app is one admin change, and nothing else moves.** No file edits. No redeploy. No consumer knows. Every parameter rule the new model has is AI Core's problem; every prompt re-renders through the new family's framework.

If an app owner can do that, then models are records, operations resolve, frameworks render, and prompts are written once. If they can't, the design has failed — nothing else in the document matters.

Every other decision in the design exists to make that one change true.

## Current Status

Stated plainly:

- **The design is complete** (July 2026). It is the single source of truth for what AI Core is.
- **The design is implemented** as module **v1.1.0** (2026-07-23): the `ai_` schema and seeds, the provider adapters, the operations resolver and `aiCore.run`, prompt assembly, disciplines with enforced authority, flows with suspend/resume, the usage ledger, the review queue, the tabbed admin page, and the security registration.
- **v1.2.0 added vision input** — media attachments on segments, handed to the provider as file parts and capability-gated.
- **v1.3.0 added the consumer-integration surface** — free-key records in schemas, `ensureOperation` / `ensureSegment` seeding, a tighten-only per-call output cap, and genuinely automatic framing for external content.
- **It installs.** `npx ai-core init` into a real app applies the schema, generates the feature, wires the registrations and runs all seven seed migrations; the generated backend and frontend type-check clean there. Runtime — app boot, the admin page, a live provider call — is the remaining gate.
- **Verification so far.** Strict type harnesses over the extracted generated code (backend against the real published SDK packages, frontend against React 18 and TanStack Query v5), then a real install — which found three defect classes the harnesses could not reach, fixed in v1.1.1–v1.1.3. **Runtime is the remaining gate**: app boot, the admin page, and a first live provider call.
- **Some designed pieces remain unbuilt:** retrieval, the semantic cache, groundedness scorer implementations, image and video generation adapters, and transcription. Each is marked where it appears in these pages, and all are tracked on the [Roadmap](/modules/ai-core/roadmap/).
- **The legacy surface has been deleted.** The earlier surface — the January 2026 release (content processors, prompt registry) and the never-released v2 work that followed (purpose keys, image and video generation) — was never installed anywhere. Nothing downstream imported it, which is what allowed v1.1.0 to replace it wholesale.

See the [Changelog](/modules/ai-core/changelog/) for what has actually happened in the repo, and the [Roadmap](/modules/ai-core/roadmap/) for the build phases ahead.
