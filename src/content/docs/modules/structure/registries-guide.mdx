This page is the definitive reference for each registry, module-registered fields, and the full set of backend hooks.

## DB-Synced Registries

These registries are backed by database tables. Their state is managed through the admin panel and synchronised to the frontend registry on load.

### 1. nodeTypeRegistry

Manages node type configurations. Each node type defines an entity in your application — Clients, Projects, Matters, etc. The registry syncs with the `structure_node_type` database table.

**Admin location:** App Structure > Structure Types

```typescript
import { nodeTypeRegistry } from '@/features/structure';
```

**Methods:**

| Method | Returns | Description |
|--------|---------|-------------|
| `getEnabledTypes()` | `NodeTypeConfig[]` | All enabled types, sorted by `sortOrder` |
| `getConfig(slug)` | `NodeTypeConfig \| undefined` | Get the bare config for a single type |
| `getAllTypes()` | `NodeTypeConfig[]` | All types (enabled and disabled), sorted by `sortOrder` |
| `get(slug)` | `NodeTypeRegistration \| undefined` | Get the wrapper (config + optional component overrides) |
| `getAll()` | `NodeTypeRegistration[]` | All registrations (wrappers) |
| `getByModule(moduleId)` | `NodeTypeConfig[]` | Types owned by a specific module |
| `getSlugs()` | `string[]` | All registered slugs |
| `has(slug)` | `boolean` | Check if a slug exists |
| `isLoadedFromDb()` | `boolean` | Whether `syncFromDatabase` has completed |
| `syncFromDatabase(types)` | `void` | Replace registry state with database records; fires `onSync` |
| `onSync(callback)` | `() => void` | Subscribe to sync events — returns an unsubscribe function |
| `getGroups()` | `NodeTypeGroup[]` | All node type groups |
| `syncGroups(groups)` | `void` | Replace group data in the registry |

<Aside type="caution" title="get() vs getConfig()">
`get(slug)` returns a `NodeTypeRegistration` wrapper that bundles the config with optional component overrides. Most of the time you want `getConfig(slug)` which returns the bare `NodeTypeConfig`. Use `getAll()` / `getAllTypes()` accordingly.
</Aside>

**Reactive hooks:**

```typescript
import { useEnabledNodeTypes, useStructureNavItems } from '@/features/structure';

// Re-renders when node types are synced from the database
const enabledTypes = useEnabledNodeTypes();
const navItems = useStructureNavItems();
```

**Registering a node type from a module (backend):**

Seed shared node types from your module's backend init via `importNodeTypes` — the only service path that creates shared (`userId: null`) rows every user can see. `createNodeType` would pin the type to the passed user, and the existence check is load-bearing: merge-mode import updates existing slugs, so an unguarded import would clobber admin customisations on every boot. The actor string is only a visibility/audit label. See [Integration — Registering Node Types](/modules/structure/integration/#registering-node-types-from-your-module) for the full pattern.

```typescript
// backend/src/features/tasks/tasks.init.ts
import { serviceRegistry } from '../../config/service.registry.js';

export async function initTasks(): Promise<void> {
  const structureService = serviceRegistry.get('structure');

  // Guard is load-bearing — merge-mode import updates existing slugs
  const existing = await structureService.listNodeTypes('tasks-init', true);
  if (existing.some(t => t.slug === 'task-board')) return;

  // Shared (userId: null) row — visible to every user
  await structureService.importNodeTypes('tasks-init', {
    nodeTypes: [{
      slug: 'task-board',
      name: 'Task Board',
      namePlural: 'Task Boards',
      icon: 'Kanban',
      moduleId: 'tasks',  // always set moduleId
      isSystem: true,
      isEnabled: true,
      // ... full config
    }],
  }, 'merge');
}
```

### Connection Types

Manages typed connection configurations. Connection types define how nodes can be linked — "Related", "Depends On", "Party", etc. Backed by the `structure_connection_type` database table.

**Admin location:** App Structure > Connection Types

Connection types are managed entirely through the admin UI and API — there is no code-level `register()` call. Use the REST endpoints for CRUD:

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/structure/connection-types` | List all connection types |
| GET | `/api/v1/structure/connection-types/:slug` | Get a single type |
| POST | `/api/v1/structure/connection-types` | Create (admin only) |
| PATCH | `/api/v1/structure/connection-types/:slug` | Update (admin only) |
| DELETE | `/api/v1/structure/connection-types/:slug` | Delete (admin only) |

**Configuration interface:**

```typescript
interface ConnectionTypeConfig {
  slug: string;
  name: string;
  namePlural: string;
  description?: string;
  icon?: string;
  color?: string;
  moduleId?: string | null;
  fromTypes?: string[];        // allowed source node types (empty = any)
  toTypes?: string[];          // allowed target node types (empty = any)
  bidirectional?: boolean;     // default true
  maxConnections?: number;     // per node (null = unlimited)
  unique?: boolean;            // one connection per pair only
  fields?: ConnectionTypeField[];
  isSystem?: boolean;
  isEnabled?: boolean;
  sortOrder?: number;
}
```

<Aside type="tip">
Connection types support the same field system as node types. You can add data fields to connections — for example, a "Party" connection between a Matter and a Client might have a "Role" select field (Plaintiff, Defendant, Witness).
</Aside>

## Content Registries

These 8 registries are populated by module code at import time via side-effect imports. Each uses the same pattern: import the registry, call `register()` with a configuration object.

<Aside type="note" title="Common properties">
All content registrations share these properties: `id` (unique identifier), `moduleId` (your module's ID), `nodeTypes` (`'*'` for all types or `string[]` of slugs), and `order` (display position, default 50). Structure's built-in items use orders 10-25, so place yours at 30+.
</Aside>

### 2. nodeTabRegistry

Adds tabs to node detail pages. This is the most common integration — your module adds a tab that appears on specific (or all) node types.

```typescript
import { nodeTabRegistry } from '@/features/structure';

nodeTabRegistry.register({
  id: 'tasks',
  moduleId: 'tasks',
  label: 'Tasks',
  icon: 'CheckSquare',
  nodeTypes: '*',           // or ['project', 'client']
  order: 30,
  component: TasksTab,
  // Optional:
  // getBadge: async (nodeId) => taskCountForNode(nodeId),
  // isVisible: (node, nodeType) => nodeType.features?.tasks !== false,
});
```

**Interface:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `id` | `string` | Yes | Unique tab identifier |
| `moduleId` | `string` | Yes | Your module's ID |
| `label` | `string` | Yes | Tab display label |
| `icon` | `string` | -- | Lucide icon name |
| `nodeTypes` | `string[] \| '*'` | Yes | Which node types show this tab |
| `order` | `number` | -- | Position in tab bar (default 50) |
| `component` | `ComponentType<NodeTabProps>` | Yes | Receives `{ node, nodeType }` |
| `getBadge` | `(nodeId: string) => Promise<string \| number \| null>` | -- | Async badge count/text |
| `isVisible` | `(node, nodeType) => boolean` | -- | Dynamic visibility |

**Admin control:** Admins can enable/disable, rename, and reorder tabs per node type on the Integrations tab.

### 6. nodeFieldTypeRegistry

Extends the field type system with custom field types beyond the 21 built-in types. Once registered, admins can assign your field type to fields in the node type configuration.

<Aside type="tip" title="Field types are genuinely module-owned (v1.21.0)">
The admin field-type picker is **driven by this registry**, not a hardcoded list — Structure registers its own built-ins here under `moduleId: 'structure'`, and your type sits alongside them, labelled with your module id. It appears when your module is installed and disappears when it is uninstalled.

Two consequences worth designing for:

- **`configSchema` renders.** Options you declare appear in the admin field editor and are stored verbatim on `field.config`. Structure never interprets that object — read it in your own renderer.
- **A field can outlive its type.** If your module is uninstalled, fields configured with your type remain in the node's `data`. Structure renders them **read-only with an explanation and preserves the value** — it never binds an unknown type to a text input, which would overwrite a stored array or object with a string.

A module-registered type **overrides a built-in with the same `id`**, which is how a storage module supplies `media` (see below).
</Aside>

```typescript
import { nodeFieldTypeRegistry } from '@/features/structure';

nodeFieldTypeRegistry.register({
  id: 'rich-text',
  moduleId: 'editor',
  label: 'Rich Text',
  icon: 'Type',
  category: 'custom',
  order: 50,
  component: RichTextEditor,
  displayComponent: RichTextDisplay,
  // Optional:
  // validate: (value, fieldConfig) => value ? null : 'Required',
  // transformBeforeSave: (value) => sanitiseHtml(value),
  // getDefaultValue: () => '',
  // configSchema: [{ id: 'toolbar', type: 'select', label: 'Toolbar', ... }],
});
```

**Interface:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `id` | `string` | Yes | Field type identifier (e.g., `'rich-text'`) |
| `moduleId` | `string` | Yes | Your module's ID |
| `label` | `string` | Yes | Name shown in admin field picker |
| `icon` | `string` | -- | Lucide icon for the picker |
| `description` | `string` | -- | Help text in the selector |
| `category` | `'text' \| 'media' \| 'reference' \| 'layout' \| 'custom'` | -- | Field categorisation |
| `order` | `number` | -- | Position in field type picker (default 50) |
| `component` | `ComponentType<FieldTypeRenderProps>` | Yes | Form field component for editing |
| `displayComponent` | `ComponentType<FieldTypeDisplayProps>` | -- | Read-only display for cards/columns |
| `validate` | `(value, fieldConfig) => string \| null` | -- | Custom validation |
| `transformBeforeSave` | `(value) => unknown` | -- | Transform before saving |
| `getDefaultValue` | `() => unknown` | -- | Default value factory |
| `configSchema` | `NodeTypeField[]` | -- | Extra admin config fields |

**Component props:**

- **Edit:** `{ field: NodeTypeField, value: unknown, onChange: (value) => void, error?: string, disabled?: boolean }`
- **Display:** `{ field: NodeTypeField, value: unknown }`

The `NodeFieldRenderer` component automatically checks `nodeFieldTypeRegistry` for module-provided types before falling back to built-in rendering — on both the edit and read-only display paths.

#### `media` is module-provided, not built in

Structure owns the media *field*, not file *storage* — storage belongs to a content/document module (see [Scope boundaries](/modules/structure/#scope-boundaries)). So **there is no built-in `media` field type**: a storage module registers one, and where no such module is installed the type simply isn't offered.

```typescript
// In your storage module's registration
nodeFieldTypeRegistry.register({
  id: 'media',
  moduleId: 'document-management',
  label: 'Media',
  icon: 'Image',
  category: 'media',
  component: MediaUploadField,      // your uploader
  displayComponent: MediaDisplay,   // thumbnail / filename / download link
  getDefaultValue: () => null,
  configSchema: [
    { id: 'accept', type: 'text', label: 'Accepted MIME types', placeholder: 'image/*, application/pdf' },
    { id: 'maxSize', type: 'number', label: 'Max size (MB)' },
    { id: 'multiple', type: 'boolean', label: 'Allow multiple files' },
  ],
});
```

Your renderer reads its options from `field.config` (shaped by the `configSchema` above) and owns the stored value shape entirely.

## Master Registry

### 10. structureRegistry

The single entry point for modules that need to register fields, lifecycle hooks, or delete validation. This is the registry you use when your module needs to store data on nodes or react to deletion.

```typescript
import { structureRegistry } from '@/features/structure';

structureRegistry.register({
  moduleId: 'billing',
  displayName: 'Billing',

  // Fields injected onto node types
  fields: [
    {
      id: 'billing.plan_id',
      nodeTypes: ['project', 'client'],
      order: 1000,
      field: {
        type: 'select',
        label: 'Billing Plan',
        options: [
          { label: 'Free', value: 'free' },
          { label: 'Pro', value: 'pro' },
          { label: 'Enterprise', value: 'enterprise' },
        ],
      },
    },
  ],

  // Optional: tabs, widgets, card extensions, actions, etc.
  // These are shortcuts that call the individual registries internally
  tabs: [/* NodeTabRegistration[] */],
  widgets: [/* NodeWidgetRegistration[] */],
  cardExtensions: [/* NodeCardExtension[] */],
  actions: [/* NodeActionRegistration[] */],
  fieldTypes: [/* NodeFieldTypeRegistration[] */],
  columns: [/* NodeColumnRegistration[] */],
  filters: [/* NodeFilterRegistration[] */],
  sidebarSections: [/* NodeSidebarSectionRegistration[] */],

  // Lifecycle hooks (optional, top-level properties)
  // onNodeCreate?: (node) => Promise<void>,
  // onBeforeNodeDelete?: (node) => Promise<DeleteValidation>,
  // onAfterNodeDelete?: (nodeId, nodeType) => Promise<void>,
  // onNodeMove?: (node, oldParentId, newParentId) => Promise<void>,
});
```

<Aside type="note">
You can register tabs, widgets, and other content items either through `structureRegistry.register()` (as arrays in the configuration object) or by calling the individual registries directly. Both approaches work — `structureRegistry` simply forwards to the individual registries. Use whichever style you prefer; direct calls are more explicit, while `structureRegistry` keeps everything in one place.
</Aside>

## Module-Registered Fields

Modules can inject their own fields onto node types without modifying the node type configuration directly. This is the mechanism for cross-module data — a Billing module adds a "Plan" field to projects, a Time Tracking module adds an "Hourly Rate" field to clients.

### How It Works

<Steps>

1. **Register fields via structureRegistry**

    ```typescript
    import { structureRegistry } from '@/features/structure';

    structureRegistry.register({
      moduleId: 'billing',
      displayName: 'Billing',
      fields: [
        {
          id: 'billing.plan_id',
          nodeTypes: ['project', 'client'],
          order: 1000,
          field: {
            type: 'select',
            label: 'Billing Plan',
            options: [
              { label: 'Free', value: 'free' },
              { label: 'Pro', value: 'pro' },
              { label: 'Enterprise', value: 'enterprise' },
            ],
          },
        },
        {
          id: 'billing.account_ref',
          nodeTypes: ['client'],
          order: 1010,
          field: {
            type: 'text',
            label: 'Account Reference',
            placeholder: 'e.g. ACC-001',
          },
        },
      ],
    });
    ```

2. **Fields appear on the Info tab** after admin-defined fields

3. **Admins control visibility** on the Integrations tab of each node type — enable/disable, override the label, and change display order

4. **Values are stored** in the node's `data` JSON alongside admin-defined field values

</Steps>

### NodeFieldRegistration Interface

```typescript
interface NodeFieldRegistration {
  id: string;                           // Namespaced ID (e.g., 'billing.plan_id')
  nodeTypes: string[] | '*';            // Which node types get this field
  order: number;                        // Display order (use 1000+ for module fields)
  field: {
    type: string;                       // Any built-in or registered field type
    label: string;
    description?: string;
    placeholder?: string;
    required?: boolean;
    options?: Array<{ label: string; value: string }>;
    validation?: {
      min?: number;
      max?: number;
      pattern?: string;
      message?: string;
    };
    width?: 'full' | 'half';
  };
}
```

### Conventions

- **Namespace field IDs** with your module ID: `billing.plan_id`, `tasks.default_assignee`, `time.hourly_rate`
- **Use order 1000+** so module fields appear after admin-defined fields (which typically use 10-100)
- **Target specific node types** rather than `'*'` where possible — a billing plan field only makes sense on projects, not folders
- **Values are stored in `node.data`** alongside admin-defined field values. The field ID is the key: `node.data['billing.plan_id']`

<Aside type="tip" title="Admin controls">
Admins see module-registered fields on the **Integrations tab** of each node type (Structure Types > Edit Type > Integrations). They can enable/disable individual fields, override the display label, and adjust the sort order. Your module does not need to do anything special to support this — the registry handles it automatically.
</Aside>

## Action Hooks

Structure fires **27 action hooks** after every mutation. Other modules subscribe to these via the starter template's `actionRegistry` to react to node events — cleaning up data on delete, setting up defaults on create, syncing state on update.

### Registering a Hook

```typescript
// backend/src/features/tasks/action.registration.ts
import { actionRegistry } from '../../config/action.registry.js';

actionRegistry.addActionAsync(
  'structure.node.deleted',       // hook name
  'tasks-cleanup-on-delete',      // unique listener ID
  async (context) => {            // handler
    const { userId, nodeId } = context;
    await prisma.task.updateMany({
      where: { nodeId, userId },
      data: { status: 'archived' },
    });
  },
  20                               // priority (lower = runs first)
);
```

### Node Hooks (15)

| Hook | Context | Fires when |
|------|---------|------------|
| `structure.node.created` | `{ userId, node, nodeType }` | A node is created |
| `structure.node.updated` | `{ userId, node, changes }` | A node is updated |
| `structure.node.deleted` | `{ userId, nodeId, nodeType }` | A node is deleted |
| `structure.node.moved` | `{ userId, node, oldParentId, newParentId }` | A node moves to a different parent |
| `structure.node.archived` | `{ userId, node }` | A node is archived |
| `structure.node.restored` | `{ userId, node }` | A node is restored from archive |
| `structure.node.favorited` | `{ userId, node, isFavorite }` | A favourite is toggled |
| `structure.node.published` | `{ userId, node }` | A node's `isPublished` changes to true |
| `structure.node.unpublished` | `{ userId, node }` | A node's `isPublished` changes to false |
| `structure.node.workflowStateChanged` | `{ userId, node, oldState, newState, oldGroup?, newGroup? }` | A node's workflow state changes |
| `structure.node.duplicated` | `{ userId, sourceNodeId, newNode, includeChildren }` | A node is duplicated |
| `structure.node.reordered` | `{ userId, parentId, nodeIds }` | Nodes are reordered within a parent |

<Aside type="caution" title="Action hooks are the key to module integration">
Without action hooks, modules are islands — they can display UI on nodes but cannot react to what happens to those nodes. Always register a `structure.node.deleted` listener to clean up your module's related data. Always register `structure.node.created` if your module needs to set up defaults for new nodes.
</Aside>

## Filter Hooks

Structure applies **4 filter hooks** at key points in the data pipeline. Filters run in sequence — each receives the output of the previous one, allowing multiple modules to transform the same data.

### Registering a Filter

```typescript
// backend/src/features/billing/filter.registration.ts
import { filterRegistry } from '../../config/filter.registry.js';

filterRegistry.register(
  'structure.node.response',       // hook name
  'billing-enrich-node',           // unique listener ID
  (node, context) => {             // handler — must return transformed data
    return {
      ...node,
      metadata: {
        ...node.metadata,
        hasBilling: true,
      },
    };
  },
  20                                // priority (lower = runs first)
);
```

### Available Filter Hooks

| Hook | Receives | Returns | Purpose |
|------|----------|---------|---------|
| `structure.node.beforeCreate` | Node input data | Transformed input | Transform or validate data before a node is saved |
| `structure.node.response` | Node object | Enriched node | Add computed properties before sending to the client |
| `structure.node.listQuery` | Query parameters | Modified query | Modify list queries before execution (add constraints, scoping) |
| `structure.nodeType.response` | Node type object | Transformed type | Enrich node type data before sending to the client |

<Aside type="tip" title="Filters vs actions">
**Actions** run after a mutation and are fire-and-forget — use them for side effects (cleanup, notifications, sync). **Filters** run during a request and must return a value — use them to transform data flowing through the pipeline.
</Aside>

## Registration Pattern

All content registries follow the same import and side-effect pattern:

```typescript
// frontend/src/features/tasks/registrations/structure-tabs.registration.ts
import { nodeTabRegistry } from '@/features/structure';
import { TasksTab } from '../components/TasksTab';

nodeTabRegistry.register({
  id: 'tasks',
  moduleId: 'tasks',
  label: 'Tasks',
  icon: 'CheckSquare',
  nodeTypes: '*',
  order: 30,
  component: TasksTab,
});
```

Then import the registration file as a side-effect in your feature's `index.ts`:

```typescript
// frontend/src/features/tasks/index.ts

// Side-effect imports (registrations) FIRST
import './registrations/structure-tabs.registration';
import './registrations/structure-widgets.registration';
import './registrations/structure-cards.registration';

// Then public exports
export { taskRoutes } from './routes.config';
```

<Aside type="caution" title="Side-effect imports must come first">
Registration imports are side-effect imports — they run code when imported. Always place them before your public exports in `index.ts`. This ensures registrations execute before any consuming code tries to read the registries.
</Aside>
