The **Structure** module provides the organisational backbone for your SaaS application. You define **node types** that represent the core entities users work with — clients, projects, matters, properties — and Structure gives each one a detail page with tabs, widgets, and actions that other modules can extend.

<Aside type="tip" title="Think of it like this">
Structure is like a filing cabinet system. You decide what goes in the cabinet (Clients? Workspaces? Cases?), how things are organised (folders, tags), and other modules add functionality to each item (billing adds invoices, tasks adds task lists, etc.).
</Aside>

## Quick Install

```bash
npx structure init
```

The CLI installs database tables, generates frontend and backend code, and installs shadcn components. Node types are configured via the admin panel or by importing a JSON configuration. See the [Configuration](/modules/structure/configuration/) page for full details.

## Architecture

### Everything Is a Node

Structure uses a **unified node model**. Every entity in your application — whether it's a client, a project, a folder, or a tag — is stored in the same `structure_node` table. The `nodeType` field determines what kind of entity it is, and the node type configuration controls how it behaves.

### Hierarchical Organisation

Nodes can contain other nodes through parent/child relationships. Structure uses **materialised paths** (e.g., `/grandparentId/parentId/selfId`) for efficient tree queries. This lets you build deep hierarchies without recursive database calls.

### Registry-Based Extensibility

Structure exposes **eleven registries** — eight content registries, two DB-synced configuration registries (`nodeTypeRegistry` and `connectionTypeRegistry`, each with a dedicated admin tab), and `structureRegistry` (module field injection). Other modules use these to extend nodes:

| Registry | Purpose |
|----------|---------|
| `nodeTypeRegistry` | Node type configurations (DB-synced; admin tab) |
| `connectionTypeRegistry` | Typed connection configurations (DB-synced; admin tab) |
| `structureRegistry` | Module field injection, lifecycle hooks, delete validation |
| `nodeTabRegistry` | Tabs on detail pages |
| `nodeWidgetRegistry` | Widgets on the Overview tab |
| `nodeCardExtensionRegistry` | Badges, actions, and content on cards |
| `nodeActionRegistry` | Action buttons on nodes |
| `nodeFieldTypeRegistry` | Custom field types |
| `nodeColumnRegistry` | Table view columns |
| `nodeFilterRegistry` | List page filter controls |
| `nodeSidebarSectionRegistry` | Detail page sidebar sections |

### Backend Hooks

The service fires **26 action hooks** after every mutation (node created, updated, deleted, moved, archived, published, unpublished, workflow state changed, duplicated, reordered, bulk created, connection CRUD, connection type CRUD, template CRUD, node type CRUD/imported, settings updated, metadata updated) and applies **4 filter hooks** at key pipeline points (before create, response enrichment, query modification, node type response). Other modules subscribe to these hooks via the starter template's `actionRegistry` and `filterRegistry`.

### Scope Boundaries

<Aside type="caution" title="What Structure does NOT do">
Structure is the **parking space** — other modules park their cars. These concerns belong in separate modules:

- **Kanban/board/calendar views** → Tasks module
- **Document storage** → Content module
- **Activity feed / audit logs** → Activity module
- **AI-discovered connections** → cog-ingest / cognitive-db
- **Rich text editing** → Content module (Plate)
- **File attachments** beyond media fields → Content module
- **Notifications** → Notifications module
- **Comments/annotations** → Comments module
</Aside>

## Examples

### Legal Practice

```
Node Types:
├── Client (container)     → /clients
│   └── Matter (primary)   → /clients/:clientId
│       ├── Folder         → organise documents
│       └── Tag            → categorise (e.g., "Litigation", "Contract")

Node Detail Page (/clients/abc123):
├── Overview tab      ← Structure module (widgets from other modules)
├── Info tab          ← Structure module (custom fields)
├── Matters tab       ← Structure module (child nodes)
├── Invoices tab      ← Billing module (adds this tab)
├── Documents tab     ← Documents module (adds this tab)
└── Settings tab      ← Structure module (node settings)
```

### Project Management

```
Node Types:
├── Project (primary)      → /projects
│   ├── Folder             → organise work
│   └── Tag                → categorise (e.g., "Active", "On Hold")

Node Detail Page (/projects/xyz789):
├── Overview tab      ← Structure module
├── Tasks tab         ← Tasks module (adds this tab)
├── Time tab          ← Time Tracking module (adds this tab)
├── Files tab         ← Files module (adds this tab)
└── Settings tab      ← Structure module
```

## Import JSON Template

Use this blank template to define your app's structure. Paste it into **Admin > App Structure > Import** or save as a `.json` file.

```json
{
  "version": "1.0",
  "description": "",
  "nodeTypes": [
    {
      "slug": "",
      "name": "",
      "namePlural": "",
      "description": "",
      "icon": "",
      "color": "",
      "rules": {
        "isRoot": true,
        "allowedChildren": []
      },
      "features": {
        "slug": true,
        "description": true,
        "color": true,
        "icon": true,
        "favorites": true,
        "archiving": true,
        "publishing": false,
        "workflowStates": false
      },
      "fields": [
        {
          "id": "",
          "type": "text",
          "label": "",
          "description": "",
          "placeholder": "",
          "required": false,
          "order": 10,
          "width": "full",
          "options": []
        }
      ],
      "workflowConfig": {
        "defaultState": "",
        "groups": [
          { "slug": "", "name": "", "sortOrder": 0 }
        ],
        "states": [
          { "slug": "", "name": "", "color": "", "icon": "", "group": "", "sortOrder": 0 }
        ],
        "transitions": [
          { "from": "", "to": [] }
        ]
      },
      "navigation": {
        "mainNav": { "enabled": true, "order": 10, "showCount": true },
        "sidebarNav": { "style": "list", "collapsible": true, "defaultExpanded": true },
        "pageNav": {
          "style": "tabs",
          "builtInTabs": {
            "overview": true,
            "info": true,
            "children": false,
            "connections": false,
            "settings": true
          },
          "defaultTab": "overview"
        },
        "listPage": { "defaultView": "grid", "showSearch": true }
      },
      "contextHints": "",
      "sortOrder": 10
    }
  ],
  "connectionTypes": [
    {
      "slug": "",
      "name": "",
      "namePlural": "",
      "description": "",
      "icon": "",
      "color": "",
      "fromTypes": [],
      "toTypes": [],
      "bidirectional": true,
      "maxConnections": null,
      "unique": false,
      "fields": [
        {
          "id": "",
          "type": "text",
          "label": "",
          "required": false,
          "options": []
        }
      ]
    }
  ]
}
```

<Aside type="tip" title="Field reference">

**Node type field types:** `text`, `textarea`, `number`, `date`, `datetime`, `select`, `multiselect`, `toggle`, `boolean`, `url`, `email`, `secret`, `node-reference`, `media`, `field-group`

**Field width:** `full` or `half`

**Navigation styles:** `mainNav.enabled` controls sidebar visibility. `pageNav.style` can be `tabs`, `sidebar`, or `none`. `listPage.defaultView` can be `grid`, `list`, or `table`.

**Workflow:** Set `features.workflowStates: true` and populate `workflowConfig`. Omit `workflowConfig` entirely if workflows aren't needed. Groups and transitions are optional.

**Icons:** Use any [Lucide icon](https://lucide.dev/icons/) name (e.g. `Briefcase`, `Building2`, `Scale`, `Tag`).

See the `examples/` directory in the module repo for complete working examples (project management, legal practice, minimal).
</Aside>

## Commands

```bash
# Install / update / remove
npx structure init                              # Install the module
npx structure init --force                      # Overwrite existing files
npx structure init --no-interactive             # Skip prompts, use defaults
npx structure update                            # Update templates + schema
npx structure update --skip-schema              # Update templates only
npx structure update --dry-run                  # Preview changes
npx structure update --force                    # Proceed through conflicts
npx structure uninstall                         # Remove the module
npx structure uninstall --keep-data             # Keep database tables
npx structure uninstall --export ./backup.json  # Dump data before removing

# Lifecycle
npx structure disable                           # Disable without removing
npx structure enable                            # Re-enable a disabled module
npx structure migrate                           # Add version headers to existing files
npx structure info                              # Print name, version, features
```

See the [Configuration page](/modules/structure/configuration/#cli-reference) for the full flag reference.
