Structure's admin page supports full JSON import and export of node type configurations, connection types, and groups. Use this to back up your configuration, migrate between environments, share setups between applications, or seed a fresh install.

---

## Overview

The **Import** and **Export** buttons sit in the admin page header (App Structure > top tab row).

- **Export** downloads a single JSON file containing all node types, connection types, and groups
- **Import** accepts any combination — only the sections present in the JSON are processed
- **Import order:** groups first, then node types, then connection types (so node types can reference groups, and connection types can reference node types)
- **Two import modes:** Merge (add new, update existing) and Replace (delete node types not in the import)

<Aside type="caution" title="Replace mode deletes missing node types">
In **Replace** mode, any existing **node type** that is not present in the import JSON will be deleted. Connection types and groups are always merge-only — they are created or updated, never deleted, regardless of mode. Replace is useful for full node-type sync but destructive if your import is incomplete. Use **Merge** mode for incremental updates.
</Aside>

## Complete Export Format

When you click **Export**, the downloaded JSON follows this structure:

```json
{
  "version": "1.0",
  "exportedAt": "2026-06-24T10:30:00.000Z",
  "nodeTypes": [
    { /* ... NodeType objects ... */ }
  ],
  "connectionTypes": [
    { /* ... ConnectionType objects ... */ }
  ],
  "groups": [
    { /* ... Group objects ... */ }
  ]
}
```

A real export contains `version`, `exportedAt` (an ISO timestamp), `nodeTypes`, `connectionTypes`, and `groups`. The exporter does not write a top-level `description` — `description` is only an optional field you may add to an import file. All three arrays are optional on import. You can import a file with only `nodeTypes`, only `connectionTypes`, only `groups`, or any combination. The `version` field is `"1.0"` for the current format.

### Field Reference

Each node type property explained:

| Property | Type | Description |
|----------|------|-------------|
| `slug` | `string` | **Required.** Unique URL identifier. Cannot change after creation. |
| `name` | `string` | **Required.** Singular display name (e.g., "Project"). |
| `namePlural` | `string` | **Required.** Plural form for navigation and lists (e.g., "Projects"). |
| `description` | `string` | Optional type description. |
| `icon` | `string` | [Lucide icon](https://lucide.dev/icons/) name (e.g., `Briefcase`, `Scale`). |
| `color` | `string` | Hex colour (e.g., `#8b5cf6`). |
| `sortOrder` | `number` | Display order in navigation and admin (default 50). |

**rules** — hierarchy and relationship constraints:

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `isRoot` | `boolean` | `false` | Can exist at root level (no parent). |
| `allowedParents` | `string[]` | `[]` | Type slugs this can nest inside. |
| `allowedChildren` | `string[]` | `[]` | Type slugs this can contain. |
| `maxDepth` | `number` | -- | Maximum nesting depth. |

**features** — toggle capabilities per type:

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `slug` | `boolean` | `false` | Allow custom URL slug. |
| `description` | `boolean` | `false` | Show rich description field. |
| `color` | `boolean` | `false` | Allow custom colour override. |
| `icon` | `boolean` | `false` | Allow custom icon override. |
| `favorites` | `boolean` | `false` | Allow marking as favourite. |
| `archiving` | `boolean` | `false` | Allow archive (soft delete). |
| `publishing` | `boolean` | `false` | Enable draft/published workflow. |
| `workflowStates` | `boolean` | `false` | Enable workflow state machine. |

**fields** — custom data fields. See the [Fields section](#fields-array-format) below.

**workflowConfig** — state machine configuration:

| Property | Type | Description |
|----------|------|-------------|
| `defaultState` | `string` | **Required when workflowStates is enabled.** Slug of the initial state. |
| `groups` | `WorkflowStateGroup[]` | Optional grouping for states (e.g., "Active", "Closed"). |
| `states` | `WorkflowState[]` | **Required.** The states in the workflow. |
| `transitions` | `WorkflowTransition[]` | Optional transition constraints. If omitted, any state change is valid. |

**navigation** — controls where and how the type appears:

| Property | Description |
|----------|-------------|
| `mainNav.enabled` | Show in the sidebar navigation. |
| `mainNav.order` | Position in sidebar (lower = higher). |
| `mainNav.showCount` | Display node count beside the label. |
| `sidebarNav.style` | `'list'`, `'tree'`, or `'none'`. |
| `pageNav.style` | Detail page layout: `'tabs'`, `'sidebar'`, or `'none'`. |
| `pageNav.builtInTabs` | Toggle visibility of built-in tabs. Each can be `true`, `false`, or `{ label?: string }`. |
| `pageNav.defaultTab` | Tab ID to show by default. |
| `listPage.defaultView` | `'grid'`, `'list'`, or `'table'`. |
| `listPage.showSearch` | Show search bar on the list page. |

**contextHints** — free-text hints for AI context assembly. Describe what information is important about this type.

**contextConfig** — controls what the `/nodes/:id/context` endpoint assembles:

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `includeAncestors` | `boolean` | `false` | Include parent chain. |
| `ancestorDepth` | `number` | -- | How many levels up to include. |
| `includeConnections` | `boolean` | `false` | Include connected nodes. |
| `includeChildCounts` | `boolean` | `false` | Include counts of child nodes by type. |

**groupSlug** — assigns this node type to a sidebar navigation group (must match a group's `slug`).

## Connection Type JSON

A complete connection type object with all possible fields.

```json
{
  "slug": "depends-on",
  "name": "Depends On",
  "namePlural": "Dependencies",
  "description": "Links items that have dependency relationships",
  "icon": "GitBranch",
  "color": "#f59e0b",
  "fromTypes": ["project"],
  "toTypes": ["project"],
  "bidirectional": false,
  "maxConnections": null,
  "unique": false,
  "fields": [
    {
      "id": "dependency_type",
      "type": "select",
      "label": "Dependency Type",
      "required": true,
      "options": [
        { "value": "blocks", "label": "Blocks" },
        { "value": "required-by", "label": "Required By" },
        { "value": "related-to", "label": "Related To" }
      ]
    }
  ]
}
```

### Field Reference

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `slug` | `string` | -- | **Required.** Unique identifier (e.g., `related`, `depends-on`). |
| `name` | `string` | -- | **Required.** Singular display name. |
| `namePlural` | `string` | -- | Plural form. |
| `description` | `string` | -- | Optional description. |
| `icon` | `string` | -- | Lucide icon name. |
| `color` | `string` | -- | Hex colour. |
| `fromTypes` | `string[]` | `[]` | Allowed source node types. An empty array (the default) means any type. Do not list `["*"]` — there is no wildcard handling, so a literal `*` matches no real type and would block every connection. |
| `toTypes` | `string[]` | `[]` | Allowed target node types. Same rules as `fromTypes`. |
| `bidirectional` | `boolean` | `true` | If true, A-to-B also means B-to-A. |
| `maxConnections` | `number \| null` | `null` | Maximum connections of this type per node. Null = unlimited. |
| `unique` | `boolean` | `false` | Only one connection of this type between any two nodes. |
| `fields` | `ConnectionTypeField[]` | `[]` | Data fields on connections. A reduced subset of the node field shape (see below). |

<Aside type="tip">
Connection type fields support a reduced subset of the node field shape — `id`, `type`, `label`, `description`, `required`, `defaultValue`, and `options` only. Node-only properties such as `order`, `width`, `placeholder`, and sub-field configs are not part of connection fields. This lets you attach data to relationships, not just to nodes. For example, a "Party" connection between a Matter and a Client can have a "Role" field (Plaintiff, Defendant, Witness).
</Aside>

## Group JSON

Groups organise node types under collapsible sidebar section headers. They are purely presentational — groups do not contain nodes.

```json
{
  "slug": "work",
  "name": "Work",
  "icon": "Briefcase",
  "sortOrder": 10
}
```

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `slug` | `string` | -- | **Required.** Unique identifier. |
| `name` | `string` | -- | **Required.** Display name (shown as sidebar section header). |
| `icon` | `string` | -- | Lucide icon for the group. |
| `sortOrder` | `number` | `50` | Display order. |

To assign a node type to a group, set the `groupSlug` property on the node type to match the group's `slug`.

## Fields Array Format

Both node types and connection types use the same field format. Each field in the `fields` array follows this structure:

```json
{
  "id": "priority",
  "type": "select",
  "label": "Priority",
  "description": "How urgent is this item?",
  "placeholder": "Choose a priority...",
  "required": false,
  "order": 20,
  "width": "half",
  "options": [
    { "value": "low", "label": "Low" },
    { "value": "medium", "label": "Medium" },
    { "value": "high", "label": "High" }
  ]
}
```

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `id` | `string` | -- | **Required.** Unique field identifier within the type. |
| `type` | `string` | -- | **Required.** One of the built-in or registered field types. |
| `label` | `string` | -- | **Required.** Display label. |
| `description` | `string` | -- | Help text below the field. |
| `placeholder` | `string` | -- | Placeholder text for inputs. |
| `required` | `boolean` | `false` | Whether the field is mandatory. |
| `order` | `number` | `50` | Display order (lower = higher). |
| `width` | `'full' \| 'half'` | `'full'` | Layout width. |
| `options` | `Array<{ value, label }>` | `[]` | Choices for `select` and `multiselect` types. |

**Built-in field types:** `text`, `textarea`, `number`, `date`, `datetime`, `time`, `select`, `multiselect`, `chained-select`, `toggle`, `boolean`, `url`, `email`, `phone`, `address`, `name-field`, `secret`, `node-reference`, `media`, `field-group`, `hidden`, `section-break`

<Aside type="note" title="Special field types">
- **boolean** and **toggle** both render as a switch (and as a Yes/No badge in read-only display)
- **field-group** is a repeater — it contains sub-fields defined via a nested `fields` array on the field-group field. Each entry adds a row with all sub-fields.
- **node-reference** supports `nodeReferenceConfig: { multiple: true, allowedTypes: [...] }` for multi-select node references
- **secret** renders as a masked input (dots) — the value is stored as plain text but hidden in the UI
</Aside>

## Fields-Only Import

You can import just a fields array directly into the Fields tab of a node type, without wrapping it in the full export format.

The **Import Fields** button on the Fields tab accepts either format:

**Bare array:**

```json
[
  {
    "id": "budget",
    "type": "number",
    "label": "Budget",
    "order": 10,
    "width": "half"
  },
  {
    "id": "priority",
    "type": "select",
    "label": "Priority",
    "order": 20,
    "width": "half",
    "options": [
      { "value": "low", "label": "Low" },
      { "value": "medium", "label": "Medium" },
      { "value": "high", "label": "High" }
    ]
  }
]
```

**Wrapped object:**

```json
{
  "fields": [
    {
      "id": "budget",
      "type": "number",
      "label": "Budget",
      "order": 10,
      "width": "half"
    }
  ]
}
```

Both formats are accepted. The imported fields are merged with any existing fields on the node type.

## Import Behaviour

### Merge Mode

- **New items** (slug not found) are created
- **Existing items** (slug matches) are updated with the imported values
- **Missing items** (in the database but not in the import) are left untouched

This is the safe default for incremental updates.

### Replace Mode

Replace mode only affects **node types**:

- **New node types** are created
- **Existing node types** are updated
- **Missing node types** (in the database but not in the import) are deleted

Connection types and groups are always merge-only — created or updated, never deleted — even in Replace mode.

<Aside type="caution" title="Deleting node types that have data">
Deleting a node type does not check for existing nodes — there is no foreign-key constraint linking a node to its type. A node type is removed silently, and any existing nodes keep an orphaned `nodeType` slug. Reassign or delete those nodes yourself if you do not want them orphaned.
</Aside>

### Processing Order

Import processes items in this order:

1. **Groups** — so that node types can reference them via `groupSlug`
2. **Node types** — so that connection types can reference them via `fromTypes` / `toTypes`
3. **Connection types**

### Validation

The import is lenient — unknown fields are ignored rather than causing errors. The system provides detailed per-item error reporting if any item fails validation. Node-type import does not check slug format; it only rejects a missing or empty `slug`, `name`, or `namePlural`.

## Starter Templates

The module repository includes example JSON configurations in the `examples/` directory:

| File | Description |
|------|-------------|
| `starter-project-management.json` | Clients, Projects, Folders, and Tags with workflow states, dependencies, and tagging connections |
| `starter-legal-matter.json` | Clients, Matters, Folders, and Tags with legal workflow states, party connections, and precedent tracking |
| `starter-minimal.json` | A single "Space" root type with Folders — the simplest starting point |

Use these as a starting point: import one via the admin panel, then customise to fit your application.

<Aside type="tip" title="Blank template">
