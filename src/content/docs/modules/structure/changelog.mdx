This page documents notable changes to the Structure module. The module uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 ‚Üí 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 ‚Üí 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 ‚Üí 1.0.1): Bug fixes, template fixes

<Aside type="tip">
Check your installed version with `npx structure info`.
</Aside>

## v1.9.8

**Revert v1.9.7 `NodeTypeForm` Tabs rewrite**

v1.9.7 replaced shadcn `<Tabs>` with a hand-rolled state-driven tab bar in `NodeTypeForm` to work around a user-reported rendering issue (six tall columns beside a cramped sidebar). On reflection that was a hack ‚Äî the module shouldn't avoid shadcn primitives just because one consuming app has a broken `@/components/ui/tabs`. The real bug lives in the host app's tabs component or its surrounding layout, and hiding it in a module template makes it harder to find the next time someone customises the form.

**Reverted:** `NodeTypeForm` is back to using `<Tabs>` / `<TabsList>` / `<TabsTrigger>` / `<TabsContent>` from `@/components/ui/tabs`. Root cause investigation is now a Known Issue on the roadmap.

## v1.9.6

**Retry the auto-prime ‚Äî critical for non-admin users**

v1.9.5 restored the auto-prime fetch but with no retry. If that first fetch fails (cold-login race with JWT landing in storage, transient network error, backend startup lag), the registry stays empty. **Admins can click into the Admin area to force a `useNodeTypes()` fire**, which re-syncs the registry. **Non-admin users can't** ‚Äî if their sidebar has no Structure items, they have no URL to navigate to, so they're stuck on whatever landing page the app dropped them on.

**Fix:** the auto-prime now retries on failure with backoff ‚Äî attempt 0 immediately, then +1s, +2s, +3s. Total ~6s window. After that it gives up silently, but by then auth has almost always initialised. Every subsequent retry uses the shared JWT-aware axios client, so it picks up the token as soon as it lands.

```ts
function primeStructureNavigation(attempt = 0): void {
  structureService.listNodeTypes()
    .then((types) => nodeTypeRegistry.syncFromDatabase(types))
    .catch(() => {
      if (attempt < 3) {
        setTimeout(() => primeStructureNavigation(attempt + 1), (attempt + 1) * 1000);
      }
    });
}
```

`<StructureInit />` and `initStructureNavigation()` are still available as explicit opt-in fallbacks, but apps shouldn't need them in the typical auth flow.

## v1.9.5

**Restore auto-priming navigation ‚Äî revert v1.9.2 over-correction**

The v1.9.2 release removed the module-scope `structureService.listNodeTypes()` call, citing a pre-auth 401 on fresh page loads. That turned out to be an over-correction ‚Äî the 401 was cosmetic (silent-caught), but removing the fetch broke nav population for any user who didn't visit a Structure page. v1.9.4's `<StructureInit />` component fixed it *if* apps wired it in, but users reported the sidebar still looked empty ‚Äî the component only helps when consuming apps mount it, and the expectation is "it should just work like it used to".

**Fix:** the auto-prime fetch at the bottom of `NAVIGATION_REGISTRATION` is back. Same behaviour as pre-v1.9.2 ‚Äî fires once on module import, uses the shared JWT-aware axios client, silent-catches errors so logged-out cold loads don't surface noise. The feature's `index.ts` is imported by the authenticated app shell, so the fetch runs post-login in practice.

```ts
structureService.listNodeTypes()
  .then((types) => nodeTypeRegistry.syncFromDatabase(types))
  .catch(() => {
    // Silent fail ‚Äî a later useNodeTypes() will populate the registry
  });
```

**`<StructureInit />` is kept** as an optional helper. Use cases documented in its docstring ‚Äî mainly apps whose auth flow means the feature module loads before the JWT is available. Post-Installation steps no longer list it as required; the sidebar just works on first render.

**Apps upgrading:** `npx structure update` pulls the revert; no app-level wiring needed. If you added `<StructureInit />` in v1.9.4, it still works and is harmless ‚Äî just optional now.

## v1.9.4

**`<StructureInit />` ‚Äî fix cold-load empty sidebar**

Fixes a user-reported bug introduced in v1.9.2: the main sidebar's Structure nav items (node types with `mainNav.enabled`) were missing on first load post-login, and only appeared after the user visited a Structure page (typically the admin page). Non-admin users who never had a reason to open admin would see an empty sidebar indefinitely.

**Root cause:** v1.9.2 removed the module-import-time `structureService.listNodeTypes()` call (it was hitting the API pre-auth and 401'ing on every fresh page load). The intended replacement was the opt-in `initStructureNavigation()` helper, documented as "call from a post-auth init point". That was too easy to overlook.

**Fix:** New `<StructureInit />` component ‚Äî renders nothing, calls `useNodeTypes()` internally. Apps mount it once in their authenticated layout, and the sidebar populates on first render instead of waiting for the user to visit a Structure page.

```tsx
// frontend/src/layouts/AuthenticatedLayout.tsx
import { StructureInit } from '@/features/structure';

export function AuthenticatedLayout({ children }) {
  return (
    <>
      <StructureInit />
      {children}
    </>
  );
}
```

Existing `initStructureNavigation()` imperative helper kept for apps that need to prime the registry outside React (e.g. from a non-component init script).

**Post-install docs updated** to make mounting `<StructureInit />` a numbered required step, with the gotcha called out that it must live below the auth boundary (it calls the API on mount).

## v1.9.2

**Standards alignment follow-up ‚Äî audit findings resolved**

Cleanup pass on findings from the v1.9.1 audit.

**`console.*` removed from generated code:**
- `pages.ts` (NodeConnectionsTab + NodeTypesAdminPage) ‚Äî 5 `console.error` in catch blocks replaced with `toast.error(err.message)`
- `registrations.ts` (NAVIGATION_REGISTRATION) ‚Äî `console.debug` removed along with its containing module-import fetch (see below)

**Module-import API call removed:**
- `NAVIGATION_REGISTRATION` no longer fires `structureService.listNodeTypes()` at module load ‚Äî that call was running pre-auth on every fresh page load and 401ing silently
- Replaced with an optional `initStructureNavigation()` function for apps that want to prime the nav before any page mounts. In most cases no call is needed ‚Äî `useNodeTypes()` on any Structure page syncs the registry and triggers the navigation listener automatically

**TanStack Query migration extended to admin + connection CRUD:**
- New `useNodeTypeMutations()` hook (in `useNodeTypes.ts`): `createNodeType`, `updateNodeType`, `deleteNodeType`, `reorderNodeTypes` ‚Äî all fire `toast.success`/`toast.error`, invalidate `['structure', 'node-types']` on success, and mirror the result back into `nodeTypeRegistry`
- New `useConnectionMutations()` hook (in `useNode.ts`): `createConnection`, `deleteConnection` ‚Äî invalidate `['structure', 'node-connections']` and `['structure', 'nodes']` (for list connection counts)
- `NodeConnectionsTab` and `NodeTypesAdminPage` rewired to consume these hooks; direct `structureService.*` calls and manual `await refetch()` are gone
- Reorder uses a single batch mutation (N parallel updates + one invalidation) instead of firing N toasts and N refetches

**Polish:**
- `useNodeMutations.updateNode` now toasts `'Node updated'` on success (previously silent ‚Äî inconsistent with create/delete/move/etc.)
- `reorderNodes` in backend `SERVICE_TEMPLATE` now logs `log.info({ userId, parentId, count }, 'Nodes reordered')` ‚Äî matches the logging pattern of every other mutation
- `mediaType` string-literal union renamed from `'any'` to `'all'` across frontend type, backend type, and admin field-config UI; user-facing label stays "Any". Removes the read-as-TypeScript-keyword confusion

## v1.9.0

**Action & Filter Hook Firing, Structured Logging, Settings Page, Lifecycle Hooks**

Major standards alignment release. Action and filter hooks that were previously registered but never fired are now fully wired into the backend service.

**Action Hooks Now Fire:**

All 16 action hooks documented in `action.registration.ts` now fire after their corresponding service mutations via `void actionRegistry.doActionAll('hook.name', context)`.

New hooks added:
- `structure.node.reordered` ‚Äî fires after batch reorder
- `structure.template.created` / `.updated` / `.deleted` ‚Äî template lifecycle events
- `structure.settings.updated` ‚Äî user settings changes

**Filter Hooks Now Applied:**

All 4 filter hooks are applied at the correct pipeline points:
- `structure.node.beforeCreate` ‚Äî applied to input before node creation
- `structure.node.response` ‚Äî applied to node data before returning from `getNode`
- `structure.node.listQuery` ‚Äî applied to query parameters before building the Prisma where clause
- `structure.nodeType.response` ‚Äî applied to each node type in `listNodeTypes` and `getNodeType`

New filter registration: `structure.nodeType.response` default pass-through filter added to `filter.registration.ts`.

**Structured Logging:**
- Backend service uses Pino structured logging via `createModuleLogger('structure')`
- All mutations log at info level with userId, nodeId/slug, and operation description
- Blocked operations (e.g., deleting a node type with existing nodes) log at warn level
- Read operations are not logged

**Settings Page:**
- User-facing settings page registered at `/settings/structure` via `settingsPagesRegistry`
- Preferences for default view, show favourites, sidebar state, item counts
- Uses `settings-blocks` components (Row, SectionHeader, Divider, Note, ToggleSwitch, Dropdown)

**Lifecycle Hooks:**
- `onInstall` hook logs success message and note about default type seeding
- `onUpdate` hook logs version transition, warns on major version changes, notes schema updates

**Scaffold Update:**
- Updated from scaffold v1.4.19 to v1.5.0

**New Hooks:**

## Versioning

Version bumps happen via the module's version scripts:

```bash
npm run version:patch  # Bug fix (x.x.X)
npm run version:minor  # New feature (x.X.0)
npm run version:major  # Breaking change (X.0.0)
```

Any change to files in `src/templates/` or `src/module.ts` requires a version bump so that apps know updates are available.
