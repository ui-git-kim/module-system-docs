This page documents notable changes to the Structure module. The module uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, template fixes

<Aside type="tip">
Check your installed version with `npx structure info`.
</Aside>

## v1.18.0

**Type Hierarchy — is-a inheritance for node types**

Node types can now declare parent types, forming a multiple-inheritance *is-a* DAG: a `court-order` is-a `legal-document` and is-a `event`; both are is-a `entity`. Subtypes inherit **fields and features** from their ancestors (nearest-first), and connection allowlists become subsumption-aware. Full guide: [Type Hierarchy](/modules/structure/type-hierarchy/).

- **`superClasses` column on `structure_node_type`** — a `Json @default("[]")` array of parent type slugs. Purely additive; the host app picks it up with a normal `prisma migrate dev` after `npx structure update`. Validated on every write path (create/update/import/propose/approve): parents must be live types, no self-inheritance, no cycles, no redundant parents, max 10 parents, duplicates deduped.
- **Resolved config on `/types` responses** — list and get now carry `superClasses` (raw) and a read-only `resolved = { fields, features, ancestors }`: own fields first, then inherited fields in linearisation order (shadowed by field id, tagged `inheritedFrom`); features merge per-key nearest-wins, and an own key *always* wins — so `cognitiveIngest: { enabled: false }` on a subtype opts out of an inherited enable. Write paths strip `resolved` and any inherited field entries. Only fields and features inherit — rules, workflow, context hints, navigation, templates, permissions and the rest deliberately don't (v1 scope).
- **Subsumption-aware connection allowlists** — `createConnection` `fromTypes`/`toTypes` now admit subtypes of listed types: a "cites" connection type allowing `legal-document` accepts a `court-order` node. Empty list still means any type.
- **`'*'` wildcard fix** — a literal `'*'` entry in `fromTypes`/`toTypes` now means "any type"; previously it was documented but matched nothing, blocking every connection that used it.
- **`includeSubtypes` on node listing** — `listNodes` gains an `includeSubtypes` option (REST query param `?includeSubtypes=true`, service option, frontend client) that expands the type filter to self + all transitive subtypes.
- **Four new service methods** — `getTypeAncestors`, `getTypeDescendants`, `isSubtypeOf`, `expandTypeSlugs` via `serviceRegistry.get('structure')`; see [Integration](/modules/structure/integration/).
- **Admin UI** — the Hierarchy tab gains a **Type Inheritance (is-a)** picker (multiple parents, cycle-guarded); the Fields tab shows inherited fields read-only with their source type. V1 limitation: inherited fields aren't selectable in card/table/filter configuration pickers.
- **Import/export/proposals** — export includes `superClasses`; import validates the final set transactionally (unknown parents, cycles, or orphaned survivors roll the whole import back — note imports use whole-row semantics, so importing a pre-v1.18 export resets `superClasses` to `[]`). `proposeNodeType` accepts `superClasses`, validated at propose time and re-validated at approval.
- **Delete guard** — `deleteNodeType` is blocked while any live type lists the slug in its `superClasses` (the error names the referencing types).
- **Hierarchy-aware embedding selection** — `listNodesNeedingEmbedding`'s default type selection now includes subtypes of `cognitiveIngest`-enabled types without restating the flag. The `publicApi` feature is deliberately *not* hierarchy-aware — a subtype of a publicly-exposed type is not auto-exposed (security decision).

<Aside type="caution">
**Behaviour changes:** (1) a literal `'*'` in `fromTypes`/`toTypes` now means **any type** — previously it matched nothing and blocked every connection using it. (2) Connection allowlists **admit subtypes** once `superClasses` are configured — a type listed in `fromTypes`/`toTypes` now also accepts its subtypes. (3) `/types` responses gained `superClasses` and `resolved` keys — additive, but strict response validators should expect them.
</Aside>

## v1.17.0

**`contextHints` / `contextConfig` now persist**

Both fields have existed on `NodeTypeConfig`, the zod schemas, and the export format since v1.10.0 — but the `structure_node_type` table had no columns for them, so every write path silently dropped them and `getNodeContext` always saw them as unset.

- **New columns on `structure_node_type`** — `contextHints` (`String? @db.Text`) and `contextConfig` (`Json @default("{}")`). Purely additive; the host app picks them up with a normal `prisma migrate dev` after `npx structure update`.
- **All write paths now include them** — `createNodeType`, `updateNodeType`, `importNodeTypes`, and `proposeNodeType` persist both fields, so JSON imports and module-proposed node types (e.g. cog-ingest's working-memory type) can ship context hints.
- **Reads are direct column reads** — the unchecked `(type as Record<string, unknown>)` casts in `listNodeTypes`/`getNodeType` are gone; `GET /nodes/:id/context` now actually honours `contextHints` and the `contextConfig` assembly toggles.
- **Proposals surface them** — `NodeTypeProposal` includes `contextHints`/`contextConfig`, so admins reviewing a proposal can see the proposed context configuration.

## v1.16.0

**Cognitive foundations (cognitive-db v2 RFC Phase 1)**

Additive groundwork for cognitive processing on top of Structure — vector embeddings, discovered-connection provenance, and module-proposed node types. Full reference: [Cognitive Foundations](/modules/structure/cognitive/).

- **Opt-in pgvector embeddings** — a new `enableEmbeddings` install prompt (default off; needs pgvector-capable Postgres, e.g. Neon) adds `embedding vector(n)` / `embeddingText` / `embeddingModel` columns to `structure_node` plus a partial HNSW cosine index. Dimensions (1024/512/1536/3072) are chosen at install.
- **`structureEmbeddings` service** — a new backend service (registered in the `serviceRegistry`, generated only when embeddings are enabled): `setNodeEmbedding`, `clearNodeEmbedding`, `listNodesNeedingEmbedding`, `semanticSearchNodes`, `findSimilarNodes`, and a memoised `ensureInfrastructure()`. In-process service API only — no HTTP routes. `structureService.getCognitiveCapabilities()` probes availability.
- **Install heal** — `onInstall`/`onUpdate` repair the first-install ordering gap (the `vector` extension must exist before Prisma can add the column): they run the setup SQL via `prisma db execute`, roll back and re-apply a failed migration, and re-ensure the index. All idempotent.
- **Connection origin/status** — `createConnection` accepts `origin` (a provenance tag stamped into `data.origin`; `'explicit'` default, well-known values `'discovered'` and `'active'`) and `status` (`'active'` default, or `'pending'`). Pending connections fire `structure.connection.proposed` and go live via `POST /connections/:id/promote`, which re-validates uniqueness/limits and endpoint liveness. New `GET /connections` listing with status/origin/type filters and pagination.
- **Node-type proposal/approval API** — modules propose node types (`POST /types/proposals` — pending, disabled, invisible to every other node-type query); admins list (`GET /types/proposals`), approve (`POST /types/proposals/:slug/approve`, user or system scope), or reject (`POST /types/proposals/:slug/reject`, which deletes the row and frees the slug). Approval takes the type live and fires `structure.nodeType.created`.
- **`features.cognitiveIngest`** — per-node-type feature flag (the "Cognitive Processing" toggle on the Properties tab) marking which types participate in cognitive processing; read by `listNodesNeedingEmbedding` and, in later phases, the cog-ingest module.
- **Six new action hooks** — `structure.connection.proposed` / `.promoted`, `structure.node.embeddingUpdated`, and `structure.nodeType.proposed` / `.proposalApproved` / `.proposalRejected`. `structure.connection.created` and `structure.nodeType.created` now mean "went live" — they fire at creation-as-active *or* at promotion/approval.
- **First module migration** — `1.16.0__cognitive-foundations.sql` ships via the scaffold's per-module migration runner (idempotent, exception-safe `DO` blocks that degrade to notices on missing privileges). Because the module now carries a DB migration, **`npx structure init` and `npx structure update` need a reachable database** via `backend/.env` (`DIRECT_DATABASE_URL`/`DATABASE_URL`).

<Aside type="caution">
**Behaviour changes:** `getNodeConnections` now returns **active connections only** by default — archived/deleted connections were previously included and are now hidden unless you pass a wider `status` filter (`getNodeContext` inherits the new default). To match, `restoreNode` now restores the connections its own delete cascade soft-deleted. Also, **`origin` is now a reserved connection field id** — connection types can no longer define a custom field with id `origin` (the provenance stamp would overwrite it).
</Aside>

## v1.15.1

**Navigation auto-prime no longer retries on rate-limit (429)**

The module-scope navigation auto-prime (`navigation.registration.ts`) retried on *any* error with backoff. That backoff exists for the JWT-not-landed-yet race (a `401`), but retrying on a `429` (rate limited) spends more of the request budget the server just asked us to back off from — amplifying a rate-limit storm. The retry is now scoped to `401`/`403` only; `429`, `5xx`, and network errors give up silently (a later `useNodeTypes()` repopulates the registry).

<Aside type="tip">
The hook queries (`useNodeTypes`, etc.) still follow TanStack Query's default retry. To stop *all* queries app-wide from retrying a `429`, set a retry predicate on your `QueryClient` (e.g. don't retry when `error.response.status === 429`) — that's an app-level concern, not per-module.
</Aside>

## v1.15.0

**Public, API-key-authed REST API — per node type**

A new external REST surface exposes node *instances* of opt-in node types to integrations, authenticated with API keys instead of the in-app session. The internal session-scoped API (`/api/v1/structure/*`) is unchanged.

- **Opt-in per type** — a node type is reachable only when its new **Public REST API** feature toggle is on; unknown/non-exposed types return `404` (no existence leak). External base: `/api/v1/public/structure/:type`.
- **Methods** — `GET` (list + one), `POST`, `PATCH`. There is intentionally **no `DELETE`**. Handlers are a thin layer over the existing `structureService`, so all field/constraint validation and per-account scoping are reused.
- **API keys** — a new **API Keys** admin tab creates keys (the raw key is shown **once**), lists them, and revokes them. Keys carry a scope (`read`, or `read` + `write`); a write with a read-only key returns `403`. Keys are stored only as a `hashSecret()` hash via the security feature, declared as a `hashedSecret`, cascaded on account deletion, and resolved by an API-key middleware (`Authorization: Bearer` / `X-API-Key`).
- New table `structure_api_keys`; new docs page [Public REST API](/modules/structure/public-api/).

<Aside type="note">
The public router registers with `requiresAuth: false` so the starter mounts it without session auth. As with all template changes, the definitive check is `npx structure update` + a build/run in a starter app.
</Aside>

## v1.14.0

**Custom fields on connections — admin-define, collect, display**

Connection types can now carry their own data fields, wired end-to-end through the UI. The data model and API already supported `structure_connection_type.fields` and stored values on the connection node's `data` column — this release makes that capability usable from the app.

- **Admin** — the connection-type form (**Admin → Connection Types**) has a new **Fields** section: add fields with a label, a type (`text`, `textarea`, `number`, `select`, `multiselect`, `toggle`, `date`), an optional comma-separated options list (for `select`/`multiselect`), and a Required toggle.
- **Collect** — the **Add Connection** dialog now chooses the connection type from a `<Select>` of defined types and renders that type's fields as inputs. Required fields gate the submit, defaults are seeded, and the values are stored on the connection's `data` keyed by field id. Arbitrary/legacy types (e.g. `related`) remain creatable when no types are defined.
- **Display** — the connections list shows an info popover on each connection badge listing that connection's field values, resolving option labels for `select`/`multiselect` and `Yes`/`No` for toggles.

<Aside type="note">
These are changes to template strings, so the definitive check is `npx structure update` + a build in a starter app.
</Aside>

## v1.13.3

**Health-audit hardening (atomicity, validation, performance, UX)**

A four-dimension health audit (standards / production / performance / security) found **0 critical and 0 security-moderate** issues — the `userId`-scoping and `requireAdmin` controls are sound — and applied 23 fixes across the templates:

- **Backend:** `deleteNode` is now atomic (`prisma.$transaction`) and also soft-deletes connections pointing at descendants; `userId` scoping is threaded through `calculatePath`/`updateDescendantPaths` and every caller; the `dataFilters` query param is validated against `dataFilterSchema` (a `400` instead of a `500` on bad input) with length/array bounds.
- **Frontend data:** `listNodes` now sends the previously-dropped `pathPrefix`/`dataFilters`/`workflow*` params (the data-filter & workflow-state filtering was silently unreachable from the hooks); connection mutations invalidate `node-context`; `nodeTypeRegistry.update()`/`unregister()` notify `onSync` subscribers so admin edits refresh the nav reactively.
- **Frontend components:** the computed-template sync in `NodeFieldRenderer` moved out of render (no more update-during-render); `NodeSelector` resolves the selected node by id and debounces its search; `NodeCard`/`NodeListItem` are memoised with stable callbacks.
- **Frontend pages:** distinct error/retry states on the list, detail, children, and connections views; `NodeInfoTab` flushes pending edits on unmount (no lost edits); settings saves surface errors; admin `NodeTypeForm` and `NodeSettingsTab` validate required fields client-side.
- **Schema/standards:** composite `@@index([userId, path])`; removed an unused `tooltip` shadcn dependency; fixed a stale `initStructure()` comment; narrowed an `as never` cast.

Two minor performance refinements (a `getTree` column `select` and raw data-filter-query pagination) were **deferred** to avoid type/correctness regressions — see the roadmap.

<Aside type="note">
These are changes to template strings, so the definitive check is `npx structure update` + a build in a starter app.

## v1.13.0

**Security feature integration — account-deletion cascade**

Structure now declares its security surface to the starter template's `securityRegistry` (scaffold v1.11.0+), so a user's Structure data is fully erased when their account is deleted — no orphaned rows.

A new generated `security.registration.ts` (a side-effect import in the feature's `index.ts`, alongside the route and service registrations) registers:

- A **manifest** declaring all six per-user data stores — `structure_nodes`, `structure_node_types`, `structure_node_type_groups`, `structure_templates`, `structure_connection_types`, `structure_settings` — as `owned` (hard-deleted on account erasure). Connections are stored as nodes, so they are covered by `structure_nodes`.
- A **deletion handler per store**, each idempotent (`deleteMany`) and scoped to the subject (`ctx.subject.userId`). For the nullable-`userId` configuration tables, only the user's own rows are removed; system and module defaults (`userId = null`) are preserved.

`ownedModels` is empty because the schema is `userId`-keyed — adopting `tenantId` + Neon RLS remains an opt-in upgrade. No field encryption or hashed secrets are declared: Structure stores only user-authored organisational data, with no credentials or sensitive columns.

**Cleanup:** removed three dead `my-feature` placeholder backend template files (`routes.ts`, `services.ts`, `types.ts`) and the duplicate, unused route/service registration exports left over from an earlier partial scaffold split.

<Aside type="note">
Requires the starter template's security feature (scaffold v1.11.0+). A frontend-only module — or a backend module that stores no per-user data — would instead delete `security.registration.ts` and its `index.ts` import.
</Aside>

## v1.9.6

**Retry the auto-prime — critical for non-admin users**

v1.9.5 restored the auto-prime fetch but with no retry. If that first fetch fails (cold-login race with JWT landing in storage, transient network error, backend startup lag), the registry stays empty. **Admins can click into the Admin area to force a `useNodeTypes()` fire**, which re-syncs the registry. **Non-admin users can't** — if their sidebar has no Structure items, they have no URL to navigate to, so they're stuck on whatever landing page the app dropped them on.

**Fix:** the auto-prime now retries on failure with backoff — attempt 0 immediately, then +1s, +2s, +3s. Total ~6s window. After that it gives up silently, but by then auth has almost always initialised. Every subsequent retry uses the shared JWT-aware axios client, so it picks up the token as soon as it lands.

```ts
function primeStructureNavigation(attempt = 0): void {
  structureService.listNodeTypes()
    .then((types) => nodeTypeRegistry.syncFromDatabase(types))
    .catch(() => {
      if (attempt < 3) {
        setTimeout(() => primeStructureNavigation(attempt + 1), (attempt + 1) * 1000);
      }
    });
}
```

`<StructureInit />` and `initStructureNavigation()` are still available as explicit opt-in fallbacks, but apps shouldn't need them in the typical auth flow.

## v1.9.5

**Restore auto-priming navigation — revert v1.9.2 over-correction**

The v1.9.2 release removed the module-scope `structureService.listNodeTypes()` call, citing a pre-auth 401 on fresh page loads. That turned out to be an over-correction — the 401 was cosmetic (silent-caught), but removing the fetch broke nav population for any user who didn't visit a Structure page. v1.9.4's `<StructureInit />` component fixed it *if* apps wired it in, but users reported the sidebar still looked empty — the component only helps when consuming apps mount it, and the expectation is "it should just work like it used to".

**Fix:** the auto-prime fetch at the bottom of `NAVIGATION_REGISTRATION` is back. Same behaviour as pre-v1.9.2 — fires once on module import, uses the shared JWT-aware axios client, silent-catches errors so logged-out cold loads don't surface noise. The feature's `index.ts` is imported by the authenticated app shell, so the fetch runs post-login in practice.

```ts
structureService.listNodeTypes()
  .then((types) => nodeTypeRegistry.syncFromDatabase(types))
  .catch(() => {
    // Silent fail — a later useNodeTypes() will populate the registry
  });
```

**`<StructureInit />` is kept** as an optional helper. Use cases documented in its docstring — mainly apps whose auth flow means the feature module loads before the JWT is available. Post-Installation steps no longer list it as required; the sidebar just works on first render.

**Apps upgrading:** `npx structure update` pulls the revert; no app-level wiring needed. If you added `<StructureInit />` in v1.9.4, it still works and is harmless — just optional now.

## v1.9.4

**`<StructureInit />` — fix cold-load empty sidebar**

Fixes a user-reported bug introduced in v1.9.2: the main sidebar's Structure nav items (node types with `mainNav.enabled`) were missing on first load post-login, and only appeared after the user visited a Structure page (typically the admin page). Non-admin users who never had a reason to open admin would see an empty sidebar indefinitely.

**Root cause:** v1.9.2 removed the module-import-time `structureService.listNodeTypes()` call (it was hitting the API pre-auth and 401'ing on every fresh page load). The intended replacement was the opt-in `initStructureNavigation()` helper, documented as "call from a post-auth init point". That was too easy to overlook.

**Fix:** New `<StructureInit />` component — renders nothing, calls `useNodeTypes()` internally. Apps mount it once in their authenticated layout, and the sidebar populates on first render instead of waiting for the user to visit a Structure page.

```tsx
// frontend/src/layouts/AuthenticatedLayout.tsx
import { StructureInit } from '@/features/structure';

export function AuthenticatedLayout({ children }) {
  return (
    <>
      <StructureInit />
      {children}
    </>
  );
}
```

Existing `initStructureNavigation()` imperative helper kept for apps that need to prime the registry outside React (e.g. from a non-component init script).

**Post-install docs updated** to make mounting `<StructureInit />` a numbered required step, with the gotcha called out that it must live below the auth boundary (it calls the API on mount).

## v1.9.2

**Standards alignment follow-up — audit findings resolved**

Cleanup pass on findings from the v1.9.1 audit.

**`console.*` removed from generated code:**
- `pages.ts` (NodeConnectionsTab + NodeTypesAdminPage) — 5 `console.error` in catch blocks replaced with `toast.error(err.message)`
- `registrations.ts` (NAVIGATION_REGISTRATION) — `console.debug` removed along with its containing module-import fetch (see below)

**Module-import API call removed:**
- `NAVIGATION_REGISTRATION` no longer fires `structureService.listNodeTypes()` at module load — that call was running pre-auth on every fresh page load and 401ing silently
- Replaced with an optional `initStructureNavigation()` function for apps that want to prime the nav before any page mounts. In most cases no call is needed — `useNodeTypes()` on any Structure page syncs the registry and triggers the navigation listener automatically

**TanStack Query migration extended to admin + connection CRUD:**
- New `useNodeTypeMutations()` hook (in `useNodeTypes.ts`): `createNodeType`, `updateNodeType`, `deleteNodeType`, `reorderNodeTypes` — all fire `toast.success`/`toast.error`, invalidate `['structure', 'node-types']` on success, and mirror the result back into `nodeTypeRegistry`
- New `useConnectionMutations()` hook (in `useNode.ts`): `createConnection`, `deleteConnection` — invalidate `['structure', 'node-connections']` and `['structure', 'nodes']` (for list connection counts)
- `NodeConnectionsTab` and `NodeTypesAdminPage` rewired to consume these hooks; direct `structureService.*` calls and manual `await refetch()` are gone
- Reorder uses a single batch mutation (N parallel updates + one invalidation) instead of firing N toasts and N refetches

**Polish:**
- `useNodeMutations.updateNode` now toasts `'Node updated'` on success (previously silent — inconsistent with create/delete/move/etc.)
- `reorderNodes` in backend `SERVICE_TEMPLATE` now logs `log.info({ userId, parentId, count }, 'Nodes reordered')` — matches the logging pattern of every other mutation
- `mediaType` string-literal union renamed from `'any'` to `'all'` across frontend type, backend type, and admin field-config UI; user-facing label stays "Any". Removes the read-as-TypeScript-keyword confusion

- `data` is now `undefined` (not `null`) while queries are pending — consumers using `if (!data)` keep working; any `data === null` check needs updating

**Action Hooks Now Fire:**

New hooks added:
- `structure.node.reordered` — fires after batch reorder
- `structure.template.created` / `.updated` / `.deleted` — template lifecycle events

**Filter Hooks Now Applied:**

All 4 filter hooks are applied at the correct pipeline points:
- `structure.node.beforeCreate` — applied to input before node creation
- `structure.node.response` — applied to node data before returning from `getNode`
- `structure.node.listQuery` — applied to query parameters before building the Prisma where clause
- `structure.nodeType.response` — applied to each node type in `listNodeTypes` and `getNodeType`

New filter registration: `structure.nodeType.response` default pass-through filter added to `filter.registration.ts`.

## Versioning

Version bumps happen via the module's version scripts:

```bash
npm run version:patch  # Bug fix (x.x.X)
npm run version:minor  # New feature (x.X.0)
npm run version:major  # Breaking change (X.0.0)
```

Any change to files in `src/templates/` or `src/module.ts` requires a version bump so that apps know updates are available.
