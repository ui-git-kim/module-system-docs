Structure ships an **internal** REST API (`/api/v1/structure/*`) that is authenticated with the in-app user session. Separately, it provides a **public REST API** authenticated with **API keys**, so external systems and integrations can read and write node instances without an app session.

The public API is **opt-in per node type** and **off by default** — a node type is only reachable once you enable its *Public REST API* toggle.

<Aside type="note">
The public API exposes node **instances** of a type (e.g. your clients, projects) — not the node-type configuration. It is a thin, authenticated layer over the same service that powers the in-app API, so all field and constraint validation still applies, and all data stays scoped to the key owner's account.
</Aside>

## Base URL

```
/api/v1/public/structure/:type
```

`:type` is a node type's **slug** (e.g. `clients`, `projects`).

## Setup

<Steps>

1. **Expose a node type.** In **Admin → Structure Types**, edit a type and turn on **Public REST API** (under Features). Its instances become reachable at `/api/v1/public/structure/<slug>`.

2. **Create an API key.** Go to the **API Keys** tab, click **New Key**, give it a name, and choose **Read only** or **Read & write**. The full key is shown **once** — copy it immediately; it is stored only as a hash and cannot be retrieved again.

3. **Call the API.** Send the key on every request (see [Authentication](#authentication)).

</Steps>

## Authentication

Send the key as a bearer token or in the `X-API-Key` header:

```bash
curl https://your-app.com/api/v1/public/structure/clients \
  -H "Authorization: Bearer sk_xxxxxxxxxxxxxxxx"

# or
curl https://your-app.com/api/v1/public/structure/clients \
  -H "X-API-Key: sk_xxxxxxxxxxxxxxxx"
```

A missing, invalid, expired, or revoked key returns **401**. A request for a node type that does not exist or is not exposed returns **404** (the API does not reveal which types exist).

## Scopes

Each key carries a scope:

| Scope | Allows |
|-------|--------|
| `read` | `GET` (list + one) |
| `read` + `write` | `GET`, `POST`, `PATCH` |

A write request made with a read-only key returns **403**.

## Endpoints

All endpoints are relative to `/api/v1/public/structure`.

| Method | Path | Scope | Description |
|--------|------|-------|-------------|
| `GET` | `/:type` | read | List instances of the type. Query params: `page`, `limit` (max 100), `search`, `sortBy`, `sortOrder`. Returns active nodes. |
| `GET` | `/:type/:id` | read | Fetch one instance (404 if it isn't of `:type`). |
| `POST` | `/:type` | write | Create an instance. Body: `name` (required), `slug`, `description`, `data`, `parentId`, `workflowState`. |
| `PATCH` | `/:type/:id` | write | Update an instance. Body: any of `name`, `slug`, `description`, `data`, `workflowState`. |

There is intentionally **no `DELETE`** — deletion is only available through the in-app interface.

The `data` object holds the type's custom field values as key/value pairs. As in the app, field-level validation (required fields, types, enums, lengths) is applied in the UI rather than enforced server-side — send values that match the type's field definitions. The payload is size-bounded (100&nbsp;KB).

### Examples

<Tabs>
<TabItem label="List">
```bash
curl "https://your-app.com/api/v1/public/structure/clients?limit=20&search=acme" \
  -H "Authorization: Bearer sk_xxxx"
```
</TabItem>
<TabItem label="Create">
```bash
curl -X POST https://your-app.com/api/v1/public/structure/clients \
  -H "Authorization: Bearer sk_xxxx" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Acme Corp", "data": { "industry": "manufacturing" } }'
```
</TabItem>
<TabItem label="Update">
```bash
curl -X PATCH https://your-app.com/api/v1/public/structure/clients/NODE_ID \
  -H "Authorization: Bearer sk_xxxx" \
  -H "Content-Type: application/json" \
  -d '{ "data": { "industry": "logistics" } }'
```
</TabItem>
</Tabs>

## Security

- **Keys are hashed.** Only a `hashSecret()` hash of each key is stored (via the [security feature](/module-starter/security/)) — never the plaintext. The raw key is returned once, at creation.
- **Account-scoped.** Every request operates only on the key owner's data; create/update set the owner server-side and can never be overridden by the request body.
- **Revoke any time.** Revoke a key from the **API Keys** tab; revoked (and expired) keys are rejected immediately. Keys are also erased when the owning account is deleted.
- **Set an expiry at creation (v1.19.0).** The create dialog offers 30 / 90 / 365 days or *Never* (default **90 days**). Expiry was always enforced server-side; before v1.19.0 the admin UI could not set it, so every key issued in-app was perpetual. The plaintext key is also cleared from the client mutation cache when the dialog closes.
- **Least privilege.** Issue read-only keys for read-only integrations, and expose only the node types an integration needs.
- **System/default types apply to everyone.** Enabling the toggle on a *system-default* node type (one shared across accounts, with no owner) exposes that type's endpoint for every account's keys — though each key still only ever sees its own instances. Prefer enabling it on the specific types you intend to publish.

<Aside type="caution">
**Not yet included (v1):** request rate limiting / quotas, OAuth, and per-field exposure control. The public endpoint bypasses session auth by design, so in production put it behind a gateway or reverse-proxy rate limiter. Treat keys like passwords and rotate them if leaked.
</Aside>
