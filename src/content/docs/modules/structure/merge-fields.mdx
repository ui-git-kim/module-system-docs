Structure supports two advanced field behaviours that work across all [field types](/modules/structure/fields/): **merge templates** for auto-computing values from other fields, and **conditional visibility** for showing or hiding fields based on form state.

---

## Merge Templates

A merge template (the `computedTemplate` setting) auto-computes a field's value from other fields on the same node. When a field has a `computedTemplate`, it becomes **read-only** — the value is resolved automatically and synced back to the node's data.

### Template Syntax

Templates use `{expression}` tokens that are replaced with actual values at render time.

#### Simple field reference

```
{field_id}
```

Resolves to the flat value of another field on the same node.

**Example:**

```json
{
  "id": "display_name",
  "type": "text",
  "label": "Display Name",
  "computedTemplate": "{first_name} {last_name}"
}
```

If `first_name` is "Kim" and `last_name` is "Schulz", the computed value is **Kim Schulz**.

Array values (e.g. from `multiselect`) are joined with commas: `"Litigation, Family Law"`.

#### Repeater filtered lookup

```
{group_id[subfield=value].target}
```

Looks inside a `field-group` repeater, finds the first row where `subfield` matches `value`, and returns the `target` field from that row.

**Example:**

```json
{
  "id": "applicant_name",
  "type": "text",
  "label": "Applicant",
  "computedTemplate": "{key_people[role=Applicant].surname}"
}
```

If `key_people` is a field-group with rows like `[{ role: "Applicant", surname: "Schulz" }, { role: "Respondent", surname: "Jones" }]`, the computed value is **Schulz**.

#### Repeater by index

```
{group_id[0].target}
```

Accesses a specific row in a repeater by its zero-based index.

**Example:**

```json
{
  "id": "primary_contact",
  "type": "text",
  "label": "Primary Contact",
  "computedTemplate": "{contacts[0].email}"
}
```

Returns the `email` field from the first row of the `contacts` field-group.

#### First item of a group

```
{group_id.subfield}
```

Shorthand for accessing the first item of a field-group. Equivalent to `{group_id[0].subfield}`.

**Example:**

```json
{
  "id": "lead_role",
  "type": "text",
  "label": "Lead Role",
  "computedTemplate": "{key_people.role}"
}
```

#### Node reference dereference

```
{field_id->node_field}
```

Dereferences a `node-reference` field — fetches the referenced node and returns one of its data fields (or top-level properties like `name`).

**Example:**

```json
{
  "id": "client_name",
  "type": "text",
  "label": "Client Name",
  "computedTemplate": "{assigned_client->name}"
}
```

If `assigned_client` points to a node with `name: "Acme Corp"`, the computed value is **Acme Corp**.

The `->` operator also works with data fields on the referenced node:

```json
{
  "id": "client_email",
  "type": "text",
  "label": "Client Email",
  "computedTemplate": "{assigned_client->contact_email}"
}
```

#### Repeater + node dereference

```
{group_id[subfield=value].ref_field->node_field}
```

Combines a repeater filtered lookup with node dereference — find a row in a field-group, read a node-reference field from that row, fetch the referenced node, and return a field from it.

**Example:**

```json
{
  "id": "matter_title",
  "type": "text",
  "label": "Matter Title",
  "computedTemplate": "{key_people[role=Applicant].person->surname} & {key_people[role=Respondent].person->surname}"
}
```

If the Applicant's person node has `surname: "Schulz"` and the Respondent's person node has `surname: "Jones"`, the computed value is **Schulz & Jones**.

Index-based dereference also works:

```
{group_id[0].ref_field->node_field}
```

### How It Works

1. When the `NodeFieldRenderer` encounters a field with `computedTemplate`, it invokes `resolveTemplateAsync`
2. The resolver parses all `{expression}` tokens and resolves them against the node's current data
3. For `->` dereferences, the resolver fetches the referenced node via the API, caching results to avoid redundant requests
4. The resolved value is set as the field's display value and automatically synced back to the node's data object
5. The field renders as a read-only `<Input>` with a muted background

<Aside type="tip">
Merge templates resolve asynchronously because node-reference dereferences require API calls. The resolver uses a per-render cache, so multiple templates referencing the same node only trigger one fetch.
</Aside>

### Admin UI

In the admin panel's **Fields** tab, every field shows a **Merge Template** text input. Type the template expression directly, or use the **Insert field...** picker beside it to append `{field_id}` tokens for fields on the same node type.

When a field has a computed template, it appears as read-only with a muted background in the create/edit form. Users cannot modify the value manually.

### Full Example

A legal matter with a computed title drawn from repeater data and node references:

```json
{
  "fields": [
    {
      "id": "key_people",
      "type": "field-group",
      "label": "Key People",
      "width": "full",
      "fields": [
        {
          "id": "role",
          "type": "select",
          "label": "Role",
          "options": [
            { "value": "Applicant", "label": "Applicant" },
            { "value": "Respondent", "label": "Respondent" }
          ]
        },
        {
          "id": "person",
          "type": "node-reference",
          "label": "Person",
          "nodeReferenceConfig": { "allowedTypes": ["person"] }
        }
      ]
    },
    {
      "id": "matter_number",
      "type": "text",
      "label": "Matter Number"
    },
    {
      "id": "matter_name",
      "type": "text",
      "label": "Matter Name",
      "width": "full",
      "computedTemplate": "{key_people[role=Applicant].person->surname} & {key_people[role=Respondent].person->surname} — {matter_number}"
    }
  ]
}
```

When both people are assigned and the matter number is "FAM-2025-042", the computed matter name becomes:

**Schulz & Jones — FAM-2025-042**

### Built-in Field Merge Templates

Beyond custom fields, the three built-in fields — **name**, **description**, and **slug** — can each carry their own merge template and an editing lock, configured through the `builtInFieldConfig` setting on the node type:

```json
{
  "builtInFieldConfig": {
    "name": {
      "computedTemplate": "{first_name} {last_name}",
      "editableBy": "system"
    },
    "description": {
      "computedTemplate": "{summary}"
    },
    "slug": {
      "computedTemplate": "{file_number}",
      "editableBy": "admin"
    }
  }
}
```

Each entry accepts:

- **`computedTemplate`** — a merge template using the same `{expression}` syntax as custom fields.
- **`editableBy`** — an editing lock with three levels: `anyone` (the default), `admin` (admin users only), and `system` (system/AI only). Omit the property to leave the field editable by anyone.

In the admin panel, the **Built-in Fields** section provides a **Merge Template** input (with the same **Insert field...** picker) and an **Editable By** dropdown for each of name, description, and slug.

<Aside type="caution">
Built-in field merge templates are persisted and edited in the admin UI, but the runtime does not currently resolve them the way it resolves custom-field `computedTemplate` values. Treat `builtInFieldConfig` templates as configuration for the `editableBy` lock; for an auto-computed value that renders on the form, use a custom field with `computedTemplate` instead.
</Aside>

## Conditional Visibility

The `visibleWhen` setting on any field controls whether the field is shown or hidden based on the current values of other fields. Hidden fields do not render at all — they are removed from the form and do not take up space.

### Single Condition

The simplest form specifies a single field and one condition. This is the original format and remains supported.

#### Equals

Show the field only when another field has a specific value:

```json
{
  "id": "school_name",
  "type": "text",
  "label": "School Name",
  "visibleWhen": {
    "field": "person_type",
    "equals": "Child"
  }
}
```

The `school_name` field only appears when `person_type` is set to "Child".

#### Not equals

Show the field when another field does NOT have a specific value:

```json
{
  "id": "professional_notes",
  "type": "textarea",
  "label": "Professional Notes",
  "visibleWhen": {
    "field": "role",
    "notEquals": "Other"
  }
}
```

#### Is not empty

Show the field when another field has any value:

```json
{
  "id": "follow_up_date",
  "type": "date",
  "label": "Follow-up Date",
  "visibleWhen": {
    "field": "notes",
    "isNotEmpty": true
  }
}
```

#### Is empty

Show the field when another field is blank:

```json
{
  "id": "reason_missing",
  "type": "textarea",
  "label": "Why is this field empty?",
  "visibleWhen": {
    "field": "reference_number",
    "isEmpty": true
  }
}
```

#### In (one of)

Show the field when another field's value is in a specific list:

```json
{
  "id": "court_name",
  "type": "text",
  "label": "Court Name",
  "visibleWhen": {
    "field": "matter_type",
    "in": ["litigation", "family", "criminal"]
  }
}
```

#### Contains

Show when a text field contains a substring:

```json
{
  "id": "international_notes",
  "type": "textarea",
  "label": "International Notes",
  "visibleWhen": {
    "field": "jurisdiction",
    "contains": "International"
  }
}
```

#### Greater than / less than

Show when a numeric field exceeds or falls below a threshold:

```json
{
  "id": "discount_reason",
  "type": "textarea",
  "label": "Discount Reason",
  "visibleWhen": {
    "field": "discount_percentage",
    "greaterThan": "0"
  }
}
```

<Aside type="note">
Comparison values are strings even for numeric comparisons — they are coerced to numbers at evaluation time.
</Aside>

### Multi-Condition Rules

For more complex visibility logic, use the `rules` array with a `logic` operator. This supports AND (all conditions must be true) and OR (any condition must be true).

#### AND — all conditions must match

```json
{
  "id": "guardian_name",
  "type": "text",
  "label": "Guardian Name",
  "required": true,
  "visibleWhen": {
    "logic": "and",
    "rules": [
      { "field": "person_type", "operator": "equals", "value": "Child" },
      { "field": "date_of_birth", "operator": "isNotEmpty" }
    ]
  }
}
```

The `guardian_name` field only appears when the person is a Child **and** the date of birth has been filled in.

#### OR — any condition must match

```json
{
  "id": "legal_representative",
  "type": "node-reference",
  "label": "Legal Representative",
  "visibleWhen": {
    "logic": "or",
    "rules": [
      { "field": "person_type", "operator": "equals", "value": "Child" },
      { "field": "requires_representative", "operator": "equals", "value": "true" }
    ]
  }
}
```

### All Operators

The following operators are available in the multi-rule format. All except `notIn` are also available in the single-condition shorthand — `notIn` is **only** supported inside the `rules`/`operator` format.

| Operator | Description | Value required? |
|----------|-------------|:---------------:|
| `equals` | Field value matches exactly | Yes |
| `notEquals` | Field value does not match | Yes |
| `in` | Field value is one of a list | Yes (array) |
| `notIn` | Field value is NOT in a list (multi-rule only) | Yes (array) |
| `isEmpty` | Field has no value (blank, null, undefined) | No |
| `isNotEmpty` | Field has a value | No |
| `contains` | Field value contains the substring | Yes |
| `greaterThan` | Numeric comparison — field value is greater | Yes |
| `lessThan` | Numeric comparison — field value is less | Yes |

### Admin UI

In the admin panel's **Fields** tab, each field has a **Visible When** editor. The editor provides:

- A **field ID** text input to specify which field to watch
- An **operator** dropdown offering only four options: **Equals**, **Not equals**, **Is one of**, and **Has value**
- A **value** input for the comparison value

Multi-rule AND/OR conditions are not editable in the admin UI — they must be authored via JSON import/export (or the API). The remaining operators (`notIn`, `isEmpty`, `contains`, `greaterThan`, `lessThan`) are also only available through JSON or the API, even though the runtime evaluator honours them.

### Real-World Example

A person intake form where different fields appear based on the person type and role:

```json
{

    {
      "id": "person_type",
      "type": "select",
      "label": "Person Type",
      "required": true,
      "options": [

        { "value": "Organisation", "label": "Organisation" }
      ],
      "order": 10
    },
    {
      "id": "date_of_birth",
      "type": "date",
      "label": "Date of Birth",
      "order": 20
    },
    {
      "id": "occupation",
      "type": "text",
      "label": "Occupation",
      "order": 30,
      "visibleWhen": {
        "field": "person_type",
        "equals": "Adult"
      }
    },
    {
      "id": "employer",
      "type": "text",
      "label": "Employer",
      "order": 40,
      "visibleWhen": {
        "field": "person_type",
        "equals": "Adult"
      }
    },
    {
      "id": "school_name",
      "type": "text",
      "label": "School Name",
      "order": 30,
      "visibleWhen": {
        "field": "person_type",
        "equals": "Child"
      }
    },
    {
      "id": "year_level",
      "type": "number",
      "label": "Year Level",
      "order": 40,
      "visibleWhen": {
        "field": "person_type",
        "equals": "Child"
      }
    },
    {
      "id": "guardian_name",
      "type": "text",
      "label": "Guardian Name",
      "required": true,
      "order": 50,
      "visibleWhen": {
        "logic": "and",
        "rules": [
          { "field": "person_type", "operator": "equals", "value": "Child" },
          { "field": "date_of_birth", "operator": "isNotEmpty" }
        ]
      }
    },
    {
      "id": "abn",
      "type": "text",
      "label": "ABN",
      "order": 30,
      "visibleWhen": {
        "field": "person_type",
        "equals": "Organisation"
      }
    },
    {
      "id": "registered_name",
      "type": "text",
      "label": "Registered Name",
      "order": 40,

        "field": "person_type",

      }

}
```

In this configuration:
- **Adult** shows occupation and employer fields
- **Child** shows school name and year level fields, plus guardian name (only after date of birth is filled in)
- **Organisation** shows ABN and registered name fields
- All three types see the date of birth field

The form dynamically adjusts as the user changes the person type dropdown — fields slide in and out without a page reload.

## Combining Merge Templates and Visibility

A single field can have both `computedTemplate` and `visibleWhen`. The merge template is always resolved — the resolver hook runs regardless of visibility, and the computed value is synced back to the node's data. The visibility check only governs whether the read-only input is rendered: when the field is hidden it is not displayed, but the value has already been computed and written back.

```json
{
  "id": "child_summary",
  "type": "text",
  "label": "Child Summary",
  "computedTemplate": "{school_name} — Year {year_level}",
  "visibleWhen": {
    "field": "person_type",
    "equals": "Child"
  }
}
```

This field only appears for children, and when visible, it auto-computes to something like **Bondi Public School — Year 4**.
