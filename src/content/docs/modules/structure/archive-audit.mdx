A running log of **archive-readiness audits**. Before a working session on the Structure module is archived, it is audited for anything that would be lost — unfinished work, loose ends, decisions, and ideas — and the findings are recorded here so a fresh session can pick them straight up.

<Aside type="note" title="Why this page exists">
Session transcripts get archived and are then awkward to search. This page is the durable, reviewable home for "what was in that session" — decisions and their reasoning, ideas raised in passing, work left mid-flight, and anything the code and the rest of these docs don't already capture. Newest entry first. Nothing here is deleted; when an item is actually done, it's ticked and left in place for provenance.
</Aside>

## How to use this page

At the end of a session you want to archive, run the archive-readiness prompt (below). It audits the session and **appends a new entry at the top** of the log, using the entry template. Then it builds and pushes the docs. Review the entry on the live site; archive the session; start the next one by reading the open items here.

### The prompt

> **Archive-readiness check.** Audit this session end-to-end for anything that would be lost if I archive it: (1) unfinished work — started-not-finished, committed-not-pushed, coded-not-validated, or left WIP (check `git status` and recent commits in every repo touched, including the `docs/` submodule); (2) loose ends — TODOs, "later" items, deferred fixes, promised follow-ups, checks I said I'd run; (3) decisions & ideas — any decision, trade-off, rejected approach and *why*, or idea raised in conversation not already written in code or docs; (4) undocumented reality — anything we changed or learned the docs don't reflect. Then **append a new entry to the top of the Archive Audit log** (`docs/src/content/docs/modules/structure/archive-audit.mdx`) using the entry template already on that page. Build the docs, push the docs submodule **and** the submodule pointer. Verify before ticking anything "done" (if a version was bumped, confirm `dist` was rebuilt after the bump). Finish by telling me plainly whether it's safe to archive.

### Entry template

```mdx
## YYYY-MM-DD — <session focus>

**Verdict:** ✅ Safe to archive  ·  ⚠️ Needs input first
**Shipped this session:** <module version(s), key commits, what went live>

**Unfinished / WIP:** <or "none">
**Loose ends & follow-ups:** <deferred fixes, promised checks, TODOs — or "none">
**Decisions & ideas:** <decisions + why, rejected approaches, ideas raised — or "none">
**Recorded where:** <docs pages / memory files updated, so it's findable>
**Open items for next session:** <the pick-up list — or "none">
```

## 2026-07-23 — Meta audit → roadmap verification → module-owned field types

**Verdict:** ✅ Safe to archive
**Shipped this session:** Structure **v1.19.0 → v1.21.1** (all pushed). A four-dimension meta audit (171 verified findings) fixed across batches A–H; 74 latent host type-errors cleared; roadmap verified against the code and found wrong in *both* directions; then Table View, server-side field filtering, three field renderers, five latent bugs, module-owned field types, and removal of the built-in `media` type. Host-validated in veradai (starter v1.64.1) at every step — 0 structure `tsc` errors, backend and frontend.

**Unfinished / WIP:** None. All working trees clean (module + `docs/` submodule), everything pushed, veradai synced to v1.21.1 with the correct version label.

**Loose ends & follow-ups:**
- veradai was left **on** v1.21.1 (upgraded, not reverted) at the user's request — a change from earlier in the session where it was reverted to pristine after each validation.
- `dataSortField` / `dataSortType` are accepted and validated by the API but remain a **no-op in the service** (server-side JSON sort). Table View sorts client-side over the current page only.
- `NodeListPage` sends no `page` / `limit`, so the backend's default limit of 50 silently truncates — pagination controls are still needed.
- `renderCompact` (NodeColumnRegistration) is declared but unread.

**Decisions & ideas:**
- **Field types are module-owned; `media` is not built in.** The user's insight — "why not have modules register field types entirely?" — was correct: `nodeFieldTypeRegistry` already had the full contract and the renderer already delegated to it, so the capability was ~80% built. `media` was removed rather than kept as a disabled/URL-fallback built-in. Shipped as a **minor with a loud breaking note, not a major**, because nothing is in production and the deprecation cycle was explicitly waived.
- **The short-lived `nodeMediaUploadRegistry` (v1.20.0 only) was removed** in v1.21.0 as redundant — a full field-type registration covers upload UI + config + display in one mechanism instead of two.
- **Unknown-field safety was a prerequisite, not a nicety.** Once field types are module-owned, uninstalling a module orphans fields; the old code bound unknown types to a text input that overwrote arrays/objects with a string. Fixed to render read-only with the value preserved before making types module-owned.
- **`getTree` bounded, not `select`-narrowed.** Added a `take` cap (depth-ascending, so truncation can't orphan children); the heavier `select` refinement stays deferred because it needs a lighter `NodeTreeItem.node` type than the public `StructureNode`.

**Recorded where:** Roadmap (`Known Issues`, `Planned Improvements`, `Next Priority`, `Completed`) rewritten for accuracy both directions; full changelog entries v1.19.0 / v1.19.1 / v1.19.2 / v1.20.0 / v1.21.0 / v1.21.1; behaviour corrections across `fields`, `configuration`, `usage`, `integration`, `registries-guide`, `public-api`, `merge-fields`, `index`. Memory: `meta-audit-2026-07-20`, `roadmap-verification-2026-07-23` (+ their findings artefacts).

**Open items for next session:**
- **`builtInFieldConfig.{name,description,slug}` editableBy is unwired** — declared, admin-configurable, persisted, but nothing reads it. Flagged **Next Priority** on the roadmap. The Name/Description/Slug inputs render unconditionally and the backend update allowlist sets all three with no permission check.
- **`inputMask`** — the one field property still genuinely inert (no format-on-type). Build it or drop the property.
- **Bulk Operations UI** — no selection state, no bulk archive/delete/move **endpoints** (only bulk *create*); `useBulkActions` is dead code.
- **Advanced Filtering UI** — text/date/number range filters, saved presets, URL-param filter state; plus the pagination controls above.
- **Public Views** — `publishedSlug` generation, an unauthenticated `/p/:slug` route + read-only page, scheduled publishing. ("View Public" button was removed until this lands.)
- **Tree View** — `StructureTree` isn't rendered anywhere internally (no `tree` view option, `sidebarNav.style === 'tree'` unimplemented).
- **Connection-type toast labels** — node toasts now name the type; `useConnectionMutations` still says "Connection".
- **Gotcha to remember:** `npm run version:*` does **not** rebuild `dist` — always `npm run build` after a version bump, or the linked CLI ships new content under the old version label (hit three times this session).
