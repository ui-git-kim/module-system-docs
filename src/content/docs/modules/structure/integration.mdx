Structure is the organisational backbone of every app built on the starter template. It defines the "things" users work with — Clients, Projects, Properties, Matters — and provides a universal system for organising, displaying, and navigating them. **Almost every module you build will integrate with Structure.**

This page is the definitive guide for module developers who need to extend, reference, or react to Structure nodes.

---

## How Modules Use Structure

There are three levels of integration, and most modules use all three:

### 1. Attach UI to Nodes (Frontend Registries)

Your module adds tabs, widgets, actions, columns, filters, sidebar sections, and fields to Structure's node pages. For example, a Tasks module adds a "Tasks" tab to every project's detail page, a task count badge on project cards, and a "Task Status" filter on list pages. This is done through Structure's **8 content registries** plus `structureRegistry` for field injection — you register components and field definitions, and Structure renders them in the right place.

<Aside type="note" title="10 registries total">
Structure exposes **10 module-specific registries**. 8 are content registries (`nodeTabRegistry`, `nodeWidgetRegistry`, `nodeCardExtensionRegistry`, `nodeActionRegistry`, `nodeColumnRegistry`, `nodeFilterRegistry`, `nodeSidebarSectionRegistry`, `nodeFieldTypeRegistry`). `nodeTypeRegistry` is a DB-synced configuration registry for node types (with a dedicated admin tab). The 10th is `structureRegistry`, which handles module field injection, lifecycle hooks, and delete validation. Connection types are not a code-based registry — they're managed through `structureService` methods and a dedicated admin tab (see below).
</Aside>

### 2. React to Node Events (Backend Action Hooks)

When nodes are created, updated, deleted, moved, archived, published, duplicated, or when workflow states change, Structure fires **33 action hooks**. Your module listens to these events and reacts: clean up tasks when a project is deleted, generate an invoice when a client is archived, send a notification when a node is moved. This is the primary mechanism for **cross-module coordination**.

### 3. Transform Node Data (Backend Filter Hooks)

Your module can transform Structure data as it flows through the system — enrich node responses with billing status, modify list queries to exclude certain types, or validate node data before creation. This uses **4 filter hooks** that run in a pipeline.

<Aside type="caution" title="Action hooks are the key to module integration">
Without action hooks, modules are islands — they can display UI on nodes but can't react to what happens to those nodes. With action hooks, your module stays in sync: when a node is deleted, your module's related data is cleaned up. When a node is created, your module can set up defaults. This is what makes the module system truly composable.
</Aside>

## Declaring the Dependency

If your module depends on Structure, declare it with `depends` in your `module.ts` (scaffold v1.16.0+; the older `requiredModules` remains honoured):

```typescript
// src/module.ts
export const moduleDefinition: ModuleDefinition = {
  id: 'tasks',
  name: 'Tasks',
  version: '1.0.0',
  depends: { structure: '^1.0.0' },
  // ...
};
```

This ensures Structure is installed first and lets the scaffold validate the dependency at install time.

<Aside type="caution" title="Don't confuse the two `dependencies` fields">
The top-level `dependencies: string[]` field doesn't exist on `ModuleDefinition`. Use `depends` (module id → semver range; the deprecated `requiredModules` still works) for module-to-module dependencies. The `frontend.dependencies` and `backend.dependencies` arrays are for npm packages only.
</Aside>

## Frontend Integration

### Adding Tabs to Node Detail Pages

This is the most common integration. Your module adds a tab that appears on specific (or all) node types.

<Steps>

1. **Create your tab component**

    ```typescript
    // frontend/src/features/tasks/components/TasksTab.tsx
    import { useParams } from 'react-router-dom';

    interface TasksTabProps {
      node: { id: string; nodeType: string };
    }

    export function TasksTab({ node }: TasksTabProps) {
      // Your tab content - fetch tasks for this node, display them, etc.
      return (
        <div>
          <h3>Tasks for {node.id}</h3>
          {/* Your task list, create button, etc. */}
        </div>
      );
    }
    ```

2. **Register the tab**

    ```typescript
    // frontend/src/features/tasks/registrations/structure-tabs.registration.ts
    import { nodeTabRegistry } from '@/features/structure';
    import { TasksTab } from '../components/TasksTab';

    nodeTabRegistry.register({
      id: 'tasks',
      moduleId: 'tasks',
      label: 'Tasks',
      icon: 'CheckSquare',
      nodeTypes: '*',           // Show on all node types, or ['project', 'client']
      order: 30,                // Position in tab bar (10-29 = primary, 30-49 = main)
      component: TasksTab,
      // Optional: dynamic visibility
      // isVisible: (node, nodeTypeConfig) => nodeTypeConfig.features?.tasks !== false,
    });
    ```

3. **Import the registration in your feature index**

    ```typescript
    // frontend/src/features/tasks/index.ts

    // Side-effect imports (registrations) FIRST
    import './registrations/structure-tabs.registration';

    // Then public exports
    export { taskRoutes } from './routes.config';
    ```

</Steps>

<Aside type="note" title="Tab ordering">
Structure's built-in tabs use orders 10-25. Place your tabs at 30+ to appear after the defaults. Use the [order value ranges](/getting-started/llm-rules/) from the LLM rules for consistency.
</Aside>

### Adding Widgets to the Overview Tab

Widgets appear on a node's Overview tab in a configurable layout.

```typescript
// frontend/src/features/activity/registrations/structure-widgets.registration.ts
import { nodeWidgetRegistry } from '@/features/structure';
import { RecentActivityWidget } from '../components/RecentActivityWidget';

nodeWidgetRegistry.register({
  id: 'recent-activity',
  moduleId: 'activity',
  title: 'Recent Activity',
  location: 'node-overview',
  nodeTypes: '*',
  order: 10,
  component: RecentActivityWidget,
});
```

Your widget component receives the same `{ node }` prop as tabs.

### Adding Badges or Actions to Node Cards

Extend how nodes appear in list/grid views.

```typescript
// frontend/src/features/tasks/registrations/structure-cards.registration.ts
import { nodeCardExtensionRegistry } from '@/features/structure';
import { TaskCountBadge } from '../components/TaskCountBadge';

nodeCardExtensionRegistry.register({
  id: 'task-count',
  moduleId: 'tasks',
  position: 'badge',       // 'badge' | 'action' | 'header' | 'content' | 'footer'
  nodeTypes: '*',
  order: 10,
  render: (node) => <TaskCountBadge node={node} />,
});
```

### Adding Action Buttons to Nodes

Add buttons to node detail headers, context menus, or bulk action bars.

```typescript
// frontend/src/features/invoicing/registrations/structure-actions.registration.ts
import { nodeActionRegistry } from '@/features/structure';

nodeActionRegistry.register({
  id: 'generate-invoice',
  moduleId: 'invoicing',
  label: 'Generate Invoice',
  icon: 'Receipt',
  nodeTypes: ['client', 'project'],
  locations: ['detail-header'],
  variant: 'default',
  order: 30,
  handler: async (node) => {
    await invoicingService.generateForNode(node.id);
  },
  confirm: {
    title: 'Generate Invoice?',
    description: 'This will create a new invoice for all unbilled items.',
  },
});
```

### Adding Filter Controls to List Pages

Register filter UI controls for list pages. Filters can operate client-side or server-side.

```typescript
// frontend/src/features/tasks/registrations/structure-filters.registration.ts
import { nodeFilterRegistry } from '@/features/structure';
import { TaskStatusFilter } from '../components/TaskStatusFilter';

nodeFilterRegistry.register({
  id: 'task-status',
  moduleId: 'tasks',
  label: 'Task Status',
  icon: 'Filter',
  nodeTypes: ['project'],
  mode: 'client',
  order: 30,
  component: TaskStatusFilter,
  filterFn: (node, filterValue) => {
    if (!filterValue) return true;
    return node.metadata?.taskStatus === filterValue;
  },
});
```

### Registering Custom Field Types

Extend the field editor system with custom field types. Structure includes 22 built-in types; your module can add more.

```typescript
// frontend/src/features/editor/registrations/structure-fields.registration.ts
import { nodeFieldTypeRegistry } from '@/features/structure';
import { RichTextEditor, RichTextDisplay } from '../components/RichTextEditor';

nodeFieldTypeRegistry.register({
  id: 'rich-text',
  moduleId: 'editor',
  label: 'Rich Text',
  category: 'text',
  icon: 'Type',
  nodeTypes: '*',
  order: 50,
  component: RichTextEditor,
  displayComponent: RichTextDisplay,
});
```

Once registered, the `NodeFieldRenderer` automatically picks up your custom field type. Admins can assign it to fields in the node type configuration.

### Using NodeSelector to Reference Nodes

If your module needs users to pick a Structure node (e.g., linking a task to a project), use the `NodeSelector` component:

```typescript
import { NodeSelector } from '@/features/structure';

function TaskForm() {
  const [selectedNodeId, setSelectedNodeId] = useState<string | null>(null);

  return (
    <NodeSelector
      value={selectedNodeId}
      onChange={setSelectedNodeId}
      allowedTypes={['project', 'client']}  // Optional: restrict to certain types
      placeholder="Select a project..."
    />
  );
}
```

`NodeSelector` provides a searchable dialog with type filtering. It's the recommended way for any module to reference Structure nodes.

### Registering Fields on Node Types

Modules can inject their own fields onto node types without modifying the node type configuration directly. This is useful when your module needs to store data on nodes (e.g., a Billing module adding a "plan" field to all projects).

```typescript
// frontend/src/features/billing/registrations/structure.registration.ts
import { structureRegistry } from '@/features/structure';

structureRegistry.register({
  moduleId: 'billing',
  displayName: 'Billing',
  fields: [
    {
      id: 'billing.plan_id',           // namespace with module ID
      nodeTypes: ['project', 'client'], // or '*' for all types
      order: 1000,                      // after admin-defined fields
      field: {
        type: 'select',
        label: 'Billing Plan',
        options: [
          { label: 'Free', value: 'free' },
          { label: 'Pro', value: 'pro' },
          { label: 'Enterprise', value: 'enterprise' },
        ],
      },
    },
  ],
});
```

Module-registered fields appear after admin-defined fields on the Info tab. Admins can control them on the **Integrations tab** of each node type — enabling/disabling, overriding the label, and changing display order. Field values are stored in the node's `data` JSON alongside admin-defined field values.

### NodeTypeRegistry (Database-Synced)

Unlike the 8 content registries above (which are code-based), the `NodeTypeRegistry` is synced with the database and provides methods for managing node types at runtime.

<Aside type="caution" title="`get` vs `getConfig`">
The registry stores `NodeTypeRegistration` wrappers (which bundle the config with optional component overrides). Use `getConfig(slug)` / `getAllTypes()` when you want the bare `NodeTypeConfig`. The `get()` / `getAll()` methods return the wrapper.
</Aside>

| Method | Returns | Description |
|--------|---------|-------------|
| `register(registration)` | `void` | Register a node type registration (wraps `config`) |
| `unregister(slug)` | `void` | Remove a type from the registry |
| `update(config)` | `void` | Update the config for an existing slug |
| `get(slug)` | `NodeTypeRegistration \| undefined` | Get the wrapper (config + optional overrides) |
| `getConfig(slug)` | `NodeTypeConfig \| undefined` | Get just the config — the common case |
| `getAll()` | `NodeTypeRegistration[]` | All registrations (wrappers) |
| `getAllTypes()` | `NodeTypeConfig[]` | All configs, sorted by `sortOrder` |
| `getEnabledTypes()` | `NodeTypeConfig[]` | Enabled configs only, sorted by `sortOrder` |
| `getByModule(moduleId)` | `NodeTypeConfig[]` | Types owned by a given module |
| `getSlugs()` | `string[]` | All registered slugs |
| `has(slug)` | `boolean` | Check existence |
| `isLoadedFromDb()` | `boolean` | Whether `syncFromDatabase` has run |
| `syncFromDatabase(types)` | `void` | Sync registry state with database records; fires `onSync` |
| `onSync(callback)` | `() => void` | Subscribe to sync events — returns an unsubscribe function |

**Reactive hooks:**
- `useEnabledNodeTypes()` — returns enabled configs, re-renders on `onSync()`
- `useStructureNavItems()` — returns sidebar navigation items, re-renders on `onSync()`

## Backend Integration

### Reacting to Structure Events (Action Hooks)

<Aside type="tip" title="Start here">
Action hooks are the most important backend integration point. If your module only does one backend integration with Structure, it should be this one.
</Aside>

Action hooks fire after every Structure mutation. Your module registers listeners that run automatically when nodes are created, updated, deleted, moved, archived, restored, or connected. This is how modules stay in sync — and it's what makes the module system truly composable.

**Common use cases:**
- **Cleanup on delete** — archive or remove your module's data when the parent node is deleted
- **Setup on create** — create default records when a new node of a specific type is created
- **Sync on update** — update cached data, trigger recalculations, or send notifications
- **Audit trail** — log significant events for compliance or debugging

<Aside type="caution" title="Per-node cleanup ≠ account-deletion cleanup">
Node-delete action hooks (`structure.node.deleted`) only fire when an *individual node* is deleted — they do **not** fire on account deletion. To erase your module's data when a user's account is deleted, use your module's own generated `security.registration.ts`, which registers a deletion handler with the starter's `securityRegistry`. Structure ships its own `security.registration.ts` that erases all Structure rows on account deletion; your integrating module needs its own to do the same for its tables. See the [Security guide](https://saas.uniqueicon.com.au/module-starter/security/) for details.
</Aside>

```typescript
// backend/src/features/tasks/action.registration.ts
import { actionRegistry } from '../../config/action.registry.js';

// Clean up tasks when a node is deleted
actionRegistry.addActionAsync(
  'structure.node.deleted',
  'tasks-cleanup-on-delete',
  async (context) => {
    const { userId, nodeId } = context;
    // Delete or archive tasks associated with this node
    await prisma.task.updateMany({
      where: { nodeId, userId },
      data: { status: 'archived' },
    });
  },
  20
);

// Set up default task list when a node is created
actionRegistry.addActionAsync(
  'structure.node.created',
  'tasks-setup-defaults',
  async (context) => {
    const { userId, node, nodeType } = context;
    if (nodeType === 'project') {
      // Create a default "To Do" task list for new projects
      await prisma.task_list.create({
        data: { nodeId: node.id, userId, name: 'To Do' },
      });
    }
  },
  20
);
```

### Transforming Structure Data (Filter Hooks)

Register filters to modify Structure data as it flows through the system. Filters run in a pipeline — each receives the output of the previous one.

```typescript
// backend/src/features/billing/filter.registration.ts
import { filterRegistry } from '../../config/filter.registry.js';

// Enrich node responses with billing data
filterRegistry.register(
  'structure.node.response',
  'billing-enrich-node',
  (node, context) => {
    // Add billing status to node metadata
    // (actual implementation would fetch from database)
    return {
      ...node,
      metadata: {
        ...node.metadata,
        hasBilling: true,
      },
    };
  },
  20
);

// Modify list queries to exclude certain types
filterRegistry.register(
  'structure.node.listQuery',
  'billing-filter-query',
  (query, context) => {
    // Example: modify the query before it executes
    return query;
  },
  20
);
```

### Accessing Structure Data via Service Registry

Other modules access Structure's backend service through the `serviceRegistry` — never by importing directly from the Structure feature folder.

```typescript
// backend/src/features/tasks/tasks.service.ts
import { serviceRegistry } from '../../config/service.registry.js';

export const tasksService = {
  async getTasksForNode(nodeId: string, userId: string) {
    // Access Structure service at runtime
    const structureService = serviceRegistry.get('structure');

    // Verify the node exists and belongs to the user
    const node = await structureService.getNode(nodeId, userId);

    // Now fetch tasks for this node
    return prisma.task.findMany({
      where: { nodeId: node.id, userId },
    });
  },
};
```

<Aside type="caution" title="Never import across module boundaries">
Do NOT import directly from `../../features/structure/structure.service.js`. Always use `serviceRegistry.get('structure')` to access other modules' services. This maintains module isolation and allows the system to detect missing dependencies.
</Aside>

### Connection Lifecycle Methods (v1.16.0+)

Connections carry provenance (`origin`) and lifecycle (`status`). Your module can create a connection as a `pending` proposal — it stays invisible to normal queries until it's promoted to active. All methods are available via `serviceRegistry.get('structure')`:

| Method | Description |
|--------|-------------|
| `createConnection(userId, fromNodeId, toNodeId, connectionType, data?, metadata?, options?)` | `options` is `{ origin?, status? }`. `origin` is a lowercase-kebab provenance tag stamped into `data.origin` (default `'explicit'`; other well-known values: `'discovered'`, `'active'`). `status` is `'active'` (default, goes live immediately) or `'pending'` (fires `structure.connection.proposed` instead of `created`) |
| `promoteConnection(id, userId)` | Promotes a pending connection to active. Re-validates that both endpoints are still active and re-checks the connection type's `isUnique`/`maxConnections` rules against the active set (`BadRequest` with machine-readable prefixes `duplicate-active-connection:`, `max-connections-reached:`, `endpoint-not-active:`). Fires `structure.connection.created` then `structure.connection.promoted` |
| `getNodeConnections(nodeId, userId, options?)` | `options` is `{ status?: string[], origin?: string }`. Returns **active connections only by default** (previously all statuses) — pass `status: ['active', 'pending']` to include proposals |
| `listConnections(userId, query?)` | Paginated listing across all nodes: `{ status?, origin?, connectionType?, page?, limit? }` → `{ connections, total, page, limit, totalPages }` |

The `NodeConnection` wire type now includes `status` and `origin` (legacy rows report `origin: 'explicit'`). Note that `origin` is a reserved connection field id — connection types can't define a custom field with that id.

### Node Type Proposals (v1.16.0+)

Instead of creating node types directly, your module can propose them for admin approval. A pending proposal reserves its slug but stays disabled and invisible to every other node-type query until approved:

| Method | Description |
|--------|-------------|
| `proposeNodeType(userId, config, { proposedBy, reason?, sourceContext? })` | Creates a pending, disabled node type from a standard `NodeTypeConfig`; `proposedBy` (your module id) becomes the `moduleId` fallback. Fires `structure.nodeType.proposed` |
| `listNodeTypeProposals(userId)` | Returns pending proposals, oldest first |
| `approveNodeTypeProposal(slug, userId, { scope? })` | Makes the proposal a live, enabled type. `scope: 'user'` (default) keeps it owned by the user; `'system'` promotes it to a shared system type. Fires `structure.nodeType.proposalApproved` and `structure.nodeType.created` — the type goes live at approval |
| `rejectNodeTypeProposal(slug, userId)` | Deletes the proposal row (freeing the slug). Fires `structure.nodeType.proposalRejected` with the full proposed config — listen for it if your module should remember not to re-propose |

### Cognitive Integration (v1.16.0+)

Structure ships opt-in cognitive foundations: pgvector node embeddings with a semantic search service, and a `features.cognitiveIngest` flag on node types. Because embeddings are an install-time opt-in, probe availability with `structureService.getCognitiveCapabilities()` — it returns `{ embeddings: { enabled, dimensions } }` without hard imports:

```typescript
const structureService = serviceRegistry.get('structure');
const caps = await structureService.getCognitiveCapabilities();

if (caps.embeddings.enabled) {
  // Generated only on embeddings-enabled installs — a separate service
  const embeddings = serviceRegistry.get('structureEmbeddings');
  const results = await embeddings.semanticSearchNodes(userId, queryEmbedding, { topK: 20 });
}
```

The `structureEmbeddings` service (registered only when the app opted into embeddings) provides `setNodeEmbedding`, `clearNodeEmbedding`, `listNodesNeedingEmbedding`, `semanticSearchNodes`, and `findSimilarNodes`. It is an in-process API only — there are no HTTP routes for embeddings or search. See the [Cognitive guide](/modules/structure/cognitive/) for full signatures, the `features.cognitiveIngest` flag, and install/opt-in details.

### Type Hierarchy Methods (v1.18.0+)

Node types can declare `superClasses` — parent type slugs forming an *is-a* DAG with field/feature inheritance (e.g. `court-order` is-a `legal-document`). Four traversal methods are available via `serviceRegistry.get('structure')`:

| Method | Description |
|--------|-------------|
| `getTypeAncestors(slug, userId)` | Returns ancestor slugs nearest-first (declared parents in declaration order, then theirs; deduped) |
| `getTypeDescendants(slug, userId)` | Returns all transitive subtype slugs (self not included) |
| `isSubtypeOf(subSlug, superSlug, userId)` | `true` when `subSlug` *is* `superSlug` or a transitive subtype (reflexive) |
| `expandTypeSlugs(slugs, userId)` | Expands a slug list to self + all transitive subtypes, deduped — the helper behind `listNodes`' `includeSubtypes` option |

`getNodeType` / `listNodeTypes` responses now also carry the raw `superClasses` and a read-only `resolved` key (`{ fields, features, ancestors }`) with inherited fields and features merged in — see the [Type Hierarchy guide](/modules/structure/type-hierarchy/) for inheritance semantics, subsumption in connection allowlists, and `includeSubtypes`.

### Registering Node Types from Your Module

If your module introduces its own node type (e.g., a "Task Board" type), seed it during backend initialisation — your module's init function, not per-user request hooks. Two rules make the seed correct:

1. **Seed via `importNodeTypes`, not `createNodeType`.** `createNodeType(userId, config)` pins the new row to the `userId` you pass, making the type invisible to every other user. `importNodeTypes` is the only service path that creates **shared rows** (`userId: null`) that every user can see.
2. **Guard with an existence check.** The guard is load-bearing: merge-mode import *updates* existing slugs, so an unguarded import would re-clobber admin customisations on every boot.

The actor string passed to both calls is only a visibility filter and audit label — any value sees shared rows, and nothing from it is written onto the row. A module-scoped label like `'tasks-init'` keeps the audit trail readable.

```typescript
// backend/src/features/tasks/tasks.init.ts
import { serviceRegistry } from '../../config/service.registry.js';

const TASK_BOARD_TYPE = {
  slug: 'task-board',
  name: 'Task Board',
  namePlural: 'Task Boards',
  icon: 'Kanban',
  moduleId: 'tasks',
  rules: {
    isRoot: false,
    allowedParents: ['project'],
    allowedChildren: [],
  },
  features: {
    favorites: true,
    archiving: true,
  },
  fields: [],
  navigation: {
    mainNav: { enabled: false },
    sidebarNav: { style: 'none' },
    pageNav: {
      style: 'tabs',
      builtInTabs: {
        overview: false,
        info: true,
        settings: true,
      },
      defaultTab: 'info',
    },
    listPage: { defaultView: 'grid' },
  },
  isSystem: true,
  isEnabled: true,
  sortOrder: 40,
};

export async function initTasks(): Promise<void> {
  const structureService = serviceRegistry.get('structure');

  // Existence check — load-bearing. Merge-mode import UPDATES existing
  // slugs, so re-running it unguarded would clobber admin customisations
  // on every boot.
  const existingTypes = await structureService.listNodeTypes('tasks-init', true);
  if (existingTypes.some(t => t.slug === 'task-board')) return;

  // importNodeTypes is the only service path that creates shared
  // (userId: null) rows visible to every user.
  await structureService.importNodeTypes('tasks-init', { nodeTypes: [TASK_BOARD_TYPE] }, 'merge');
}
```

<Aside type="caution" title="Don't seed shared types with createNodeType">
`createNodeType(userId, config)` writes the `userId` onto the row (and validates it against a real user), so the type exists for that one user only. It's the right call for user-initiated type creation — never for module seeding.
</Aside>

For a live example, the Cog Ingest module seeds its shared `working-memory` node type at backend boot with exactly this guard-then-import pattern — see the [Cog Ingest integration guide](/modules/cog-ingest/integration/).

<Aside type="note" title="moduleId matters">
Always set `moduleId` to your module's ID when creating node types. This lets the system know which module owns the type, and enables cleanup on module uninstall. Default node types seeded by Structure itself set `moduleId: 'structure'`.
</Aside>

### Namespaced Metadata Writes

When your module needs to store small amounts of data on a node without defining custom fields, use the metadata isolation API. This performs an atomic `jsonb_set` operation scoped to your module's namespace, preventing cross-module data corruption.

```typescript
// backend/src/features/billing/billing.service.ts
import { serviceRegistry } from '../../config/service.registry.js';

const structureService = serviceRegistry.get('structure');

// Write billing data to node.metadata.billing namespace
await structureService.updateNodeMetadata(
  nodeId,
  userId,
  'billing',  // moduleId — becomes the namespace key
  { planId: 'pro', trialEndsAt: '2025-06-01' }
);
// Result: node.metadata = { billing: { planId: 'pro', trialEndsAt: '2025-06-01' }, ...otherModules }
```

The API endpoint is `PATCH /api/v1/structure/nodes/:id/metadata` with body `{ moduleId, data }`. Each module's writes are isolated — updating the `billing` namespace never touches the `tasks` namespace. Fires the `structure.node.metadataUpdated` action hook.

<Aside type="tip" title="Metadata vs custom fields">
Use **custom fields** (defined in node type config or via module field registration) for data that users see and edit. Use **metadata** for internal module state that drives logic but isn't directly user-facing — like sync status, cached counts, or external IDs.
</Aside>

## Storing Node References in Your Schema

If your module's database tables need to reference Structure nodes, use a plain `nodeId` string field — not a Prisma `@relation`:

```prisma
model task {
  id        String   @id @default(cuid())
  userId    String
  nodeId    String              // References structure_node.id
  title     String
  status    String   @default("todo")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([userId])
  @@index([nodeId])             // Index for fast lookups by node
  @@index([userId, nodeId])     // Composite for user+node queries
}
```

<Aside type="caution" title="No cross-module @relation">
Module schemas must be self-contained. Use plain string fields with `@@index` for foreign keys to other modules' tables. Never use `@relation` across module boundaries — this would create tight coupling and break independent installation.
</Aside>

## Admin Control of Module Integrations

Admins can control module-provided content per node type in two places:

1. **Integrations tab** in the node type form (Structure Types > Edit Type > Integrations) — control tabs, widgets, card extensions, and module-registered fields per type
2. **Registries tab** on the App Structure admin page — browse all registered items across the 8 content registries (actions, columns, filters, sidebar sections, field types, tabs, widgets, card extensions)

Your module's registered items can be:

- **Enabled/disabled** per node type
- **Renamed** — admins can override the label you provide
- **Reordered** — admins can change the display order
- **Slot reassigned** (card extensions only) — admins can move an extension between card positions

Your module doesn't need to do anything special to support this. The registries automatically check the `integrations` config on each node type before returning results. If an admin disables your action for a particular node type, `getActionsForNode()` simply won't include it.

<Aside type="tip">
Module integrations are discovered at render time from the registries. As modules are installed or removed, the admin UI updates automatically to show available integrations. The Registries tab provides a central view of everything that's been registered across all modules.
</Aside>

## Integration Checklist

Use this checklist when building a module that integrates with Structure:

- [ ] **Dependency declared** — `depends: { structure: '^1.0.0' }` in `module.ts`
- [ ] **Frontend registrations** — Tabs, widgets, actions, columns, filters, sidebar sections, and/or card extensions registered via side-effect imports
- [ ] **Backend service access** — Use `serviceRegistry.get('structure')`, never direct imports
- [ ] **Event hooks** — Action registrations for cleanup on node delete, setup on node create
- [ ] **Schema isolation** — Plain `nodeId` string field with `@@index`, no `@relation`
- [ ] **Node types** — If registering types, set `moduleId` to your module ID
- [ ] **Order values** — Tabs/widgets/actions use order 30+ (after Structure's built-in items)
- [ ] **Labels** — Use `getLabel(nodeType, key)` for user-facing text instead of hardcoding type names
