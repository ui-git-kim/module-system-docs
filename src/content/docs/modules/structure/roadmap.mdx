## Next Priority

### Read-Only Node Detail Page with Edit Mode

**Priority:** High ŌĆö Blocks production use

The node detail page currently renders as an always-editable form. This needs to change to a read-only display by default with an Edit button to switch to form mode. Key requirements:

- **View mode** (default) ŌĆö fields render as formatted text/badges, not inputs
- **Edit button** on the NodeDetailHeader to toggle edit mode
- **Field-level permissions** ŌĆö `editableBy` controls which fields are editable (anyone/admin/system)
- **Node-level permissions** ŌĆö who can create, edit, archive, delete (configurable per node type)
- **Built-in field permissions** ŌĆö editableBy on Name, Description, Slug
- **System fields** (editableBy: 'system') always render as read-only, even in edit mode
- **NodeFieldRenderer** needs a `readOnly` render mode (text/badges instead of inputs)

### Conditional Colour Field Picker

**Priority:** High

The conditional colour admin UI currently uses a text input for the field ID. Needs:

- Combobox showing only select/multiselect fields (since colour rules map to option values)
- Auto-populate colour rules from the selected field's options
- Each option gets a colour picker
- Built-in field permissions (editableBy dropdown on Name/Description/Slug)

### Groups Tab Navigation Management

**Priority:** Medium

- Reorder groups via drag-and-drop
- Drag node types between groups
- Configure nav order from Groups tab directly

### Cross-Module Developer Experience

**Priority:** High

Improvements to make Structure easier and more powerful for other modules to integrate with. Identified from real-world integration work across modules like Builder, Tasks, and others.

#### NodeSelector Enhancements

- **Inline quick-create mode** ŌĆö Built-in minimal creation form within the NodeSelector popover, so users can create a node without leaving the current dialog. Currently requires `onCreateClick` callback which forces the consuming module to implement its own creation flow
- **Dialog/modal variant** ŌĆö Alternative `NodeSelector` rendered as a dialog instead of a popover, for use inside other dialogs where popover z-index and overflow issues arise
- **`onNodeCreated` callback** ŌĆö Fire a callback after a node is created inline, with the new node auto-selected. Simplifies the consuming module's state management

#### Frontend API Surface

- **`useNodeName(id)` hook** ŌĆö Lightweight hook that resolves a node ID to its display name without fetching the full node object. Useful for badges, breadcrumbs, and labels throughout consuming modules
- **`useNodesByType(type)` hook** ŌĆö Fetch all nodes of a given type for the current user. Common pattern that currently requires manual service calls
- **Export `StructureService` type** ŌĆö Typed interface for the structure service, so consuming modules can type their `serviceRegistry.get('structure')` calls without importing internals
- **`useNodeAncestors(id)` hook** ŌĆö Resolve the full parent chain for breadcrumb rendering in consuming modules

#### Backend API Surface

- **Frontend `structureService.createNode()`** ŌĆö A frontend service method for programmatic node creation (not just via the admin/detail page UI). Allows modules like Builder to create project nodes inline
- **Typed action/filter contexts** ŌĆö Export TypeScript interfaces for all action hook contexts (e.g., `NodeCreatedContext`, `NodeDeletedContext`) so consuming modules get autocomplete and type safety in their registrations

#### General DX

- **Integration testing utilities** ŌĆö Provide mock/stub helpers for `serviceRegistry.get('structure')` so module developers can unit test their Structure integrations without a running database
- **Node field value access helper** ŌĆö Simple utility to extract a typed field value from a node's data blob by field slug, handling missing fields gracefully

## Known Issues

### 2026-07-22 meta audit ŌĆö nine criticals: install blockers, an IDOR, and a data-loss guard

**Priority:** Critical ŌĆö fix before any new production install (batches AŌĆōC)

A four-dimension meta audit (standards / production readiness / performance / security) of v1.18.1 ran 2026-07-20 ŌåÆ 2026-07-22 with adversarial verification: 171 confirmed findings from 312 raw candidates, every finding surviving a three-lens refute-by-default review. The full register (issue / fix / evidence per item) is maintained in the module workspace audit artefact; this page records the actionable groupings. The nine criticals:

**Security and data integrity**

- **Cross-user connection read (IDOR).** `createNode` writes client-supplied `fromNodeId`/`toNodeId` with no ownership check (`backend/index.ts` ~L2431); `getNodeConnections` / `listConnections` then hydrate `include: { fromNode, toNode }` with no `userId` predicate (~L4155 / ~L4203), so a connection pointed at another user's node id returns that user's full node row ŌĆö name, description, and the entire `data` blob. Fix at the write side (verify both FKs resolve to caller-owned rows) and scope the relation includes.
- **Deletion handlers can wipe entire tables.** Every erasure handler derives `const userId = ctx.subject.userId ?? ctx.subject.tenantId` then calls `deleteMany({ where: { userId } })` (`backend/registrations.ts` ~L83 onward). If both fields are absent, Prisma treats `undefined` as *omit the filter* ŌĆö the handler deletes **every row for every user**. Reachable only via a host contract violation, but the blast radius is unrecoverable and the guard is one line per handler.
- **`bulkCreateNodes` reports partial success from an aborted transaction** (`backend/index.ts` ~L2994). Per-row errors are caught *inside* an interactive transaction Prisma has already rolled back, so rows reported "created" were never committed.
- **`NodeForm` never resets between opens** (`frontend/components.ts` ~L616). The router reuses a single `:nodeType` route, so navigating `/clients` ŌåÆ `/projects` keeps the stale form state and submits `nodeType: 'clients'` ŌĆö a Project is persisted as a Client, with the previous entry's values pre-filled.

**Install blockers ŌĆö generated code fails to compile or build on a current host**

- Zero-argument `useRef<T>()` ├Ś3 (`pages.ts` ~L58 / ~L481, `components.ts` ~L865) ŌĆö the zero-arg overload was removed in `@types/react` 19.
- `.filter(Boolean)` does not narrow (`components.ts` ~L117) ŌĆö strict type-check failure; needs a type-guard predicate.
- `handleExport` passes `{ includeDisabled: true }` where `listConnectionTypes(includeDisabled?: boolean)` is declared (`pages.ts` ~L1716) ŌĆö type error (works at runtime only because an object is truthy).
- `typeof formData.fields[0]` (`pages.ts` ~L4030) ŌĆö `formData` is `Partial<NodeTypeConfig>`, so this resolves as `(NodeTypeField[] | undefined)[0]` ŌåÆ TS2339, confirmed by compiler repro. Use `NonNullable<typeof formData.fields>[number]`.
- **`@dnd-kit/{core,sortable,utilities}` imported but undeclared** (`pages.ts` ~L1429ŌĆō1437 vs `module.ts` `frontend.dependencies`, which lists only `react-hook-form`) ŌĆö a fresh `npx structure init` does not build. Near-term fix: declare the three packages in `module.ts`. Direction decided 2026-07-22: drag-and-drop dependencies move **into the starter template** as base dependencies (tracked on the [starter roadmap](/starter-template/roadmap/)); the module declaration then remains as a harmless redundancy for older hosts.

### 2026-07-22 meta audit ŌĆö moderate findings (53)

**Priority:** HighŌĆōMedium (batches DŌĆōG)

Grouped by fix area, with representative sites:

- **Backend ownership and input validation (Security).** `getNode` / `listNodes` `children` includes carry no `userId` predicate; `duplicateNode` trusts `targetParentId` unverified; `bulkCreateNodes` repeats the unvalidated FK write; the public API route omits `allowNoOrigin` so production CORS rejects every headless API client (`backend/index.ts` ~L67); `dataFilters` numeric operators coerce unvalidated values into raw SQL and crash the request (~L2158); `importNodeTypes` creates imported types as shared `userId: null` rows; field-level `editableBy` and node-type `permissions` are enforced only in the browser ŌĆö no server-side counterpart.
- **Contract and dead wiring (Production).** `labels` is accepted, typed, and exported but never persisted or read back; `createNode` silently drops the `settings` field its own schema accepts; `StructureRegistry.getFieldsForNodeType` is never called, so module-registered fields never render; frontend lifecycle hooks are declared but never invoked (a delete veto is ignored); the Structure settings page saves preferences nothing reads; `NodeForm` omits `allData`, so conditional visibility and computed fields are dead in create/edit; `syncFromDatabase` wipes module-registered node types with no re-application path.
- **Frontend error surfaces (Production).** `useConnectionTypeMutations`, `useApiKeyMutations`, and `useTemplateMutations` have no `onError` ŌĆö a failed key revoke reads as success; the list filter dropdown renders a `SelectItem` with an empty `value` (hard render crash); the node-type config page renders a false 404 during load; the admin page has no error state (a failed load renders "No types yet"); config import double-submits and swallows per-item failures; the connection-type form submits with no required-field validation; the published-slug input fires a PATCH per keystroke.
- **Data integrity (Production).** Node slug uniqueness is never pre-checked, so `P2002` surfaces as an unhandled error; `addField` mints duplicate field ids after any deletion; `getNextSortOrder` runs on the global Prisma client from inside transactions; `@@unique([userId, parentId, name])` is NULL-asymmetric with no conflict handling; card/header delete fires with no confirmation.
- **Performance (backend).** `updateDescendantPaths` issues one UPDATE per descendant on every move; `duplicateNode` runs a SELECT per descendant; `getNodeConnections` is unbounded and hydrates both full endpoint rows per connection; `bulkCreateNodes` can issue ~1,500 sequential round trips inside one default-timeout transaction; `listPage.defaultView` is silently ignored (lazy initial state races the config load).
- **Secrets and keys (Security).** The `secret` field type stores credentials in plaintext while the masked UI implies protection ŌĆö move to the starter's `encryptField`, with a data migration for existing values; the `url` field renders an unvalidated `href` (`javascript:` XSS, reachable via public-API writes); the admin UI cannot set a key expiry, so every issued key is perpetual; the plaintext API key is retained in the TanStack mutation cache after the create dialog closes.

### Several built-in field types render as plain text

**Priority:** Medium

A v1.13.2 documentation-accuracy audit found that several field types declared in the `NodeTypeFieldType` union have no `case` in the `NodeFieldRenderer` switch and fall through to a plain text `<Input>`: **`media`** (no file uploader ŌĆö `mediaConfig` is inert), **`chained-select`** (`chainedSelectConfig` inert), and the **`multiselect`** _edit_ control (read-only display renders the stored array as badges correctly, but the editor is a text box). Related inert field properties: **`inputMask`**, textarea **`size`**, and **`validation.pattern`** / text length (persisted by the admin editor but never enforced at render time ŌĆö only `number` min/max are wired). The docs now describe these honestly; implementing the real renderers is tracked under Planned Improvements below.

The `NodeTypeForm` tabs rendering bug flagged during v1.9.7 / v1.9.8 turned out to be caused by a `@neondatabase/*` global CSS import in the host app leaking layout rules onto shadcn primitives ŌĆö **not** a Structure issue. Resolution tracked in the [starter template repo](/starter-template/roadmap/) instead. The `tabs.tsx` overwrite via `npx shadcn@latest add tabs --overwrite` alone wasn't enough; the real cause is in `frontend/src/index.css` (or equivalent global stylesheet import chain).

### Field-group repeater renders literal `${...}` placeholders

**Priority:** Ō£ģ Resolved (v1.15.3, reconfirmed 2026-07-19)

> Fixed: `NODE_FIELD_RENDERER` now renders the correct JSX expressions `#{itemIdx + 1}` (`components.ts` ~L1819) and `Add {field.label}` (~L1857) instead of the over-escaped `#\${itemIdx + 1}` / `Add \${field.label}`. Retained below for provenance; safe to archive.

The `field-group` field renderer emits two **over-escaped** template expressions as JSX _text_, so a user editing a field-group (repeater) sees the literal strings `#${itemIdx + 1}` and `Add ${field.label}` instead of the item number and the field label. In `src/templates/frontend/components.ts` (`NODE_FIELD_RENDERER`, ~L1813 and ~L1851) the JSX reads `#\${itemIdx + 1}` and `Add \${field.label}` ŌĆö these should be JSX expressions (`#{itemIdx + 1}` and `Add {field.label}`), not escaped template-literal interpolations. `npm run check:templates` only flags _un_-escaped `${`, so it does not catch this _over_-escape. Fix is a two-character change per site.

### Deferred performance refinements (from the v1.13.3 health audit)

**Priority:** Low

Two performance findings were intentionally **not** auto-applied because the naive fix would regress correctness; each needs a careful, generated-app-verified implementation:

- **`getTree` column `select`** ŌĆö `getTree` fetches full node rows (including the heavy `data`/`settings`/`metadata` JSON) just to build a tree from structural columns. A `select` would shrink the payload, but the method returns `NodeTreeItem.node` typed as a full `StructureNode` that consumers read ŌĆö narrowing the query without narrowing the type is unsound. Introduce a dedicated lightweight tree-node type (or accept the wider type) first.
- **Data-filter query pagination** ŌĆö when `dataFilters` are present, `listNodes` runs a raw `SELECT "id"` with no `LIMIT`, materialising every matching id into an `IN (...)`. Pushing `LIMIT`/`OFFSET` into that raw query is wrong (it doesn't carry the other where-clauses or the `orderBy`); the correct fix folds the data filter into a single paged query. A `.max(50)` bound on `dataFilters` now caps filter *complexity* as an interim mitigation.

### Implement the inert built-in field renderers

**Priority:** Medium

Surfaced by the v1.13.2 docs-accuracy audit. These field types are declared and selectable but not implemented in `NodeFieldRenderer` (they render as a plain text input):

- **`media`** ŌĆö a real file-upload widget driven by `mediaConfig` (accept, maxSize, multiple). Currently falls through to a text input.
- **`chained-select`** ŌĆö dependent cascading `<Select>`s driven by `chainedSelectConfig` (levels, options).
- **`multiselect`** (edit) ŌĆö a multi-pick control. The read-only display already renders the stored array as badges; only the editor is missing.
- **`inputMask`** ŌĆö enforce the mask (e.g. `AAA-999`) on `text` input.
- **textarea `size`** ŌĆö map `small`/`medium`/`large` to row counts (currently hardcoded to 3 rows).
- **`validation.pattern` / length** ŌĆö enforce regex and min/max length for `text` (currently stored but not applied; only `number` min/max are wired as HTML attributes).

Until implemented, modules needing these can register a custom field type via `nodeFieldTypeRegistry`.

### Toast copy should use the node type's label, not "Node"

**Priority:** Medium

The mutation hooks currently toast generic messages like `"Node created"`, `"Node updated"`, `"Node deleted"`, `"Node moved"`, etc. End users don't think in terms of "nodes" ŌĆö they think in terms of Projects, Clients, Folders, Tags, whatever the app has configured. A toast saying `"Node updated"` is technically accurate but confusingly abstract.

The module already has the infrastructure to fix this:

- `nodeType.name` and `nodeType.namePlural` are available on every node (e.g. "Project", "Client")
- The `getLabel()` utility in `utils/labels.ts` supports admin-customised labels

The fix lives in `useNodeMutations` (in `src/templates/frontend/index.ts`'s `USE_NODE_MUTATIONS` constant). Each mutation already receives or can look up the node ŌĆö thread the `nodeType.name` (or `getLabel('node.created', nodeType.name)`) through the toast message:

```ts
// Before
toast.success('Node created');

// After ŌĆö needs the nodeType name on the mutation context
toast.success(\`\${nodeType.name} created\`);   // "Project created"
toast.success(\`\${nodeType.name} updated\`);   // "Client updated"
```

Same pattern applies to `useConnectionMutations` ("Connection" ŌåÆ could use the connection type label) and `useNodeTypeMutations` (admin-only ŌĆö "Node type" is acceptable there since admins are the audience).

Approach options to consider:
1. **Pass `nodeType` into each mutation call** ŌĆö consumer component provides it. Mutation shape changes, type-safe.
2. **Look up `nodeType` from the node result/context** ŌĆö mutations resolve it from `nodeTypeRegistry.getConfig(node.nodeType)`. Call-site stays unchanged.
3. **Expose a `getToastLabel()` overridable via admin config** ŌĆö gives each app full control over wording.

Option 2 is probably the least disruptive ŌĆö no consumer changes, just richer toasts.

### Server-Side Field Filtering

**Priority:** Medium

Field filters on the list page are applied client-side after fetching. Should pass filter parameters to the API and apply in Prisma query.

### Reconcile scaffold reference templates with Structure-specific ones

**Priority:** Ō£ģ Resolved

An earlier scaffold split added generic `my-feature` templates in `src/templates/backend/{routes,services,types}.ts` and `src/templates/frontend/{routes,services,hooks,types}.ts`. Structure keeps its real templates inline in the per-area `index.ts` files, so these were dead, unimported exports.

**Resolved:** the dead `my-feature` files (backend and frontend) have been removed, and the merge-noise concern no longer applies ŌĆö scaffold **v1.12.0** treats `src/templates/**` and `src/module.ts` as **module-owned**, so `scaffold:update` never auto-merges upstream edits into them (it reports them as advisory via `scaffold:check`) and never re-introduces files you've deleted.

### styleRegistry Contribution (deferred ŌĆö handled by Style module)

**Priority:** Deferred

Node-type colours, tree-view indent tokens, and card accent colours could be exposed as contributable design tokens via the scaffold v1.6.0 `styleRegistry`. Deferred: theming is the responsibility of a dedicated Style module, not Structure.

### Searchable Encrypted Fields

**Priority:** Low ŌĆö depends on a starter-provided primitive

A future capability to let a node-type field be flagged `encrypted` (AES-256-GCM at rest, key never in the database) while still being **filterable within a single user's own data space**. Today an encrypted field is genuinely un-filterable and un-sortable: `buildDataFilterQuery` emits raw SQL against `data->>'field'`, and a non-deterministic GCM envelope is opaque to `=`, `ILIKE`, and range comparisons. This entry scopes how we will restore *limited* search without ever implying false security. Nothing here ships until the dependency below lands.

- **Dependency on the starter.** This will require a starter-provided **per-tenant blind-index primitive** ŌĆö a keyed `HMAC(indexKey, normalize(value))` helper, with the index key derived *separately* from the encryption key. It does not exist yet. It also requires the starter to **actually derive per-tenant keys (HKDF)** first: the shipped envelope's `{ tenantId }` option is currently ignored by the env adapter, so until that is implemented every tenant shares one global key ŌĆö building an index on that today would produce a single global tag space, not per-tenant isolation.
- **Per-field `encrypted` flag.** Admins will be able to flag a field encrypted; an additional opt-in will enable a blind-index sibling tag column (`dataIndex`) for that field. The blind index will be **restricted to high-entropy fields** ŌĆö never booleans, status enums, country, or short codes, whose plaintext is recoverable from a deterministic tag via frequency analysis.
- **Exact-match / keyword only on encrypted fields.** A blind index will support `equals`, `in`, membership, dedupe, and whole-word keyword lookups only. **Range, sort, and substring (`contains`/`starts_with`) cannot use it** ŌĆö these will fall back to **decrypt-in-memory** (the default, viable because per-user datasets are small and bounded) or be **disabled** in the filter UI for encrypted fields.
- **Honest leakage model.** A deterministic blind index *always* leaks the equality pattern and value frequency, and a per-tenant key only **limits cross-user correlation** ŌĆö it does not remove within-user leakage. The admin UI will warn accordingly; decrypt-in-memory (which leaks nothing to the database) stays the recommended default, with the blind index reserved for the narrow case where dataset size forces SQL-side equality.
- **Safe backfill.** Adding the flag, rotating keys, or building the index will re-encrypt existing values, so the work must guarantee a fresh random IV per encryption (GCM nonce reuse is catastrophic) and maintain the tag transactionally alongside the ciphertext.
