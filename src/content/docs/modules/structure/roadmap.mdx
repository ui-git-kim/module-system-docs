## Next Priority

### Built-in field permissions (`builtInFieldConfig`)

**Priority:** High — the last unwired piece of the read-only/edit-mode work

`builtInFieldConfig.{name,description,slug}` is fully declared, fully configurable in the admin node-type editor, and fully persisted — but **nothing reads it**. The Name/Description/Slug inputs render unconditionally in `NodeForm`, and the backend update allowlist sets all three with no permission check, so the setting has no effect anywhere.

- Derive `readOnly`/`disabled` for the three built-in inputs from `editableBy` (`system` always locked, `admin` locked for non-admins)
- Mirror the existing custom-field guard server-side in `updateNode`, so a direct API call cannot bypass the UI gate
- Resolve `builtInFieldConfig.*.computedTemplate` at render/save — `useResolvedTemplate` currently only covers custom-field `computedTemplate`
- Apply field-level `editableBy` inside `NodeForm`'s custom-field loop too, matching the rule `NodeInfoTab` already applies
- Gate `NodeChildrenTab`'s create buttons on the child type's `permissions.create` (they currently render from `allowedChildren` alone)

<Aside type="note">
The rest of the original "Read-Only Node Detail Page with Edit Mode" entry is **done** (v1.19.x and earlier) and was verified against the code on 2026-07-23: the detail page is read-only by default, `NodeFieldRenderer` has a real `readOnly` mode, `isEditing` threads from `NodeDetailPage` through `NodeDetailHeader`'s Edit toggle into every registered tab via `NodeTabProps`, field-level `editableBy` gates the detail page (with a server-side backstop), and node-level create/edit/archive/delete permissions gate their UI. Only the built-in-field half above remains.
</Aside>

### Conditional Colour — admin polish

**Priority:** Low

The picker itself is **done** — it is a Combobox limited to select/multiselect/boolean/toggle fields, with a colour picker per rule — and as of **v1.20.0** the saved rules are actually applied at render by `resolveNodeColor()`. Remaining polish:

- Show the option *label* rather than the raw value in each rule row
- Warn when a rule value no longer matches any option after the source field's options are edited

### Groups Tab Navigation Management

**Priority:** Medium

- Reorder groups via drag-and-drop
- Drag node types between groups
- Configure nav order from Groups tab directly

### Node Merge

**Priority:** Medium

A first-class merge primitive — fold one node instance into another when two records turn out to be the same real-world thing:

- `mergeNode(fromId, intoId)` on the service: redirect the loser's connections to the winner, carry over missing `data` keys, archive the loser
- Machine-readable conflict handling when both nodes hold different values for the same field
- Fires a `structure.node.merged` action hook so modules can react

Primary consumer is the [cognitive-db v2](/cognitive-db/v2-rfc/) engine's entity reconciliation (two existing entities later discovered to be duplicates — e.g. "Bob" and "Robert Smith"); equally useful for user-created duplicates in any app. Until this lands, the engine buffers merge candidates in `cog_intelligence_discovery` rather than executing them — a deferral agreed on the v2 track, so nothing blocks on it.

### Tree View Improvements

**Priority:** Medium

`StructureTree` today is a ~125-line recursive renderer: indentation, chevron toggles (self-managing since v1.20.0), selection highlight, icon + name, and a `maxDepth` guard. It is also **not rendered anywhere in the module** — there is no `tree` option in the list-page view switcher and `sidebarNav.style === 'tree'` is unimplemented, so the component and `useNodeTree` are available to consuming modules but unused internally.

- **Render it somewhere** — add a `tree` view option and/or implement the already-configurable `sidebarNav.style === 'tree'`
- Inline node creation in tree view (respecting `rules.allowedChildren`)
- Drag-and-drop reparenting, with drop-on-node and drop-between-sibling zones, calling the existing `moveNode`
- Expand/collapse state persistence per user (persist expanded ids to `structure_settings`, rehydrate on mount)
- Tree search and filtering, auto-expanding ancestors of matches
- Keyboard navigation with `role="tree"`/`treeitem`, `aria-expanded`, roving tabindex and arrow-key handling
- Remove or populate the vestigial `NodeTreeItem.isExpanded` / `isSelected` fields — `buildTree` never sets them and the tree never reads them

## Known Issues

### 2026-07-22 meta audit — nine criticals: install blockers, an IDOR, and a data-loss guard

**Priority:** ✅ Resolved (v1.19.0–v1.19.2) — all nine fixed, host-validated

> **Shipped.** Every critical below is fixed and verified by a real host install (veradai, starter
> v1.62.0, Prisma 7.8 / React 19): connection FK ownership + relation scoping (v1.19.0), the five
> install blockers (v1.19.0, `@dnd-kit` auto-installs on update), the deletion-handler guard, the
> `NodeForm` reset and bulk-create semantics (v1.19.0). Host type-check caught one regression during
> validation — `ErrorAlert` takes a string, not an `Error` — fixed in **v1.19.1**. A follow-up pass
> (**v1.19.2**) then cleared all 74 pre-existing latent type errors in the generated code, so a
> fresh install now type-checks **0 errors** on both backend and frontend. Original detail retained
> below for provenance.

A four-dimension meta audit (standards / production readiness / performance / security) of v1.18.1 ran 2026-07-20 → 2026-07-22 with adversarial verification: 171 confirmed findings from 312 raw candidates, every finding surviving a three-lens refute-by-default review. The full register (issue / fix / evidence per item) is maintained in the module workspace audit artefact; this page records the actionable groupings. The nine criticals:

**Security and data integrity**

- **Cross-user connection read (IDOR).** `createNode` writes client-supplied `fromNodeId`/`toNodeId` with no ownership check (`backend/index.ts` ~L2431); `getNodeConnections` / `listConnections` then hydrate `include: { fromNode, toNode }` with no `userId` predicate (~L4155 / ~L4203), so a connection pointed at another user's node id returns that user's full node row — name, description, and the entire `data` blob. Fix at the write side (verify both FKs resolve to caller-owned rows) and scope the relation includes.
- **Deletion handlers can wipe entire tables.** Every erasure handler derives `const userId = ctx.subject.userId ?? ctx.subject.tenantId` then calls `deleteMany({ where: { userId } })` (`backend/registrations.ts` ~L83 onward). If both fields are absent, Prisma treats `undefined` as *omit the filter* — the handler deletes **every row for every user**. Reachable only via a host contract violation, but the blast radius is unrecoverable and the guard is one line per handler.
- **`bulkCreateNodes` reports partial success from an aborted transaction** (`backend/index.ts` ~L2994). Per-row errors are caught *inside* an interactive transaction Prisma has already rolled back, so rows reported "created" were never committed.
- **`NodeForm` never resets between opens** (`frontend/components.ts` ~L616). The router reuses a single `:nodeType` route, so navigating `/clients` → `/projects` keeps the stale form state and submits `nodeType: 'clients'` — a Project is persisted as a Client, with the previous entry's values pre-filled.

**Install blockers — generated code fails to compile or build on a current host**

- Zero-argument `useRef<T>()` ×3 (`pages.ts` ~L58 / ~L481, `components.ts` ~L865) — the zero-arg overload was removed in `@types/react` 19.
- `.filter(Boolean)` does not narrow (`components.ts` ~L117) — strict type-check failure; needs a type-guard predicate.
- `handleExport` passes `{ includeDisabled: true }` where `listConnectionTypes(includeDisabled?: boolean)` is declared (`pages.ts` ~L1716) — type error (works at runtime only because an object is truthy).
- `typeof formData.fields[0]` (`pages.ts` ~L4030) — `formData` is `Partial<NodeTypeConfig>`, so this resolves as `(NodeTypeField[] | undefined)[0]` → TS2339, confirmed by compiler repro. Use `NonNullable<typeof formData.fields>[number]`.
- **`@dnd-kit/{core,sortable,utilities}` imported but undeclared** (`pages.ts` ~L1429–1437 vs `module.ts` `frontend.dependencies`, which lists only `react-hook-form`) — a fresh `npx structure init` does not build. Near-term fix: declare the three packages in `module.ts`. Direction decided 2026-07-22: drag-and-drop dependencies move **into the starter template** as base dependencies (tracked on the [starter roadmap](/starter-template/roadmap/)); the module declaration then remains as a harmless redundancy for older hosts.

### 2026-07-22 meta audit — moderate findings (53)

**Priority:** ✅ Resolved (v1.19.0) — batches D–G shipped

> **Shipped.** All six groups below are fixed in v1.19.0. Highlights worth knowing as behaviour
> changes: `secret` fields are now [encrypted at rest](/modules/structure/fields/#secret) with lazy
> migration of existing plaintext; field-level `editableBy` is
> [enforced server-side](/modules/structure/fields/) (a direct API write to a locked field now 400s);
> [bulk create is partial-success](/modules/structure/configuration/), not all-or-nothing;
> module-registered fields actually render; the frontend delete veto and lifecycle notifies fire;
> `labels` [persists end-to-end](/modules/structure/usage/#labels-utility); and API keys can be given
> an [expiry from the admin UI](/modules/structure/public-api/#security). Original detail retained
> below for provenance.

- **Backend ownership and input validation (Security).** `getNode` / `listNodes` `children` includes carry no `userId` predicate; `duplicateNode` trusts `targetParentId` unverified; `bulkCreateNodes` repeats the unvalidated FK write; the public API route omits `allowNoOrigin` so production CORS rejects every headless API client (`backend/index.ts` ~L67); `dataFilters` numeric operators coerce unvalidated values into raw SQL and crash the request (~L2158); `importNodeTypes` creates imported types as shared `userId: null` rows; field-level `editableBy` and node-type `permissions` are enforced only in the browser — no server-side counterpart.
- **Contract and dead wiring (Production).** `labels` is accepted, typed, and exported but never persisted or read back; `createNode` silently drops the `settings` field its own schema accepts; `StructureRegistry.getFieldsForNodeType` is never called, so module-registered fields never render; frontend lifecycle hooks are declared but never invoked (a delete veto is ignored); the Structure settings page saves preferences nothing reads; `NodeForm` omits `allData`, so conditional visibility and computed fields are dead in create/edit; `syncFromDatabase` wipes module-registered node types with no re-application path.
- **Frontend error surfaces (Production).** `useConnectionTypeMutations`, `useApiKeyMutations`, and `useTemplateMutations` have no `onError` — a failed key revoke reads as success; the list filter dropdown renders a `SelectItem` with an empty `value` (hard render crash); the node-type config page renders a false 404 during load; the admin page has no error state (a failed load renders "No types yet"); config import double-submits and swallows per-item failures; the connection-type form submits with no required-field validation; the published-slug input fires a PATCH per keystroke.
- **Data integrity (Production).** Node slug uniqueness is never pre-checked, so `P2002` surfaces as an unhandled error; `addField` mints duplicate field ids after any deletion; `getNextSortOrder` runs on the global Prisma client from inside transactions; `@@unique([userId, parentId, name])` is NULL-asymmetric with no conflict handling; card/header delete fires with no confirmation.
- **Performance (backend).** `updateDescendantPaths` issues one UPDATE per descendant on every move; `duplicateNode` runs a SELECT per descendant; `getNodeConnections` is unbounded and hydrates both full endpoint rows per connection; `bulkCreateNodes` can issue ~1,500 sequential round trips inside one default-timeout transaction; `listPage.defaultView` is silently ignored (lazy initial state races the config load).
- **Secrets and keys (Security).** The `secret` field type stores credentials in plaintext while the masked UI implies protection — move to the starter's `encryptField`, with a data migration for existing values; the `url` field renders an unvalidated `href` (`javascript:` XSS, reachable via public-API writes); the admin UI cannot set a key expiry, so every issued key is perpetual; the plaintext API key is retained in the TanStack mutation cache after the create dialog closes.

### 2026-07-23 roadmap verification — five latent bugs

**Priority:** ✅ Four resolved (v1.20.0), one deferred by design

Every open roadmap item was re-verified against the code on 2026-07-23. Besides correcting the entries themselves, that pass surfaced five defects nobody had logged:

- ✅ **`conditionalColor` was write-only.** Admins could configure colour rules that no code ever read — the entire feature was inert. `resolveNodeColor()` now applies them (explicit node colour wins, then a matching rule, then the configured fallback, then the type colour) across `NodeIcon`, `NodeCard` and `NodeDetailHeader`.
- ✅ **The "View Public" button dead-ended.** `NodeDetailHeader` linked to `/p/:publishedSlug`, a route the module does not ship, so it fell into the protected catch-all. Removed until public views land.
- ✅ **Three field types could corrupt data.** `media`, `multiselect` and `chained-select` fell through to a text input bound to `onChange(e.target.value)`, letting a user overwrite an array/object value with a string. All three are now implemented properly.
- ✅ **`StructureTree` chevrons never worked when uncontrolled.** `expandedIds` defaulted to a fresh empty `Set` with no internal state, so unless a caller supplied both `expandedIds` and `onToggleExpand` the tree could not expand. Now controlled when both props are given, self-managing otherwise.
- ✅ **`getTree` was unbounded** — every active node for the user, no `take`, no type scope. Capped at 5,000; because rows are ordered depth-ascending, truncation drops the deepest nodes first and so can never orphan a child from a loaded parent.

**Deferred by design:** `StructureTree` is still not rendered anywhere in the module — there is no `tree` option in the list-page view switcher and `sidebarNav.style === 'tree'` is unimplemented, so the component (and `useNodeTree`) remain available to consuming modules but unused internally. That belongs with the Tree View Improvements entry below.

### Several built-in field types render as plain text

**Priority:** Medium

A v1.13.2 documentation-accuracy audit found that several field types declared in the `NodeTypeFieldType` union have no `case` in the `NodeFieldRenderer` switch and fall through to a plain text `<Input>`: **`media`** (no file uploader — `mediaConfig` is inert), **`chained-select`** (`chainedSelectConfig` inert), and the **`multiselect`** _edit_ control (read-only display renders the stored array as badges correctly, but the editor is a text box). Related inert field properties: **`inputMask`**, textarea **`size`**, and **`validation.pattern`** / text length (persisted by the admin editor but never enforced at render time — only `number` min/max are wired). The docs now describe these honestly; implementing the real renderers is tracked under Planned Improvements below.

The `NodeTypeForm` tabs rendering bug flagged during v1.9.7 / v1.9.8 turned out to be caused by a `@neondatabase/*` global CSS import in the host app leaking layout rules onto shadcn primitives — **not** a Structure issue. Resolution tracked in the [starter template repo](/starter-template/roadmap/) instead. The `tabs.tsx` overwrite via `npx shadcn@latest add tabs --overwrite` alone wasn't enough; the real cause is in `frontend/src/index.css` (or equivalent global stylesheet import chain).

### Field-group repeater renders literal `${...}` placeholders

**Priority:** ✅ Resolved (v1.15.3, reconfirmed 2026-07-19)

> Fixed: `NODE_FIELD_RENDERER` now renders the correct JSX expressions `#{itemIdx + 1}` (`components.ts` ~L1819) and `Add {field.label}` (~L1857) instead of the over-escaped `#\${itemIdx + 1}` / `Add \${field.label}`. Retained below for provenance; safe to archive.

The `field-group` field renderer emits two **over-escaped** template expressions as JSX _text_, so a user editing a field-group (repeater) sees the literal strings `#${itemIdx + 1}` and `Add ${field.label}` instead of the item number and the field label. In `src/templates/frontend/components.ts` (`NODE_FIELD_RENDERER`, ~L1813 and ~L1851) the JSX reads `#\${itemIdx + 1}` and `Add \${field.label}` — these should be JSX expressions (`#{itemIdx + 1}` and `Add {field.label}`), not escaped template-literal interpolations. `npm run check:templates` only flags _un_-escaped `${`, so it does not catch this _over_-escape. Fix is a two-character change per site.

### Structure-only styling compliance sweep (v1.15.2 standards-alignment audit)

**Priority:** ✅ Resolved — narrow §7 opacity residue remains (see 2026-07-19 audit below)

> The typography / weight / colour-literal / shadow / `!important` sweep is **done**: a 2026-07-19 grep across all templates found **0** hits in every one of those categories. What remains is a narrower **§7 opacity-for-effect** residue (25 `bg-muted/NN`, `bg-background/95`, `bg-destructive/90`, `border-muted-foreground/30` modifiers) — tracked in the v1.18.x audit entry. Original detail retained below.

An LLM-rules §5 ("structure-only styling") audit of the generated UI templates found ~209 forbidden typography classes plus a handful of colour/elevation literals. Template feature code must express **layout + state only** — type scale, weight, colour and elevation belong to the shadcn primitives, `globals.css`, or a Style Registry.

- **Typography scale (~157)** — `text-xs/sm/lg/xl` across `frontend/components.ts` (~26) and `frontend/pages.ts` (~131).
- **Font weight (~52)** — `font-medium/semibold/normal` (mostly `pages.ts`).
- **Hard colour literals (~6)** — `text-amber-600` soft-warning (`pages.ts:5103`); `text-yellow-500` favourite-star ×4 and `text-green-500` copied-check (`components.ts`). Move to semantic tokens (`text-muted-foreground` / `text-destructive`, or a success token).
- **Decorative shadows (3)** — `hover:shadow-md` ×2 and a drag-state `shadow-lg`.
- **`!important` + arbitrary value** — the NodeForm `SheetContent` width override `!w-1/3 !max-w-[33vw]` (`components.ts:630`) needs a non-`!` approach.

This sweep changes rendered output, so it must be **verified in a generated app**, not just built. Acceptable exceptions (leave as-is): idiomatic shadcn opacity idioms (`hover:bg-muted/50`, `bg-background/95 backdrop-blur`), and the ColorPicker's data-driven swatches / `PRESET_COLORS` hex array / inline `style={{ backgroundColor }}` (colours-as-data, not chrome).

### Node-type-group CRUD is missing action hooks

**Priority:** ✅ Resolved (v1.15.3)

> `createNodeTypeGroup` / `updateNodeTypeGroup` / `deleteNodeTypeGroup` now fire `actionRegistry.doActionAll('structure.nodeTypeGroup.created' | '…updated' | '…deleted', …)` (`backend/index.ts` L3124 / L3155 / L3184). Confirmed by the 2026-07-19 audit. Original detail retained below.

`createNodeTypeGroup` / `updateNodeTypeGroup` / `deleteNodeTypeGroup` (`src/templates/backend/index.ts` ~L2207–2276) only `log.info` — they never fire `actionRegistry.doActionAll('structure.nodeTypeGroup.created' | '…updated' | '…deleted', …)`, unlike every sibling config CRUD (node, nodeType, connectionType, template dispatch ~28 `doActionAll` calls between them). Other modules can't react to group changes. Add the three hooks and document them in the action-registration hook table.

### Deferred performance refinements (from the v1.13.3 health audit)

**Priority:** Low

Two performance findings were intentionally **not** auto-applied because the naive fix would regress correctness; each needs a careful, generated-app-verified implementation:

- **`getTree` column `select`** — `getTree` fetches full node rows (including the heavy `data`/`settings`/`metadata` JSON) just to build a tree from structural columns. A `select` would shrink the payload, but the method returns `NodeTreeItem.node` typed as a full `StructureNode` that consumers read — narrowing the query without narrowing the type is unsound. Introduce a dedicated lightweight tree-node type (or accept the wider type) first. *(v1.20.0 added the safe half — a 5,000-row bound — since the query was previously unbounded. The `select` itself still needs the type change.)*
- **Data-filter query pagination** — when `dataFilters` are present, `listNodes` runs a raw `SELECT "id"` with no `LIMIT`, materialising every matching id into an `IN (...)`. Pushing `LIMIT`/`OFFSET` into that raw query is wrong (it doesn't carry the other where-clauses or the `orderBy`); the correct fix folds the data filter into a single paged query. A `.max(50)` bound on `dataFilters` now caps filter *complexity* as an interim mitigation.

### Toast copy should use the node type's label, not "Node"

**Priority:** ✅ Resolved (v1.20.0) — connection-type labels remain

> Node toasts now name the type: created / updated / deleted / moved / duplicated / archived / restored read as "Client created", resolved through `nodeTypeRegistry` with an `Item` fallback for a type the registry hasn't synced yet. `deleteMutation` returns the type slug it already fetched for the delete veto, so no extra lookup was needed. **Still generic:** `useConnectionMutations` ("Connection created"/"Connection removed") could name the connection type, and `favoriteMutation`/`reorderMutation` fire no success toast at all — decide whether that silence is intentional. Original detail retained below.

The mutation hooks currently toast generic messages like `"Node created"`, `"Node updated"`, `"Node deleted"`, `"Node moved"`, etc. End users don't think in terms of "nodes" — they think in terms of Projects, Clients, Folders, Tags, whatever the app has configured. A toast saying `"Node updated"` is technically accurate but confusingly abstract.

The module already has the infrastructure to fix this:

- `nodeType.name` and `nodeType.namePlural` are available on every node (e.g. "Project", "Client")
- The `getLabel()` utility in `utils/labels.ts` supports admin-customised labels

The fix lives in `useNodeMutations` (in `src/templates/frontend/index.ts`'s `USE_NODE_MUTATIONS` constant). Each mutation already receives or can look up the node — thread the `nodeType.name` (or `getLabel('node.created', nodeType.name)`) through the toast message:

```ts
// Before
toast.success('Node created');

// After — needs the nodeType name on the mutation context
toast.success(\`\${nodeType.name} created\`);   // "Project created"
toast.success(\`\${nodeType.name} updated\`);   // "Client updated"
```

Same pattern applies to `useConnectionMutations` ("Connection" → could use the connection type label) and `useNodeTypeMutations` (admin-only — "Node type" is acceptable there since admins are the audience).

Approach options to consider:
1. **Pass `nodeType` into each mutation call** — consumer component provides it. Mutation shape changes, type-safe.
2. **Look up `nodeType` from the node result/context** — mutations resolve it from `nodeTypeRegistry.getConfig(node.nodeType)`. Call-site stays unchanged.
3. **Expose a `getToastLabel()` overridable via admin config** — gives each app full control over wording.

Option 2 is probably the least disruptive — no consumer changes, just richer toasts.

### Searchable Encrypted Fields

**Priority:** Low — depends on a starter-provided primitive

A future capability to let a node-type field be flagged `encrypted` (AES-256-GCM at rest, key never in the database) while still being **filterable within a single user's own data space**. Today an encrypted field is genuinely un-filterable and un-sortable: `buildDataFilterQuery` emits raw SQL against `data->>'field'`, and a non-deterministic GCM envelope is opaque to `=`, `ILIKE`, and range comparisons. This entry scopes how we will restore *limited* search without ever implying false security. Nothing here ships until the dependency below lands.

- **Dependency on the starter.** This will require a starter-provided **per-tenant blind-index primitive** — a keyed `HMAC(indexKey, normalize(value))` helper, with the index key derived *separately* from the encryption key. It does not exist yet. It also requires the starter to **actually derive per-tenant keys (HKDF)** first: the shipped envelope's `{ tenantId }` option is currently ignored by the env adapter, so until that is implemented every tenant shares one global key — building an index on that today would produce a single global tag space, not per-tenant isolation.
- **Per-field `encrypted` flag.** Admins will be able to flag a field encrypted; an additional opt-in will enable a blind-index sibling tag column (`dataIndex`) for that field. The blind index will be **restricted to high-entropy fields** — never booleans, status enums, country, or short codes, whose plaintext is recoverable from a deterministic tag via frequency analysis.
- **Exact-match / keyword only on encrypted fields.** A blind index will support `equals`, `in`, membership, dedupe, and whole-word keyword lookups only. **Range, sort, and substring (`contains`/`starts_with`) cannot use it** — these will fall back to **decrypt-in-memory** (the default, viable because per-user datasets are small and bounded) or be **disabled** in the filter UI for encrypted fields.
- **Honest leakage model.** A deterministic blind index *always* leaks the equality pattern and value frequency, and a per-tenant key only **limits cross-user correlation** — it does not remove within-user leakage. The admin UI will warn accordingly; decrypt-in-memory (which leaks nothing to the database) stays the recommended default, with the blind index reserved for the narrow case where dataset size forces SQL-side equality.
- **Safe backfill.** Adding the flag, rotating keys, or building the index will re-encrypt existing values, so the work must guarantee a fresh random IV per encryption (GCM nonce reuse is catastrophic) and maintain the tag transactionally alongside the ciphertext.
