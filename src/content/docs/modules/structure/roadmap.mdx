## Next Priority

### Read-Only Node Detail Page with Edit Mode

**Priority:** High — Blocks production use

The node detail page currently renders as an always-editable form. This needs to change to a read-only display by default with an Edit button to switch to form mode. Key requirements:

- **View mode** (default) — fields render as formatted text/badges, not inputs
- **Edit button** on the NodeDetailHeader to toggle edit mode
- **Field-level permissions** — `editableBy` controls which fields are editable (anyone/admin/system)
- **Node-level permissions** — who can create, edit, archive, delete (configurable per node type)
- **Built-in field permissions** — editableBy on Name, Description, Slug
- **System fields** (editableBy: 'system') always render as read-only, even in edit mode
- **NodeFieldRenderer** needs a `readOnly` render mode (text/badges instead of inputs)

### Conditional Colour Field Picker

**Priority:** High

The conditional colour admin UI currently uses a text input for the field ID. Needs:

- Combobox showing only select/multiselect fields (since colour rules map to option values)
- Auto-populate colour rules from the selected field's options
- Each option gets a colour picker
- Built-in field permissions (editableBy dropdown on Name/Description/Slug)

### Groups Tab Navigation Management

**Priority:** Medium

- Reorder groups via drag-and-drop
- Drag node types between groups
- Configure nav order from Groups tab directly

### Cross-Module Developer Experience

**Priority:** High

Improvements to make Structure easier and more powerful for other modules to integrate with. Identified from real-world integration work across modules like Builder, Tasks, and others.

#### NodeSelector Enhancements

- **Inline quick-create mode** — Built-in minimal creation form within the NodeSelector popover, so users can create a node without leaving the current dialog. Currently requires `onCreateClick` callback which forces the consuming module to implement its own creation flow
- **Dialog/modal variant** — Alternative `NodeSelector` rendered as a dialog instead of a popover, for use inside other dialogs where popover z-index and overflow issues arise
- **`onNodeCreated` callback** — Fire a callback after a node is created inline, with the new node auto-selected. Simplifies the consuming module's state management

#### Frontend API Surface

- **`useNodeName(id)` hook** — Lightweight hook that resolves a node ID to its display name without fetching the full node object. Useful for badges, breadcrumbs, and labels throughout consuming modules
- **`useNodesByType(type)` hook** — Fetch all nodes of a given type for the current user. Common pattern that currently requires manual service calls
- **Export `StructureService` type** — Typed interface for the structure service, so consuming modules can type their `serviceRegistry.get('structure')` calls without importing internals
- **`useNodeAncestors(id)` hook** — Resolve the full parent chain for breadcrumb rendering in consuming modules

#### Backend API Surface

- **Frontend `structureService.createNode()`** — A frontend service method for programmatic node creation (not just via the admin/detail page UI). Allows modules like Builder to create project nodes inline
- **Typed action/filter contexts** — Export TypeScript interfaces for all action hook contexts (e.g., `NodeCreatedContext`, `NodeDeletedContext`) so consuming modules get autocomplete and type safety in their registrations

#### General DX

- **Integration testing utilities** — Provide mock/stub helpers for `serviceRegistry.get('structure')` so module developers can unit test their Structure integrations without a running database
- **Node field value access helper** — Simple utility to extract a typed field value from a node's data blob by field slug, handling missing fields gracefully

## Known Issues

### Several built-in field types render as plain text

**Priority:** Medium

A v1.13.2 documentation-accuracy audit found that several field types declared in the `NodeTypeFieldType` union have no `case` in the `NodeFieldRenderer` switch and fall through to a plain text `<Input>`: **`media`** (no file uploader — `mediaConfig` is inert), **`chained-select`** (`chainedSelectConfig` inert), and the **`multiselect`** _edit_ control (read-only display renders the stored array as badges correctly, but the editor is a text box). Related inert field properties: **`inputMask`**, textarea **`size`**, and **`validation.pattern`** / text length (persisted by the admin editor but never enforced at render time — only `number` min/max are wired). The docs now describe these honestly; implementing the real renderers is tracked under Planned Improvements below.

The `NodeTypeForm` tabs rendering bug flagged during v1.9.7 / v1.9.8 turned out to be caused by a `@neondatabase/*` global CSS import in the host app leaking layout rules onto shadcn primitives — **not** a Structure issue. Resolution tracked in the [starter template repo](/starter-template/roadmap/) instead. The `tabs.tsx` overwrite via `npx shadcn@latest add tabs --overwrite` alone wasn't enough; the real cause is in `frontend/src/index.css` (or equivalent global stylesheet import chain).

### Field-group repeater renders literal `${...}` placeholders

**Priority:** Medium — found by the v1.15.2 standards-alignment audit

The `field-group` field renderer emits two **over-escaped** template expressions as JSX _text_, so a user editing a field-group (repeater) sees the literal strings `#${itemIdx + 1}` and `Add ${field.label}` instead of the item number and the field label. In `src/templates/frontend/components.ts` (`NODE_FIELD_RENDERER`, ~L1813 and ~L1851) the JSX reads `#\${itemIdx + 1}` and `Add \${field.label}` — these should be JSX expressions (`#{itemIdx + 1}` and `Add {field.label}`), not escaped template-literal interpolations. `npm run check:templates` only flags _un_-escaped `${`, so it does not catch this _over_-escape. Fix is a two-character change per site.

## Planned Improvements

### Structure-only styling compliance sweep (v1.15.2 standards-alignment audit)

**Priority:** Medium

An LLM-rules §5 ("structure-only styling") audit of the generated UI templates found ~209 forbidden typography classes plus a handful of colour/elevation literals. Template feature code must express **layout + state only** — type scale, weight, colour and elevation belong to the shadcn primitives, `globals.css`, or a Style Registry.

- **Typography scale (~157)** — `text-xs/sm/lg/xl` across `frontend/components.ts` (~26) and `frontend/pages.ts` (~131).
- **Font weight (~52)** — `font-medium/semibold/normal` (mostly `pages.ts`).
- **Hard colour literals (~6)** — `text-amber-600` soft-warning (`pages.ts:5103`); `text-yellow-500` favourite-star ×4 and `text-green-500` copied-check (`components.ts`). Move to semantic tokens (`text-muted-foreground` / `text-destructive`, or a success token).
- **Decorative shadows (3)** — `hover:shadow-md` ×2 and a drag-state `shadow-lg`.
- **`!important` + arbitrary value** — the NodeForm `SheetContent` width override `!w-1/3 !max-w-[33vw]` (`components.ts:630`) needs a non-`!` approach.

This sweep changes rendered output, so it must be **verified in a generated app**, not just built. Acceptable exceptions (leave as-is): idiomatic shadcn opacity idioms (`hover:bg-muted/50`, `bg-background/95 backdrop-blur`), and the ColorPicker's data-driven swatches / `PRESET_COLORS` hex array / inline `style={{ backgroundColor }}` (colours-as-data, not chrome).

### Node-type-group CRUD is missing action hooks

**Priority:** Low

`createNodeTypeGroup` / `updateNodeTypeGroup` / `deleteNodeTypeGroup` (`src/templates/backend/index.ts` ~L2207–2276) only `log.info` — they never fire `actionRegistry.doActionAll('structure.nodeTypeGroup.created' | '…updated' | '…deleted', …)`, unlike every sibling config CRUD (node, nodeType, connectionType, template dispatch ~28 `doActionAll` calls between them). Other modules can't react to group changes. Add the three hooks and document them in the action-registration hook table.

### Adopt scaffold v1.18.0 (per-module DB migration runner)

**Priority:** ✅ Resolved (v1.16.0) — one §-rule touch-up remaining

The scaffold is now at **v1.18.0** (module contract **v1.2.0**), and the per-module DB **migration runner** is no longer dormant: the `migrations` field is **in use**, shipping `1.16.0__cognitive-foundations.sql` (pgvector extension + HNSW index provisioning for the cognitive foundations). A consequence worth knowing: because the module carries a migration, `npx structure init`/`update` now need a reachable database via `backend/.env`.

Of the two §-rule touch-ups noted in the same audit, the missing `@@index` on the four soft-FK template columns of `structure_node_type` (`cardTemplateId`, `listItemTemplateId`, `detailHeaderTemplateId`, `overviewTemplateId`) is **done**. The other remains: pre-existing generated file headers still use the two-field `// FILE:` / `// PURPOSE:` block without the labelled `// DESCRIPTION:` third field (§1) — the new v1.16.0 templates carry it, so this is a mechanical sweep over the older headers for a future housekeeping pass.

### Deferred performance refinements (from the v1.13.3 health audit)

**Priority:** Low

Two performance findings were intentionally **not** auto-applied because the naive fix would regress correctness; each needs a careful, generated-app-verified implementation:

- **`getTree` column `select`** — `getTree` fetches full node rows (including the heavy `data`/`settings`/`metadata` JSON) just to build a tree from structural columns. A `select` would shrink the payload, but the method returns `NodeTreeItem.node` typed as a full `StructureNode` that consumers read — narrowing the query without narrowing the type is unsound. Introduce a dedicated lightweight tree-node type (or accept the wider type) first.
- **Data-filter query pagination** — when `dataFilters` are present, `listNodes` runs a raw `SELECT "id"` with no `LIMIT`, materialising every matching id into an `IN (...)`. Pushing `LIMIT`/`OFFSET` into that raw query is wrong (it doesn't carry the other where-clauses or the `orderBy`); the correct fix folds the data filter into a single paged query. A `.max(50)` bound on `dataFilters` now caps filter *complexity* as an interim mitigation.

### Implement the inert built-in field renderers

**Priority:** Medium

Surfaced by the v1.13.2 docs-accuracy audit. These field types are declared and selectable but not implemented in `NodeFieldRenderer` (they render as a plain text input):

- **`media`** — a real file-upload widget driven by `mediaConfig` (accept, maxSize, multiple). Currently falls through to a text input.
- **`chained-select`** — dependent cascading `<Select>`s driven by `chainedSelectConfig` (levels, options).
- **`multiselect`** (edit) — a multi-pick control. The read-only display already renders the stored array as badges; only the editor is missing.
- **`inputMask`** — enforce the mask (e.g. `AAA-999`) on `text` input.
- **textarea `size`** — map `small`/`medium`/`large` to row counts (currently hardcoded to 3 rows).
- **`validation.pattern` / length** — enforce regex and min/max length for `text` (currently stored but not applied; only `number` min/max are wired as HTML attributes).

Until implemented, modules needing these can register a custom field type via `nodeFieldTypeRegistry`.

### Toast copy should use the node type's label, not "Node"

**Priority:** Medium

The mutation hooks currently toast generic messages like `"Node created"`, `"Node updated"`, `"Node deleted"`, `"Node moved"`, etc. End users don't think in terms of "nodes" — they think in terms of Projects, Clients, Folders, Tags, whatever the app has configured. A toast saying `"Node updated"` is technically accurate but confusingly abstract.

The module already has the infrastructure to fix this:

- `nodeType.name` and `nodeType.namePlural` are available on every node (e.g. "Project", "Client")
- The `getLabel()` utility in `utils/labels.ts` supports admin-customised labels

The fix lives in `useNodeMutations` (in `src/templates/frontend/index.ts`'s `USE_NODE_MUTATIONS` constant). Each mutation already receives or can look up the node — thread the `nodeType.name` (or `getLabel('node.created', nodeType.name)`) through the toast message:

```ts
// Before
toast.success('Node created');

// After — needs the nodeType name on the mutation context
toast.success(\`\${nodeType.name} created\`);   // "Project created"
toast.success(\`\${nodeType.name} updated\`);   // "Client updated"
```

Same pattern applies to `useConnectionMutations` ("Connection" → could use the connection type label) and `useNodeTypeMutations` (admin-only — "Node type" is acceptable there since admins are the audience).

Approach options to consider:
1. **Pass `nodeType` into each mutation call** — consumer component provides it. Mutation shape changes, type-safe.
2. **Look up `nodeType` from the node result/context** — mutations resolve it from `nodeTypeRegistry.getConfig(node.nodeType)`. Call-site stays unchanged.
3. **Expose a `getToastLabel()` overridable via admin config** — gives each app full control over wording.

Option 2 is probably the least disruptive — no consumer changes, just richer toasts.

### Server-Side Field Filtering

**Priority:** Medium

Field filters on the list page are applied client-side after fetching. Should pass filter parameters to the API and apply in Prisma query.

### Reconcile scaffold reference templates with Structure-specific ones

**Priority:** ✅ Resolved

An earlier scaffold split added generic `my-feature` templates in `src/templates/backend/{routes,services,types}.ts` and `src/templates/frontend/{routes,services,hooks,types}.ts`. Structure keeps its real templates inline in the per-area `index.ts` files, so these were dead, unimported exports.

**Resolved:** the dead `my-feature` files (backend and frontend) have been removed, and the merge-noise concern no longer applies — scaffold **v1.12.0** treats `src/templates/**` and `src/module.ts` as **module-owned**, so `scaffold:update` never auto-merges upstream edits into them (it reports them as advisory via `scaffold:check`) and never re-introduces files you've deleted.

### Searchable Encrypted Fields

**Priority:** Low — depends on a starter-provided primitive

A future capability to let a node-type field be flagged `encrypted` (AES-256-GCM at rest, key never in the database) while still being **filterable within a single user's own data space**. Today an encrypted field is genuinely un-filterable and un-sortable: `buildDataFilterQuery` emits raw SQL against `data->>'field'`, and a non-deterministic GCM envelope is opaque to `=`, `ILIKE`, and range comparisons. This entry scopes how we will restore *limited* search without ever implying false security. Nothing here ships until the dependency below lands.

- **Dependency on the starter.** This will require a starter-provided **per-tenant blind-index primitive** — a keyed `HMAC(indexKey, normalize(value))` helper, with the index key derived *separately* from the encryption key. It does not exist yet. It also requires the starter to **actually derive per-tenant keys (HKDF)** first: the shipped envelope's `{ tenantId }` option is currently ignored by the env adapter, so until that is implemented every tenant shares one global key — building an index on that today would produce a single global tag space, not per-tenant isolation.
- **Per-field `encrypted` flag.** Admins will be able to flag a field encrypted; an additional opt-in will enable a blind-index sibling tag column (`dataIndex`) for that field. The blind index will be **restricted to high-entropy fields** — never booleans, status enums, country, or short codes, whose plaintext is recoverable from a deterministic tag via frequency analysis.
- **Exact-match / keyword only on encrypted fields.** A blind index will support `equals`, `in`, membership, dedupe, and whole-word keyword lookups only. **Range, sort, and substring (`contains`/`starts_with`) cannot use it** — these will fall back to **decrypt-in-memory** (the default, viable because per-user datasets are small and bounded) or be **disabled** in the filter UI for encrypted fields.
- **Honest leakage model.** A deterministic blind index *always* leaks the equality pattern and value frequency, and a per-tenant key only **limits cross-user correlation** — it does not remove within-user leakage. The admin UI will warn accordingly; decrypt-in-memory (which leaks nothing to the database) stays the recommended default, with the blind index reserved for the narrow case where dataset size forces SQL-side equality.
- **Safe backfill.** Adding the flag, rotating keys, or building the index will re-encrypt existing values, so the work must guarantee a fresh random IV per encryption (GCM nonce reuse is catastrophic) and maintain the tag transactionally alongside the ciphertext.

## Contributing Ideas

Have a feature request or improvement idea?

1. **Check existing items** - Review this roadmap and the [module repository](https://github.com/ui-git-kim/structure)
2. **Open an issue** - Describe the feature and use cases
3. **Discuss** - Engage with feedback to refine the proposal

<Aside type="tip">
When proposing features, consider how they integrate with the registry system. Features that other modules can extend through registries are preferred.
</Aside>
