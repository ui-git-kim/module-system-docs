### Cross-Module Developer Experience

**Priority:** High

Improvements to make Structure easier and more powerful for other modules to integrate with. Identified from real-world integration work across modules like Builder, Tasks, and others.

#### NodeSelector Enhancements

- **Inline quick-create mode** — Built-in minimal creation form within the NodeSelector popover, so users can create a node without leaving the current dialog. Currently requires `onCreateClick` callback which forces the consuming module to implement its own creation flow
- **Dialog/modal variant** — Alternative `NodeSelector` rendered as a dialog instead of a popover, for use inside other dialogs where popover z-index and overflow issues arise
- **`onNodeCreated` callback** — Fire a callback after a node is created inline, with the new node auto-selected. Simplifies the consuming module's state management
- **Multi-select mode** — Allow selecting multiple nodes at once (e.g., linking a task to several projects). Return `string[]` instead of `string | null`

#### Frontend API Surface

- **`useNodeName(id)` hook** — Lightweight hook that resolves a node ID to its display name without fetching the full node object. Useful for badges, breadcrumbs, and labels throughout consuming modules
- **`useNodesByType(type)` hook** — Fetch all nodes of a given type for the current user. Common pattern that currently requires manual service calls
- **Export `StructureService` type** — Typed interface for the structure service, so consuming modules can type their `serviceRegistry.get('structure')` calls without importing internals
- **`useNodeAncestors(id)` hook** — Resolve the full parent chain for breadcrumb rendering in consuming modules

#### Backend API Surface

- **Frontend `structureService.createNode()`** — A frontend service method for programmatic node creation (not just via the admin/detail page UI). Allows modules like Builder to create project nodes inline
- **Typed action/filter contexts** — Export TypeScript interfaces for all action hook contexts (e.g., `NodeCreatedContext`, `NodeDeletedContext`) so consuming modules get autocomplete and type safety in their registrations

#### General DX

- **Integration testing utilities** — Provide mock/stub helpers for `serviceRegistry.get('structure')` so module developers can unit test their Structure integrations without a running database
- **Node field value access helper** — Simple utility to extract a typed field value from a node's data blob by field slug, handling missing fields gracefully

## Known Issues

### Documentation interface reference is inaccurate

**Priority:** High

Comparing `configuration.mdx` against the actual TypeScript interfaces in `src/templates/frontend/index.ts` uncovered substantive drift:

- **`NodeTypeFeatures`** (configuration.mdx:259-268) documents seven `can*` fields (`canFavorite`, `canArchive`, `canPublish`, `canManualSort`, `customSlug`, `customColor`, `customIcon`). The real interface has eight un-prefixed fields: `slug`, `description`, `color`, `icon`, `favorites`, `archiving`, `publishing`, `manualSort`.
- **`NodeTypeRules`** (configuration.mdx:246-254) invents `canBeRoot`, `canNestSameType`, `canContainSameType` — none of these exist. The real interface has `isRoot`, `maxChildren`, `isHierarchical`, `allowMultipleParents`, `isConnection`, `connectionConfig` that aren't documented.
- **`NodeTypeNavigation.sidebarNav.style`** documented as `'list' | 'tree'` — actual is `'none' | 'list' | 'tree'`. Missing sub-fields `parentType`, `maxDepth`, `showEmpty`, `collapsible`, `defaultExpanded`.
- **`NodeTypeConfig`** marks `fields`, `icon`, `isSystem`, `sortOrder` as required; actual interface has them all optional. `moduleId` is typed `string`, actual is `string | null`.
- **`NodeTypeNavigation.pageNav.builtInTabs`** shows each key as `boolean`; actual is `boolean | { label?; widgetLayout? }`. Missing the sibling `tabOrder` and `hideTabs` arrays.
- **`structure_settings`** table (configuration.mdx:209-213) omits `createdAt` and `updatedAt` columns.

Developers following these docs will write code that fails the type-checker. Fix: rewrite the affected blocks to match the interfaces in `frontend/index.ts:180-330` and the schema in `schema.ts`.

### `NodeSelector` example in usage.mdx is wrong

**Priority:** High

`usage.mdx:193-205` shows the primary integration example using `nodeTypes={['client']}` and an `onChange` that receives a node object. The actual props are `allowedTypes?: string[]` and `onChange: (value: string | null) => void` — `onChange` receives the selected **ID string**, not a node. Copy-pasted code won't compile.

`integration.mdx:329-336` has the correct example; `usage.mdx` needs to match it.

### `dependencies: ['structure']` in module.ts is invalid

**Priority:** High

`integration.mdx:84` and `:901` both tell module authors to declare the Structure dependency as `dependencies: ['structure']`. That field doesn't exist at the top level of `ModuleDefinition` — the correct field is `requiredModules?: string[]`. Top-level `dependencies` arrays only live inside `frontend` / `backend` and are for npm packages.

Fix: global find-and-replace `dependencies: ['structure']` → `requiredModules: ['structure']` in `integration.mdx` and the Integration Checklist.

### Outdated hook signatures in usage.mdx

**Priority:** Medium

- `useNodeTree(rootId)` documented — actual signature is `useNodeTree(nodeType?: string)`.
- `useNodeMutations()` described as "Create, update, and delete with optimistic updates" — since v1.9.1 it uses cache invalidation, not optimistic updates. Description should be updated.
- `useNodeConnections` listed under "Navigation Hooks" — it's a data hook.
- Missing from the hook tables: `useNodePath`, `useNodeChildren`, `useConnectionMutations` (v1.9.2), `useNodeTypeMutations` (v1.9.2).

### NodeTypeRegistry API reference is misleading

**Priority:** Medium

`integration.mdx:549-556` documents `get(slug)` as "Get a single node type by slug" and `getAll()` as "Get all registered node types". Both are misleading — `get` returns a `NodeTypeRegistration` wrapper, `getAll` returns an array of wrappers. The methods that return bare configs are `getConfig(slug)` and `getAllTypes()`. Readers expecting `get` to return a config will get a wrapper and have to dig for `.config`.

### Invalid init example in integration.mdx

**Priority:** Medium

`integration.mdx:718-761` shows `structureService.listNodeTypes('system', true)` and `structureService.createNodeType('system', {...})`. Both methods actually take `userId: string` as the first argument. Passing a literal `'system'` string would not match any real user and will fail at the FK check. Replace with the real pattern (accept a system user ID as an init parameter, or document that `moduleId` scoping is the supported mechanism for module-owned types).

### Registry count is inconsistent across pages

**Priority:** Low

`index.mdx` says "9 extensible registries"; `usage.mdx` says "8 registry types"; `integration.mdx` says "8 frontend registries". Truth: 9 module-specific registries total — 8 content registries (Tabs, Widgets, Card Extensions, Actions, Columns, Filters, Sidebar Sections, Field Types) visible in the Registries admin tab, plus `nodeTypeRegistry` which is DB-synced and has its own Structure Types admin tab. Pages should reference this explanation consistently.

### Stale comment in INDEX_TEMPLATE

**Priority:** Low

`src/templates/frontend/registrations.ts:479` has a code comment reading "Navigation: adds items to sidebar (auto-fetches node types at import)". The auto-fetch was removed in v1.9.2. Template change, not a docs-site change, but ships to every consuming app.

## Planned Improvements

### Toast copy should use the node type's label, not "Node"

**Priority:** Medium

The mutation hooks currently toast generic messages like `"Node created"`, `"Node updated"`, `"Node deleted"`, `"Node moved"`, etc. End users don't think in terms of "nodes" — they think in terms of Projects, Clients, Folders, Tags, whatever the app has configured. A toast saying `"Node updated"` is technically accurate but confusingly abstract.

The module already has the infrastructure to fix this:

- `nodeType.name` and `nodeType.namePlural` are available on every node (e.g. "Project", "Client")
- The `getLabel()` utility in `utils/labels.ts` supports admin-customised labels

The fix lives in `useNodeMutations` (in `src/templates/frontend/index.ts`'s `USE_NODE_MUTATIONS` constant). Each mutation already receives or can look up the node — thread the `nodeType.name` (or `getLabel('node.created', nodeType.name)`) through the toast message:

```ts
// Before
toast.success('Node created');

// After — needs the nodeType name on the mutation context
toast.success(\`\${nodeType.name} created\`);   // "Project created"
toast.success(\`\${nodeType.name} updated\`);   // "Client updated"
```

Same pattern applies to `useConnectionMutations` ("Connection" → could use the connection type label) and `useNodeTypeMutations` (admin-only — "Node type" is acceptable there since admins are the audience).

Approach options to consider:
1. **Pass `nodeType` into each mutation call** — consumer component provides it. Mutation shape changes, type-safe.
2. **Look up `nodeType` from the node result/context** — mutations resolve it from `nodeTypeRegistry.getConfig(node.nodeType)`. Call-site stays unchanged.
3. **Expose a `getToastLabel()` overridable via admin config** — gives each app full control over wording.

Option 2 is probably the least disruptive — no consumer changes, just richer toasts.

### Server-Side Field Filtering

**Priority:** Medium

Field filters on the list page are applied client-side after fetching. Should pass filter parameters to the API and apply in Prisma query.

### Complete the CLI reference in docs

**Priority:** Low

`index.mdx:187-194` and `configuration.mdx:468-482` list a subset of subcommands and flags. Missing:

- Subcommands: `migrate`, `disable`, `enable` (all in `cli.ts`)
- `init --no-interactive`
- `update --force`, `update --dry-run`
- `uninstall --keep-data`, `uninstall --export <path>`

Add a complete CLI reference table pulled from `src/cli.ts`.

### Document post-install routing + system-level seeding

**Priority:** Low

`configuration.mdx`'s Post-Installation steps (73-86) show `import { initStructure }` and `...structureRoutes` but don't explain:

- `initStructure()` is only generated when `hasInit: true` in module.ts
- The routes use catch-all patterns — must be spread at the END of the route table so more specific app routes win
- Default node types are seeded with `userId: null` as system types; modules registering types pass a real userId but tag with `moduleId`

These are the gotchas that trip up new integrations.

### Document schema details that SQL consumers need

**Priority:** Low

`configuration.mdx` table blocks don't mention:

- Actual table names are plural via `@@map` (`structure_node` model → `structure_nodes` table)
- Unique constraints on `structure_node`: `(userId, slug)` and `(userId, parentId, name)` — these affect API error responses
- Indexes that power performance

Add a short "Raw SQL access notes" section.

### Import snippets for all registry interfaces

**Priority:** Low

`integration.mdx`'s 8 registry reference tables document properties but don't show the interface name (`NodeTabRegistration`, `NodeActionRegistration`, etc.). Adding a one-line import snippet above each table lets IDE users peek the full type.

### Reconcile scaffold reference templates with Structure-specific ones

**Priority:** Low

The scaffold v1.9.0 merge added generic `my-feature` templates in `src/templates/backend/{routes,services,types,registrations}.ts` and `src/templates/frontend/{routes,services,types,hooks}.ts`. Structure uses its own Structure-specific `index.ts` files instead, so most of these new files are dead exports. They'll cause noise in future `npm run scaffold:update` merges. Decide whether to keep them as scaffold reference or drop them and accept the merge divergence.

### styleRegistry Contribution (deferred — handled by Style module)

**Priority:** Deferred

Node-type colours, tree-view indent tokens, and card accent colours could be exposed as contributable design tokens via the scaffold v1.6.0 `styleRegistry`. Deferred: theming is the responsibility of a dedicated Style module, not Structure.

## Under Consideration

Ideas being evaluated for feasibility and demand.

### Node Versioning

Track changes to node data over time:

- Version history for field changes
- Diff view between versions
- Restore to previous version
- Audit trail per node

### Computed Fields

Fields that derive their value from other fields or related data:

- Formula fields (concatenation, arithmetic)
- Aggregation fields (count children, sum values)
- Lookup fields (pull from connected nodes)
- Auto-refresh on dependency changes

### Import/Export

Bulk data management:

- CSV import for node creation
- CSV/JSON export of node lists
- Field mapping during import
- Validation and error reporting

### Node Permissions

Granular access control per node:

- Owner, editor, viewer roles per node
- Inherit permissions from parent
- Share individual nodes with specific users
- Permission checks in API and UI

### Workflow Automation

Trigger actions based on node state changes:

- Status change triggers (e.g., archived → notify)
- Field value triggers (e.g., due date approaching)
- Integration with action registry
- Configurable via admin panel
