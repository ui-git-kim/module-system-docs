Structure ships with **21 built-in field types** that cover text, dates, selections, composites, references, and special-purpose inputs. Every field type shares a common set of [field settings](#field-settings) and can optionally use [merge templates](/modules/structure/merge-fields/) and [conditional visibility](/modules/structure/merge-fields/#conditional-visibility).

---

## Text Fields

### text

Single-line text input. The most common field type.

Supports `placeholder`. `validation.pattern` and `validation.min`/`max` (as min/max **length**) are enforced on the input as of v1.20.0, with `validation.message` surfaced as the title. `inputMask` remains a planned property that is not yet wired up тАФ see [Field Settings](#field-settings).

```json
{
  "id": "company_name",
  "type": "text",
  "label": "Company Name",
  "placeholder": "Enter company name",
  "required": true,
  "validation": {
    "pattern": "^[A-Za-z0-9 ]+$",
    "message": "Only letters, numbers, and spaces allowed"
  }
}
```

Renders as a standard `<input type="text">`.

### textarea

Multi-line text input for longer content.

The `size` setting maps to height as of v1.20.0: `small` = 2 rows, `medium` (default) = 4, `large` = 8. `validation.min`/`max` apply as min/max length.

```json
{
  "id": "notes",
  "type": "textarea",
  "label": "Notes",
  "placeholder": "Additional details...",
  "width": "full"
}
```

Renders as a `<textarea>` element.

### number

Numeric input with optional min/max validation.

```json
{
  "id": "estimated_hours",
  "type": "number",
  "label": "Estimated Hours",
  "placeholder": "0",
  "validation": {
    "min": 0,
    "max": 9999,
    "message": "Must be between 0 and 9999"
  }
}
```

Renders as `<input type="number">` with `min` and `max` attributes set from validation.

### email

Email address input with built-in browser validation.

```json
{
  "id": "contact_email",
  "type": "email",
  "label": "Contact Email",
  "placeholder": "name@example.com",
  "required": true
}
```

Renders as `<input type="email">`.

### url

URL input field.

```json
{
  "id": "website",
  "type": "url",
  "label": "Website",
  "placeholder": "https://example.com"
}
```

Renders as `<input type="url">`.

### phone

Telephone input with optional phone configuration.

**phoneConfig options:**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `format` | `'international' \| 'national' \| 'any'` | `'any'` | Expected phone format |
| `defaultCountryCode` | `string` | тАФ | Default country code (e.g. `'AU'`, `'US'`) |

```json
{
  "id": "mobile",
  "type": "phone",
  "label": "Mobile",
  "phoneConfig": {
    "format": "international",
    "defaultCountryCode": "AU"
  }
}
```

Renders as `<input type="tel">`. The placeholder auto-adjusts based on `format` тАФ international shows `+61 400 000 000`, national shows `0400 000 000`.

## Date & Time

### date

Date picker for selecting a calendar date.

```json
{
  "id": "start_date",
  "type": "date",
  "label": "Start Date",
  "required": true
}
```

Renders as `<input type="date">`. Stores the value as an ISO date string (e.g. `2025-03-15`).

### datetime

Combined date and time picker.

```json
{
  "id": "appointment",
  "type": "datetime",
  "label": "Appointment",
  "required": true
}
```

Renders as `<input type="datetime-local">`. Stores the value as an ISO datetime string (e.g. `2025-03-15T14:30`).

### time

Standalone time picker for selecting a time without a date.

```json
{
  "id": "office_hours_start",
  "type": "time",
  "label": "Office Hours Start"
}
```

Renders as `<input type="time">`. Stores the value as `HH:MM` (e.g. `09:00`).

## Selection

### select

Single-select dropdown with a defined list of options.

Each option is an object with `value` (stored in data) and `label` (displayed to user).

```json
{
  "id": "priority",
  "type": "select",
  "label": "Priority",
  "placeholder": "Select priority...",
  "options": [
    { "value": "low", "label": "Low" },
    { "value": "medium", "label": "Medium" },
    { "value": "high", "label": "High" },
    { "value": "critical", "label": "Critical" }
  ],
  "defaultValue": "medium"
}
```

Renders as a standard shadcn `<Select>` dropdown (not searchable).

<Aside type="note">
Only `select` field types can be used as filterable fields on list pages. The list page renders a dropdown filter for each filterable field.
</Aside>

### multiselect

Multi-select allowing the user to choose multiple values from an options list.

Implemented in v1.20.0 as a toggle-button group тАФ each option is a button that toggles in and out of the selection, with the chosen options shown as badges in read-only mode. A legacy scalar value is tolerated by coercing it to a single-entry array on read.

```json
{
  "id": "tags",
  "type": "multiselect",
  "label": "Categories",
  "options": [
    { "value": "litigation", "label": "Litigation" },
    { "value": "property", "label": "Property" },
    { "value": "family", "label": "Family Law" },
    { "value": "commercial", "label": "Commercial" }
  ]
}
```

Stores the value as an array of selected option values (e.g. `["litigation", "family"]`).

### chained-select

Cascading (dependent) selects where each level's options depend on the previous selection, configured with `chainedSelectConfig`.

Implemented in v1.20.0: each level is disabled until its parent is chosen, and picking a level clears any deeper selections. The value is stored as an array of selections, one entry per level, and displays read-only as a breadcrumb.

<Aside type="note">
There is no admin editor for `chainedSelectConfig` yet тАФ configure it via JSON import (see the example below) until one ships.
</Aside>

**chainedSelectConfig:**

| Property | Type | Description |
|----------|------|-------------|
| `levels` | `Array<{ label, placeholder? }>` | One entry per select in the cascade |
| `options` | `Record<string, Option[]>` | Hierarchical option tree тАФ each option can have `children` |

```json
{
  "id": "location",
  "type": "chained-select",
  "label": "Location",
  "chainedSelectConfig": {
    "levels": [
      { "label": "State", "placeholder": "Select state..." },
      { "label": "City", "placeholder": "Select city..." }
    ],
    "options": {
      "root": [
        {
          "value": "nsw",
          "label": "New South Wales",
          "children": {
            "nsw": [
              { "value": "sydney", "label": "Sydney" },
              { "value": "newcastle", "label": "Newcastle" }
            ]
          }
        },
        {
          "value": "vic",
          "label": "Victoria",
          "children": {
            "vic": [
              { "value": "melbourne", "label": "Melbourne" },
              { "value": "geelong", "label": "Geelong" }
            ]
          }
        }
      ]
    }
  }
}
```

Implemented as dependent selects in v1.20.0 тАФ each level is disabled until its parent is chosen, picking a level clears deeper selections, and the value stores as an array of selections (one per level), displaying read-only as a breadcrumb. There is no admin editor for `chainedSelectConfig` yet тАФ configure it via JSON import as shown above.

## Toggle

### toggle

Switch component for on/off boolean values.

```json
{
  "id": "is_active",
  "type": "toggle",
  "label": "Active",
  "defaultValue": true
}
```

Renders as a shadcn `<Switch>` component inline with the label.

### boolean

Yes/No switch. Functionally identical to `toggle` but with clearer semantics when the field represents a yes/no question rather than an on/off state.

```json
{
  "id": "has_dependants",
  "type": "boolean",
  "label": "Has Dependants?",
  "defaultValue": false
}
```

Renders as a shadcn `<Switch>` component inline with the label, same as toggle.

<Aside type="tip">
Use `toggle` for on/off states (e.g. "Active", "Enabled") and `boolean` for yes/no questions (e.g. "Has Children?", "Is Australian Citizen?"). Both store a boolean value тАФ the distinction is semantic.
</Aside>

## Composite

### address

Composite address field with configurable sub-fields. Stores value as an object with `street1`, `street2`, `city`, `state`, `postcode`, and `country` keys.

**addressConfig:**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `showStreet2` | `boolean` | `true` | Show the second street address line |
| `showState` | `boolean` | `true` | Show the state/region field |
| `showCountry` | `boolean` | `true` | Show the country field |
| `defaultCountry` | `string` | тАФ | Pre-fill the country (e.g. `'Australia'`) |

```json
{
  "id": "office_address",
  "type": "address",
  "label": "Office Address",
  "required": true,
  "width": "full",
  "addressConfig": {
    "showStreet2": true,
    "showState": true,
    "showCountry": true,
    "defaultCountry": "Australia"
  }
}
```

Renders as a stacked group of inputs:
- Street Address (full width)
- Street Address Line 2 (full width, if enabled)
- City + State (two columns)
- Postcode + Country (two columns)

**Stored value:**

```json
{
  "street1": "123 Pitt Street",
  "street2": "Level 4",
  "city": "Sydney",
  "state": "NSW",
  "postcode": "2000",
  "country": "Australia"
}
```

### name-field

Composite name field with configurable sub-fields for prefix, first, middle, last, and suffix. Stores value as an object.

**nameConfig:**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `showPrefix` | `boolean` | `false` | Show the prefix/title field |
| `showMiddle` | `boolean` | `false` | Show the middle name field |
| `showSuffix` | `boolean` | `false` | Show the suffix field |
| `prefixOptions` | `string[]` | тАФ | Dropdown options for prefix (e.g. `['Mr', 'Mrs', 'Ms', 'Dr']`). If empty, renders as a text input |
| `suffixOptions` | `string[]` | тАФ | Dropdown options for suffix |

```json
{
  "id": "full_name",
  "type": "name-field",
  "label": "Full Name",
  "required": true,
  "nameConfig": {
    "showPrefix": true,
    "showMiddle": true,
    "showSuffix": false,
    "prefixOptions": ["Mr", "Mrs", "Ms", "Miss", "Dr", "Prof"]
  }
}
```

Renders as:
- Prefix (narrow dropdown or text input, if enabled)
- First Name + Last Name (two columns)
- Middle Name (full width, if enabled)
- Suffix (full width, if enabled)

**Stored value:**

```json
{
  "prefix": "Dr",
  "first": "Kim",
  "middle": "James",
  "last": "Schulz"
}
```

### field-group

Repeatable group of sub-fields. Each group entry is a row containing all defined sub-fields. Users can add and remove rows.

The `fields` array on the field definition contains the sub-field schema тАФ each sub-field uses the same `NodeTypeField` interface (excluding `field-group` itself to prevent nesting).

```json
{
  "id": "key_people",
  "type": "field-group",
  "label": "Key People",
  "width": "full",
  "fields": [
    {
      "id": "role",
      "type": "select",
      "label": "Role",
      "options": [
        { "value": "Applicant", "label": "Applicant" },
        { "value": "Respondent", "label": "Respondent" },
        { "value": "Solicitor", "label": "Solicitor" }
      ],
      "required": true
    },
    {
      "id": "person",
      "type": "node-reference",
      "label": "Person",
      "nodeReferenceConfig": {
        "allowedTypes": ["person"]
      }
    },
    {
      "id": "notes",
      "type": "text",
      "label": "Notes"
    }
  ]
}
```

Renders as a list of `<Card>` components, each containing the sub-fields. An "Add" button appends a new empty row. Each row has a remove button.

**Stored value:**

```json
[
  { "role": "Applicant", "person": "node_abc123", "notes": "" },
  { "role": "Respondent", "person": "node_def456", "notes": "Self-represented" }
]
```

<Aside type="tip">
Field groups are powerful when combined with [merge templates](/modules/structure/merge-fields/). You can pull values from specific repeater rows using filtered lookups like `{key_people[role=Applicant].person->surname}`.
</Aside>

## Reference

### node-reference

Picks one or more nodes from the system. Uses the `NodeSelector` component with optional type filtering.

**nodeReferenceConfig:**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `allowedTypes` | `string[]` | тАФ | Restrict to specific node type slugs. Omit to allow any type |
| `multiple` | `boolean` | `false` | Allow selecting multiple nodes |

<Tabs>
  <TabItem label="Single reference">
```json
{
  "id": "assigned_client",
  "type": "node-reference",
  "label": "Client",
  "required": true,
  "nodeReferenceConfig": {
    "allowedTypes": ["client"]
  }
}
```

Stores a single node ID string: `"clu1abc..."`.
  </TabItem>
  <TabItem label="Multi reference">
```json
{
  "id": "related_matters",
  "type": "node-reference",
  "label": "Related Matters",
  "nodeReferenceConfig": {
    "allowedTypes": ["matter"],
    "multiple": true
  }
}
```

Stores an array of node ID strings: `["clu1abc...", "clu2def..."]`.
  </TabItem>
</Tabs>

Renders as a `<NodeSelector>` component тАФ a searchable dropdown that fetches nodes from the API. When `multiple` is true, each selected node appears as a row with its own selector and a remove button, plus an "Add" button to append more.

## Special

### hidden

Stored in the node's `data` but not rendered on user-facing forms. Useful for system-managed values, module-injected fields, or fields whose value is set programmatically.

```json
{
  "id": "external_id",
  "type": "hidden",
  "label": "External ID",
  "defaultValue": ""
}
```

In the admin form editor, hidden fields show a disabled text input. On user-facing forms, they render nothing.

<Aside type="note">
The `hidden` type is distinct from the `isHidden` flag. Any field type can be hidden from forms by setting `isHidden: true` тАФ the field still stores data but is not rendered. The `hidden` type is specifically for fields that should never have a visible input.
</Aside>

### section-break

Visual divider used to group fields visually. Does not store any data тАФ it exists purely for layout.

Has a label (shown as a heading) and optional description (shown as helper text below the heading). Always renders full-width across both columns.

```json
{
  "id": "section_contact",
  "type": "section-break",
  "label": "Contact Information",
  "description": "How this person can be reached"
}
```

Renders as a horizontal rule with the label as a `<h3>` heading and the description as muted text below.

### secret

Password/API-key field with generate, reveal/hide, and copy-to-clipboard functionality. Configured with `secretConfig`.

**secretConfig:**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `length` | `number` | `32` | Length of the generated secret |
| `prefix` | `string` | `''` | Prefix prepended to generated secrets (e.g. `'sk_'`, `'pk_'`) |
| `charset` | `'alphanumeric' \| 'hex' \| 'base64'` | `'alphanumeric'` | Character set for generation |

```json
{
  "id": "api_key",
  "type": "secret",
  "label": "API Key",
  "placeholder": "Click generate to create a key",
  "secretConfig": {
    "length": 40,
    "prefix": "sk_live_",
    "charset": "alphanumeric"
  }
}
```

Renders as a masked `<input type="password">` with three action buttons:
- **Reveal/hide** тАФ toggle visibility of the secret value
- **Generate** тАФ create a new random secret using the configured charset and length
- **Copy** тАФ copy the current value to clipboard (shows a checkmark on success)

<Aside type="note" title="Encrypted at rest (v1.19.0)">
Values of `secret` fields are **encrypted before they are stored**, using the starter's
`encryptField` (AES-256-GCM, key from env/KMS, never in the database тАФ see
[Security Feature](/starter-template/security-feature/)). The stored shape is
`{ "__sealed": "<envelope>" }`, so a sealed value is unambiguously detectable.

- **Decryption is narrow**: only the owner's node **detail** read (`GET /nodes/:id`) returns the
  plaintext. List and tree responses deliberately keep the sealed envelope.
- **Legacy values migrate lazily**: a plaintext value written before v1.19.0 keeps working and is
  sealed on its next write тАФ no bulk migration step.
- **Inherited fields count**: secret detection walks the `superClasses` hierarchy, so a `secret`
  field declared on a parent type is sealed on the child too.
- A tampered or key-rotated envelope decrypts to an empty string rather than leaking the envelope.
</Aside>

### media тАФ provided by a storage module

**Structure has no built-in `media` field type.** Structure owns organisational structure, not file storage тАФ per its [scope boundaries](/modules/structure/#scope-boundaries), file attachments belong to a content/document module.

A storage module registers `media` through [`nodeFieldTypeRegistry`](/modules/structure/registries-guide/#6-nodefieldtyperegistry), supplying the uploader, the read-only display and its own `configSchema` (accepted types, max size, multiple). Where no such module is installed, `media` simply does not appear in the admin field-type picker.

<Aside type="caution" title="Changed in v1.21.0">
Earlier versions shipped a built-in `media` entry whose renderer fell through to a plain text `<Input>` тАФ so editing one could overwrite a stored array or object with a string. The built-in type, its `mediaConfig`, and the short-lived `nodeMediaUploadRegistry` (v1.20.0 only) have all been removed in favour of a single mechanism: the field-type registry.

If a node type still references `type: "media"` with no storage module installed, the field renders **read-only with an explanation and its stored value preserved** тАФ it is never bound to an editable text input.
</Aside>

## Field Settings

These settings apply to **all field types**. They control identity, display, layout, validation, and advanced behaviours.

### Identity & Type

| Setting | Type | Description |
|---------|------|-------------|
| `id` | `string` | **Required.** Unique identifier in `snake_case`. Used as the key in the node's `data` JSON object |
| `type` | `string` | **Required.** One of the 21 built-in field types listed above, or a custom type registered via `nodeFieldTypeRegistry` |
| `label` | `string` | **Required.** Display label shown beside the input |

### Help Text

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `description` | `string` | тАФ | Help text shown near the input |
| `descriptionPlacement` | `'above' \| 'below'` | `'below'` | Whether the description appears above or below the input |
| `placeholder` | `string` | тАФ | Placeholder text inside the input |

### Value & Validation

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `required` | `boolean` | `false` | Whether the field must have a value |
| `defaultValue` | `unknown` | тАФ | Default value when creating a new node |
| `validation` | `object` | тАФ | Validation rules (see below) |

**Validation object:**

| Property | Type | Description |
|----------|------|-------------|
| `min` | `number` | Minimum value. Applied only to the `number` input (as the HTML `min` attribute) by the built-in renderer; no length enforcement is wired for text |
| `max` | `number` | Maximum value. Applied only to the `number` input (as the HTML `max` attribute) by the built-in renderer; no length enforcement is wired for text |
| `pattern` | `string` | Regular expression pattern. Stored in the admin editor but not enforced at render time by the built-in renderer |
| `message` | `string` | Custom error message shown when validation fails |

### Layout

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `order` | `number` | тАФ | Sort order within the form. Lower numbers appear first |
| `width` | `'full' \| 'half'` | `'half'` | Field width. `full` spans both columns; `half` takes one column |
| `section` | `string` | тАФ | Group fields visually by assigning the same section name |
| `size` | `'small' \| 'medium' \| 'large'` | `'medium'` | For `textarea`, maps to height (as of v1.20.0): `small` = 2 rows, `medium` = 4, `large` = 8. No effect on other field types |
| `cssClass` | `string` | тАФ | Custom CSS class applied to the field wrapper element |

### Admin

| Setting | Type | Description |
|---------|------|-------------|
| `adminLabel` | `string` | Internal admin-only label. Not shown to end users тАФ used for reference in the admin panel |

### Visibility & Behaviour

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `isHidden` | `boolean` | `false` | Hide the field from user-facing forms. The field still stores data in the node's `data` object, but the input is not rendered. Works on any field type |
| `editableBy` | `'anyone' \| 'admin' \| 'system'` | `'anyone'` | Per-field access control. `anyone` lets any user edit; `admin` restricts editing to admins; `system` makes the field API/module-only and read-only in forms. Fields with a `computedTemplate` become `system` automatically. **Enforced server-side as of v1.19.0** тАФ see below |
| `inputMask` | `string` | тАФ | Planned property for an input mask pattern (e.g. `'(999) 999-9999'`). Not yet wired up тАФ the built-in renderer does not read it, so it currently has no effect on what users can type |
| `computedTemplate` | `string` | тАФ | [Merge template](/modules/structure/merge-fields/) that auto-computes this field's value from other fields. When set, the field becomes read-only |
| `visibleWhen` | `object` | тАФ | [Conditional visibility](/modules/structure/merge-fields/#conditional-visibility) rules that show or hide this field based on other field values |

<Aside type="tip">
Use `section` to group related fields together visually, and `section-break` fields to add labelled dividers between groups. Combine `width: 'half'` fields to lay out two fields side by side in the form.
</Aside>

<Aside type="caution" title="editableBy is enforced server-side (v1.19.0)">
`PATCH /nodes/:id` now rejects writes to locked fields with **400**, naming the offending field тАФ
`system` fields are refused for everyone, `admin` fields for non-admins. Before v1.19.0 the lock was
**browser-only**: the form hid the input, but a direct API call wrote the value happily.

Two consequences for integrators:
- A module or script that writes a `system` field through the node API must move to a path that
  legitimately owns it (a service call, or an `admin` lock plus an admin caller).
- **Public API-key callers are never treated as admins**, so a key cannot write an `admin`-locked
  field regardless of who owns the key.
</Aside>
