Roadmap for the [cognitive-db engine](/modules/cog-ingest/engine/) that powers Cog Ingest. For the module (UI, sources, jobs) see the [Cog Ingest Roadmap](/modules/cog-ingest/roadmap/).

<Aside type="note">
The engine is **tenantId === userId** by design (solo-business focus). Anything that would add teams/organisations or a separate tenant layer is deliberately out of scope — see below.
</Aside>

---

## Intentionally out of scope

### Multi-tenant / teams isolation

**Status:** Won't do (by design)

The engine owns every row by `userId` and treats one user as one tenant. Audit items that asked to thread a separate tenant identity through validation (e.g. `ONT-01`) were deliberately **not** implemented — ownership isolation via `userId` is the model, and it maps directly onto the starter's security foundation. Adding an org/team layer would be a different product.

## Planned — needs product direction

These are real, known gaps left as documented stubs during the 2026-07-14 audit remediation (rather than implemented speculatively). Each needs a product decision before building.

### Discovery Tier 2 / Tier 3

**Priority:** High

`discovery.service` currently returns Tier 1 results only; the semantic **entity-matching** and **potential-contradiction** tiers are stubs (`entityMatches` / `potentialContradictions` always empty). Building these means defining how aggressively the engine should cross-reference and flag conflicts.

### Budget persistence

**Priority:** Medium

Token/budget tracking (`resources/budget.service`) is currently an in-process stub — per-user budgets and operation-scope alert thresholds are computed but never persisted or invalidated. Needs a storage model and a policy for what happens when a budget is exceeded.

### Hybrid lexical + semantic search

**Priority:** Medium

`hybridSearch` is presently *filtered semantic* search — there is no lexical/full-text score blended with the vector distance (the unused `semanticWeight` option was removed). A true hybrid would combine `ts_rank`/`ILIKE` scoring with cosine distance under a tunable weight.

### Merge reversal (`undoMerge`) connection replay

**Priority:** Low

`condensation.undoMerge` restores node status but does **not** reverse connections redirected during the merge. A full restore needs the merge step to record what it redirected so it can be replayed.

### Cross-item batch entity dedup

**Priority:** Low

Entity dedup is now correct *within* a single ingest, but concurrent items in `ingestBatch` sharing a brand-new entity can still race and create duplicates. Closing this needs cross-item serialisation or a post-batch dedupe pass.

## Under consideration

- Repository-level pushdown for the remaining in-memory graph scans (large-graph scale).
- Confidence-decay scheduling (time-based belief weakening) beyond evidence-driven propagation.
- Pluggable extraction schemas per source, driven from the module's Structure field mappings.

---

## Known issues

No open correctness or security issues after the 2026-07-14 audit. All 84 verified findings — including a SQL-injection fix, a graph-adjacency correctness fix, tenant-isolation (`userId`) gaps, and non-atomic writes — were remediated. Residual, non-blocking items are the *Planned* stubs above and a couple of scale-oriented optimisations (`F8` connection fetch is now scoped to the loaded graph; large-graph analyses still load into memory).
