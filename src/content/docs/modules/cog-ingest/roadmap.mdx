## Known Issues

Issues discovered but not yet fixed.

### Processing Is Synchronous

**Severity:** Medium
**Where:** `backend/src/features/cog-ingest/cog-ingest.service.ts` — the `ingest()` method

The API route waits for cognitive-db to finish processing before responding. For large documents or deep processing, this can cause request timeouts. The job record is created with `status: 'processing'` but the response isn't sent until processing completes or fails.

**Workaround:** Keep content size reasonable and use `'standard'` processing level for large documents. Batch ingestion processes items sequentially, compounding the issue.

### Settings Page State Is Local Only

**Severity:** Low
**Where:** `frontend/src/features/cog-ingest/pages/CogIngestSettingsPage.tsx`

The settings page (default processing level, show token usage) uses `useState` with simulated save — it doesn't persist preferences to the backend. Values reset on page reload.

**Workaround:** None. Settings are cosmetic only in the current version.

### Source Mapping UI Not Implemented

**Severity:** Low
**Where:** Source Config Dialog — no tab for managing Structure node-type mappings

The backend supports source-to-Structure-node-type mappings (CRUD endpoints exist and work), but the source config dialog does not yet include a tab for managing them. The mapping endpoints must be called directly.

**Workaround:** Use the API directly (`POST /api/v1/cog-ingest/sources/:id/mappings`) to create mappings. The source config dialog (v1.5.0) handles context fields, extraction focus, and prompt overrides — mapping management will be added as a future tab.

### Job List Missing Source Names

**Severity:** Low
**Where:** `backend/src/features/cog-ingest/cog-ingest.service.ts` — `getJobs()` method

The job list response includes a `sourceName` field that's always an empty string. The query doesn't join against the sources table to populate it.

**Workaround:** Frontend can cross-reference with `useIngestSources()` to display source names.

### Template Standards Drift

**Severity:** Low
**Where:** generated-file headers across `src/templates/`

One residual deviation from the current LLM rules (the aesthetic opacity modifiers and `as any` casts flagged in the 2026-07-14 audit were fixed in v1.11.7):

- **Generated-file headers** emit only `FILE` / `PURPOSE`; the llm-rules standard is three fields (add `DESCRIPTION`). A repo-wide sweep of ~56 headers, deferred as low/cosmetic (even the scaffold's own generated headers are two-field).

No behavioural impact — a mechanical template sweep when prioritised.

## Planned Improvements

### Short-Term (Next Version)

#### RSS Polling Progress and Entry List

Currently the RSS poll returns a summary (entries found/new/ingested/skipped) with no detail on which entries were processed or why. Improve this:
- Poll results should include entry titles, links, and per-entry status (ingested/skipped/error with reason)
- The Feed tab should display a scrollable list of recent entries after polling with status indicators
- Consider server-sent events (SSE) for real-time progress during long polls
- Entry list should link to the ingest job detail page for ingested entries

**Files to change:**
- `src/templates/backend/rss.ts` — include entry details in `RssPollResult`
- `src/templates/frontend/components.ts` — expand the Feed tab status section to show entry list

#### Ingest History Per Source

Add a way to browse all ingested content for a source — not just stats, but the actual entries, their extraction results, and created nodes.
- Add an expandable "History" section or tab to the source config showing recent ingest jobs
- Each job shows: title, date, status, entities extracted, connections found, link to job detail page
- For RSS sources: link entries to their ingest jobs and show extraction summaries inline
- Filter by status, date range, content type

**Files to change:**
- `src/templates/frontend/components.ts` — add History tab or section to SourceConfigPanel
- `src/templates/frontend/hooks.ts` — may need a source-filtered jobs hook

#### Content Parsing Configuration

Add admin-configurable content parsing/splitting before ingestion. The `ContentHandling` section of `SourceConfigJson` already defines the schema (`chunkingStrategy`, `chunkSize`, `chunkOverlap`, etc.) but has no UI or backend implementation.

<Aside type="note">
**Partially addressed in v1.8.0.** The new Acquisition tab covers content fetching and preparation (readability parsing, screenshot capture, OCR, vision analysis). What remains is the *splitting* step — breaking large documents into chunks before cognitive-db processing.
</Aside>

Use cases:
- Text message logs → split into individual conversations by timestamp/sender patterns
- Email threads → split into individual messages
- Long documents → split into sections by heading
- Mixed content → detect and separate different content types

Implementation:
- Add a "Parsing" section to the Acquisition tab or a standalone tab
- UI for selecting splitting strategy (none, by section, by paragraph, sliding window, auto)
- Configure chunk size, overlap, and content format
- Implement the parsing logic in the ingest service (before cognitive-db processing)
- Each chunk becomes a separate ingest job linked to the parent

**Files to change:**
- `src/templates/frontend/components.ts` — add Parsing section to AcquisitionTab or SourceConfigPanel
- `src/templates/backend/services.ts` — apply content handling config in the `ingest()` method

#### Persist User Preferences

Connect the settings page to actual user preferences storage. Use the starter template's user preferences system (via `useUserProfile` and `updatePreferences`) so settings survive page reloads.

**Files to change:**
- `src/templates/frontend/pages.ts` — `SETTINGS_PAGE_COMPONENT_TEMPLATE`

#### Populate Source Names in Job List

Join the `cog_ingest_source` table when querying jobs to populate the `sourceName` field. This is a one-line Prisma change but needs the job list response type updated too.

**Files to change:**
- `src/templates/backend/services.ts` — `getJobs()` method

#### Source Mapping Tab in Config Dialog

<Aside type="note">

### Medium-Term — Document Management upload pipeline (auto-ingest)

Consume Document Management's reusable upload pipeline so files uploaded through the documents module are automatically ingested. Document Management (v1.2.3) now emits an **awaited async** `documents.document.uploaded` event carrying a storage **reference** (not bytes): `{ userId, providerId, remotePath, fileName, mimeType, size, contentHash }`. cog-ingest is the first intended consumer.

**Shape (event → job → ingest):**

1. **Subscribe** to `documents.document.uploaded` with `actionRegistry.addActionAsync`. Keep the callback O(1): resolve a source id, `enqueueJob('cog-ingest.ingest-file', { ...ref, sourceId }, { tenantId: userId })`, and return. Document Management *awaits* this fan-out, so no heavy work in the callback.
2. **Job handler** (`registerJobHandler('cog-ingest.ingest-file', …)` from the starter `features/jobs` seam) pulls the bytes on demand via the Document Management service — `serviceRegistry.getOptional('document-management')?.getDownloadStreamByPath(providerId, userId, remotePath)` — then calls `cogIngestService.ingest(userId, …)`. Running in a job keeps the synchronous, LLM-bound ingest off the upload response and gives durability/back-pressure under `JOBS_RUNNER=pg-boss`.
3. **Source resolution:** `ingest()` requires a pre-existing user-scoped `sourceId`. Resolve one per user (create-once a `document-management` source, or a provider/folder → source mapping so different folders map to different sources). If none resolves, skip — auto-ingest is opt-in per location.
4. **Content-type mapping** from `mimeType`: `application/pdf` → `contentType: 'pdf'` + `pdfData` (base64); `image/*` → `'image'` + `imageData` (base64) + `imageMimeType`; `text/*` / `application/json` → `'text'` + decoded string. Office formats (DOCX/XLSX/PPTX) have no runtime extractor yet — extract to text first (see **New Content Types → Document Files** below) or skip.
5. **Guarded + inert by default:** guard the download via `serviceRegistry.getOptional('document-management')`; the whole feature is inert when Document Management is absent (nothing emits the event).

**Manual counterpart:** Document Management is also adding a manual "Ingest to Knowledge Base" action (document browser + Structure node) that calls the same job path for on-demand or already-referenced files.

**Files to change (cog-ingest):**
- `src/templates/backend/registrations.ts` + `index.ts` (`INDEX_TEMPLATE`) — a new `action.registration.ts` (subscribe) and `jobs.registration.ts` (handler), imported as side-effects before the public exports.
- A source-resolution helper, and optionally a provider/folder → source mapping.

<Aside type="note">
Depends on Document Management **≥ v1.2.3** — the producer half (the async `documents.document.uploaded` event with the enriched reference payload, plus `getDownloadStreamByPath`) shipped there. See the Document Management roadmap → *Next up → Reusable upload → ingest pipeline*. Two follow-ups remain on the **producer** side: populating `contentHash` for streamed uploads (best-effort/null today) for dedup, and the manual action.
</Aside>

### Medium-Term — New Content Types

The `allowedContentTypes` system (v1.6.0) makes it straightforward to add new content types per source. Each new type needs a backend processor, frontend input UI, and an entry in the content type enum.

#### Audio/Video Recordings

Add `recording` as a content type. Accept audio (MP3, WAV, M4A) and video (MP4, WebM) files. Process via speech-to-text transcription (Whisper API or similar), then ingest the transcript through the standard pipeline. Optionally extract speaker diarisation for conversation-style content.

- Backend: recording processor with transcription provider (configurable — Whisper, Deepgram, AssemblyAI)
- Frontend: file upload with audio/video preview, playback controls, transcription status
- Config: transcription provider selection, language, speaker detection toggle
- Per-source: toggle `recording` on in the allowed content types

#### Email Ingestion

Add `email` as a content type. Accept forwarded emails (EML/MSG files or via webhook) and extract subject, sender, recipients, body, and attachments. Each attachment can be recursively ingested as its own content type (PDF, image, etc.).

- Backend: email parser (e.g. `mailparser`), attachment extraction, recursive ingestion
- Frontend: email upload, or webhook endpoint for automated email forwarding
- Config: which parts to ingest (headers, body, attachments), attachment handling (inline vs separate jobs)
- Integration: could pair with RSS for email newsletter feeds

#### Document Files (Word, Excel, PowerPoint)

Add `document` as a content type covering Office formats (DOCX, XLSX, PPTX) and other rich document types. Extract text content with structure preservation (headings, tables, slides).

- Backend: document parsers (`mammoth` for DOCX, `xlsx` for spreadsheets, `pptx-parser` for presentations)
- Frontend: file upload with format detection and preview
- Config: which elements to extract (text, tables, images, speaker notes)

#### Web Scraping / Site Crawl

Extend the existing URL processor to optionally crawl linked pages from a starting URL. Ingest multiple pages from a site with depth and scope controls.

- Backend: crawler with configurable depth, URL pattern filters, rate limiting
- Config: max pages, crawl depth, URL whitelist/blacklist patterns, follow external links toggle
- Difference from RSS: crawls a site structure rather than polling a feed

### Medium-Term — Processing Improvements

#### Asynchronous Processing

Move ingestion to a background job system so the API responds immediately with a `queued` status. Options:

1. **Simple polling** — return the job ID immediately, frontend polls `GET /jobs/:id` for status updates
2. **WebSocket/SSE** — push status updates to the frontend in real time
3. **Queue integration** — use BullMQ or similar for reliable job processing

This is the most impactful improvement for user experience with large documents.

#### Suggestions Inbox

Aggregate unresolved questions and suggestions from cognitive-db across all ingest jobs into a dedicated view. Currently questions are only visible on individual job detail pages.

- Dashboard or standalone page showing all pending questions/suggestions
- Filter by source, priority, date
- Action buttons: answer, dismiss, create follow-up task, link to Structure node
- Mark as resolved/dismissed with notes

#### Ingest History on Main Page

Add pagination and filtering to the recent jobs list on the main ingest page:
- Filter by source, status, date range
- Cursor-based pagination (backend already supports it)
- Click-through to job detail page

### Medium-Term — cognitive-db v2 (Structure-native) Track

Working memory (**v1.12.0**, see *Completed*) delivered this module's share of the [cognitive-db v2 RFC](/cognitive-db/v2-rfc/)'s early phases. The remaining phases of the accepted RFC run through this module:

- **Phase 3 — engine core retarget:** cognitive-db reads and writes Structure nodes and connections instead of its own `cog_node` tables. *Substrate shipped:* the three `cog_intelligence_*` process tables (in this module's schema since **v1.12.2**), the engine's `StructureAdapter` injection seam and its `storage` flag (defaults to v1 `cog_node`), and the self-contained StructureStore. *Remaining:* wire the ingest/perception/dedup path to the flag and build the adapter from the service registry in `initCogIngest` — unblocked now that Structure's Type Hierarchy has landed
- **Phase 4 — cog-ingest rewire + presentation:** point the ingest pipeline at the retargeted engine and add dedicated presentation for working memory and cognitive content (currently rendered through the generic Structure node UI)
- **Phase 5 — retire v1:** remove the legacy engine tables (`cog_node`, `cog_working_memory_event`, `cog_processing_log`) once nothing reads them

### Long-Term / Ideas

#### AI-Assisted Admin Configuration

Admin uploads a sample document/URL, and AI analyses it to suggest CSS selectors for field scraping, entity types and attributes, output schema sections, and preprocessing steps. A configuration wizard that significantly reduces setup time for new sources.

#### Ontology Management Admin Page

Dedicated admin page for managing ontologies: built-in ontologies shipped with the engine, custom ontologies defined via Structure types and entity type definitions, and AI-suggested ontologies proposed during ingestion. Ability to promote AI-suggested to custom, or merge with built-in.

#### Built-In Format Intelligence Templates

Ship pre-configured acquisition strategies for common formats so admins don't start from scratch. Facebook export JSON structure (encoding fixes, section keys, timestamp format), WhatsApp export format, Instagram/Twitter/Reddit export structures, common website patterns.

#### NotebookLLM-Style Deep Document Understanding

Using the pipeline for multi-pass extraction (overview, details, cross-references), conversational interface over ingested content, and cross-document relationship discovery. Builds on the existing pipeline — about extraction depth and post-processing.

#### Content Deduplication

Detect and flag duplicate or near-duplicate content before processing. Use embedding similarity to identify content that has already been ingested, saving tokens and avoiding duplicate entities.

#### Multi-File Upload

Support uploading multiple files at once in the ingest UI. Process them as a batch with shared context and cross-document connection discovery.

#### Webhook Ingestion

Accept content via webhook endpoint for automated ingestion from external systems (email forwarders, document management systems, CRM triggers).

## Completed

Items that have been done.

### v1.12.0 — Working Memory

Shared `working-memory` Structure node type (seeded at boot as a `userId: null` row), the `cog-ingest.working-memory.*` filter/action hook surface, and `workingMemoryService` — rebuild-semantics assembly reflected onto `origin: 'active'` connections, structured context assembly, pinned items, per-user and per-session granularity. Implements cognitive-db v2 RFC §7 (item 2 of the §13 roadmap). Requires Structure ≥ 1.16.0 (`depends: { structure: '>=1.16.0' }` plus a runtime version gate).

### v1.11.10 — Engine v1.0.0 propagated

Re-vendored the cognitive-db engine at its first stable release (**v1.0.0**, after a full code audit of 84 findings) and bumped the module so hosts pick it up on `update`.

### v1.11.9 — Vendored cognitive-db engine

cognitive-db is now **vendored** into the module (`vendor/cognitive-db.tgz` packed at build via `scripts/vendor-engine.cjs`) and installed into the host backend by an `onInstall`/`onUpdate` lifecycle hook as a portable `file:` dependency — **no npm registry or GitHub auth**, engine version locked to the module release. (v1.11.8 briefly used a `github:` git dependency; superseded because it needed GitHub auth at install.) A local `npm link cognitive-db` still takes precedence for engine development — the hook skips when the target is a symlink.

### v1.11.7 — Standards alignment & host integration

Brought the module in line with the current module-system standards (from the 2026-07-14 audit):

- **Scaffold updated 1.10.2 → 1.18.0** (contract v1.2.0) — adds `module-contract.ts`, the `migrate-db` per-module migration runner, and the `enable`/`disable`/`migrate-db` CLI commands.
- **Registered with the host modules foundation** — new `module.registration.ts` registers cog-ingest with the starter's `moduleRegistry`, so it now creates an `InstalledModule` row and appears on `/admin/modules`.
- **Account-erasure deletion handler** — new `security.registration.ts` declares the module's per-user data and erases it on account deletion (right-to-erasure).
- **`depends: { structure: '*' }`** replaces the deprecated `requiredModules`.
- **Template fixes** — removed forbidden aesthetic opacity modifiers and three `as any` casts. The contract v1.2.0 migration surface is available but unused (the schema is purely additive).

### v1.11.0 — 5-Stage Pipeline Redesign

Complete redesign of the admin panel around a 5-stage pipeline (Intake, Acquisition, Preprocessing, Extraction, Actions). New Input tab with methods + format toggles, new Preprocessing tab with chained AI steps, content integrity config on General tab, updated output schema with ordered lists and enum fields, updated Actions tab with new field mapping model. Full type system for ~30 content formats and per-format acquisition configs. Supersedes the medium-term "New Content Types" items — the type system now supports all planned formats.

### v1.7.0 — Post-Ingest Actions + Field Mapping

Configurable post-ingest actions via Actions tab. Auto-create Structure nodes with field mapping from extraction results and context field values. Mapping sources: context fields, AI summary, entity names (filterable by type), title, raw content.

### v1.6.0 — RSS Feeds + Per-Source Content Types

RSS/Atom feed polling as an automated content type. Per-source content type toggles (Text, URL, Image, PDF, RSS) replace the old source type dropdown — each source configures which types it accepts. Structure node type selector replaces manual slug entry, with fields auto-imported as context fields with enabled toggles. Split-panel admin layout matching Structure module. Background polling service with entry tracking, deduplication, keyword filtering, auto-ingest and manual review workflows.

### v1.5.0 — Source Configuration System

Per-source configuration via the `config` JSON field. Source Config Dialog with four tabs (General, Context Fields, Extraction, Prompts). Context field builder with 8 field types, options builder, auto-generated keys. Dynamic context fields on the ingest page. Backend validation, context string assembly, auto-tag application. Full `SourceConfigJsonSchema` with Zod validation.

### v1.4.0 — Admin Config Management + Deep Multi-Modal Extraction

Runtime engine configuration via admin UI with JSON import/export. Deep multi-modal extraction pipeline with optional URL screenshot capture, PDF page rendering, and enhanced image analysis via vision models. New `cog_ingest_config` table, 4 new API routes, 4 new action hooks.

### v1.3.6 — Settings-Blocks API Migration

Migrated settings page from deprecated `SettingsSectionHeader` / `SettingsToggleRow` / `SettingsDropdownRow` to the current composition API (`SectionHeader`, `Row` + `Dropdown`, `Row` + `ToggleSwitch`). Fixed the runtime `SettingsDivider not found` error.

### v1.3.5 — Template Alignment

Removed all prohibited typography and font weight classes from templates. Replaced raw HTML form elements with shadcn components. Fixed colour literal and stale imports. Brought all generated output into compliance with current LLM rules.
