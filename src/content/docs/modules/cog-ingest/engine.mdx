[**cognitive-db**](https://github.com/ui-git-kim/cognitive-db) is the brain-like knowledge layer that powers Cog Ingest. Where the module handles the admin UI, sources, jobs, and Structure integration, the engine does the actual thinking: perceiving content, extracting entities, discovering connections, growing an ontology, detecting patterns, and answering questions over a single unified graph.

You never install it separately — it ships **inside** the module.

## Delivery: vendored into the module

As of Cog Ingest **v1.11.9**, cognitive-db is **vendored** into the module rather than pulled from a registry:

- The module build packs a pinned engine build into `vendor/cognitive-db.tgz`.
- On `init`/`update`, a lifecycle hook copies that tarball into the host's `backend/.vendor/` and installs it as a portable `file:` dependency.
- The engine version is **locked to the module release**, and installs need **no npm registry or GitHub authentication**.

<Aside type="tip" title="Developing the engine locally">
A local `npm link cognitive-db` in the host backend takes precedence — the install hook detects the symlink and skips the vendored copy, so you can iterate on the engine against a running app. Re-link after a bare `npm install` if the symlink gets dropped.
</Aside>

cognitive-db is also a standalone library. To use it outside Cog Ingest, install it directly from its (private) Git repo — see the engine's own `README` / `INTEGRATION` docs:

```bash
npm install github:ui-git-kim/cognitive-db
```

## Tenant model: `tenantId === userId`

The engine is deliberately **single-user-per-tenant**: one user *is* the tenant. There is no team/organisation layer and no separate `tenantId` column — every row is owned by `userId`, and that is the isolation boundary.

This is a product decision, not a limitation: the target is apps for **solo business owners** who want the full intelligence stack without the teams/enterprise machinery that gets in the way. It also means the engine slots cleanly into the starter's `userId`-ownership security foundation — which is why the module's `security.registration.ts` declares `ownedModels: []` and relies on ownership isolation.

## The unified graph

Everything the engine stores lives in one table, `cog_node`, as typed nodes — entities, connections, concepts, emotions, ontology definitions, and versioned history all share the schema. Two supporting tables complete the set:

| Table | Purpose |
|-------|---------|
| `cog_node` | The unified graph — nodes *and* connections (as connection-type nodes with `fromNodeId`/`toNodeId`), with pgvector embeddings for semantic search |
| `cog_working_memory_event` | Short-term event stream (retrievals, co-occurrences) feeding consolidation |
| `cog_processing_log` | Per-operation audit + job/propagation bookkeeping |

## Configuration

cognitive-db is initialised once at app startup with your Prisma client and model providers. Cog Ingest generates this wiring for you during `init`; the shape is:

```typescript
import { initConfig } from 'cognitive-db';

initConfig({
  database: { mode: 'integrated', prisma },   // your app's PrismaClient
  llm: { model: anthropic('claude-...') },      // Vercel AI SDK LanguageModel
  embedding: { /* provider + dimensions */ },
});
```

See [Configuration](/modules/cog-ingest/configuration/) for the module-level prompts and environment variables that feed this.

## Quality

The engine reached its **first stable release (v1.0.0)** on 2026-07-14 after a comprehensive code audit (84 verified findings across 96 files, all remediated — including a SQL-injection fix and graph-traversal correctness fixes). It follows semantic versioning from here, inline with the module system. See the [Engine Roadmap](/modules/cog-ingest/engine-roadmap/) for what's next and what's intentionally out of scope.
