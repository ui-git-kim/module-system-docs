This page documents notable changes to the Cog Ingest module. The module uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, template fixes

## v1.12.3 — 2026-07-23

**Structure v1.19 alignment + host-compatibility pass — generated code now type-checks clean on Prisma 7.8 / React 19 / zod 4 hosts**

The same class of clearing pass structure ran in its v1.19.2: a full `tsc --noEmit` validation of cog-ingest's generated output inside a live host (veradai) went from **29 frontend + 33 backend errors to 0/0**. Ships with the vendored engine **cognitive-db v1.0.1** (ai peer `^6 || ^7`, zod peer `^3.25 || ^4`, zod-4-safe `z.record` — the old peers ERESOLVE-failed the vendored engine install on current hosts).

**Structure v1.19 drift fixes** (from the 2026-07-23 alignment audit — all 33 touchpoints verified, 0 misaligned, 3 drift-risks fixed):
- `IngestNodeWidget`'s `node` prop is now optional with a null guard — the required prop against structure's optional `NodeWidgetProps.node` was a live TS2322 under `strictFunctionTypes`
- Working memory passes `limit: 500` to `getNodeConnections` (structure v1.19.0 bounded it at `min(500, limit ?? 200)` newest-first; without it, active sets beyond 200 edges would silently drop the oldest from prune/sweep/pinned reads). The working set now has an explicit 500-edge ceiling
- The three Structure field pickers (Acquisition, Output Schema, Actions) read `resolved.fields ?? fields`, so fields inherited via v1.18 `superClasses` appear in mapping UIs

**RSS could never start — fixed.** RSS was gated on `config.allowedContentTypes`, a field the config schema **strips at parse** (nothing writes it since the v1.11 pipeline redesign), so `initRssPolling` never started a poller and `pollRssSource` always threw. RSS now keys off an enabled Stage-1 `feed` input method — the admin dialog's own definition. Ingest content-type validation and the user page's visible tabs derive from enabled input methods the same way (`paste`/`api`/`in_app_capture` → text, `url` → url, `file_upload` → image/pdf, `feed` → rss); sources with no methods configured stay unrestricted.

**Host-compatibility sweep:**
- zod 4: two-argument `z.record(z.string(), …)` everywhere; `.default({})` → `.prefault({})` (22 schemas) — the v3 forms throw or fail to compile on zod 4
- Prisma 7: the `Prisma` namespace imports from `../../generated/prisma/client.js` (no longer exported by `@prisma/client`); `InputJsonValue` casts on source create/update; the engine call bridges through the engine's structural Prisma interface
- pdf-parse v2: `PDFParse` class + `getText()` (the v1 default-function export no longer exists — PDF ingestion was broken at runtime on current hosts, not just in types)
- rss-parser items properly typed (`FeedItem`); react-query keys serialise options objects; unused imports/locals pruned; `ContextFieldsTab`'s add-field object now includes `enabled: true`; fixed an unbound `fields` reference in `ContextFieldGroup` (move-down button disable logic)
- `testScrape` declared its `results` map after the Puppeteer error path wrote to it (TDZ crash on that path)

## v1.12.2 — 2026-07-16

**Action hooks now reach async listeners (`doAction` → `doActionAll`)**

All eleven ingest/config/RSS lifecycle hooks were emitted with `actionRegistry.doAction()`, which runs **sync listeners only** — handlers registered with `addActionAsync` were silently skipped. The module's own RSS poller subscribes to `source.created` / `source.updated` / `source.deleted` asynchronously, so dynamic polling start/stop never fired on source changes and polling only reconciled at app restart. The documented consumer pattern (listening with `addActionAsync`, e.g. on `ingest.completed`) was affected the same way. Starter v1.54.0's dev-mode warning flags exactly this mismatch.

- All 13 emission sites across the backend service, processors, and RSS templates now fire `await actionRegistry.doActionAll()` — sync then async listeners, in order; the registry logs a throwing listener without propagating, so a bad subscriber can't break the producer
- RSS polling now starts, reconfigures, and stops immediately when a source is created, updated, or deleted — no app restart needed
- Affected hooks: `ingest.completed`, `ingest.failed`, `source.created`, `source.updated`, `source.deleted`, `config.updated`, `config.imported`, `extraction.visual-completed`, `extraction.screenshot-taken`, `rss.polled`, `rss.entryDetected`
- Docs: the [Integration hook reference](/modules/cog-ingest/integration/#action-hooks) now lists the two RSS hooks (16 total) and documents the dispatch semantics

## v1.12.0 — 2026-07-15

**Working memory — node type, hook surface, and service**

Implements working memory from the cognitive-db v2 RFC (§7). Backend-only in this release — no HTTP routes or dedicated UI; working memory nodes render through the generic Structure node UI (dedicated presentation lands with the v2 RFC's Phase 4).

**`working-memory` Node Type:**
- Seeded at backend boot (and lazily on first use) as a **shared** row (`userId: null`) via `importNodeTypes`, guarded by an existence check so admin customisations survive restarts
- System type (`isSystem: true`), hidden from navigation, single `notes` textarea field ("Recent Thought Process")
- The active set is stored as outbound `working-memory` connections with `origin: 'active'` (Structure connection provenance); module state lives in `metadata['cog-ingest']` (`lastActiveAt`, `itemCount`)
- Two granularities: a rolling per-user root node, or one node per session (a child of the session node) — nodes are stamped `data.scope: 'user' | 'session'`

**Hook Surface (`cog-ingest.working-memory.*`):**
- Filter `assemble` — modules contribute `WorkingMemoryItem[]` with **rebuild semantics**: contribute what is relevant now, every run; items no longer contributed drop out, pinned items persist
- Filter `context` — shapes the structured `WorkingMemoryContext` handed to chat/LLM consumers; `maxTokens` is advisory (token-aware filters own trimming)
- Actions: `updated` (typed change payload: `assemble`, `context`, `item-added`, `item-removed`, `notes`), `item.added` / `item.removed`, `session.started` / `session.ended` — all fired catch-guarded fire-and-forget with async listener support

**Working Memory Service** (`serviceRegistry.get('cog-ingest').workingMemory`):
- `ensureNode`, `get`, `assemble`, `getContext`, `addItem` / `removeItem`, `setNotes`, `startSession` / `endSession`
- `assemble` seeds pinned items, runs the assemble filter, dedupes, and reflects the diff onto the active-set connections; `endSession` sweeps the ephemeral edges and archives the session node
- Manual `addItem` pins by default; assemble-created edges are never pinned
- Archived or deleted per-session nodes are restored when a session re-opens instead of colliding with the unique constraint

**Types & Exports:**
- New types: `WorkingMemoryItem`, `WorkingMemoryContext`, `WorkingMemoryContextSection`, `WorkingMemoryChange`, `WorkingMemoryNodeRef`, `WorkingMemorySnapshot`
- New exports: `workingMemoryService`, `ensureWorkingMemoryNodeType`, `WORKING_MEMORY_HOOKS`, `WORKING_MEMORY_NODE_TYPE_SLUG`, `WORKING_MEMORY_CONNECTION_TYPE`

**Requires Structure ≥ 1.16.0** — `depends` tightened to `{ structure: '>=1.16.0' }`, and the service enforces a runtime version gate against the Structure service's v1.16 surface.

## v1.11.0 — 2026-05-21

**5-stage pipeline redesign with new admin tabs**

Complete redesign of the admin Ingest Configuration panel around a 5-stage pipeline: Intake, Acquisition, Preprocessing, Extraction, Actions.

**New Pipeline Architecture:**
- Every content piece flows through up to 5 stages, each mapped to an admin tab
- Full tab bar: General | Input | Acquisition | Preprocessing | Feed\* | Context | Extraction | Prompts | Actions

**Input Tab (Stage 1: Intake) — redesigned:**
- Input methods: paste, file upload, URL, feed, API, in-app capture — independently toggled
- Content format categories: text, documents, email/messaging, app exports, social media exports, data, media, web
- ~30 content formats with per-format validation settings (file size, page limits, duration)

**Acquisition Tab (Stage 2) — per-format configs:**
- Per-format acquisition strategies: URL scraping, PDF extraction, image vision, audio transcription, video processing
- `includeAsAiContext` replaces `includeInPrompt` on scraped field rules

**Preprocessing Tab (Stage 3) — new:**
- Optional AI-powered step that organises content before extraction
- Step chain builder — multiple preprocessing steps in sequence
- Five modes: split, organise, classify, enrich, custom
- Per-step prompt and output configuration (title source, max items, keep original)

**Extraction Tab (Stage 4) — enhanced:**
- Output schema: `OutputField` gains `enumValues` and `required` fields
- Output schema: `OutputSection` gains `ordered` (for numbered items like court orders) and `indexField`
- Boolean field type added to output fields

**Actions Tab (Stage 5) — updated:**
- Updated `FieldMapping` type: `source` is now `'scraped' | 'extracted' | 'context' | 'computed'` with `sourceKey`
- Node name from: added `ai_title` and `scraped_field` options
- Content integrity display when locked mode is configured

**General Tab — enhanced:**
- Content integrity configuration: none, content\_locked, full\_locked modes
- Hash algorithm selection, access tracking, correction audit trail

**Types:**
- New: `InputConfig`, `FormatConfig`, `ContentFormat`, `PreprocessingConfig`, `PreprocessingStep`, `ContentIntegrityConfig`, `ActionsConfig`, `FieldMapping`, `AcquisitionResult`, `PreprocessedItem`, `ParsedMessage`, `SpeakerSegment`
- New acquisition configs per format: text, PDF, Word, spreadsheet, presentation, ebook, email, email archive, chat export, app export, social media export, CSV, JSON, calendar, contacts, image, audio, video, video URL, URL, RSS
- `SourceConfigJson` restructured with `input`, `preprocessing`, `integrity`, `postIngest` fields
- Backend Zod schemas updated to match all new types

**Prompt Assembly:**
- New template variables: `{preprocessingMeta}`, `{contentType}`
- Output schema prompt includes ordered list markers, enum values, required field indicators

## v1.9.0 — 2026-05-13

**Field scraping, content selection, and auto-mapping**

Two-layer acquisition system that separates deterministic scraping from AI extraction:

**Field Scraping (Acquisition tab):**
- New `ScrapedFieldRule` type: CSS selector-based field extraction that runs before AI — no tokens spent on structured data
- Inline scraping rule editor with CSS selector, type (text/date/number/list/html), attribute extraction, regex transforms, required flag
- "Add from Structure" dropdown pre-populates rules from the Structure node type's field definitions
- Content selection controls what goes to AI: CSS selector for content area, exclude scraped regions, or full page

**Structure Field Integration:**
- `structureFieldKey` on `ScrapedFieldRule` and `OutputSection` links fields to Structure node type fields
- Cross-tab awareness: fields assigned as scraped in Acquisition are greyed out in Extraction's "Add from Structure" dropdown, and vice versa
- Each Structure field goes to exactly one place: scraped deterministically or extracted by AI

**Auto-Mapping (Actions tab):**
- Auto-mapped entries display read-only from scraped fields and output schema sections with `structureFieldKey`
- New `scraped_field` and `source_url` mapping sources in `NodeFieldMapping`
- `autoMapped` flag tracks which mappings were auto-generated

**Backend:**
- `buildSourceContext` tells AI what's already scraped so it doesn't re-extract
- `processUrl` receives `contentSelection` and `scrapedFields` config
- Post-ingest auto-maps scraped values and output schema sections to Structure node fields

## v1.8.0 — 2026-05-13

**Input, Acquisition tabs and reworked Extraction tab**

Major redesign of the admin source configuration panel. The tab architecture changes from `General | Feed* | Context Fields | Extraction | Prompts | Actions` to `General | Input | Acquisition | Feed* | Context | Extraction | Prompts | Actions`.

**New Input Tab:**
- Per-content-type validation settings with toggle UI
- Text: min/max content length, input mode (paste/upload/both)
- URL: single/batch mode, max URLs, allowed/blocked domain lists
- PDF: max file size, max pages, scanned PDF (OCR) toggle
- Image: allowed formats, max file size
- Content type toggles moved here from General tab

**New Acquisition Tab:**
- Configures how raw content becomes extractable text — the pipeline stage between receiving content and running AI extraction
- URL: capture mode (text only / text + screenshot / screenshot only), readability or custom CSS selector parsing, screenshot viewport and full-page settings, output composition
- RSS: entry content strategy (entry only / fetch page / both), which entry fields to include (title, author, date, description, full content)
- PDF: text extraction method (standard / OCR / auto-detect), layout preservation, page rendering strategy with resolution settings
- Image: vision analysis type (general / document / design / custom prompt), EXIF extraction, colour palette extraction, OCR toggle

**Reworked Extraction Tab:**
- Replaced comma-separated entity/relationship type strings with rich definition cards
- Entity type definitions: name, description, priority badge, typed attributes (text/date/number/enum/list), required flags, example values
- Relationship type definitions: name, description, from/to entity type constraints, priority
- Output schema builder: define structured JSON sections with typed fields and extraction hints, required section toggles, live JSON preview
- Feature toggles: dates, emotions, questions, action items — presented clearly with descriptions

**Context Tab (renamed from Context Fields):**
- Quick-add intent field presets: "Why are you saving this?", "What stands out?", "How will you use this?"
- One-click to add pre-configured intent fields that capture ingestion purpose

**Actions Tab:**
- Field mapping now includes `output_field` source option when an output schema is defined
- Dropdown lists all output schema sections and their fields for mapping

**Prompts Tab:**
- New template variables: `{outputSchema}`, `{entityTypes}`, `{relationshipTypes}`, `{acquisitionMeta}`

**Backend:**
- `buildSourceContext()` assembles rich entity type definitions (with attributes and examples), relationship type definitions (with directionality), and output schema into the extraction prompt
- Post-ingest field mapping supports `output_field` source for extracting values from structured AI output

## v1.5.0 — 2026-05-12

**Source configuration system with context fields**

The Settings button on each source row now opens a rich configuration dialog. Admins configure per-source processing behaviour through the existing `config` JSON field — no schema migration needed.

**Source Config Dialog:**
- Four-tab sheet: General, Context Fields, Extraction, Prompts
- Context field builder with 8 field types (text, textarea, select, multiselect, date, number, boolean, tags)
- Add/edit/remove/reorder fields with auto-generated camelCase keys from labels
- Options builder for select/multiselect fields (value + label pairs)
- Extraction focus — priority/ignore entity types, relationship types, feature toggles
- Prompt overrides — system context, content description, extraction instructions
- Auto-tag rules — always or conditional based on context field values
- UI config — Lucide icon name and hex colour for source badges

**Dynamic Context Fields on Ingest Page:**
- When a user selects a source, its configured context fields appear on the ingest form
- Type-appropriate inputs: text, textarea, select dropdowns, multiselect toggle badges, date picker, number, switch, comma-separated tags
- Required field validation prevents submission until filled
- Default values applied from field config when source is selected

**Backend Processing:**
- `SourceConfigJsonSchema` — full Zod validation for the source config JSON
- `buildSourceContext()` — assembles a labelled prompt string from source config + user context values
- Ingest service validates required context fields, checks allowed content types, applies auto-tag rules, collects metadata-usage fields, and passes assembled context to cognitive-db
- `contextFields` parameter added to the ingest request schema

**New shadcn components:** `sheet`, `switch`

## v1.4.0 — 2026-05-11

**Admin configuration management and deep multi-modal extraction pipeline**

Two major features in this release:

**Runtime Engine Configuration:**
- New `cog_ingest_config` database table for persisting configuration overrides
- Admin page redesigned with tabbed interface: Ingest Sources + Engine Configuration
- Config sections: LLM, embedding, processing defaults, feature flags, prompts, extraction depth
- JSON import/export for transferring configuration between environments
- `initCogIngest()` now loads DB overrides and merges with static defaults on startup
- `reinitialiseEngine()` applies config changes at runtime without server restart
- 4 new API routes: `GET/PUT /config/:key`, `POST /config/import`, `GET /config/export`
- Zod validation schemas for all config sections

**Deep Multi-Modal Extraction:**
- Enhanced image processor with richer structured vision prompts (toggleable)
- URL processor with optional Puppeteer screenshot capture + vision analysis for design/layout extraction
- PDF processor with optional page rendering via `pdf-to-img` + per-page vision analysis
- New `EnrichedContent` type with structured text sections (metadata, text, visual analysis, observations)
- Configurable PDF page sampling strategies (first-only, first-last, every-5th, all)
- All visual features off by default, toggleable per content type in admin UI
- Graceful fallback to text-only when optional dependencies unavailable

**New hooks:** `cog-ingest.config.updated`, `cog-ingest.config.imported`, `cog-ingest.extraction.visual-completed`, `cog-ingest.extraction.screenshot-taken`

**New dependency:** `pdf-to-img` (for PDF page rendering). Puppeteer is optional — install separately for URL screenshots.
