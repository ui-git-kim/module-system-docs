## Frontend Pages

### Ingest Page (`/cog-ingest`)

The main page where users submit content for processing. It includes:

- **Content type tabs** — switch between Text, URL, Image, and PDF input
- **Source selector** — choose from admin-configured ingest sources
- **Dynamic context fields** — if the selected source has configured context fields, they appear between the source selector and the content input. Each field renders with a type-appropriate UI component (text input, textarea, select dropdown, multiselect toggle badges, date picker, number input, switch, or comma-separated tags input). Required fields must be filled before submission. Default values from the source config are applied when the source is selected. (v1.5.0)
- **Stats row** — total jobs, nodes created, connections found, tokens used
- **Recent jobs** — list of recent ingestion jobs with status badges

Users select a source, fill in any context fields, choose a content type, and submit. For text and URL inputs, the form uses the standard mutation hook. For image and PDF uploads, it uses `FormData` with the file upload service method. Context field values are sent as `contextFields` in the request body.

### Job Detail Page (`/cog-ingest/jobs/:id`)

Shows the full results of a single ingest job:

- **Summary stats** — entity count, connection count, emotion count, questions raised
- **Extracted entities** — displayed as badges with sub-type labels
- **Discovered connections** — listed with connection type
- **Questions raised** — questions the engine identified with priority levels

The page sets breadcrumbs via `useBreadcrumbs` for navigation context.

### Settings Page

Registered in the user settings area. Provides:

- **Default processing level** — dropdown to set the default depth for new sources
- **Show token usage** — toggle to display token consumption in results

Uses the starter template's `SectionHeader`, `Row`, `Dropdown`, `Divider`, and `ToggleSwitch` components from `@/components/settings-blocks`.

### Admin Page

Registered in the admin area with a tabbed interface:

**Ingest Sources tab:**
- Stats overview — active sources, total jobs, nodes created, tokens used
- Source management — create, configure, and delete ingest sources
- Quick-add form — inline form for creating new sources with slug and name
- Source configuration — click the Settings button on a source row to open the Source Config Dialog with four tabs: General (icon, colour, auto-tags), Context Fields (field builder), Extraction (entity/relationship priorities), and Prompts (LLM instructions). See [Source Configuration](/modules/cog-ingest/configuration/#source-configuration-per-source) for details. (v1.5.0)

**Engine Configuration tab (v1.4.0):**
- LLM configuration — provider, model, API key status
- Embedding configuration — provider, model, dimensions (read-only)
- Processing defaults — default level, batch size, pattern growth
- Feature flags — toggle switches for each cognitive processing feature
- Prompts — system context, content description, custom prompt overrides
- Extraction depth — visual analysis toggles, PDF sampling, screenshot settings
- Import/export — JSON backup and transfer between environments

## Frontend Hooks

All hooks use TanStack Query with module-namespaced query keys via `createModuleQueryKeys('cog-ingest')`.

### Ingest Operations (`useCogIngest.ts`)

| Hook | Type | Purpose |
|------|------|---------|
| `useIngestStats()` | Query | Fetches aggregate stats (total jobs, nodes, connections, tokens, active sources) |
| `useSubmitIngest()` | Mutation | Submits content for ingestion; invalidates `jobs` and `stats` queries on success |

### Source CRUD (`useCogIngestSources.ts`)

| Hook | Type | Purpose |
|------|------|---------|
| `useIngestSources()` | Query | Fetches all ingest sources for the current user |
| `useCreateSource()` | Mutation | Creates a new ingest source; invalidates `sources` query |
| `useUpdateSource()` | Mutation | Updates an existing source; invalidates `sources` query |
| `useDeleteSource()` | Mutation | Deletes a source; invalidates `sources` query |
| `useSourceMappings(sourceId)` | Query | Fetches mappings for a specific source (enabled when `sourceId` is provided) |
| `useUpsertMapping()` | Mutation | Creates or updates a source mapping; invalidates `mappings` query |

### Engine Config (`useCogIngestConfig.ts`)

| Hook | Type | Purpose |
|------|------|---------|
| `useEngineConfig()` | Query | Fetches merged engine configuration (defaults + overrides) |
| `useUpdateEngineConfig()` | Mutation | Updates a config section; invalidates `config` query |
| `useImportConfig()` | Mutation | Imports configuration from JSON; invalidates `config` query |
| `useExportConfig()` | Mutation | Exports current configuration as JSON |

### Job Queries (`useCogIngestJobs.ts`)

| Hook | Type | Purpose |
|------|------|---------|
| `useIngestJobs(options?)` | Query | Fetches paginated job list with optional filters |
| `useIngestJob(id)` | Query | Fetches a single job by ID (enabled when `id` is provided) |
| `useIngestedForNode(nodeId)` | Query | Fetches completed jobs for a Structure node (enabled when `nodeId` is provided) |

**Job list options:**

```typescript
interface IngestJobListOptions {
  limit?: number;          // Page size (default: 50, max: 200)
  cursor?: string;         // Cursor for pagination
  sourceId?: string;       // Filter by source
  status?: string;         // Filter by status
  structureNodeId?: string; // Filter by Structure node
}
```

## Frontend Service

The API service (`services/cog-ingest.service.ts`) wraps all backend endpoints using the shared `api` client from `@/lib/api`. Every method accepts an optional `{ signal?: AbortSignal }` for cancellation.

| Method | Endpoint | Description |
|--------|----------|-------------|
| `getSources()` | `GET /sources` | List all sources |
| `createSource(input)` | `POST /sources` | Create a source |
| `updateSource(id, input)` | `PATCH /sources/:id` | Update a source |
| `deleteSource(id)` | `DELETE /sources/:id` | Delete a source |
| `getMappings(sourceId)` | `GET /sources/:id/mappings` | List mappings for a source |
| `upsertMapping(sourceId, input)` | `POST /sources/:id/mappings` | Create or update a mapping |
| `ingest(input)` | `POST /ingest` | Submit content for ingestion |
| `uploadAndIngest(file, options)` | `POST /ingest` | Upload a file and ingest (FormData) |
| `getJobs(options?)` | `GET /jobs` | List jobs with pagination/filters |
| `getJobById(id)` | `GET /jobs/:id` | Get a single job |
| `getStats()` | `GET /stats` | Get aggregate stats |
| `getIngestedForNode(nodeId)` | `GET /nodes/:nodeId/ingested` | Get completed jobs for a node |
| `getEngineConfig()` | `GET /config` | Get merged engine configuration |
| `updateEngineConfig(key, value)` | `PUT /config/:key` | Update a config section |
| `importConfig(config)` | `POST /config/import` | Import config from JSON |
| `exportConfig()` | `GET /config/export` | Export config as JSON |

All endpoints are prefixed with `/api/v1/cog-ingest`.

All routes require authentication (`requiresAuth: true`). The base path is `/api/v1/cog-ingest`.

```
GET  /api/v1/cog-ingest/sources/:id/mappings

```

**Create/Update body:**

### Engine Configuration

```
GET  /api/v1/cog-ingest/config           — Get merged config (defaults + DB overrides)
PUT  /api/v1/cog-ingest/config/:key      — Update a config section (engine, processing, features, prompts, extraction)
POST /api/v1/cog-ingest/config/import    — Import config from JSON body
GET  /api/v1/cog-ingest/config/export    — Export full config as JSON
```

The PUT endpoint validates against Zod schemas per section. The import endpoint accepts the full JSON format (see [Configuration](/modules/cog-ingest/configuration/#json-import--export)) and returns `{ importedKeys, warnings }`.

Fires actions: `cog-ingest.config.updated` (on PUT), `cog-ingest.config.imported` (on import)

### Structure Integration

```
GET /api/v1/cog-ingest/nodes/:nodeId/ingested
```

Returns up to 50 completed ingest jobs associated with a Structure node, ordered by most recent.

## Content Processors

The backend includes three content processors that transform non-text content before passing it to cognitive-db. Each processor supports optional **deep multi-modal extraction** via vision models (configurable in the admin UI under Extraction Depth).

### Image Processor

Uses the configured LLM's vision capability to describe the image. When **enhanced image analysis** is enabled, uses a richer structured prompt that extracts composition, colours (with hex values), style, text, technical details, context type, and design elements. Supports optional user-provided context (people, location, event, notes) to ground the description.

### URL Processor

Fetches the URL, parses the HTML with [jsdom](https://github.com/jsdom/jsdom) and [@mozilla/readability](https://github.com/mozilla/readability) to extract article text. Falls back to raw body text if Readability parsing fails.

When **URL screenshots** is enabled and Puppeteer is installed, additionally captures a full-page screenshot and runs vision analysis to extract layout, colour scheme, typography, imagery, UI components, and design patterns. The text and visual analysis are combined into an `EnrichedContent` format with structured sections.

### PDF Processor

Parses the PDF buffer with [pdf-parse](https://www.npmjs.com/package/pdf-parse) to extract text content.

When **PDF page rendering** is enabled, additionally renders selected pages as images using [pdf-to-img](https://www.npmjs.com/package/pdf-to-img) and runs vision analysis to extract document structure, tables, charts, signatures, formatting, and annotations. The page sampling strategy (first-only, first-last, every-5th, all) is configurable.

### EnrichedContent Format

When visual analysis is enabled, processors produce enriched content with structured text sections that cognitive-db's extraction prompts can parse naturally:

```
=== SOURCE METADATA ===
Type: url
Source: https://example.com
Captured: 2026-05-11T10:30:00Z

=== TEXT CONTENT ===
[extracted text content]

=== VISUAL ANALYSIS ===
[vision model description]

=== STRUCTURED OBSERVATIONS ===
Layout: [description]
Colours: [description]
Typography: [description]
```

## Preprocessors Registry

The module includes a custom **preprocessors registry** for transforming content before it reaches cognitive-db. This is separate from the starter template's filter registry — it's a module-level concern for content splitting and transformation.

```typescript
import { preprocessorsRegistry } from './preprocessors.registry';

preprocessorsRegistry.register({
  id: 'text-message-splitter',
  name: 'Text Message Splitter',
  contentTypes: ['text'],
  detect: (content) => /^\[?\d{1,2}[:/]\d{2}/.test(content),
  process: async (content, metadata) => ({
    content: content.replace(/\n(?=\[?\d{1,2}[:/]\d{2})/g, '\n---\n'),
    metadata: { ...metadata, preprocessor: 'text-message-splitter' },
  }),
});
```

Preprocessors run in registration order on content that matches their `contentTypes` and passes their `detect` function.

## TypeScript Types

The module exports all types from `types/index.ts`. Key interfaces:

```typescript
// Content types
type ContentType = 'text' | 'url' | 'image' | 'pdf';

// Source configuration (v1.5.0) — stored in IngestSource.config
interface SourceConfigJson {
  contextFields: ContextField[];
  extraction?: ExtractionFocus;
  prompts?: SourcePromptOverrides;
  autoTags: AutoTagRule[];
  ui?: SourceUIConfig;
  validation?: SourceValidationConfig;
}

interface ContextField {
  key: string;
  label: string;
  description?: string;
  type: 'text' | 'textarea' | 'select' | 'multiselect' | 'date' | 'number' | 'boolean' | 'tags';
  options?: Array<{ value: string; label: string }>;
  required: boolean;
  defaultValue?: string | number | boolean | string[];
  usage: 'context' | 'metadata' | 'filter';
  order: number;
}

// Source
interface IngestSource {
  id: string;
  userId: string;
  slug: string;
  name: string;
  description: string | null;
  sourceType: string;
  processingLevel: 'minimal' | 'standard' | 'deep';
  ontologyGrowthMode: 'off' | 'propose' | 'auto';
  isActive: boolean;
  config: SourceConfigJson;
  createdAt: string;
  updatedAt: string;
}

// Job
interface IngestJob {
  id: string;
  userId: string;
  sourceId: string;
  status: 'queued' | 'processing' | 'completed' | 'failed';

  result: Record<string, unknown> | null;
  tokensUsed: number;
  nodesCreated: number;
  connectionsFound: number;
  durationMs: number;
  error: string | null;
  structureNodeId: string | null;
  createdAt: string;
  updatedAt: string;
}

// Stats
interface IngestStats {
  totalJobs: number;
  completedJobs: number;
  failedJobs: number;
  totalTokensUsed: number;
  totalNodesCreated: number;
  totalConnectionsFound: number;

}
```
