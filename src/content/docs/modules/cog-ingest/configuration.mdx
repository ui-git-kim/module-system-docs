This page covers everything you configure when installing and running Cog Ingest — the init prompts, environment variables, provider setup, and runtime configuration.

---

## Init Prompts

During `npx cognitive-ingest init`, you answer four questions. Your answers determine which dependencies are installed, which environment variables are required, and how the config template is generated.

| Prompt | Options | Default | Effect |
|--------|---------|---------|--------|
| **LLM provider** | Anthropic (Claude), OpenAI (GPT) | Anthropic | Installs `@ai-sdk/anthropic` or `@ai-sdk/openai`; sets the model in config |
| **Embedding provider** | Voyage AI, OpenAI | Voyage AI | Installs `voyageai` or uses OpenAI SDK; configures embed functions |
| **Embedding dimensions** | 512, 1024, 2048 | 1024 | Sets `vector(N)` column size in schema; affects accuracy vs speed |
| **Default processing level** | Minimal, Standard, Deep | Standard | Default depth for new ingest sources |

<Aside type="note">
These prompt answers are stored as template variables (`{{llmProvider}}`, `{{embeddingProvider}}`, `{{embeddingDimensions}}`, `{{defaultProcessingLevel}}`) and baked into the generated config file at install time. These become the **static defaults**. From v1.4.0, you can override most settings at runtime via the admin UI — see [Runtime Configuration](#runtime-configuration-admin-ui) below.
</Aside>

## Environment Variables

All environment variables are backend-only. Which ones are required depends on your prompt answers.

| Variable | Required When | Description |
|----------|--------------|-------------|
| `ANTHROPIC_API_KEY` | LLM provider = Anthropic | API key for Claude models |
| `OPENAI_API_KEY` | LLM provider = OpenAI, **or** Embedding provider = OpenAI | API key for GPT and/or OpenAI embeddings |
| `VOYAGE_API_KEY` | Embedding provider = Voyage AI | API key for Voyage embeddings |

## Provider Setup

### Anthropic (Claude)

1. Create an account at [console.anthropic.com](https://console.anthropic.com)
2. Generate an API key under **API Keys**
3. Add it to `backend/.env` as `ANTHROPIC_API_KEY`

The module uses `claude-sonnet-4-20250514` by default. To change the model, edit the `llmModel` assignment in `cog-ingest.config.ts`.

### OpenAI (GPT)

1. Create an account at [platform.openai.com](https://platform.openai.com)
2. Generate an API key under **API Keys**
3. Add it to `backend/.env` as `OPENAI_API_KEY`

The module uses `gpt-4o` for LLM and `text-embedding-3-small` or `text-embedding-3-large` for embeddings (depending on dimensions).

### Voyage AI (Embeddings)

1. Create an account at [dash.voyageai.com](https://dash.voyageai.com)
2. Generate an API key
3. Add it to `backend/.env` as `VOYAGE_API_KEY`

The module uses `voyage-3` for dimensions ≤ 1024 and `voyage-3-lite` for larger dimensions.

## Runtime Configuration (Admin UI)

From v1.4.0, engine configuration is managed at runtime through the admin page. The static config file provides **defaults**; the database provides **overrides**. Database overrides always win.

The admin page has two tabs:
- **Ingest Sources** — source CRUD (existing)
- **Engine Configuration** — runtime settings management (new)

### Config Sections

The Engine Configuration tab is organised into collapsible sections:

| Section | Key | What It Controls |
|---------|-----|-----------------|
| **LLM Configuration** | `engine` | Provider (Anthropic/OpenAI), model, API key status (read-only) |
| **Embedding Configuration** | `engine` | Provider (Voyage/OpenAI), model. Dimensions are read-only (changing requires re-embedding) |
| **Processing Defaults** | `processing` | Default level, batch size, pattern growth mode/threshold |
| **Feature Flags** | `features` | Toggle switches for: understanding, patterns, proactive, entity extraction, relationship discovery, question detection, emotion detection |
| **Prompts** | `prompts` | System context, content description, custom prompt overrides |
| **Extraction Depth** | `extraction` | Visual analysis master toggle, per-type toggles (URL screenshots, PDF rendering, enhanced images), PDF sampling strategy |
| **Import / Export** | — | JSON file import with preview, JSON export download |

### Config Merging

On backend startup, `initCogIngest()`:
1. Loads static defaults from the generated config file
2. Queries the `cog_ingest_config` table for overrides
3. Merges overrides on top of defaults
4. Initialises cognitive-db with the merged config

When an admin saves a config change:
1. The override is persisted to `cog_ingest_config`
2. `reinitialiseEngine()` calls cognitive-db's `updateConfig()` to apply immediately
3. No server restart required

### Read-Only Fields

Some settings cannot be changed at runtime and are shown as read-only in the admin UI:

| Field | Reason |
|-------|--------|
| Embedding dimensions | Changing would require re-embedding all existing data |
| Database mode | Set at install time (always `integrated`) |
| API keys | Shown as status only (`configured` / `missing`) — never exposed |

### JSON Import / Export

Export the current configuration as a JSON file for backup or transfer between environments. Import a JSON file to bulk-apply settings with a preview diff before confirming.

**JSON format** (all sections optional — only included sections update on import):

```json
{
  "$schema": "cog-ingest-config-v1",
  "engine": {
    "llm": { "provider": "anthropic", "model": "claude-sonnet-4-20250514" },
    "embedding": { "provider": "voyage", "model": "voyage-3" }
  },
  "processing": {
    "defaultLevel": "standard",
    "mode": "sync",
    "batchSize": 10,
    "patternGrowthMode": "off",
    "patternGrowthThreshold": 0.85
  },
  "features": {
    "understanding": true,
    "patterns": true,
    "proactive": true,
    "entityExtraction": true,
    "relationshipDiscovery": true,
    "questionDetection": true,
    "emotionDetection": true
  },
  "prompts": {
    "systemContext": "",
    "contentDescription": "",
    "customPrompts": {
      "entityExtraction": null,
      "contentUnderstanding": null,
      "relationshipExtraction": null,
      "patternDescription": null,
      "connectionExplanation": null,
      "insightGeneration": null
    }
  },
  "extraction": {
    "visualAnalysisEnabled": false,
    "urlScreenshots": false,
    "screenshotViewport": { "width": 1440, "height": 900 },
    "screenshotMaxHeight": 5000,
    "pdfPageRendering": false,
    "pdfSamplePages": "first-last-every-5th",
    "pdfMaxPagesToRender": 20,
    "enhancedImageAnalysis": false
  }
}
```

<Aside type="note" title="Vision model choice moved to AI Core (v1.13.0)">
The `visionAnalysisModel` key was removed. Vision analysis runs as named [AI Core](/modules/ai-core/) operations (`cog-ingest/analyse-image`, `-image-enhanced`, `-url-screenshot`, `-pdf-page`), so the model — and the prompt — are chosen in the AI Core admin. The flags above still decide *whether* visual analysis runs; AI Core decides *what runs it*.
</Aside>

## Source Configuration (Per-Source)

From v1.5.0, each ingest source has a rich, typed configuration stored in its `config` JSON field. Admins configure this via the **Settings** button on each source row in the admin page, which opens a multi-tab config dialog.

From v1.8.0, the tab layout is: **General | Input | Acquisition | Feed* | Context | Extraction | Prompts | Actions** (*Feed tab only appears when RSS is enabled in Input).

The source config controls:
- **Input** — which content types are accepted, with per-type validation settings
- **Acquisition** — how raw content becomes extractable text (parsing, screenshots, OCR)
- **Context fields** — user-facing form fields shown on the ingest page
- **Entity & relationship types** — rich definitions for what the AI should extract
- **Output schema** — the structured JSON shape the AI must return
- **Prompt overrides** — custom instructions for the LLM
- **Auto-tags** — automatic tag application rules
- **Actions** — post-ingest field mapping with output schema reference
- **UI** — icon and colour for the source badge

All sections are optional — an empty `config: {}` uses engine defaults. No schema migration is needed; everything goes into the existing JSON field.

### Context Fields

Context fields are the primary feature. They define form fields that appear on the ingest page when a user selects a source. Values from fields with usage `'context'` are assembled into a labelled section in the extraction prompt, giving the LLM critical human-provided context.

| Property | Type | Required | Default | Description |
|----------|------|----------|---------|-------------|
| `key` | `string` | Yes | — | camelCase identifier (auto-generated from label) |
| `label` | `string` | Yes | — | Display label |
| `description` | `string` | No | — | Help text / placeholder |
| `type` | enum | Yes | — | `text`, `textarea`, `select`, `multiselect`, `date`, `number`, `boolean`, `tags` |
| `options` | `{value, label}[]` | For select/multiselect | — | Option list |
| `required` | `boolean` | No | `false` | Prevents submission until filled |
| `defaultValue` | mixed | No | — | Pre-populated when user selects the source |
| `usage` | enum | No | `'context'` | `context` (→ prompt), `metadata` (→ stored only), `filter` (→ filtering) |
| `order` | `number` | No | `0` | Display order on the form |

**How context fields flow:**

1. Admin configures fields on a source via the config dialog
2. User selects that source on the ingest form → dynamic fields appear
3. User fills in values alongside their content
4. Backend validates required fields are provided
5. Fields with `usage: 'context'` are assembled into a labelled prompt section:
   ```
   ADDITIONAL CONTEXT PROVIDED BY USER:
   - Case Number: VCC-2025-001
   - Court: County Court of Victoria
   - Filing Type: Affidavit
   ```
6. This context string is passed to cognitive-db as `IngestInput.context`
7. Fields with `usage: 'metadata'` are stored in the job metadata only

### Input Config (v1.8.0)

Per-content-type validation settings. Each enabled content type can have its own config:

**Text:** `minContentLength` (default 10), `maxContentLength` (default 100000), `inputMode` (`paste` | `upload` | `both`)

**URL:** `mode` (`single` | `batch`), `maxUrls` (default 10), optional `allowedDomains` and `blockedDomains` arrays

**PDF:** `maxFileSize` in MB (default 50), `maxPages` (default 200), `allowScanned` toggle for OCR

**Image:** `allowedFormats` array (`png`, `jpg`, `webp`, `gif`), `maxFileSize` in MB (default 20)

### Acquisition Config (v1.9.0 — Two-Layer)

The acquisition tab has two distinct jobs: **field scraping** (deterministic CSS selector extraction) and **content selection** (what goes to AI). Only sections for enabled content types are shown.

**URL Acquisition:**
- **Field Scraping** — `ScrapedFieldRule[]`: CSS selector-based extraction for structured data (case name, citation, judge, etc.). Each rule has: `key`, `label`, `selector`, `type` (text/date/number/list/html), optional `attribute` extraction, optional `transform` (regex, prefix/suffix), `required` flag, `structureFieldKey` for auto-mapping.
- **Content Selection** — `method` controls what goes to AI: `selector` (CSS selector for content area), `exclude_scraped` (everything except scraped regions), `full_page` (Readability parse). Additional `removeSelectors` and `preserveFormatting` options.
- **Visual Capture** — optional screenshot with viewport, full-page, and wait settings.
- **"Add from Structure"** — dropdown lists fields from the Structure node type, pre-populates key/label/type. Fields already assigned are greyed out.

**RSS Acquisition:** entry content strategy (`entry_only` / `fetch_page` / `both`), linked page config (`inherit` URL config or custom), entry fields to include.

**PDF Acquisition:** text extraction method, layout preservation, page rendering, optional `scrapedFields` for structured PDF data.

**Image Acquisition:** vision analysis type, EXIF, colour palette, OCR.

### Extraction (v1.8.0 — Reworked)

The extraction tab is now organised into four accordion sections:

**Entity Type Definitions** — rich definitions replacing the old comma-separated `priorityEntityTypes`. Each entity type has: `name`, `description`, `priority` (high/normal/ignore), typed `attributes` (key, label, type, required, enum values), and optional `examples`.

**Relationship Type Definitions** — replaces `priorityRelationshipTypes`. Each type has: `name`, `description`, `priority`, `fromTypes` and `toTypes` arrays constraining which entity types can participate.

**Output Schema** — defines the JSON structure the AI must return. Format is `sections` (default), `flat`, or `custom`. Each section has a key, label, description, type (object/list/text/key_value), and typed fields with extraction hints. Required sections can be toggled. A preview shows the expected JSON shape.

**Feature Toggles** — `extractDates` (default true), `extractEmotions` (default true), `extractQuestions` (default false), `extractActionItems` (default false).

### Prompt Overrides

Custom instructions for the LLM when processing content from this source.

| Property | Type | Max Length | Description |
|----------|------|-----------|-------------|
| `systemContext` | `string` | 4000 | Prepended to the system prompt — describe the domain |
| `contentDescription` | `string` | 2000 | Describes what the content IS — helps the model understand |
| `extractionInstructions` | `string` | 4000 | Appended to the extraction prompt — specific extraction instructions |

### Auto-Tag Rules

Automatically apply tags to ingested content based on rules.

```json
{
  "autoTags": [
    { "tag": "legal", "when": "always" },
    { "tag": "confidential", "when": "condition", "conditionField": "confidential", "conditionValue": true }
  ]
}
```

Rules with `when: 'always'` add the tag to every ingest from this source. Rules with `when: 'condition'` check a context field value — the tag is only applied if the field's value matches `conditionValue`.

### UI Config

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `icon` | `string` | `'file-text'` | Lucide icon name (e.g. `gavel`, `folder-open`) |
| `colour` | `string` | `'#6366f1'` | Hex colour for the source badge |

### Full Example

A "Court Filing" source with context fields, extraction focus, prompt overrides, and auto-tags:

```json
{
  "ui": { "icon": "gavel", "colour": "#dc2626" },
  "contextFields": [
    {
      "key": "caseNumber", "label": "Case Number", "type": "text",
      "required": true, "usage": "context", "order": 0
    },
    {
      "key": "court", "label": "Court", "type": "select",
      "options": [
        { "value": "county_vic", "label": "County Court of Victoria" },
        { "value": "supreme_vic", "label": "Supreme Court of Victoria" }
      ],
      "required": true, "usage": "context", "order": 1
    },
    {
      "key": "filingType", "label": "Filing Type", "type": "select",
      "options": [
        { "value": "affidavit", "label": "Affidavit" },
        { "value": "defence", "label": "Defence" }
      ],
      "required": true, "usage": "context", "order": 2
    },
    {
      "key": "confidential", "label": "Confidential", "type": "boolean",
      "defaultValue": false, "usage": "metadata", "order": 3
    }
  ],
  "extraction": {
    "priorityEntityTypes": ["person", "organisation", "event"],
    "ignoreEntityTypes": ["colour"],
    "extractEmotions": false,
    "extractActionItems": true
  },
  "prompts": {
    "systemContext": "You are analysing Australian court filings. Preserve legal terminology exactly.",
    "extractionInstructions": "Extract all parties. Identify court orders and directions. Note all dates and cited legislation."
  },
  "autoTags": [
    { "tag": "legal", "when": "always" },
    { "tag": "confidential", "when": "condition", "conditionField": "confidential", "conditionValue": true }
  ]
}
```

With this configuration, when a user selects "Court Filing" and fills in the context fields, the backend assembles a context string like:

```
CONTENT TYPE: Court Filing
Legal filings lodged with Australian courts

FOCUS ON THESE ENTITY TYPES: person, organisation, event

IGNORE THESE ENTITY TYPES: colour

ADDITIONAL CONTEXT PROVIDED BY USER:
- Case Number: VCC-2025-001
- Court: County Court of Victoria
- Filing Type: Affidavit

ADDITIONAL INSTRUCTIONS:
Extract all parties. Identify court orders and directions. Note all dates and cited legislation.
```

## Extraction Configuration

The extraction section controls the deep multi-modal extraction pipeline. All visual analysis features are **off by default** and must be explicitly enabled.

| Setting | Default | Description |
|---------|---------|-------------|
| `visualAnalysisEnabled` | `false` | Master toggle — must be on for any visual analysis |
| `urlScreenshots` | `false` | Capture page screenshots for URL content (requires Puppeteer) |
| `pdfPageRendering` | `false` | Render PDF pages as images for visual analysis |
| `enhancedImageAnalysis` | `false` | Use richer structured vision prompts for images |
| `pdfSamplePages` | `'first-last-every-5th'` | Which pages to render: `first-only`, `first-last`, `first-last-every-5th`, `all` |
| `pdfMaxPagesToRender` | `20` | Maximum pages to send through vision analysis |
| `screenshotViewport` | `{ width: 1440, height: 900 }` | Browser viewport for URL screenshots |
| `screenshotMaxHeight` | `5000` | Max pixel height for full-page screenshots |

<Aside type="caution" title="Puppeteer is optional">
URL screenshot capture requires Puppeteer, which is NOT installed by default (~400 MB with Chrome). Install it separately: `npm install puppeteer`. If not installed, URL processing falls back to text-only extraction with a log warning.
</Aside>

## Config File Reference

The config file initialises cognitive-db with your app's Prisma client and provider settings. It is generated at `backend/src/features/cog-ingest/cog-ingest.config.ts` and runs as a side-effect import from the feature barrel file.

```typescript
// Key exports from cog-ingest.config.ts
export const llmModel;             // AI SDK model instance
export const cogIngestConfig;      // Runtime settings object
export async function initCogIngest();    // Initialises cognitive-db (loads DB overrides)
export async function reinitialiseEngine(); // Re-applies config at runtime
```

The `initCogIngest()` function is called during backend startup via the `hasInit: true` flag in the module definition. It:

1. Loads static defaults from the config file
2. Queries `cog_ingest_config` for runtime overrides
3. Merges overrides on top of defaults
4. Configures cognitive-db with database, LLM, embedding, and features
5. Logs which sections are using overrides vs defaults
6. Seeds the `working-memory` Structure node type if it doesn't already exist

<Aside type="note" title="Working-memory node type">
The `working-memory` node type is seeded automatically at backend boot as a shared row (`userId: null`). Admins can customise it afterwards — the seed is guarded by an existence check and never overwrites admin changes. If boot-time seeding fails, the type is created lazily on first working-memory use.
</Aside>

## Database Schema

The module creates 8 Prisma models — 3 cognitive-db engine tables and 5 module-specific tables.

### Engine Tables

| Model | Table Name | Purpose |
|-------|-----------|---------|
| `cog_node` | `cog_nodes` | Unified node model — entities, connections, patterns, ontology |
| `cog_working_memory_event` | `cog_working_memory_events` | Working memory tracking for sessions |
| `cog_processing_log` | `cog_processing_logs` | Processing operation log with stats |

### Module Tables

| Model | Table Name | Purpose |
|-------|-----------|---------|
| `cog_ingest_source` | `cog_ingest_sources` | Admin-configured ingest source definitions |
| `cog_ingest_job` | `cog_ingest_jobs` | Individual ingest job records with results |
| `cog_ingest_mapping` | `cog_ingest_mappings` | Source-to-Structure-node-type mappings |
| `cog_ingest_config` | `cog_ingest_configs` | Runtime configuration overrides (key-value JSON) |
| `cog_ingest_rss_entry` | `cog_ingest_rss_entries` | RSS/Atom feed entry tracking — prevents duplicate ingestion, supports review workflows |

<Aside type="caution">
Engine tables use literal names (not the `my_feature` placeholder pattern) because the cognitive-db runtime depends on the exact column layout. The module tables also use literal names since they share the `cog_` prefix.
</Aside>

### Key Fields

**cog_ingest_source:**
- `slug` — URL-friendly identifier (unique per user)
- `processingLevel` — `'minimal'`, `'standard'`, or `'deep'`
- `ontologyGrowthMode` — `'off'`, `'propose'`, or `'auto'`
- `config` — JSON field for per-source settings

**cog_ingest_job:**
- `sourceId` — references the ingest source used
- `status` — `'queued'`, `'processing'`, `'completed'`, or `'failed'`
- `result` — full JSON result from cognitive-db
- `structureNodeId` — optional link to a Structure node

**cog_ingest_mapping:**
- `sourceId` + `structureNodeTypeSlug` — unique compound key
- `entityTypes` / `relationshipTypes` — JSON arrays of types to extract
- `customPrompt` — optional LLM prompt for extraction context

**cog_ingest_config:**
- `key` — unique config section key (`'engine'`, `'processing'`, `'features'`, `'prompts'`, `'extraction'`)
- `value` — JSON object with the section's configuration values
- `userId` — who last modified the config (audit trail)
