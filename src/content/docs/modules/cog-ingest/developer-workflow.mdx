## Setup

### Clone and Install

```bash
git clone <cog-ingest-repo-url>
cd cognitive-ingest
npm install
```

### Build

```bash
npm run build     # One-time build
npm run dev       # Watch mode (rebuilds on file changes)
```

### Link for Local Testing

```bash
npm link          # Makes 'cognitive-ingest' command available globally
```

<Aside type="note">
You also need to link `cognitive-db` locally if it's not published to npm. In the cognitive-db repo: `npm run build && npm link`. Then in the test app's backend: `npm link cognitive-db`.
</Aside>

## Development Loop

The standard workflow when making changes:

```
Edit templates → Build → Test in app → Version bump → Commit → Push
```

### 1. Edit Templates

All your work happens in these locations:

| Location | What You Edit |
|----------|--------------|
| `src/templates/frontend/` | Frontend component, hook, and service templates |
| `src/templates/backend/` | Backend route, service, and registration templates |
| `src/templates/schema.ts` | Prisma schema (database tables) |
| `src/module.ts` | Module configuration (routes, dependencies, features list) |

<Aside type="caution">
Never edit files in `src/scaffold/` or `src/cli.ts`. These are shared infrastructure — report bugs upstream to the module starter template.
</Aside>

### 2. Build

```bash
npm run build
```

This compiles TypeScript to the `dist/` folder. Template string constants are bundled as-is — they're not transpiled, just included in the build output.

### 3. Test in a Starter App

In a separate terminal, navigate to a starter template app:

```bash
# Fresh install
npx cognitive-ingest init

# Update existing installation (after template changes)
npx cognitive-ingest update

# Preview what would change (dry run)
npx cognitive-ingest update --dry-run

# Force overwrite all files
npx cognitive-ingest init --force

# Check installed version and module info
npx cognitive-ingest info
```

### 4. Version Bump

After verifying your changes work, bump the version:

```bash
npm run version:patch    # Bug fix, template fix
npm run version:minor    # New feature, new template
npm run version:major    # Breaking change
```

This updates `package.json` and `src/module.ts` and creates a commit.

### 5. Commit and Push

```bash
git add src/templates/ src/module.ts
git commit -m "feat: description

Co-Authored-By: Claude <noreply@anthropic.com>"
git push
```

<Aside type="caution" title="Commit directly to main">
This is an upstream module repo. Downstream apps pull changes via `npm update`. Commit directly to `main` — feature branches delay changes reaching downstream.
</Aside>

## Version Management

### When to Bump

**Always bump after ANY change to templates or module.ts.** This is how the update system detects changes.

| Changed File | Bump Required? |
|-------------|----------------|
| `src/templates/**/*.ts` | ✅ Yes |
| `src/module.ts` | ✅ Yes |
| `CLAUDE.md`, `README.md` | ❌ No |
| `docs/**` | ❌ No (push docs submodule instead) |

### Version Commands

| Command | When to Use |
|---------|------------|
| `npm run version:patch` | Bug fixes, template fixes, minor adjustments (no changelog needed) |
| `npm run version:minor` | New features, new templates, enhancements (changelog required) |
| `npm run version:major` | Breaking changes requiring app migration (changelog required) |

### Tagged Releases

For publishing to npm with git tags:

```bash
npm run release:patch    # Bumps, commits, and creates git tag
npm run release:minor
npm run release:major
```

After a tagged release, push tags: `git push --tags`.

### Why Bumping Matters

1. **Version headers** — generated files include the module version in their headers. Without a bump, regenerated files show the old version.
2. **Update detection** — `npx cognitive-ingest update` compares the installed version against the module version. No bump = no update detected.
3. **Hash comparison** — the update system compares file content hashes. But it only checks if the module version has changed first.

## Scaffold Updates

The module's infrastructure (CLI, init logic, update system) comes from the **module starter template**. You can pull improvements with scaffold updates.

### Check for Updates

```bash
npm run scaffold:check
```

This shows what's available without changing anything. Output tells you your current scaffold version and whether updates are available.

### Apply Updates

```bash
npm run scaffold:update
```

This fetches from the scaffold upstream remote and merges changes into your repo.

### What Gets Updated

| Updated | Not Updated |
|---------|-------------|
| `src/scaffold/**` (init, update, types, utils) | `src/templates/**` (your templates) |
| `src/cli.ts` | `src/module.ts` (your config) |
| `scripts/**` | `CLAUDE.md` (module-specific sections) |
| CLAUDE.md scaffold sections (between markers) | Your customisations |
| `tsup.config.ts`, `tsconfig.json` | `package.json` (name, description, bin) |

### Handling Merge Conflicts

Scaffold updates use `git merge`, so conflicts are possible if you've modified scaffold files (which you shouldn't have). If you get conflicts:

1. Check if the conflicting file is in `src/scaffold/` — if so, accept the upstream version
2. If the conflict is in a shared config file, review both sides and merge manually
3. Run `npm run build` after resolving to verify

## Common Development Scenarios

### Adding a New Frontend Page

1. Create the template string in `src/templates/frontend/pages.ts`:
   ```typescript
   export const MY_NEW_PAGE_TEMPLATE = `// ========================================================================
   // FILE: frontend/src/features/my-feature/pages/MyNewPage.tsx
   // PURPOSE: Description
   // ========================================================================
   // ... component code
   `;
   ```

2. Export it from `src/templates/frontend/index.ts`

3. Register it in `src/module.ts` under `frontend.templates`:
   ```typescript
   'pages/MyNewPage.tsx': frontend.MY_NEW_PAGE_TEMPLATE,
   ```

4. If it needs a route, add it to `frontend.routes.protectedRoutes`

5. Remember: no typography scale or font weight classes in template code — use semantic tokens only

### Adding a New Backend Endpoint

1. Add the route handler to the routes template in `src/templates/backend/routes.ts`

2. Add the service method to the service template in `src/templates/backend/services.ts`

3. If the endpoint needs new validation, add Zod schemas to `src/templates/backend/types.ts`

4. Remember: use `.js` extensions in all relative imports

### Adding a New Database Table

1. Add the model to `src/templates/schema.ts`

2. Follow module database conventions:
   - Plain `userId String` foreign keys (no `@relation`)
   - `@@index` on foreign key columns
   - `@@map("table_name")` for the table name
   - `cog_` prefix for consistency with engine tables

3. Add the model name to `schema.modelNames` in `src/module.ts`

### Adding a New Registry Registration

1. Create the registration template in `src/templates/frontend/registrations.ts` (or backend equivalent)

2. Export it from the template index file

3. For core starter registries, add to `frontend.registrations` in `src/module.ts`:
   ```typescript
   {
     type: 'dashboard',
     path: 'registrations/my-registration.ts',
     content: frontend.MY_REGISTRATION,
   },
   ```

4. For module-specific registries (like Structure's), add as a regular template file instead

5. Add the side-effect import to the index template

### Adding a New Environment Variable

1. Add it to `envVariables` in `src/module.ts`:
   ```typescript
   envVariables: {
     backend: [
       {
         name: 'MY_NEW_KEY',
         description: 'Description shown during init',
         required: true,
         secret: true,
         when: (answers) => answers.somePrompt === 'value',
       },
     ],
   },
   ```

2. Use it in your config or service template via `process.env.MY_NEW_KEY`

3. Frontend variables automatically receive a `VITE_` prefix

### Adding a Prompt Variable

1. Add the prompt to `prompts` in `src/module.ts`:
   ```typescript
   {
     id: 'myVariable',
     type: 'select',
     question: 'Choose an option:',
     options: [
       { value: 'a', label: 'Option A' },
       { value: 'b', label: 'Option B' },
     ],
     default: 'a',
   },
   ```

2. Use it in templates with `{{myVariable}}` or conditionals:
   ```typescript
   {{#if myVariable === 'a'}}
   // Code for option A
   {{else}}
   // Code for option B
   {{/if}}
   ```

3. Use it for conditional env variables with the `when` callback

### Adding Lifecycle Hooks

Add a `lifecycle` property to your module definition:

```typescript
lifecycle: {
  onInstall: async (context) => {
    // Run after all files are generated
  },
  onUpdate: async (context) => {
    // Run after update applies changes
  },
  onUninstall: async (context) => {
    // Run during uninstall
  },
},
```

## Debugging and Troubleshooting

### Inspecting Generated Output

After running `npx cognitive-ingest init` in a test app, check:

- `frontend/src/features/cog-ingest/` — all frontend files
- `backend/src/features/cog-ingest/` — all backend files
- `backend/prisma/schema.prisma` — schema additions

Compare generated files against your template strings to verify placeholder replacement worked correctly.

### Common Template Mistakes

| Symptom | Cause | Fix |
|---------|-------|-----|
| `my-feature` appears literally in generated code | Placeholder not being replaced | Check you're using the correct case variant (`my-feature`, `myFeature`, `MyFeature`) |
| Backtick syntax error in generated file | Unescaped backtick in template | Escape as `` \` `` inside template strings |
| `${variable}` evaluated at build time | Unescaped template expression | Escape as `\${variable}` |
| Import not found in generated backend | Missing `.js` extension | Add `.js` to all relative imports in backend templates |

### Build Errors

| Error | Meaning | Fix |
|-------|---------|-----|
| `Cannot find module './templates/...'` | Missing export from template index | Add the export to `src/templates/frontend/index.ts` or `backend/index.ts` |
| Type error in module.ts | Template constant doesn't match expected type | Check the template is exported as a `string` constant |

### Update Not Detecting Changes

If `npx cognitive-ingest update` says "already up to date" but you've changed templates:

1. **Did you bump the version?** The update system checks version first. Run `npm run version:patch`.
2. **Did you rebuild?** Run `npm run build` after bumping.
3. **Is the link stale?** Run `npm link` again in the module repo.
4. **Check with info:** Run `npx cognitive-ingest info` to see the installed vs available version.

### Verifying Backend Import Extensions

All backend template imports must use `.js` extensions for ESM compatibility:

```typescript
// ✅ Correct
import { prisma } from '../../lib/prisma.js';
import { myFeatureService } from './my-feature.service.js';

// ❌ Wrong — will fail at runtime
import { prisma } from '../../lib/prisma';
import { myFeatureService } from './my-feature.service';
```

## Documentation Workflow

The docs live in a git submodule at `docs/`. They deploy automatically via GitHub Actions when pushed.

### Making Documentation Changes

1. Edit `.mdx` files in `docs/src/content/docs/modules/cog-ingest/`
2. Push the docs submodule:
   ```bash
   cd docs
   git add .
   git commit -m "Docs: description of change"
   git push
   cd ..
   ```
3. Update the submodule reference in the parent repo:
   ```bash
   git add docs
   git commit -m "docs: update docs submodule"
   git push
   ```

### Adding a New Docs Page

1. Create the `.mdx` file in the module's docs directory
2. Add the sidebar entry in `docs/astro.config.mjs` under the Cog Ingest section
3. Follow the push workflow above

### Sidebar Configuration

The sidebar is configured in `docs/astro.config.mjs`. Each module has its own section under the "Modules" label with entries for each page.
