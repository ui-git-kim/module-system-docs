This page covers what you work with after installing the Style module — the two project-theme tabs, the visual className designer, the admin theme page, and the hooks, service, and API endpoints that back them.

What you get depends on the `installMode` prompt you answered at install:

<CardGrid>
  <Card title="Admin mode" icon="setting">
    Just the app-wide **Theme Settings** page at `/admin/theme`. No `builder` or `structure` dependency. The designer and node tabs are not generated.
  </Card>
  <Card title="Full mode (default)" icon="puzzle">
    Everything in Admin mode **plus** the builder **Style** property-panel tab, the two **Shadcn Config** + **Theme** structure node tabs, the `theme_config` mirror field, and the builder-canvas CSS filter.
  </Card>
</CardGrid>

<Aside type="note" title="Placeholders resolve to `style`">
Templates use the `my-feature` placeholder, which is replaced with the module id at install. In a real app the service is `styleService`, the API base is `/api/v1/style`, query keys are `['style', …]`, and the theme-saved event is `style:theme-saved`. This page uses the resolved names.
</Aside>

## Project Theme Tabs (full mode)

In a full install the module contributes two tabs to project structure nodes through structure's `nodeTabRegistry`. Both attach to the node type(s) in `PROJECT_NODE_TYPES` — set from the `projectNodeType` prompt (default `projects`) and editable in `registrations/structure.registration.ts`. They share **one** stored `StyleConfig`, read as two layers (see [Design philosophy](/modules/style/design/) for the model):

- **Shadcn Config tab** edits Layer 1 — the shadcn preset — which resolves to the base palette.
- **Theme tab** edits Layer 2 — hand-tuned colour tokens and non-palette variables — on top of that base.

The preset controls map to these `StyleConfig` fields:

| Field | Values | Edited in |
|-------|--------|-----------|
| `componentLibrary` | `radix` \| `base` | Shadcn Config |
| `styleName` | `nova` \| `vega` \| `maia` \| `lyra` \| `mira` \| `luma` \| `sera` \| `rhea` | Shadcn Config |
| `baseColor` | `neutral` \| `stone` \| `zinc` \| `mauve` \| `olive` \| `mist` \| `taupe` | Shadcn Config |
| `colorPreset` | theme colour name (e.g. `blue`, `emerald`, `neutral`) | Shadcn Config |
| `chartColor` *(optional)* | theme colour name | Shadcn Config |
| `iconLibrary` | `lucide` \| `hugeicons` \| `phosphor` \| `tabler` \| `remixicon` | Shadcn Config |
| `font`, `fontHeading` *(optional)* | font id (`inter`, `geist`, …) | Shadcn Config |
| `radius` | `default` \| `none` \| `small` \| `medium` \| `large` | both tabs |
| `menuColor` | `default` \| `inverted` \| `default-translucent` \| `inverted-translucent` | Shadcn Config |
| `menuAccent` | `subtle` \| `bold` | Shadcn Config |
| `sidebarAccent` | `subtle` \| `bold` *(module extension — not a shadcn preset field)* | Shadcn Config |
| `cssVars` | `{ light, dark, custom }` token maps | Theme |
| `savedColors`, `savedGradients` *(optional)* | swatch/gradient recall | Theme |
| `defaultStyling` *(optional)* | `data-slot` default rules | (theme output — authoring UI planned) |

### Shadcn Config tab

`ShadcnConfigTab.tsx` is a side-by-side editor: a settings panel (`ShadcnSettingsPanel`) on the left and a **live iframe preview** (`ThemePreviewFrame`) on the right that renders the effective palette, font, radius, menu, and sidebar accent as you change them. Picking a `styleName` cascades that style's shadcn defaults (`STYLE_DEFAULTS`) alongside it.

The toolbar exposes:

| Control | What it does |
|---------|--------------|
| Preset code + copy | Shows the current base62 preset code (`encodePreset`); the copy button writes it to the clipboard. Codes round-trip with `shadcn.com/create`. |
| Paste preset + **Apply** | Decodes a pasted preset code (`decodePreset`) and applies it through the two-layer flow. Invalid codes show an inline error. |
| **CSS** toggle | Reveals the generated `global.css` snippet (`GlobalCssSnippet`). |
| **History** | The shadcn-config (preset-layer) history popover — see below. |
| **Revert** | *Reset to base snapshot* (last-saved `StyleConfig`) or *Reset to shadcn defaults* (re-applies the `nova` preset). |
| **Update** | Persists the config; the button reflects dirty/saving/saved/error state. |

#### Ask-per-change rebase

Changing a palette-affecting field (`baseColor`, `colorPreset`, `chartColor`, `menuAccent`, or `styleName`) regenerates the base palette. What happens next depends on whether the project has a custom delta:

<Steps>
1. **No palette impact** — the field is applied directly.
2. **Palette impact, no custom delta** — the field is applied and the base palette is silently reseeded (`reseedCssVars`).
3. **Palette impact with a custom delta** — the field is applied so the controls reflect the choice, then a **"Keep your custom colours?"** prompt asks how to treat the hand-tuned tokens.
</Steps>

The prompt offers three choices:

| Choice | Effect |
|--------|--------|
| **Rebase** | Recompute the base from the new preset and re-apply the delta on top (`rebaseCssVars`) — keep your edits on the new palette. |
| **Keep** | Leave the colours exactly as they are; only the preset field changes. |
| **Discard** | Take the clean new base and drop the delta (`reseedCssVars`). |

Restoring a preset snapshot from history and pasting a preset code both flow through this same check.

#### Shadcn-config history

The tab keeps its **own** history, distinct from the Theme tab's combined version snapshots. It lists preset-layer snapshots (`getShadcnConfigVersions`) with relative timestamps and editable labels; restoring one re-applies its preset through the rebase flow above, so custom colours are preserved unless you discard them.

### Theme tab

`ProjectThemeTab.tsx` is a split-panel visual editor — controls on the left (three section tabs), the same live `ThemePreviewFrame` on the right. Switching the preview between light and dark drives which mode the controls edit.

| Section | Contents |
|---------|----------|
| **Colours** | Per-token colour editing for the active mode (`ThemeColorSection`). Editing a token auto-generates its **opposite-mode counterpart** (`generateDarkCounterpart` / `generateLightCounterpart`), so a light edit updates dark and vice versa. Includes saved colours and saved gradients recall. |
| **Typography** | Font family/heading, letter-spacing, radius, and custom CSS variables (`ThemeLayoutSection`). Mode-independent keys (`font-sans`, `font-mono`, `font-serif`, `spacing`, `letter-spacing`) update both light and dark at once. |
| **Effects** | Spacing scale and shadow tokens (mode-dependent), radius, and custom CSS variables. |

Custom CSS variables are edited here and stored on `cssVars.custom`; radius is a top-level `StyleConfig.radius` field.

**Import / Export JSON** — export writes the whole `cssVars` object (`light`, `dark`, `custom`) to `theme.json`; import expects an object with `light` and `dark` maps (and optionally `custom`) and rejects anything else with an inline error.

**Version history** is the tab's combined snapshot store (`getVersions` → `restoreVersion` / `updateVersionLabel`) — the full `StyleConfig`, distinct from the Shadcn Config tab's preset-only history.

<Aside type="tip" title="Live builder preview">
Saving a project config dispatches a `style:theme-saved` window event (`{ detail: { projectId } }`). The builder-canvas filter listens for it and refetches the project's CSS, so the builder's iframe preview updates without a page refresh. See the [builder integration](/modules/builder/integration/) page.
</Aside>

## Visual className Designer

In a full install the module contributes a **Style** tab to the builder's property panel via `propertyPanelTabRegistry`. `StylePanel.tsx` receives `{ node, block, onUpdate }` and edits the selected node's `className` string (a direct property on the node, not nested in `props`). It renders nine collapsible sections:

| Section | Purpose |
|---------|---------|
| **Layout** | Display, flex/grid, direction, alignment, wrap, gap |
| **Sizing** | Width, height, min/max |
| **Spacing** | Padding and margin |
| **Typography** | Font size, weight, alignment, colour, line height |
| **Background** | Background colour, opacity, gradient |
| **Border & Shadow** | Border width/style/colour, radius, shadow, ring |
| **Effects** | Opacity, cursor, transition, animation |
| **Position** | Position, inset, z-index |
| **Classes & Tokens** | Raw class entry (`CustomClassInput`) with merge feedback |

**Layout** is open by default. Each section reads/writes a slice of a structured `StyleState`.

**Bidirectional sync** — `useStyleState` parses the incoming `className` into a `StyleState` (`classesToStyleState`) and, when you edit a section, converts it back to a class string (`styleStateToClasses`) after a 150 ms debounce, delivering it through `onClassNameChange`. `StylePanel` then clones the block tree, mutates the node's `className`, and calls `onUpdate` — the same clone → mutate → update pattern the builder's Properties tab uses.

**tailwind-merge** — the generated class output runs through `twMerge`, so a generated `p-4` and a hand-typed `p-2` resolve deterministically (the last-appended class wins its group) instead of both surviving.

**Base-class merge + variant warnings** — the panel reads `projectId` from the builder route (`?projectId=`) and fetches the builder's per-project base-class manifest via `useBaseClassManifest`. When the node's `type` matches a `data-slot` in the manifest, `computeClassMergeInfo` resolves the component's active cva variant, `twMerge`s the shipped base classes against your classes, and **warns on variant-group mismatches** the merge cannot fix — e.g. `max-w-2xl` can't beat a shipped `sm:max-w-sm` (different conflict groups), so the Classes field suggests `sm:max-w-2xl` with one-click apply. With no manifest (builder absent, no snapshot, or no matching slot) it degrades to merging the designer's own output only, with no false indicators.

<Aside type="note" title="Manifest source">
The manifest comes from the builder: `GET /api/v1/builder/projects/:id/component-set` → `set.baseClasses`, keyed by `data-slot` (builder v1.18.0+). See [builder integration](/modules/builder/integration/).
</Aside>

## Admin Theme Settings Page

Available in **both** install modes at `/admin/theme` (registered through `adminPagesRegistry`). `AppThemeSettingsPage.tsx` drives the **app-wide** theme via `useProjectTheme({ appWide: true })` — the same settings panel, preset toolbar, CSS toggle, revert, and live preview as the Shadcn Config tab, but persisting the single active app theme rather than a project config.

- **Save** persists the config and immediately applies it to the running app through `styleRegistry.update(STYLE_REGISTRY_ID, { css })`, so changes take effect without a reload.
- `useMenuColorBinding` runs live, so dropdown/popover inversion is visible in the preview as you change `menuColor`.
- Selecting **Sidebar accent: Bold** toggles the `sidebar-accent-bold` catalogue entry via `styleRegistry.setOverride(...)`.
- Picking a `styleName` cascades that style's shadcn defaults.

<Aside type="caution" title="Admin only">
The app-wide write endpoints (`PUT /app/config`, `PUT /app/activate/:id`) require an admin role. The page is registered as an admin page accordingly.
</Aside>

## Hooks

All hooks live under `features/style/hooks/`.

### `useStyleState`

Bidirectional sync between the visual `StyleState` and a node's `className`. Backs the designer.

```typescript
const {
  styleState,          // structured StyleState (layout, sizing, …, customClasses)
  updateSection,       // (section, value) => void — update one section
  updateCustomClasses, // (classes: string[]) => void — replace the raw class list
  generatedClassName,  // the current generated class string
  reset,               // re-parse from the className prop
} = useStyleState({
  nodeId,              // builder node id
  className,           // current className on the node
  onClassNameChange,   // (className: string) => void — emitted (debounced 150ms)
  customTokens,        // optional: project custom token names so twMerge groups bg-<token>
});
```

### `useProjectTheme`

Read/write access to a project config (`{ nodeId }`) or the app-wide config (`{ appWide: true }`). Loads from the API on mount and persists via `save`.

```typescript
const {
  theme,             // StyleConfig | null (null while loading)
  updateTheme,       // (updates: Partial<StyleConfig>) => void
  setCssVars,        // (mode, vars) => void — merge tokens into light/dark
  updateCssVars,     // (mode, vars) => void — partial token merge
  addCustomVariable, // (variable) => void
  removeCustomVariable, // (name) => void
  setCustomVariables,   // (variables: CustomVariable[]) => void
  baseSnapshot,      // full StyleConfig at load/save time
  loadTheme,         // (config) => void — e.g. after restoring a version
  resetToBase,       // revert cssVars to baseSnapshot
  effectiveCssVars,  // merged ThemeConfig used by the preview
  globalCss,         // generated global.css string
  isLoading, isDirty, isSaving,
  savedConfig,       // last-saved config (for per-field changed indicators)
  revertField,       // (key) => void — revert one field
  revertAll,         // revert every field to saved
  save,              // (label?) => Promise<boolean>
  saveError, loadError, // save is blocked while loadError is set
  isNewConfig,       // never persisted yet
  configName, setConfigName,
} = useProjectTheme({ nodeId /* or */ appWide: true });
```

### `useThemeInjection`

Injects theme CSS variables into the builder's preview iframe (`:root` / `.dark` blocks only, via `generateInjectableCSS`). Returns nothing.

```typescript
useThemeInjection({
  iframeRef, // React.RefObject<HTMLIFrameElement | null>
  theme,     // StyleConfig | null
  enabled,   // default true
});
```

### `useThemePresets`

CRUD wrapper around the saved-theme presets API (used by the parked Style Library page).

```typescript
const {
  themes, isLoading, error,
  loadThemes,  // () => Promise<void>
  saveTheme,   // (input: CreateThemeInput) => Promise<SavedTheme>
  updateTheme, // (id, input: UpdateThemeInput) => Promise<SavedTheme>
  deleteTheme, // (id) => Promise<void> — optimistic, rolls back on failure
  applyTheme,  // (theme: SavedTheme) => StyleConfig
} = useThemePresets();
```

### `useMenuColorBinding`

Applies shadcn's runtime menu semantics: adds `.dark` to `.cn-menu-target` elements when `menuColor` is inverted, and sets `data-menu-translucent` on `[data-menu-translucent]` elements when translucent. Re-applies as portalled menus mount. Returns nothing.

```typescript
useMenuColorBinding(theme?.menuColor);
```

<Aside type="note" title="Menu ≠ sidebar">
This binding targets dropdowns, popovers, and selects — **not** the sidebar. The sidebar is themed via `--sidebar-*` tokens and the `sidebarAccent` field. See [Design philosophy](/modules/style/design/).
</Aside>

### TanStack Query hooks (`useStyleQueries`)

Module-namespaced query/mutation wrappers around the service. Query keys come from `styleKeys = createModuleQueryKeys('style')`, so they are prefixed `['style', …]` per the [data-fetching convention](/starter-template/data-fetching/).

| Hook | Kind | Key / effect |
|------|------|--------------|
| `useThemes()` | query | `['style','themes']` |
| `useTheme(id)` | query | `['style','theme', id]` |
| `useProjectConfig(nodeId)` | query | `['style','project', nodeId]` |
| `useActiveAppTheme()` | query | `['style','app-theme']` |
| `useBaseClassManifest(projectId)` | query | `['style','builder-component-set', projectId]` (5-min `staleTime`; degrades to null) |
| `useSaveProjectConfig()` | mutation | invalidates project key + fires `style:theme-saved` |
| `useSaveAppConfig()` | mutation | applies via `styleRegistry.update` + invalidates `app-theme` |
| `useCreateTheme()` / `useUpdateTheme()` / `useDeleteTheme()` | mutation | invalidate `themes` (delete is optimistic) |
| `useActivateProjectTheme()` / `useActivateAppTheme()` | mutation | invalidate project / app-theme keys |

## Frontend Service

`styleService` (`services/style.service.ts`) wraps the API; every method accepts a trailing optional `AbortSignal` for cancellation.

| Method | Calls |
|--------|-------|
| `getThemes` / `getTheme` / `createTheme` / `updateTheme` / `deleteTheme` | Theme CRUD |
| `getProjectConfig` / `saveProjectConfig` | Project config (get returns `null` on 404) |
| `setActiveProjectTheme` | Activate a theme for a project |
| `getActiveAppTheme` / `saveAppConfig` / `setActiveAppTheme` | App-wide override |
| `getVersions` / `getVersion` / `restoreVersion` / `updateVersionLabel` | Combined project version history |
| `getShadcnConfigVersions` / `getShadcnConfigVersion` / `updateShadcnConfigVersionLabel` | Preset-layer history |
| `getProjectComponentSet` | Builder base-class manifest (`/api/v1/builder/projects/:id/component-set`; any failure → `null`) |

## Backend API Endpoints

All endpoints are under `/api/v1/style` and require authentication. App-wide writes additionally require an admin role.

### Themes (CRUD)

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | List the user's saved themes |
| `POST` | `/` | Create a theme (app-override / global themes require admin) |
| `GET` | `/:id` | Get a single theme |
| `PATCH` | `/:id` | Update a theme |
| `DELETE` | `/:id` | Delete a theme (`204`) |

### Project config

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/projects/:nodeId/config` | Get the saved project config |
| `PUT` | `/projects/:nodeId/config` | Save (upsert) the project config |
| `PUT` | `/projects/:nodeId/activate/:themeId` | Set the active theme for a project |

### Project versions (combined history)

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/projects/:nodeId/versions` | List version history (`[]` if no config) |
| `GET` | `/projects/:nodeId/versions/:version` | Get one version with its full config |
| `PUT` | `/projects/:nodeId/versions/:version/restore` | Restore a version |
| `PATCH` | `/projects/:nodeId/versions/:version` | Update a version label |

### Shadcn-config versions (preset-layer history)

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/projects/:nodeId/shadcn-versions` | List preset-layer snapshots (`[]` if no config) |
| `GET` | `/projects/:nodeId/shadcn-versions/:version` | Get one preset-layer snapshot |
| `PATCH` | `/projects/:nodeId/shadcn-versions/:version` | Update a snapshot label |

<Aside type="note" title="Restore is client-side here">
Preset-layer restore has no dedicated endpoint — the Shadcn Config tab fetches the snapshot detail and re-applies it through the ask-per-change rebase flow. There is no `.../shadcn-versions/:version/restore` route.
</Aside>

### App override

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/app/active` | Get the active app-wide theme (`null` if none) |
| `PUT` | `/app/config` | Save the app-wide config *(admin)* |
| `PUT` | `/app/activate/:id` | Set the active app-wide theme *(admin)* |

<Aside type="caution" title="Route order">
`/projects/*` and `/app/*` routes are declared before the `/:id` catch-alls so those segments aren't matched as a theme id.
</Aside>

## Utilities

The generated `features/style/utils/` folder holds the pure functions behind the UI.

| File | Exports | Role |
|------|---------|------|
| `css-generator.ts` | `resolveTheme`, `generateThemeCSS`, `generateInjectableCSS`, `computePaletteDelta`, `hasPaletteDelta`, `affectsPalette`, `rebaseCssVars`, `reseedCssVars`, `isSafeCustomVar` | Resolve the base palette, emit CSS, and drive the two-layer delta/rebase logic |
| `class-parser.ts` | `classesToStyleState` | Parse a Tailwind class string into structured `StyleState` |
| `class-generator.ts` | `styleStateToClasses`, `computeClassMergeInfo`, `resolveSlotBaseClasses`, `detectVariantMismatches` | Emit classes (via `twMerge`) and compute base-class merge + variant-mismatch warnings |
| `preset-codec.ts` | `encodePreset`, `decodePreset`, `styleConfigToPreset`, `presetToStyleConfigPartial` | shadcn-compatible preset-code encode/decode |
| `oklch-utils.ts` | `hexToOklch`, `oklchToHex`, `parseOklchString`, `formatOklch`, `generateDarkCounterpart`, `generateLightCounterpart` | Hand-rolled OKLCH conversion and light/dark counterpart generation (no external colour library — `culori` was removed in v1.14.10) |
| `shadcn-data.ts` | `PRESET_STYLES`, `PRESET_BASE_COLORS`, `PRESET_THEMES`, `PRESET_FONTS`, `STYLE_LABELS`, `STYLE_DEFAULTS`, `FONT_FAMILIES`, … | Single synced source of truth for shadcn colour/preset/style data, regenerated by `npm run sync:shadcn` |

<Aside type="tip" title="Palette generation lives elsewhere">
`resolveTheme` produces the *base* palette from a preset. Bespoke palette generation (tonal scales, harmony, gamut mapping) belongs to the separate [colour-palette module](/colour-palette/integration/), not this one.
</Aside>
