## Planned Improvements

Enhancements with committed direction.

### Per-user project theme activation

**Priority:** Medium

`setActiveProjectTheme` still allows activating an `isSystem` theme, which flips `isActive` on a row shared by all users, while the preceding deactivation only clears the caller's own rows. It's latent today (`getProjectConfig` filters on `projectId`), but activation is genuinely per-user state modelled as a boolean on a shared record. Proper fix: a dedicated per-user/per-project activation join row, or excluding system themes from the activation gate. Flagged by the July 2026 verification pass.

### Multi-tenant erasure for personal themes

**Priority:** Medium

`my_feature_theme` keys personal themes on `userId` only, with no `tenantId` owner column, so the security deletion handler can erase a specific member but cannot resolve a pure tenant-wide offboard. It now returns `{ action: 'failed' }` in that case (surfacing the gap instead of silently orphaning rows), but full multi-tenant support needs a `tenantId` owner column plus `ownedModels` RLS declaration.

### Rule 7 typography in the generated designer UI

**Priority:** Medium

Colour literals (`emerald`/`amber`) were replaced with semantic status tokens, but ~300 typography-scale occurrences (`text-xs`, `text-[10px]`, `font-medium`…) and aesthetic-opacity utilities remain across the generated designer UI. This needs a deliberate decision rather than a blind sweep — the compact panel UI arguably warrants a sanctioned exception (its own catalogue entry / `--typography-*` tokens) rather than forcing everything to defaults.

### Extend the shadcn sync to the remaining option lists

**Priority:** Low

`npm run sync:shadcn` now regenerates the colour data, preset arrays, per-style defaults and font maps into `utils/shadcn-data.ts`. The Shadcn settings panel's `FONT_OPTIONS` (with the sans/mono/serif grouping), `ICON_LIBRARIES`, `RADIUS_OPTIONS` and `MENU_*` option lists are still hand-copied locals in `theme-settings.ts`. Emit these from the sync (the font `type` and radius rem values are already in the fetched sources) and import them, so nothing about the shadcn config is hand-maintained. Also worth adding: a guard that fails the sync if a preset array's order changes (only appends are safe — the codec encodes indices) and a CI check comparing the pinned tag against the latest `shadcn` release.

### Adopt starter primitives in generated UI

**Priority:** Low

Hand-rolled controls remain where shadcn primitives are declared or available: ColourPicker's tab strip and overlay dropdowns (Tabs/Popover), style-section toggle groups and range input (ToggleGroup/Slider), and raw `<button>`s with inline style resets in the structure tabs. (Page loading/error/empty states were migrated to the starter page-blocks in the July 2026 pass.)

### American-spelling identifier renames

**Priority:** Low

Exported identifiers `ColourTokenEditor`, `ColourSwatch`, `ColourPicker` use Australian spelling against the American-identifiers convention (the codebase already mixes both). Renames are breaking for target apps, so they need the deprecation-shim path (old export + `@deprecated` + dev warning, removal a minor later).

### Full TanStack Query migration of legacy hooks

**Priority:** Medium

`useProjectTheme` and `useThemePresets` still manage their own loading/error/dirty state and never invalidate the `useStyleQueries` cache. The July 2026 pass surfaced version-history errors in `ProjectThemeTab` (no more silent `catch {}`) and blocked save-over-defaults on load failure, but the full migration to the query hooks remains.

### Bootstrap setOverride vs admin toggles

**Priority:** Low

`style.registration.ts` bootstrap and the admin theme page both call `styleRegistry.setOverride()` for `sidebar-accent-bold` whenever an active app theme exists, silently overwriting any toggle the admin made through the starter's styleOverrides UI. Needs a precedence design (saved theme wins vs admin override wins).

### projectId null vs '' sentinel design pass

**Priority:** Low

The schema comment says `projectId` null = global/reusable theme, but `saveAppConfig` uses an `''` sentinel, and Postgres treats NULLs as distinct in the compound unique — themes created with `projectId` omitted fall outside the dedupe. Internally consistent today, but worth unifying.

### App Typography tab

**Priority:** High

Typography was removed from the shadcn config admin tab in v1.13.2 because shadcn's own typography handling needs research before committing to a shape. Once we know what we want, typography becomes its own admin tab separate from the shadcn config.

Design questions to settle before implementation:

- **Scope of control:** which slots get typography knobs — badges only, every shadcn primitive with text, or something in between? shadcn itself doesn't expose typography controls in `shadcn.com/create`, so this is a custom layer.
- **Shape of the knobs:** fixed preset list (compact / default / generous / bold) vs. per-slot overrides vs. ratio-based scale vs. a typographic grid with base size × ratio. The v1.12 approach was four fixed presets — probably too coarse for real use.
- **Relationship to the font selection in the shadcn config:** font family is already in shadcn config. Does the typography tab own size/weight/tracking/line-height and leave font family to shadcn? Or does it own everything typographic?
- **Per-slot data-slot coverage:** which slots do we target with `data-slot` selectors — `[data-slot="badge"]`, `[data-slot="card-title"]`, sidebar menu buttons, page headers, tables? Need a stable list.
- **Interaction with Tailwind v4 `@theme inline` block:** shadcn emits `--text-*` variables in some patterns. Do we feed into those, override them, or live alongside?
- **Storage:** alongside `appTheme` in AppSettings, or a separate `appTypography` field for independent versioning/lifecycle?
- **Project-level typography:** does the project-level tab get its own typography control, or is typography admin-only? If project-level, how does it interact with the shadcn registry the project generates?
- **Runtime live preview:** use the same catalogue-entry + `styleRegistry.update({ tokens })` pattern we established for sidebar-accent, or something else?

The v1.12 catalogue entry (`my-feature.admin-typography`) targeted four slots — `badge`, `card-title`, `admin-role-badge` / `status-pill`, `page-title` — driven by CSS variables with fallbacks. That pattern is worth keeping as a reference even though the specific preset shape isn't.

### Catalogue under review

**Priority:** Medium

The non-core catalogue entries (`tabs-line-primary`, `button-bold`, `card-elevated`, `sidebar-rounded`) may be pulled out of v1.x and reintroduced later when the theme generator roadmap item lands. The core shipped in v1.13 is:

- shadcn-config fields (`baseColor`, `theme`, `chartColor`, `radius`, `font`, `fontHeading`, `iconLibrary`, `style`, `menuColor`, `menuAccent`)
- `sidebarAccent` extension (our one core addition)

Everything else is catalogue — review whether those additions belong in the core `style` module or in a future `theme-generator` / `style-extensions` module.

### @theme inline block alignment with shadcn

**Priority:** High

The module's generated `global.css` has the right tokens in `:root` and `.dark`, but the `@theme inline` block structure differs slightly from shadcn's `buildInstructions` output. Shadcn puts `--font-sans`/`--font-heading`/`--font-mono` first, then colour mappings, then multiplicative radius derivations (`calc(var(--radius) * 0.6)` rather than `calc(var(--radius) - 4px)`). Full alignment needed for byte-perfect diff against `shadcn init`.

### Route 2 — admin storage migrates to AppSettings

**Priority:** High (starter template work in progress)

Admin app theme should live in `AppSettings.appTheme` (starter-template field, pending) rather than `my_feature_theme` with `isAppOverride: true`. When the starter lands the `appTheme` field, the module migrates:

- Backend: drop the `isAppOverride` path entirely; keep project path untouched
- `style.registration.ts` bootstrap: read `AppSettingsContext.appTheme` instead of `GET /app/active`
- `AppThemeSettingsPage`: swap `useProjectTheme({ appWide: true })` for a new `useAppTheme()` hook over `useAppSettings()`
- Admin Shadcn Config and eventual Admin Theme tabs both read/write the same appTheme
- Catalogue-entry toggles continue going through `AppSettings.styleOverrides` as they already do

### Bootstrap cache for styleRegistry tokens

**Priority:** Medium

`style.registration.ts` fetches the active app theme asynchronously on load. Until the fetch resolves, the page renders with starter defaults. A `localStorage` cache keyed by app id would let us apply last-known tokens synchronously at module load.

### Catalogue browser UI in admin theme settings

**Priority:** Medium

Catalogue entries currently toggle through the starter's generic `styleOverrides` UI. A Style-module-owned browser in the admin theme settings page would group entries by primitive, show live previews, and surface the CSS for audit.

### Per-slot typography overrides

**Priority:** Low

The typography preset is a global choice. Future work: let admins override individual `--typography-*` tokens (e.g. keep Default but bump `--typography-page-title-size`).

### More catalogue entries

**Priority:** Low

Current catalogue covers tabs, buttons, cards, sidebar-rounded, sidebar-accent-bold, admin-typography. Expand to cover dialog/sheet/alert-dialog, table row density, form field sizing, etc.

### Flesh out the module docs

**Priority:** Low

Sister Structure and Builder modules each have Configuration, Usage, Integration, Developer Workflow, and Changelog pages. Style currently ships Overview, Design philosophy, Roadmap, and a module README (added July 2026).

## Known Issues

Technical debt and minor issues to be addressed.

| Issue | Status | Notes |
|-------|--------|-------|
| **No generated-code type-check in CI** | Medium priority | The module's `npm run type-check` only lints the template **strings**, not the generated code inside them — which is how ~143 type errors accumulated undetected (now fixed, v1.14.4). A CI step that generates into a throwaway app and runs `tsc` would catch regressions. Until then, verify with `npx style update` into a test app + `tsc`. |
| **`@theme inline` block differs from shadcn** | High priority | Tokens are correct; the surrounding structure (font vars at top, multiplicative radius derivations) isn't yet an exact match. Tracked above under Planned Improvements. |
| **`setActiveProjectTheme` can activate a shared system theme** | Medium priority | Latent cross-user write on the shared `isActive` boolean; needs a per-user activation model. Tracked above under Planned Improvements |
| **Legacy hooks still fetch outside TanStack Query** | Low priority | `useProjectTheme` / `useThemePresets` fetch with raw `useState`/`useEffect` (no dedup/cache, refetch every mount). The save-coherence half was fixed in v1.14.5 (`save()` now invalidates the cache and dispatches `my-feature:theme-saved`, so the builder preview refreshes); the remaining work is migrating the *reads* onto the `useProjectConfig`/`useActiveAppTheme` query hooks |
| **First-paint flash on app theme load** | Low priority | Registry tokens apply once async fetch resolves |
