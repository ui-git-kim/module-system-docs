## Planned Improvements

Enhancements with committed direction.

### Security registration for the generated backend

**Priority:** High

Scaffold v1.11 made `security.registration.ts` mandatory for modules storing per-user data, and this module stores `my_feature_theme` (userId-owned) plus `my_feature_theme_version` snapshots. The backend templates ship no `securityRegistry.register()` manifest, no deletion handlers, and no `user.deleted` cleanup â€” a deleted user's themes and version snapshots persist forever.

Design notes before implementation:

- `my_feature_theme` mixes personal rows, `isSystem` shared rows, and `isAppOverride` app-level rows under one `userId` column â€” the manifest needs separate `dataCategories` and the owned deletion handler must scope with `isSystem: false, isAppOverride: false` (system/app rows are preserved or anonymised, not deleted).
- Version snapshots have no `userId` â€” deletion cascades through `themeId`, so the handler (and the service's own `delete()`) should remove versions and theme in one `prisma.$transaction`.
- No `encryptedFields`/`hashedSecrets` needed â€” theme JSON is not sensitive data.

### Ownership checks on activate / restore / label routes

**Priority:** High

`setActiveProjectTheme` (`PUT /projects/:nodeId/activate/:themeId`) performs no ownership or membership check, and the version-restore/version-label routes gate writes on a read-scope query that matches shared system themes â€” any authenticated user can deactivate another user's active theme or overwrite a shared theme's config and labels. Service-layer ownership checks should also move to atomic `$transaction` + idempotent `deleteMany` per current backend conventions.

### Rule 7 compliance in generated designer UI

**Priority:** Medium

The current LLM rules restrict generated component code to layout utilities + semantic tokens. The audit found ~300 typography-scale occurrences (`text-xs`, `text-[10px]`, `font-medium`â€¦) across all 15 generated UI files, plus opacity-for-aesthetics (`bg-muted/30`, `hover:bg-accent/50`) and a handful of colour literals (`emerald`/`amber` instead of the module's own `success`/`warning` tokens). Colour literals are straightforward fixes; the typography scale needs a deliberate decision â€” the compact designer-panel UI arguably warrants a sanctioned exception (e.g. its own catalogue entry / `--typography-*` tokens) rather than a blind sweep to defaults.

### AbortSignal support in the style service

**Priority:** Medium

None of the 15 generated service methods accept an `AbortSignal`, and the TanStack Query `queryFn`s discard the `({ signal })` context â€” current conventions require every service method to take an optional signal and thread it to axios.

### Adopt starter primitives in generated UI

**Priority:** Low

Hand-rolled controls exist where shadcn primitives are declared or available: ColourPicker's tab strip and overlay dropdowns (Tabs/Popover), style-section toggle groups and range input (ToggleGroup/Slider), raw `<button>`s with inline style resets in the structure tabs, and hand-rolled loading/error/empty states in pages where the starter's `LoadingState`/`ErrorAlert`/`EmptyState` page-blocks exist.

### Shared constants module for font/style maps

**Priority:** Low

`FONT_FAMILIES`, `STYLE_LABELS`, `FONT_LABELS` are duplicated verbatim across three generated files (ShadcnConfigTab, ProjectThemeTab, AppThemeSettingsPage) and `STYLE_DEFAULTS` across two â€” extract to one generated shared module.

### American-spelling identifier renames

**Priority:** Low

Exported identifiers `ColourTokenEditor`, `ColourSwatch`, `ColourPicker` use Australian spelling against the American-identifiers convention (the codebase already mixes both). Renames are breaking for target apps, so they need the deprecation-shim path (old export + `@deprecated` + dev warning, removal a minor later).

### Bootstrap setOverride vs admin toggles

**Priority:** Low

`style.registration.ts` bootstrap and the admin theme page both call `styleRegistry.setOverride()` for `sidebar-accent-bold` whenever an active app theme exists, silently overwriting any toggle the admin made through the starter's styleOverrides UI. Needs a precedence design (saved theme wins vs admin override wins).

### projectId null vs '' sentinel design pass

**Priority:** Low

The schema comment says `projectId` null = global/reusable theme, but `saveAppConfig` uses an `''` sentinel, and Postgres treats NULLs as distinct in the compound unique â€” themes created with `projectId` omitted fall outside the dedupe. Internally consistent today, but worth unifying.

### Rewrite the README

**Priority:** Low

`README.md` is still the generic module-starter template README (including its own "After Cloning: Update This README" instruction) â€” it documents nothing about the Builder Style module's features, install modes, or usage.

### App Typography tab

**Priority:** High

Typography was removed from the shadcn config admin tab in v1.13.2 because shadcn's own typography handling needs research before committing to a shape. Once we know what we want, typography becomes its own admin tab separate from the shadcn config.

Design questions to settle before implementation:

- **Scope of control:** which slots get typography knobs â€” badges only, every shadcn primitive with text, or something in between? shadcn itself doesn't expose typography controls in `shadcn.com/create`, so this is a custom layer.
- **Shape of the knobs:** fixed preset list (compact / default / generous / bold) vs. per-slot overrides vs. ratio-based scale vs. a typographic grid with base size Ă— ratio. The v1.12 approach was four fixed presets â€” probably too coarse for real use.
- **Relationship to the font selection in the shadcn config:** font family is already in shadcn config. Does the typography tab own size/weight/tracking/line-height and leave font family to shadcn? Or does it own everything typographic?
- **Per-slot data-slot coverage:** which slots do we target with `data-slot` selectors â€” `[data-slot="badge"]`, `[data-slot="card-title"]`, sidebar menu buttons, page headers, tables? Need a stable list.
- **Interaction with Tailwind v4 `@theme inline` block:** shadcn emits `--text-*` variables in some patterns. Do we feed into those, override them, or live alongside?
- **Storage:** alongside `appTheme` in AppSettings, or a separate `appTypography` field for independent versioning/lifecycle?
- **Project-level typography:** does the project-level tab get its own typography control, or is typography admin-only? If project-level, how does it interact with the shadcn registry the project generates?
- **Runtime live preview:** use the same catalogue-entry + `styleRegistry.update({ tokens })` pattern we established for sidebar-accent, or something else?

The v1.12 catalogue entry (`my-feature.admin-typography`) targeted four slots â€” `badge`, `card-title`, `admin-role-badge` / `status-pill`, `page-title` â€” driven by CSS variables with fallbacks. That pattern is worth keeping as a reference even though the specific preset shape isn't.

### Catalogue under review

**Priority:** Medium

The non-core catalogue entries (`tabs-line-primary`, `button-bold`, `card-elevated`, `sidebar-rounded`) may be pulled out of v1.x and reintroduced later when the theme generator roadmap item lands. The core shipped in v1.13 is:

- shadcn-config fields (`baseColor`, `theme`, `chartColor`, `radius`, `font`, `fontHeading`, `iconLibrary`, `style`, `menuColor`, `menuAccent`)
- `sidebarAccent` extension (our one core addition)

Everything else is catalogue â€” review whether those additions belong in the core `style` module or in a future `theme-generator` / `style-extensions` module.

### @theme inline block alignment with shadcn

**Priority:** High

The module's generated `global.css` has the right tokens in `:root` and `.dark`, but the `@theme inline` block structure differs slightly from shadcn's `buildInstructions` output. Shadcn puts `--font-sans`/`--font-heading`/`--font-mono` first, then colour mappings, then multiplicative radius derivations (`calc(var(--radius) * 0.6)` rather than `calc(var(--radius) - 4px)`). Full alignment needed for byte-perfect diff against `shadcn init`.

### Route 2 â€” admin storage migrates to AppSettings

**Priority:** High (starter template work in progress)

Admin app theme should live in `AppSettings.appTheme` (starter-template field, pending) rather than `my_feature_theme` with `isAppOverride: true`. When the starter lands the `appTheme` field, the module migrates:

- Backend: drop the `isAppOverride` path entirely; keep project path untouched
- `style.registration.ts` bootstrap: read `AppSettingsContext.appTheme` instead of `GET /app/active`
- `AppThemeSettingsPage`: swap `useProjectTheme({ appWide: true })` for a new `useAppTheme()` hook over `useAppSettings()`
- Admin Shadcn Config and eventual Admin Theme tabs both read/write the same appTheme
- Catalogue-entry toggles continue going through `AppSettings.styleOverrides` as they already do

### Full TanStack Query migration of legacy hooks

**Priority:** Medium

`useProjectTheme` and `useThemePresets` still manage their own loading/error/dirty state. The `useStyleQueries` hooks shipped in v1.11.5 provide the right pattern; legacy hooks should migrate.

### Bootstrap cache for styleRegistry tokens

**Priority:** Medium

`style.registration.ts` fetches the active app theme asynchronously on load. Until the fetch resolves, the page renders with starter defaults. A `localStorage` cache keyed by app id would let us apply last-known tokens synchronously at module load.

### Catalogue browser UI in admin theme settings

**Priority:** Medium

Catalogue entries currently toggle through the starter's generic `styleOverrides` UI. A Style-module-owned browser in the admin theme settings page would group entries by primitive, show live previews, and surface the CSS for audit.

### Per-slot typography overrides

**Priority:** Low

The typography preset is a global choice. Future work: let admins override individual `--typography-*` tokens (e.g. keep Default but bump `--typography-page-title-size`).

### More catalogue entries

**Priority:** Low

Current catalogue covers tabs, buttons, cards, sidebar-rounded, sidebar-accent-bold, admin-typography. Expand to cover dialog/sheet/alert-dialog, table row density, form field sizing, etc.

### Conditional template generation for install modes

**Priority:** High *(unblocked â€” scaffold v1.14 shipped `{ content, when }`)*

`installMode: admin` still emits structure/property-panel/builder-canvas registrations, the structure tabs, and the StylePanel suite even though they're never imported â€” and the registrations import builder/structure registries that don't exist in admin-only apps, breaking `tsc` there. The scaffold's file-level conditional generation landed in v1.14 (this repo is on 1.15.0), so `when: "installMode !== 'admin'"` gating is now implementable. On update, newly-false `when` files are offered as orphans for removal.

### Flesh out the module docs

**Priority:** Low

Sister Structure and Builder modules each have Configuration, Usage, Integration, Developer Workflow, and Changelog pages. Style currently ships Overview, Design philosophy, and Roadmap.
