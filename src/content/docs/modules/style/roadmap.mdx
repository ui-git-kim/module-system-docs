## Planned Improvements

Enhancements with committed direction.

### Full TanStack Query migration of legacy hooks

**Priority:** Medium

`useProjectTheme` and `useThemePresets` still manage their own loading/error/dirty state with `useState` + `useEffect`. The v1.12 alignment added a new `useStyleQueries` file with the correct pattern, but the legacy hooks haven't been rewritten against it yet. Consumers can migrate piecemeal; the two should eventually be fully replaced by composition of the query hooks with pure UI state for dirty tracking.

### Bootstrap cache for styleRegistry tokens

**Priority:** Medium

`style.registration.ts` fetches the active app theme asynchronously on module load. Until the fetch resolves, the page renders with starter defaults, which causes a first-paint flash to the admin-chosen theme. A `localStorage` cache keyed by app id would let us apply the last-known tokens synchronously at module load, then reconcile once the network response arrives. The starter's style-system docs flag this same gap for the framework itself.

### Catalogue browser UI in admin theme settings

**Priority:** Medium

Catalogue entries currently toggle through the starter's generic `styleOverrides` UI. A Style-module-owned browser inside the admin theme settings page would:

- Group entries by primitive (Tabs, Buttons, Cards, Sidebar…)
- Show a live preview of each entry's effect
- Surface the CSS so admins can audit what they're enabling
- Track which entries were toggled by whom and when

### Admin-triggered `npm run components:sync`

**Priority:** Low

Changes to `componentsJson.style`, `componentsJson.tailwind.baseColor`, or `componentsJson.tailwind.cssVariables` require `npm run components:sync` to take effect. Today that runs manually. The admin page could expose a "Re-sync shadcn components" action that triggers the script via a backend endpoint.

### Per-slot typography overrides

**Priority:** Low

The typography preset is a single global choice. Future work: let admins override individual `--typography-*` tokens from the settings UI (e.g. keep the Default preset but bump the page-title size).

### Conditional template generation for install modes

**Priority:** Low

The module exposes an `installMode` prompt (admin vs. full) but only the barrel file's side-effect imports are conditionally emitted. The `structure/` and `components/structure/` templates are always generated. When a user picks **Admin only**, those files sit unused. Either add `when:` gating on individual templates (pending scaffold support), or split the module into `style` and `style-builder`.

### Flesh out the module docs

**Priority:** Low

Sister Structure and Builder modules each have Configuration, Usage, Integration, Developer Workflow, and Changelog pages. Style currently ships Overview, Design philosophy, and Roadmap. Filling the rest in would bring it to parity.

## Known Issues

Technical debt and minor issues to be addressed.

| Issue | Status | Notes |
|-------|--------|-------|
| **Legacy hooks bypass TanStack Query cache** | Low priority | `useProjectTheme` and `useThemePresets` use local state; cache hits are available through `useStyleQueries` but legacy-hook consumers re-fetch on every mount |
| **First-paint flash on app theme load** | Low priority | Registry tokens apply once async fetch resolves; tracked above under Planned Improvements |
| **Unused files when install mode is admin-only** | Low priority | Structure and property-panel component files generate but are never imported in admin mode |
