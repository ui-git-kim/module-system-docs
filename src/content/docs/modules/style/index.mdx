The **Style** module is the sole contributor of app-wide styling across the system. It gives admins a visual theme editor, ships a catalogue of opt-in style registry entries that target shadcn's stable data attributes, and gives builders a structured Tailwind className designer for individual nodes. Themes are persisted per-project and per-app, and every styling decision flows through the starter's [styleRegistry](/starter-template/style-system/) — no other module contributes styling.

See [Design philosophy](/modules/style/design/) for the catalogue pattern and the tokens-vs-css decision matrix.

<Aside type="tip" title="Two install modes">
Run `npx style init` and choose **Admin only** to get just the app theme settings page, or **Admin + Builder** to also wire the style panel and structure tabs used by the builder and structure modules.
</Aside>

## Quick Install

```bash
npx style init
```

The CLI creates the Prisma models (`style_theme`, `style_theme_version`, `style_shadcn_config_version`), generates the frontend feature folder, installs required shadcn components, and registers the app theme admin page.

## Architecture

The module contributes to six frontend registries:

| Registry | Purpose |
|----------|---------|
| `styleRegistry` | App-wide theme tokens (one active theme per app) plus the opt-in style catalogue |
| `adminPagesRegistry` | Admin settings page at `/admin/theme` |
| `nodeTabRegistry` *(structure)* | "Shadcn Config" and "Theme" tabs on project nodes (full mode only) |
| `propertyPanelTabRegistry` *(builder)* | "Style" tab on selected builder elements (full mode only) |
| `filterRegistry` | Filter on the builder's `builder.canvas.globalCss` hook injecting the project's generated CSS into the iframe preview (full mode only) |
| `structureRegistry` *(structure)* | Registers the hidden `theme_config` mirror field on the project node type (full mode only — see the contract below) |

The backend exposes `/api/v1/style` with routes for theme CRUD, project configs, app overrides, combined version history, and a separate shadcn-config (preset-layer) history (`/projects/:nodeId/shadcn-versions`). Routes register through `routeRegistry`; the service is exposed via `serviceRegistry.register('style', ...)`; and the module ships a `security.registration.ts` (data-category manifest + account-erasure deletion handler).

## Builder integration — the `theme_config` contract

The builder renders a project's components using the project's own shadcn configuration. Rather than have the builder call the style module directly, the style module **mirrors the active configuration onto the project's structure node**, and the builder reads it from there. This keeps the two modules decoupled — the style module owns the data, the builder just reads a field.

### The field

The style module registers a field on the project node type through structure's `structureRegistry` (full install mode only):

```typescript
// PROJECT_NODE_TYPES is set at install from the `projectNodeType` prompt
// (default 'projects') and is editable in structure.registration.ts — point
// it at whatever node-type slug(s) your builder projects use.
const PROJECT_NODE_TYPES = ['projects'];

structureRegistry.register({
  moduleId: 'style',
  displayName: 'Style',
  fields: [{
    id: 'theme_config',            // NOT namespaced — the builder reads this exact id
    moduleId: 'style',             // the field's owning module
    nodeTypes: PROJECT_NODE_TYPES, // the project node-type slug(s)
    order: 1000,
    field: { type: 'hidden', label: 'Shadcn Config' },
  }],
});
```

The same `PROJECT_NODE_TYPES` drives the Shadcn Config + Theme node tabs, so all three attach to the node type your builder reads. The mirror writes to the node you're editing regardless (it's a `data` merge), but the tabs and the structure admin UI only appear on the configured type — so set this to match your builder's project node-type slug.

Because structure stores field values in the node's `data` JSON, the value lands at **`node.data.theme_config`**. The field is `hidden` — it is system-managed, not user-edited on the Info tab.

### Who writes it

The style backend mirrors the full `StyleConfig` onto the node on **every change to the active project configuration** — `saveProjectConfig`, `setActiveProjectTheme`, and project-scoped `restoreVersion` — via structure's node-update API:

```typescript
await structureService.updateNode(nodeId, userId, { data: { theme_config: config } });
```

The style module's own `style_theme` table remains the editor/history store; `node.data.theme_config` is the **canonical mirror everyone reads**. The write is best-effort (failure is logged, never blocks the save), so the builder should tolerate a missing field on projects that predate the style module or haven't been saved yet.

### The shape the builder reads

At `node.data.theme_config` the builder needs at minimum these keys (the style module mirrors the whole `StyleConfig`, so extra keys are present and can be ignored):

```typescript
{
  componentLibrary: 'radix' | 'base',   // ← builder uses this
  styleName: 'nova' | 'vega' | 'maia' | 'lyra' | 'mira' | 'luma' | 'sera' | 'rhea',
  cssVars: {                            // ← builder uses this as the preview theme
    light: { [token: string]: string },
    dark?: { [token: string]: string },
  },
  // iconLibrary, radius, font, baseColor, colorPreset, chartColor, savedColors, defaultStyling, … — ignored by the builder
}
```

The builder derives the shadcn registry style path as **`${componentLibrary}-${styleName}`** (e.g. `radix-luma`) and fetches `ui.shadcn.com/r/styles/{that}/{component}.json`. All eight `styleName` values map 1:1 to real registry slugs for both `radix-` and `base-` — no translation table is needed. `cssVars` is applied directly as the preview theme.

<Aside type="note" title="Read fallbacks">
For resilience against older data, the builder reads `node.data.theme_config` first, then falls back to `shadcn_config` → `style_config` → the legacy `data.theme`.
</Aside>

## Dependencies
