The Style module is the **sole stylist** for apps built on the module system. No other module contributes styling. Everything flows through the starter's [`styleRegistry`](/starter-template/style-system/), and the Style module is the registry's single writer.

<Aside type="caution" title="One module, one voice">
Other modules must not call `styleRegistry.register()` or edit `components/ui/*` files. If a module needs a visual tweak, it requests a catalogue entry here and the Style module ships it.
</Aside>

<Aside type="tip" title="Match shadcn, then extend">
The core `baseColor + theme + radius + menuAccent + menuColor + font + iconLibrary + style + chartColor + fontHeading` pipeline produces CSS **byte-identical to `shadcn init`** for the same preset code. Our own features (typography presets, sidebar accent, the other catalogue entries) are layered on top as additive registry entries — never baked into the shadcn baseline.
</Aside>

## Menu ≠ Sidebar — the most important distinction

Shadcn's "Menu" controls in `shadcn.com/create` govern **dropdowns, popovers, and select menus** — not the sidebar.

| You set | What changes |
|---------|--------------|
| Menu Colour: **Inverted** | Dropdowns, popovers, select menus render with dark-mode tokens on a light page (and vice versa). The sidebar is **not affected**. |
| Menu Colour: **Translucent** | Those same menus get a translucent backdrop. Sidebar not affected. |
| Menu Accent: **Bold** | The global `--accent` token is redirected to `--primary`. Hover/active states on buttons, menu items, and selection highlights across the whole app pick up the primary colour. |

Menu colour is applied **at runtime**, not in the generated CSS. The module installs a binding that toggles `.dark` on every element with the `.cn-menu-target` class and sets `data-menu-translucent` on `[data-menu-translucent]` elements. Starter template dropdowns must carry `.cn-menu-target` for this to take effect.

Menu accent IS in the generated CSS — `buildRegistryTheme` (our `resolveTheme`) mutates `--accent` and `--accent-foreground` to match `--primary` when bold.

## Sidebar Accent — our extension

Shadcn ships `--sidebar-*` tokens but no UI to make the active nav item bold. That's a reasonable design knob that sits cleanly alongside shadcn without breaking preset compatibility.

Our `sidebarAccent` field (values `'subtle' | 'bold'`):

- Stored in `StyleConfig`, not encoded in the shadcn preset code — codes still round-trip cleanly with `shadcn.com/create`
- Implemented as the `my-feature.sidebar-accent-bold` catalogue entry:

```css
:root, .dark {
  --sidebar-accent: var(--primary);
  --sidebar-accent-foreground: var(--primary-foreground);
}
```

- Toggled on/off via `styleRegistry.setOverride()` by the admin settings page when the user picks Bold. Pure additive layer — the baseline `:root` / `.dark` blocks stay byte-identical to shadcn's output.

## The catalogue pattern

Beyond the shadcn baseline, the module ships opt-in `styleRegistry` entries targeting shadcn's **stable public API**: the `data-slot`, `data-variant`, and `data-state` attributes every primitive renders.

```html
<div data-slot="tabs" data-orientation="horizontal">
  <div data-slot="tabs-list" data-variant="line">
    <button data-slot="tabs-trigger" data-variant="line" data-state="active">…</button>
  </div>
</div>
```

A catalogue entry hooks those attributes:

```typescript
styleRegistry.register({
  id: 'my-feature.tabs-line-primary',
  order: 60,
  enabled: false,
  css: `
    [data-slot="tabs-trigger"][data-variant="line"][data-state="active"] {
      color: var(--primary);
      border-color: var(--primary);
    }
  `,
});
```

Zero changes to `components/ui/tabs.tsx`. Zero template edits. Purely additive.

## Preset codec — byte-compatible with shadcn

The module's `preset-codec.ts` is a line-for-line port of `shadcn/preset/preset.ts`. A preset code generated in `shadcn.com/create` decodes identically here, and vice versa.

**Format:** version character prefix (`"b"` = current v2) + base62-encoded bit-packed integer.

**V2 field layout (51 bits):**

```
menuColor (3) | menuAccent (3) | radius (4) | font (6) | iconLibrary (6)
  | theme (6) | baseColor (6) | style (6) | chartColor (6) | fontHeading (5)
```

Value arrays (`PRESET_STYLES`, `PRESET_FONTS`, etc.) follow shadcn's ordering exactly. Rule: **never reorder existing arrays — only append**. Reordering breaks all existing codes.

Our `sidebarAccent` and `typographyPreset` are NOT encoded in the preset code. They live in `StyleConfig` and persist separately through the normal theme save path.

## Why the cascade wins

Catalogue entries land on top of shadcn's CSS reliably because:

1. **Source order** — the style pipeline appends `<style id="module-styles">` to `<head>` **after** `globals.css` and any Tailwind utilities. On ties, later wins.
2. **Specificity on attribute selectors** — shadcn's `data-[state=active]:text-foreground` compiles to class + attribute, specificity `(0,2,0)`. A catalogue selector like `[data-slot="tabs-trigger"][data-variant="line"][data-state="active"]` is `(0,3,0)` — strictly higher.

You don't need `!important` outside of rare edge cases (see caveats).

## Tokens vs CSS — decision matrix

When writing a new catalogue entry, decide:

| Property type | Use | Why |
|---------------|-----|-----|
| Colours (primary, background, sidebar) | `tokens` → `--primary`, `--sidebar-*` | shadcn reads these via `@theme inline`; modules just redefine them |
| Radius | `tokens` → `--radius-*` | Same |
| Font family | `tokens` → `--font-sans`, `--font-mono` | Same |
| Typography sizing, weight, spacing | `css` | shadcn hardcodes these as Tailwind utilities, no CSS var to override |
| Layout (gaps, padding on a variant) | `css` | Same |
| State-driven styles (hover, active, disabled) | `css` | Radix exposes state via data attributes |
| Animations / transitions | `css` | Same |

**Rule of thumb:** if shadcn's `@theme inline` block lists a CSS variable for it, use `tokens`. Otherwise use `css` targeting data attributes.

---

## Hybrid pattern — CSS that reads module-owned tokens

Sometimes you want user-facing toggles to drive typography or spacing (sizing, weight, tracking), which is `css` territory per the matrix. The pattern is:

1. Define a catalogue `css` entry that targets data-slots and **reads module-owned custom properties** with sensible fallbacks
2. Keep the property values in a preset map keyed by a user choice
3. Write the active preset's tokens into the catalogue entry via `styleRegistry.update(id, { tokens })`

This keeps the CSS static and auditable while letting user choices travel entirely through tokens. It's the approach we'll use for the future App Typography tab (see roadmap).

## Status colour tokens

The module ships four semantic status tokens alongside shadcn's standards:

| Token | Purpose | Light default |
|-------|---------|---------------|
| `destructive` | Errors, delete actions | `oklch(0.577 0.245 27.325)` |
| `success` | Confirmations | `oklch(0.517 0.174 142.5)` |
| `warning` | Cautions | `oklch(0.769 0.188 70.08)` |
| `info` | Informational states | `oklch(0.55 0.18 250)` |

Downstream components should consume semantic tokens:

```tsx
// ❌ before
<Badge className="bg-red-100 text-red-800">Expired</Badge>

// ✅ after
<Badge className="bg-destructive/10 text-destructive">Expired</Badge>
```

The Style module only ships the tokens. Migrating starter-template badges to use them is a separate starter PR.

## Shadow pass

Default shadcn primitives ship no shadow utilities — apps opt into `my-feature.card-elevated` for card elevation. Keeps elevation a deliberate choice and lets an app tune the elevation scale globally.

## Adding a new catalogue entry

1. Export the CSS from `utils/catalogue.ts`:
   ```typescript
   export const CATALOGUE_MY_THING = `
     [data-slot="..."][data-variant="..."] { … }
   `;
   ```
2. Register it in `registrations/catalogue.registration.ts`:
   ```typescript
   styleRegistry.register({
     id: 'my-feature.my-thing',
     order: 60,
     enabled: false,
     tokens: {},
     css: CATALOGUE_MY_THING,
   });
   ```
3. Bump the module version and ship. Apps pick up the new entry with `npx style update`.

## Caveats

### `!important` as escape hatch
If a shadcn class has higher specificity than your selector (rare — happens with some `[&>...]` Tailwind compound selectors), use `!important` for that rule. Document why in a comment.

### `@source` in module CSS
Tailwind v4 refuses `@source` directives inside module CSS that goes through `@layer`. If an entry needs Tailwind utilities, generate them up-front at the module's build time and ship the compiled CSS string. Raw CSS (what the catalogue ships today) is always safe.

### Value arrays are append-only
The preset codec's value arrays (`PRESET_STYLES`, `PRESET_FONTS`, etc.) mirror shadcn's. Reordering or removing entries breaks every existing preset code. Only append new values to the end.
