The Builder Style module is configured almost entirely at **install time** and thereafter through the admin theme editor and per-project tabs — its runtime state lives in the database, not in config files. There are no environment variables. The only install-time decisions are which **install mode** to run and, in full mode, which **structure node type** represents your builder projects.

---

## Environment Variables

**None.** The module reads and writes all of its state (themes, version history, app-wide overrides) through the backend at `/api/v1/style`, backed by its own Prisma tables. There is nothing to put in `.env`.

---

## Installation Prompts

`npx style init` asks two questions.

| Prompt | ID | Type | Default | Shown when |
|--------|----|------|---------|-----------|
| What would you like to install? | `installMode` | select | `full` | always |
| Structure node-type slug for builder projects? | `projectNodeType` | input | `projects` | `installMode !== 'admin'` |

### `installMode`

<Tabs>
  <TabItem label="admin">
    **Admin theme settings only.** Installs the `/admin/theme` page, the app-wide `styleRegistry` contribution, the style catalogue, the three Prisma models, and the backend. It has **no dependency** on the builder or structure modules — the builder property panel tab, the structure node tabs, the `theme_config` mirror field, and the builder-canvas CSS filter are all skipped. Pick this for apps that only need one app-wide theme.
  </TabItem>
  <TabItem label="full (default)">
    **Admin + builder integration.** Everything in `admin` mode, plus the builder `StylePanel` property-panel tab, the two structure node tabs (**Shadcn Config** and **Theme**), the hidden `theme_config` mirror field, and the `builder.canvas.globalCss` filter that themes the builder preview. Requires the [`builder`](/modules/builder/integration/) and [`structure`](/modules/structure/integration/) modules to be installed.
  </TabItem>
</Tabs>

<Aside type="caution" title="Full mode needs builder + structure">
The full-mode files import `@/features/builder` and `@/features/structure`. If those modules are absent, choose `admin` — the scaffold uses a conditional-file guard (`when: installMode !== 'admin'`) to skip the builder/structure files so an admin-only app never ships components that fail type-checking.
</Aside>

### `projectNodeType`

Full mode only. The structure node-type slug that represents builder "projects". It becomes `PROJECT_NODE_TYPES` in the generated `structure.registration.ts` and drives three things that must all attach to the same node type your builder reads:

- the **Shadcn Config** and **Theme** node tabs,
- the hidden **`theme_config`** mirror field (the builder reads `node.data.theme_config`).

It is **editable after install** — open `registrations/structure.registration.ts` and change the `PROJECT_NODE_TYPES` array to target a different slug or multiple node types. See the [`theme_config` contract](/modules/style/integration/) for how the builder consumes it.

## What Gets Installed

<Steps>

1. **Three Prisma models** — `style_theme`, `style_theme_version`, `style_shadcn_config_version` (see [Database Schema](#database-schema)). Self-contained: `userId` foreign keys with `@@index`, no `@relation` directives.

2. **The frontend feature folder** — `frontend/src/features/style/` with the theme editor, hooks, services, utils, the style catalogue, and (full mode) the builder/structure integration components.

3. **shadcn components** (installed if missing): `tabs`, `select`, `input`, `button`, `popover`, `switch`, `separator`, `scroll-area`, `tooltip`, `badge`, `label`, `card`.

4. **Frontend npm dependencies** — `@tanstack/react-query` (query-backed data hooks with module-namespaced cache keys `['style', …]`) and `tailwind-merge` (the class designer merges conflicting utilities via `twMerge`). The backend adds no dependencies.

5. **The admin theme page** — registered on `adminPagesRegistry` with id `app-theme`, title **Theme**, at **`/admin/theme`**.

6. **The backend** at **`/api/v1/style`** — theme CRUD, project configs, app overrides, and version history.

</Steps>

<Aside type="note" title="culori removed">
As of v1.14.10 the module no longer depends on `culori`; OKLCH conversions are hand-rolled. The only runtime frontend dependencies are `@tanstack/react-query` and `tailwind-merge`.
</Aside>

### Registries the module contributes to

| Registry | Side | Install mode | Purpose |
|----------|------|:------------:|---------|
| `styleRegistry` | frontend | all | The app-wide active theme (`style.app-theme`) **plus** the opt-in style catalogue entries |
| `adminPagesRegistry` | frontend | all | The `/admin/theme` settings page |
| `nodeTabRegistry` | frontend *(structure)* | full | The **Shadcn Config** and **Theme** tabs on project nodes |
| `propertyPanelTabRegistry` | frontend *(builder)* | full | The **Style** tab (`StylePanel`) on the selected builder element |
| `filterRegistry` | frontend | full | A filter on `builder.canvas.globalCss` injecting the project's generated CSS into the iframe preview |
| `structureRegistry` | frontend *(structure)* | full | The hidden `theme_config` mirror field on the project node type |
| `routeRegistry` | backend | all | Mounts the `/api/v1/style` routes |
| `serviceRegistry` | backend | all | `serviceRegistry.register('style', …)` — cross-module service access |
| `securityRegistry` | backend | all | Data-category manifest + account-erasure deletion handler |
| `actionRegistry` | backend | all | Fires theme lifecycle hooks (`style.theme.created`, `style.appTheme.saved`, `style.projectConfig.saved`, …) |

## The StyleConfig Shape

Every theme — app-wide, per-project, or a saved reusable preset — is stored as a single `StyleConfig` JSON blob in the `config` column. It is the canonical unit the editor reads, the backend mirrors to `node.data.theme_config`, and the CSS generator turns into `global.css`.

| Field | Type | Purpose |
|-------|------|---------|
| `componentLibrary` | `'radix' \| 'base'` | Which shadcn component library the project renders against |
| `styleName` | `'vega' \| 'nova' \| 'maia' \| 'lyra' \| 'mira' \| 'luma' \| 'sera' \| 'rhea'` | The shadcn style preset |
| `baseColor` | `string` | Base neutral tone (see [Base Colour](#base-colour)) |
| `colorPreset` | `string` | The theme / accent colour |
| `chartColor?` | `string` | Chart palette accent (`'default'` to inherit the theme) |
| `iconLibrary` | `'lucide' \| 'hugeicons' \| 'phosphor' \| 'tabler' \| 'remixicon'` | Icon set for generated code and preview |
| `font` | `string` | Body font (a known font key or a custom family name) |
| `fontHeading?` | `string` | Heading font override (`'inherit'` to match the body font) |
| `radius` | `'default' \| 'none' \| 'small' \| 'medium' \| 'large'` | Corner-radius scale (`'default'` emits no `--radius`, using the style's own) |
| `menuColor` | `'default' \| 'inverted' \| 'default-translucent' \| 'inverted-translucent'` | Colour treatment for **dropdowns, popovers, selects** — **not** the sidebar |
| `menuAccent` | `'subtle' \| 'bold'` | `'bold'` redirects the global `--accent` token to `--primary` |
| `sidebarAccent?` | `'subtle' \| 'bold'` | Our extension (not a shadcn preset). `'bold'` toggles the `style.sidebar-accent-bold` catalogue entry |
| `cssVars` | `ThemeConfig` | The custom colour-token layer — `{ light, dark, custom }` (see below) |
| `savedColors?` | `string[]` | Saved solid colours (hex) persisted with the project |
| `savedGradients?` | `string[]` | Saved gradient CSS strings persisted with the project |
| `defaultStyling?` | `DefaultStyleRule[]` | Per-slot base-component restyle layer (see below) |

### `cssVars` — the custom colour layer

```typescript
interface ThemeConfig {
  light: Record<string, string>;   // token -> value, light mode
  dark: Record<string, string>;    // token -> value, dark mode
  custom: CustomVariable[];         // { name, light, dark, description? }
}
```

`cssVars` is **Layer 2** of the two-layer model. The base palette (Layer 1) is resolved from `baseColor`, `colorPreset`, `chartColor`, and `menuAccent`; `cssVars` holds only the *delta* the user has hand-tuned on top of that base. The effective theme is `base ⊕ delta`. Changing a preset while a delta exists triggers an **ask-per-change** prompt (keep / rebase / discard). The `custom` array carries arbitrary user-defined CSS variables, each mapped into `@theme inline` automatically.

<Aside type="tip" title="Two histories">
Saving a theme snapshots the combined `StyleConfig` into `style_theme_version`. Separately, whenever the shadcn-config (preset) layer changes, a snapshot is written to `style_shadcn_config_version` — a preset-only history surfaced in the **Shadcn Config** tab, independent of the **Theme** tab's combined history.
</Aside>

### `defaultStyling` — per-slot base restyles

```typescript
interface DefaultStyleRule {
  slot: string;                       // a component's data-slot, e.g. "sheet-content"
  properties: Record<string, string>; // CSS property -> value, e.g. { "max-width": "48rem" }
  scope?: 'light' | 'dark';           // omitted = both colour modes
}
```

Each rule restyles **every instance** of a shadcn primitive via its `data-slot` attribute, emitted inside an `@layer components` block so per-instance utility classes still win. This layer is independent of the palette layers and is always preserved across preset changes and rebases.

## Status Colour Tokens

Beyond the shadcn palette, the module manages four semantic status tokens, each with a `-foreground` counterpart, generated for both light and dark modes:

| Token | Foreground token |
|-------|------------------|
| `--destructive` | `--destructive-foreground` |
| `--success` | `--success-foreground` |
| `--warning` | `--warning-foreground` |
| `--info` | `--info-foreground` |

Catalogue entries and module UI reference these semantic tokens (e.g. `var(--warning)`) rather than Tailwind colour utilities, so they stay theme-safe across apps that define no `text-warning`/`text-success` utilities.

## The Style Catalogue

The module ships a catalogue of **opt-in** `styleRegistry` entries (`utils/catalogue.ts` + `registrations/catalogue.registration.ts`). Every entry targets shadcn's stable `data-slot` / `data-variant` / `data-state` attributes — **never class names** — so component refactors don't break them.

| Catalogue ID | What it does | Default |
|--------------|--------------|:-------:|
| `style.tabs-line-primary` | Larger line-variant tabs with primary hover/active states | off |
| `style.button-bold` | Heavier default-variant buttons with more weight and padding | off |
| `style.card-elevated` | Opt-in card elevation (shadow + hover lift) owned by the module | off |
| `style.sidebar-rounded` | Rounds sidebar menu items to `calc(var(--radius) + 2px)` | off |
| `style.sidebar-accent-bold` | Redirects the sidebar's selected/hover accent to `--primary` | off |

<Aside type="tip" title="How entries are toggled">
The first four entries are flipped on and off by admins from the starter's `AppSettings.styleOverrides` UI (each ships `enabled: false`). The fifth, `style.sidebar-accent-bold`, is **driven by `config.sidebarAccent`** rather than manual toggling — the theme page calls `styleRegistry.setOverride(…, config.sidebarAccent === 'bold')`, and it registers at `order: 70` so it layers *after* the baseline app-theme entry (`order: 50`).
</Aside>

The style module is the **sole stylist**: it is the only module that calls `styleRegistry.register()` or edits `components/ui/*`. Other modules request a catalogue entry rather than contributing styling directly. See [Integration](/modules/style/integration/) and the starter's [Style System](/starter-template/style-system/).

## Database Schema

Three self-contained models. Table names derive from the `my_feature_*` schema templates via the module-id placeholder (`my_feature` → `style`), so the installed tables are:

| Model | Holds |
|-------|-------|
| `style_theme` | A saved theme (app-wide, per-project, personal, or system). Columns include `userId`, `projectId?`, `name`, `displayName`, `config` (the `StyleConfig` JSON), and the scope flags `isGlobal`, `isSystem`, `isAppOverride`, `isActive` |
| `style_theme_version` | Combined-theme version snapshots (`themeId`, auto-incrementing `version`, optional `label`, full `config` snapshot) |
| `style_shadcn_config_version` | Preset-layer-only version snapshots — written only when the shadcn-config layer actually changes (deduped in the service) |

<Aside type="note" title="Self-contained by design">
Per the module database convention, the schema uses plain foreign-key columns (`userId`, `themeId`, `projectId`) with `@@index`/`@@unique` — never Prisma `@relation` directives — so it never modifies the shared User model. See [Module Database](/module-starter/database/).
</Aside>

The backend `securityRegistry` manifest classifies `style_theme` personal rows and both version tables as **owned** (hard-deleted on account erasure) and system themes / app-theme overrides as **preserved** (they outlive the authoring account). The paired deletion handler erases a subject's personal themes and their snapshots in one transaction.

## Template Placeholders

This repo authors templates with placeholders; the scaffold substitutes them with the module id (`style`) at install. Understanding the mapping helps when reading generated code:

| Placeholder | Replaced with | Example (in the installed app) |
|-------------|---------------|-------------------------------|
| `my-feature` | `style` (kebab-case / module id) | `features/style/`, catalogue id `style.button-bold` |

| `myFeature` | `style` (camelCase) | `styleService`, event `style:theme-saved` |
| `my_feature` | `style` (snake_case) | tables `style_theme`, `style_theme_version` |

## Planned Configuration

<Aside type="note" title="In development">
**Admin-only per-managed-app styling.** The intent is to reuse a managed app's shadcn config and custom theme to style the app an admin *manages* — so an operator can theme a downstream app from the same editor. Today only two styling surfaces exist: the app-wide `/admin/theme` page (one active theme per app, contributed through `styleRegistry` at bootstrap) and per-project themes via the structure tabs (full mode). Cross-app admin styling is not built yet.
</Aside>
