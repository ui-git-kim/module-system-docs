Once Document Management is installed (`npx document-management init`), it adds a **Documents** item to your main navigation, a **Storage Providers** admin page, a **Recent Documents** dashboard widget, and a preferences panel. This page walks you through the day-to-day flow — connecting a provider, browsing, previewing, favouriting, and linking documents to Structure nodes — and then gives you the complete reference for the frontend components, data hooks, and backend API.

<Aside type="tip" title="Connector, not storage">
Remember the core principle: this module is a **connector, not a store**. The database keeps references, metadata, and your organisation (favourites, tags, node links) — it never holds file content. Files always live in the storage provider you connect.
</Aside>

## Getting Started: The Core Flow

The typical journey runs from the admin page (connect a provider) to the browser (find a file) to a document reference (organise it). Here is the end-to-end sequence.

<Steps>

1. **Add a storage provider** at `/admin/documents`

    Open the **Storage Providers** admin page (registered as `adminPagesRegistry` id `documents`, so the route is `/admin/documents`). Click **Add Provider**, choose a `providerType` (`s3`, `webdav`, or `local` — whichever your install enabled), and fill in the connection details. Credentials are encrypted with AES-256-GCM before they hit the database and are redacted from every API response.

    ```ts
    // The dialog calls useCreateProvider() under the hood:
    const createProvider = useCreateProvider();

    await createProvider.mutateAsync({
      name: 'Evidence Bucket',
      providerType: 's3',
      isDefault: true,
      isReadOnly: true,            // legal/evidence stores — writes throw ReadOnlyError
      config: {
        bucket: 'evidence-au',
        region: 'ap-southeast-2',
        accessKeyId: 'AKIA...',
        secretAccessKey: '...',    // encrypted before the DB write, never returned
      },
    });
    ```

2. **Test the connection**

    Click **Test** on the provider card. This calls `POST /providers/:id/test`, which connects through the adapter, refreshes `status` (`connected` | `error` | `pending` | `unavailable`) and `statusMessage`, and stamps `lastCheckedAt`. A green status means you are ready to browse.

3. **Browse files** at `/documents`

    The **Documents** nav item (`navigationRegistry` id `document-management`, icon `FileText`, order 25) opens `DocumentBrowserPage`. The `ProviderSidebar` lists your providers and folder bookmarks; the `FileBrowser` shows the current folder in grid or list view. Navigating into a folder routes to `/documents/browse/:providerId/*` (`FolderViewPage`).

4. **Preview a file**

    Selecting a file opens `FilePreview` (or routes to `/documents/preview/:documentId` via `DocumentPreviewPage`). The component picks a renderer from the file's MIME category — inline image, `react-pdf`, code block, HTML5 video/audio, or a download prompt. See [File Preview Strategy](#file-preview-strategy) below.

5. **Favourite and tag it**

    Marking a file as a favourite (or adding tags/notes) creates a `doc_document` reference via `POST /documents` and flips `isFavourite`. The reference stores `remotePath`, `fileName`, `mimeType`, `fileSize`, a server-computed `contentHash` (SHA-256), and your `tags`/`notes`. Favourites appear under `GET /documents/favourites` and in the `RecentDocumentsWidget`.

6. **Link it to a Structure node** (optional)

    If the Structure module is installed, the `DocumentLinkDialog` (which uses Structure's `NodeSelector`) links a document to a node via `POST /nodes/:nodeId/documents`, creating a `doc_node_link` row with a `linkType` of `attachment`, `reference`, or `primary`. The link fires the `documents.document.linked` action hook so other modules can react.

</Steps>

<Aside type="caution" title="Structure integration is opt-in">
The Structure side-effect imports in the generated `index.ts` are **commented out** by default. Uncomment them only once the Structure module is installed — then the Documents tab, node widgets, badges, actions, and sidebar section appear on node detail pages. Without Structure, the module runs perfectly well as a standalone document browser.
</Aside>

## Adding a Storage Provider

The admin page at `/admin/documents` is where you register the external stores the module connects to. Each provider maps to one row in `doc_storage_provider` and one of three adapters.

<CardGrid>
  <Card title="S3 / Cloudflare R2" icon="cloud-download">
    S3-compatible cloud storage via `@aws-sdk/client-s3`. Supports presigned preview and thumbnail URLs. Needs `bucket`, `region`, `accessKeyId`, `secretAccessKey` (and an optional `endpoint` for R2).
  </Card>
  <Card title="WebDAV" icon="external">
    ownCloud, Nextcloud, or a NAS via the `webdav` npm package. Authenticates with basic auth or a bearer token. Needs `baseUrl` plus credentials.
  </Card>
  <Card title="Local filesystem" icon="laptop">
    Node `fs`, for development only. Restricted to a base directory (`DOC_LOCAL_BASE_PATH`) and rejects `../` and absolute path traversal.
  </Card>
</CardGrid>

Key fields you set when creating a provider:

| Field | Type | Purpose |
|-------|------|---------|
| `name` | `string` | Human-readable label shown in the sidebar |
| `providerType` | `s3` \| `webdav` \| `local` | Which adapter backs this provider |
| `config` | object (encrypted) | Connection details + credentials — encrypted at rest, redacted in responses |
| `isDefault` | `boolean` | Pre-selected provider in the browser |
| `isReadOnly` | `boolean` | Injected into `createAdapter(...,{ readOnly })`; write ops throw `ReadOnlyError` at the adapter level |
| `basePath` | `string` | Optional path prefix scoping the provider to a subtree |

<Aside type="note" title="Which adapters are available?">
The install prompt (`storageAdapters`: `s3` | `webdav` | `local` | `all`, default `s3`) decides which adapter types the UI offers, exposed at runtime as the `ENABLED_ADAPTERS` constant in `document-management.config.ts`. All three adapter SDKs are **always** installed regardless of the selection, so the generated app always type-checks — only the offered set changes.
</Aside>

## Browsing and Organising

After a provider tests green, head to `/documents`. From here you can:

- **Navigate folders** — click into folders; the URL updates to `/documents/browse/:providerId/*` and the breadcrumb tracks your path.
- **Switch views** — toggle between `FileGrid` and `FileList` with the `viewMode` control.
- **Bookmark a folder** — pin a frequently used folder with `useCreateBookmark()`, writing a `doc_folder_bookmark` row that then appears in the `ProviderSidebar`.
- **Upload a file** — drop a file to upload it via the raw-body endpoint (`POST /upload/:providerId?path=...`); size is enforced on the byte stream at `DOC_MAX_UPLOAD_SIZE` (50MB default), not the spoofable `Content-Length`.
- **Favourite / tag** — create a `doc_document` reference to organise the file independently of where it lives.
- **Bulk download** — select several files and `POST /bulk-download` to stream a zip (built with `archiver`).

Every view, download, upload, link, and unlink is recorded in `doc_recent_access`, which doubles as a chain-of-custody audit trail and feeds the recent-documents lists.

## Data Hooks

All data access goes through the TanStack Query hooks in `hooks/useDocumentManagement.ts`. Consume these from components rather than calling the service directly — the shared `QueryClient` handles caching, deduplication, background refetch, and request cancellation. Keys are namespaced with `document-management`.

| Hook | Kind | Description |
|------|------|-------------|
| `useProviders()` | query | List all storage providers for the user. |
| `useProvider(id)` | query | Fetch a single provider by id. |
| `useCreateProvider()` | mutation | Create a provider; invalidates the providers list. |
| `useUpdateProvider()` | mutation | Update a provider (`{ id, input }`). |
| `useDeleteProvider()` | mutation | Delete a provider by id. |
| `useTestProvider()` | mutation | Run a connection test; refreshes status/capabilities. |
| `useFolder(providerId, path, options?)` | query | List a folder's contents (browse). |
| `useFileInfo(providerId, path)` | query | Fetch metadata for a single file. |
| `useUpload()` | mutation | Upload a file (`{ providerId, path, file, onProgress? }`); invalidates folder + recent. |
| `useDocuments(options?)` | query | List document references with filters (`limit`, `cursor`, `providerId`, `status`, `isFavourite`, `tag`). |
| `useDocument(id)` | query | Fetch a single document reference. |
| `useRecentDocuments()` | query | Recently accessed documents. |
| `useFavourites()` | query | Documents flagged `isFavourite`. |
| `useCreateDocument()` | mutation | Create a document reference; invalidates documents + favourites. |
| `useUpdateDocument()` | mutation | Update a reference (`{ id, input }`); patches the cache. |
| `useDeleteDocument()` | mutation | Delete a reference by id. |
| `useNodeDocuments(nodeId)` | query | Documents linked to a Structure node. |
| `useNodeDocumentCount(nodeId)` | query | Count of documents linked to a node (drives the badge). |
| `useLinkDocument()` | mutation | Link a document to a node (`{ nodeId, input }`). |
| `useUnlinkDocument()` | mutation | Remove a node link (`{ nodeId, documentId }`). |
| `useSearch(q, params?)` | query | Search across files (`providerId`, `limit`); disabled when `q` is empty. |
| `useBookmarks(providerId?)` | query | List folder bookmarks, optionally scoped to a provider. |
| `useCreateBookmark()` | mutation | Create a folder bookmark. |
| `useDeleteBookmark()` | mutation | Delete a bookmark by id. |

```tsx
// Example: a favourites list using two hooks
import { useFavourites, useUpdateDocument } from '@/features/document-management';

function FavouritesList() {
  const { data: favourites, isLoading } = useFavourites();
  const updateDocument = useUpdateDocument();

  if (isLoading) return <Skeleton />;

  return (
    <ul>
      {favourites?.map((doc) => (
        <li key={doc.id}>
          {doc.fileName}
          <button
            onClick={() =>
              updateDocument.mutate({ id: doc.id, input: { isFavourite: false } })
            }
          >
            Unfavourite
          </button>
        </li>
      ))}
    </ul>
  );
}
```

## Backend API Reference

All endpoints are served under the base path **`/api/v1/document-management`** and require authentication (`requireAuth: true`). The design spec defines **22 endpoints across 5 groups**; this implementation adds **3 folder-bookmark endpoints**. Paths shown below are relative to the base path. The `*` in browse/file paths is a wildcard capturing the remote path within the provider.

<Aside type="caution" title="Credentials are never returned">
Provider responses are produced by `toProviderResponse`, which omits the encrypted `config`. The frontend never receives long-lived credentials. For S3 previews, the client may receive a short-lived presigned URL — for preview only.
</Aside>

<Aside type="note" title="Uploads use a raw body, not multipart">
`POST /upload/:providerId` reads the raw request body as the file and takes the destination (including filename) from `?path=`, deliberately avoiding a dependency on a specific multipart middleware. The upload size is enforced by a counting transform on the byte stream that aborts at `DOC_MAX_UPLOAD_SIZE` — not the spoofable `Content-Length` header.
</Aside>

## File Preview Strategy

`FilePreview` derives a category from the file name and MIME type (via `fileCategory` in `file-utils`) and renders accordingly. Where bytes are needed, the client blob-fetches them through `/download` so authentication is preserved — which is why the `/preview` endpoint returns a JSON directive rather than streaming inline.

| MIME category | Example extensions | Rendering approach | Library |
|---------------|--------------------|--------------------|---------|
| Image | `jpg`, `png`, `gif`, `webp`, `svg` | Inline `<img>` with zoom | native |
| PDF | `pdf` | Paginated document viewer | `react-pdf` |
| Text | `txt`, `md`, `csv` | Code block (bytes fetched via `/download`, read as text) | native |
| Code | `js`, `ts`, `py`, `json`, and more | Code block (bytes fetched via `/download`, read as text) | native |
| Video | `mp4`, `webm` | HTML5 `<video>` player | native |
| Audio | `mp3`, `wav`, `ogg` | HTML5 `<audio>` player | native |
| Office | `docx`, `xlsx`, `pptx` | Download prompt + file icon (no inline preview in v1) | — |
| Archive | `zip`, `tar`, `gz` | Download prompt + file icon (no inline preview in v1) | — |
| Other | anything unrecognised | Download prompt + file icon | — |

```tsx
import { FilePreview } from '@/features/document-management';

// FilePreview picks the renderer from the file's MIME category internally
<FilePreview file={selectedFile} providerId={providerId} />
```

<Aside type="tip" title="Why preview returns JSON">
For presigned providers (S3/R2) the preview directive is `{ kind: "url", url }` with a short-lived URL the client loads directly. For everyone else it is `{ kind: "inline" }`, signalling the client to blob-fetch through the authenticated `/download` route. This keeps long-lived credentials off the client while still allowing fast in-browser previews.
</Aside>

## When a Provider Is Offline

A provider's `status` is one of `connected`, `error`, `pending`, or `unavailable`. Storage providers go down, credentials rotate, and NAS boxes reboot — the UI is built to degrade gracefully rather than break.

<CardGrid>
  <Card title="Unavailable card" icon="warning">
    When a provider is unreachable, `FileBrowser` swaps the file grid for an **unavailable card** that explains the problem and offers a **Retry** button instead of showing an empty or broken folder.
  </Card>
  <Card title="Cached references survive" icon="document">

  </Card>

    The Retry button (and the admin **Test** action) call `POST /providers/:id/test`, which re-runs the connection, refreshes `status`, `statusMessage`, `capabilities`, and `lastCheckedAt`. A successful test restores normal browsing.
  </Card>

```tsx
import { useFolder, useTestProvider } from '@/features/document-management';

function FolderArea({ providerId, path }: { providerId: string; path: string }) {
  const { data, error } = useFolder(providerId, path);
  const testProvider = useTestProvider();

  if (error) {
    return (
      <div className="unavailable-card">
        <p>This storage provider is currently unavailable.</p>
        <button onClick={() => testProvider.mutate(providerId)}>Retry</button>
      </div>
    );
  }

  return <FileBrowser providerId={providerId} initialPath={path} />;
}
```

<Aside type="note" title="Read-only providers">
A provider created with `isReadOnly: true` rejects every write (upload, delete, move, copy) at the **adapter level** with a `ReadOnlyError` — `createAdapter(type, config, { readOnly })` injects the flag. This is the recommended setting for legal and evidence stores, where the SHA-256 `contentHash` and the `doc_recent_access` audit trail together give you tamper-evidence and chain of custody.
</Aside>
