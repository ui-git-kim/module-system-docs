This is the authoritative roadmap for the **cognitive-db** engine — the intelligence layer that powers [Cog Ingest](/modules/cog-ingest/). It reconciles the original concept-and-roadmap design map against the code audit that shipped **v1.0.0**, so every item below is marked with an honest status: **built**, **partial/stub**, **planned**, or **out of scope**.

<Aside type="note" title="How to read status labels">
The engine's design documents describe an ambitious end-state. Not all of it is built. Where a design feature is not yet shipped, it is labelled **planned** or **stub** here — never presented as if it ships. The v1.0.0 line is the ground truth for what exists today.
</Aside>

---

## 1. Completed — v1.0.0

The **1.0.0** release (2026-07-14) is the engine's first stable release: a stable public API under semantic versioning, hardened by a full multi-agent code audit and moved onto the module system's vendored delivery.

<CardGrid>
  <Card title="First stable release" icon="rocket">
    v1.0.0 marks the stable public API. Versioning follows `major.minor.patch` in line with the module ecosystem, tracked in the engine's `CHANGELOG.md`.
  </Card>
  <Card title="Audit remediation" icon="approve-check">
    A multi-agent audit of 96 files (~28k LOC) produced **84 findings** (2 critical, 12 high, 22 medium) — all verified and fixed before release.
  </Card>
  <Card title="Vendored delivery" icon="cloud-download">
    The engine now ships inside Cog Ingest (`vendor/cognitive-db.tgz`) and installs via a lifecycle hook — no npm registry or GitHub authentication, version locked to the module release.
  </Card>
  <Card title="Tenant model confirmed" icon="approve-check">
    `tenantId === userId` was formally adopted as the tenant model — ownership isolation by `userId`, no multi-tenant layer.
  </Card>
</CardGrid>

### What the audit fixed

The remediation touched correctness, security, and performance across the whole engine:

| Area | Representative fixes |
|------|----------------------|
| **Security** | SQL injection in graph search closed — every filter value now binds as a numbered placeholder; the scaffold's `execSync` no longer interpolates a path; `retentionDays` bound in cleanup SQL. |
| **Correctness** | Graph adjacency now resolves the far endpoint (centrality, community detection, and pathfinding had been operating on connection-node ids); atomic `createVersion` under `$transaction`; `userId` threaded onto relationship connections; connection-limit truncation removed; version-chain cycle guards; preference-detection valence parsing — and ~40 more. |
| **Performance** | DB-side aggregation instead of full table scans; a single batched connection fetch replacing an N+1 query storm; windowed trend queries. |

### Capabilities that are built and shipping

These are confirmed present in v1.0.0 by the audit:

- **Unified node model** — one `cog_node` table for content, entities, connections, patterns, questions, and ontology.
- **Embeddings + semantic search** over pgvector, including *filtered* semantic search (marketed as "hybrid" — see §5).
- **Connections as first-class, versioned nodes** with atomic `previousVersion` / `supersededBy` chains.
- **Ontology-as-data** — `concept` and `relationship_type` nodes, a base ontology, and `superClass` / `domain` / `range` constraints.
- **Perception** — graded LLM extraction of entities, relationships, emotions, concepts, and questions, with graceful degradation when the LLM is unavailable.
- **Tier-1 immediate connection discovery**, confidence propagation (learning), and working-memory event tracking with consolidation.
- **Condensation** — duplicate merging, version compression, archival, and summarisation.
- **Pattern detection** (preference / behavioural / temporal), **proactive gap and question detection**, and some contradiction detection.
- **Graph algorithms** — centrality, community detection, pathfinding, and similarity — plus batch processing and scheduling, token/metrics accounting, and entity matching and dedup.

<Aside type="tip">
For the architecture behind these capabilities, see the [engine overview](/modules/cog-ingest/engine/). This roadmap focuses on direction, not mechanics.
</Aside>

## 2. Intentionally out of scope

### Multi-tenant / teams isolation — won't do (by design)

The engine is **solo-business**: one user *is* one tenant. Every table is keyed on `userId`, and that ownership *is* the isolation boundary. There is no teams or organisation layer and no separate `tenantId` column.

<Aside type="caution" title="This reframes the original security section">
An early security assessment (concept doc §9) proposed enforced multi-tenant isolation — non-nullable `userId`, mandatory tenant-scoping guards, and PostgreSQL Row-Level Security. That predates the `tenantId === userId` decision and is **not** the plan. Isolation is by `userId` **ownership**: the starter's security foundation isolates owned rows, and the consuming module's `security.registration.ts` declares `ownedModels: []` and relies on that ownership. Adding an org/team layer would be a different product.
</Aside>

`userId` being **nullable** is intentional, not a gap: system-level and ontology nodes are shared, not user-owned, so they carry a null `userId`. Audit items that asked to thread a *separate* tenant identity through validation (e.g. `ONT-01`) were deliberately **not** implemented — the fix, where relevant, is to scope by `userId` ownership, which the account-erasure pathway already respects.

The lawful **deletion / erasure pathway is implemented** — the consuming module's account-erasure deletion handler cascades through the module's per-user data. "Never decay" governs *relevance ranking over time*; it never meant "cannot delete on lawful request".

## 3. Near-term

Actionable next steps, with anything multi-tenant removed per §2.

### Resolve the open compliance decisions

Four decisions from the security profile still need a product answer before the corresponding controls can be built:

| Decision | Status |
|----------|--------|
| Desired retention periods | Needs a decision |
| Whether EU / UK users are in scope | Needs a decision |
| Key-management preferences | Needs a decision |
| Session-policy specifics | Needs a decision |

Related: **third-party AI processor disclosure** — content leaving to US-based LLM and embedding providers is a data-flow disclosure consideration under Australian privacy law (and potentially others depending on where an app's users sit).

### Append-only access-audit trail

**Status:** Planned

The existing `cog_processing_log` records *what happened during processing*, not *who read what, when*. A separate, append-only read/access audit trail is planned for apps handling sensitive content (the primary consumer, Evidence Manager, handles legal evidence). This is additive and does **not** depend on any multi-tenant layer.

### Structure-integration gaps

The [Structure module](/modules/structure/) is the foundational container/hierarchy module that Cog Ingest maps sources into. Several integration points identified in a cross-reference review remain open — see §4.

## 4. Structure integration gaps

These are the concrete integration gaps between the engine and the Structure module. All are **planned**, none are built.

<Steps>

1. **`contexts` parameter on ingest** — Structure's materialised paths (its hierarchy) should propagate into a `validInContexts`-style mechanism on the engine. The engine's ingest inputs do not yet accept a `contexts` parameter.

2. **Bulk context-update on reparenting** — when a Structure node is reparented, everything beneath it changes context. There is no bulk context-update operation to propagate that shift into the graph.

3. **Callback / event system back into Structure** — there is currently no pathway for cognitive intelligence (suggested connections, notifications) to flow *back into* Structure. Structure's 16 action hooks are a planned integration surface, not yet fully wired.

4. **Node-type → ontology mapping** — Structure's node types should auto-map to cognitive-db ontology concepts. This is not yet formalised. (The `node-reference` field type was identified as a plausible bridge for entity-to-Structure-node linking.)

</Steps>

<Aside type="note">
Structure's explicit, user-created connections and the engine's semantically discovered connections are **complementary but distinct** — the integration must keep them separate in both the data model and the UI, not conflate them.
</Aside>

## 5. Feature completions

These are the deferred stubs the 2026-07-14 audit documented rather than implemented speculatively. Each was left as an honest stub; each needs a product decision before building.

### Discovery Tier 2 / Tier 3

**Priority:** High · **Status:** Stub

`discovery.service` returns **Tier 1** results only. The semantic **entity-matching** and **potential-contradiction** tiers are empty stubs (`entityMatches` / `potentialContradictions` always empty). Building them means defining how aggressively the engine cross-references entities and flags conflicts.

### Budget persistence

**Priority:** Medium · **Status:** In-process stub

Token / budget tracking (`resources/budget.service`) is computed **in-process only** — per-user budgets and alert thresholds are never persisted or invalidated. This needs a storage model and a policy for what happens when a budget is exceeded.

### Hybrid lexical + semantic search

**Priority:** Medium · **Status:** Partial

`hybridSearch` is presently *filtered semantic* search — cosine-distance vector ordering plus `WHERE` filters, with no lexical / full-text score blended in. (The unused `semanticWeight` option was removed in v1.0.0.) A true hybrid would combine `ts_rank` / `ILIKE` scoring with vector distance under a tunable weight.

### `undoMerge` connection replay

**Priority:** Low · **Status:** Status-only

`condensation.undoMerge` restores node **status** but does not reverse the connections redirected during the merge. A full restore needs the merge step to record what it redirected so it can be replayed.

### Cross-item batch entity dedup

**Priority:** Low · **Status:** Single-ingest only

Entity dedup is correct *within* a single ingest, but concurrent items in `ingestBatch` sharing a brand-new entity can still race and create duplicates. Closing this needs cross-item serialisation or a post-batch dedupe pass.

## 6. Research agenda (R1–R7)

A neuroscience-informed refinements plan (24 January 2026) layers seven enhancements onto the node model. **None of this is built.** The whole layer is designed to be **additive and opt-in** (config flags), non-breaking to existing APIs — no phase requires the others to deliver value.

| Phase | Name | Core idea |
|-------|------|-----------|
| **R1** | Hebbian Foundations | Co-activated connections strengthen (`Δw = η × activation_i × activation_j`); never decay, only strengthen from a baseline. |
| **R2** | Spreading Activation | Pre-compute activation weights at ingest so retrieval needs no on-the-fly graph traversal. |
| **R3** | Enhanced Perception | Experiential-feature extraction (sensory / motor / affective / situational) plus late chunking, so document-level context survives chunk boundaries. |
| **R4** | Consolidation Engine | Replay-based background processing that periodically re-activates and strengthens important recent knowledge, mimicking sleep consolidation. |
| **R5** | Temporal Patterns | Sequential pattern mining (PrefixSpan) — detects "A is reliably followed by B", not just co-occurrence. |
| **R6** | Attractor Retrieval | Hopfield-style convergence for vague / partial queries ("that thing about… last month"). |
| **R7** | Importance Weighting | Fisher-information importance scoring feeding Elastic Weight Consolidation, so important connections resist being overwritten. |

### The connection-strength formula

The headline of the research is that shared **experiential** features should outweigh raw embedding similarity — a deliberate correction to a pure-RAG approach that over-weights taxonomic and semantic closeness:

```typescript
connectionStrength =
  experiential * 0.40 +
  semantic     * 0.30 +
  temporal     * 0.20 +
  taxonomic    * 0.10;
```

`taxonomic` draws on the ontology's `is-a` hierarchy; the four components are weighted, not collapsed into a single cosine score.

### Sequencing

R1, R3, and R5 have no dependency on each other and can proceed in parallel. R4 and R6 both depend on R2 (but not on each other). R7 depends on R4.

```text
Existing phases (A–F)
        │
   ┌────┼──────────────┐
   ▼    ▼              ▼
  R1   R3             R5
   │
   ▼
  R2
   │
 ┌─┴───────┐
 ▼         ▼
 R4        R6
 │
 ▼
 R7
```

<Aside type="note">
None of R1–R7 blocks app-level work. Because every phase is opt-in behind a config flag and additive at the schema level (new columns, new tables), existing apps keep working unchanged until they enable a refinement.
</Aside>

## 7. Implementation phases (A–F)

The V2 "everything is a node" rewrite was scoped in six phases. The original design map could only *infer* their status; the 2026-07-14 audit **verified** it. Reconciled status:

| Phase | Focus | Status | Notes (verified by audit) |
|-------|-------|--------|---------------------------|
| **A — Core Graph** | Unified node model, embeddings, semantic search, basic connections | **Built** | The `cog_node` scaffold, embeddings, and pgvector search are all present. |
| **B — Deep Perception** | Multi-level perception, ontology typing, Tier-1 discovery | **Built** | Graded perception, ontology-as-data, and immediate discovery all ship. |
| **C — Patterns & Learning** | Pattern detection, version propagation, confidence updates | **Partial** | Pattern detection, confidence propagation, and version chains exist and were audited; propagation had correctness fixes applied. |
| **D — Working Memory & Proactive** | Session tracking, consolidation, contradiction / question / surprise detection | **Partial** | Working-memory events, consolidation, and proactive gap / question / contradiction detection exist and were verified. |
| **E — Resource Optimisation** | Token budgets, condensation, graceful degradation | **Partial** | Condensation and graceful degradation ship; **budget persistence is a stub** (§5). |
| **F — Full Intelligence** | Gap analysis, temporal reminders, advanced graph algorithms | **Mostly not started** | Graph algorithms exist; most advanced-intelligence work (and the R1–R7 layer) is not built. |

<Aside type="tip">
The earlier design map marked C, D, and E as "status unconfirmed — verify". The audit was that verification: the learning, pattern, working-memory, and proactive subsystems are real. What remains for C–E is completing the stubs in §5, not building from scratch.
</Aside>

## 8. Documents module — out of scope for the engine

A **Documents module** has been discussed for the app layer, but it is **out of scope for the engine**. cognitive-db is not a content-management system.

| Concern | Owner |
|---------|-------|
| Content reference (`sourceId` / `sourceType`) | cognitive-db |
| Knowledge evolution — *why understanding changed* | cognitive-db |
| Document versioning (v1 → v2 → v3 of a draft) | Documents module (app layer) |
| File storage, templates, export, annotations, rich-text editing, review workflows | Documents module (app layer) |

Two versioning concepts must not be conflated: **knowledge evolution** (how the engine's *understanding* changed, with a reason — an engine concern) versus **document versioning** (how a *document* changed draft-to-draft — an app concern). The Documents module remains at the design stage; it is not on the engine's roadmap.
