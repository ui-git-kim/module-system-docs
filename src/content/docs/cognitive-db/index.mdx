**cognitive-db** is a reusable TypeScript/npm package that gives any application a brain-like knowledge and intelligence layer. Its job is to *understand* content, not just store and retrieve it — and to surface connections a person might not make themselves.

It is the intelligence engine vendored inside the [Cog Ingest module](/modules/cog-ingest/). If you are building on that module, cognitive-db is the machinery doing the perception, connection discovery, and ontology work underneath.

<Aside type="tip" title="Not a vector DB, not a RAG library">
cognitive-db *uses* embeddings and retrieval as building blocks, but it is neither a vector database nor a RAG library. A vector DB finds text similar to a query. cognitive-db builds understanding — typed entities, versioned connections, patterns, and proactive insight — at the moment content arrives.
</Aside>

## The problem it solves

The engine's founding motivation was personal: compensating for how ADHD affects attention and working memory — particularly the **object-permanence** problem, where information that is out of sight stops existing in functional awareness.

But the design deliberately follows the **curb-cut effect**: every accommodation built for that specific challenge turns out to help everyone. Nobody reliably remembers to search for something they have forgotten exists. Everyone has "that thing about... the project... last month" moments. So cognitive-db is built as **universal design** from the outset, not a niche accessibility feature bolted onto a knowledge base.

| ADHD challenge | Accommodation built | Universal benefit |
|---|---|---|
| Working-memory limits | Proactive surfacing | Nobody remembers to search for forgotten things |
| Object permanence | Visible / active items | Out of sight = out of mind for everyone |
| Time blindness | Temporal anchoring | "When was this?" helps all recall |
| Difficulty starting | Pre-computed connections | Reduced cognitive load for all |
| Hyperfocus → losing context | Session / working-memory tracking | Context restoration benefits everyone |
| Pattern blindness | Automatic pattern detection | Humans miss statistical patterns generally |

## Core philosophy

These principles recur across every design conversation in the project and are treated as close to non-negotiable. They are in force in v1.0.0.

| Principle | What it means |
|---|---|
| **Build at ingest, not retrieval** | Connections, entity extraction, and understanding happen when content arrives — not on the fly at query time. Retrieval is fast because the thinking already happened. |
| **Never decay** | Biological forgetting serves metabolic and capacity limits that do not apply to a database. Relevance *ranking* may downscale, but information is never deleted — a decade-old note can be the single most important thing in the system. (The one exception is the lawful erasure pathway; see below.) |
| **Proactive over reactive** | The system surfaces contradictions, forgotten context, and connections without being asked, rather than waiting for a query. |
| **Deep perception** | Extraction goes beyond keywords: intention, emotion, concepts, questions raised, and action items, all captured once at ingest. |
| **Everything is a node** | Entities, connections, patterns, ontology concepts, and content all live in one unified, versioned graph structure — a single `cog_node` table. |
| **Generic, not app-specific** | No hardcoded domain logic. Entity types, relationship types, prompts, and models are all supplied by the consuming app via `initConfig()`. |
| **Apps own their data** | cognitive-db never stores your original content — only its understanding of it, referenced back via `sourceId` + `sourceType`. |

## How this differs from "just RAG"

A plain vector database finds text similar to a query. cognitive-db additionally:

- Types entities against an app-defined ontology, not just tags.
- Builds a persistent graph of relationships as first-class, versioned objects.
- Tracks how understanding of something evolves over time, recording *why* each change happened (`previousVersion` / `supersededBy` chains).
- Runs **tiered discovery** (immediate → background → periodic → on-demand) so expensive connection-finding never blocks ingest.
- Treats connections and patterns themselves as discoverable, versionable things — not just edges in a lookup table.

<Aside type="note" title="Isolation is by ownership, not multi-tenancy">
cognitive-db follows a solo-business model: `tenantId === userId`. One user *is* the tenant — there is no separate teams or organisation layer and no multi-tenant scoping. Rows are isolated by `userId` **ownership** through the starter's security foundation; the consuming module's `security.registration.ts` declares `ownedModels: []` and relies on that ownership. A nullable `userId` is intentional — system and ontology nodes are shared, not user-owned. Lawful erasure is implemented via the module's account-erasure deletion handler.
</Aside>

## Status

<Aside type="caution" title="v1.0.0 — first stable release">
cognitive-db reached **v1.0.0** on 2026-07-14, the first stable release after a full code audit. The unified node model, embeddings and semantic search, versioned connections, ontology-as-data, LLM perception, Tier-1 connection discovery, working-memory tracking, condensation, and pattern detection are **built**. Some discovery tiers, budget persistence, and the R1–R7 neuroscience refinements are **stubbed or planned** — each is labelled as such where it appears in these docs. The engine is delivered vendored into Cog Ingest (no npm or GitHub auth required); standalone install is available via `npm install github:ui-git-kim/cognitive-db`.
</Aside>
