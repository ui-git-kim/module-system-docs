## Planned

Features planned for upcoming releases.

### Multiple Layouts

**Priority:** High

Currently the template has a single dashboard layout with sidebar navigation. This enhancement will add:

- **Builder/Editor Layout** - Full-bleed canvas for visual editors (already have `FullBleed` component)
- **Wizard Layout** - Step-based flows for onboarding or multi-step forms
- **Focus Layout** - Minimal chrome for focused tasks (reading, writing)
- **Landing Layout** - Marketing pages without app navigation

**Implementation approach:**
- Layout registry for registering layout components
- Route-level layout selection
- Nested layouts for flexibility

**Related:** the `styleRegistry` (implemented alongside тАФ see changelog for the landing version) is the sibling feature for module-contributed styling; the two should compose (a module may want to ship both a layout and a matching style profile).

### In-App Component Reinstall for Style Registry

**Priority:** Medium

The `styleRegistry` landed with a terminal-only reinstall flow тАФ when a module's `componentsJson` contributions change (new `style`, `baseColor`, `cssVariables`, added registries, etc.), admins need to run `npm run components:sync` in a terminal to apply the changes. This enhancement brings that flow into the admin UI.

**Scope:**

- Backend endpoint that spawns `scripts/sync-components.{ps1,sh}` as a child process.
- Streams stdout/stderr back to the admin UI over Server-Sent Events or WebSocket.
- Admin page surfaces: "Reinstall N components to apply changes? [Reinstall] [Keep staged]".
- "Keep staged" lets an admin batch multiple `styleRegistry` changes before a single reinstall.
- Progress feedback (which component is currently being reinstalled, any errors).

**Depends on:** [Admin Module Management](#admin-module-management) landing first. That feature will establish the node-spawn-and-stream pattern for module install/update/uninstall, which this reinstall flow will reuse rather than reinvent.

**Constraint:** some deploy environments (locked-down hosting, containers without a writable filesystem) will block shelling out to `npx`. This feature is therefore opt-in тАФ apps that can't support it can continue using the terminal sync.

### Admin Module Management

**Priority:** High

Replace CLI-based module operations with an admin panel UI. Currently modules are installed, updated, and removed via terminal commands (`npx my-module init`, `update`, `uninstall`). This feature brings that workflow into the app itself:

- **Browse & install** тАФ Discover available modules (from a registry or local packages) and install them from the admin panel
- **Configure** тАФ Module-specific settings and prompt answers managed through a UI instead of interactive CLI prompts
- **Deactivate / activate** тАФ Toggle modules on or off without removing files (replaces `npx my-module disable/enable`)
- **Uninstall** тАФ Remove modules cleanly with confirmation, data export options, and rollback (replaces `npx my-module uninstall`)
- **Update** тАФ Check for module updates and apply them with a diff preview (replaces `npx my-module update`)
- **Module status dashboard** тАФ View installed modules, versions, health, and dependencies at a glance

**Implementation considerations:**
- Backend service that wraps scaffold operations (init, update, uninstall, enable, disable)
- Admin page registered via `adminPagesRegistry`
- Module manifest (`.module-manifests.json`) read/write from the backend
- Restart or hot-reload mechanism after module changes take effect
- Permission-gated to admin role

### Neon Plugin Integration & Live DB Audit

**Priority:** Medium

Install Neon's official Claude Code plugin (`neon@claude-plugins-official`) and run a full database audit against the actual Neon project тАФ not just the public docs. Parked after v1.22.2 because the plugin install hit SSH-auth friction on a fresh machine and the doc-only audit (see v1.22.2 changelog) already confirmed the backend stack matches Neon's published recommendations.

**What the plugin unlocks (once installed):**

- Query the actual project via MCP tools rather than inferring from docs.
- Verify `DATABASE_URL` vs `DIRECT_DATABASE_URL` resolve to the `-pooler` and direct endpoints Neon expects.
- Compare the configured `NEON_AUTH_URL` against the project's JWKS endpoint and trusted domains list.
- List branches and check for schema drift between `main` and any preview branches.
- Read the current Auth settings (email verification on/off, "Allow Localhost" toggle, custom SMTP configured, custom OAuth creds) against the production checklist added in v1.22.2.
- Inspect any existing RLS policies тАФ relevant if modules start adopting the Data API pattern documented in [Database & Prisma](/starter-template/database/#alternative-the-neon-data-api).

**Install note:** on machines without GitHub SSH keys, set `git config --global url."https://github.com/".insteadOf git@github.com:` before `claude plugin install`, otherwise the marketplace clone fails with `Permission denied (publickey)`.

**Expected outcome:** either "current setup matches project state, nothing to change" (most likely) or a short punch-list of Neon Console configuration items to update for a specific environment. No template code changes anticipated.

### styleOverrides Bootstrap Cache Fix

**Priority:** Medium

Module-registered style entries apply synchronously at module-load time; `AppSettings.styleOverrides` is fetched asynchronously during React mount. If an admin disabled an entry whose tokens visibly change the shell, users see a brief flash from the "on" state to the "off" state during the fetch.

**Fix:**

- Cache the last-known `styleOverrides` map in `localStorage` from `AppSettingsContext` on every update.
- In `main.tsx`, read the cache before React mounts and call `styleRegistry.applyOverrides(cached)` synchronously тАФ same pattern as the existing theme-class init that prevents the dark/light mode flash.
- On server response, reconcile and re-apply if the server-side map differs from the cache.

**Scope:** ~15 min. Fully backwards-compatible тАФ users without a cached value still get the current behaviour.

Related: [Style System ┬з Known limitation](/starter-template/style-system/#known-limitation--first-paint-flash), [Changelog v1.21.0](/starter-template/changelog/#v1210).

### Frontend Env Var Build-Time Validation

**Priority:** Low

The backend uses Zod to validate env vars at startup; the frontend currently doesn't validate `VITE_*` vars at all. A missing or malformed `VITE_NEON_AUTH_URL` surfaces as a runtime auth failure, not a build error.

**Fix:** add a Vite plugin (or a `tsx` preflight) that parses `import.meta.env` against a Zod schema and fails the build if required vars are missing. Mirrors the backend's `config/env.ts` pattern.

### App Documentation

**Priority:** Low

Optional documentation setup for apps created from the template:

- Create script prompts: "Would you like to set up documentation?"
- Scaffolds Astro/Starlight docs folder (same setup as module system docs)
- App-specific docs, not linked to module system shared docs
- Pre-configured with app name and basic structure
- Separate from the shared module system documentation

**Current state:** Apps inherit the `docs/` submodule from the template, which points to the shared module system docs. Apps should either have no docs folder or their own dedicated docs.

## Known Performance Issues

Findings from the v1.20.4 performance audit. Grouped by where the cost actually lands.

<Aside type="danger" title="Blocker before production">
  **Group C тАФ Backend list perf** is a pre-production blocker. The `getAllUsers()` endpoint returns every user row in the database with no `take`/`skip` pagination, and `User` has no index on `createdAt` (used for ORDER BY in the admin users list) or `role` (used for the "last admin" check). This is fine at dev scale but will degrade quickly past a few thousand rows and must be fixed before shipping to real traffic.
</Aside>

**Status as of v1.20.7:**
- тЬЕ Group A тАФ Context + callback hygiene (v1.20.5)
- тЬЕ Group B тАФ Bundle + debounce (v1.20.6)
- ЁЯЪз **Group C тАФ Backend list perf (pre-production blocker)**
- тЬЕ Group D тАФ Correctness cleanup (v1.20.7)

### Context & callback hygiene

**Priority:** High тАФ quick wins, zero behavioural risk

- **`AppSettingsContext.value` is not memoised** ([`frontend/src/features/admin/context/AppSettingsContext.tsx:159`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/context/AppSettingsContext.tsx)) тАФ the context wraps the entire app shell; every render of `AppSettingsProvider` creates a new object identity, forcing every consumer to re-render. Fix: wrap in `useMemo`.
- **`useNavigationOverrides.save` and `useToolbarOverrides.save` depend on `[overrides]`** тАФ the callbacks are recreated on every toggle/swap. Pages memoise their toolbar `items` against these callbacks via `useAdminToolbarActions`, which triggers a registry unregisterтЖТregister cycle on every interaction. Fix: pass the overrides to save as an argument, not a closure dep.
- **`NavItem` is not `React.memo`-wrapped** ([`frontend/src/features/dashboard/components/sidebar/NavMain.tsx`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/dashboard/components/sidebar/NavMain.tsx)) тАФ sidebar re-renders all items on every parent render. Low impact today (fewer than 50 items), worth fixing for module authors who might add many items.

### User-visible responsiveness

**Priority:** High

- **`AdminDataTable` search has no debounce** ([`frontend/src/features/admin/components/AdminDataTable.tsx:143`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/components/AdminDataTable.tsx)) тАФ every keystroke triggers a full TanStack table re-filter. Jank at 100+ rows. Fix: 200ms debounce.
- **Admin pages are in the main bundle** ([`frontend/src/features/admin/admin-pages.registration.ts`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/admin-pages.registration.ts)) тАФ the route already wraps in `<Suspense>`, so switching to `React.lazy()` is a clean win for the 98% of users who aren't admins.

### Backend list scalability

**Priority:** Medium тАФ only matters past a few thousand rows, but cheap to fix now

- **`admin.service.ts -> getAllUsers()` has no pagination** ([`backend/src/features/admin/admin.service.ts:321`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/src/features/admin/admin.service.ts)) тАФ returns every user row on every admin-users-page visit. Fix: add `page`/`limit` params and return `{ users, total }`; update `adminService` and `useAdminUsers` in lockstep (breaking API change тЖТ minor version bump, downstream modules that call this need updating).
- **`User` model missing indexes on `createdAt` and `role`** ([`backend/prisma/schema.prisma`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/prisma/schema.prisma)) тАФ `getAllUsers` orders by `createdAt` and `updateUserRole` counts admins by `role`. Add `@@index([createdAt])` and `@@index([role])`. Ships as a migration downstream apps need to run.

### Duplicate fetches & cleanup

**Priority:** Low тАФ correctness more than performance, but should be fixed

<Aside type="tip" title="Systemic fix landed in v1.22.0">
Adopting TanStack Query as the [standard data fetching layer](/starter-template/data-fetching/) supersedes most of the individual fixes in this section. Providers that migrate to `useQuery` get deduplication (no more double-fetch), built-in cancellation (no stale `setState` on unmount), and a shared cache as a matter of course. Migrate remaining hooks (`useAdminUsers`, `useNavigationOverrides`, `useToolbarOverrides`) as they're next touched тАФ the reference pattern is `AppSettingsContext`.
</Aside>

- тЬЕ **`AppSettingsPage` fetches app settings twice** тАФ page now reads from the global context (already landed); the context itself is backed by a shared `useQuery` since v1.22.0, so any other consumer that later mounts `['app-settings']` hits the cache.
- **Fetch effects without `AbortController`** in remaining hooks (`useAdminUsers`, `useNavigationOverrides`, `useToolbarOverrides`) тАФ stale `setState` after unmount. Preferred fix: migrate each to `useQuery` / `useMutation`, which handles cancellation and re-use for free.
- **`matchMedia` listener in `main.tsx` is never removed** тАФ module-level, runs once in production; HMR in dev can accumulate listeners. Low impact but worth fixing.
