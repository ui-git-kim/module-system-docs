## Planned

Features planned for upcoming releases.

### Security floor (v2.0.0)

**Priority:** Highest ŌĆö the foundation security architecture every app inherits. **Philosophy: delegate to expert providers, don't build security.** The starter is the thin, correct wiring of **Neon, Cloudflare, a managed KMS, and standard libraries**; bespoke code is a short, named allow-list only (app-layer field encryption, the deletion-cascade wiring, a thin `owned()` defence-in-depth). Landing in phases under a **v2.0.0** umbrella. See the [Security feature](/starter-template/security-feature/) for the full delegation map.

Phases (each implemented, reviewed, and stopped for sign-off before the next):

- Ō£ģ **P1 ŌĆö Security feature shell + `securityRegistry` + tier config** (v1.28.0). The declaration layer + `securityProfile` (`standard`/`hardened`/`strict`, default `hardened`) ŌåÆ `SecurityConfig` with env overrides.
- Ō£ģ **Auth hardening (Neon Auth, no DIY)** (v1.29.0). Locked-down API CSP (helmet) + documented edge-CSP/WAF guidance + reliance on Neon's short-lived/refresh tokens + cookie-mode-where-possible + MFA/step-up by tier ŌĆö *not* a custom session/cookie/CSRF system. (An earlier custom-cookie attempt was reverted as DIY.)
- Ō£ģ **Isolation floor ŌĆö Neon RLS + the `tenantId` convention** (v1.30.0). Owned models carry `tenantId` (isolation key) + `userId` (actor); `req.user.tenantId` (= user id in solo); `withTenantContext` (sets `app.tenant_id`) as the RLS seam + `owned()` keyed on `tenantId` as defence-in-depth; Database docs reframed to the RLS floor with the `FORCE RLS` + `tenantId` policy template. `tenantId` is present from the start (forward-compat) so going multi-user is a data change, not a schema rewrite. **Database-per-tenant** (hard isolation) remains a deferred opt-in capability ŌĆö see below.
- Ō£ģ **Field encryption (app-layer AES-256-GCM)** (v1.31.0). `encryptField`/`decryptField` over Node's `crypto` (random IV per call, GCM auth tag verified), plus `hashSecret`/`verifySecret` (SHA-256 + constant-time) for high-entropy tokens. Key from a provider abstraction ŌĆö **`env` adapter shipped; `kms` adapter as an interface** (`setKeyProvider`); per-tenant keys + crypto-shred are a KMS-adapter capability (strict). The key is **never sent to the database** (app-layer, not `pgcrypto`), so Neon only ever holds ciphertext. Off at `standard`, on at `hardened`+.
- Ō£ģ **Deletion cascade (provider-API wiring)** (v1.32.0). `deleteAccount()` runs registered deletion handlers in a fail-safe, classification-driven order (`credential` crypto-shred ŌåÆ `owned` hard-delete ŌåÆ `referenced` unlink; `preserved` deferred as a legal hold), then removes the **Neon Auth identity last** via the `AuthIdentityDeleter` seam (default `unconfigured` = visible, not silent ŌĆö closes the gap where `deleteUser` left the identity behind), and returns a consent + classification compliance bundle. Modules register handlers via `securityRegistry.registerDeletionHandler`; both the self-service and admin deletion paths run the cascade. A retention job to erase `preserved` holds when they lift remains out of scope (app-level).
- Ō£ģ **Field-encryption key custody (AWS KMS)** (v1.33.0ŌĆōv1.35.0). Resolves the `kms` adapter the field-encryption phase shipped as an interface. **Envelope encryption**: KMS holds the root key, the app unwraps a data key **once** at startup and caches it, AES-256-GCM runs locally (KMS isn't called per field); the plaintext key is never in env, the DB, or a Neon dump. `createAwsKmsKeyProvider({region,keyId,wrappedDek})` reference adapter at `@/features/security/key-providers/aws-kms`; `@aws-sdk/client-kms` is an **optional, app-installed** dep (dynamically imported, so non-KMS apps build without it). One-command `npm run security:setup-kms` (v1.34.0) provisions the wrapped key + injects the env; the provider then **auto-wires from env at bootstrap** (v1.35.0 ŌĆö no `app.ts` edit, survives `template:update`). Crypto-shred = disable/delete the KMS key (global); per-tenant crypto-shred is the documented growth path. Guide: [Field encryption with AWS KMS](/getting-started/aws-kms-setup/).
- Ō£ģ **Cloudflare edge** (v1.36.0). The perimeter (DDoS, WAF, bot, edge rate limiting, TLS/HSTS, the frontend XSS-CSP, optional Zero Trust) delegated to **Cloudflare** ŌĆö [setup guide](/getting-started/cloudflare-edge/) + a per-tier edge checklist. The code half is a **startup posture guardrail** (`runStartupPostureCheck()` in `features/security/posture.ts`, wired at the end of `app.ts` bootstrap): it evaluates the resolved posture once every feature + module has registered ŌĆö edge declaration, trust-proxy sanity, field-encryption key readiness, auth-identity deleter, module `minimumTier` requirements, production hardening ŌĆö logs each finding, and **halts at `strict` in production** on an error finding (override: `SECURITY_POSTURE_ENFORCE`). A one-shot `edgePostureMiddleware` warns when the first real request bypasses the edge (no `cf-ray`/`cf-connecting-ip`). New env knobs `SECURITY_EDGE` + `SECURITY_POSTURE_ENFORCE`; non-throwing `getKeyProviderStatus()` readiness probe. **The edge is additive ŌĆö its absence warns, never fails; the app-layer floor stands alone.**

**Deferred (no provider exists / heavier capability):**

- ŌÅ│ **Zero-knowledge encryption ŌĆö "the operator can't read the data".** No provider offers this: at-rest and field encryption still let the key-holder decrypt, and Neon Data Masking doesn't hide data from the project owner. The only mechanism is **end-to-end / client-side encryption** keyed by the user's secret, with hard tradeoffs (no server-side search/processing/AI on content; difficult key recovery). **Opt-in, per-app, designed later.** Today's baseline for the same intent is ciphertext-in-DB via app-layer field encryption (above) + access controls.
- ŌÅ│ **Database-per-tenant control-plane.** The provisioning (Neon API), tenantŌåÆconnection routing, and fan-out-migration machinery that makes database-per-tenant operable at scale for multi-tenant apps.

### Observability (core): System admin page + Audit/Errors

**Priority:** High ŌĆö **decision (2026-06-25): audit + error logging live in the starter core, not a separate module.** Consolidates the former *Database Health Page*, the *Audit & Errors module*, and the error-console half of *Error Pages & Offline Handling*. The seams already exist in core ŌĆö 9 lifecycle action hooks (`user.*` / `admin.*`), the single error-middleware choke point, `securityRegistry.getAuditEvents()` (declaration layer, already shipped), and the `SECURITY_AUDIT_MODE` knob ŌĆö so the foundation belongs here. Compliance-grade depth stays an **opt-in sink** so apps that don't need it carry no weight.

**Architecture ŌĆö one pipeline, swappable sink** (mirrors the `setKeyProvider` pattern):

- `AuditSink` interface ŌĆö `recordEvent()` / `recordError()` / `queryRecent()`, with `setAuditSink()` as the swap seam.
- **`MemoryAuditSink`** (default, always on) ŌĆö bounded ring buffers (~200 errors / ~500 activity), ephemeral, zero-config; powers the live console.
- **`PrismaAuditSink`** (optional, `hardened`+) ŌĆö `AuditEvent` / `ErrorLog` models; survives restarts, supports retention. A composite fans out to memory + DB so the live console always works.
- **Compliance sink** (opt-in; strict / `SECURITY_AUDIT_MODE=compliance`) ŌĆö immutable / hash-chained, no TTL, export hooks. The former "module", reframed as this sink.

**Capture (almost entirely additive):**

- **Errors** ŌĆö one line in `backend/src/middleware/error.middleware.ts` after `log.error()` ŌåÆ `auditSink.recordError(...)`. Optional `POST /api/v1/admin/system/client-error` from a React error boundary.
- **Activity** ŌĆö a new `backend/src/features/audit/audit.registration.ts` subscribes the sink to the 9 existing hooks via `actionRegistry.addActionAsync(hook, 'audit', cb)` ŌĆö no changes to the emitting services. `securityRegistry.getAuditEvents()` is the canonical filter set.

**The `/admin/system` page** (mirrors the v1.37.0 Security page ŌĆö `*-admin.routes/service/types` + page + `adminPagesRegistry`, admin-gated reads):

- **Database** ŌĆö `checkDatabaseHealth()` (connected / latency / adapter) + last migration (`_prisma_migrations`) + table row counts.
- **Services / connections** ŌĆö reuses `evaluateStartupPosture()` (the 7 startup findings) + `getKeyProviderStatus()`: Neon Auth/JWKS, KMS readiness, edge, DB adapter. A *window*, not a reimplementation.
- **Console** ŌĆö errors + activity tabs reading `auditSink.queryRecent()` via `GET /api/v1/admin/system/errors` and `.../activity`, rendered with the existing `AdminDataTable`.

**Security / redaction:** all reads are **admin-only** (posture + errors expose internals). Redaction reuses `securityRegistry.getEncryptedFields()` / `getHashedSecrets()` to scrub activity `changes`; stack traces are server-side / dev only; secrets shown set-or-not. DB-backed audit is owned data ŌåÆ `tenantId` / RLS applies.

**Phasing:**

- Ō£ģ **P1 (shipped v1.39.0)** ŌĆö `/admin/system` page + sink abstraction + `MemoryAuditSink` + the error-middleware hook + activity listeners (**errors and activity** console) + DB / services health. Ephemeral, in-memory, no schema change. See [System page](/starter-template/system/).
- Ō£ģ **P2 (shipped v1.40.0)** ŌĆö durable `SqlAuditSink` persisting to a **self-provisioned** `"AuditLog"` table (raw SQL `CREATE TABLE IF NOT EXISTS`, no Prisma model / migration / propagation ŌĆö a deliberate change from the original model-based plan to avoid the 5-file schema hand-sync tax) + TTL retention (`SECURITY_AUDIT_RETENTION_DAYS`) + a History tab. Opt-in via `SECURITY_AUDIT_SINK=database`; best-effort (degrades to memory).
- Ō£ģ **P3 (shipped v1.40.0)** ŌĆö `ComplianceAuditSink`: immutable, SHA-256 hash-chained (the exact canonical payload is stored per row, so verify is round-trip-safe), advisory-locked, never pruned; `audit/verify` + `audit/export` endpoints + a System-page integrity check. Auto-selected at the strict tier / `SECURITY_AUDIT_MODE=compliance`.
- **P4** (optional) ŌĆö client-error boundary capture into the console.

**Supersedes** the former separate *Audit & Errors module* and *Database Health Page* entries. The rate limiter stays unchanged ŌĆö 429s become visible in the console instead of being silenced.

### In-App Component Reinstall for Style Registry

**Priority:** Medium

The `styleRegistry` landed with a terminal-only reinstall flow ŌĆö when a module's `componentsJson` contributions change (new `style`, `baseColor`, `cssVariables`, added registries, etc.), admins need to run `npm run components:sync` in a terminal to apply the changes. This enhancement brings that flow into the admin UI.

**Scope:**

- Backend endpoint that spawns `scripts/sync-components.{ps1,sh}` as a child process.
- Streams stdout/stderr back to the admin UI over Server-Sent Events or WebSocket.
- Admin page surfaces: "Reinstall N components to apply changes? [Reinstall] [Keep staged]".
- "Keep staged" lets an admin batch multiple `styleRegistry` changes before a single reinstall.
- Progress feedback (which component is currently being reinstalled, any errors).

**Depends on:** [Admin Module Management](#admin-module-management) landing first. That feature will establish the node-spawn-and-stream pattern for module install/update/uninstall, which this reinstall flow will reuse rather than reinvent.

**Constraint:** some deploy environments (locked-down hosting, containers without a writable filesystem) will block shelling out to `npx`. This feature is therefore opt-in ŌĆö apps that can't support it can continue using the terminal sync.

### Admin Module Management

**Priority:** High

Replace CLI-based module operations with an admin panel UI. Currently modules are installed, updated, and removed via terminal commands (`npx my-module init`, `update`, `uninstall`). This feature brings that workflow into the app itself:

- **Browse & install** ŌĆö Discover available modules (from a registry or local packages) and install them from the admin panel
- **Configure** ŌĆö Module-specific settings and prompt answers managed through a UI instead of interactive CLI prompts
- **Deactivate / activate** ŌĆö Toggle modules on or off without removing files (replaces `npx my-module disable/enable`)
- **Uninstall** ŌĆö Remove modules cleanly with confirmation, data export options, and rollback (replaces `npx my-module uninstall`)
- **Update** ŌĆö Check for module updates and apply them with a diff preview (replaces `npx my-module update`)
- **Module status dashboard** ŌĆö Ō£ģ Shipped read-only (v1.50.0): the admin **Modules** page (`/admin/modules`, `adminPagesRegistry` order 16) shows registered modules merged with `InstalledModule` state, dependency issues (presence + range, with excusal flags), cycles and the computed boot order. Everything below still requires the trust-boundary decision first

**Implementation considerations:**
- Backend service that wraps scaffold operations (init, update, uninstall, enable, disable)
- Admin page registered via `adminPagesRegistry`
- Module manifest (`.module-manifests.json`) read/write from the backend
- Restart or hot-reload mechanism after module changes take effect
- Permission-gated to admin role

**Trust boundary (flagged in the v1.26.0 audit):** installing a module from the admin UI means executing untrusted code ŌĆö arbitrary side-effect imports run on the frontend at registration time, and the backend would spawn `npx`. There is no sandboxing or isolation today, and none is yet designed. Before this feature ships, decide what runs untrusted module code and how it is contained (signed modules / allowlist registry / out-of-process execution). Also note today's **build-time coupling** that blocks true runtime install-uninstall: features are wired via static side-effect imports in `app.ts` and `routeRegistry.mountAll()` is one-shot (it sets `mounted = true` and warns that late registration "may not work"). Runtime module management needs a dynamic mount/unmount story, not just a UI over the CLI.

### Finalise the forgot-password / reset-password flow

**Priority:** High ŌĆö blocks production for v1.23.0-based apps.

v1.23.0 shipped `ForgotPasswordPage.tsx` and `ResetPasswordPage.tsx` as custom shadcn forms that call `authClient.requestPasswordReset()` and `authClient.resetPassword()` respectively ŌĆö the method names from Better Auth 1.6. - Submit a new password, confirm it's actually rotated on the server (sign in with the new password).

If any step fails, fall back to Neon's standalone `<ForgotPasswordForm>` / `<ResetPasswordForm>` components under a scoped CSS loader (to keep the Tabs fix intact on every other route) until Neon's 1.6 upgrade lands and the SDK path works cleanly.

### Template update mechanism review

**Priority:** High ŌĆö every point fix to date has been a patch on top of the wrong abstraction.

`npm run template:update` uses a git three-way merge as its update mechanism. That choice has cost real time across the v1.22.0 ŌåÆ v1.23.1 arc: v1.22.2 leaked a withdrawn experimental change into a downstream app, v1.23.0's merge left stale imports behind, the docs submodule at various points copied its contents into apps that had no business tracking docs, and every template release has required a post-ship patch to unwind one or another merge surprise. The v1.23.1 post-merge force-sync is a patch on the symptom, not the cause.

The cause is that **git merge is the wrong tool for this job.** Merge assumes bidirectional collaboration between peers ŌĆö two branches that have legitimately diverged and want to reconcile. Template updates are unidirectional: the template is authoritative, the app is a consumer, and the app's job is to receive updates, not negotiate with them. Every scenario where git's three-way resolution "helpfully" keeps a local line is a scenario where the template lost.

This entry is a **review, not an implementation plan** ŌĆö the point is to step back and question every assumption before writing more code. Concrete deliverable: a design doc (not a PR) that answers the questions below with data from real update scenarios, then proposes an architecture.

**Questions the review should answer:**

- **What are the actual scenarios `template:update` has to handle?** Enumerate every real case ŌĆö fresh app off `create-app`, app that's merged every release, app that skipped several releases, app that pulled a withdrawn experimental release, app that forked a core file legitimately (e.g. custom admin dashboard), app with user-owned files that the template also ships (`components.json`, `schema.prisma`), app that has `.claude/` / `docs/` leaked in from a prior bad merge, app on a disconnected network with a local template path, app tracking multiple modules with their own update cadences. What does each need?
- **Which scenarios does today's git-merge-based flow actually handle correctly?** The evidence so far says the happy path works and almost everything else takes manual intervention. Let's write this down with examples rather than guess.
- **Is there a reason we're using git merge beyond "git is available"?** The update flow doesn't need most of what merge provides (conflict markers, interactive resolution, rebase compatibility). It needs: "replace template-owned files with the new versions, leave user-owned files alone, stop pretending these two categories are negotiable."
- **Could a simpler mechanism replace it?** Candidates to evaluate:
  - **Copy-then-sync** ŌĆö `template:update` fetches the template at the target version, walks a manifest of template-owned paths, copies each into the app's working tree. No merge. User-owned paths are never touched. File deletions in the template become deletions in the app. This is how GitHub's "Template repository" feature works in spirit. Simpler, harder to mess up, easy to preview accurately. Cost: loses git's conflict detection for apps that have forked a template file ŌĆö need an opt-in mechanism (the `.template-ignore` from the sibling roadmap entry).
  - **Versioned patches** ŌĆö the template publishes a patch file per release (`patches/v1.22.3-to-v1.23.0.patch`). `template:update` applies the patch sequentially. Reproducible, reviewable, auditable. Cost: patch files drift if the user's files diverge; needs a reject-file review step when a patch doesn't apply.
  - **Manifest-driven per-file sync** ŌĆö the template ships `.template-manifest.json` listing every file it owns and its current content hash. `template:update` compares app file hashes against the manifest and syncs the diff. User-owned files are absent from the manifest, so they're never touched. Clean separation of concerns. Cost: manifest has to be kept in sync with the template tree ŌĆö solvable with a pre-commit hook on the template side.
  - **Hybrid** ŌĆö force-sync for template-authoritative files (the majority), git merge for shared-ownership files like `schema.prisma`, skip for user-owned files. Essentially today's v1.23.1 behaviour formalised with explicit categorisation in a manifest rather than a hardcoded allow-list.
- **What testing harness does this need?** The update script currently has zero automated coverage. Every new scenario is discovered by a downstream user in production. At minimum we should have a test that spins up a simulated app at version N, runs `template:update` to N+1, and asserts specific files are in specific states. Running the full matrix (every supported starting version ├Ś every current version) on every release is how we catch the v1.22.2 class of bug before it ships.
- **How does preview mode accurately represent what will happen?** Today's `--preview` shows a git diff between branches ŌĆö but the actual merge behaviour diverges from a naive diff. Preview should run the full update in a throwaway worktree and show the true before/after for the app's tree, including post-merge force-syncs and path removals.
- **What's the rollback story?** If `template:update` goes wrong, there's no clean undo beyond `git reset --hard`, which the script can't offer because it'd also discard uncommitted user work. `template:revert-to v<N>` should be a first-class command ŌĆö snapshot the app's tree before the update, restore on request.
- **How does the app signal "I've genuinely customised this template file and I know what I'm doing"?** The `.template-ignore` roadmap entry is one answer. There might be others ŌĆö per-file YAML frontmatter markers, a fork registry, whatever.
- **What's the relationship to the Module Starter's update flow?** Modules already have an update path (`npx my-module update`). Does the template update mechanism share code / principles / primitives with it? If not, should it? A unified "update-something-from-upstream" primitive used by both would be worth the refactor cost.

**Scope for the review:**

- Write the design doc answering the above. Publish to `docs/src/content/docs/starter-template/update-mechanism-review.mdx` with a draft banner.
- Implement a test harness for `template:update` covering at least the five known-problematic scenarios from this session (v1.22.2 ŌåÆ v1.23.0 stale import, docs submodule leak, .claude leak, user-customised `schema.prisma`, fresh unrelated-histories merge).
- Propose an architecture change based on the design doc's conclusions. Plan the migration from today's git-merge flow to the chosen architecture.

**Out of scope for the review** (explicitly ŌĆö keep it bounded):

- Shipping the new architecture in the same release as the design doc. This is a review, not a rewrite. Ship the doc first, discuss, then plan the implementation.
- Refactoring `create-app` in the same pass ŌĆö different concerns, can come later.

**Expected outcome:** at minimum, an honest catalogue of what works and what doesn't. At best, a clear architectural direction that closes this class of bug permanently so we stop patching the symptom.

### Auto-merge schema field additions in `template:update`

**Priority:** High ŌĆö fills the last piece of the "no manual changes per release" promise.

v1.25.0 added the `appTheme` field to the `AppSettings` model. The update script already detects and auto-adds **new models** (via `Extract-ModelSection` in `update-template.{ps1,sh}`), but doesn't handle **new fields on existing models**. Downstream apps that pull v1.25.0 have to manually add `appTheme Json?` to their local `AppSettings` and run `prisma migrate dev` ŌĆö small, but the kind of friction this template's versioning philosophy keeps saying shouldn't exist.

**What this would do:**

- **Parse both schemas after merge.** For each model the template ships (User, AppSettings, plus anything future releases add), diff the field lists. Field present in template but not in local ŌåÆ "missing field."
- **Insert missing fields into the local schema** with their full Prisma definition (type, modifiers, default, attributes). Preserves any local fields the user has added to the same model.
- **Set `$needsMigration = true`** so the existing auto-migrate step (`prisma migrate dev --name template_update_vX_X_X`) fires and generates the migration for the downstream app's DB.
- **Handle the "field removed in template but user added it locally" case** ŌĆö leave the user's field alone, warn about the mismatch at the end of the update so the user can decide.

**Edge cases:**

- **Field renames** ŌĆö template changes `oldName String` to `newName String`. Diff algorithm sees "remove old, add new" ŌĆö not a rename. Ship a data-migration SQL snippet alongside the schema change for renames; document as a breaking change.
- **Type changes** ŌĆö template changes `field Int` to `field BigInt`. Diff sees same field name, different definition. Warn the user, don't auto-overwrite. Type changes force a major bump anyway per the [deprecation policy](/starter-template/versioning/).
- **Multi-file schema mode** ŌĆö the `Extract-ModelSection` approach works per-file; extend to iterate `backend/prisma/schema/*.prisma` in multi-file mode.
- **User has removed the model entirely** ŌĆö if downstream deleted `AppSettings` (unusual but possible), respect that; the existing new-model auto-add kicks in. Don't force a re-add.

**Why this belongs in the update script, not a codemod:**

Codemods (per the separate [Deprecation codemods](#deprecation-codemods-shipped-with-templateupdate) roadmap item) are for consumer-code changes ŌĆö rewriting imports and JSX shapes. Schema additions are infrastructure changes the downstream app needs regardless of what its own code looks like. Handling them in the update script keeps the flow atomic: one `template:update` run pulls the release and makes the downstream whole.

**Target: v1.25.1 or v1.26.0.** The `appTheme` field already requires a manual schema step for apps pulling v1.25.0 ŌĆö documented in the changelog. Shipping this fix in v1.25.1 means apps that update in the intervening window do the manual step once; apps that jump v1.24.X ŌåÆ v1.25.1 get it for free.

### Structure-only styling migration

**Priority:** High ŌĆö retires the ~170 legacy style-in-template violations the v1.24.7 audit catalogued.

v1.24.7 codified the rule that template components use layout + semantic tokens only (no typography scale, font weights, colour literals, decorative shadows). The rule protects downstream apps from fighting the template's design decisions ŌĆö but the template itself has accumulated violations over the pre-1.24.7 releases:

- **~158 font-size / font-weight occurrences** across 30 files (`text-sm`, `font-semibold`, etc.) in `settings-blocks/`, `page-blocks/`, feature pages, sidebar + header components, auth pages.
- **~12 colour-literal occurrences** in 5 files ŌĆö admin role badges (`bg-red-100 text-red-800` for admin, blue for user, purple for managed), plan status badges (green-100), security status pills (green/blue), personal-info verified badge (green), table badges (blue).
- **A handful of `shadow-*` / `tracking-*` / `leading-relaxed`** occurrences.

Migration plan (one or two entries per release until cleared):

- **Typography pass** ŌĆö replace `text-sm` / `font-medium` / `font-semibold` usage with structural-only classes. Badge components move to using the new tokens.
- **Shadow pass** ŌĆö remove `shadow-sm` / `shadow-md` from template code; expose via Style Registry entries contributed by a "card elevation" style module.
- **Audit script** ŌĆö ship `scripts/audit-template-styling.{ps1,sh}` that greps for the banned class patterns and reports violations. Run as part of CI so regressions get caught.

Each release in the migration plan ships a **non-breaking** change ŌĆö existing visual output is preserved through Style Registry entries registered on template bootstrap, consumers see no difference. Once the audit script reports zero violations, the rule becomes enforceable via CI.

### Deprecation codemods shipped with `template:update`

**Priority:** Medium ŌĆö reduces the manual-migration burden even further once deprecation shims retire.

The v1.24.6 [deprecation policy](/starter-template/versioning/) ensures downstream apps never break at head ŌĆö public exports go through a shim cycle before removal. That solves the "broke in the field" problem, but downstream apps still need to do a one-time find-and-replace when they want to move off a deprecated export (rename imports, swap JSX shapes, etc.). The template can do that work for them.

**What this would explore:**

- **Codemods per deprecation** ŌĆö when a release deprecates a public export, ship a jscodeshift (or ts-morph) codemod alongside the shim that rewrites consumer code to use the replacement. Codemod lives at `codemods/<release>/<export-name>.ts` in the template tree.
- **`template:update` prompts to run them** ŌĆö after pulling a release that deprecates something, the update script detects any codemods shipped by that release and prompts: "v1.24.4 deprecated `TabBar`. Run the migration codemod now? [y/N]". Running it does the rewrite across the downstream app's `src/` and stages the changes for review.
- **Skip-if-already-migrated** ŌĆö codemod is idempotent, so running it twice is safe. If the downstream app has already migrated (no TabBar imports found), codemod reports "nothing to do."
- **Catalogue page in docs** ŌĆö `/starter-template/codemods/` lists every codemod the template has ever shipped, what it does, and which release introduced it. Downstream apps catching up from multiple versions behind can see the full migration sequence.

**Scope note:**

- Codemods only for public-export shape changes (rename, prop signature change, file-path move). Not for semantic behaviour changes ŌĆö those still need human review.
- Downstream apps opt-in per codemod at `template:update` time. Never auto-run without confirmation.
- Module-contributed codemods are a follow-up ŌĆö modules that ship their own deprecations can contribute codemods via a module manifest field once the primitive is in place for the starter template itself.

**Why this is "planned" not "now":**

Useful, but the deprecation shim + warning flow already covers the core "don't break downstream" requirement. Codemods are a second-order UX improvement that makes migrating off deprecated exports less tedious. Land [per-change / per-hunk template updates](#per-change--per-hunk-template-updates) first ŌĆö that addresses the more common daily friction.

### Per-change / per-hunk template updates

**Priority:** Medium ŌĆö complements the [Template update mechanism review](#template-update-mechanism-review).

Today's `template:update` is whole-file in both directions: a conflict on any part of a file surfaces as "the whole file is in conflict," and the force-sync branch replaces the whole file with the template's current version even if only one line of that file actually changed between releases. Both behaviours are blunt.

A downstream user hit this after v1.24.4 pulled the new `AppSettingsPage.tsx`: they'd made no local changes to that file, but the three-way merge still surfaced the whole file as unresolved because git saw "both sides modified." The user-owned allow-list preserves their version; the force-sync replaces with the template's ŌĆö fine for template-owned files, but a maintenance overhead for anyone tracking local churn.

**What this would explore:**

- **Per-hunk apply** ŌĆö instead of letting git merge the whole file, walk the diff of template-changed files between the old merge-base and the new template HEAD, and apply each hunk individually with `git apply --3way` (or equivalent). A hunk that doesn't apply cleanly leaves that specific range of the file in conflict; unaffected ranges stay pristine. Means: a 2-line template tweak to a 400-line file doesn't surface as "the whole file is conflicted."
- **Skip-identical optimisation** ŌĆö if the app's version of a file is byte-identical to the template's pre-change version (no local edits since the last merge), just swap to the new version. No merge needed.
- **Whole-file opt-out** ŌĆö for files where per-hunk doesn't make sense (large JSON configs, serialised data, generated code), mark them in a manifest to fall back to today's whole-file behaviour.

**Why this is "planned" not "now":**

This is essentially "reinvent `git merge` with a narrower contract," and the per-hunk application logic is the hard bit ŌĆö needs robust handling of context-lines, binary files, whitespace-only changes, deleted-then-recreated files, and so on. The sibling [Template update mechanism review](#template-update-mechanism-review) roadmap item is the right place to weigh per-hunk apply against the other candidate architectures (copy-then-sync, versioned patches, manifest-driven per-file sync). Land the review's design doc first, then pick.

### `.template-ignore` for per-app merge customisation

**Priority:** Medium ŌĆö follow-up to v1.23.1's force-sync behaviour.

v1.23.1's `template:update` force-syncs every template-changed file to the template's current version, using a hardcoded user-owned allow-list. Works for every known case today, but there's no way for an app to say "I've customised `features/admin/pages/AdminDashboardPage.tsx` on purpose, please don't overwrite it." The escape hatch should be a file in the app root:

```
# .template-ignore ŌĆö extends the user-owned allow-list for this app
frontend/src/features/admin/pages/AdminDashboardPage.tsx
frontend/src/features/dashboard/
```

Each line is a path (file or directory prefix) that's added to the user-owned allow-list at merge time. `update-template.{ps1,sh}` reads this file if present and extends `USER_OWNED_PATHS` / user-owned directory checks accordingly.

**Scope:**

- Read `.template-ignore` from the app root before the post-merge force-sync step.
- Treat each non-comment, non-blank line as a user-owned path. Support glob syntax (`frontend/src/features/admin/pages/*.tsx`) for flexibility.
- Warn when an app adds paths that the template has actually changed since the last merge ŌĆö so the user knows they might be missing a fix.
- Document in the [Updating guide](/starter-template/updating/) with worked examples.

**Why not by default:** force-sync as the default is the right call ŌĆö most apps will never need to diverge from a template-owned file. Making `.template-ignore` the opt-in keeps the common case clean and the escape hatch explicit.

### Archive removed files during template update

**Priority:** Medium ŌĆö safety net for major refactors like v1.23.0.

When `npm run template:update` pulls a template release that deletes files (e.g. `frontend/src/features/auth/components/AccountSettings.tsx` in v1.23.0), the merge straight-deletes them locally. If the downstream app had customised the deleted file, the customisation is gone ŌĆö recoverable from git history, but only if the user knows to look there and hasn't already committed the deletion. There's no diffable record in the working tree.

This feature borrows the "non-destructive uninstall" pattern from the Module Starter's [uninstall flow](/module-starter/building-modules/) ŌĆö instead of hard-deleting, move removed files to an archive tree under `.template-archive/v<version>/<original-path>`. The working tree stays clean (the archive is gitignored by default), users have a two-week window to review what went and salvage customisations, and a follow-up `template:archive:clean` command drops the archive after review.

**Scope:**

- **Update `update-template.{ps1,sh}`** ŌĆö before running the git subtree merge, detect files the incoming release deletes (via `git diff --name-only --diff-filter=D <old>..<new>`). For each, copy the current working-tree version to `.template-archive/v<previous-version>/<original-path>` before the merge runs. The merge then deletes the original.
- **Gitignore entry** ŌĆö `.template-archive/` added to the root `.gitignore` shipped by the starter. Users who want to archive into a committed path can opt out.
- **Archive manifest** ŌĆö write `.template-archive/v<version>/manifest.json` listing what was moved, why (which template commit deleted it), and the suggested action (review / delete / merge into new location). LLM-friendly so `claude-code` / CI bots can read it.
- **`npm run template:archive:clean`** ŌĆö drops archives older than N versions (default: keep the last 2). Lets users maintain a small review window without unbounded archive growth.

**Out of scope:**

- Archiving *modified-then-removed* files differently from unmodified removals. Treat them all the same ŌĆö the archive IS the diff-from-what-the-template-had. Users can use `git diff` against the previous template tag to find their customisations.
- Restoring from archive automatically. That's a manual `cp` back into the working tree ŌĆö we don't want magic that re-creates deleted files unless the user explicitly asks.

**v1.23.0 retrospective:** `AccountSettings.tsx` was deleted and `auth.css` (which only ever existed in the withdrawn v1.22.2) was also removed. Neither was customised in any known downstream app, so no loss occurred this time ŌĆö but the risk exists for every future refactor. Archive-on-removal should land before the next large restructuring change (custom settings pages / layout registry / similar).

### Module system as a foundation feature (`ModuleContract` home)

## Known Security Issues

Findings from the v1.26.0 security & architecture audit (backend + frontend, audited against live code). Application-layer access control is sound for the current surface ŌĆö the `User` model is the only user-owned data and every access path is scoped to `req.user.id` ŌĆö but the items below must be closed before any deployment that stores sensitive or multi-tenant data, especially the legal/evidence-grade apps on this stack.

<Aside type="tip" title="Mostly addressed in v1.27.0">
  The error-leak, rate-limiting, TOCTOU-race, and CORS items below were fixed in **v1.27.0**, and the user-data isolation backstop now ships (`owned()` + opt-in RLS). Remaining: the frontend `better-auth` Critical (coupled to Neon's auth upgrade ŌĆö see Dependency vulnerabilities) and explicit CSP (see [Production security hardening](#production-security-hardening)). Per-item status is noted below.
</Aside>

### Database-level user-data isolation backstop

**Priority:** Critical (strict tier) ŌĆö the headline gap for legal/evidence apps.

**Status:** Ō£ģ Addressed in **v1.27.0** ŌĆö `@/lib/owned` ships `owned(userId, where)` for app-layer scoping and `withUserContext()` for the opt-in RLS backstop; the strict-tier RLS profile is documented. Adopt both for any user-owned model.

User-data isolation is enforced **only** at the application layer (JWT ŌåÆ `req.user.id` ŌåÆ `where: { userId }`). There is no Postgres Row-Level Security and no reusable ownership-check helper shipped by the template. The only user-owned model today is `User`, and every path is correctly scoped ŌĆö but the commented example relation (`prisma.post.findMany({ where: { userId } })` in `user.service.ts`) is the pattern downstream developers copy by hand, so the first user-owned model added is one forgotten `where` clause away from full cross-user leakage.

- Ship an ownership-scoped query helper (e.g. `owned(userId, where)`) so isolation is not hand-rolled per query.
- Document the **RLS floor**: `FORCE ROW LEVEL SECURITY` plus a per-request `set_config('app.tenant_id', ŌĆ”)` inside `withTenantContext` (the Prisma table-owner connection bypasses RLS otherwise). See [Database & Prisma ŌåÆ User-data isolation](/starter-template/database/#user-data-isolation-neon-rls).

### Error-message leakage in production

**Priority:** High. **Status:** Ō£ģ Fixed in **v1.27.0** ŌĆö non-operational errors now return a generic message in production.

`backend/src/middleware/error.middleware.ts` returns `err.message` to the client for **all** errors including unexpected 500s; only `stack` is gated behind `NODE_ENV === 'development'`. Unwrapped Prisma/internal errors leak schema and query detail to clients in production. Fix: when the error is not an operational `AppError`, return a generic message and log the real one server-side only.

### No global rate limiting

**Priority:** High. **Status:** Ō£ģ Fixed in **v1.27.0** ŌĆö global `express-rate-limit` on `/api` (env-tunable, health checks skipped).

`backend/src/app.ts` carries only a commented-out `express-rate-limit` TODO; the sole limiter anywhere is the 1-per-hour data-export check in `user.service.ts`. Auth, role-change, and delete endpoints have no throttle. The `TooManyRequestsError` + `Retry-After` plumbing already exists ŌĆö only the limiter middleware is missing. (New dependency goes through the `create-app` set, not a direct install.)

### Non-atomic role checks (TOCTOU races)

**Priority:** High. **Status:** Ō£ģ Fixed in **v1.27.0** ŌĆö both guards now run in Serializable transactions (`withTransaction({ isolationLevel: 'Serializable' })`).

Two role invariants are checked non-atomically (`count()` then mutate):

- **First-user-becomes-admin** (`user.service.ts` `getOrCreateProfile`) ŌĆö concurrent first logins can mint multiple admins; a pre-admin attacker who calls the API first gains admin.
- **Last-admin guard** (`admin.service.ts` `updateUserRole` / `deleteUser`) ŌĆö concurrent demotions/deletes can leave zero admins and lock the system.

Fix: wrap each in `withTransaction` with row locking, or enforce via a DB constraint / unique partial index.

### CORS allows no-origin requests with credentials

**Priority:** High. **Status:** Ō£ģ Fixed in **v1.27.0** ŌĆö no-origin requests are rejected in production. **Amended in v1.54.0:** routes may opt back in per-path via `allowNoOrigin: true` in the route registry ŌĆö reserved for endpoints authenticated by other means (webhook HMAC over the raw body) and the health probes; the default remains reject.

`backend/src/app.ts` returns `callback(null, true)` for requests with no `Origin` header while `credentials: true` is set (an acknowledged, unresolved `DECISION NEEDED` in the code). The API authenticates via `Authorization: Bearer`, so classic CSRF exposure is currently limited ŌĆö but this is the wrong default and becomes critical once any cookie/session auth is in play (`better-auth` uses session cookies). Reject no-origin requests in production.

### Dependency vulnerabilities

**Priority:** High. **Status:** ŌÜĀ’ĖÅ Partially fixed in **v1.27.0** ŌĆö backend now reports **0 high / 0 critical**; the frontend `better-auth` Critical remains (see below).

Original audit (`npm audit`, lockfile-only): frontend 1 critical + 15 high, backend 10 high. v1.27.0 bumped `axios` (`^1.18.0`), `react-router-dom` (`^7.18.0`), `ws` (`^8.21.0`) and regenerated all lockfiles ŌĆö the existing `^` ranges already permitted the patched versions, so the highs were stale-lockfile pins. Result: **backend 0 high / 0 critical** (3 moderate dev-tooling remain); frontend highs reduced.

**Still open:** the **`better-auth`** Critical (2FA bypass, OAuth `state`/PKCE) is transitive via `@neondatabase/neon-js` and pinned to Neon's Better Auth 1.4.6 server ŌĆö it tracks the same Neon upgrade as the [forgot-password flow](#finalise-the-forgot-password--reset-password-flow) and **cannot be unilaterally bumped without breaking auth**. The remaining frontend highs are that auth cluster (`fastify` / `kysely` / `fast-uri`) or dev-only build tooling (`vite` / `rollup`). Re-run `npm audit` after the Neon upgrade lands, and add a recurring audit gate (CI).
