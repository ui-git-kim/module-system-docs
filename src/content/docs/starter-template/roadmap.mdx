## Planned

Features planned for upcoming releases.

### Multiple Layouts

**Priority:** High

Currently the template has a single dashboard layout with sidebar navigation. This enhancement will add:

- **Builder/Editor Layout** - Full-bleed canvas for visual editors (already have `FullBleed` component)
- **Wizard Layout** - Step-based flows for onboarding or multi-step forms
- **Focus Layout** - Minimal chrome for focused tasks (reading, writing)
- **Landing Layout** - Marketing pages without app navigation

**Implementation approach:**
- Layout registry for registering layout components
- Route-level layout selection
- Nested layouts for flexibility

### Dashboard Widget Admin

**Priority:** High

Admin panel page for managing dashboard widgets:

- View all registered widgets
- Toggle widget visibility globally (hide from all users)
- Reorder widget default positions
- Set default visibility for new users

### Database Health Page

**Priority:** High

Admin panel page for monitoring database health:

- Connection status and pool metrics
- Table sizes and row counts
- Recent migration history
- Basic query performance stats
- Prisma connection info

### Permissions Module Integration

**Priority:** High

Prepare the template for granular permissions beyond basic roles:

- `managed` role support is already in place
- Hook points for permission checks in navigation
- Permission-aware UI components
- Backend middleware integration points

### Admin Module Management

**Priority:** High

Replace CLI-based module operations with an admin panel UI. Currently modules are installed, updated, and removed via terminal commands (`npx my-module init`, `update`, `uninstall`). This feature brings that workflow into the app itself:

- **Browse & install** — Discover available modules (from a registry or local packages) and install them from the admin panel
- **Configure** — Module-specific settings and prompt answers managed through a UI instead of interactive CLI prompts
- **Deactivate / activate** — Toggle modules on or off without removing files (replaces `npx my-module disable/enable`)
- **Uninstall** — Remove modules cleanly with confirmation, data export options, and rollback (replaces `npx my-module uninstall`)
- **Update** — Check for module updates and apply them with a diff preview (replaces `npx my-module update`)
- **Module status dashboard** — View installed modules, versions, health, and dependencies at a glance

**Implementation considerations:**
- Backend service that wraps scaffold operations (init, update, uninstall, enable, disable)
- Admin page registered via `adminPagesRegistry`
- Module manifest (`.module-manifests.json`) read/write from the backend
- Restart or hot-reload mechanism after module changes take effect
- Permission-gated to admin role

### Error Pages & Offline Handling

**Priority:** Medium

Proper error boundaries and user-facing error pages:

- Network failure page (no internet connection — currently auth silently fails)
- Auth error page (JWKS fetch failure, token expired, invalid session)
- 404 page (replace current plain `<div>404</div>` with styled page)
- 500 / unexpected error boundary with retry action
- React Router `errorElement` integration on protected routes
- Offline detection banner or toast notification

### Deployment Guide

**Priority:** Medium

Documentation page covering production deployment:

- Pre-deployment checklist (env vars, migrations, build verification)
- Deployment steps (backend first, then frontend)
- Post-deployment verification (health check, auth flow, critical paths)
- Security hardening checklist (rate limiting, CORS, monitoring)
- Error monitoring integration patterns (Sentry, structured logging)

### Internationalisation (i18n)

**Priority:** Medium

Add foundation for multi-language support:

- Language options registry (already exists)
- Translation file structure
- React context for current locale
- Backend response localisation hooks

### Notification System

**Priority:** Medium

Built-in notification infrastructure:

- Toast notifications (already using sonner)
- Persistent notification centre
- Notification preferences (registry exists)
- Backend notification service hooks

### Command Palette

**Priority:** Low

Global keyboard-driven navigation:

- `Cmd+K` / `Ctrl+K` trigger
- Search across navigation items
- Recent items and actions
- Extensible via registry

### App Documentation

**Priority:** Low

Optional documentation setup for apps created from the template:

- Create script prompts: "Would you like to set up documentation?"
- Scaffolds Astro/Starlight docs folder (same setup as module system docs)
- App-specific docs, not linked to module system shared docs
- Pre-configured with app name and basic structure
- Separate from the shared module system documentation

**Current state:** Apps inherit the `docs/` submodule from the template, which points to the shared module system docs. Apps should either have no docs folder or their own dedicated docs.

## Known Performance Issues

Findings from the v1.20.4 performance audit. Grouped by where the cost actually lands.

<Aside type="danger" title="Blocker before production">
  **Group C — Backend list perf** is a pre-production blocker. The `getAllUsers()` endpoint returns every user row in the database with no `take`/`skip` pagination, and `User` has no index on `createdAt` (used for ORDER BY in the admin users list) or `role` (used for the "last admin" check). This is fine at dev scale but will degrade quickly past a few thousand rows and must be fixed before shipping to real traffic.
</Aside>

**Status as of v1.20.7:**
- ✅ Group A — Context + callback hygiene (v1.20.5)
- ✅ Group B — Bundle + debounce (v1.20.6)
- 🚧 **Group C — Backend list perf (pre-production blocker)**
- ✅ Group D — Correctness cleanup (v1.20.7)

### Context & callback hygiene

**Priority:** High — quick wins, zero behavioural risk

- **`AppSettingsContext.value` is not memoised** ([`frontend/src/features/admin/context/AppSettingsContext.tsx:159`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/context/AppSettingsContext.tsx)) — the context wraps the entire app shell; every render of `AppSettingsProvider` creates a new object identity, forcing every consumer to re-render. Fix: wrap in `useMemo`.
- **`useNavigationOverrides.save` and `useToolbarOverrides.save` depend on `[overrides]`** — the callbacks are recreated on every toggle/swap. Pages memoise their toolbar `items` against these callbacks via `useAdminToolbarActions`, which triggers a registry unregister→register cycle on every interaction. Fix: pass the overrides to save as an argument, not a closure dep.
- **`NavItem` is not `React.memo`-wrapped** ([`frontend/src/features/dashboard/components/sidebar/NavMain.tsx`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/dashboard/components/sidebar/NavMain.tsx)) — sidebar re-renders all items on every parent render. Low impact today (<50 items), worth fixing for module authors who might add many items.

### User-visible responsiveness

**Priority:** High

- **`AdminDataTable` search has no debounce** ([`frontend/src/features/admin/components/AdminDataTable.tsx:143`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/components/AdminDataTable.tsx)) — every keystroke triggers a full TanStack table re-filter. Jank at 100+ rows. Fix: 200ms debounce.
- **Admin pages are in the main bundle** ([`frontend/src/features/admin/admin-pages.registration.ts`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/admin-pages.registration.ts)) — the route already wraps in `<Suspense>`, so switching to `React.lazy()` is a clean win for the 98% of users who aren't admins.

### Backend list scalability

**Priority:** Medium — only matters past a few thousand rows, but cheap to fix now

- **`admin.service.ts -> getAllUsers()` has no pagination** ([`backend/src/features/admin/admin.service.ts:321`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/src/features/admin/admin.service.ts)) — returns every user row on every admin-users-page visit. Fix: add `page`/`limit` params and return `{ users, total }`; update `adminService` and `useAdminUsers` in lockstep (breaking API change → minor version bump, downstream modules that call this need updating).
- **`User` model missing indexes on `createdAt` and `role`** ([`backend/prisma/schema.prisma`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/prisma/schema.prisma)) — `getAllUsers` orders by `createdAt` and `updateUserRole` counts admins by `role`. Add `@@index([createdAt])` and `@@index([role])`. Ships as a migration downstream apps need to run.

### Duplicate fetches & cleanup

**Priority:** Low — correctness more than performance, but should be fixed

- **`AppSettingsPage` fetches app settings twice** — once via `AppSettingsContext` at the app root, again via the page-local `useAppSettings` hook. Fix: page reads from the global context; the hook becomes a form/save helper only.
- **Fetch effects without `AbortController`** in `useAppSettings`, `useAdminUsers`, `useNavigationOverrides`, `useToolbarOverrides` — stale `setState` after unmount. Fix: add an `isMounted` flag or pass an `AbortSignal` through `adminService`.
- **`matchMedia` listener in `main.tsx` is never removed** — module-level, runs once in production; HMR in dev can accumulate listeners. Low impact but worth fixing.
