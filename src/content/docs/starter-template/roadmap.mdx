## Planned

Features planned for upcoming releases.

### Multiple Layouts

**Priority:** High

Currently the template has a single dashboard layout with sidebar navigation. This enhancement will add:

- **Builder/Editor Layout** - Full-bleed canvas for visual editors (already have `FullBleed` component)
- **Wizard Layout** - Step-based flows for onboarding or multi-step forms
- **Focus Layout** - Minimal chrome for focused tasks (reading, writing)
- **Landing Layout** - Marketing pages without app navigation

**Implementation approach:**
- Layout registry for registering layout components
- Route-level layout selection
- Nested layouts for flexibility

**Related:** the `styleRegistry` (implemented alongside — see changelog for the landing version) is the sibling feature for module-contributed styling; the two should compose (a module may want to ship both a layout and a matching style profile).

### In-App Component Reinstall for Style Registry

**Priority:** Medium

The `styleRegistry` landed with a terminal-only reinstall flow — when a module's `componentsJson` contributions change (new `style`, `baseColor`, `cssVariables`, added registries, etc.), admins need to run `npm run components:sync` in a terminal to apply the changes. This enhancement brings that flow into the admin UI.

**Scope:**

- Backend endpoint that spawns `scripts/sync-components.{ps1,sh}` as a child process.
- Streams stdout/stderr back to the admin UI over Server-Sent Events or WebSocket.
- Admin page surfaces: "Reinstall N components to apply changes? [Reinstall] [Keep staged]".
- "Keep staged" lets an admin batch multiple `styleRegistry` changes before a single reinstall.
- Progress feedback (which component is currently being reinstalled, any errors).

**Depends on:** [Admin Module Management](#admin-module-management) landing first. That feature will establish the node-spawn-and-stream pattern for module install/update/uninstall, which this reinstall flow will reuse rather than reinvent.

**Constraint:** some deploy environments (locked-down hosting, containers without a writable filesystem) will block shelling out to `npx`. This feature is therefore opt-in — apps that can't support it can continue using the terminal sync.

### Admin Module Management

**Priority:** High

Replace CLI-based module operations with an admin panel UI. Currently modules are installed, updated, and removed via terminal commands (`npx my-module init`, `update`, `uninstall`). This feature brings that workflow into the app itself:

- **Browse & install** — Discover available modules (from a registry or local packages) and install them from the admin panel
- **Configure** — Module-specific settings and prompt answers managed through a UI instead of interactive CLI prompts
- **Deactivate / activate** — Toggle modules on or off without removing files (replaces `npx my-module disable/enable`)
- **Uninstall** — Remove modules cleanly with confirmation, data export options, and rollback (replaces `npx my-module uninstall`)
- **Update** — Check for module updates and apply them with a diff preview (replaces `npx my-module update`)
- **Module status dashboard** — View installed modules, versions, health, and dependencies at a glance

**Implementation considerations:**
- Backend service that wraps scaffold operations (init, update, uninstall, enable, disable)
- Admin page registered via `adminPagesRegistry`
- Module manifest (`.module-manifests.json`) read/write from the backend
- Restart or hot-reload mechanism after module changes take effect
- Permission-gated to admin role

### Email OTP / magic link sign-in

**Priority:** Medium — builds on v1.23.0's custom auth pages.

v1.23.0 shipped with email+password and Google OAuth. Adding OTP brings the starter back up to feature parity with the original Neon `AuthView` without reintroducing the Neon UI package or its Tailwind preset. Scope adds one more page (OTP code verification step) and a sign-in-with-OTP toggle on `SignInPage.tsx`. The underlying client already supports OTP via `authClient.emailOtp.*` methods.

### Finalise the forgot-password / reset-password flow

**Priority:** High — blocks production for v1.23.0-based apps.

v1.23.0 shipped `ForgotPasswordPage.tsx` and `ResetPasswordPage.tsx` as custom shadcn forms that call `authClient.requestPasswordReset()` and `authClient.resetPassword()` respectively — the method names from Better Auth 1.6. Neon's server is on Better Auth 1.4.6 until their upgrade lands ("coming weeks" per the Neon team on Discord, 2026-04-15) and their docs explicitly flag SDK methods for password reset as "not fully supported yet." The forms are wired but unverified end-to-end. Before any v1.23.0-based app ships to production:

- Trigger a reset from the live app, confirm the email arrives.
- Click the link, confirm it lands on `/auth/reset-password` with a valid `?token=...`.
- Submit a new password, confirm it's actually rotated on the server (sign in with the new password).

If any step fails, fall back to Neon's standalone `<ForgotPasswordForm>` / `<ResetPasswordForm>` components under a scoped CSS loader (to keep the Tabs fix intact on every other route) until Neon's 1.6 upgrade lands and the SDK path works cleanly.

### Template update mechanism review

**Priority:** High — every point fix to date has been a patch on top of the wrong abstraction.

`npm run template:update` uses a git three-way merge as its update mechanism. That choice has cost real time across the v1.22.0 → v1.23.1 arc: v1.22.2 leaked a withdrawn experimental change into a downstream app, v1.23.0's merge left stale imports behind, the docs submodule at various points copied its contents into apps that had no business tracking docs, and every template release has required a post-ship patch to unwind one or another merge surprise. The v1.23.1 post-merge force-sync is a patch on the symptom, not the cause.

The cause is that **git merge is the wrong tool for this job.** Merge assumes bidirectional collaboration between peers — two branches that have legitimately diverged and want to reconcile. Template updates are unidirectional: the template is authoritative, the app is a consumer, and the app's job is to receive updates, not negotiate with them. Every scenario where git's three-way resolution "helpfully" keeps a local line is a scenario where the template lost.

This entry is a **review, not an implementation plan** — the point is to step back and question every assumption before writing more code. Concrete deliverable: a design doc (not a PR) that answers the questions below with data from real update scenarios, then proposes an architecture.

**Questions the review should answer:**

- **What are the actual scenarios `template:update` has to handle?** Enumerate every real case — fresh app off `create-app`, app that's merged every release, app that skipped several releases, app that pulled a withdrawn experimental release, app that forked a core file legitimately (e.g. custom admin dashboard), app with user-owned files that the template also ships (`components.json`, `schema.prisma`), app that has `.claude/` / `docs/` leaked in from a prior bad merge, app on a disconnected network with a local template path, app tracking multiple modules with their own update cadences. What does each need?
- **Which scenarios does today's git-merge-based flow actually handle correctly?** The evidence so far says the happy path works and almost everything else takes manual intervention. Let's write this down with examples rather than guess.
- **Is there a reason we're using git merge beyond "git is available"?** The update flow doesn't need most of what merge provides (conflict markers, interactive resolution, rebase compatibility). It needs: "replace template-owned files with the new versions, leave user-owned files alone, stop pretending these two categories are negotiable."
- **Could a simpler mechanism replace it?** Candidates to evaluate:
  - **Copy-then-sync** — `template:update` fetches the template at the target version, walks a manifest of template-owned paths, copies each into the app's working tree. No merge. User-owned paths are never touched. File deletions in the template become deletions in the app. This is how GitHub's "Template repository" feature works in spirit. Simpler, harder to mess up, easy to preview accurately. Cost: loses git's conflict detection for apps that have forked a template file — need an opt-in mechanism (the `.template-ignore` from the sibling roadmap entry).
  - **Versioned patches** — the template publishes a patch file per release (`patches/v1.22.3-to-v1.23.0.patch`). `template:update` applies the patch sequentially. Reproducible, reviewable, auditable. Cost: patch files drift if the user's files diverge; needs a reject-file review step when a patch doesn't apply.
  - **Manifest-driven per-file sync** — the template ships `.template-manifest.json` listing every file it owns and its current content hash. `template:update` compares app file hashes against the manifest and syncs the diff. User-owned files are absent from the manifest, so they're never touched. Clean separation of concerns. Cost: manifest has to be kept in sync with the template tree — solvable with a pre-commit hook on the template side.
  - **Hybrid** — force-sync for template-authoritative files (the majority), git merge for shared-ownership files like `schema.prisma`, skip for user-owned files. Essentially today's v1.23.1 behaviour formalised with explicit categorisation in a manifest rather than a hardcoded allow-list.
- **What testing harness does this need?** The update script currently has zero automated coverage. Every new scenario is discovered by a downstream user in production. At minimum we should have a test that spins up a simulated app at version N, runs `template:update` to N+1, and asserts specific files are in specific states. Running the full matrix (every supported starting version × every current version) on every release is how we catch the v1.22.2 class of bug before it ships.
- **How does preview mode accurately represent what will happen?** Today's `--preview` shows a git diff between branches — but the actual merge behaviour diverges from a naive diff. Preview should run the full update in a throwaway worktree and show the true before/after for the app's tree, including post-merge force-syncs and path removals.
- **What's the rollback story?** If `template:update` goes wrong, there's no clean undo beyond `git reset --hard`, which the script can't offer because it'd also discard uncommitted user work. `template:revert-to v<N>` should be a first-class command — snapshot the app's tree before the update, restore on request.
- **How does the app signal "I've genuinely customised this template file and I know what I'm doing"?** The `.template-ignore` roadmap entry is one answer. There might be others — per-file YAML frontmatter markers, a fork registry, whatever.
- **What's the relationship to the Module Starter's update flow?** Modules already have an update path (`npx my-module update`). Does the template update mechanism share code / principles / primitives with it? If not, should it? A unified "update-something-from-upstream" primitive used by both would be worth the refactor cost.

**Scope for the review:**

- Write the design doc answering the above. Publish to `docs/src/content/docs/starter-template/update-mechanism-review.mdx` with a draft banner.
- Implement a test harness for `template:update` covering at least the five known-problematic scenarios from this session (v1.22.2 → v1.23.0 stale import, docs submodule leak, .claude leak, user-customised `schema.prisma`, fresh unrelated-histories merge).
- Propose an architecture change based on the design doc's conclusions. Plan the migration from today's git-merge flow to the chosen architecture.

**Out of scope for the review** (explicitly — keep it bounded):

- Shipping the new architecture in the same release as the design doc. This is a review, not a rewrite. Ship the doc first, discuss, then plan the implementation.
- Refactoring `create-app` in the same pass — different concerns, can come later.

**Expected outcome:** at minimum, an honest catalogue of what works and what doesn't. At best, a clear architectural direction that closes this class of bug permanently so we stop patching the symptom.

### `.template-ignore` for per-app merge customisation

**Priority:** Medium — follow-up to v1.23.1's force-sync behaviour.

v1.23.1's `template:update` force-syncs every template-changed file to the template's current version, using a hardcoded user-owned allow-list. Works for every known case today, but there's no way for an app to say "I've customised `features/admin/pages/AdminDashboardPage.tsx` on purpose, please don't overwrite it." The escape hatch should be a file in the app root:

```
# .template-ignore — extends the user-owned allow-list for this app
frontend/src/features/admin/pages/AdminDashboardPage.tsx
frontend/src/features/dashboard/
```

Each line is a path (file or directory prefix) that's added to the user-owned allow-list at merge time. `update-template.{ps1,sh}` reads this file if present and extends `USER_OWNED_PATHS` / user-owned directory checks accordingly.

**Scope:**

- Read `.template-ignore` from the app root before the post-merge force-sync step.
- Treat each non-comment, non-blank line as a user-owned path. Support glob syntax (`frontend/src/features/admin/pages/*.tsx`) for flexibility.
- Warn when an app adds paths that the template has actually changed since the last merge — so the user knows they might be missing a fix.
- Document in the [Updating guide](/starter-template/updating/) with worked examples.

**Why not by default:** force-sync as the default is the right call — most apps will never need to diverge from a template-owned file. Making `.template-ignore` the opt-in keeps the common case clean and the escape hatch explicit.

### Archive removed files during template update

**Priority:** Medium — safety net for major refactors like v1.23.0.

When `npm run template:update` pulls a template release that deletes files (e.g. `frontend/src/features/auth/components/AccountSettings.tsx` in v1.23.0), the merge straight-deletes them locally. If the downstream app had customised the deleted file, the customisation is gone — recoverable from git history, but only if the user knows to look there and hasn't already committed the deletion. There's no diffable record in the working tree.

This feature borrows the "non-destructive uninstall" pattern from the Module Starter's [uninstall flow](/module-starter/creating-modules/) — instead of hard-deleting, move removed files to an archive tree under `.template-archive/v<version>/<original-path>`. The working tree stays clean (the archive is gitignored by default), users have a two-week window to review what went and salvage customisations, and a follow-up `template:archive:clean` command drops the archive after review.

**Scope:**

- **Update `update-template.{ps1,sh}`** — before running the git subtree merge, detect files the incoming release deletes (via `git diff --name-only --diff-filter=D <old>..<new>`). For each, copy the current working-tree version to `.template-archive/v<previous-version>/<original-path>` before the merge runs. The merge then deletes the original.
- **Gitignore entry** — `.template-archive/` added to the root `.gitignore` shipped by the starter. Users who want to archive into a committed path can opt out.
- **Archive manifest** — write `.template-archive/v<version>/manifest.json` listing what was moved, why (which template commit deleted it), and the suggested action (review / delete / merge into new location). LLM-friendly so `claude-code` / CI bots can read it.
- **`npm run template:archive:clean`** — drops archives older than N versions (default: keep the last 2). Lets users maintain a small review window without unbounded archive growth.

**Out of scope:**

- Archiving *modified-then-removed* files differently from unmodified removals. Treat them all the same — the archive IS the diff-from-what-the-template-had. Users can use `git diff` against the previous template tag to find their customisations.
- Restoring from archive automatically. That's a manual `cp` back into the working tree — we don't want magic that re-creates deleted files unless the user explicitly asks.

**v1.23.0 retrospective:** `AccountSettings.tsx` was deleted and `auth.css` (which only ever existed in the withdrawn v1.22.2) was also removed. Neither was customised in any known downstream app, so no loss occurred this time — but the risk exists for every future refactor. Archive-on-removal should land before the next large restructuring change (custom settings pages / layout registry / similar).

### styleOverrides Bootstrap Cache Fix

**Priority:** Medium

Module-registered style entries apply synchronously at module-load time; `AppSettings.styleOverrides` is fetched asynchronously during React mount. If an admin disabled an entry whose tokens visibly change the shell, users see a brief flash from the "on" state to the "off" state during the fetch.

**Fix:**

- Cache the last-known `styleOverrides` map in `localStorage` from `AppSettingsContext` on every update.
- In `main.tsx`, read the cache before React mounts and call `styleRegistry.applyOverrides(cached)` synchronously — same pattern as the existing theme-class init that prevents the dark/light mode flash.
- On server response, reconcile and re-apply if the server-side map differs from the cache.

**Scope:** ~15 min. Fully backwards-compatible — users without a cached value still get the current behaviour.

Related: [Style System § Known limitation](/starter-template/style-system/#known-limitation--first-paint-flash), [Changelog v1.21.0](/starter-template/changelog/#v1210).

### Frontend Env Var Build-Time Validation

**Priority:** Low

The backend uses Zod to validate env vars at startup; the frontend currently doesn't validate `VITE_*` vars at all. A missing or malformed `VITE_NEON_AUTH_URL` surfaces as a runtime auth failure, not a build error.

**Fix:** add a Vite plugin (or a `tsx` preflight) that parses `import.meta.env` against a Zod schema and fails the build if required vars are missing. Mirrors the backend's `config/env.ts` pattern.

## Known Performance Issues

Findings from the v1.20.4 performance audit. Grouped by where the cost actually lands.

### Context & callback hygiene

**Priority:** High — quick wins, zero behavioural risk

- **`AppSettingsContext.value` is not memoised** ([`frontend/src/features/admin/context/AppSettingsContext.tsx:159`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/context/AppSettingsContext.tsx)) — the context wraps the entire app shell; every render of `AppSettingsProvider` creates a new object identity, forcing every consumer to re-render. Fix: wrap in `useMemo`.
- **`useNavigationOverrides.save` and `useToolbarOverrides.save` depend on `[overrides]`** — the callbacks are recreated on every toggle/swap. Pages memoise their toolbar `items` against these callbacks via `useAdminToolbarActions`, which triggers a registry unregister→register cycle on every interaction. Fix: pass the overrides to save as an argument, not a closure dep.
- **`NavItem` is not `React.memo`-wrapped** ([`frontend/src/features/dashboard/components/sidebar/NavMain.tsx`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/dashboard/components/sidebar/NavMain.tsx)) — sidebar re-renders all items on every parent render. Low impact today (fewer than 50 items), worth fixing for module authors who might add many items.

### User-visible responsiveness

**Priority:** High

- **`AdminDataTable` search has no debounce** ([`frontend/src/features/admin/components/AdminDataTable.tsx:143`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/components/AdminDataTable.tsx)) — every keystroke triggers a full TanStack table re-filter. Jank at 100+ rows. Fix: 200ms debounce.
- **Admin pages are in the main bundle** ([`frontend/src/features/admin/admin-pages.registration.ts`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/frontend/src/features/admin/admin-pages.registration.ts)) — the route already wraps in `<Suspense>`, so switching to `React.lazy()` is a clean win for the 98% of users who aren't admins.

### Backend list scalability

**Priority:** Medium — only matters past a few thousand rows, but cheap to fix now

- **`admin.service.ts -> getAllUsers()` has no pagination** ([`backend/src/features/admin/admin.service.ts:321`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/src/features/admin/admin.service.ts)) — returns every user row on every admin-users-page visit. Fix: add `page`/`limit` params and return `{ users, total }`; update `adminService` and `useAdminUsers` in lockstep (breaking API change → minor version bump, downstream modules that call this need updating).
- **`User` model missing indexes on `createdAt` and `role`** ([`backend/prisma/schema.prisma`](https://github.com/ui-git-kim/unique-react-fullstack-starter/blob/main/backend/prisma/schema.prisma)) — `getAllUsers` orders by `createdAt` and `updateUserRole` counts admins by `role`. Add `@@index([createdAt])` and `@@index([role])`. Ships as a migration downstream apps need to run.

### Duplicate fetches & cleanup

**Priority:** Low — correctness more than performance, but should be fixed

<Aside type="tip" title="Systemic fix landed in v1.22.0">
Adopting TanStack Query as the [standard data fetching layer](/starter-template/data-fetching/) supersedes most of the individual fixes in this section. Providers that migrate to `useQuery` get deduplication (no more double-fetch), built-in cancellation (no stale `setState` on unmount), and a shared cache as a matter of course. Migrate remaining hooks (`useAdminUsers`, `useNavigationOverrides`, `useToolbarOverrides`) as they're next touched — the reference pattern is `AppSettingsContext`.
</Aside>

- ✅ **`AppSettingsPage` fetches app settings twice** — page now reads from the global context (already landed); the context itself is backed by a shared `useQuery` since v1.22.0, so any other consumer that later mounts `['app-settings']` hits the cache.
- **Fetch effects without `AbortController`** in remaining hooks (`useAdminUsers`, `useNavigationOverrides`, `useToolbarOverrides`) — stale `setState` after unmount. Preferred fix: migrate each to `useQuery` / `useMutation`, which handles cancellation and re-use for free.
- **`matchMedia` listener in `main.tsx` is never removed** — module-level, runs once in production; HMR in dev can accumulate listeners. Low impact but worth fixing.
