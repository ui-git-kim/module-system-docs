The starter template uses **Prisma 7** with **Neon PostgreSQL**. This page covers the Prisma 7 architecture, runtime client setup, and **critical guidelines for modules adding database tables**.

<Aside type="tip">
  For initial Neon setup (creating a project, getting connection strings, configuring environment variables), see [Neon Database Setup](/getting-started/neon-setup/).
</Aside>

## Overview

Prisma 7 separates configuration into multiple files:

```
backend/
├── prisma.config.ts          # CLI configuration (migrations)
├── prisma/
│   ├── schema.prisma         # Model definitions only (no URL!)
│   └── migrations/           # Migration history
└── src/
    ├── lib/prisma.ts         # Runtime client with adapters
    └── generated/prisma/     # Generated Prisma client
```

**Key points:**
- `schema.prisma` - Defines models only, no connection URL
- `prisma.config.ts` - Configures CLI operations (uses direct/non-pooled URL for migrations)
- `lib/prisma.ts` - Creates runtime client with appropriate adapter (uses pooled URL)
- Neon uses pooled connections for runtime, direct for migrations

## Prisma 7 Configuration

### Schema File

In Prisma 7, the schema only defines models—no connection URL:

```prisma
// backend/prisma/schema.prisma
datasource db {
  provider = "postgresql"
  // No URL here - configured in prisma.config.ts
}

generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma"
}

model User {
  id    String @id
  email String @unique
  // ...
}
```

### CLI Configuration

The `prisma.config.ts` file configures CLI operations:

```typescript
// backend/prisma.config.ts
import 'dotenv/config';
import { defineConfig } from 'prisma/config';

export default defineConfig({
  schema: 'prisma/schema.prisma',
  migrations: {
    path: 'prisma/migrations',
  },
  datasource: {
    // Direct URL for migrations (non-pooled)
    url: process.env.DIRECT_DATABASE_URL || process.env.DATABASE_URL,
  },
});
```

## Neon Connection Architecture

Neon requires different connection strategies for different operations:

```
┌─────────────────────────────────────────────────────────────┐
│                       Neon Database                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────┐         ┌─────────────────┐          │
│   │  Pooled URL     │         │  Direct URL     │          │
│   │  (DATABASE_URL) │         │ (DIRECT_URL)    │          │
│   │                 │         │                 │          │
│   │  -pooler in     │         │  No -pooler     │          │
│   │  hostname       │         │                 │          │
│   └────────┬────────┘         └────────┬────────┘          │
│            │                           │                    │
│            ▼                           ▼                    │
│   ┌─────────────────┐         ┌─────────────────┐          │
│   │  Runtime        │         │  Migrations     │          │
│   │  (lib/prisma.ts)│         │  (CLI)          │          │
│   │                 │         │                 │          │
│   │  WebSocket via  │         │  TCP via        │          │
│   │  PrismaNeon     │         │  prisma.config  │          │
│   └─────────────────┘         └─────────────────┘          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

<Aside type="tip">
  The pooled URL (with `-pooler` in hostname) is used for runtime queries. The direct URL is used for migrations. See [Neon Database Setup](/getting-started/neon-setup/) for full environment variable configuration.
</Aside>

## Runtime Client Setup

The `lib/prisma.ts` file creates the Prisma client with the appropriate adapter:

```typescript
// backend/src/lib/prisma.ts
import { PrismaClient, Prisma } from '../generated/prisma/client.js';
import { PrismaNeon } from '@prisma/adapter-neon';
import { PrismaPg } from '@prisma/adapter-pg';
import { neonConfig } from '@neondatabase/serverless';
import ws from 'ws';
import { env } from '../config/env.js';

// Cache the client across hot reloads (see singleton note below)
declare global {
  var prisma: PrismaClient | undefined;
}

// Select the adapter based on the DATABASE_ADAPTER env var
function createAdapter() {
  if (env.DATABASE_ADAPTER === 'neon') {
    // Configure WebSockets for Node.js (not needed in edge runtimes)
    neonConfig.webSocketConstructor = ws;
    return new PrismaNeon({ connectionString: env.DATABASE_URL });
  }
  return new PrismaPg({ connectionString: env.DATABASE_URL });
}

// Log level derived from NODE_ENV / DEBUG_PRISMA
function getLogConfig(): Prisma.LogLevel[] {
  if (env.NODE_ENV === 'production') return ['error'];
  return env.DEBUG_PRISMA ? ['query', 'error', 'warn'] : ['error', 'warn'];
}

// Reuse the cached client in non-production to survive hot reloads
export const prisma =
  globalThis.prisma ??
  new PrismaClient({ adapter: createAdapter(), log: getLogConfig() });

if (env.NODE_ENV !== 'production') {
  globalThis.prisma = prisma;
}
```

The client is a **global singleton**: `globalThis.prisma ?? new PrismaClient(...)` reuses a cached instance in non-production so repeated dev hot reloads don't each open a new client and exhaust the connection pool. It is constructed with a `log` config derived from `DEBUG_PRISMA` / `NODE_ENV`, and adapter selection lives in a `createAdapter()` helper reading `env.DATABASE_ADAPTER`.

### Using Prisma in Features

```typescript
// Import from lib/prisma.ts, not from @prisma/client
import { prisma } from '../lib/prisma.js';

export async function getUser(id: string) {
  return prisma.user.findUnique({ where: { id } });
}
```

<Aside type="danger">
  **Do not create separate Prisma clients.** Always import from `lib/prisma.ts` to reuse the global singleton — creating extra clients (especially across hot reloads) exhausts the connection pool.
</Aside>

## Database Commands

All commands run from the `backend/` directory:

```bash
# Generate Prisma client after schema changes
npx prisma generate

# Create and apply migrations (development)
npx prisma migrate dev --name your_migration_name

# Apply pending migrations (production)
npx prisma migrate deploy

# Reset database (drops all data!)
npx prisma migrate reset

# Open Prisma Studio (GUI)
npx prisma studio
```

<Aside type="caution">
  **Never use `prisma db push` in this template.** Always use `prisma migrate dev` to create proper migration files that can be tracked and deployed.
</Aside>

## Schema Structure

### Core Models

The template includes three core models:

```prisma
// User - integrates with Neon Auth
model User {
  id          String   @id           // From Neon Auth JWT
  email       String   @unique
  role        String   @default("user")
  name        String?
  avatarUrl   String?
  bio         String?
  preferences Json     @default("{}")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

// AppSettings - singleton for global config
model AppSettings {
  id        String   @id @default("default")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // Branding
  appName         String  @default("My App")
  logoUrl         String?
  logoDisplayMode String  @default("logo-and-name") // "logo-only" | "logo-and-name" | "name-only"

  // Navigation & features
  showDashboardInNav   Boolean @default(true)
  enableMarketingPages Boolean @default(false)

  // JSON override columns (public schema — modules read/write these)
  navigationOverrides Json @default("{}")
  toolbarOverrides    Json @default("{}")
  styleOverrides      Json @default("{}")
  dashboardOverrides  Json @default("{}") // added v1.27.0
  appTheme            Json?               // added v1.25.0
}

// Space - the user's account boundary ("space"); auto-created 1:1 with the user
model Space {
  id                  String    @id      // == User.id == tenantId in solo mode
  deletionState       String    @default("active") // "active" | "pending_deletion" | "deleted"
  deletionRequestedAt DateTime?
  createdAt           DateTime  @default(now())
  updatedAt           DateTime  @updatedAt

  @@index([deletionState])
}
```

`Space` is the user's account boundary — their **[space](/getting-started/product-model/)** — auto-provisioned 1:1 with the user in `getOrCreateProfile` (`Space.id === User.id === tenantId` in solo mode). It holds only foundation-owned lifecycle state; provider billing data (Stripe customer id, plan) lives in the billing module, keyed on `tenantId`. See [User-data isolation](#user-data-isolation-neon-rls).

## Module Database Guidelines

<Aside type="danger">
  **Critical: Modules must use independent tables with NO relations to core models.**
</Aside>

### Why No Relations?

Modules are installed/uninstalled independently. If a module creates foreign key relations to the User table:

1. **Uninstalling breaks** - Can't remove module tables without dropping the FK
2. **Migration conflicts** - Module and template migrations interfere
3. **Circular dependencies** - User model would need to know about all modules

### Correct Pattern for Modules

```prisma
// ✅ CORRECT - Independent table with userId as plain string
model BillingSubscription {
  id        String   @id @default(cuid())
  userId    String   // Just a string, no @relation
  plan      String
  status    String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([userId])  // Index for queries
}
```

```prisma
// ❌ WRONG - Foreign key relation to User
model BillingSubscription {
  id     String @id @default(cuid())
  userId String
  user   User   @relation(fields: [userId], references: [id])  // NO!
}
```

### Querying Without Relations

```typescript
// Get user's subscription
const subscription = await prisma.billingSubscription.findFirst({
  where: { userId: user.id },
});

// Get user data separately if needed
const user = await prisma.user.findUnique({
  where: { id: subscription.userId },
});
```

### Module Migration Strategy

Modules should:

1. **Prefix table names** - e.g., `BillingSubscription`, `BillingInvoice`
2. **Use string IDs** - Reference User.id as plain string
3. **Add indexes** - `@@index([userId])` for query performance
4. **Document dependencies** - State which core models they reference

## Transactions

For operations that need atomicity:

```typescript
import { withTransaction, batchTransaction } from '../lib/prisma.js';

// Interactive transaction
const result = await withTransaction(async (tx) => {
  const user = await tx.user.create({ data: { ... } });
  const settings = await tx.appSettings.update({ ... });
  return { user, settings };
});

// Batch transaction (multiple operations)
const [user, log] = await batchTransaction([
  prisma.user.create({ data: { ... } }),
  prisma.auditLog.create({ data: { ... } }),
]);
```

## Troubleshooting

<Aside type="tip">
  For connection string issues and Neon-specific troubleshooting, see [Neon Database Setup - Troubleshooting](/getting-started/neon-setup/#troubleshooting).
</Aside>

### "Cannot find module '@prisma/client'"

1. Run `npx prisma generate` after schema changes
2. Check import path: use `../generated/prisma/client.js`
3. Ensure `.js` extension in imports (ESM requirement)

### "Prisma client not initialized"

1. Import from `lib/prisma.ts`, not `@prisma/client`
2. Ensure adapter is created before client
3. Check `DATABASE_ADAPTER` env var is set

### "Migration failed"

1. Ensure `DIRECT_DATABASE_URL` is set (non-pooled URL)
2. Check migration status: `npx prisma migrate status`
3. Reset if needed (development only): `npx prisma migrate reset`

## User-data isolation (Neon RLS)

Cross-user isolation is **delegated to the database** — Neon's [Row-Level Security](https://neon.com/docs/guides/neon-rls) is the wall, with the `owned()` helper as application-layer defence-in-depth. Both layers scope on the **isolation key `tenantId`**.

<Aside type="note" title="Two ids on every owned model">
Owned models carry **`tenantId`** (the isolation key — what RLS scopes on) and **`userId`** (the actor — who created the row). In **solo apps `tenantId === userId`** (the deployment's own database is the wall); the column is present from the start so growing to multi-user is a *data* change, not a schema rewrite. `req.user.tenantId` is set at auth time (= the user id in solo mode). The starter ships no owned *resource* models yet — `User` is the actor/identity and `AppSettings` is a global singleton — so these policies apply to the owned models your modules add.
</Aside>

### Neon RLS — the wall

RLS makes the database itself refuse to return another tenant's rows. Because the Prisma `pg`/`neon` adapter connects as the **table owner** (which bypasses RLS), every owned table needs `FORCE ROW LEVEL SECURITY` plus a policy keyed on `tenantId`. `@/lib/owned` provides `withTenantContext()` to publish the caller's tenant to the transaction so the policy can read it:

```typescript
import { withTenantContext } from '../../lib/owned.js';

// RLS policy: USING ("tenantId" = current_setting('app.tenant_id', true))
const docs = await withTenantContext(req.user!.tenantId!, (tx) =>
  tx.document.findMany(), // RLS restricts rows to the caller's tenant
);
```

<Aside type="note" title="Deprecated: withUserContext()">
`@/lib/owned` still exports `withUserContext(userId, fn)` — a deprecated alias from the v1.27.0 opt-in API that sets the legacy `app.user_id` session variable instead of `app.tenant_id`. Apps that adopted it keep working, but should migrate their RLS policies to `current_setting('app.tenant_id', true)` and switch to `withTenantContext()`.
</Aside>

Add the policy in a migration (per-app, since the starter ships no owned models):

```sql
-- backend/prisma/migrations/<timestamp>_rls_documents/migration.sql
ALTER TABLE "Document" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "Document" FORCE ROW LEVEL SECURITY; -- owners (Prisma) are NOT exempt

CREATE POLICY tenant_isolation ON "Document"
  USING ("tenantId" = current_setting('app.tenant_id', true))
  WITH CHECK ("tenantId" = current_setting('app.tenant_id', true));
```

### `owned()` — application-layer defence-in-depth

```typescript
import { owned } from '../../lib/owned.js';

// Read by id, but only within the caller's tenant — no cross-tenant IDOR:
const doc = await prisma.document.findFirst({ where: owned(req.user!.tenantId!, { id }) });

// List only the caller's tenant's rows:
const docs = await prisma.document.findMany({ where: owned(req.user!.tenantId!) });
```

Both layers active together are the backstop: a forgotten `WHERE` can't leak (RLS catches it); a misapplied RLS context can't leak (`owned()` scopes it).

<Aside type="caution">
Cross-tenant reads (admin reports, the last-admin count) must run **outside** `withTenantContext()` (or under a role exempt from the policy), or they will see zero rows.
</Aside>

<Aside type="tip" title="Need hard isolation? database-per-tenant">
For strict/multi-tenant apps, **database-per-tenant** (a Neon **project** per tenant) makes cross-tenant leakage *structurally impossible* — no shared tables — with per-tenant residency and point-in-time restore. RLS is the shared-DB floor; database-per-tenant is the opt-in capability. See the [security roadmap](/starter-template/roadmap/#security-floor-v200).
</Aside>

## Alternative: the Neon Data API

The starter uses **Prisma + Express** for all database access: the frontend never talks to the database directly, and every authenticated query goes through a route handler that has already verified the JWT. This is the recommended architecture for this template and for any module that ships inside it — it keeps business logic, validation, and cross-module composition in one place (the service registry) and matches the rest of the starter's patterns.

Neon also publishes a **[Data API](https://neon.com/docs/data-api/get-started)** — a PostgREST-based REST endpoint in front of your database that accepts a Neon Auth JWT and enforces [row-level security](https://neon.com/docs/data-api/rls) policies at the database level. With the Data API you can query Postgres directly from the frontend:

```typescript
import { createClient } from '@neondatabase/neon-js';

const client = createClient({
  auth:    { url: import.meta.env.VITE_NEON_AUTH_URL },
  dataApi: { url: import.meta.env.VITE_NEON_DATA_API_URL },
});

// The current user's JWT is attached automatically.
// RLS policies that reference auth.user_id() gate access per-row.
const { data } = await client.from('posts').select('*').eq('is_published', true);
```

<Aside type="caution">
**This is not how the starter's Prisma + Express flow works.** The Data API is an alternative architecture — if you adopt it for a feature, that feature bypasses `requireAuth`, the route registry, service registry, error middleware, and Zod validation. You are trading the starter's composition guarantees for fewer moving parts. That tradeoff can be the right call for read-heavy, per-user data where RLS expresses the access rule cleanly; it is almost never the right call for cross-user operations, business-logic-heavy writes, or anything a module expects to compose with.
</Aside>

**When the Data API might make sense in a module:**

- Public read-only dashboards where every row has an RLS policy.
- User-owned content (notes, settings) where `auth.user_id() = row.user_id` is the entire access rule.
- Prototypes and internal tools where you want to skip building an Express route.

**When it will not:** anywhere the starter's conventions matter — shared services, admin-role gating via `requireAdmin`, request validation, global error handling, cross-feature composition. Stick with Prisma + Express for those.

The template does not ship a Data API client by default. If a module wants to use it, the module should own the `VITE_NEON_DATA_API_URL` variable, own the RLS migrations, and document clearly that its data path is Data-API-scoped rather than Express-scoped so app authors know what they are opting into.
