The admin feature provides a comprehensive admin panel for managing app branding, navigation, toolbar items, and users. It also provides two contexts that expose these settings to the rest of the app:

- **`BrandingProvider`** (mounted in `App.tsx` at the app root, since v1.24.0) — fetches the **public** subset of settings (`appName`, `logoUrl`, `logoDisplayMode`, `styleOverrides`) from `GET /api/v1/branding`, no auth required. Available on every route, including unauthenticated auth pages. Consumed via `useBranding()`.
- **`AppSettingsProvider`** (mounted in `DashboardPage`) — fetches the **full** authenticated settings surface (adds nav overrides, toolbar overrides, feature flags, admin-side view of style overrides) from `GET /api/v1/admin/settings`. Only available inside the authenticated shell. Consumed via `useAppSettingsValue()` / `useGlobalAppSettings()`.

Use `useBranding()` in any code that might render on a public route. Use `useAppSettingsValue()` when you specifically need admin fields that are only meaningful inside the shell.

---

## Overview

The admin panel is accessible at `/admin` and is only visible to users with the `admin` role. It provides:

- **App Settings** - Configure app name, logo, and display options
- **User Management** - View, modify roles, and delete users
- **Navigation Management** - Customise sidebar items, sections, and user menu
- **Header Toolbar** - Show/hide toolbar items

<Aside type="tip">
  The first user to sign up automatically becomes an admin. You can promote other users in Admin → Users.
</Aside>

## Architecture

### The Settings Flow

When a user loads the dashboard:

<Steps>
1. `DashboardPage` renders with `AppSettingsProvider` wrapper
2. `AppSettingsProvider` fetches settings from `GET /api/v1/admin/settings`
3. Settings are stored in React context and made available via hooks
4. `AppSidebar` reads logo, app name, and applies navigation overrides
5. `HeaderToolbar` applies toolbar overrides to hide items
6. When admin saves changes, `refetch()` updates the context and UI updates immediately
</Steps>

### Why Settings GET is Public

The admin API has a unique permission structure:

```
GET  /api/v1/admin/settings  → All authenticated users (for sidebar branding)
PATCH /api/v1/admin/settings → Admin only (to modify settings)
```

This design allows:
- **All users** can see the app logo and name in the sidebar
- **Only admins** can change branding and configuration

## App Settings

### Branding Configuration

The App Settings page allows admins to configure:

| Setting | Description |
|---------|-------------|
| **App Name** | Displayed in sidebar header and page titles |
| **Logo** | Custom logo (uploaded as base64 or URL) |
| **Logo Display Mode** | Logo only, logo + name, or name only |
| **Show Dashboard in Nav** | Whether dashboard appears in main nav |
| **Enable Marketing Pages** | Feature flag for public pages |

### Using App Settings in Your Components

```tsx
import { useAppSettingsValue } from '@/features/admin';

function MyComponent() {
  const settings = useAppSettingsValue();

  return (
    <header>
      {settings.logoUrl && <img src={settings.logoUrl} alt={settings.appName} />}
      <h1>{settings.appName}</h1>
    </header>
  );
}
```

### Available Hooks

| Hook | Purpose | Provider |
|------|---------|----------|
| `useBranding()` | Public subset: `appName`, `logoUrl`, `logoDisplayMode`, `styleOverrides`. Forgiving — returns `DEFAULT_BRANDING` if provider absent. | `BrandingProvider` (root) |
| `useBrandingState()` | Full Branding context with loading/error state. | `BrandingProvider` (root) |
| `useGlobalAppSettings()` | Full authenticated context — `settings`, `isLoading`, `error`, `refetch`, `updateSettings`, `setAppTheme`. Throws if used outside the provider. | `AppSettingsProvider` (shell) |
| `useAppSettingsValue()` | Just settings with defaults (most common authenticated read). | `AppSettingsProvider` (shell) |
| `useGlobalNavigationOverrides()` | Navigation + section overrides. | `AppSettingsProvider` (shell) |
| `useGlobalToolbarOverrides()` | Toolbar hidden items. | `AppSettingsProvider` (shell) |

## Navigation Management

The Navigation Management page has three tabs for managing different parts of the navigation system.

### Main Navigation Tab

Displays navigation items grouped by their sections. Each section is a collapsible container showing:

- **Section header** with label, item count, and customisation options
- **Navigation items** within that section

**Section actions:**
- Create new sections to organise navigation items
- Edit the section label
- Toggle whether the section label is visible in the sidebar
- Hide the entire section (items move to the default section)
- Delete admin-created sections (if empty)
- Reset customisations to defaults

**Item actions:**
- Drag to reorder within the section
- Toggle visibility on/off

<Aside type="note">
**Sections can be created by admins** or registered by modules. Admins can create custom sections to organise navigation items. Module-registered sections cannot be deleted (they're tied to the module), but admin-created sections can be removed if empty.
</Aside>

### Secondary Navigation Tab

Items that appear at the bottom of the sidebar (e.g., Support, Admin links).

- Drag to reorder
- Toggle visibility on/off

### User Menu Tab

Items that appear in the user dropdown menu (accessed by clicking the user avatar).

- Drag to reorder
- Toggle visibility on/off

## User Management

The Users page provides:

- **View all users** - List with email, name, role, and join date
- **Change roles** - Promote to admin or demote to user
- **Delete users** - Remove accounts (with confirmation)

### Role Types

| Role | Description |
|------|-------------|
| `user` | Standard user access to all user-facing features |
| `admin` | Full access, can manage all settings and users. Bypasses most role restrictions. |
| `managed` | Permissions controlled by external module. Use this when you need granular access control beyond user/admin. |

<Aside type="tip">
**About the `managed` role:** Set a user's role to `managed` when you want their permissions controlled by a permissions module rather than the simple user/admin system. This enables scenarios like "can edit but not delete" or "can view reports but not export". Without a permissions module installed, `managed` users have equivalent access to regular users.
</Aside>

### Safety Rules

The backend enforces these rules:

- Cannot remove the last admin
- Cannot delete your own account from admin panel
- Cannot demote yourself if you're the last admin

## Adding Admin Pages

Features can register custom admin pages using the admin pages registry. Two patterns are supported — choose based on how often the page is visited.

### Lazy-loaded (recommended for most admin pages)

The built-in admin pages are code-split out of the main bundle via `React.lazy` so users who never visit `/admin` don't pay for the code. The admin route at `features/admin/routes.config.tsx` already wraps children in `<Suspense>`, so no extra wiring is needed.

```tsx
// features/my-feature/admin-pages.registration.ts
import { lazy } from 'react';
import { adminPagesRegistry } from '@/config/admin-pages.registry';
import { Settings } from 'lucide-react';

// Named-export → default-export shim required by React.lazy
const MyAdminPage = lazy(() =>
  import('./pages/MyAdminPage').then((m) => ({ default: m.MyAdminPage })),
);

adminPagesRegistry.register({
  id: 'my-feature-settings',
  title: 'My Feature',
  description: 'Configure my feature',
  icon: Settings,
  component: MyAdminPage,
  order: 30,
});
```

### Static import (for frequently-accessed pages or small pages)

If the admin page is small and/or central to the experience, a plain static import is also fine. The registry accepts any `ComponentType`.

```tsx
// features/my-feature/admin-pages.registration.ts
import { adminPagesRegistry } from '@/config/admin-pages.registry';
import { MyAdminPage } from './pages/MyAdminPage';
import { Settings } from 'lucide-react';

adminPagesRegistry.register({
  id: 'my-feature-settings',
  title: 'My Feature',
  description: 'Configure my feature',
  icon: Settings,
  component: MyAdminPage,
  order: 30,
});
```

Either way, the page will automatically appear in the admin sidebar at `/admin/my-feature-settings`.

### Building Admin Pages with Shared Components

Admin pages use two shared component libraries for consistent layout. Both libraries are shared between settings pages and admin pages.

#### Page Blocks (`@/components/page-blocks`)

Use these for page-level structure:

```tsx
import { PageHeader, LoadingState, ErrorAlert, EmptyState } from '@/components/page-blocks';

function MyAdminPage() {
  const { data, isLoading, error } = useMyData();

  return (
    <>
      <PageHeader
        icon={Settings}
        title="My Feature"
        description="Configure module-specific settings"
      />
      <ErrorAlert error={error} />
      {isLoading ? (
        <LoadingState />
      ) : !data ? (
        <EmptyState
          icon={PackageOpen}
          title="No data yet"
          action={<Button>Create first item</Button>}
        />
      ) : (
        <MyContent data={data} />
      )}
    </>
  );
}
```

| Component | Props |
|-----------|-------|
| `PageHeader` | `icon`, `title`, `description`, `badge` |
| `LoadingState` | *(none — renders a centred spinner)* |
| `ErrorAlert` | `error` (renders `null` when falsy) |
| `EmptyState` | `icon`, `title`, `description`, `action` |

#### Settings Blocks (`@/components/settings-blocks`)

Use these for form content within admin pages:

```tsx
import { SectionHeader, Row, Divider, ToggleSwitch } from '@/components/settings-blocks';
import { Tabs, TabsList, TabsTrigger, TabsContent } from '@/components/ui/tabs';

function MyAdminPage() {
  const [activeTab, setActiveTab] = useState('general');

  return (
    <>
      <PageHeader icon={Settings} title="My Feature" />
      <Tabs value={activeTab} onValueChange={setActiveTab}>
        <TabsList variant="line">
          <TabsTrigger value="general">General</TabsTrigger>
          <TabsTrigger value="advanced">Advanced</TabsTrigger>
        </TabsList>

        <TabsContent value="general">
          <SectionHeader title="Feature Settings" />
          <Row
            label="Enable feature"
            description="Turn this feature on for all users"
            action={<ToggleSwitch checked={enabled} onChange={setEnabled} />}
          />
          <Divider />
        </TabsContent>

        <TabsContent value="advanced">
          {/* advanced settings */}
        </TabsContent>
      </Tabs>
    </>
  );
}
```

<Aside type="note">
The `variant="line"` prop on `TabsList` ships with shadcn's `radix-nova` style (underline on active tab, bottom border across the row). If your app is on a different shadcn style, drop the prop to get the default pill/box tabs instead. The `settings-blocks` library previously exposed a custom `TabBar` component — removed in v1.24.4 in favour of stock shadcn `Tabs` to avoid fighting the style your generator / `shadcn init` produces.
</Aside>

<Aside type="tip">
  The admin's built-in pages (Navigation Management, App Settings, etc.) all use these same components. Follow the same patterns when building your own admin pages.
</Aside>

## API Endpoints

### Public Endpoints (All Authenticated Users)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/v1/admin/settings` | GET | Get all app settings (branding + all overrides) |
| `/api/v1/admin/style-overrides` | GET | Get style registry enable/disable map |
| `/api/v1/branding` | GET | **Public** (no auth) — public branding subset for auth pages |

<Aside type="note">
The `/settings` endpoint returns **everything** the frontend needs to render the shell: app name, logo, navigation overrides (including section overrides), toolbar overrides, style registry overrides, and (since v1.25.0) the module-owned `appTheme` blob. This is what the `AppSettingsProvider` fetches for all users on mount.

`/style-overrides` exists as a dedicated read because the frontend `styleRegistry` bootstrap may want to fetch and apply overrides before the full settings payload arrives — useful for preventing a brief unstyled flash on first paint.
</Aside>

### Admin-Only Endpoints (Require Admin Role)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/v1/admin/settings` | PATCH | Update app branding settings |
| `/api/v1/admin/navigation` | PATCH | Update navigation + section overrides |
| `/api/v1/admin/toolbar` | PATCH | Update toolbar overrides |
| `/api/v1/admin/style-overrides` | **PUT** | Replace the full style registry overrides map |
| `/api/v1/admin/users` | GET | List all users |
| `/api/v1/admin/users/:id/role` | PATCH | Update user role |
| `/api/v1/admin/users/:id` | DELETE | Delete user |

`/style-overrides` uses **PUT** (full replace), not PATCH — sending the entire map each time keeps persistence in sync with the in-memory `styleRegistry` state and automatically drops stale keys for entries that have been unregistered.

<Aside type="tip">
The admin panel also uses a GET endpoint for `/navigation` and `/toolbar` to fetch data for editing. These are admin-only because regular users get everything from the single `/settings` endpoint.
</Aside>

<Aside type="tip" title="Pattern: public reads, admin writes">
Several endpoints here use a **public-read, admin-write** structure: GET is accessible to all authenticated users (so the frontend can render the shell correctly), but PATCH/PUT/DELETE require the admin role. `/settings` and `/style-overrides` both follow this pattern.

If you build a module that needs admins to configure app-wide state that every user reads, replicate this pattern: one route registration for the public reader, one admin-only writer.
</Aside>

## Database Schema

```prisma
model AppSettings {
  id                   String   @id @default("default")
  appName              String   @default("My App")
  logoUrl              String?
  logoDisplayMode      String   @default("logo-and-name")
  showDashboardInNav   Boolean  @default(true)
  enableMarketingPages Boolean  @default(false)
  navigationOverrides  Json     @default("{}")  // Items + sections
  toolbarOverrides     Json     @default("{}")  // Toolbar hidden
  styleOverrides       Json     @default("{}")  // Style registry enable/disable
  createdAt            DateTime @default(now())
  updatedAt            DateTime @updatedAt
}
```

The `navigationOverrides` JSON field contains both item overrides and section overrides:

```typescript
interface NavigationOverrides {
  hidden: string[];                        // Hidden item IDs
  orderOverrides: Record<string, number>;  // Item order
  sections: Record<string, SectionOverride>; // Section customisation
  hiddenSections: string[];                // Hidden section IDs
}
```

The `styleOverrides` JSON field is a flat map from `styleRegistry` entry id to boolean enabled flag. Modules ship with `enabled: true` by default; an override here wins. Absence of a key means "use the module's default". See the [Style System page](/starter-template/style-system/) for the full flow.

```typescript
type StyleOverrides = Record<string, boolean>;
```

The table uses a singleton pattern — there's only ever one row with `id = "default"`. The service auto-creates this row on first access.

## Style Registry Overrides

The admin panel is the natural control surface for module-contributed style entries. Modules register entries in the `styleRegistry` with their own `enabled` default; admins can override that per-entry. Overrides persist in `AppSettings.styleOverrides` and are applied by `AppSettingsProvider` during bootstrap.

### Reading the current state

```typescript
import { useStyleRegistry, styleRegistry } from '@/config/registry.hooks';

function StyleOverridesList() {
  // Reactive — re-renders when entries are registered/updated/unregistered.
  const entries = useStyleRegistry();

  return entries.map((entry) => (
    <StyleEntryRow
      key={entry.id}
      entry={entry}
      enabled={styleRegistry.isEnabled(entry.id)}
    />
  ));
}
```

`styleRegistry.isEnabled(id)` composes the module default with any admin override in one place, so admin UI doesn't need to duplicate that logic.

### Writing an override

```typescript
// 1. Flip locally so the pipeline re-applies immediately (hot side responds)
styleRegistry.setOverride(entry.id, false);

// 2. Persist the full map — server replaces atomically
await api.put('/api/v1/admin/style-overrides', {
  overrides: styleRegistry.getOverrides(),
});
```

`clearOverride(id)` reverts to the module's registered default. Calling `getOverrides()` returns only *explicit* overrides (not the computed enabled state for every entry), which keeps the persisted map tight and means a module changing its default later is picked up by any entry that wasn't explicitly overridden.

See also:

- [Style System](/starter-template/style-system/) — full flow from module registration through admin override to reinstall
- [Registry Reference → Style Registry](/starter-template/registries/#style-registry) — full entry shape and API

## App Theme (`appTheme`)

Added in **v1.25.0**. A dedicated field on `AppSettings` for **module-owned theme configuration**. The starter treats it as opaque JSON storage — the shape is defined and validated by the consuming style module (e.g. useFoundry), not the template.

### Why it's separate from `styleOverrides`

`styleOverrides` is a `Record<string, boolean>` — purpose-built for "is catalogue entry X enabled." `appTheme` is a free-shape blob holding the **base app theme** that the module derives from (shadcn preset codes like `baseColor`, `radius`, `font`, plus whatever extensions the module needs).

The split keeps each storage field cleanly purposed:

| Field | Purpose | Shape |
|-------|---------|-------|
| `appTheme` | Base theme config owned by the style module | `unknown` (opaque, module-defined) |
| `styleOverrides` | On/off switches for individual catalogue entries (`tabs-line-primary`, `button-bold`, etc.) | `Record<string, boolean>` |

A module's typical flow: on mount, read `appTheme` from context, derive CSS (or use it directly if the module writes pre-rendered CSS), register into `styleRegistry` as a single entry (e.g. `{ id: 'app-theme', css: ... }`). The existing styleRegistry pipeline then injects everything via its `<style id="module-styles">` tag — no new injection primitives in the starter.

### Reading it

```tsx
import { useGlobalAppSettings } from '@/features/admin';

function MyThemeConsumer() {
  const { settings } = useGlobalAppSettings();
  const appTheme = settings?.appTheme ?? null;

  // Cast to the module's expected shape (module owns the schema)
  const themeConfig = appTheme as MyModuleThemeConfig | null;
  // ...
}
```

The type comes through as `unknown` from the starter — modules cast on read. If you want compile-time protection, define the expected shape in the module and use a type guard.

### Writing it

```tsx
import { useGlobalAppSettings } from '@/features/admin';

function ThemeSaveButton({ newTheme }: { newTheme: MyModuleThemeConfig }) {
  const { setAppTheme } = useGlobalAppSettings();

  const handleSave = async () => {
    // Optimistic update + PATCH + refetch, packaged into one call.
    await setAppTheme(newTheme);
  };

  return <Button onClick={handleSave}>Save theme</Button>;
}
```

`setAppTheme(theme)`:
1. Updates the query cache optimistically so consumers re-render immediately.
2. PATCHes `/api/v1/admin/settings` with `{ appTheme: theme }`.
3. Refetches on completion to reconcile with whatever the server normalised.

Pass `null` to clear the stored theme.

### Shape contract

**The starter does not enforce a shape.** The Zod schema on `UpdateAppSettingsSchema.appTheme` accepts `z.unknown().optional()`. The database column is `Json?` — anything JSON-serialisable is valid storage.

This is deliberate. The consuming style module evolves its schema independently of starter releases; keeping `appTheme` loose on the starter side means a module can ship v1.0, v1.1, v2.0 without the starter ever needing to re-version. Validation is the module's responsibility on both the write path (before calling `setAppTheme`) and the read path (shape-guard after `useGlobalAppSettings().settings?.appTheme`).

<Aside type="tip" title="Typical shapes a style module might write">
- A **pre-rendered CSS string** — module generates CSS from its internal config on save, writes the string to `appTheme`. Starter-side read → `styleRegistry.update('app-theme', { css: theme as string })` → pipeline injects.
- A **structured JSON config** (shadcn preset codes + module extensions) — module re-compiles to CSS on every read. More flexible if the module wants to re-render on palette changes without a round-trip.
- **Anything JSON-serialisable** — strings, objects, arrays. Storage is neutral.
</Aside>

### Why not let modules register their own `AppSettings` fields?

Extensible settings would mean modules could each add their own columns, and the template's `AppSettings` type would become a union that depends on which modules are installed. Nightmare for typing and API evolution.

Instead: one well-known `appTheme` slot, reserved for whichever style module the app is using. An app with no style module simply has `appTheme: null` and ignores the field. An app with *multiple* style modules picks one as the owner of `appTheme` — the others either share the same shape convention or use their own `styleOverrides` entries.

## Troubleshooting

### Logo Not Showing After Upload

1. Ensure you clicked "Save Changes" after uploading
2. Check browser console for errors
3. Verify the `AppSettingsProvider` wraps your dashboard layout

### Navigation Overrides Not Applied

1. Verify the `AppSettingsProvider` is in your component tree
2. Ensure the item IDs match (check registration files)
3. Check that the admin panel shows your changes as saved

### Section Not Appearing

1. Check that your module's `navigation.registration.ts` is imported
2. Verify at least one item is assigned to the section
3. Check the admin panel - is the section hidden?

### Can't Access Admin Panel

1. Verify your user has `role: 'admin'` in the database
2. Check the User table: `SELECT role FROM "User" WHERE email = 'your@email.com'`
3. First user is auto-promoted to admin on first API call

### Settings Don't Persist

1. Check that the AppSettings migration has run
2. Run: `cd backend && npx prisma migrate dev`
