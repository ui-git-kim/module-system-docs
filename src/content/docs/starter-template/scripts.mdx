The starter template includes shell scripts for creating new apps and keeping them updated with template changes. Both PowerShell (`.ps1`) and Bash (`.sh`) versions are provided.

<Aside type="tip">
  For a step-by-step walkthrough of the create script prompts, see [Create Script Walkthrough](/starter-template/create-script/).
</Aside>

### Scripts in this template

| Command | Purpose | Details |
|---------|---------|---------|
| `./create-app.{ps1,sh}` | Clone the template into a new app | [create-app](#create-app) |
| `npm run template:check` / `template:update` | Preview / merge upstream template updates | [update-template](#update-template) |
| `npm run template:version:{patch,minor,major}` | Bump the template `VERSION` file + root/backend `package.json` | [Version Tracking](#version-tracking) |
| `npm run components:sync` / `components:sync:dry` | Merge module `styleRegistry` contributions into `components.json` and reinstall shadcn components | [Sync shadcn components](#sync-shadcn-components) |
| `npm run deps:check` / `deps:update` / `deps:update:latest` / `deps:lockfile` | Check or apply dependency updates across root/frontend/backend (cross-platform; preserves `npm link`s) | [Dependency Updates](#dependency-updates) |
| `npm run release:{patch,minor,major}` | Bump the **app's** version (not the template's) in apps cloned from this starter | [Version Tracking](#app-version) |
| `npm run security:setup-kms` | Interactive AWS KMS field-encryption setup (prompts for KMS/IAM values, generates the wrapped data key, writes to `backend/.env`) | [Field encryption with AWS KMS](/getting-started/aws-kms-setup/) |

## create-app

Creates a new app from the template, preserving git history for future updates.

### Usage

<Tabs>
  <TabItem label="Windows">
    ```powershell
    # Run the PowerShell script directly
    .\create-app.ps1

    # Or use the batch wrapper (if execution policy blocks .ps1)
    .\create-app.bat
    ```

    <Aside type="note">
      The `.bat` file is a wrapper that runs the `.ps1` script. Use it if PowerShell execution policy prevents running `.ps1` files directly.
    </Aside>
  </TabItem>
  <TabItem label="macOS/Linux">
    ```bash
    ./create-app.sh
    ```
  </TabItem>
</Tabs>

### What It Does

1. **Prompts for configuration:**
   - App name
   - Output directory
   - Neon database credentials (paste from Neon dashboard)
   - Neon Auth URL
   - Backend port

2. **Clones the template:**
   - Preserves full git history (enables updates)
   - Removes `origin` remote (local-only by default)
   - Removes template-only files and docs submodule

3. **Installs dependencies:**
   - Root dependencies (concurrently)
   - Frontend dependencies (React, Vite, Tailwind, etc.)
   - Backend dependencies (Express, Prisma, etc.)

4. **Initialises shadcn/ui:**
   - Runs `shadcn init`
   - Adds required components (sidebar, dropdown-menu, etc.)

5. **Configures environment:**
   - Creates `.env` files for frontend and backend
   - Sets database adapter to `neon`

6. **Sets up database:**
   - Runs `prisma generate`
   - Runs `prisma migrate dev --name init`

7. **Creates initial commit:**
   - Commits all files with "Initialize [app] from template"

8. **Optional: configures app remote:**
   - Prompts for your app's git repository URL

### Database Configuration

The template uses **Neon PostgreSQL**. During setup, paste the connection snippet from your Neon dashboard:

```
DATABASE_URL="postgresql://user:pass@ep-xxx-pooler.../neondb?..."
# DATABASE_URL_UNPOOLED="postgresql://user:pass@ep-xxx.../neondb?..."
```

The script parses both URLs automatically:

| Variable | Description |
|----------|-------------|
| `DATABASE_URL` | Pooled connection (has `-pooler` in hostname) |
| `DIRECT_DATABASE_URL` | Direct connection for migrations |

### Output Structure

After running, you'll have:

```
my-app/
├── frontend/
│   ├── .env              # Frontend environment
│   └── ...
├── backend/
│   ├── .env              # Backend environment
│   └── ...
├── .git/                 # Preserves template history
└── ...
```

## update-template

Merges updates from the upstream template into your app.

### Usage

<Tabs>
  <TabItem label="Windows">
    ```powershell
    # Interactive mode
    .\update-template.ps1

    # Preview changes only
    .\update-template.ps1 -Preview

    # Merge directly
    .\update-template.ps1 -Merge
    ```
  </TabItem>
  <TabItem label="macOS/Linux">
    ```bash
    # Interactive mode
    ./update-template.sh

    # Preview changes only
    ./update-template.sh --preview

    # Merge directly
    ./update-template.sh --merge
    ```
  </TabItem>
</Tabs>

Or use npm scripts:

```bash
npm run template:check   # Preview changes
npm run template:update  # Merge changes
```

<Aside type="note" title="After a template update">
If the update touches any `componentsJson` contributions (module-starter changes, template-level shadcn config changes, style-module updates), run `npm run components:sync` afterwards so `frontend/components.json` is re-merged and shadcn components are reinstalled with the new config. Check the template changelog entry for the version you pulled — it will call out any cold-side changes. Most template updates are runtime-only and don't require a sync.
</Aside>

### What It Does

1. **Checks prerequisites:**
   - Git installed
   - In a git repository
   - Working directory is clean

2. **Sets up template remote:**
   - Adds `template-upstream` remote (if not exists)
   - Configures as pull-only (push disabled)

3. **Fetches updates:**
   - Fetches from `template-upstream/main`

4. **Self-updates:**
   - If the update script itself changed, it updates and re-runs

5. **Shows available changes:**
   - Lists commits that will be merged
   - Shows version difference

6. **Offers options:**
   - **Merge** (recommended): Creates merge commit, preserves history
   - **Preview**: Shows changes without applying

7. **Handles conflicts:**
   - Auto-removes template-only files (create-app.*, etc.)
   - Provides conflict resolution guidance
   - Lists files with remaining conflicts

8. **Post-merge tasks:**
   - Checks for new schema models
   - Auto-adds missing models (e.g., AppSettings)
   - Preserves `npm link` symlinks across `install:all` (snapshot + restore)
   - Reconciles foundation npm scripts into your `package.json` (so new `deps:*` aliases reach older apps)
   - Reconciles foundation env keys into your `.env` files (appends missing security knobs; never overwrites)
   - Provides next steps

### Configuration

Edit the script to configure template source:

<Tabs>
  <TabItem label="Bash">
    ```bash
    # update-template.sh

    # Relative path (template is sibling folder)
    TEMPLATE_REPO="../unique-react-fullstack-starter"

    # Or absolute path
    TEMPLATE_REPO="/path/to/unique-react-fullstack-starter"

    # Or SSH URL
    TEMPLATE_REPO="git@github.com:ui-git-kim/unique-react-fullstack-starter.git"

    # Or HTTPS URL
    TEMPLATE_REPO="https://github.com/ui-git-kim/unique-react-fullstack-starter.git"
    ```
  </TabItem>
  <TabItem label="PowerShell">
    ```powershell
    # update-template.ps1

    # Relative path (template is sibling folder)
    $TEMPLATE_REPO = "../unique-react-fullstack-starter"

    # Or absolute path
    $TEMPLATE_REPO = "C:/Projects/unique-react-fullstack-starter"

    # Or SSH URL
    $TEMPLATE_REPO = "git@github.com:ui-git-kim/unique-react-fullstack-starter.git"

    # Or HTTPS URL
    $TEMPLATE_REPO = "https://github.com/ui-git-kim/unique-react-fullstack-starter.git"
    ```
  </TabItem>
</Tabs>

### First-Time Merge

If your app has unrelated git history (common when cloned without preserved history), the script handles this:

1. Detects unrelated histories
2. Uses `--allow-unrelated-histories` flag
3. Provides guidance on resolving conflicts
4. Auto-removes template-only files

### Conflict Resolution

When conflicts occur:

1. Open conflicting files in your editor
2. Look for conflict markers:
   ```
   <<<<<<< HEAD
   Your changes
   =======
   Template changes
   >>>>>>> template-upstream/main
   ```
3. Choose which changes to keep (or combine them)
4. Stage resolved files: `git add <file>`
5. Complete the merge: `git commit`

Or abort: `git merge --abort`

### Files Auto-Removed

These template-only files are automatically removed during merge:

- `create-app.ps1`
- `create-app.sh`
- `create-app.bat`

In addition, the excluded template-only paths `docs/` (the docs submodule), `.claude/`, and `.template-archive/` are stripped from the merge — both proactively and on conflict — so they never leak from the template into your app.

### Schema Updates

The script automatically adds new models from the template:

```bash
# If AppSettings model is added to template but missing from your schema:
[INFO] Adding AppSettings model to schema...
[OK] Added AppSettings model

# You'll need to run the migration:
cd backend && npx prisma migrate dev --name add_template_models
```

## Sync shadcn components

Merge module `styleRegistry` contributions into `frontend/components.json` and reinstall shadcn components with the new configuration. Use after installing, updating, or disabling any module that carries `componentsJsonContribution` — any change to `style`, `tailwind.baseColor`, `tailwind.cssVariables`, `aliases`, or `registries`.

### npm scripts

```bash
npm run components:sync       # Merge + reinstall
npm run components:sync:dry   # Preview the merged components.json (no writes, no reinstall)
```

### What it does

The script walks `.module-manifests.json` and for every **enabled** module (according to `.module-state.json`):

1. Reads the `componentsJsonContribution` field.
2. Enforces exclusivity on `style`, `tailwind.baseColor`, `tailwind.cssVariables` — fails with exit code 2 if two enabled modules claim the same field with different values.
3. Merges additive fields (`aliases`, `registries`) with last-write-wins and a warning on overlap.
4. Writes the merged `frontend/components.json` atomically (temp file + rename).
5. Collects the union of `installedShadcnComponents` across enabled modules and runs `npx shadcn@latest add --overwrite --yes <union>` so components pick up the new base colour / style / variables.

### Direct usage

<Tabs>
  <TabItem label="Windows">
    ```powershell
    .\scripts\sync-components.ps1                  # Merge + reinstall
    .\scripts\sync-components.ps1 -DryRun          # Preview only
    .\scripts\sync-components.ps1 -SkipReinstall   # Update components.json but don't reinstall
    ```
  </TabItem>
  <TabItem label="macOS/Linux">
    ```bash
    ./scripts/sync-components.sh                   # Merge + reinstall
    ./scripts/sync-components.sh --dry-run         # Preview only
    ./scripts/sync-components.sh --skip-reinstall  # Update components.json but don't reinstall
    ```

    **Requires `jq`** — install via `brew install jq` (macOS) or `apt install jq` (Debian/Ubuntu).
  </TabItem>
</Tabs>

### Safe to run anytime

The script is idempotent: running it when nothing has changed is a no-op (components.json matches the module merge; shadcn add with `--overwrite` rewrites the same files). It's also safe to run on a fresh app with no modules installed yet — it reports "Nothing to sync." and exits 0.

### In-app version

An admin-UI reinstall flow (confirmation dialog, streamed progress) is [planned as a follow-up](/starter-template/roadmap/#in-app-component-reinstall-for-style-registry). For now, this is a terminal-only operation.

## Dependency Updates

Keep the root, frontend and backend dependencies up to date. The updater is a cross-platform **Node** script (`scripts/update-deps.mjs`) — it runs the same on Windows, macOS and Linux with no PowerShell dependency. Four modes cover everything from a quick peek to a full major-version sweep to a `node_modules`-free lockfile refresh.

<Aside type="note" title="Runs in apps AND in the template repo">
  In a **cloned app**, use `deps:update` / `deps:update:latest` (they install into `node_modules`). In the **template repo itself** — which ships no `node_modules` — use `deps:lockfile` to refresh the lockfiles without installing. Earlier versions wired these to PowerShell and to a user-owned `package.json`, so on some host apps `npm run deps:update` did nothing; both are fixed in v1.32.0 (see [Reaching host apps](#reaching-host-apps)).
</Aside>

### npm scripts

```bash
npm run deps:check          # Preview pending updates (no changes)
npm run deps:update         # Safe: npm update in root/frontend/backend (respects semver ^ and ~)
npm run deps:update:latest  # Bump everything to latest via npm-check-updates (may break)
npm run deps:lockfile       # Regenerate lockfiles from package.json — no node_modules install
npm run deps:relink         # Re-create npm-link symlinks from the last snapshot
```

### What each mode does

| Mode | Respects semver? | Tool | Needs `node_modules`? | Changes lockfiles? |
|------|------------------|------|:--:|:--:|
| `deps:check` | n/a (read-only) | `npm outdated` + `npx npm-check-updates` | No | No |
| `deps:update` | Yes — won't cross majors | `npm update` | Yes (skips packages without it) | Yes |
| `deps:update:latest` | **No** — jumps to latest | `npx npm-check-updates -u` + `npm install` | Yes | Yes |
| `deps:lockfile` | Yes | `npm install --package-lock-only` | **No** | Yes |

`deps:check` is informational — it prints what's pending and always exits 0 (it's a preview, not a CI drift gate).

### npm-link preservation

`deps:update` and `deps:update:latest` run broad `npm install`s, which prune `npm link` symlinks to locally-developed modules. The updater **snapshots** those links first (`.module-links.json`, git-ignored) and **restores** them after the install, so your linked modules survive. `template:update` does the same around its `install:all` step. If a link is ever lost, `npm run deps:relink` restores it from the last snapshot.

### Direct usage

<Tabs>
  <TabItem label="Windows">
    ```powershell
    node scripts\update-deps.mjs              # Safe update
    node scripts\update-deps.mjs --check       # Preview only
    node scripts\update-deps.mjs --latest      # Bump to latest (may break)
    node scripts\update-deps.mjs --lockfile-only  # Refresh lockfiles, no install
    ```

    The legacy `.\scripts\update-deps.ps1` still works — it's now a thin shim that calls the Node script.
  </TabItem>
  <TabItem label="macOS/Linux">
    ```bash
    node scripts/update-deps.mjs               # Safe update
    node scripts/update-deps.mjs --check       # Preview only
    node scripts/update-deps.mjs --latest      # Bump to latest (may break)
    node scripts/update-deps.mjs --lockfile-only  # Refresh lockfiles, no install
    ```

    The legacy `./scripts/update-deps.sh` still works — it's now a thin shim that calls the Node script.
  </TabItem>
</Tabs>

### Reaching host apps

The root `package.json` is **user-owned**, so `template:update` never overwrites it — which means `deps:*` aliases added to the template never reached apps created before they existed (`npm run deps:update` → `Missing script`). To fix that, `template:update` runs `scripts/sync-package-scripts.mjs` after merging, which ensures the foundation `deps:*` aliases exist and point at the canonical Node commands. Only those foundation keys are touched; your own scripts are left alone.

**Env keys** have the same problem: `.env` is user-owned, so new foundation knobs the template introduces (e.g. `SECURITY_PROFILE`, `SECURITY_ENCRYPTION_KEY`, the deletion-cascade `NEON_AUTH_*`) never reach existing apps. `template:update` also runs `scripts/sync-env.mjs`, which **appends** any missing foundation key to `frontend/.env` / `backend/.env` — the security tier active (matching the code default), secrets as commented placeholders to fill if needed. It only ever appends; existing values are never touched, and `.env` stays git-ignored (nothing is committed). You don't hand-add these per app.

<Aside type="tip">
  If `template:update` stops on a merge conflict, the post-merge reconcilers don't run. After resolving the conflict, run `node scripts/sync-package-scripts.mjs`, `node scripts/sync-env.mjs` (and `npm run deps:relink` if needed) to finish.
</Aside>

### Recommended workflow

1. `npm run deps:check` — see what's pending across all three packages.
2. `npm run deps:update` — apply safe, semver-respecting updates. Commit.
3. When you're ready to take breaking bumps: `npm run deps:update:latest`, run the app, fix fallout, then bump the template version (`template:version:minor` for new baseline, `template:version:major` if your downstream modules will break).

<Aside type="tip">
  When shadcn/ui components update, re-running `npx shadcn@latest add <component>` may prompt to overwrite. After updating, check that required providers (e.g. `TooltipProvider`) are still mounted — the template already mounts `TooltipProvider` at the root in `main.tsx`, so tooltip additions "just work".
</Aside>

## Version Tracking

<Aside type="caution">
  **Version tracking is critical for the update system.** The template's `VERSION` file enables the update script to show what version you're updating from/to. Always bump the version when making infrastructure changes.
</Aside>

### Template Version

The template uses a `VERSION` file at the root:

```
1.13.0
```

The update script displays version information:

```
[INFO] Current template version: 1.12.0
[INFO] Template version available: 1.13.0
```

### Bumping Template Version

<Tabs>
  <TabItem label="npm Scripts">
    ```bash
    npm run template:version:patch  # 1.2.0 -> 1.2.1
    npm run template:version:minor  # 1.2.0 -> 1.3.0
    npm run template:version:major  # 1.2.0 -> 2.0.0
    ```
  </TabItem>
  <TabItem label="Direct">
    ```bash
    ./scripts/bump-version.sh patch --commit
    ./scripts/bump-version.sh minor --commit
    ./scripts/bump-version.sh major --commit
    ```
  </TabItem>
</Tabs>

### When to Bump

| Change Type | Bump | Changelog |
|-------------|------|-----------|
| Bug fixes in `config/`, `lib/`, `middleware/` | Patch | No |
| New features, new registries, new utilities | Minor | Yes |
| Breaking changes to registries or APIs | Major | Yes |

### App Version

Apps use `package.json` version (separate from template version). This is a distinct script from the template `bump-version.*` — app versions are bumped by `scripts/bump-app-version.*`, which the `release:*` aliases call:

<Tabs>
  <TabItem label="npm Scripts">
    ```bash
    npm run release:patch  # 1.0.0 -> 1.0.1
    npm run release:minor  # 1.0.0 -> 1.1.0
    npm run release:major  # 1.0.0 -> 2.0.0
    ```
  </TabItem>
  <TabItem label="Direct">
    ```bash
    # Bash
    ./scripts/bump-app-version.sh patch --commit --tag
    ./scripts/bump-app-version.sh minor --commit --tag
    ./scripts/bump-app-version.sh major --commit --tag
    ```

    ```powershell
    # PowerShell
    pwsh -File ./scripts/bump-app-version.ps1 -Type patch -Commit -Tag
    pwsh -File ./scripts/bump-app-version.ps1 -Type minor -Commit -Tag
    pwsh -File ./scripts/bump-app-version.ps1 -Type major -Commit -Tag
    ```
  </TabItem>
</Tabs>

## Troubleshooting

### "Template path not found"

Edit the script and update `TEMPLATE_REPO` to point to your template location.

### "You have uncommitted changes"

Commit or stash your changes before updating:

```bash
git stash
./update-template.sh
git stash pop
```

### "Merge conflicts"

1. View conflicts: `git status`
2. Open each conflicting file
3. Resolve conflicts manually
4. Stage and commit

### "Cannot merge unrelated histories"

The script handles this automatically. If running git manually:

```bash
git merge template-upstream/main --allow-unrelated-histories
```

### PowerShell Execution Policy

If `.ps1` files won't run, use the `.bat` wrapper:

```powershell
.\create-app.bat
```

Or temporarily allow scripts:

```powershell
Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope Process
.\create-app.ps1
```
