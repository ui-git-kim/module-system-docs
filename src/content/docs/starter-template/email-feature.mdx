The starter ships email as a **foundation feature** (`backend/src/features/email/`, backend only â€” a sibling to `auth`, `user` and `admin`, **not a module**). It is a **seam, not a mail system**: one `sendEmail()` entry point, a provider interface, a default that logs instead of sending, and four optional adapters whose SDKs are **app-installed, never template dependencies**.

<Aside type="tip" title="Philosophy">
Delivery is delegated to expert providers (Resend, SendGrid, AWS SES, or any SMTP host via Nodemailer). The foundation owns the *mechanism* â€” one correct wiring of "send this message" â€” while templates, rendering and per-feature email *content* stay with callers and modules. Same pattern as the [security feature's](/starter-template/security-feature/) key-provider and auth-identity seams.
</Aside>

<Aside type="note" title="Auth emails are not sent through this seam">
Verification, password-reset and change-email messages are sent by **Neon Auth itself**; a custom SMTP provider for those is configured in the **Neon dashboard**, not here (see [Authentication](/starter-template/auth-feature/)). This seam serves everything else: notifications, billing receipts, and module-sent mail.
</Aside>

## How it works

- **`sendEmail(message)`** is the one entry point. It resolves the active provider, applies the `email.outbound` filter, sends, fires the `email.sent` action, and **never throws** â€” failures normalise to `{ sent: false, detail }` so callers treat non-delivery as data.
- **Unconfigured = visible, not silent.** With no provider wired, the **console provider** logs the envelope (body at `debug` level) and returns `sent: false` â€” it never fakes success. In development this doubles as a "what would have been sent" console.
- **Adapters are optional dependencies.** Each SDK is imported dynamically via a variable specifier (the same idiom as the [AWS KMS key provider](/getting-started/aws-kms-setup/) and [Sentry](/getting-started/error-tracking/)), so the template â€” and any app not sending email â€” type-checks and builds without any mail SDK installed.
- **Wiring is configuration, not code.** Set the `EMAIL_*` env keys and the feature auto-wires the matching adapter at startup; a bad or incomplete config logs an error and leaves the console default active â€” it never crashes bootstrap.

## Sending email

```typescript
import { sendEmail } from '../email/index.js';

const result = await sendEmail({
  to: user.email,
  subject: 'Your receipt',
  text: 'Thanks! Your receipt is attached.',          // plain text â€” required
  html: '<p>Thanks! Your receipt is attached.</p>',    // optional alternative
  attachments: [{ filename: 'receipt.pdf', content: pdfBuffer, contentType: 'application/pdf' }],
});

if (!result.sent) {
  log.warn({ detail: result.detail }, 'Receipt not delivered');
}
```

`EmailMessage` is deliberately small: `to`, `subject`, `text`, optional `html` / `from` / `replyTo` / `attachments`. Single recipient â€” the product is solo, so fan-out is a caller loop, never seam surface. Attachment `content` follows the Nodemailer convention: a **string is UTF-8 text**, **binary content must be a `Buffer`** (adapters convert per provider).

`EmailResult` reports `sent`, the `provider` name, the provider's `messageId` when available, and a human-readable `detail`.

## Enabling a provider

Set `EMAIL_PROVIDER` + `EMAIL_FROM` (plus the provider's key) in `backend/.env`, and install the matching SDK **in your app** â€” the stubs are already in your generated `.env`:

```bash frame="none"
# EMAIL_PROVIDER=              # resend | sendgrid | ses | smtp
# EMAIL_FROM=                  # e.g. My App <noreply@yourdomain.com>
```

    <Aside type="caution">

  <TabItem label="SMTP (Nodemailer)">
    <Steps>
      1. `cd backend && npm install nodemailer`
      2. Set `EMAIL_PROVIDER=smtp`, `EMAIL_FROM=...`, `SMTP_URL=smtp://user:pass@host:587` (any SMTP host â€” a mail provider, SES's SMTP interface, or Mailpit/MailHog in dev).
      3. Restart.
    </Steps>
  </TabItem>
</Tabs>

A declared-but-broken config (key missing, SDK not installed) stays **visible**: startup logs an error, `getEmailProviderStatus()` reports `configured: true, ready: false`, and sends return `sent: false` with an actionable message naming the missing piece.

## The admin Email page (v1.58.0)

The frontend gained an email feature (`frontend/src/features/email`) whose admin page â€” **Email**, in the admin sidebar's Platform section â€” surfaces the seam without touching its service-only design:

- **Provider status** â€” renders the `getEmailProviderStatus()` probe (provider / declared / ready, plus the seam's own remediation detail) with env-key guidance when not ready.
- **Test delivery** â€” `POST /api/v1/admin/email/test` sends a test email **to the signed-in admin's own address only** (the body carries no recipient, so the endpoint cannot be used as a relay) and renders the returned `EmailResult` verbatim â€” including honest non-delivery (`sent: false` + detail), since `sendEmail` never throws.

Both endpoints (`GET /api/v1/admin/email/status`, `POST /api/v1/admin/email/test`) live in the backend email feature (`email.routes.ts`, admin-gated) â€” the sending seam itself remains service-only. Deliberately **not** included: an outbound log viewer (the seam is stateless by design), template previews (templates are module territory), and admin editing of `EMAIL_*` env (secrets stay out of the browser); see the roadmap if demand materialises.

## Custom adapters

Any provider can be wired by implementing the two-member `EmailProvider` interface and registering it once at bootstrap:

```typescript
import { setEmailProvider, type EmailProvider } from '../email/index.js';

const myProvider: EmailProvider = {
  name: 'my-provider',
  async send(message) {
    // one provider API call
    return { sent: true, provider: 'my-provider', messageId: '...' };
  },
};
setEmailProvider(myProvider);
```

## Hooks

| Hook | Kind | Fires |
|------|------|-------|
| `email.outbound` | filter | Before send â€” transform the outgoing `EmailMessage` (e.g. append a footer) |
| `email.sent` | action | After a successful send, with `{ message, result }` â€” fire-and-forget, never delays the send path |

```typescript
import { filterRegistry } from '../../config/filter.registry.js';

filterRegistry.register<EmailMessage>('email.outbound', 'my-module.footer', (m) => ({
  ...m,
  text: `${m.text}\n\nâ€” Sent by My App`,
}));
```

## Enabling on an existing app

Run `npm run template:update` â€” the merge brings the feature in, and `sync-env` appends the commented `EMAIL_*` stubs to your `backend/.env`. Fill them in and install the SDK per the tabs above.

## What the foundation deliberately does not do

- **Templates / rendering** â€” callers pass `text` (+ optional `html`); template systems are module territory (the planned Email module).
- **Bulk / marketing send** â€” one message, one recipient; campaign tooling belongs with a provider.
- **Queuing and retry** â€” `sendEmail` is a direct call. Durable retry arrives with the background-jobs seam (roadmap) or via your provider.
- **Auth emails** â€” Neon Auth sends its own (see the note above).

The seam is stateless â€” it stores no rows, so it registers a security manifest declaring only the `email.sent` audit event, and needs no deletion handler.
