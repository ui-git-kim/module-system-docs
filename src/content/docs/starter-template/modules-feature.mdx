The starter ships the **module system itself** as a foundation feature (`backend/src/features/modules/`, backend only — a sibling to `auth`, `user`, [`email`](/starter-template/email-feature/), [`jobs`](/starter-template/jobs-feature/) and [`notifications`](/starter-template/notifications-feature/), **not a module**). This is where you work on *how modules work in the system*: it owns the **`ModuleContract`** (the single source of truth for "what a module is"), the runtime **`moduleRegistry`** installed modules register with, the **`InstalledModule`** state table and its startup sync, and the **dependency-check seam**.

<Aside type="note" title="Two things are called 'manifest' — they stay separate">
The **security manifest** (`ModuleSecurityManifest`) declares a feature's or module's *security surface* and keeps living in per-feature `security.registration.ts` files — foundation features use it too, without being modules. The **module contract** (`ModuleContract`) is different: it is the module system's *identity and lifecycle* contract. Different consumers, different timing; see the [Security feature](/starter-template/security-feature/).
</Aside>

## The contract — one source of truth, two repos

`ModuleContract` lives canonically at `backend/src/features/modules/module-contract.ts` — a deliberately **self-contained file** (zero imports; types plus one `MODULE_CONTRACT_VERSION` const). The module starter consumes a **hash-stamped synced copy** (`src/scaffold/module-contract.ts`, refreshed by its `contract:sync` script) and extends it with its scaffold-time codegen fields to form the full `ModuleDefinition` — so "your module definition **extends** the foundation module contract". The copy propagates to every downstream module automatically on `scaffold:update` (it is scaffold infrastructure, not module-owned).

```typescript
export interface ModuleContract {
  id: string;           // kebab-case unique id
  name: string;
  description: string;
  version: string;      // semver

  /** Module id → semver range. Checked as WARNINGS at install
   *  (CLI) and at boot, which also computes a topological order
   *  and reports cycles. Hard enforcement is a future step. */
  depends?: Record<string, string>;

  /** @deprecated Use `depends`. Still honoured everywhere depends is checked. */
  requiredModules?: string[];

  lifecycle?: LifecycleHooks;   // CLI-time hooks (install/update/activate/…)
}
```

Lifecycle hooks (`onInstall`, `onUpdate`, `onActivate`, `onDeactivate`, `onUninstall`) execute in the **module CLI's Node process** inside the host app's working directory — never in the app server. Their contexts are generic over the definition type, so scaffold-extended definitions keep typed access to their own fields inside hooks. If you change the contract, **bump `MODULE_CONTRACT_VERSION`** and re-sync the module starter — the version-bump script reminds you when the file changed since the last release. The contract also carries the shared **semver range** semantics (v1.1.0) and the **module-migration** table shape + filename parse/order (v1.2.0, see [Module migrations](#module-migrations-contract-v120)), so the CLI and the host never disagree about a range's meaning or a migration's identity.

## How the runtime learns about modules

- **Registration at import time.** An installed module's backend feature ships a generated `module.registration.ts` that registers `{ id, name, version, depends }` with the `moduleRegistry` as a side effect of being imported — exactly the `security.registration.ts` pattern. Foundation features do **not** register: they are not modules.
- **Startup state sync.** After all features and modules have registered, `app.ts` fires `runModuleStartupTasks()` (fire-and-forget, never throws): the feature diffs the registry against the `InstalledModule` table and records each transition. This diff is the *only* way the runtime can observe CLI-side install, update and uninstall — they happen while the server is down.
- **Audit events at each transition** (declared in the feature's security manifest): `module.installed` (new row), `module.updated` (version change), `module.missing` (a row whose code no longer registers — uninstalled?), and `module.dependencyIssue`. Each also fires on the `actionRegistry`, so other features and modules can react.
- **Dependency check.** Every boot, each registered module's `depends` (and legacy `requiredModules`) ids are presence-checked, and — since v1.49.0 — registered dependencies are **range-checked** against the declared semver range (the contract's own `satisfiesRange`, so the CLI and the host can never disagree about what a range means) and the registry computes a **dependency-first topological order** (Kahn's algorithm), reporting cycles as readable paths (`a → b → a`). The result partitions the registered ids three ways: `order`, cycle members, and `blocked` — modules that cannot be ordered because they sit behind a cycle or on one entangled with an already-reported cycle; blocked ids get their own startup warning. Unlike the CLI's install-time check this catches **uninstall drift** — a dependency removed after its dependents were installed. A dependency that is *installed but not registered* (an older-vintage module, or a frontend-only module with no backend code) is excused via a feature-folder fallback rather than warned about; a version mismatch gets no such excusal. Everything is a **warning** — nothing hard-fails yet.

```typescript
// generated per module: backend/src/features/my-feature/module.registration.ts
import { moduleRegistry } from '../modules/index.js';

moduleRegistry.register({
  id: 'my-feature',
  name: 'My Feature',
  version: '1.2.0',
  depends: { 'ai-core': '^1.2.0' },
});
```

## The `InstalledModule` table

Deployment-level state — like `AppSettings`, it carries **no `tenantId`**, is not account data, and is untouched by the account-deletion cascade (modules are installed per deployment, not per space). The sync **never deletes rows**: a module whose code disappears is stamped `missingSince` (cleared if it returns), so "was installed, code now absent" stays visible for the future Admin Module Management UI. The `enabled` column is that UI's seam for *runtime* enablement.

<Aside type="caution" title="Registered = installed ≠ enabled">
The module CLI's `enable`/`disable` commands flip a flag in `.module-state.json` that the runtime cannot see — a "disabled" module's routes, navigation and registrations still execute today. Do not read `moduleRegistry.getAll()` as enablement; the `InstalledModule.enabled` column is where runtime enablement will live.
</Aside>

## Module migrations (contract v1.2.0)

A module can ship ordered migration files (`src/templates/migrations/<version>__<slug>.{sql,ts}`); its CLI copies them into the host at `backend/prisma/module-migrations/<module-id>/` and — the module starter's job — **applies** the pending ones at `npx <module> update`, recording each in a runner-owned `_module_migrations` table. That table is deliberately **not** a Prisma model (it is runner plumbing, not app data; a model would cost the multi-site embedded-schema hand-sync — the same call made for the security `AuditLog` table), so both the runner and the host reach it through raw SQL.

The **starter's** share is read-only. Contract v1.2.0 adds the shared surface both sides agree on — the table name/DDL/row shape (`MODULE_MIGRATIONS_TABLE`, `MODULE_MIGRATIONS_DDL`, `ModuleMigrationRecord`) and the `<version>__<slug>.{sql,ts}` filename parse + ordering (`parseMigrationFilename`, `compareMigrations`) — so the CLI's applied order and the host's displayed order can never drift. `getModuleMigrationStatuses()` diffs the on-disk files against the `_module_migrations` rows per registered module and reports applied/pending counts (never-throwing; disk-only when the table can't be read). The host **never applies** a migration — that stays a CLI operation.

## The admin Modules page

`/admin/modules` (v1.50.0, `adminPagesRegistry` order 16, between Security and System) is a **read-only** window over all of the above: registered modules merged with the `InstalledModule` rows (flagging `missingSince` and recorded-but-not-registered states), dependency issues with their excusal flags, cycle paths, the computed order, and — since v1.52.0 — a **Migrations** section showing each module's applied/pending counts with the pending filenames. The backend surface is one admin-gated endpoint (`GET /api/v1/admin/modules/overview`) served by the side-effect-free `getModulesOverview()` — unlike the startup check it never logs or fires actions. Install, update, uninstall and enable stay CLI operations until the Admin Module Management trust boundary (executing module code from the UI) is decided — see the [roadmap](/starter-template/roadmap/).

## What the foundation deliberately does not do (yet)

- **Hard dependency enforcement and ordered install/boot** — ranges are checked and the topological order is computed (`moduleRegistry.computeModuleOrder()`), but mismatches and cycles only warn, and nothing consumes the order at runtime yet ([roadmap](/starter-template/roadmap/)).
- **Module management from the UI** — the admin page is read-only; install/update/uninstall/enable from the browser means executing module code, and that trust boundary is deliberately undecided. Runtime enable/disable also needs a dynamic mount/unmount story (features are wired by static imports).
- **Owning the CLI's state files** — `.module-state.json` / `.module-manifests.json` remain CLI bookkeeping; the runtime's record is the `InstalledModule` table.
- **Applying module migrations** — the starter reads migration status (above); the **runner** that applies them lives in the module CLI (module starter), since migrations run at CLI time against `DIRECT_DATABASE_URL`, never at server boot.

The feature's own security manifest declares only the four audit events above — it stores no per-account rows, so it needs no owned models, data categories or deletion handler.
