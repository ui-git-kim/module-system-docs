---
title: Filter & Action Hooks
description: How modules use filters to transform data and actions to run side effects at defined hook points
---

import { Tabs, TabItem, Aside } from '@astrojs/starlight/components';

The starter template provides two hook systems that let modules extend behaviour without modifying core code: **filters** transform data as it passes through a pipeline, and **actions** execute side effects at defined hook points.

Both are available on frontend and backend. The action registry is identical on both sides — frontend and backend both support async actions (`addActionAsync`, `doActionAsync`, `doActionAll`, `doActionParallel`). The filter registry's async variants (`registerAsync`, `applyAsync`, `applyAll`) are backend-only.

---

## Filters vs Actions

| | Filters | Actions |
|---|---------|---------|
| **Purpose** | Transform data | Run side effects |
| **Returns** | Modified value (same type) | Nothing |
| **Pipeline** | Each filter receives previous output | Independent execution |
| **Example** | Modify a user's display name | Send an email after user creation |

---

## Filter Registry

Filters let modules transform data at defined hook points. Each filter receives the output of the previous filter, creating a pipeline.

### Registering a Filter

<Tabs>
  <TabItem label="Frontend">
```typescript
import { filterRegistry } from '@/config/filter.registry';

// Transform user display name (e.g., add title prefix)
filterRegistry.register(
  'user.displayName',      // Hook name
  'titles-module',         // Unique ID for this filter
  (name, ctx) => {         // Transform function
    return ctx.user.title ? `${ctx.user.title} ${name}` : name;
  },
  5  // Order (lower = earlier in pipeline)
);
```
  </TabItem>
  <TabItem label="Backend">
```typescript
import { filterRegistry } from '../../config/filter.registry.js';

// Sync filter
filterRegistry.register(
  'api.user.profile.response',
  'privacy-module',
  (profile) => ({ ...profile, email: maskEmail(profile.email) }),
  20
);

// Async filter (for database lookups or external calls)
filterRegistry.registerAsync(
  'api.user.profile.response',
  'enrichment-module',
  async (profile, ctx) => {
    const stats = await statsService.getUserStats(profile.id);
    return { ...profile, stats };
  }
);
```
  </TabItem>
</Tabs>

### Applying Filters

<Tabs>
  <TabItem label="Frontend">
```typescript
// In a component or utility
const displayName = filterRegistry.apply(
  'user.displayName',
  user.name,
  { user }  // Optional context passed to all filters
);
```
  </TabItem>
  <TabItem label="Backend">
```typescript
// Sync only
const result = filterRegistry.apply('hook.name', value, context);

// Async only
const result = await filterRegistry.applyAsync('hook.name', value, context);

// All filters (sync first, then async)
const result = await filterRegistry.applyAll('hook.name', value, context);
```
  </TabItem>
</Tabs>

### Filter Methods

**Frontend:**

| Method | Description |
|--------|-------------|
| `register(hook, id, callback, order)` | Register a sync filter |
| `unregister(hook, id)` | Remove a filter |
| `apply(hook, value, context)` | Run all filters on a value |
| `hasFilters(hook)` | Check if any filters are registered |
| `getHooks()` | List all hook names |
| `getFilterCount(hook)` | Count filters for a hook |
| `getFilterIds(hook)` | List filter IDs for a hook |
| `subscribe(listener)` | Subscribe to registry changes; returns an unsubscribe function (React integration) |
| `getSnapshot()` | Get a snapshot of registered filters for `useSyncExternalStore` |

**Backend (additional methods):**

| Method | Description |
|--------|-------------|
| `registerAsync(hook, id, callback, order)` | Register an async filter |
| `unregisterAsync(hook, id)` | Remove an async filter |
| `applyAsync(hook, value, context)` | Run async filters only |
| `applyAll(hook, value, context)` | Run sync then async filters |
| `hasSyncFilters(hook)` | Check if any sync filters are registered |
| `hasAsyncFilters(hook)` | Check if any async filters are registered |

### Hook Naming Convention

| Pattern | Examples |
|---------|----------|
| Frontend: `user.{property}` | `user.displayName`, `user.avatarUrl` |
| Frontend: `dashboard.{element}` | `dashboard.widgets` |
| Frontend: `theme.cssVariables` | CSS variable injection |
| Backend: `api.{feature}.{action}.response` | `api.user.profile.response` |
| Backend: `api.{feature}.{action}.request` | `api.user.update.request` |
| Backend: `{entity}.{direction}` (message pipelines) | `email.outbound` — transform an outgoing [email message](/starter-template/email-feature/#hooks) |

## Action Registry

Actions let modules run side effects at defined hook points. Unlike filters, actions don't return values — they execute independently.

### Registering an Action

<Tabs>
  <TabItem label="Frontend">
```typescript
import { actionRegistry } from '@/config/action.registry';

// Analytics tracking when user logs in
actionRegistry.addAction('user.loggedIn', 'analytics', (ctx) => {
  trackEvent('login', { userId: ctx.user.id });
});

// Show toast when settings are saved
actionRegistry.addAction('settings.saved', 'notifications', (ctx) => {
  showToast('Settings saved successfully');
});
```
  </TabItem>
  <TabItem label="Backend">
```typescript
import { actionRegistry } from '../../config/action.registry.js';

// Sync action (fire-and-forget)
actionRegistry.addAction('user.created', 'audit', (ctx) => {
  auditLog.record('USER_CREATED', { userId: ctx.userId });
});

// Async action (awaited when using doActionAll)
actionRegistry.addActionAsync('user.created', 'welcome-email', async (ctx) => {
  await sendWelcomeEmail(ctx.email);
});
```
  </TabItem>
</Tabs>

### Firing Actions

<Tabs>
  <TabItem label="Frontend">
```typescript
// Fire all sync actions
actionRegistry.doAction('user.loggedIn', { user, method: 'email' });

// The frontend registry also supports async actions
await actionRegistry.doActionAll('user.loggedIn', { user, method: 'email' });
```
  </TabItem>
  <TabItem label="Backend">
```typescript
// All actions (sync first, then async sequentially) — the right default
// for hooks modules subscribe to
await actionRegistry.doActionAll('user.created', { userId, email, role });

// Fire-and-forget from a sync context: sync handlers run inline, async
// handlers run in parallel in the background (errors caught internally)
void actionRegistry.doActionParallel('notifications.send', { items });

// Sync pool only — dev-warns if async handlers would be skipped
actionRegistry.doAction('cache.invalidated', { keys });

// Async pool only (sequential) — dev-warns if sync handlers would be skipped
await actionRegistry.doActionAsync('user.created', { userId, email });
```
  </TabItem>
</Tabs>

### Action Methods

| Method | Description |
|--------|-------------|
| `addAction(hook, id, callback, order)` | Register a sync action |
| `addActionAsync(hook, id, callback, order)` | Register an async action |
| `removeAction(hook, id)` | Remove an action |
| `removeAllActions(hook)` | Remove all actions for a hook; returns the number removed |
| `doAction(hook, context)` | Fire sync actions only — since v1.54.0, a once-per-hook **dev-mode warning** fires when async handlers would be skipped |
| `doActionAsync(hook, context)` | Fire async actions sequentially — dev-warns symmetrically when sync handlers would be skipped |
| `doActionAll(hook, context)` | Fire sync then async (never warns — it runs both pools) |
| `doActionParallel(hook, context)` | Fire sync inline, then async in parallel |
| `doActionWithResults(hook, context)` | Backend only — fire all actions and collect per-action success/error results |
| `hasActions(hook)` | Check if any actions are registered |
| `countActions(hook)` | Count actions for a hook |
| `getHooks()` | List all hook names |
| `getActionIds(hook)` | List action IDs for a hook |
| `getActionInfo(hook)` | Backend only — list `{ id, order, async }` for each action on a hook |
| `clear()` | Remove all registered actions (useful for testing) |

## The Typed Hook Catalogue (v1.57.0)

Hook names used to be magic strings you discovered by grepping docs. Both sides now ship a **catalogue module** — the single source of truth for every hook the template fires, with typed context payloads:

```typescript
// Backend
import { CoreHooks, type CoreHookContexts } from '../../config/hooks.catalog.js';

actionRegistry.addActionAsync(
  CoreHooks.userRoleUpdated,          // autocomplete, no typos
  'my-module',
  async (ctx: CoreHookContexts['user.roleUpdated']) => {
    await syncPermissions(ctx.userId, ctx.newRole);
  },
);

// Frontend
import { CoreHooks, type CoreHookContexts } from '@/config/hooks.catalog';
```

The string **values** are the contract (they never change outside the deprecation cycle); the constants are conveniences — plain string hook names keep working forever. Each entry's JSDoc names the dispatch semantics and the emitting call site.

**Dev-mode dispatch tracing** (v1.57.0): every dispatch logs one debug line — hook, dispatcher, and per-pool listener counts — so "did my hook fire, and did anyone listen?" is answerable from `LOG_LEVEL=debug` (backend) or the browser console's verbose level (frontend). Silent in production.

**Job lifecycle hooks** (v1.57.0): both job runners now dispatch `job.completed` and `job.failed` (`void doActionParallel`, context `{ jobName, jobId?, attempt, runner, tenantId?, error? }`). `job.failed` was declared in the jobs security manifest since v1.46 but never fired — subscribe to it for background-work failure alerts (e.g. from a notifications module).

## Available Backend Hooks

The starter template fires these action hooks in the user and admin services. All fire after the successful database operation.

### User Service Hooks

| Hook | Execution | Context |
|------|-----------|---------|
| `user.created` | `await doActionAll` | `{ userId, email, role, isFirstUser }` |
| `user.roleUpdated` | `await doActionAll` | `{ userId, previousRole, newRole }` |
| `user.profileUpdated` | `void doActionParallel` | `{ userId, fields }` |
| `user.preferencesUpdated` | `void doActionParallel` | `{ userId }` |
| `user.deleted` | `await doActionAll` | `{ userId, email, deletionRecord, report }` |

### Admin Service Hooks

| Hook | Execution | Context |
|------|-----------|---------|
| `admin.settingsUpdated` | `await doActionAll` | `{ settings }` |
| `admin.navigationUpdated` | `await doActionAll` | `{ overrides }` |
| `admin.toolbarUpdated` | `await doActionAll` | `{ overrides }` |
| `admin.styleOverridesUpdated` | `await doActionAll` | `{ overrides }` |
| `admin.dashboardOverridesUpdated` | `await doActionAll` | `{ overrides }` |
| `user.roleUpdated` | `await doActionAll` | `{ userId, previousRole, newRole, updatedBy }` |
| `user.deleted` | `await doActionAll` | `{ userId, deletedBy, report }` |

### Foundation Seam Hooks

| Hook | Kind / Execution | Context |
|------|------------------|---------|
| `email.outbound` | filter, `applyAll` before send | the outgoing `EmailMessage` — see [Email feature](/starter-template/email-feature/#hooks) |
| `email.sent` | action, `doActionParallel` after a successful send | `{ message, result }` |
| `app.shutdown` | action, `await doActionAll` on SIGTERM/SIGINT (bounded by `SHUTDOWN_TIMEOUT`) | `{ signal }` — drain buffered work while the DB is still connected; see [Jobs feature](/starter-template/jobs-feature/#the-appshutdown-seam) |
| `notification.created` | action, `doActionParallel` after the in-app row is written | `{ id, tenantId, userId, category }` — see [Notifications feature](/starter-template/notifications-feature/) |
| `error:unhandled` | action, `void doActionParallel` from the global error middleware | `{ statusCode, name, message, operational, method, path, ip, tenantId, stack }` — feeds the admin System console audit sink |
| `job.completed` / `job.failed` | action, `void doActionParallel` from both job runners (v1.57.0) | `{ jobName, jobId?, attempt, runner, tenantId?, error? }` — `job.failed` closes the declared-but-never-fired gap in the jobs security manifest |
| `security.dataAccessed` | action, `void doActionParallel` from `auditedDecrypt` (v1.64.0) | `{ userId, tenantId, resource, purpose?, crossTenant, success, envelopeVersion }` — every audited read of encrypted content, success or failure; `userId` is the **actor**, `tenantId` the data **owner**, and `crossTenant` (actor ≠ owner) is the flag the compliance trail exists to prove absent. Captured by the audit sink, indexed under the owner — see [Security feature](/starter-template/security-feature/#field-encryption) |

<Aside type="tip">
**Awaited vs fire-and-forget (since v1.54.0 every core hook runs BOTH pools):**

- **`await doActionAll`** hooks (`user.created`, `user.roleUpdated`, `user.deleted`, all five `admin.*Updated`) are awaited — the response waits for your async handler, giving read-after-write consistency with the client's post-mutation refetch. Keep these handlers fast: a slow one delays the admin response (handler *errors* are caught inside the dispatcher and never fail the request).
- **`void doActionParallel`** hooks (`user.profileUpdated`, `user.preferencesUpdated`, `email.sent`, `notification.created`, `error:unhandled`, `security.dataAccessed`, frontend `theme.changed`) are fire-and-forget — sync handlers still run inline, async handlers run in the background and won't delay the response, but in-flight work can be lost on process exit.
- Before v1.54.0, `doAction` emissions **silently dropped** `addActionAsync` handlers on ~10 core hooks — those handlers now fire. If yours was registered but never ran, verify it's idempotent under real traffic.
</Aside>

<Aside type="note">
The admin service fires the **same hook names** as the user service for shared operations (role updates, user deletion). Context fields like `updatedBy` and `deletedBy` distinguish the source.
</Aside>

### Frontend Hooks

The frontend action registry currently fires the `theme.changed` hook via `void doActionParallel` — async subscribers run fire-and-forget, sync subscribers inline (see [Theme Utilities](/starter-template/theme/)). Additional frontend hooks are planned.

## Order Convention

Both filter and action registries use the unified **0-100 ordering scale**:

| Range | Label | Use for |
|-------|-------|---------|
| 0-25 | Core | Validation, security checks |
| 26-50 | Feature | Main business logic |
| 51-75 | Secondary | Enrichment, formatting |
| 76-100 | Utility | Logging, analytics |

The default order is **50** if not specified. Lower numbers execute first.

## Example: Module Using Both

Here's a complete example of a module using both filters and actions:

```typescript
// modules/analytics/hooks.registration.ts
import { actionRegistry } from '../../config/action.registry.js';
import { filterRegistry } from '../../config/filter.registry.js';

// Track user creation (action — side effect)
actionRegistry.addActionAsync('user.created', 'analytics', async (ctx) => {
  await analytics.track('user_signed_up', {
    userId: ctx.userId,
    isFirstUser: ctx.isFirstUser,
  });
});

// Add analytics metadata to profile responses (filter — transforms data)
filterRegistry.register(
  'api.user.profile.response',
  'analytics',
  (profile) => ({
    ...profile,
    lastActiveAt: analyticsCache.getLastActive(profile.id),
  }),
  60  // Run after core filters
);
```
