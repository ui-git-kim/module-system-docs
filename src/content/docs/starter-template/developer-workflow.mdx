This page covers the workflow for developing the starter template itself — not for building apps or modules from it.

<Aside type="note">
  The starter template is a **template repository**, not a runnable app. It has no `node_modules` installed. You cannot run `npm run dev` or `npm run build` here. Changes are validated through code review and by creating test apps.
</Aside>

---

## What You Can Edit

| Path | Editable? | Notes |
|------|-----------|-------|
| `frontend/src/features/**` | Yes | Feature modules |
| `frontend/src/config/**` | Yes | Registries (bump version) |
| `frontend/src/lib/**` | Yes | Utilities (bump version) |
| `frontend/src/components/settings-blocks/**` | Yes | Shared settings UI (bump version) |
| `frontend/src/components/page-blocks/**` | Yes | Shared page UI (bump version) |
| `frontend/src/components/ui/**` | No | shadcn — regenerate, never edit |
| `frontend/src/components/layout/**` | Yes | Layout components (bump version) |
| `backend/src/**` | Yes | Backend code (bump version) |
| `scripts/**` | Yes | CLI scripts (bump version) |
| `docs/**` | Yes | Documentation (push submodule) |
| `CLAUDE.md`, `README.md` | Yes | No version bump needed |

## Making Changes

<Steps>
1. **Edit the code** in the appropriate directory
2. **Review your changes** — since you cannot run the app, carefully verify:
   - TypeScript types are correct
   - Imports use the right paths (`@/` prefix for frontend, `.js` for backend)
   - File headers are present on new files
3. **Bump the version** if you changed any infrastructure files:
   ```bash
   npm run template:version:patch    # Bug fixes
   npm run template:version:minor    # New features
   npm run template:version:major    # Breaking changes
   ```
4. **Update the changelog** for minor/major bumps:
   - Add entry to `docs/src/content/docs/starter-template/changelog.mdx`
5. **Push documentation** if you edited files in `docs/`:
   ```bash
   cd docs && git add . && git commit -m "Docs: description" && git push
   cd .. && git add docs && git commit -m "Update docs submodule"
   ```
6. **Commit your changes** to the main repository
</Steps>

## Version Bumping

The version bump scripts update the `VERSION` file at the project root. Downstream apps use this to detect available updates.

```bash
# Most common — bug fixes, small improvements
npm run template:version:patch

# New features, enhancements
npm run template:version:minor

# Breaking changes (rare)
npm run template:version:major
```

### When to Bump

**Bump for ANY code change** that git will merge into downstream apps:

| Changed | Bump? |
|---------|-------|
| `frontend/src/**/*.ts(x)` | Yes |
| `backend/src/**/*.ts` | Yes |
| `scripts/*` | Yes |
| Root scripts (`.ps1`, `.sh`, `.bat`) | Yes |
| `package.json` | Yes |
| `docs/*` | No (push submodule instead) |
| `CLAUDE.md`, `README.md` | No |

<Aside type="danger">
  Never skip version bumping. Missing bumps mean downstream apps won't know updates are available.
</Aside>

## Documentation Workflow

The `docs/` folder is a git submodule pointing to the shared documentation repository. Changes to docs must be pushed separately.

<Steps>
1. Edit files in `docs/src/content/docs/`
2. Commit and push the docs submodule:
   ```bash
   cd docs && git add . && git commit -m "Docs: description" && git push
   ```
3. Update the submodule reference in the main repo:
   ```bash
   cd .. && git add docs && git commit -m "Update docs submodule"
   ```
</Steps>

<Aside type="tip">
  Pushing the docs submodule triggers an automatic deploy to the live documentation site.
</Aside>

## Testing Changes

Since the starter template has no `node_modules`, you cannot run the app directly. To validate changes:

### 1. Code Review

Carefully review for:
- Correct TypeScript types
- Correct import paths
- Consistent patterns with existing code
- File headers on new files

### 2. Create a Test App

The most thorough validation is to create a test app from the template:

<Tabs>
  <TabItem label="Windows">
```powershell
# Create a test app from the current template state
.\create-app.ps1
```
  </TabItem>
  <TabItem label="macOS/Linux">
```bash
# Create a test app from the current template state
./create-app.sh
```
  </TabItem>
</Tabs>

Then run the test app to verify your changes work:

```bash
cd your-test-app
npm run dev
```

### 3. Check Type Errors

In the test app (not the template), run:

```bash
npm run type-check
```

## How Changes Reach Apps

When you commit changes to the starter template, downstream apps receive them through the update system:

```
1. You push changes to the starter template
2. You bump the version (VERSION file)
3. App developer runs update-template script
4. Script pulls from template's git remote
5. Changes merge into the app
6. App developer resolves any conflicts
```

See [Updating from Template](/starter-template/updating/) for details on the update mechanism.

## Key Principles

1. **Registry pattern** — Never modify core files for features. Use registries.
2. **Feature-based organisation** — Keep related code together in feature folders.
3. **Shared component libraries** — Use `@/components/settings-blocks` and `@/components/page-blocks` for consistent UI.
4. **Forward momentum** — No backwards compatibility layers. Clean breaks with major versions.
5. **Version tracking** — Always bump so downstream apps know about updates.
