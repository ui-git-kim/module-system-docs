Issues reported from downstream repos (apps built from the starter template, and modules that build on it) that need to be fixed **here** in the starter template. When an issue is fixed and propagated via `template:update`, move it from **Open Issues** to **Completed**.

## How Upstream Issues Are Tracked

When Claude Code (or a developer) discovers a bug or limitation while working in an **app** or **module** that actually belongs in an **upstream** repo, it logs the issue to a dedicated page in the docs submodule — **one page per upstream repo**:

| Issue belongs in… | Log it to… | Visible at… |
|-------------------|-----------|-------------|
| **Starter Template** | `docs/src/content/docs/starter-template/upstream-issues.mdx` | [/starter-template/upstream-issues/](/starter-template/upstream-issues/) |
| **Module Starter Template** | `docs/src/content/docs/module-starter/upstream-issues.mdx` | [/module-starter/upstream-issues/](/module-starter/upstream-issues/) |

Because the `docs/` folder is a **shared submodule** across all repos, issues reported from *any* app or module are visible in one place per upstream repo. When working on the starter template, check this page first; when working on a downstream app or module, log starter-template bugs here so they get fixed at the source and propagate to every consumer on the next `template:update`.

<Aside type="tip">
This page is the canonical log of downstream-surfaced issues that belong in the **starter template**. The [Roadmap](/starter-template/roadmap/) tracks the starter's own audit findings and planned work — the two are complementary. Use **this page** for "a downstream consumer hit this and it should be fixed upstream," and the roadmap for the starter's own backlog.
</Aside>

**Entry format:**

```text
Open:      **[YYYY-MM-DD] [app/module name]** — Description. Manifests in `path/to/file`. Severity: High/Medium/Low.
Completed: **[YYYY-MM-DD, vX.X.X] [app/module name]** — Description — fixed in vX.X.X, propagated via template:update.
```

- **[2026-06-26, v1.54.0] structure** — The shared TanStack Query `QueryClient` uses the default retry policy, which **retries on `429` (Too Many Requests)**. Retrying a rate-limited request spends more of the budget the app-layer `express-rate-limit` floor just told clients to back off from, so a brief limit snowballs into a 429 storm across *every* module's bootstrap queries (branding, style, user, settings, structure nav-prime, …). **Fix:** set a `retry` predicate on the app's `QueryClient` that returns `false` when `error.response?.status === 429` (arguably any 4xx), keeping retries for `5xx`/network. One change fixes it for all modules at once; per-module retry overrides are only a workaround. Surfaced while testing the structure public-API release — the module-side auto-prime retry was fixed in **structure v1.15.1**, but the hook-query retry is app-wide and belongs here. Manifests in the shared QueryClient setup (wherever `new QueryClient(...)` is configured). Severity: Medium. — **Fixed in v1.54.0** (retry predicate: 4xx never retries, network/5xx/408 retry once), propagated via `template:update`.

- **[2026-07-14, v1.54.0] document-management** — `@types/express-serve-static-core` **5.1.x** redefined `ParamsDictionary` param values from `string` to **`string | string[]`** (`[key: string]: string | string[]`), so `const { userId } = req.params` is now typed `string | string[]` and no longer assignable to a `string` parameter. This fails `npm run type-check` on a **clean install of the starter itself** — e.g. `backend/src/features/admin/admin.routes.ts:210` (`adminService.updateUserRole(userId, role)`) and the sibling `*-admin.routes.ts` handlers — and cascades to **every module's** route handlers/controllers that destructure `req.params` (surfaced in the `document-management` controller during its v1.2.2 alignment E2E on the `veradai` app; the starter's lockfile pins `@types/express-serve-static-core` 5.1.0, veradai resolved 5.1.1 — both carry the change). Runtime is unaffected (non-wildcard params are strings at runtime); this is a type-check/CI breakage only. **Fix (one place, fixes all consumers):** pin `@types/express-serve-static-core` `<5.1.0` in the starter backend, **or** coerce at the edge via a shared helper (`getParam(req, 'userId'): string`, or `String(req.params.userId)`) rather than every handler and module casting individually. Manifests in `backend/src/features/admin/admin.routes.ts` and all core `req.params` handlers. Severity: Medium. — **Fixed in v1.54.0** (`overrides` pin `~5.0.0` in `backend/package.json`; `sync-deps.mjs` gains a presence-only foundation-overrides leg so existing apps receive it), propagated via `template:update`.

- **[2026-07-14, v1.54.0] document-management** — **The action registry's dispatch API is a footgun and needs an audit + cleanup.** `actionRegistry` exposes five overlapping dispatchers — `doAction` (sync only), `doActionAsync`, `doActionAll`, `doActionParallel`, `doActionWithResults` — and the sync `doAction` **silently ignores** every handler registered with `addActionAsync` (`config/action.registry.ts:218-231` runs only `if (!action.async)`, and its own JSDoc says *"Async actions are ignored"*). There is no dev-mode warning, so a module that does `addActionAsync('user.roleUpdated', …)` simply never fires. This is systemic: ~10 core call sites emit via the sync `doAction` — `user.roleUpdated` (`user.service.ts:216`, `admin.service.ts:560`) and `admin.settingsUpdated` / `navigationUpdated` / `toolbarUpdated` / `styleOverridesUpdated` / `dashboardOverridesUpdated` (`admin.service.ts:270-480`) — so async consumers of any of those hooks are dropped today with no error. document-management hit the same bug on its own `documents.document.uploaded` event (fixed in v1.2.3 by switching to `doActionParallel`). **Audit + cleanup:** (1) sweep every core `doAction` call site and switch hooks meant to support module consumers to `doActionAll` / `doActionParallel`; (2) make `doAction` emit a dev-mode warning when the target hook has async handlers registered, so the silent drop is visible; (3) consider consolidating the five dispatchers to a smaller, clearly-named set (one async fan-out as the default, a sync variant only for perf-critical hot paths) and document ordering + error-isolation semantics consistently. Manifests in `backend/src/config/action.registry.ts` and the core `doAction` call sites in `admin.service.ts` / `user.service.ts`. Severity: Medium. — **Fixed in v1.54.0** (core sites sweep to `doActionAll`/`doActionParallel`; once-per-hook dev warnings in both registries; dispatcher consolidation deferred to the [roadmap](/starter-template/roadmap/)), propagated via `template:update`.

- **[2026-07-14, v1.54.0] document-management** — **The `/admin/modules` migration-status read logs a scary `prisma:error` on any app where no module has run a `migrate-db` migration.** `readAppliedMigrations()` (`backend/src/features/modules/modules.service.ts:395`) runs `SELECT … FROM _module_migrations` via `prisma.$queryRawUnsafe` (`:398-401`). That bookkeeping table is created lazily by the scaffold migration runner on a module's **first** `migrate-db` migration; until then it does not exist, so the query fails with Postgres `42P01` (`relation "_module_migrations" does not exist`). The app HANDLES this correctly — the catch (`:424-438`) treats the missing table as routine (`log.debug`, returns null) and the admin UI shows *"migration-tracking table could not be read … shown as pending"* — **but Prisma's client logs the raw query failure at `error` level before the app's catch runs**, so every boot / `/admin/modules` load emits a red `prisma:error 42P01`. Cosmetic (nothing is broken) but alarming, and it hits every deployment with the modules feature until some module first ships a `migrate-db` migration — the common case, since additive Prisma-only schema changes never create the table. Surfaced on a fresh Neon app running document-management v1.2.4 (which ships no `migrate-db` migrations). **Fix:** probe existence before the read so Prisma never errors — e.g. `SELECT to_regclass('_module_migrations')` (returns NULL without raising) and skip the SELECT when null; or run `MODULE_MIGRATIONS_DDL` (`CREATE TABLE IF NOT EXISTS`, already defined in `module-contract.ts`) once at modules-feature startup so the read always succeeds. Manifests in `backend/src/features/modules/modules.service.ts` (`readAppliedMigrations`). Severity: Low. — **Fixed in v1.54.0** (`to_regclass()` existence probe before the SELECT — Prisma never sees the failing query), propagated via `template:update`.

- **[2026-07-14, v1.54.0] billing** — `template:update` doesn't propagate new **sub-package (`backend`/`frontend`) `package.json` scripts** to existing apps, so `npm run type-check` breaks after an update. The root `type-check:frontend`/`type-check:backend` run `cd frontend|backend && npm run type-check`, but the sub-package `type-check` scripts (`backend`: `tsc --noEmit`, `frontend`: `tsc -b --noEmit`) exist only in freshly-created apps: on update the sub-package `package.json`s are kept-local on conflict (merge log: `backend/package.json (kept local)`), and `sync-package-scripts.mjs` reconciles **only the root `package.json`** (its `FOUNDATION_SCRIPTS` set is root-scoped and omits `type-check`). Manifests in `scripts/sync-package-scripts.mjs` (root-only scope) and `backend/package.json` / `frontend/package.json`. Severity: Low (type-check/CI convenience; runtime unaffected — workaround: `cd backend && npx tsc --noEmit`, `cd frontend && npx tsc -b`). - **[2026-07-14, v1.54.0] billing** — **No way to verify inbound third-party webhook signatures (e.g. Stripe HMAC-SHA256) — a spoofable, unauthenticated webhook.** The global `express.json({ limit: '10mb' })` at `backend/src/app.ts:160` has no `verify` callback and runs before `routeRegistry.mountAll` (`app.ts:244`), so the raw request bytes are parsed and discarded before any handler runs. `PublicRouteConfig` (`backend/src/config/route.registry.ts:56-65`) exposes no `middleware`/`rawBody` hook and `registerPublic` mounts the handler directly (`route.registry.ts:182`), so a public webhook route can't attach `express.raw` ahead of the JSON parser (and the auth-gated `RouteConfig.middleware` stack mounts after the global parser anyway). There is no HMAC primitive: `features/security/crypto.ts` imports `createHash`+`timingSafeEqual` but not `createHmac`, and `hashSecret`/`verifySecret` are plain SHA-256 token hashing (no keyed MAC, no raw payload) — they verify "you hold a token we stored", not "this payload was signed by the provider". Only an unauthenticated POST is supported (documented at `starter-template/auth-feature.mdx` with an empty handler — the "webhook support" that exists is a public-route pattern, not verification). With no raw body, a correct handler cannot verify the provider signature, so the starter forces a bad choice: either **reject every event** (the billing module's choice — safe, but the webhook can't sync subscription state) or **skip verification** — which leaves the public endpoint **spoofable**, letting any caller forge subscription/entitlement changes. **Fix (one place, benefits every integration module):** add `verify: (req, _res, buf) => { req.rawBody = buf; }` to the global `express.json` in `app.ts` and type `rawBody?: Buffer` on the Express `Request`; OR add a `rawBody: true` flag to `PublicRouteConfig` that mounts `express.raw` for that path before the global JSON parser. Billing is already forward-compatible (its handler reads `req.rawBody` with a `req.body` fallback), so either host fix enables verification with no module change. Manifests in `backend/src/app.ts` and `backend/src/config/route.registry.ts`. Severity: High. — **Fixed in v1.54.0** (`req.rawBody` via `express.json` verify hook; `computeHmac`/`timingSafeEqualStrings` + `verifyWebhookSignature`/`verifyTimestampedWebhookSignature` in the security feature; `allowNoOrigin` route flag consulted by the production CORS gate — the fix also uncovered that production CORS rejected *all* no-Origin server-to-server requests, so webhooks were undeliverable in production regardless of raw body; health endpoints opted in too). *Verification* needs no billing-module change (its handler already reads `req.rawBody`); production *delivery* needs one line in billing's webhook registration: `allowNoOrigin: true`. Propagated via `template:update`.
