Issues reported from downstream repos (apps built from the starter template, and modules that build on it) that need to be fixed **here** in the starter template. When an issue is fixed and propagated via `template:update`, move it from **Open Issues** to **Completed**.

## How Upstream Issues Are Tracked

When Claude Code (or a developer) discovers a bug or limitation while working in an **app** or **module** that actually belongs in an **upstream** repo, it logs the issue to a dedicated page in the docs submodule — **one page per upstream repo**:

| Issue belongs in… | Log it to… | Visible at… |
|-------------------|-----------|-------------|
| **Starter Template** | `docs/src/content/docs/starter-template/upstream-issues.mdx` | [/starter-template/upstream-issues/](/starter-template/upstream-issues/) |
| **Module Starter Template** | `docs/src/content/docs/module-starter/upstream-issues.mdx` | [/module-starter/upstream-issues/](/module-starter/upstream-issues/) |

Because the `docs/` folder is a **shared submodule** across all repos, issues reported from *any* app or module are visible in one place per upstream repo. When working on the starter template, check this page first; when working on a downstream app or module, log starter-template bugs here so they get fixed at the source and propagate to every consumer on the next `template:update`.

<Aside type="tip">
This page is the canonical log of downstream-surfaced issues that belong in the **starter template**. The [Roadmap](/starter-template/roadmap/) tracks the starter's own audit findings and planned work — the two are complementary. Use **this page** for "a downstream consumer hit this and it should be fixed upstream," and the roadmap for the starter's own backlog.
</Aside>

**Entry format:**

```text
Open:      **[YYYY-MM-DD] [app/module name]** — Description. Manifests in `path/to/file`. Severity: High/Medium/Low.
Completed: **[YYYY-MM-DD, vX.X.X] [app/module name]** — Description — fixed in vX.X.X, propagated via template:update.
```

## Open Issues

- **[2026-06-26] structure** — The shared TanStack Query `QueryClient` uses the default retry policy, which **retries on `429` (Too Many Requests)**. Retrying a rate-limited request spends more of the budget the app-layer `express-rate-limit` floor just told clients to back off from, so a brief limit snowballs into a 429 storm across *every* module's bootstrap queries (branding, style, user, settings, structure nav-prime, …). **Fix:** set a `retry` predicate on the app's `QueryClient` that returns `false` when `error.response?.status === 429` (arguably any 4xx), keeping retries for `5xx`/network. One change fixes it for all modules at once; per-module retry overrides are only a workaround. Surfaced while testing the structure public-API release — the module-side auto-prime retry was fixed in **structure v1.15.1**, but the hook-query retry is app-wide and belongs here. Manifests in the shared QueryClient setup (wherever `new QueryClient(...)` is configured). Severity: Medium.

- **[2026-07-14] document-management** — `@types/express-serve-static-core` **5.1.x** redefined `ParamsDictionary` param values from `string` to **`string | string[]`** (`[key: string]: string | string[]`), so `const { userId } = req.params` is now typed `string | string[]` and no longer assignable to a `string` parameter. This fails `npm run type-check` on a **clean install of the starter itself** — e.g. `backend/src/features/admin/admin.routes.ts:210` (`adminService.updateUserRole(userId, role)`) and the sibling `*-admin.routes.ts` handlers — and cascades to **every module's** route handlers/controllers that destructure `req.params` (surfaced in the `document-management` controller during its v1.2.2 alignment E2E on the `veradai` app; the starter's lockfile pins `@types/express-serve-static-core` 5.1.0, veradai resolved 5.1.1 — both carry the change). Runtime is unaffected (non-wildcard params are strings at runtime); this is a type-check/CI breakage only. **Fix (one place, fixes all consumers):** pin `@types/express-serve-static-core` `<5.1.0` in the starter backend, **or** coerce at the edge via a shared helper (`getParam(req, 'userId'): string`, or `String(req.params.userId)`) rather than every handler and module casting individually. Manifests in `backend/src/features/admin/admin.routes.ts` and all core `req.params` handlers. Severity: Medium.

- **[2026-07-14] document-management** — **The action registry's dispatch API is a footgun and needs an audit + cleanup.** `actionRegistry` exposes five overlapping dispatchers — `doAction` (sync only), `doActionAsync`, `doActionAll`, `doActionParallel`, `doActionWithResults` — and the sync `doAction` **silently ignores** every handler registered with `addActionAsync` (`config/action.registry.ts:218-231` runs only `if (!action.async)`, and its own JSDoc says *"Async actions are ignored"*). There is no dev-mode warning, so a module that does `addActionAsync('user.roleUpdated', …)` simply never fires. This is systemic: ~10 core call sites emit via the sync `doAction` — `user.roleUpdated` (`user.service.ts:216`, `admin.service.ts:560`) and `admin.settingsUpdated` / `navigationUpdated` / `toolbarUpdated` / `styleOverridesUpdated` / `dashboardOverridesUpdated` (`admin.service.ts:270-480`) — so async consumers of any of those hooks are dropped today with no error. document-management hit the same bug on its own `documents.document.uploaded` event (fixed in v1.2.3 by switching to `doActionParallel`). **Audit + cleanup:** (1) sweep every core `doAction` call site and switch hooks meant to support module consumers to `doActionAll` / `doActionParallel`; (2) make `doAction` emit a dev-mode warning when the target hook has async handlers registered, so the silent drop is visible; (3) consider consolidating the five dispatchers to a smaller, clearly-named set (one async fan-out as the default, a sync variant only for perf-critical hot paths) and document ordering + error-isolation semantics consistently. Manifests in `backend/src/config/action.registry.ts` and the core `doAction` call sites in `admin.service.ts` / `user.service.ts`. Severity: Medium.

<Aside type="note">
Third-party / provider blockers — e.g. the `better-auth` version pinned to Neon's auth server, or the forgot/reset-password flow pending Neon's Better Auth 1.6 upgrade — are **not** logged here, because the starter template cannot fix them directly. Those live in the roadmap under [Known Security Issues](/starter-template/roadmap/#known-security-issues). This page is for issues the starter template *can* fix.
</Aside>
