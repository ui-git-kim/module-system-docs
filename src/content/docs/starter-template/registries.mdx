The starter template provides a comprehensive set of registries for extending your application. This page documents every registry, its API, and usage patterns.

---

## Registry Types

There are two types of registries:

| Type | Purpose | Examples |
|------|---------|----------|
| **Collection Registries** | Store items (extend BaseRegistry) | Navigation, Dashboard, Settings Pages |
| **Pipeline Registries** | Transform data or execute side effects | Filter, Action |

Collection registries extend `BaseRegistry<T>` and share a consistent API. Pipeline registries have specialised APIs for their use cases.

## Frontend Registries

### Navigation Registry

Register items for the sidebar and user menu. The navigation registry supports **sidebar sections** for grouping main navigation items visually.

#### Basic Registration

```typescript
import { navigationRegistry } from '@/config/navigation.registry';
import { Folder, HelpCircle, Settings } from 'lucide-react';

// Main sidebar item (goes into 'main' section by default)
navigationRegistry.registerMain({
  id: 'projects',
  title: 'Projects',
  url: '/projects',
  icon: Folder,
  order: 20,
  requiredRole: 'user',  // Optional: restrict visibility by role
});

// Secondary sidebar (bottom section, not grouped by sections)
navigationRegistry.registerSecondary({
  id: 'help',
  title: 'Help',
  url: '/help',
  icon: HelpCircle,
  order: 10,
});

// User menu item (dropdown when clicking avatar)
navigationRegistry.registerUserMenu({
  id: 'settings',
  title: 'Settings',
  url: '/settings/profile',
  icon: Settings,
  order: 10,
});
```

#### Sidebar Sections

Sections are visual groupings for main navigation items. Modules can register their own sections and assign items to them.

```typescript
import { navigationRegistry } from '@/config/navigation.registry';
import { Folder, FileText, BarChart } from 'lucide-react';

// Register a section (optional - sections auto-create when referenced)
navigationRegistry.registerSection({
  id: 'workspace',
  label: 'Workspace',
  showLabel: true,  // Show "Workspace" header in sidebar
  order: 20,        // Section order (0-100)
});

// Register items into the section
navigationRegistry.registerMain({
  id: 'projects',
  title: 'Projects',
  url: '/projects',
  icon: Folder,
  order: 10,
  sidebarSection: 'workspace',  // Assign to 'workspace' section
});

navigationRegistry.registerMain({
  id: 'documents',
  title: 'Documents',
  url: '/documents',
  icon: FileText,
  order: 20,
  sidebarSection: 'workspace',  // Same section
});

// Items without sidebarSection go into 'main' section
navigationRegistry.registerMain({
  id: 'analytics',
  title: 'Analytics',
  url: '/analytics',
  icon: BarChart,
  order: 30,
  // No sidebarSection = defaults to 'main'
});
```

<Aside type="tip">
**Auto-creation:** If you assign an item to a section that doesn't exist, the registry auto-creates it with a default label (capitalised section ID) and `showLabel: true`.
</Aside>

#### Section Properties

| Property | Type | Description |
|----------|------|-------------|
| `id` | `string` | Unique identifier (lowercase, no spaces) |
| `label` | `string` | Display text shown in sidebar (e.g., "Workspace") |
| `showLabel` | `boolean` | Whether to show the section header |
| `order` | `number` | Sort position (0-100, lower = higher in sidebar) |

#### Order Convention

The template uses a **0-100 scale** for ordering items and sections:

| Range | Use For | Examples |
|-------|---------|----------|
| **0-25** | Primary features | Dashboard, main app features |
| **26-50** | Core features | Projects, documents, analytics |
| **51-75** | Secondary features | Settings, reports |
| **76-100** | Utilities | Help, admin, support |

<Aside type="note">
Default order is `50` if not specified. Secondary navigation uses the same scale but is displayed separately at the bottom of the sidebar.
</Aside>

#### How Sections Work at Runtime

1. **Modules register sections** via `registerSection()` or implicitly via `sidebarSection` on items
2. **Items specify their section** via `sidebarSection` property (defaults to `'main'`)
3. **Admin can override** section labels, visibility, and order via the admin panel
4. **At render time**, `NavMain` merges registry sections with admin overrides

```
Registry (in-memory)          +    Admin Overrides (database)
--------------------------         --------------------------
Section: workspace                 workspace:
  label: "Workspace"                 label: "My Workspace"  ← Admin changed label
  showLabel: true                    order: 5               ← Admin moved it up
  order: 20

Section: main                      (no overrides)
  label: "Application"
  showLabel: true
  order: 0
                                   ═════════════════════════
                                            ↓
                                   Merged at render time
                                            ↓
                                   Section: workspace
                                     label: "My Workspace"  ← Override wins
                                     showLabel: true        ← Registry default
                                     order: 5               ← Override wins
```

#### Feature Registration Pattern

The recommended pattern for modules:

```typescript
// features/workspace/navigation.registration.ts
import { navigationRegistry } from '@/config/navigation.registry';
import { Folder, FileText, Users } from 'lucide-react';

// 1. Register your section
navigationRegistry.registerSection({
  id: 'workspace',
  label: 'Workspace',
  showLabel: true,
  order: 20,
});

// 2. Register items into your section
navigationRegistry.registerMain({
  id: 'workspace-projects',
  title: 'Projects',
  url: '/workspace/projects',
  icon: Folder,
  order: 10,
  sidebarSection: 'workspace',
});

navigationRegistry.registerMain({
  id: 'workspace-documents',
  title: 'Documents',
  url: '/workspace/documents',
  icon: FileText,
  order: 20,
  sidebarSection: 'workspace',
});

navigationRegistry.registerMain({
  id: 'workspace-team',
  title: 'Team',
  url: '/workspace/team',
  icon: Users,
  order: 30,
  sidebarSection: 'workspace',
});
```

#### Item Interface

```typescript
interface NavigationItem {
  id: string;
  title: string;
  url: string;
  icon: LucideIcon;
  order?: number;              // Default: 50
  sidebarSection?: string;     // Default: 'main' (main nav only)
  requiredRole?: 'user' | 'admin' | 'managed' | UserRole[];
  badge?: string | number;     // Optional badge text/count
  external?: boolean;          // Opens in new tab
  items?: SubNavItem[];        // Collapsible sub-items
}

interface SidebarSection {
  id: string;
  label: string;
  showLabel: boolean;
  order?: number;  // Default: 50
}
```

#### Methods

| Method | Description |
|--------|-------------|
| `registerMain(item)` | Add to main sidebar |
| `registerSecondary(item)` | Add to bottom sidebar |
| `registerUserMenu(item)` | Add to user dropdown |
| `registerSection(section)` | Register a sidebar section |
| `registerFeature(config)` | Register all at once |
| `getMain()` | Get main nav items (sorted by order) |
| `getSecondary()` | Get secondary nav items (sorted by order) |
| `getUserMenu()` | Get user menu items |
| `getSections()` | Get all registered sections |
| `getMainGroupedBySection()` | Get main items grouped by section |
| `getGrouped()` | Get all items grouped by type |

#### Hooks

| Hook | Returns |
|------|---------|
| `useNavigationRegistry()` | All items |
| `useNavigationGrouped()` | Grouped by type (main/secondary/user) |
| `useMainNavigation()` | Main nav items only |
| `useMainNavigationGrouped()` | Main items grouped by section |
| `useSecondaryNavigation()` | Secondary nav items only |
| `useUserMenuNavigation()` | User menu items only |
| `useRegistrySections()` | All registered sections |

### Dashboard Registry

Register widgets for the dashboard.

```typescript
import { dashboardRegistry } from '@/config/dashboard.registry';
import { AnalyticsWidget } from './components/AnalyticsWidget';

dashboardRegistry.register({
  id: 'analytics-overview',
  title: 'Analytics Overview',
  component: AnalyticsWidget,
  size: '2x1',
  order: 10,
  category: 'analytics',
  description: 'Key metrics at a glance',
  canHide: true,    // User can hide (default: true)
  canResize: false, // User can resize (default: false)
});
```

**Widget Sizes:**

| Size | Columns | Rows | Use For |
|------|---------|------|---------|
| `1x1` | 1 | 1 | Small stats, quick actions |
| `2x1` | 2 | 1 | Charts, summaries |
| `1x2` | 1 | 2 | Lists, feeds |
| `2x2` | 2 | 2 | Complex visualisations |
| `full` | 4 | 1 | Full-width content |

**Methods:**
| Method | Description |
|--------|-------------|
| `register(widget)` | Register a widget |
| `getByCategory(category)` | Filter by category |
| `getCategories()` | Get unique categories |
| `getVisible(userVisibility)` | Get visible widgets respecting user preferences |

**Hooks:**
| Hook | Returns |
|------|---------|
| `useDashboardRegistry()` | All widgets |
| `useVisibleDashboardWidgets(userVisibility)` | Visible widgets only |

### Settings Pages Registry

Register pages in the settings section.

```typescript
import { settingsPagesRegistry } from '@/config/settings-pages.registry';
import { TeamSettingsPage } from './pages/TeamSettingsPage';
import { Users, CreditCard } from 'lucide-react';

// Add a new settings page
settingsPagesRegistry.register({
  id: 'team',               // URL: /settings/team

  component: StripeBillingPage,

```

<Aside type="note">
**Replaceable Pages:** `billing` (placeholder exists for modules to replace)

**Non-Extensible Pages:** `security`, `data` (for security/compliance reasons)
</Aside>

**Hooks:**
| Hook | Returns |
|------|---------|
| `useSettingsPagesRegistry()` | All pages |
| `useSettingsNavigation()` | Pages formatted for nav |

### Header Toolbar Registry

Add items to the header toolbar.

```typescript
import { headerToolbarRegistry } from '@/config/header-toolbar.registry';
import { Search, Bell, Moon, Sun } from 'lucide-react';

// Action button
headerToolbarRegistry.register({
  id: 'global-search',
  type: 'action',
  position: 'left',
  order: 10,
  icon: Search,
  label: 'Search (Cmd+K)',
  onClick: () => openSearchModal(),
});

// Toggle button
headerToolbarRegistry.register({
  id: 'theme-toggle',
  type: 'toggle',
  position: 'right',
  order: 100,
  icon: Moon,
  activeIcon: Sun,
  label: 'Toggle theme',
  isActive: isDarkMode,
  onClick: () => toggleTheme(),
});

// Dropdown with badge
headerToolbarRegistry.register({
  id: 'notifications',
  type: 'dropdown',
  position: 'right',
  order: 50,
  icon: Bell,
  label: 'Notifications',
  badge: 3,
  component: NotificationDropdown,
});

// Custom component (full control)
headerToolbarRegistry.register({
  id: 'custom-widget',
  type: 'custom',
  position: 'right',
  order: 80,
  component: MyCustomToolbarWidget,
});
```

**Item Types:**

| Type | Description | Required Props |
|------|-------------|----------------|
| `action` | Button with onClick | `onClick`, `icon`, `label` |
| `toggle` | Toggle button with state | `onClick`, `icon`, `isActive`, `label` |
| `dropdown` | Opens dropdown menu | `icon`, `component`, `label` |
| `custom` | Fully custom component | `component` |

**Order Convention:**

| Range | Use For |
|-------|---------|
| 1-20 | Primary actions (search, new item) |
| 21-40 | Feature-specific actions |
| 41-60 | Notifications, status |
| 61-80 | Secondary utilities |
| 81-99 | Settings, help |
| 100+ | Theme toggle, far-right items |

**Hooks:**
| Hook | Returns |
|------|---------|
| `useHeaderToolbarRegistry()` | All items |
| `useHeaderToolbarGrouped()` | Grouped by position |

## Backend Registries

### Route Registry

Register API routes with versioning.

```typescript
import { routeRegistry } from '@/config/route.registry';
import { billingRoutes } from './billing.routes';

routeRegistry.register({
  path: '/billing',
  router: billingRoutes,
  feature: 'billing',
  versions: ['v1'],           // Mounted at /api/v1/billing
  requiresAuth: true,         // Apply auth middleware
  middleware: [customMiddleware], // Additional middleware
  description: 'Billing API',
  deprecated: false,          // Mark as deprecated if needed
});

// Public route (no auth required)
routeRegistry.registerPublic({
  path: '/api/health',
  method: 'get',
  handler: healthCheckHandler,
  description: 'Health check endpoint',
});
```

**Route Configuration:**

```typescript
interface RouteConfig {
  path: string;              // Base path (e.g., '/billing')
  router: Router;            // Express router
  feature: string;           // Feature name for grouping
  versions: string[];        // API versions ['v1', 'v2']
  requiresAuth?: boolean;    // Apply auth middleware (default: true)
  middleware?: Middleware[]; // Additional middleware
  description?: string;      // Documentation
  deprecated?: boolean;      // Mark as deprecated
}
```

**Methods:**
| Method | Description |
|--------|-------------|
| `register(config)` | Register feature routes |
| `registerPublic(config)` | Register public (no auth) routes |
| `mountAll(app, authMiddleware)` | Mount all routes to Express app |
| `getByFeature(feature)` | Get routes by feature name |
| `getByVersion(version)` | Get routes by API version |
| `generateDocs()` | Generate route documentation |

### Service Registry

Register services for cross-feature communication.

```typescript
import { serviceRegistry } from '@/config/service.registry';
import { userService } from './user.service';

// Register with type augmentation for type safety
serviceRegistry.register('user', userService, {
  feature: 'user',
  description: 'User profile management',
});

// Access from another feature
const userService = serviceRegistry.get('user');
const profile = await userService.getProfile(userId);
```

**Type Augmentation:**

For TypeScript type safety, augment the `ServiceTypes` interface:

```typescript
// In your feature's types file
declare module '@/config/service.registry' {
  interface ServiceTypes {
    'user': UserService;
    'billing': BillingService;
  }
}
```

This provides autocomplete and type checking when using `serviceRegistry.get()`.

## Filter Registry

Transform data at hook points. Filters run in a pipeline where each filter receives the output of the previous one.

<Tabs>
  <TabItem label="Frontend">
```typescript
import { filterRegistry } from '@/config/filter.registry';

// Transform user display name
filterRegistry.register(
  'user.displayName',        // Hook name
  'titles-module',           // Unique ID
  (name, ctx) => {           // Transform function
    return ctx.user.title ? `${ctx.user.title} ${name}` : name;
  },
  5  // Order (lower = earlier)
);

// Apply filters
const displayName = filterRegistry.apply(
  'user.displayName',
  user.name,
  { user }
);
```
  </TabItem>
  <TabItem label="Backend">
```typescript
import { filterRegistry } from '@/config/filter.registry';

// Synchronous filter
filterRegistry.register(
  'api.user.profile.response',
  'privacy-module',
  (profile) => ({ ...profile, email: maskEmail(profile.email) }),
  20  // Run late
);

// Async filter (for DB lookups, external calls)
filterRegistry.registerAsync(
  'api.user.profile.response',
  'enrichment-module',
  async (profile, ctx) => {
    const stats = await statsService.getUserStats(profile.id);
    return { ...profile, stats };
  }
);

// Apply all filters (sync + async)
const response = await filterRegistry.applyAll(
  'api.user.profile.response',
  profileData,
  { req }
);
```
  </TabItem>
</Tabs>

**Hook Naming Convention:**

| Pattern | Examples |
|---------|----------|
| Frontend: `user.{property}` | `user.displayName`, `user.avatarUrl` |
| Frontend: `dashboard.{element}` | `dashboard.widgets`, `dashboard.layout` |
| Frontend: `nav.{section}.items` | `nav.main.items`, `nav.user.items` |
| Backend: `api.{feature}.{action}.response` | `api.user.profile.response` |
| Backend: `api.{feature}.{action}.request` | `api.user.update.request` |
| Backend: `service.{feature}.{method}` | `service.user.getProfile` |

**Methods:**

| Method | Description |
|--------|-------------|
| `register(hook, id, callback, order)` | Register sync filter |
| `registerAsync(hook, id, callback, order)` | Register async filter (backend only) |
| `unregister(hook, id)` | Remove a filter |
| `apply(hook, value, context)` | Apply sync filters |
| `applyAsync(hook, value, context)` | Apply async filters (backend only) |
| `applyAll(hook, value, context)` | Apply all filters (backend only) |
| `hasFilters(hook)` | Check if filters exist |
| `getHooks()` | Get all hook names |

## Action Registry

Execute side effects at hook points. Unlike filters, actions don't transform data.

<Tabs>
  <TabItem label="Frontend">
```typescript
import { actionRegistry } from '@/config/action.registry';

// Analytics tracking
actionRegistry.addAction('user.loggedIn', 'analytics', (ctx) => {
  trackEvent('login', { userId: ctx.user.id });
});

// Show notification
actionRegistry.addAction('settings.saved', 'notifications', (ctx) => {
  showToast('Settings saved successfully');
});

// Execute all actions for a hook
actionRegistry.doAction('user.loggedIn', { user, method: 'email' });
```
  </TabItem>
  <TabItem label="Backend">
```typescript
import { actionRegistry } from '@/config/action.registry';

// Sync action
actionRegistry.addAction('user.created', 'audit', (ctx) => {
  auditLog.record('USER_CREATED', { userId: ctx.user.id });
});

// Async action
actionRegistry.addActionAsync('order.completed', 'email', async (ctx) => {
  await sendOrderConfirmationEmail(ctx.order, ctx.user);
});

// Execute all actions (sync first, then async)
await actionRegistry.doActionAll('user.created', { user, req });

// Execute async actions in parallel (for independent operations)
await actionRegistry.doActionParallel('notifications.send', { notifications });
```
  </TabItem>
</Tabs>

**Hook Naming Convention:**

| Pattern | Examples |
|---------|----------|
| `{entity}.{event}` | `user.loggedIn`, `user.loggedOut`, `user.created` |
| `{feature}.{action}` | `settings.saved`, `theme.changed` |
| `{component}.{lifecycle}` | `dashboard.mounted`, `modal.opened` |
| `before:{action}` | `before:save`, `before:delete` |
| `after:{action}` | `after:save`, `after:delete` |

**Methods:**

| Method | Description |
|--------|-------------|
| `addAction(hook, id, callback, order)` | Register sync action |
| `addActionAsync(hook, id, callback, order)` | Register async action |
| `removeAction(hook, id)` | Remove an action |
| `doAction(hook, context)` | Execute sync actions |
| `doActionAsync(hook, context)` | Execute async actions sequentially |
| `doActionAll(hook, context)` | Execute sync then async |
| `doActionParallel(hook, context)` | Execute async in parallel |
| `hasActions(hook)` | Check if actions exist |
| `getHooks()` | Get all hook names |

## Self-Registration Pattern

Features register themselves using side-effect imports. This happens automatically when the feature is imported:

```typescript
// features/billing/index.ts
// Side-effect imports trigger registration
import './navigation.registration';
import './settings.registration';
import './route.registration';

// Public exports
export { BillingPage } from './pages/BillingPage';
export { billingRoutes } from './routes.config';
```

```typescript
// features/billing/navigation.registration.ts
import { navigationRegistry } from '@/config/navigation.registry';
import { CreditCard } from 'lucide-react';

navigationRegistry.registerMain({
  id: 'billing',
  title: 'Billing',
  url: '/billing',
  icon: CreditCard,
  order: 40,
});
```

When the app imports the feature (in `main.tsx` or `app.ts`), registrations happen automatically.
