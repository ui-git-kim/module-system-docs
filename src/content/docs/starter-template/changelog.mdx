This page documents notable changes to the starter template. The template uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, documentation updates

Check your current version in the `VERSION` file at the project root.

## v1.57.0

**Hooks clarity — typed catalogue, dispatch tracing, job lifecycle hooks, phantom purge**

The hook-adoption audit found module authors had no reliable way to discover what fires (magic strings, four drifting hand-maintained tables, docs advertising eight hooks that never existed). This release makes the hook surface honest and discoverable:

- **Typed hook catalogue** — new `hooks.catalog.ts` on both sides (`backend/src/config/`, `@/config/`): `CoreHooks` constants (autocomplete-safe names; string values are the unchanging contract) + `CoreHookContexts` (typed context payloads mirroring the actual emissions, each entry's JSDoc naming dispatch semantics and the emitting call site). See [Hooks → The Typed Hook Catalogue](/starter-template/hooks/#the-typed-hook-catalogue-v1570).
- **Dev-mode dispatch tracing** — every dispatch logs one debug line (hook, dispatcher, per-pool listener counts) on both registries; answers "did it fire, did anyone listen?" from the log. Production: no-op.
- **`job.completed` / `job.failed` fired** — both runners (inline + pg-boss) now dispatch job lifecycle hooks (`void doActionParallel`); `job.failed` was **declared in the jobs security manifest since v1.46 but never dispatched**. Context: `{ jobName, jobId?, attempt, runner, tenantId?, error? }`.
- **Phantom hooks purged from docs** — `settings.saved`, `route.changed`, `user.loggedIn`/`loggedOut`, and four filter hooks were documented (module-starter lifecycle-hooks page, registry JSDoc examples) but never fired; subscribing modules got silent dead code. Removed everywhere rather than implemented speculatively — **if your module subscribed to any of these, the handler never ran**; auth session hooks are roadmapped against real demand. The drifted module-starter execution tables are also synced with the v1.54 dispatch semantics.

Minor bump: two new public config modules. No signature changes; all existing hook names and dispatchers unchanged.

## v1.56.0

**Admin console cleanup — sectioned IA, real Overview, Widgets editor, styling sweep**

Ticks the roadmap's **Admin console cleanup & consolidation** and **Dashboard Widget Admin** (High) entries. Real installs run 12–14 admin pages in one flat list; this release gives the admin area an information architecture.

- **Sectioned sidebar** — `AdminPage` gains an optional `section` field (`overview` / `users` / `appearance` / `platform` / `extensions`; canonical list: `ADMIN_SECTIONS`), `adminPagesRegistry.getNavSections()` groups pages, and `AdminLayout` renders grouped sections. **Fully backwards compatible**: unsectioned module pages land in Extensions automatically. Order bands documented: built-ins reserve 1–29, modules use 30+ (sorted within a section).
- **Overview page** — the `/admin` landing page (retitled from "Dashboard") is rebuilt from a placeholder with hardcoded fake stats (including a lying green "Healthy") into a live dashboard: user/database/module/error stat cards linking into their pages, a **"needs attention"** list merging security findings, module dependency issues, and blocked modules (worst first), and the most recent audit-sink activity/errors. Stat values carry `data-slot="admin-stat-value"` so style modules can scale them via the Style Registry.
- **Widgets editor** *(new page, Appearance)* — `/admin/widgets` lists every registered dashboard widget (including currently-hidden ones) with visibility toggles writing the already-plumbed `AppSettings.dashboardOverrides`; saves apply to the running registry immediately. New `useDashboardOverrides` hook (draft-over-cache).
- **Renames & tidy-ups** — "App Settings" → **Branding** (id/URL unchanged); "Header Toolbar" → **Toolbar**, its stale "overrides are not applied at runtime" note corrected (they are) and its mislabelled "Center Items" tab fixed to Left; the `showDashboardInNav` toggle moved from Branding to the **Navigation** page (same persistence path); the sidebar Admin icon no longer shares Security's Shield.
- **Structure-only styling sweep** — ~45 catalogued violations cleared across the admin feature (typography scale, font weights, `shadow-lg`, `bg-primary/5`, and the `roleColors` colour-literal map → semantic Badge variants). Security/System/Modules pages verified already compliant.
- **Deprecations** — the six static admin-page exports from `@/features/admin` are deprecated (JSDoc, removal **v2.0.0**): they defeat the admin panel's `React.lazy` code-splitting; import from `@/features/admin/pages/<Page>` directly if needed. The v1.55.0 `backend/src/lib/audit-sink*` shims are **removed on their stated v1.56.0 target** (zero downstream importers, verified again at removal).

Verified: backend + frontend `tsc` clean against real installed dependencies.

## v1.55.0

**Quick-wins batch + containment tidy — six roadmap items closed**

The first release out of the July 2026 containment audit: the audit's verdict on `lib/` was that it is largely *correct* (the audited placement rule is now written into the [LLM rules](/getting-started/llm-rules/) — lib is for code ≥2 features or the bootstrap path import; single-feature seams live in their feature; tests may import internals directly), with one real move and a batch of verified roadmap quick wins.

- **Audit sinks moved into the admin feature** — `audit-sink.ts` and `audit-sql-sink.ts` were feature-grade observability code living in `backend/src/lib/`; every consumer was already in `features/admin/`. They now live there too, the admin barrel exports the seam (`auditSink`, `setAuditSink`, `SqlAuditSink`, `ComplianceAuditSink` + types), and the old `lib/` paths are **deprecated re-export shims** (dev-mode warning; removal target v1.56.0 — no downstream repo imported them, verified at move time). See the [versioning table](/starter-template/versioning/#currently-deprecated-exports).
- **Admin users pagination** *(roadmap: Backend list scalability)* — `adminService.listUsers({ page, pageSize })` (transactional `findMany` + `count`, pageSize clamped 1–100); `GET /api/v1/admin/users` returns a `UserListPage` envelope when paging params are present and the **legacy array otherwise** (non-breaking); `AdminDataTable` gains an optional `serverPagination` prop (client-side behaviour unchanged when absent); the admin Users page drives it end-to-end. Also `@@index([createdAt])` + `@@index([role])` on `User` — ships as a migration on your next update.
- **Last three admin hooks migrated to TanStack Query** *(roadmap: Duplicate fetches & cleanup)* — `useAdminUsers` (with `keepPreviousData` for flicker-free page flips and a read-after-write refetch before the spinner stops), `useNavigationOverrides`, `useToolbarOverrides` (draft-over-cache pattern so background refetches can't clobber in-progress edits; save-identity stability preserved). Return shapes unchanged; `useAdminUsers` additionally returns `total/page/pageSize/pageCount/setPage/setPageSize`. Behaviour note: lists now serve cached data within the 60&nbsp;s staleTime on remount instead of refetching every mount.
- **styleOverrides first-paint flash fixed** *(roadmap item)* — the admin's last-known map is cached in localStorage (`cacheStyleOverrides`/`readCachedStyleOverrides` from `@/config/style.registry`) and re-applied synchronously in `main.tsx` before React mounts; the server map replaces it wholesale on arrival.
- **Frontend env validation** *(roadmap item)* — `frontend/src/config/env.schema.ts` (zod) evaluated in `vite.config.ts`: **production builds fail fast** on missing/malformed `VITE_API_URL` / `VITE_NEON_AUTH_URL`; dev warns only.
- **Real 404 page** *(roadmap: Error Pages, partial)* — structure-only `components/NotFoundPage.tsx` (unmatched path shown, back/home actions) at the router fallback.

Verified: full backend + frontend `tsc` clean against real installed dependencies. Minor bump: additive public API (`UserListPage`, `serverPagination`, style-cache helpers, admin sink barrel exports) + the deprecation shims.

## v1.54.0

**Upstream-issues batch — all six open downstream reports fixed**

One release closes every open entry on the [upstream issues](/starter-template/upstream-issues/) page: reports from the **structure**, **document-management** and **billing** modules, fixed at the source so the whole fleet inherits them on the next `template:update`. Minor bump: new public exports and registry surface (webhook verification, `allowNoOrigin`) ride along with the fixes.

**1. Query retry policy — no retries on 4xx client errors** *(structure)*

The shared `QueryClient` retried every failed query once, including `429 Too Many Requests` — so when the API rate limiter started shedding load, every module's bootstrap queries immediately re-sent, amplifying the storm. `frontend/src/lib/query-client.ts` replaces the numeric `retry: 1` with a predicate: **4xx client errors never retry** (the request is unauthorised, invalid, missing, or rate-limited — a retry cannot succeed); network errors, 5xx, and `408 Request Timeout` keep the single retry with the default backoff. Mutations were already `retry: 0`; per-hook overrides still win. Deterministic 4xx failures now surface immediately instead of after a ~1s silent retry — if a query relied on that retry to paper over a session-bootstrap 401 race, gate it with `enabled` instead.

**2. `@types/express-serve-static-core` pinned to 5.0.x** *(document-management)*

The newly published 5.1.x widened `ParamsDictionary` values from `string` to `string | string[]` (modelling Express 5 wildcard `*name` params), failing `npm run type-check` on every clean install in each handler that destructures `req.params` — in the template and in every module. Runtime is unaffected (named `:param` segments are always strings). `backend/package.json` gains an `overrides` block pinning `~5.0.0` (resolves 5.0.7, still satisfying `@types/express`'s `^5.0.0`), and the backend lockfile is regenerated. **Existing apps** receive the pin on `template:update`: `sync-deps.mjs` now reconciles a foundation **overrides** manifest alongside foundation dependencies — presence-only (an override you already declare for the same package is never touched), running before `install:all` so the pinned version installs in the same pass. The pin is a **hold, not the destination** — the eventual migration is per-route param generics, tracked on the [roadmap](/starter-template/roadmap/); if you retype your own handlers sooner, delete the override and it won't come back.

**3. Action registry — async handlers no longer silently dropped** *(document-management)*

`doAction` ran only sync-registered handlers and **silently ignored** everything registered with `addActionAsync` — so a module's async subscriber on ~10 core hooks (`user.roleUpdated`, `user.profileUpdated`, `user.preferencesUpdated`, the five `admin.*Updated` hooks, `error:unhandled`, and frontend `theme.changed`) simply never fired. Both registries (backend + frontend) now:

- emit a **once-per-hook dev-mode warning** when `doAction` (or `doActionAsync`) would skip handlers sitting in the other pool — the silent drop is gone (production stays silent);
- route `doActionAll` through private execution loops, so it can never double-fire and never warns spuriously.

The core emission sites now actually run async handlers: admin mutations (`admin.settingsUpdated`, `navigationUpdated`, `toolbarUpdated`, `styleOverridesUpdated`, `dashboardOverridesUpdated`) and both `user.roleUpdated` emissions use **`await doActionAll`** (read-after-write consistency before the admin UI refetches; handler errors are caught inside the dispatcher and can never fail the request); the hot paths `user.profileUpdated` / `user.preferencesUpdated`, `error:unhandled`, and frontend `theme.changed` use **`void doActionParallel`** (fire-and-forget — sync handlers still run inline). **Behaviour note:** async handlers that were dead code until now START FIRING after update — verify yours are idempotent; a slow awaited handler adds latency to admin mutations, so keep those fast. Dispatcher-set consolidation is deliberately deferred to the [roadmap](/starter-template/roadmap/). See [Hooks](/starter-template/hooks/).

**4. `_module_migrations` read no longer logs a spurious `prisma:error`** *(document-management)*

Until a module runs its first `migrate-db` migration the bookkeeping table doesn't exist, and the admin Modules read failed with `42P01` — handled correctly, but Prisma printed the failed query at error level *before* the catch could downgrade it (an alarming red line on every `/admin/modules` load). `readAppliedMigrations()` now probes existence first with `to_regclass()` (returns NULL, never raises) and skips the SELECT when absent. Same UI semantics; zero noise.

**5. Sub-package foundation scripts now propagate on `template:update`** *(billing)*

`sync-package-scripts.mjs` reconciled only the root `package.json`, so apps created before v1.38/v1.43 came out of an update with a CI workflow calling `type-check`/`test` scripts their user-owned `frontend`/`backend` `package.json` never received ("Missing script: type-check", red CI). The reconciler now manages per-target manifests: root (previous set plus the `type-check`/`type-check:frontend`/`type-check:backend` wrappers, absent from pre-v1.38 apps), `frontend` (`type-check`, `test`, `test:watch`) and `backend` (`type-check`, `pretest`, `test`, `test:watch`); `update-template.{ps1,sh}` stage and commit all three files — per-file, so a layout missing a sub-package file can't silently abort the whole staging. Managed keys are **forced to the canonical command** (CI depends on their exact behaviour) — deliberately different from `sync-deps`' presence-only rule, because dependency *versions* are user-owned while foundation script *commands* are template-owned. Your own scripts are never touched; a customised managed key (e.g. a hand-edited `test`) will be reset to canonical on update.

**6. Webhook signature verification infrastructure** *(billing, High)*

Public webhook endpoints were unverifiable — the global JSON parser discarded the raw request bytes, no keyed-MAC primitive existed, and (discovered during the fix) **production CORS rejected all no-Origin server-to-server requests**, so webhook deliveries could never reach a handler in production at all:

- **`req.rawBody`** — `express.json` now captures the exact raw bytes via a `verify` hook (`middleware/raw-body.middleware.ts`); a reference, not a copy, bounded by the existing 10&nbsp;mb limit.
- **Security feature**: new `computeHmac` / `timingSafeEqualStrings` primitives (`crypto.ts`) and `verifyWebhookSignature` / `verifyTimestampedWebhookSignature` helpers (`webhooks.ts`) — the Stripe-style `t=...,v1=...` scheme with a replay window (default 300&nbsp;s, rotation candidates accepted) and the plain GitHub-style scheme (`prefix: 'sha256='`); constant-time comparisons, `node:crypto` only. See [Webhook signature verification](/starter-template/security-feature/#webhook-signature-verification).
- **Route registry**: `allowNoOrigin?: boolean` on `RouteConfig`/`PublicRouteConfig` + `routeRegistry.allowsNoOrigin(path)`; the CORS layer (now the options-delegate form) admits production no-Origin requests **only** for opted-in routes. Flag it exclusively on endpoints authenticated by other means (webhook HMAC) or safe probes.
- **Health endpoints** opt in: production no-Origin monitor probes of `/api/health` and `/api/health/db` previously got CORS-rejected 5xx — they now return real status. Retrain any monitor that was trained on the failure.
- **Module note**: signature *verification* needs no module changes (billing already reads `req.rawBody`), but production *delivery* needs one registration line in the module — `allowNoOrigin: true` on the webhook's `registerPublic` call.

**Also:** two latent frontend type errors fixed (`settings-blocks/Table.tsx` unused import; `AdminComponents.tsx` React 19 portal cast) — restores a clean frontend `type-check` under current `@types/react`.

**Propagation caveat (one-update lag):** an app's update run executes its *locally checked-out* `update-template` script, so apps whose script predates v1.51.0 (`sync-deps`) or this release (widened `sync-package-scripts`) won't have the pin/scripts fully applied+committed on their first update — run `node scripts/sync-deps.mjs && node scripts/sync-package-scripts.mjs` manually, or update twice.

## v1.53.0

**Vite config shadow — now auto-healed before dev/build**

The compiled-`vite.config.js`-shadows-`vite.config.ts` trap (v1.51.0 shipped a *warning* for it during `template:update`) actually bites at `npm run dev` — where a stale compiled config silently overrides every `vite.config.ts` change, with a bewildering symptom (e.g. the optional `@sentry/react` alias not applying, so the dev dependency scan can't resolve it).

- **`scripts/check-vite-shadow.mjs`** now runs as a `predev`/`prebuild` guard: if a `frontend/vite.config.{js,mjs,cjs,d.ts}` sits next to `vite.config.ts`, it **removes** the artifact when git confirms it's untracked (the gitignored `tsc` output — nothing versioned is lost) and prints a one-line notice. A *tracked* `vite.config.js` (a deliberate choice) or a checkout with no git repo is left alone with a warning. It never exits non-zero, so it can't block dev/build.
- Reconciled into existing apps: `predev`/`prebuild` join `FOUNDATION_SCRIPTS` in `sync-package-scripts.mjs`, so `template:update` merges them into the user-owned root `package.json` — the guard reaches apps created before it existed.

Non-breaking; one new script + two npm pre-hooks.

## v1.52.0

**Module migrations — the host's read-only half (contract v1.2.0)**

Groundwork for [per-module schema migrations](/starter-template/roadmap/): the module CLI will apply ordered migration files at `npx <module> update` and record them in a runner-owned `_module_migrations` table (the runner itself ships next, in the module starter). This release adds the **starter's** share — the shared contract surface plus a read-only status view:

- **Contract v1.2.0** (`module-contract.ts`, MODULE_CONTRACT_VERSION 1.1.0 → 1.2.0): the `_module_migrations` table name/DDL/row shape (`MODULE_MIGRATIONS_TABLE`, `MODULE_MIGRATIONS_COLUMNS`, `MODULE_MIGRATIONS_DDL`, `ModuleMigrationRecord`) and the `<version>__<slug>.{sql,ts}` filename parse + ordering (`parseMigrationFilename`, `compareMigrations`). Both the CLI's runner and the host reference these, so applied order and displayed order can never drift. The table is **not** a Prisma model — it is runner plumbing, reached by raw SQL (the same call made for the security `AuditLog` table, avoiding the multi-site embedded-schema hand-sync). Module starters must re-run `contract:sync`.
- **`getModuleMigrationStatuses()`** in the modules feature — diffs the on-disk migration files (`backend/prisma/module-migrations/<id>/`) against the `_module_migrations` rows per registered module, reporting applied/pending counts. Read-only and never-throwing: it reads the table via raw SQL (independent of the generated Prisma client) and falls back to disk-only (everything pending) when the table can't be read. Folded into `getModulesOverview()`.
- **Admin Modules page** gains a **Migrations** section: per-module applied/pending with the pending filenames, structure-only styling (icon + semantic token).

Non-breaking (additive contract constants + read-only surface + one admin section). The host never applies a migration — that stays a CLI operation against `DIRECT_DATABASE_URL`.

## v1.51.0

**Update-script resume (`-Continue`), foundation dependency reconciliation, Vite config shadow warning**

Three more field-proven gaps from the same v1.25→v1.50 app update that produced v1.50.2 — this time the class of failure where a merge conflict (or a user-owned `package.json`) quietly leaves an app half-updated:

- **`-Continue` / `--continue` flag on `update-template.{ps1,sh}`.** A merge conflict aborts the update *before* the whole post-merge pipeline (schema auto-add, dependency install, script/env/dep reconciliation, migration, `prisma generate`, shadcn check) — and after you resolve and commit, a plain re-run exits "already up to date" without resuming any of it. `npm run template:update -- -Continue` (or `./update-template.sh --continue`) now resumes the pipeline: it skips the already-done merge, refuses while a merge is still in progress (with step-by-step guidance — in any mode, replacing the misleading "commit or stash" advice mid-merge; unmerged files *without* a merge in progress get a neutral rebase/cherry-pick diagnosis instead), **refuses when the template has commits you haven't merged** (an aborted merge, or a release published while you were resolving — resuming would reconcile schema/deps against never-merged code; a normal update is the answer there), and runs the migration unconditionally since the post-commit merge-base diff can no longer reveal whether the merge carried schema changes (`migrate dev` no-ops cleanly). The conflict-exit message and the "already up to date" exit both name the command. See [Resuming after a conflict](/starter-template/updating/#resuming-after-a-conflict--continue).
- **`scripts/sync-deps.mjs` — foundation dependencies now reach existing apps.** `package.json` files are user-owned (never force-synced), `install:all` installs only what's listed, and `deps:update` bumps existing deps but never *adds* missing ones — so a dependency the template starts relying on (confirmed in the field: `vitest`, added v1.43, absent from every pre-v1.43 app) never arrives. The new reconciler is the third leg beside `sync-package-scripts.mjs` and `sync-env.mjs`: presence-only (your placement and version choices are never touched), runs before `install:all` so additions install in the same pass, commits scoped to the three `package.json` files. Initial manifest: `vitest ^3.0.0` (devDependency, frontend + backend).
- **Shadowed `vite.config.ts` warning.** Vite resolves `vite.config.js`/`.mjs` *before* `vite.config.ts`, so a stray compiled config (a one-off `tsc` artefact) silently swallows every template fix to the real config — no error, just changes that "don't take". The update script now ends every completed update run — and the "already up to date" fast path, where a diagnostic re-run lands — by naming any `frontend/vite.config.{js,mjs,cjs}` sitting next to `vite.config.ts` (a lone `.cjs` gets a softer "stale, not shadowing" note since `.ts` outranks `.cjs`) and printing a per-file removal command chosen by tracked-ness (`git rm` for tracked files, plain delete for untracked-and-gitignored artefacts, where `git rm` would fail). Detection only — it never deletes user files.

Non-breaking; update-script hardening + one new reconciler script.

## v1.50.2

**Template update no longer leaves a half-applied app (Prisma 7 + Vite 8)**

Found in the field on the first app to update across v1.48–v1.50: `template:update` completed but the app booted with a stale Prisma client, and the dev server showed a Vite overlay for the optional Sentry package. Three root causes, all fixed:

- **`prisma generate` now runs explicitly at the end of every update** (both scripts). The old flow assumed the client regenerated during `install:all` — true under Prisma 6's postinstall hook, gone in Prisma 7 (and `migrate dev` does not reliably regenerate either). A merged schema change could therefore leave the app importing a client that predates new models — the "Cannot read properties of undefined (reading 'findMany')" TypeError at boot.
- **The migration step retries once after a short wait.** A suspended Neon compute wakes on first contact, so the initial `prisma migrate dev` frequently dies with `P1001` mid-update; the retry lands. The manual-fallback warning remains for genuine failures.
- **Optional `@sentry/react` resolves without being installed under Vite 8.** Vite 8's stricter dev-time import analysis resolves every dynamic import — even inside the dead `__SENTRY_ENABLED__` branch — erroring the overlay when the optional package is absent. `vite.config.ts` now aliases `@sentry/react` to a local no-op stub (`src/lib/sentry-stub.ts`) when no `VITE_SENTRY_DSN` is set; with a DSN, no alias and the real package resolves. The stub never executes — the build-time guard still folds the block away.

Non-breaking; update-script + frontend-config hardening.

## v1.50.1

**Modules feature — friendly remediation for a stale Prisma client**

Right after a template update — before `npx prisma migrate dev` has run — the app's generated Prisma client predates the `InstalledModule` model, so `prisma.installedModule` is `undefined` and the startup state sync warned with a raw `TypeError` ("Cannot read properties of undefined (reading 'findMany')"). The server still booted (never-throw held), but the message didn't say what to do.

Both the startup sync and the admin overview now detect the missing delegate up front and log the actual remediation: *"Prisma client predates the InstalledModule model — run `npx prisma migrate dev` in backend/ (creates the table if needed and regenerates the client)"*. Quiet (debug) when no modules are registered; a warning otherwise, and always a warning on the admin overview path where someone is actively looking.

## v1.49.0

**Dependency resolution (warn-only) — semver ranges validated, topological order computed**

The [modules feature](/starter-template/modules-feature/)'s dependency seam grows real teeth — as **warnings** (hard enforcement and ordered install/boot remain future steps):

- **Contract v1.1.0** — `module-contract.ts` now carries the semver range semantics themselves: pure, zero-import `compareSemver()` + `satisfiesRange()` (documented subset: `*`/`x`, exact, X-ranges, `^`, `~`, comparators, space-AND, `||`-OR; divergences from full npm semver — no hyphen ranges, no prerelease gating, comparator partials zero-fill — are listed in the file). Because the module-starter consumes a hash-synced copy, the CLI's install-time check and the host's boot-time check can never disagree about what a range means.
- **`validateDependencies()` now range-checks** — a registered dependency whose version falls outside the declared range produces a `version-mismatch` issue (logged with the audit flag + fired on `module.dependencyIssue`). The feature-folder excusal applies only to `missing` — a mismatch means the dependency IS present, at the wrong version.
- **`moduleRegistry.computeModuleOrder()`** — dependency-first topological order (Kahn's algorithm, deterministic alphabetical tie-break) over the registered modules; modules on a cycle are excluded and cycles report as **readable paths** (`a → b → a`), never a vague "unresolved dependencies"; ids stuck behind (or entangled with) a reported cycle surface in a `blocked` list, so `{ order, cycles, blocked }` always partitions the registry. Startup warns per cycle and for blocked ids; nothing consumes the order at runtime yet — it is the seam for ordered install/boot and the admin dashboard.
- Validated during development against a throwaway executable scenario harness (90+ assertions, strict-compiled and run against the real implementation — including adversarial cases found in review: operator whitespace, bare operators failing closed, wildcard-minor X-ranges); the vitest suite (`modules.test.ts`) carries the shipped CI coverage.

Non-breaking (additive contract functions + additive issue kind; `requiredModules` remains presence-only). Cross-repo: module-starter scaffold **v1.17.0** syncs the contract and warns on range mismatches at install.

## v1.48.0

**Module system as a foundation feature — `ModuleContract` home, runtime registry, installed-module state**

The module system itself is now a foundation feature (`backend/src/features/modules/`, backend only) — the home of "how modules work in the system", per the [roadmap decision](/starter-template/roadmap/#module-system-as-a-foundation-feature-modulecontract-home) from the v1.44.0 manifest review. See the new [Modules feature](/starter-template/modules-feature/) page.

- **`ModuleContract`** — the canonical, cross-repo module contract (id / name / description / `version`, new **`depends`** map of module id → semver range, deprecated-but-honoured `requiredModules`, and the lifecycle hook surface with contexts generic over the definition type). Lives in a deliberately self-contained `module-contract.ts` (zero imports) with its own `MODULE_CONTRACT_VERSION`, because the module-starter consumes a **hash-stamped synced copy** and extends it into its full `ModuleDefinition` — one source of truth, two repos. `depends` ranges are **recorded, not yet validated**: enforcement + topological install/boot ordering land with the dependency-resolution effort.
- **`moduleRegistry`** — runtime registry installed modules register with at import time via a generated `module.registration.ts` (the `security.registration.ts` pattern). Foundation features don't register — they're not modules. Registered = installed ≠ enabled (the CLI's enable/disable state is invisible to the runtime today).
- **New `InstalledModule` model + startup state sync** — deployment-level (no `tenantId`, like `AppSettings`; untouched by account deletion). At boot the feature diffs the registry against the table and emits the module lifecycle audit events — `module.installed`, `module.updated`, `module.missing` — via audit-tagged logs + `actionRegistry`; this diff is the only way the runtime can observe CLI-side install/uninstall, which happen while the server is down. Rows are never deleted: disappearing code is stamped `missingSince` (cleared on return); `enabled` is the Admin Module Management seam. Auto-added to existing apps by `template:update` (both scripts) and embedded in create-app's multi-file mode.
- **Dependency-check seam** — every boot, each registered module's `depends`/`requiredModules` ids are presence-checked (`module.dependencyIssue` on genuine gaps). Unlike the CLI's install-time check, this catches **uninstall drift**; a feature-folder fallback excuses installed-but-unregistered dependencies (old-vintage or frontend-only modules) from false warnings.
- **Contract-sync release gate** — `scripts/bump-version.ps1/.sh` now warn at release time when `module-contract.ts` changed since the last bump ("run `contract:sync` in the module-starter"), closing the silent-drift gap the embedded-AppSettings-schema precedent suffered.
- The `modules` security manifest declares the four audit events; no owned models, categories or deletion handler (deployment-level state only). Wired in `app.ts` via the standard foundation import + a fire-and-forget `runModuleStartupTasks()` (never throws; an unmigrated table is skipped quietly).

**Separation preserved:** the `ModuleSecurityManifest` stays in per-feature `security.registration.ts` — this feature owns module *identity/lifecycle*, not security declarations. **Cross-repo next leg:** module-starter scaffold v1.16.0 consumes the contract (`contract:sync` + drift check, `ModuleDefinition extends ModuleContract`, host-gated `module.registration.ts` generation).

Non-breaking (additive feature + additive model; existing modules and `requiredModules` keep working unchanged).

**Cloudflare edge + startup posture guardrail**

The final security-floor phase. The perimeter — DDoS, WAF, bot management, edge rate limiting, TLS/HSTS, the frontend XSS-CSP, optional Zero Trust — is delegated to **Cloudflare**, with almost no app code: it's dashboard configuration plus two env knobs. The code half is a **startup posture guardrail** that tells you whether the posture is actually in place. **The edge is additive — its absence warns, never fails; the app-layer floor (RLS, auth, validation, rate limiting, field encryption) stands alone.**

### Startup posture guardrail

- **`features/security/posture.ts`** — `runStartupPostureCheck()` is called at the end of `app.ts` bootstrap, once every feature and module has registered. It evaluates the resolved posture and surfaces misconfiguration you'd otherwise hit at runtime: **edge declaration** (warn at `hardened`+ in prod with no edge), **trust-proxy sanity** (`SECURITY_EDGE=cloudflare` but `TRUST_PROXY_HOPS=0` breaks IP keying), **field-encryption key readiness** (on but no key / unwired KMS), **auth-identity deleter** (unwired ⇒ deletion leaves the login behind), **module tier requirements** (`findTierViolations` — a module declaring a `minimumTier` above the active tier), and **production hardening** (`REQUEST_TIMEOUT=0`).
- **Enforcement** — findings log at their severity; an `error` finding **halts startup** only at the `strict` tier in production by default. `SECURITY_POSTURE_ENFORCE` overrides (`true` everywhere, `false` never — which at `strict` defeats the guard). This wires the seam `securityRegistry.findTierViolations()` left open since v1.28.0.
- **`evaluateStartupPosture()`** — the pure, no-side-effect form (returns a `PostureReport`) for tests and a future health surface. **`edgePostureMiddleware`** — a one-shot check that the first real inbound request actually carries Cloudflare headers (`cf-ray` / `cf-connecting-ip`); warns on a bypass, pure pass-through otherwise.
- **`getKeyProviderStatus()`** — a non-throwing readiness probe added to the key provider (env key present + 32 bytes, or a registered/auto-wired KMS adapter), consumed by the guardrail.

### Edge setup docs + per-tier checklist

- New guide: **[Cloudflare edge](/getting-started/cloudflare-edge/)** — proxied DNS, Full (strict) TLS + HSTS, the WAF, edge rate-limiting rules, the frontend CSP (where XSS protection actually lives — the API's `helmet` CSP serves JSON), locking the origin to Cloudflare, declaring `SECURITY_EDGE`, and optional Zero Trust on `/admin`. Includes a **per-tier edge checklist** (`standard` → `strict`).

### New env knobs

- **`SECURITY_EDGE`** (`cloudflare` \| `none`) and **`SECURITY_POSTURE_ENFORCE`** (`true` \| `false`). Both are appended to existing apps' `backend/.env` on `template:update` (via `sync-env.mjs`) and written by `create-app`.

Non-breaking. Existing apps at `standard`/`hardened` only ever see logged findings — the guardrail never exits below `strict`-in-production (or `SECURITY_POSTURE_ENFORCE=true`). Apps run exactly as before until they choose to put Cloudflare in front and set `SECURITY_EDGE`.

## v1.35.0

**KMS provider auto-wires from env (no bootstrap edit)**

The last manual step of AWS KMS setup is gone. When the env declares KMS key custody (`SECURITY_KEY_PROVIDER=kms` + the `SECURITY_KMS_*` vars that `npm run security:setup-kms` writes), the security feature **auto-wires** the AWS KMS provider at bootstrap — no `app.ts` edit.

- **`features/security/key-providers/auto-wire.ts`** — a side-effect module (imported by the security barrel) that registers `createAwsKmsKeyProvider` from env when fully configured. The wiring lives in a **template-owned file** (so it survives `template:update`), and your **`.env`** (user-owned, preserved) is what switches it on — the robust version of "wire it in code, activate on setup" *without* the force-sync revert that editing `app.ts` would hit. The AWS adapter is imported statically but only loads the SDK lazily inside `getKey()`, so non-KMS apps never pull it; registration is synchronous (no startup race). A manual `setKeyProvider(...)` still overrides.
- **`security:setup-kms`** now ends with "restart your dev server" instead of a bootstrap snippet, no longer trips Node's `DEP0190` warning (the npm install runs as a single command string), and — like `template:update` / `deps:update` — **preserves your `npm link` symlinks** across its SDK install (snapshot + restore), so a linked module dev setup isn't clobbered.

Non-breaking: apps not using KMS are unaffected; KMS apps that wired `app.ts` manually keep working (the manual call wins).

## v1.34.0

**One-command AWS KMS setup + field-addition migrations on update**

Two follow-ups that take the v1.33.0 KMS work from "documented" to "wired", and close a real update gap.

### `npm run security:setup-kms` — prompt-and-inject KMS setup

The AWS KMS walkthrough was a lot of manual steps. Now it's one interactive command — same shape as the Neon setup: you create the key + IAM principal in the AWS console, then the script prompts for the values and **does the rest**.

- **`scripts/setup-kms.mjs`** — prompts for region, KMS key id/ARN, and AWS access key + secret (secret masked; leave blank to use an IAM role / `aws configure`). ### `template:update` migrates new *fields*, not just new *models*

`template:update` auto-migrated new **models** but missed new **fields** on existing models — the merge brought the field into `schema.prisma` with no migration, silently drifting the DB (the `AppSettings.dashboardOverrides` → `column does not exist` class of bug). Now, whenever the template changed any Prisma schema file, the update runs `prisma migrate dev` so added fields are migrated (additive; a removal/rename prompts interactively). Closes the drift for good.

No new template dependency; no schema change.

## v1.33.0

**Field-encryption key custody (AWS KMS) + env reconciliation + Better Auth fix**

### Field encryption: provider-held key custody (AWS KMS)

The field-encryption master key can now live in **AWS KMS** instead of an env var — closing the "the key sits on my server" gap. Envelope encryption: KMS holds the root key, the app unwraps a data key **once** at startup and caches it, and AES-256-GCM runs locally (KMS isn't called per field). The plaintext key is never in env, the database, or a Neon dump.

- **`createAwsKmsKeyProvider({ region, keyId, wrappedDek })`** — an opt-in reference `KeyProvider` at `@/features/security/key-providers/aws-kms`. Set `SECURITY_KEY_PROVIDER=kms` and wire it at bootstrap. The AWS SDK (`@aws-sdk/client-kms`) is an **optional dependency you install in your app** (dynamically imported, so apps not using KMS still build without it) — *not* a template dependency.
- **Crypto-shred** = disable/delete the KMS key (global); per-tenant crypto-shred is the documented growth path (the seam already accepts `tenantId`).
- Vetted by an adversarial crypto review (single-flight unwrap, `KeyId`-pinned `Decrypt`, 32-byte fail-closed, no key material logged).
- New guide: [Field encryption with AWS KMS](/getting-started/aws-kms-setup/).

### Better Auth deleter correction

Neon Auth migrated from **Stack Auth → Better Auth**. The v1.32.0 `createNeonAuthIdentityDeleter` baked the legacy Stack Auth endpoint and is replaced by a provider-agnostic **`createRestAuthIdentityDeleter({ buildUrl, headers })`** (you supply your provider's admin endpoint). For Neon Auth the recommended path is now implementing `AuthIdentityDeleter` directly against the Neon Auth server SDK (`@neondatabase/auth`, admin `removeUser`) — see the [Security feature](/starter-template/security-feature/#the-neon-auth-identity-closing-the-gap). The provider-agnostic seam (`setAuthIdentityDeleter`) is unchanged.

### Foundation env-key reconciliation

`.env` is user-owned, so new foundation env knobs never reached existing apps — you had to hand-add them per app. **`scripts/sync-env.mjs`** (run by `template:update`) now **appends** any missing foundation key to `frontend/.env` / `backend/.env`: `SECURITY_PROFILE` / `VITE_SECURITY_PROFILE` active at the tier default; `SECURITY_ENCRYPTION_KEY`, the deletion-cascade `NEON_AUTH_*`, and the new KMS `SECURITY_KMS_*` / `AWS_REGION` as commented placeholders. Append-only — existing values are never touched, and `.env` stays git-ignored. `create-app` writes the same placeholders for fresh apps.

No new template dependency; no schema change.

## v1.32.0

**Deletion cascade + cross-platform dependency tooling**

Two things ship together: the next security-floor phase (provider-delegated account/tenant erasure) and a rebuild of the dependency-update tooling so it actually works in host apps.

### Deletion cascade (provider-API wiring)

The "named bespoke = thin provider wiring" exception in the [delegation model](/starter-template/security-feature/). On account/tenant deletion, `deleteAccount()` runs every registered deletion handler in a **classification-driven, fail-safe order**, then removes the auth identity **last**, and returns a `DeletionReport` (it never throws — every outcome is captured).

- **`deleteAccount(subject, opts)`** — the orchestrator. Order: capture a compliance bundle **before** any destruction → `credential` (crypto-shred) → `owned` (hard-delete; the core `User` row runs late so module child tables clear first) → `referenced` (unlink, never touch the source) → `preserved` recorded as a **deferred legal hold** → auth identity **last**, and only if no `owned`/`credential` handler failed (so a partial failure keeps the login alive for an idempotent retry). There is deliberately **no wrapping DB transaction** — the cascade spans external providers (R2/Stripe/KMS) that can't enrol in one.
- **`securityRegistry.registerDeletionHandler({ id, moduleId, classification, order?, run })`** — modules contribute deletion units via a side-effect `security.registration.ts`. Handlers must be **idempotent** and must throw / return `{ action: 'failed' }` on non-completion.
- **Auth-identity seam (`auth-identity.ts`)** — closes the historical gap where `deleteUser` left the **Neon Auth identity** behind. The default deleter is `unconfigured` (a visible WARN + a reported `unconfigured` status — it does *not* fake success); wire the real provider call once at bootstrap via `setAuthIdentityDeleter(createNeonAuthIdentityDeleter({ projectId, secretServerKey }))`. Mirrors the `key-provider` pattern.
- **Compliance bundle** — the data-classification manifest + consent/decision record + legal-hold deferrals, captured before destruction (GDPR Art. 17 / Art. 30). Returned in the report for the app to persist or email.
- **Both deletion paths wired** — self-service (`user.service.deleteUser`) and admin (`admin.service.deleteUser`) now run the same cascade. The admin path keeps its atomic last-admin guard (the row is deleted inside the Serializable transaction; the idempotent `user.account` handler then no-ops).
- New public exports from `@/features/security`: `deleteAccount`, `buildComplianceBundle`, `registerDeletionHandler`/`getDeletionHandlers`, the `AuthIdentityDeleter` seam, and the deletion types. `deleteUser`'s return is **widened additively** (`status` + `cascade` added; existing fields kept). No new dependency (Node `crypto`/`fetch`). Non-breaking, no migration.

<Aside type="caution" title="Migration note for modules">
If a module moves its cleanup from the `user.deleted` action hook into a registered deletion handler, **remove its `user.deleted` subscriber in the same step** — otherwise the cleanup runs twice. The legacy hook is kept (it fires after the cascade) only for back-compat during migration.
</Aside>

### Cross-platform dependency tooling

Fixes two real host-app problems: `npm run deps:update` did nothing (the root `package.json` is user-owned, so the `deps:*` aliases never reached older apps — and they were wired to `pwsh`, not cross-platform), and `template:update` wiped locally `npm link`-ed modules.

- **`deps:*` are now Node-based + cross-platform** — `scripts/update-deps.mjs` (no PowerShell dependency). The old `update-deps.{ps1,sh}` are thin shims that delegate to it.
- **`deps:lockfile` (new)** — `npm install --package-lock-only` across root/frontend/backend: regenerates lockfiles **without** installing `node_modules`. This is the mode for refreshing the **template repo's own** stale locks (it ships no `node_modules`).
- **npm-link preservation** — `scripts/link-modules.mjs` snapshots `npm link` symlinks before, and restores them after, any install that prunes them. `template:update` and `deps:update`/`deps:update:latest` now keep your locally-linked modules intact. Manual re-link: `npm run deps:relink`.
- **Foundation-script reconciliation** — `template:update` runs `scripts/sync-package-scripts.mjs`, which ensures the `deps:*` aliases exist + point at the canonical commands in the host app's `package.json` (the fix for "`Missing script: deps:update`" on older apps). Only foundation keys are touched; your own scripts are untouched.

## v1.31.0

**Field encryption (app-layer AES-256-GCM)**

The foundation crypto primitive — modules declare `encryptedFields` / `hashedSecrets` and call it; they never hand-roll crypto.

- **`encryptField` / `decryptField`** — AES-256-GCM via Node's `crypto` (random 96-bit IV per call; 128-bit GCM auth tag verified on decrypt → tamper throws). Versioned envelope (`v1:…`) for rotation.
- **`hashSecret` / `verifySecret`** — SHA-256 + constant-time compare, for high-entropy tokens / license keys (not passwords — those are Neon Auth's).
- **Key provider abstraction** — the **`env`** adapter ships working (`SECURITY_ENCRYPTION_KEY`, base64 32 bytes; `openssl rand -base64 32`); a **`kms`** adapter is an interface, registered per-deployment via `setKeyProvider()`. `kms` selected-but-unwired throws at `strict`, warns + falls back to env elsewhere.
- **The key is never sent to the database** — app-layer encryption (deliberately *not* `pgcrypto`, which would send the key to Postgres), so Neon only ever holds ciphertext. Field encryption is off at `standard`, on at `hardened`+.
- New env var `SECURITY_ENCRYPTION_KEY` (optional; added commented to the create-app `.env` template). No new dependency (Node `crypto`). Non-breaking.

Stronger zero-knowledge ("even the app can't read it") has no provider and stays on the [roadmap](/starter-template/roadmap/#security-floor-v200).

## v1.30.0

**Isolation floor — Neon RLS + the `tenantId` convention**

Reorients data isolation to a provider-delegated model: **Neon RLS is the wall**, the `owned()` helper is application-layer defence-in-depth, both scoped on the isolation key **`tenantId`**. Light and non-breaking for the starter (no owned *resource* models yet — `User` is the actor/identity, `AppSettings` is global); this ships the convention + helpers + guidance that modules build on.

- **Convention:** owned models carry **`tenantId`** (isolation key, NOT NULL, indexed) + **`userId`** (actor). In solo apps `tenantId === userId`; the column is present from the start so growing to multi-user is a *data* change, not a schema rewrite. The `schema.prisma` example model shows the pattern.
- **`req.user.tenantId`** — set at auth time (= the user id in solo mode); the isolation key all data-scoping reads.
- **`withTenantContext(tenantId, fn)`** — the Neon RLS seam (sets `app.tenant_id` inside a transaction); pair with `FORCE ROW LEVEL SECURITY` + a `tenantId` policy. `withUserContext()` is now **`@deprecated`** (use `withTenantContext`; it still sets `app.user_id` so existing policies keep working).
- **`owned()` now keys on `tenantId`** (the isolation key) instead of `userId`. *Heads-up* for anyone who adopted the v1.27.0 opt-in helper: pass `req.user.tenantId` and ensure the owned model has a `tenantId` column.
- **Backend CSP** unchanged from v1.29.0; no new dependencies.
- **Docs:** the [Database page](/starter-template/database/#user-data-isolation-neon-rls) reframed from "opt-in strict-tier RLS" to the **Neon RLS floor** — the `FORCE RLS` + `tenantId` policy template, `withTenantContext`/`owned()` usage, and the database-per-tenant pointer for hard isolation.

Non-breaking for the starter's own schema (no migration). Modules add the `tenantId` column to their owned models going forward.

## v1.25.0

**`AppSettings.appTheme` — module-owned theme storage on the singleton**

A new dedicated slot on `AppSettings` for **style modules** (useFoundry, or any future alternative) to persist their theme configuration. The starter treats it as opaque JSON storage — the shape is entirely owned and validated by the consuming module. This closes the loop for the style-registry conversation: modules now have a single well-known location for their app-level theme, cleanly separated from the `styleOverrides` boolean map used for catalogue entry toggles.

### The two-bucket model

| Field | What lives here | Shape | Who owns shape |
|-------|-----------------|-------|----------------|
| `appTheme` **(new)** | Base app-level theme config — shadcn preset codes, module extensions, whatever the style module needs | `unknown` (opaque) | The consuming style module |
| `styleOverrides` | On/off flags for individual catalogue entries (`tabs-line-primary`, `button-bold`, etc.) | `Record<string, boolean>` | Starter (boolean-map convention) |

`appTheme` is the **base**; `styleOverrides` is **adhoc opt-ins layered on top**. Matches the "shadcn core + module extensions in appTheme, catalogue entries separate" split that surfaced during the style-module architecture conversation.

### What shipped

**Backend:**

- **Schema**: `appTheme Json?` added to the `AppSettings` model in `backend/prisma/schema.prisma`. Nullable so a freshly-created app starts with no theme ("use module defaults").
- **Types** (`admin.types.ts`): new `AppTheme = unknown` type alias; `appTheme: AppTheme | null` added to `AppSettings` interface; `appTheme: z.unknown().optional()` added to `UpdateAppSettingsSchema`.
- **Service** (`admin.service.ts`): `getAppSettings()` returns `appTheme` via Prisma spread; `updateAppSettings()` handles `data.appTheme === null` → `Prisma.DbNull` (clears the column), non-null → `Prisma.InputJsonValue`, `undefined` → field omitted from update (standard Prisma semantics).

**Frontend:**

- **Service types** (`features/admin/services/admin.service.ts`): `AppTheme` type + `appTheme: AppTheme | null` on `AppSettings` + `appTheme?: AppTheme | null` on `UpdateAppSettingsInput`. Re-exported from `features/admin/index.ts` and (for consistency) from `features/admin/context/AppSettingsContext.tsx`.
- **Context** (`AppSettingsContext.tsx`): new `setAppTheme(config: AppTheme | null): Promise<void>` on the context value. Optimistic cache write → PATCH `/api/v1/admin/settings` with `{ appTheme }` → refetch to reconcile. `DEFAULT_SETTINGS.appTheme = null`.

**Usage from a style module:**

```tsx
import { useGlobalAppSettings, type AppTheme } from '@/features/admin';

function MyStyleModuleBootstrap() {
```

## v1.24.7

**Structure-only styling rule — template components express layout + state, not visual design**

The template had accumulated ~170 small visual-design decisions (typography scales, font weights, hardcoded status colours, decorative shadows) baked into component code. Every one of them imposes a choice on downstream apps that have their own brand, palette, and density preferences. That's the opposite of the template's value proposition — a starter template should give apps structure, not force them to fight for their own look.

v1.24.7 codifies the rule. Going forward, template code (everything under `frontend/src/` outside `components/ui/`) can only use layout utilities and shadcn's semantic tokens. Typography, colour literals, shadows, and decorative opacity belong in your generated globals.css, your shadcn component variants, or Style Registry entries.

### The policy in one table

| Category | Template allowed? | Where it belongs instead |
|----------|------------------|--------------------------|
| Layout (`flex`, `grid`, `gap-*`, `p-*`) | ✅ | — |
| Sizing (`w-*`, `h-*`, `size-*`) | ✅ | — |
| Display / overflow | ✅ | — |
| Text alignment (`text-left`, `text-balance`) | ✅ | — |
| Shadcn semantic tokens (`bg-primary`, `text-foreground`) | ✅ | — |
| Semantic radius (`rounded-md`, `rounded-lg`) | ✅ | — |
| Typography scale (`text-sm`, `text-lg`) | ❌ | globals.css body rules; Style Registry entry targeting a `data-slot` |
| Font weight (`font-semibold`, `font-bold`) | ❌ | Same |
| Colour literals (`bg-blue-500`, `bg-[#hex]`) | ❌ | Style Registry entry; new semantic token in globals.css |
| Decorative shadows (`shadow-md`, `shadow-lg`) | ❌ | Style Registry entry |
| Opacity modifiers as aesthetic effect (`bg-primary/50`) | ❌ | Blend modes or `color-mix()` or Style Registry |
| `!important` | ❌ | Never — blocks Style Registry overrides |

### What changed

- **LLM rules page**: new rule 7, "Template Components Use Structure + Semantic Tokens Only." Full allow/deny list, layered-ownership diagram (globals.css / shadcn components / Style Registry / template code), exception criteria (decoration-only components, icon sizing, structural leading), audit state. Common Mistakes table row added.
- **CLAUDE.md**: rule 8 added to Critical Rules Summary. New "Structure-Only Styling Policy" section summarising the rule and linking to the full LLM rules entry.
- **Style System docs page**: caution callout at the top cross-referencing rule 7, so anyone landing on the style-system docs sees the scope of what template code is expected to do (nothing) vs what Style Registry entries are expected to do (everything visual).
- **Roadmap**: new "Structure-only styling migration" entry detailing the ~170 violations catalogued (~158 font-size/weight across 30 files, ~12 colour literals in 5 files, a handful of shadow/tracking/leading) and the migration plan — typography pass, status colour pass, shadow pass, audit script. Each migration ships non-breaking via Style Registry entries registered on template bootstrap.

### No code changes in this release

v1.24.7 ships policy only. Existing template code that violates the rule stays as-is until the migration roadmap items land — no visual regressions, no breaking changes for downstream apps.

**Going forward**: new template code (anything added in PRs from v1.24.7 onward) must comply. Reviewers (human or LLM) flag any PR that adds `text-sm` / `font-semibold` / colour literals / decorative shadows to template files outside `components/ui/`.

### Why this matters for the style module

The style module you're building (useFoundry / downstream) is where these visual decisions actually belong. Rule 7 creates the space for the style module to own them without fighting the template. Style Registry entries register CSS that targets shadcn's `data-slot` / `data-variant` / `data-state` attributes — no component source edits, no template changes, no build-time codegen. The template becomes a structural skeleton; the style module becomes the skin. Swapping skins becomes registry-level, not code-level.

See the [Style System](/starter-template/style-system/) page for the registry architecture and the [worked example of the tabs line variant in the user conversation that prompted this rule](https://saas.uniqueicon.com.au/starter-template/versioning/) for how a style module would ship this specific kind of override.

### Migration for downstream apps

No action required. v1.24.7 is pure policy — no template files change hands, no exports move or disappear. `npm run template:update` pulls the new docs; your app keeps rendering identically.

## v1.24.6

**Deprecation policy + `TabBar` restored as a shim — the template never breaks downstream at head again**

A downstream user pulling v1.24.4 hit `Uncaught SyntaxError: does not provide an export named 'TabBar'` and raised the principled objection: the whole point of a starter template is that consumers shouldn't have to make a wave of manual changes every release. v1.24.4 deleted `TabBar` cold; downstream pages and modules importing it broke at head. That's the wrong contract.

v1.24.6 codifies the right contract — **deprecate, then remove** — and applies it consistently, not just to `TabBar`.

### The policy

Every public export now follows `deprecate → shim → warn → remove in next minor`:

1. **Convert to a shim** that keeps the old API working on top of the new API internally.
2. **Emit a one-time dev-mode warning** via the new `deprecationWarning()` utility, pointing at the replacement.
3. **Mark `@deprecated`** in JSDoc with a concrete migration snippet.
4. **Document in the "Currently deprecated" table** on the new [Versioning & Deprecation](/starter-template/versioning/) page.
5. **Wait at least one minor version** before removing the shim. Deprecating in v1.N.X means the earliest legal removal is v1.(N+1).0.

Breaking changes that genuinely can't be shimmed (rare) force a major bump and still get a one-cycle `@deprecated` warning period before the file is deleted.

**What counts as public**: named exports from `index.ts` files under `@/components`, `@/features`, `@/config`, `@/lib`; registry APIs; `/api/v1/*` routes; Prisma schema models the template ships. **Not public**: internal files not re-exported from a feature's `index.ts`; `@internal` / `@experimental` tagged exports; shadcn-generated `components/ui/*`; test helpers.

### What changed

**New file:** `frontend/src/lib/deprecation.ts` — centralised warning emitter. Dedups per-`id` per-session, gated behind `import.meta.env.DEV` so production bundles contain neither the warnings nor the dedup state. Used by every shim the template ships.

**Restored:** `frontend/src/components/settings-blocks/components/TabBar.tsx` — now a deprecation shim. Takes the original `TabBar` props (`tabs`, `activeTab`, `onTabChange`, `className`) and renders `<Tabs value={...} onValueChange={...}><TabsList variant="line">…</TabsList></Tabs>` internally. Emits a one-time dev-mode warning on first mount:

```
[DEPRECATED] settings-blocks.TabBar: TabBar is deprecated since v1.24.4
and scheduled for removal in v1.25.0. Use <Tabs><TabsList variant="line">…
</TabsList></Tabs> from @/components/ui/tabs instead. See
/starter-template/changelog/#v1244 for the migration snippet.
```

Downstream code importing `TabBar` keeps working with zero edits.

**Restored:** `TabBar`, `TabBarProps`, `TabItem` exports in `frontend/src/components/settings-blocks/index.ts`, marked `@deprecated` so editor tooltips show the strikethrough.

**New docs page:** [`/starter-template/versioning/`](/starter-template/versioning/) — full policy, release cadence, shim pattern reference implementation, currently-deprecated table, FAQ, and LLM-assisted-change guidance. Linked from the sidebar under Starter Template.

**LLM rules page** (`/getting-started/llm-rules/`): **rule 6 rewritten**. Was: "No backwards compatibility — clean removal, bump major version instead." That's the rule that produced the v1.24.4 break. Now: "Deprecate before removing public exports" with the full shim pattern, warning convention, and docs-trail requirements. The old rule had a `❌ WRONG` example of the exact shim pattern v1.24.6 now requires — inverted to `✅ CORRECT` with the TabBar implementation as the reference.

**CLAUDE.md**: added a "Deprecation Policy for Public Exports" section summarising the rules, linking to the full policy page. Added to the Critical Rules Summary list as rule 7.

**Roadmap**: added new Planned entry for ["Deprecation codemods shipped with `template:update`"](/starter-template/roadmap/#deprecation-codemods-shipped-with-templateupdate) — jscodeshift-style migrations that `template:update` can offer to run on first pull of a deprecating release, rewriting consumer imports/JSX so the manual find-and-replace step goes away too.

### Currently deprecated

Reflected on the new versioning page. At the time of this release:

| Export | Deprecated since | Removal target | Replacement |
|--------|-----------------|----------------|-------------|
| `TabBar` (and `TabBarProps`, `TabItem`) from `@/components/settings-blocks` | v1.24.4 (removed) / v1.24.6 (shim reinstated) | v1.25.0 | `<Tabs>` + `<TabsList variant="line">` + `<TabsTrigger>` from `@/components/ui/tabs` |

Nothing else is currently deprecated.

### Migration

Run `npm run template:update` to pick up v1.24.6. Downstream apps with `import { TabBar } from '@/components/settings-blocks'` will stop erroring immediately — the shim resolves and your pages render with shadcn's `variant="line"` treatment under the hood. You'll see the `[DEPRECATED] settings-blocks.TabBar: …` warning once per page in the dev console. Migrate to the shadcn `<Tabs>` composition at your own pace before v1.25.0 ships.

## v1.24.3

**`AuthBackground` slabs now use blend modes — true shades and tints of primary, hue preserved**

Follow-up to v1.24.2. v1.24.2 went bold (primary fills the page) but used `bg-black/N` and `bg-white/N` overlays for the angular slabs. That produces visible tonal variation, but the overlays are rgba-blended, which mixes the primary toward grey and desaturates it. Users who'd configured a vivid brand colour still saw a faintly washed-out version of it on the slabs.

v1.24.3 switches the slab colour strategy to **blend modes on `bg-primary`**. Every slab (including the base) uses the exact same `bg-primary` class. Blend modes do the tonal work:

- `mix-blend-multiply` on a primary slab over a primary base → **primary × primary = darker shade**. Same hue, lower lightness.
- `mix-blend-screen` → **1 − (1 − primary)² = lighter tint**. Same hue, higher lightness.
- `opacity` on the slab scales the effect — a half-opacity multiply slab is a softer shade.

Because the only colour in play is `--color-primary`, the hue is mathematically preserved across every facet. A vivid cyan primary gives cyan + dark-cyan + lighter-cyan facets, no drift toward grey. A near-black primary gives near-black + even-darker + slightly-lighter — still restrained, still correct hue.

### Why blend modes beat color-mix and overlays

| Approach | Hue | Browser support | Token touch points |
|----------|-----|-----------------|--------------------|
| `bg-primary/[0.05]` on `bg-background` (v1.23.0) | Reads grey | Universal | One token |
| `bg-black/N` / `bg-white/N` overlays (v1.24.2) | Desaturates toward grey | Universal | One token |
| `color-mix(in oklch, ..., black X%)` inline styles | Preserved | Chrome 111+ / Safari 16.2+ / FF 113+ | Computed, not a token |
| **Blend modes on `bg-primary`** (v1.24.3) | **Preserved** | **Universal since 2016** | **One token** |

Blend modes were the right hammer for this nail — one colour token everywhere, the browser does the math, works in every shipping browser.

### Migration

Run `npm run template:update` to pick up v1.24.3 and reload the window. The sign-in page should now show your primary colour with darker and lighter facets that stay on-hue.

Same `.auth-page-background*` Style Registry override hooks preserved. If blend modes are wrong for your design (e.g. you want a contrasting accent slab in a different colour), target the individual slab class and set your own `background-color` plus `mix-blend-mode: normal` to override.

## v1.24.1

**Scrub template-level style overrides; let shadcn defaults and your tokens do the work**

Follow-up cleanup to v1.24.0. While wiring up the `BrandingProvider` we realised the template was carrying a handful of CSS rules that overrode shadcn's defaults — harmless in isolation but they made it hard to tell whether a theming issue was "my generated tokens" or "the template is fighting me." This release removes them and leaves the template components using only shadcn's semantic tokens, so your `globals.css` (whether hand-written or produced by a theme generator) is the only source of truth for appearance.

### `globals.css` is now a true stub

Previous template versions shipped:

```css
@theme {
  --color-background: hsl(0 0% 100%);
  --color-foreground: hsl(240 10% 3.9%);
  /* ... 20 more tokens, all in the 'zinc' palette ... */
}

.lucide {
  stroke-width: 1 !important;
}

* { border-color: var(--color-border); }

body {
  background-color: var(--color-background);
  color: var(--color-foreground);
  font-family: system-ui, -apple-system, sans-serif;
}
```

None of that was load-bearing:

- **`@theme { ... }`** — `shadcn init` (which `create-app` runs) rewrites this block with the palette you chose at init time. The template's version was a stub that got overwritten immediately and was only there to make `globals.css` parse in editors. Dropping it avoids any risk that the stub's tokens subtly differ from shadcn's output and confuse consumers.
- **`.lucide { stroke-width: 1 !important }`** — a template-author preference for thinner icons. Not a shadcn default. Apps that want it back can add it to their own `globals.css`.
- **`* { border-color: var(--color-border) }`** — a Tailwind v3 pattern that v4 resolves through token mapping directly. Not needed.
- **`body { ... }`** — `shadcn init` writes its own body rules that incorporate the chosen font, background token, and foreground token. The template's overrides fought shadcn's output.

v1.24.1 strips all of it. `globals.css` in the template is now a single `@import 'tailwindcss';` plus a comment explaining what the file deliberately doesn't contain. The authoritative version on every downstream app is whatever `shadcn init` produced (and v1.23.2's force-sync allow-list protects it from being reset by future `template:update` runs).

### `AuthBackground` opacity bumped 0.03–0.07 → 0.08–0.15

The auth-page decorative slabs used `bg-primary/[0.03]` through `bg-primary/[0.07]` — so faint that even a vivid primary colour read as "hint of tint on gray." If you configured a colourful primary via your theme generator, the sign-in page still looked grey, because the slabs barely expressed the colour.

v1.24.1 bumps the three slabs to `0.08` / `0.12` / `0.15`. Still atmospheric (the `<Card>` remains the focus), but the primary colour now actually reads as primary on any palette. Overridable as before via the `.auth-page-background*` classes if you want a different treatment.

### Why it didn't show up sooner

Default shadcn `zinc` has a near-black primary (`hsl(240 5.9% 10%)`). At 5% opacity that looks like a light gray wash — which reads as "subtle slab," the intended effect. The problem only became obvious once people configured a vivid primary (cyan, violet, teal, etc.) and got a slightly-less-subtle wash of the same character instead of their actual brand colour. Bumping to 8–15% keeps the "subtle" read for dark primaries while actually delivering the colour for vivid ones.

### Migration

## v1.24.0

**Split branding from AppSettings — auth pages now follow admin-configured branding via a public endpoint**

Fixes a v1.23.0 regression. The custom shadcn auth pages that shipped in v1.23.0 (`SignInPage`, `SignUpPage`, `ForgotPasswordPage`, `ResetPasswordPage`) started calling `useAppSettingsValue()` to display the app's name and logo on sign-in — but `AppSettingsProvider` is mounted inside `DashboardPage`, which only renders for authenticated users. Hitting `/auth/sign-in` signed-out threw `useGlobalAppSettings must be used within AppSettingsProvider` and crashed the page with the React Router "Unexpected Application Error" screen.

v1.24.0 splits the public-facing subset of settings into its own provider at the app root, backed by a new public backend endpoint. Auth pages now render with the admin's configured branding without ever needing a session.

### What changed

**Backend:**
- New **public** `GET /api/v1/branding` endpoint (no auth required). Returns `{ appName, logoUrl, logoDisplayMode, styleOverrides }` — the subset auth pages can safely see. Registered as a separate router in `admin.routes.ts` so the existing `adminRouter` can keep `requiresAuth: true`.
- Nav overrides, toolbar overrides, user management, and all write endpoints stay authenticated. No changes to the admin DB schema.

**Frontend:**
- New `features/admin/context/BrandingContext.tsx` — `BrandingProvider` + `useBranding()`. Fetches from `/api/v1/branding` via TanStack Query with a 5-minute stale window. On resolution, also calls `styleRegistry.applyOverrides(styleOverrides)` so admin-configured theme tokens and CSS (e.g. `.auth-page-background` targeting) apply to auth pages too.
- `App.tsx` — wraps `<RouterProvider>` with `<BrandingProvider>` at the root.
- Auth pages (`SignInPage`, `SignUpPage`, `ForgotPasswordPage`, `ResetPasswordPage`) switched from `useAppSettingsValue()` to `useBranding()`. The hook is more forgiving — returns `DEFAULT_BRANDING` (app name from `VITE_APP_NAME`) if the provider isn't mounted, rather than throwing.
- `AppSettingsProvider` stays where it is (inside `DashboardPage`). Still owns nav overrides, toolbar overrides, feature flags, and the admin-side view of styleOverrides. Still calls `applyOverrides` on its own fetch — idempotent with Branding's call, and the re-apply flow for admin UI updates is unchanged.
- New `branding` key on `appQueryKeys` so the two providers don't collide on cache entries.

### When auth pages see the real branding

`BrandingProvider` fetches on mount — i.e. the first paint of any page, including sign-in. So:

- **Fresh visitor (never signed in):** first paint uses `DEFAULT_BRANDING` (from `VITE_APP_NAME`); the fetch resolves within ~100ms and the page re-renders with DB-stored branding.
- **Returning signed-out visitor:** TanStack Query serves the cached branding from memory, so the first paint already has the admin's branding. styleOverrides stay consistent (no reset-on-sign-out weirdness).
- **Authenticated visitor:** same public fetch runs, plus `AppSettingsProvider` inside the shell fetches the full admin surface. Both providers call `applyOverrides` — idempotent.

### Exposure tradeoff

`styleOverrides` on the public endpoint is a map of `{ entryId: boolean }` — which styleRegistry entries the admin has toggled. It's necessary for auth pages to theme correctly (tokens + CSS from module-contributed entries), but it does reveal *which* style entries exist and their on/off state. That's a small information leak (module names only, no module code or admin data). If that's a concern for your deployment, override the endpoint in a downstream module to filter the map to a known-public allowlist. The typical starter deployment accepts the tradeoff for the sign-in UX.

### Migration

1. Run `npm run template:update` to pick up v1.24.0 (will also run `install:all`).
2. No database migration required — the backend reads the same `AppSettings` row it always has; only the frontend provider topology changed.
3. No action needed for existing modules. Code calling `useAppSettingsValue()` still works unchanged inside the authenticated shell. If a module has code that could render on unauthenticated routes (rare — most module UI is behind auth) and it reads only branding fields, prefer `useBranding()` — but it's a soft recommendation, not a breaking change.

### Why minor, not patch

New public API surface (`/api/v1/branding`), new frontend hook (`useBranding`), new provider in the root tree. Downstream apps that pulled v1.23.0 or later need a frontend change (provider at the root) on merge — minor captures "new capability plus a small structural shift" more honestly than patch.

## v1.23.3

**Fix: `template:update` leaves the merge stuck when the `docs` submodule pointer bumps**

Second follow-up to v1.23.1. The script already knows `docs/`, `.claude/`, and `.template-archive/` are template-only paths that shouldn't leak into downstream apps — it strips them post-merge in the success branch. But that cleanup never ran when git raised a modify/delete conflict during the merge itself, which happens every time the template's `docs` submodule pointer moves and a downstream app has (correctly) removed its `docs` directory. The symptom: `npm run template:update` reports a conflict, VS Code's source control panel shows `docs` as a pending merge change with a `!`, and the user is stuck until they manually run `git rm -rf --cached docs && rm -rf docs && git commit`.

### What changed

Extended the conflict auto-resolution branch in both `update-template.ps1` and `update-template.sh` to handle the excluded paths — same shape as the existing `create-app.*` auto-removal. For each of `docs`, `.claude`, `.template-archive`: if the path is in an unmerged state or present in the working tree, remove it (including the submodule registration for `docs` — `git rm -rf --cached`, then wipe the directory, then strip the `.gitmodules` entry). Once the conflict is cleared, the script completes the merge and continues to the normal post-merge flow.

### Why the v1.23.1 logic missed this

The post-merge cleanup is gated on `git merge` returning exit code 0. A modify/delete conflict on `docs` makes `git merge` exit non-zero, so execution diverts into the conflict-handling branch and the excluded-paths cleanup is skipped. The original conflict branch only auto-resolved `create-app.*` — which meant `docs` conflicts always required manual intervention, even though the template never intends the app to carry `docs` in the first place.

### When you'll hit this

- Every release where the docs submodule pointer moves (which is most of them — docs updates ship alongside code changes).
- First `template:update` after a docs-pointer change if your app has ever removed `docs`.
- Apps on v1.23.1 or v1.23.2 pulling v1.23.3 specifically — both prior versions have the docs-pointer bump.

### Migration

Run `npm run template:update` to pick up v1.23.3. If you're currently **stuck mid-merge** with a `docs` conflict from a v1.23.1/v1.23.2 update, the one-off manual unstick is:

```bash
git rm -rf --cached docs
rm -rf docs
git config -f .gitmodules --remove-section submodule.docs 2>/dev/null || true
git add -A
git commit --no-edit
```

That completes the current merge. From v1.23.3 onward the script does this automatically.

## v1.23.1

**`template:update` now enforces template authority on core files**

Fixes a class of merge bug where `npm run template:update` leaves stale code in downstream apps because git's three-way merge resolves the update surprisingly. Specifically, when a previous template version added lines to a file and a newer release rewrote the same file, git can keep the local (pre-rewrite) version even though the template's current version is the intended state.

This was discovered when a downstream app updated v1.22.2 → v1.23.0 and came out of the merge still carrying the `import '@/styles/auth.css';` line in `frontend/src/features/auth/index.ts` — a line added by the withdrawn v1.22.2 attempt and removed in v1.23.0. The dev server failed to start because `styles/auth.css` itself no longer existed.

### What changed

**Post-merge, `update-template.{sh,ps1}` now force-syncs template-owned files to the template's current version.** The algorithm:

1. Git merge runs as before.
2. After a successful merge, the script walks every file the template changed between the merge-base and the template's new HEAD (`git diff --name-only <base>..<template-head>`).
3. For each changed file:
   - If the path is in a **user-owned allow-list** (see below), it's preserved as-is.
   - Otherwise, the template's version is force-checked-out and staged (`git checkout <template-head> -- <file>`).
4. The merge commit is amended to include the force-synced content, so history stays clean — one merge commit per `template:update`.

### User-owned allow-list

These paths are template-provided but user-customised, so they're preserved even if the template changed them. The script warns when any of them differ from the template's current version so you can reconcile manually if needed.

- `frontend/components.json` — mutated by `npm run components:sync` as modules contribute shadcn components
- `frontend/src/styles/globals.css` — _(added in v1.23.2)_ the template ships a minimal stub; `shadcn init` rewrites it with the full light + dark token set
- `backend/prisma/schema.prisma` and `backend/prisma/schema/*.prisma` — user models live here
- `backend/prisma/migrations/*` — always user-owned, generated from user's schema changes
- `package.json` / `frontend/package.json` / `backend/package.json` — app-specific name, version, dependencies
- `*-lock.json` — regenerated by npm anyway
- `.env` / `.env.local` — secrets
- `README.md` / `CLAUDE.md` — per-app project readmes and Claude Code instructions

### Excluded paths

These are stripped from the working tree if the merge pulls them in. They're template-only and have no business in downstream apps:

- `docs/` — the module-system docs submodule, only useful in the template repo itself
- `.claude/` — user-specific Claude Code config and permission allowlists
- `.template-archive/` — reserved for the [archive-on-removal](/starter-template/roadmap/#archive-removed-files-during-template-update) roadmap feature

### Why force-take is safe

The template's contract has always been "use registries, don't edit core files." Template-authoritative files (auth, admin, user, dashboard features; shared UI libraries; config registries; middleware; base library code) are not meant to be customised per-app — that's what registries, filter/action hooks, and module manifests are for. Force-syncing these on update makes the contract load-bearing at `template:update` time instead of relying on discipline.

If your app genuinely needs to diverge from a template-owned file beyond registry extension points, the upcoming [`.template-ignore`](/starter-template/roadmap/#exclude-template-only-paths-from-templateupdate-merges) mechanism (roadmapped) will let you opt paths out of the force-sync. Until then, if you hit a case where force-sync discards a customisation you actually need, please flag it — it probably means either the path should be in the allow-list, or we need a registry extension point.

### Retrospective fix for existing affected apps

This release **also fixes any app already stuck from the v1.22.2 → v1.23.0 merge pattern**. Running `npm run template:update` to pick up v1.23.1 will re-sync every file the template has changed since your last clean merge — including `features/auth/index.ts`, which will be overwritten with the v1.23.0 version that doesn't import `auth.css`. You should not need to manually edit anything.

### Migration

No action required beyond running `template:update`. The new behaviour is automatic from v1.23.1 onward.

## v1.22.3

**Withdraw the v1.22.2 cascade-layer attempt; fix the root cause in v1.23.0**

No code changes beyond a version bump. This release exists to give downstream apps a clean floor to roll off the experimental v1.22.2 work onto, which only shipped locally (never published to the starter remote) but reached at least one downstream app through a local template pull.

### What v1.22.2 attempted

`@neondatabase/neon-js/ui/tailwind` is the Neon-documented CSS entry point for Tailwind v4 SPAs. Sitting in the unlayered default of `globals.css`, its base/utility rules targeting ARIA roles (`tablist`, `tab`, `tabpanel`) were leaking onto shadcn primitives that share those roles — the reported shadcn `Tabs` layout bug downstream modules hit. v1.22.2 tried to scope the preset into its own CSS cascade layer (`@layer neon-auth`) so shadcn and any Style Registry contribution would win cascade battles.

### Why it didn't work

Tailwind v4 does not allow `@source` directives to be nested inside a `@layer`. The Neon preset contains `@source` directives (to scan Neon's own component files for class usage), so wrapping `@import '@neondatabase/neon-js/ui/tailwind' layer(neon-auth)` triggers a build-time error:

```
[plugin:@tailwindcss/vite:generate:serve] `@source` cannot be nested.
```

We can't layer the preset. And we can't layer Tailwind either — `@import 'tailwindcss' layer(...)` fails for the same reason (Tailwind's own entry point uses `@theme` declarations that also can't be nested in a layer). Shadcn itself has no bundled CSS to layer — it's utility classes inline in React components. There is no layer-based fix that keeps the Neon preset.

### The real fix (in v1.23.0)

Replace Neon's pre-built UI components (`AuthView`, `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, `RedirectToSignIn`) and the `NeonAuthUIProvider` wrapper with custom shadcn-based forms that call the underlying auth client (`@neondatabase/neon-js/auth`) directly. The CSS conflict evaporates entirely — no Neon preset needed, `globals.css` goes back to a single `@import 'tailwindcss'`, auth pages match the rest of the app's design. Scope and plan are captured in the [Custom shadcn auth pages](/starter-template/roadmap/#custom-shadcn-auth-pages) roadmap entry.

### Why scope of v1.22.2 stayed small in practice

The audit documented in the v1.23.0 roadmap entry found that `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, and `RedirectToSignIn` are re-exported from `features/auth/index.ts` but not imported anywhere in the codebase — the settings pages are already fully custom shadcn (see `SecurityPage.tsx`, `PersonalInformationPage.tsx`, `DataPrivacyPage.tsx`). Only `AuthView` (inside `AuthPage.tsx`) and `NeonAuthUIProvider` (inside `main.tsx`) have real consumers. The v1.23.0 work is therefore bounded to four new auth page files plus a small amount of cleanup.

### Migration

Apps on the experimental v1.22.2 will see `frontend/src/styles/auth.css` deleted, `frontend/src/styles/globals.css` restored to its v1.22.1 shape (single `@import 'tailwindcss';`), and the `import '@/styles/auth.css'` line removed from `frontend/src/features/auth/index.ts`. ## v1.22.1

**Auto-run `prisma migrate dev` on schema-updating template pulls**

`update-template.{sh,ps1}` now mirrors the module starter's post-schema-update flow ([`src/scaffold/update.ts`](https://github.com/ui-git-kim/unique-react-fullstack-module-starter) — `runPrismaAfterSchemaUpdate`): when a template update introduces new Prisma models and sets `NEEDS_MIGRATION=true`, the script runs `cd backend && npx prisma migrate dev --name template_update_v<version>` automatically after `npm run install:all` completes. Previously the scripts only printed the command and left the user to run it manually.

### What's new

- **Auto-migrate on schema changes** — when the update merges in new models (detected via the existing `MODELS_ADDED` logic), the script runs the migration with a versioned name derived from the template version (e.g. `template_update_v1_22_1`) so migration history is traceable.
- **Non-fatal failure** — wrapped in try/catch; if Prisma fails (missing `DIRECT_DATABASE_URL`, dev DB unreachable, conflicting state) the script logs a warning with the exact command to retry and continues to the "Next Steps" summary. The update merge itself is never reverted on migration failure.
- **Interactive stdio** — stdio is inherited so Prisma's own prompts (data-loss confirmations, generator downloads) work the same as running the command by hand.
- **Dedup with Next Steps** — the "Run database migration" reminder is suppressed when the auto-run succeeded, and falls back to the versioned command (not the old `add_template_models` name) when it didn't.

### Why

## v1.21.0

**Style Registry + `components.json` sync pipeline**

A single `styleRegistry` for modules to contribute both **runtime CSS overrides** (tokens + raw CSS) and **build-time shadcn `components.json`** values (style, base colour, cssVariables, registries, aliases). Complements the existing `theme.cssVariables` filter and `theme.changed` action hooks from v1.20.1.

Full reference on the [Registry Reference page](/starter-template/registries/#style-registry); terminal script usage on [CLI Scripts](/starter-template/scripts/#sync-shadcn-components); toggle reinstall UI is [on the roadmap](/starter-template/roadmap/#in-app-component-reinstall-for-style-registry).

### What's new

**Frontend — `frontend/src/config/style.registry.ts`** + **`frontend/src/lib/style-pipeline.ts`**

- Imperative API: `register`, `update`, `unregister`, `setOverride`, `clearOverride`, `applyOverrides`.
- Exclusivity enforced at registration for `componentsJson.style`, `componentsJson.tailwind.baseColor`, `componentsJson.tailwind.cssVariables` — a second entry claiming the same field throws `StyleRegistryError` with code `EXCLUSIVE_FIELD_CLAIMED`. Additive fields (`aliases`, `registries`) merge with last-write-wins and a `console.warn` on overlap.
- Pipeline merges tokens by `order`, pipes through the existing `theme.cssVariables` filter, applies via `applyThemeVariables()`, and writes concatenated raw CSS into a single `<style id="module-styles">` tag appended to `<head>` (so module CSS wins over `globals.css` on specificity ties).
- **Bug fixed in the process:** `applyThemeVariables` only ever *set* CSS variables on `:root` — stale values were never cleared. The new pipeline tracks previously-applied tokens and clears any that aren't in the next merge, so disabling a module reliably reverts its tokens.

**Backend — `AppSettings.styleOverrides`**

New `Json` field on the `AppSettings` model persisting per-entry admin toggles (`Record<entryId, boolean>`). Modules ship `enabled: true` by default; admin overrides win. `AppSettingsContext` calls `styleRegistry.applyOverrides(map)` on load, so the pipeline honours the admin's state before modules finish registering.

New endpoints:

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| `GET` | `/api/v1/admin/style-overrides` | Authenticated | Read full overrides map (frontend bootstrap) |
| `PUT` | `/api/v1/admin/style-overrides` | Admin only | Replace full overrides map |

Action hook `admin.styleOverridesUpdated` fires after a successful update so modules can react (e.g. invalidate caches).

**Sync script — `npm run components:sync`**

PowerShell + Bash scripts at `scripts/sync-components.{ps1,sh}`. Reads `componentsJsonContribution` from each enabled module in `.module-manifests.json`, enforces exclusivity, merges into `frontend/components.json` atomically, then runs `npx shadcn@latest add --overwrite` for the union of `installedShadcnComponents` from `.module-state.json`. A `--dry-run` / `-DryRun` mode previews without writing. Bash version requires `jq`.

```bash
npm run components:sync        # merge manifests → components.json → reinstall
npm run components:sync:dry    # preview only
```

### Migration for downstream apps

1. `npm run template:update` — merges the new `AppSettings.styleOverrides` field into your Prisma schema (also updates the multi-file `app-settings.prisma` heredoc for apps that were on multi-file mode but hadn't added AppSettings yet).
2. `cd backend && npx prisma migrate dev --name add_style_overrides` — creates the DB migration.
3. `npm run components:sync` — no-op if you have no modules contributing to `componentsJson`; otherwise applies their contributions.

### Module contract

Modules that want to contribute `componentsJson` fields declare a `componentsJsonContribution` entry in their `.module-manifests.json` **in addition to** calling `styleRegistry.register({ componentsJson: {...} })` at module load time. The registry copy is for runtime validation (exclusivity) and admin UI display; the manifest copy is what the sync script reads. These should match — a divergence means the runtime app sees different config than shadcn actually wrote out.

### Known limitation

There's a brief flash on first paint if an admin disabled a module-registered entry — the module's tokens apply at registration time, then the `applyOverrides` call removes them once the AppSettings fetch completes. Fixable with a localStorage cache of `styleOverrides` at bootstrap (same pattern as the theme class init in `main.tsx`). Not critical; can land in a follow-up patch.

```json
}
```

**After:**

```json
"paths": {
  "@/*": ["./src/*"]
}
```

No runtime or build behaviour changes — this only silences the deprecation warning and future-proofs for TypeScript 7.0.

The `create-app` script now asks whether to use single-file or multi-file Prisma schema during app creation.

**Impact on existing apps:** None. Existing single-file apps continue to work unchanged. The update scripts auto-detect which mode your app uses.

## v1.16.0

**Personal Info Fields Registry: Module Preferences Now Save Correctly**

Fixed a critical issue where module-registered personal info fields with `storage: 'preferences'` (the default) were rejected by the backend Zod validation and failed silently in the UI.

**Root Cause (3 compounding issues):**

1. **Backend Zod schema used `.strict()`** — The `UpdatePreferencesSchema` rejected any field not explicitly listed, returning a 400 error for module-registered preference keys (e.g. `handle`, `socialTwitter`). Changed to `.passthrough()` so module-registered fields pass through validation.

2. **`handleFieldChange` swallowed errors** — When the API returned an error, the `onChange` callback caught and logged the error but did not re-throw. Custom field components (like the Blocks Builder's `HandleField`) that called `await onChange(value)` never saw the failure, showing "Saved" when the save actually failed.

3. **Text/textarea fields fired API calls on every keystroke** — No debounce, causing rapid-fire `PATCH` requests with potential race conditions.

**Fixes:**

- Backend `UpdatePreferencesSchema` now uses `.passthrough()` instead of `.strict()`
- Backend `UserPreferences` and `UpdatePreferencesInput` types now include `[key: string]: unknown` index signatures, matching the frontend types and the Zod `.passthrough()` behaviour
- `handleFieldChange` now re-throws errors after reverting, so custom components can catch failures and display their own error UI (e.g. toast notifications)
- `PersonalInfoFieldProps.onChange` type changed from `(value: unknown) => void` to `(value: unknown) => Promise<void>` to reflect async save behaviour
- Text and textarea registry fields now use debounced saves (600ms delay) instead of firing API calls on every keystroke
- Built-in field types (text, textarea, select) gracefully handle re-thrown errors

**Files Modified:**
- `backend/src/features/user/user.types.ts` — Zod schema + type index signatures
- `frontend/src/config/personal-info-fields.registry.ts` — `onChange` type signature
- `frontend/src/features/user/pages/PersonalInformationPage.tsx` — Error re-throw + debounced text fields

**Impact on modules:**
- Modules registering `personalInfoFieldsRegistry` fields with `storage: 'preferences'` now work correctly out of the box
- Custom field components should `await onChange(value)` and handle errors (already the expected pattern)
- No breaking changes — existing module code that calls `onChange()` without `await` continues to work

## v1.14.8

**Dynamic Breadcrumb Support**

Added a new `BreadcrumbContext` that allows pages to set dynamic breadcrumbs that display in the site header.

**New Components:**
- `BreadcrumbProvider` - Wraps the app and manages breadcrumb state
- `useBreadcrumbs()` - Hook to set dynamic breadcrumbs from any component
- `useBreadcrumbsOptional()` - Safe hook that returns undefined if not in provider

**Features:**
- Dynamic breadcrumbs automatically override route-based breadcrumbs when set
- Breadcrumbs auto-clear on route change (no stale breadcrumbs)
- Supports clickable links via optional `path` property on breadcrumb items

**Usage:**
```tsx
import { useBreadcrumbs } from '@/components/layout';

function MyDetailPage() {
  const { setBreadcrumbs } = useBreadcrumbs();
  const { data } = useMyData();

  useEffect(() => {
    if (data) {
      setBreadcrumbs([
        { label: 'Items', path: '/items' },
        { label: data.name }
      ]);
    }
  }, [data, setBreadcrumbs]);
}
```

**Files Added:**
- `frontend/src/components/layout/BreadcrumbContext.tsx`

**Files Modified:**
- `frontend/src/components/layout/index.ts` - Added exports
- `frontend/src/features/dashboard/pages/DashboardPage.tsx` - Added provider
- `frontend/src/features/dashboard/components/header/SiteHeader.tsx` - Uses dynamic breadcrumbs

## v1.14.4

**Admin Routes Registration Fix**

Fixed a bug where admin routes (PATCH, navigation, toolbar, users) were not being mounted due to duplicate path detection in the route registry.

**Problem:**
- Admin feature registered two routers on `/admin`: one for public GET (settings) and one for admin-only operations
- Route registry's duplicate detection skipped the second registration
- Result: Only `GET /api/v1/admin/settings` worked; all write operations returned 404

**Fix:**
- Merged both routers into a single router
- Applied `requireAdmin` middleware per-route instead of per-router
- `GET /settings` remains accessible to all authenticated users (for sidebar branding)
- All other admin routes properly require admin role

**Affected endpoints now working:**
- `PATCH /api/v1/admin/settings`
- `GET/PATCH /api/v1/admin/navigation`
- `GET/PATCH /api/v1/admin/toolbar`
- `GET /api/v1/admin/users`
- `PATCH /api/v1/admin/users/:userId/role`
- `DELETE /api/v1/admin/users/:userId`

## Versioning Infrastructure

Version bumps happen automatically for infrastructure changes. See [LLM Rules](/getting-started/llm-rules/) for the complete policy.

**Files requiring version bump:**
- `frontend/src/config/*.ts` - Registry files
- `frontend/src/lib/*.ts` - Utility libraries
- `backend/src/config/*.ts` - Registry files
- `backend/src/lib/*.ts` - Utility libraries
- `backend/src/middleware/*.ts` - Middleware
- `scripts/*.ps1`, `scripts/*.sh` - CLI scripts

**Bump commands:**
```bash
# Patch (bug fixes)
npm run template:version:patch

# Minor (new features)
npm run template:version:minor

# Major (breaking changes)
npm run template:version:major
```
