This page documents notable changes to the starter template. The template uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, documentation updates

## v1.23.2

**Fix: `template:update` force-sync was wiping shadcn-generated `globals.css`**

A follow-up to v1.23.1. The force-sync allow-list was missing `frontend/src/styles/globals.css`, so every app that had run `shadcn init` (which all of them have, via `create-app`) saw its fully-populated `globals.css` — with the light **and** dark token palette, `@custom-variant dark`, font imports, any shadcn-added chart colours — replaced by the template's minimal stub on every `template:update`. The symptom downstream: the app looks like it's rendered in "no theme" mode, dark-mode toggle does nothing visible, and anything relying on `--color-*` tokens under `.dark` breaks silently.

### What changed

Added `frontend/src/styles/globals.css` to the user-owned allow-list in both `update-template.ps1` and `update-template.sh`. Same semantics as `components.json`, `schema.prisma`, `package.json`, etc. — preserved as-is, with a warning if the template also changed it so you can reconcile manually if needed.

### Why it's user-owned

The template only ships a minimal stub for `globals.css` to pass lint and TypeScript in the template repo itself (where it's never run). `shadcn init` rewrites it on downstream install with the full shadcn v4 token set. From that point on, the downstream app owns `globals.css` — any changes the template makes to its stub should not silently overwrite the user's populated version.

This is the same category of file as `frontend/components.json` (also rewritten by `shadcn init`, already on the allow-list). v1.23.1 caught `components.json` but missed `globals.css`.

### Retrospective fix for affected apps

If your app's `globals.css` was stomped during a v1.23.1 `template:update`, recover it by re-running:

```bash
cd frontend
npx shadcn@latest init --yes --defaults --force
```

`--force` is required because `components.json` already exists. This rewrites `globals.css` with the full palette (and leaves everything else alone). Then pull v1.23.2:

```bash
cd ..
npm run template:update
```

From v1.23.2 onward the allow-list will protect it.

### Migration

No action required beyond running `template:update` to pick up v1.23.2, plus the one-off shadcn re-init if your app's CSS is currently broken (see above).

## v1.23.1

**`template:update` now enforces template authority on core files**

Fixes a class of merge bug where `npm run template:update` leaves stale code in downstream apps because git's three-way merge resolves the update surprisingly. Specifically, when a previous template version added lines to a file and a newer release rewrote the same file, git can keep the local (pre-rewrite) version even though the template's current version is the intended state.

This was discovered when a downstream app updated v1.22.2 → v1.23.0 and came out of the merge still carrying the `import '@/styles/auth.css';` line in `frontend/src/features/auth/index.ts` — a line added by the withdrawn v1.22.2 attempt and removed in v1.23.0. The dev server failed to start because `styles/auth.css` itself no longer existed.

### What changed

**Post-merge, `update-template.{sh,ps1}` now force-syncs template-owned files to the template's current version.** The algorithm:

1. Git merge runs as before.
2. After a successful merge, the script walks every file the template changed between the merge-base and the template's new HEAD (`git diff --name-only <base>..<template-head>`).
3. For each changed file:
   - If the path is in a **user-owned allow-list** (see below), it's preserved as-is.
   - Otherwise, the template's version is force-checked-out and staged (`git checkout <template-head> -- <file>`).
4. The merge commit is amended to include the force-synced content, so history stays clean — one merge commit per `template:update`.

### User-owned allow-list

These paths are template-provided but user-customised, so they're preserved even if the template changed them. The script warns when any of them differ from the template's current version so you can reconcile manually if needed.

- `frontend/components.json` — mutated by `npm run components:sync` as modules contribute shadcn components
- `frontend/src/styles/globals.css` — _(added in v1.23.2)_ the template ships a minimal stub; `shadcn init` rewrites it with the full light + dark token set
- `backend/prisma/schema.prisma` and `backend/prisma/schema/*.prisma` — user models live here
- `backend/prisma/migrations/*` — always user-owned, generated from user's schema changes
- `package.json` / `frontend/package.json` / `backend/package.json` — app-specific name, version, dependencies
- `*-lock.json` — regenerated by npm anyway
- `.env` / `.env.local` — secrets
- `README.md` / `CLAUDE.md` — per-app project readmes and Claude Code instructions

### Excluded paths

These are stripped from the working tree if the merge pulls them in. They're template-only and have no business in downstream apps:

- `docs/` — the module-system docs submodule, only useful in the template repo itself
- `.claude/` — user-specific Claude Code config and permission allowlists
- `.template-archive/` — reserved for the [archive-on-removal](/starter-template/roadmap/#archive-removed-files-during-template-update) roadmap feature

### Why force-take is safe

The template's contract has always been "use registries, don't edit core files." Template-authoritative files (auth, admin, user, dashboard features; shared UI libraries; config registries; middleware; base library code) are not meant to be customised per-app — that's what registries, filter/action hooks, and module manifests are for. Force-syncing these on update makes the contract load-bearing at `template:update` time instead of relying on discipline.

If your app genuinely needs to diverge from a template-owned file beyond registry extension points, the upcoming [`.template-ignore`](/starter-template/roadmap/#exclude-template-only-paths-from-templateupdate-merges) mechanism (roadmapped) will let you opt paths out of the force-sync. Until then, if you hit a case where force-sync discards a customisation you actually need, please flag it — it probably means either the path should be in the allow-list, or we need a registry extension point.

### Retrospective fix for existing affected apps

This release **also fixes any app already stuck from the v1.22.2 → v1.23.0 merge pattern**. Running `npm run template:update` to pick up v1.23.1 will re-sync every file the template has changed since your last clean merge — including `features/auth/index.ts`, which will be overwritten with the v1.23.0 version that doesn't import `auth.css`. You should not need to manually edit anything.

### Migration

No action required beyond running `template:update`. The new behaviour is automatic from v1.23.1 onward.

## v1.23.0

**Custom shadcn auth pages — replaces the Neon UI package, fixes the shadcn Tabs layout bug**

Replaces every Neon-provided auth UI component the starter used (`AuthView`, `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, `RedirectToSignIn`) and the `NeonAuthUIProvider` wrapper with four custom shadcn-based auth pages. The `@neondatabase/neon-js/ui/tailwind` preset is no longer imported from `globals.css`, which is the real fix for the shadcn `Tabs` layout bug that v1.22.2 tried and failed to scope via CSS cascade layers (Tailwind v4 disallows `@source` inside a `@layer` — the preset contains `@source` directives, so layering it was never going to work).

### Why this shape

The [audit in this changelog entry's roadmap precursor](/starter-template/roadmap/#finalise-the-forgot-password--reset-password-flow) found that `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, and `RedirectToSignIn` were re-exported from `features/auth/index.ts` but never imported anywhere — settings pages already call `authClient` methods directly (`SecurityPage.tsx`, `PersonalInformationPage.tsx`, `DataPrivacyPage.tsx`). Only `AuthView` (in `AuthPage.tsx`) and `NeonAuthUIProvider` (in `main.tsx`) had real consumers. So replacing the UI layer was a four-new-page rewrite plus a small cleanup, not a sweeping migration.

### New files

- **`frontend/src/features/auth/components/AuthBackground.tsx`** — decorative angular slabs tinted with the `primary` token. Three overlapping polygons at varying opacities (0.03 / 0.05 / 0.07). Subtle by default so it reads as atmosphere, not decoration. `.auth-page-background` class and per-slab classnames are available as Style Registry override hooks.
- **`frontend/src/features/auth/pages/SignInPage.tsx`** — email + password + Google OAuth + "Forgot password?" link. Uses `authClient.signIn.email` and `authClient.signIn.social({ provider: 'google' })`. Logo + app name pulled from `useAppSettingsValue()` so auth pages follow whatever branding the admin has configured.
- **`frontend/src/features/auth/pages/SignUpPage.tsx`** — name + email + password + Google OAuth. Uses `authClient.signUp.email`. Shows a "check your inbox" state when Neon's project has email verification required.
- **`frontend/src/features/auth/pages/ForgotPasswordPage.tsx`** — email input, calls `authClient.requestPasswordReset({ email, redirectTo })`. See "Pre-production TODOs" below.
- **`frontend/src/features/auth/pages/ResetPasswordPage.tsx`** — reads the reset token from `?token=...`, submits new password via `authClient.resetPassword`. See "Pre-production TODOs" below.

### Rewritten / modified files

- **`frontend/src/features/auth/pages/AuthPage.tsx`** — was a thin wrapper around Neon's `<AuthView pathname={pathname} />`; now a switch statement that matches `pathname` to one of the four page components above. Unknown pathnames redirect to sign-in.
- **`frontend/src/main.tsx`** — removed the `<NeonAuthUIProvider>` wrapper. `authClient.useSession` and all client methods work standalone via `BetterAuthReactAdapter` on the client itself — the provider only existed to configure `AuthView`'s appearance, which we no longer render.
- **`frontend/src/features/auth/index.ts`** — removed the dead `SignedIn` / `SignedOut` / `UserButton` / `RedirectToSignIn` re-exports and the unused `AccountSettings` wrapper. Added exports for the four new pages.
- **`frontend/src/styles/globals.css`** — removed `@import '@neondatabase/neon-js/ui/tailwind';`. A comment in its place documents why it's not imported, so anyone reintroducing a Neon UI component understands the tradeoff they're making with Tabs.

### Deleted files

- **`frontend/src/features/auth/components/AccountSettings.tsx`** — unused wrapper around Neon's `<AccountView>`. Settings pages handle account management directly via custom shadcn UI calling `authClient` methods.

### Pre-production TODOs (forgot / reset password)

`ForgotPasswordPage` and `ResetPasswordPage` call `authClient.requestPasswordReset()` and `authClient.resetPassword()` — the method names documented in Better Auth 1.6. **Neon's server is on Better Auth 1.4.6 until their upgrade lands** ("coming weeks" per the Neon team on Discord, 2026-04-15), and Neon's docs flag SDK methods for password reset as "not fully supported yet." The forms are wired correctly against the documented SDK surface, but haven't been verified against Neon's live server.

Before any v1.23.0-based app ships to production:

1. Trigger a forgot-password from the running app and confirm the reset email arrives.
2. Click the emailed link and confirm it lands on `/auth/reset-password?token=...` with a real token.
3. Submit a new password, sign out, and confirm the new password works on sign-in.

If any step fails, the fallback plan is Neon's standalone `<ForgotPasswordForm>` / `<ResetPasswordForm>` components under a scoped CSS loader until Neon 1.6 ships. Tracked in the [Finalise the forgot-password / reset-password flow](/starter-template/roadmap/#finalise-the-forgot-password--reset-password-flow) roadmap entry.

### Also in v1.23.0

- **"Remember this device" checkbox present but hidden** (feature flag `SHOW_REMEMBER_ME = false` in `SignInPage.tsx`). Better Auth documents `rememberMe` on `signIn.email()` — flip the flag once session-persistence behaviour is verified end-to-end on Neon's server.
- **Logo / app name** on every auth page come from `useAppSettingsValue()` — auth pages follow whatever branding the admin panel has configured.

### Migration

Public API breaking: `features/auth` no longer exports `SignedIn`, `SignedOut`, `UserButton`, `RedirectToSignIn`, or `AccountSettings`. Codebase-wide grep at audit time found zero consumers across the template and the in-tree modules, so no actual migration should be needed. If `npm run template:update` reports a module or app-level file importing any of these names, replace with one of:

- `SignedIn` / `SignedOut` → `const { data } = useSession(); if (!data?.user) return null;` (and the inverse)
- `UserButton` → the starter's own `NavUser` component in the dashboard sidebar, or build a shadcn dropdown with the session data
- `RedirectToSignIn` → `<Navigate to="/auth/sign-in" replace />`
- `AccountSettings` → the existing `SecurityPage` / `PersonalInformationPage` / `DataPrivacyPage` under `/settings`, which were already doing this work without the Neon wrapper

Everything else is transparent — `useSession`, `authClient`, `getAuthToken`, protected-route middleware, backend JWKS verification all unchanged.

## v1.22.3

**Withdraw the v1.22.2 cascade-layer attempt; fix the root cause in v1.23.0**

No code changes beyond a version bump. This release exists to give downstream apps a clean floor to roll off the experimental v1.22.2 work onto, which only shipped locally (never published to the starter remote) but reached at least one downstream app through a local template pull.

### What v1.22.2 attempted

`@neondatabase/neon-js/ui/tailwind` is the Neon-documented CSS entry point for Tailwind v4 SPAs. Sitting in the unlayered default of `globals.css`, its base/utility rules targeting ARIA roles (`tablist`, `tab`, `tabpanel`) were leaking onto shadcn primitives that share those roles — the reported shadcn `Tabs` layout bug downstream modules hit. v1.22.2 tried to scope the preset into its own CSS cascade layer (`@layer neon-auth`) so shadcn and any Style Registry contribution would win cascade battles.

### Why it didn't work

Tailwind v4 does not allow `@source` directives to be nested inside a `@layer`. The Neon preset contains `@source` directives (to scan Neon's own component files for class usage), so wrapping `@import '@neondatabase/neon-js/ui/tailwind' layer(neon-auth)` triggers a build-time error:

```
[plugin:@tailwindcss/vite:generate:serve] `@source` cannot be nested.
```

We can't layer the preset. And we can't layer Tailwind either — `@import 'tailwindcss' layer(...)` fails for the same reason (Tailwind's own entry point uses `@theme` declarations that also can't be nested in a layer). Shadcn itself has no bundled CSS to layer — it's utility classes inline in React components. There is no layer-based fix that keeps the Neon preset.

### The real fix (in v1.23.0)

Replace Neon's pre-built UI components (`AuthView`, `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, `RedirectToSignIn`) and the `NeonAuthUIProvider` wrapper with custom shadcn-based forms that call the underlying auth client (`@neondatabase/neon-js/auth`) directly. The CSS conflict evaporates entirely — no Neon preset needed, `globals.css` goes back to a single `@import 'tailwindcss'`, auth pages match the rest of the app's design. Scope and plan are captured in the [Custom shadcn auth pages](/starter-template/roadmap/#custom-shadcn-auth-pages) roadmap entry.

### Why scope of v1.22.2 stayed small in practice

The audit documented in the v1.23.0 roadmap entry found that `AccountView`, `SignedIn`, `SignedOut`, `UserButton`, and `RedirectToSignIn` are re-exported from `features/auth/index.ts` but not imported anywhere in the codebase — the settings pages are already fully custom shadcn (see `SecurityPage.tsx`, `PersonalInformationPage.tsx`, `DataPrivacyPage.tsx`). Only `AuthView` (inside `AuthPage.tsx`) and `NeonAuthUIProvider` (inside `main.tsx`) have real consumers. The v1.23.0 work is therefore bounded to four new auth page files plus a small amount of cleanup.

### Migration

Run `npm run template:update`. Apps on the experimental v1.22.2 will see `frontend/src/styles/auth.css` deleted, `frontend/src/styles/globals.css` restored to its v1.22.1 shape (single `@import 'tailwindcss';`), and the `import '@/styles/auth.css'` line removed from `frontend/src/features/auth/index.ts`. The Tabs layout bug that v1.22.2 tried to fix returns until v1.23.0 ships — modules that need a working tabs primitive in the interim can continue to ship a local workaround. Apps that never touched v1.22.2 see no effective change.

## v1.22.1

**Auto-run `prisma migrate dev` on schema-updating template pulls**

`update-template.{sh,ps1}` now mirrors the module starter's post-schema-update flow ([`src/scaffold/update.ts`](https://github.com/ui-git-kim/unique-react-fullstack-module-starter) — `runPrismaAfterSchemaUpdate`): when a template update introduces new Prisma models and sets `NEEDS_MIGRATION=true`, the script runs `cd backend && npx prisma migrate dev --name template_update_v<version>` automatically after `npm run install:all` completes. Previously the scripts only printed the command and left the user to run it manually.

### What's new

- **Auto-migrate on schema changes** — when the update merges in new models (detected via the existing `MODELS_ADDED` logic), the script runs the migration with a versioned name derived from the template version (e.g. `template_update_v1_22_1`) so migration history is traceable.
- **Non-fatal failure** — wrapped in try/catch; if Prisma fails (missing `DIRECT_DATABASE_URL`, dev DB unreachable, conflicting state) the script logs a warning with the exact command to retry and continues to the "Next Steps" summary. The update merge itself is never reverted on migration failure.
- **Interactive stdio** — stdio is inherited so Prisma's own prompts (data-loss confirmations, generator downloads) work the same as running the command by hand.
- **Dedup with Next Steps** — the "Run database migration" reminder is suppressed when the auto-run succeeded, and falls back to the versioned command (not the old `add_template_models` name) when it didn't.

### Why

Template schema changes are rare but high-friction: the old script left an easy-to-miss "IMPORTANT" message buried between dependency install output and the Next Steps list. Aligning with the module starter's behaviour means downstream apps get consistent post-update ergonomics whether a template pull or a module update introduced the schema drift.

### No behavioural change when schema is unchanged

If `NEEDS_MIGRATION=false` (the common case), the script is identical to v1.22.0. Only apps picking up a template release with new models will notice — and only to see the migration applied for them automatically instead of being asked to run it.

## v1.22.0

**TanStack Query as the standard data fetching layer**

Adds TanStack Query (React Query v5) as the declared standard for server state across the starter template and every module. Introduces a shared `QueryClient` at the app root, documents the query-key namespacing convention, and refactors `AppSettingsContext` as the reference implementation. Every future module should use `useQuery` / `useMutation` instead of raw `fetch` + `useEffect`.

### What's new

- **Dependencies** — `@tanstack/react-query@^5` (dependency) and `@tanstack/react-query-devtools@^5` (devDependency) added to `frontend/package.json` and to `create-app.{sh,ps1}` so newly cloned apps install them.
- **Configured client** — `frontend/src/lib/query-client.ts` exports a single `QueryClient` with opinionated defaults: `staleTime: 60s`, `gcTime: 5min`, query `retry: 1`, mutation `retry: 0`, `refetchOnWindowFocus: false`, `refetchOnReconnect: true`. Each default documented with rationale.
- **Query keys** — `frontend/src/lib/query-keys.ts` publishes app-level keys (`['app-settings']`, `['session']`) as `appQueryKeys` and exports an optional `createModuleQueryKeys('<module-id>')` helper. Modules **must** namespace query keys under their module ID as the first array element.
- **Provider wiring** — `main.tsx` now mounts `<QueryClientProvider>` as the outermost provider (outside `NeonAuthUIProvider`), and mounts `<ReactQueryDevtools>` gated behind `import.meta.env.DEV` so production bundles pay nothing.
- **Reference implementation** — `frontend/src/features/admin/context/AppSettingsContext.tsx` rewritten from `useState` + `useEffect` + `useRef` to `useQuery`. Public API (`useGlobalAppSettings`, `useAppSettingsValue`, `useGlobalNavigationOverrides`, `useGlobalToolbarOverrides`, `AppSettingsProvider`) is unchanged — every consumer continues to work with zero migration. 401 handling still returns defaults silently; `updateSettings` is now an optimistic `queryClient.setQueryData` write.

### Docs

- **New page** — [Data Fetching](/starter-template/data-fetching/) is the authoritative reference for TanStack Query usage in this system: server-vs-client state split, key conventions with good/bad examples, mutation patterns (invalidate and optimistic), and a worked billing-module example.
- **LLM rules** — new **Data Fetching** section with the short version of the convention, plus four new Common Mistakes rows (raw fetch in `useEffect`, unnamespaced keys, second `QueryClient`, server data in Zustand).
- **Configuration** — new "TanStack Query Client Defaults" section documenting every default, the rationale, and when to override per-hook.
- **Architecture** — tech-stack box mentions TanStack Query; `lib/` tree adds `query-client.ts` and `query-keys.ts`; Key Files table updated.
- **Module starter → Building Modules** — new "Data Fetching (TanStack Query)" subsection in Starter Components with the module-specific rules and example hooks.

### Migration

No migration required. Existing downstream apps pick up the layer via `npm run template:update`; existing modules continue to work and should migrate their data-fetching hooks to `useQuery` as they're next touched. There is no deprecation shim — follow the convention on all new code.

## v1.21.1

**Documentation accuracy audit + `useStyleRegistry` hook**

Audited all 23 starter-template documentation pages against the v1.21.0 code and fixed everything the audit surfaced. Also added a `useStyleRegistry` React hook for consistency with every other registry in the template.

### Code — `useStyleRegistry` hook

Every other registry in the template is paired with a `createRegistryHook`-backed `useXxx` hook. The Style Registry shipped in v1.21.0 without one. Fixed:

```typescript
// frontend/src/config/style.registry.ts
export const useStyleRegistry = createRegistryHook(styleRegistry);
```

Re-exported from `@/config/registry.hooks`. Returns all registered entries sorted by `order`, re-renders on `register` / `update` / `unregister`. Pair with `styleRegistry.isEnabled(id)` to get the effective enabled state (which composes the module default with any admin override).

### Docs — new Style System page

Cross-cutting overview page that ties the runtime (`style-pipeline.ts`), build-time (`components:sync`), admin surface (`AppSettings.styleOverrides`), and module contract into one coherent picture. Lives at `/starter-template/style-system/` as a top-level sidebar entry. [Read it here](/starter-template/style-system/).

The page is the single entry point for understanding "which styling layer do I use for my use case". Existing pages (`theme.mdx`, `registries.mdx § Style Registry`, `scripts.mdx § Sync shadcn components`, `admin-feature.mdx § Style registry overrides`) remain the reference material for each layer — the Style System page is the map.

### Docs — fixes landed

- **admin-feature.mdx** — missing `/api/v1/admin/style-overrides` GET and PUT endpoints added; "settings returns everything" claim corrected to include `styleOverrides`; Prisma schema block updated with `styleOverrides` field; new `§Style registry overrides` section; public-read/admin-write pattern called out as a tip; "Adding Admin Pages" now shows both lazy and static-import patterns (lazy is the template's own default since v1.20.6).
- **auth-feature.mdx** — `main.tsx` code sample updated to show the `TooltipProvider` wrap (landed in v1.20.2 but never reached this example).
- **architecture.mdx** — frontend/backend file trees add `style.registry.ts`, `style-pipeline.ts`, `admin.types.ts`, and `service.registration.ts`; the stale `admin.routes.ts # GET/PATCH /settings` comment now reflects all the endpoints it actually exposes.
- **theme.mdx** — reframed as the low-level primitive page. Prominent pointer to the Style Registry for module authors. The "Style Module Integration" section no longer describes a future/planned state — v1.21.0 has shipped the module-facing API.
- **registries.mdx** — Style Registry section gains a React hook subsection documenting `useStyleRegistry` + the admin-UI methods on the singleton.
- **scripts.mdx** — "Scripts in this template" TOC at the top; an `<Aside>` on the `template:update` section explaining when to run `components:sync` afterwards.
- **configuration.mdx** — frontend env vars table with defaults and required flags, matching the level of detail in the backend section.
- **llm-rules.mdx** — `styleRegistry` row in the Available Registries table; two new Common Mistakes entries: forgetting `components:sync` after a cold-side change, and two modules claiming the same exclusive `componentsJson` field.
- **roadmap.mdx** — two new Planned items (styleOverrides bootstrap-cache fix for the first-paint flash, frontend env-var build-time validation); audit marked Completed.

### No behavioural change

The `useStyleRegistry` hook is purely additive — any code that used `useSyncExternalStore(styleRegistry.subscribe, styleRegistry.getSnapshot)` directly still works. Docs changes are correction/additions. No migration required; downstream apps get this patch via `npm run template:update` with no further action.

## v1.21.0

**Style Registry + `components.json` sync pipeline**

A single `styleRegistry` for modules to contribute both **runtime CSS overrides** (tokens + raw CSS) and **build-time shadcn `components.json`** values (style, base colour, cssVariables, registries, aliases). Complements the existing `theme.cssVariables` filter and `theme.changed` action hooks from v1.20.1.

Full reference on the [Registry Reference page](/starter-template/registries/#style-registry); terminal script usage on [CLI Scripts](/starter-template/scripts/#sync-shadcn-components); toggle reinstall UI is [on the roadmap](/starter-template/roadmap/#in-app-component-reinstall-for-style-registry).

### What's new

**Frontend — `frontend/src/config/style.registry.ts`** + **`frontend/src/lib/style-pipeline.ts`**

- Imperative API: `register`, `update`, `unregister`, `setOverride`, `clearOverride`, `applyOverrides`.
- Exclusivity enforced at registration for `componentsJson.style`, `componentsJson.tailwind.baseColor`, `componentsJson.tailwind.cssVariables` — a second entry claiming the same field throws `StyleRegistryError` with code `EXCLUSIVE_FIELD_CLAIMED`. Additive fields (`aliases`, `registries`) merge with last-write-wins and a `console.warn` on overlap.
- Pipeline merges tokens by `order`, pipes through the existing `theme.cssVariables` filter, applies via `applyThemeVariables()`, and writes concatenated raw CSS into a single `<style id="module-styles">` tag appended to `<head>` (so module CSS wins over `globals.css` on specificity ties).
- **Bug fixed in the process:** `applyThemeVariables` only ever *set* CSS variables on `:root` — stale values were never cleared. The new pipeline tracks previously-applied tokens and clears any that aren't in the next merge, so disabling a module reliably reverts its tokens.

**Backend — `AppSettings.styleOverrides`**

New `Json` field on the `AppSettings` model persisting per-entry admin toggles (`Record<entryId, boolean>`). Modules ship `enabled: true` by default; admin overrides win. `AppSettingsContext` calls `styleRegistry.applyOverrides(map)` on load, so the pipeline honours the admin's state before modules finish registering.

New endpoints:

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| `GET` | `/api/v1/admin/style-overrides` | Authenticated | Read full overrides map (frontend bootstrap) |
| `PUT` | `/api/v1/admin/style-overrides` | Admin only | Replace full overrides map |

Action hook `admin.styleOverridesUpdated` fires after a successful update so modules can react (e.g. invalidate caches).

**Sync script — `npm run components:sync`**

PowerShell + Bash scripts at `scripts/sync-components.{ps1,sh}`. Reads `componentsJsonContribution` from each enabled module in `.module-manifests.json`, enforces exclusivity, merges into `frontend/components.json` atomically, then runs `npx shadcn@latest add --overwrite` for the union of `installedShadcnComponents` from `.module-state.json`. A `--dry-run` / `-DryRun` mode previews without writing. Bash version requires `jq`.

```bash
npm run components:sync        # merge manifests → components.json → reinstall
npm run components:sync:dry    # preview only
```

### Migration for downstream apps

1. `npm run template:update` — merges the new `AppSettings.styleOverrides` field into your Prisma schema (also updates the multi-file `app-settings.prisma` heredoc for apps that were on multi-file mode but hadn't added AppSettings yet).
2. `cd backend && npx prisma migrate dev --name add_style_overrides` — creates the DB migration.
3. `npm run components:sync` — no-op if you have no modules contributing to `componentsJson`; otherwise applies their contributions.

### Module contract

Modules that want to contribute `componentsJson` fields declare a `componentsJsonContribution` entry in their `.module-manifests.json` **in addition to** calling `styleRegistry.register({ componentsJson: {...} })` at module load time. The registry copy is for runtime validation (exclusivity) and admin UI display; the manifest copy is what the sync script reads. These should match — a divergence means the runtime app sees different config than shadcn actually wrote out.

### Known limitation

There's a brief flash on first paint if an admin disabled a module-registered entry — the module's tokens apply at registration time, then the `applyOverrides` call removes them once the AppSettings fetch completes. Fixable with a localStorage cache of `styleOverrides` at bootstrap (same pattern as the theme class init in `main.tsx`). Not critical; can land in a follow-up patch.

## v1.16.0

**Personal Info Fields Registry: Module Preferences Now Save Correctly**

Fixed a critical issue where module-registered personal info fields with `storage: 'preferences'` (the default) were rejected by the backend Zod validation and failed silently in the UI.

**Root Cause (3 compounding issues):**

1. **Backend Zod schema used `.strict()`** — The `UpdatePreferencesSchema` rejected any field not explicitly listed, returning a 400 error for module-registered preference keys (e.g. `handle`, `socialTwitter`). Changed to `.passthrough()` so module-registered fields pass through validation.

2. **`handleFieldChange` swallowed errors** — When the API returned an error, the `onChange` callback caught and logged the error but did not re-throw. Custom field components (like the Blocks Builder's `HandleField`) that called `await onChange(value)` never saw the failure, showing "Saved" when the save actually failed.

3. **Text/textarea fields fired API calls on every keystroke** — No debounce, causing rapid-fire `PATCH` requests with potential race conditions.

**Fixes:**

- Backend `UpdatePreferencesSchema` now uses `.passthrough()` instead of `.strict()`
- Backend `UserPreferences` and `UpdatePreferencesInput` types now include `[key: string]: unknown` index signatures, matching the frontend types and the Zod `.passthrough()` behaviour
- `handleFieldChange` now re-throws errors after reverting, so custom components can catch failures and display their own error UI (e.g. toast notifications)
- `PersonalInfoFieldProps.onChange` type changed from `(value: unknown) => void` to `(value: unknown) => Promise<void>` to reflect async save behaviour
- Text and textarea registry fields now use debounced saves (600ms delay) instead of firing API calls on every keystroke
- Built-in field types (text, textarea, select) gracefully handle re-thrown errors

**Files Modified:**
- `backend/src/features/user/user.types.ts` — Zod schema + type index signatures
- `frontend/src/config/personal-info-fields.registry.ts` — `onChange` type signature
- `frontend/src/features/user/pages/PersonalInformationPage.tsx` — Error re-throw + debounced text fields

**Impact on modules:**
- Modules registering `personalInfoFieldsRegistry` fields with `storage: 'preferences'` now work correctly out of the box
- Custom field components should `await onChange(value)` and handle errors (already the expected pattern)
- No breaking changes — existing module code that calls `onChange()` without `await` continues to work

## v1.14.8

**Dynamic Breadcrumb Support**

Added a new `BreadcrumbContext` that allows pages to set dynamic breadcrumbs that display in the site header.

**New Components:**
- `BreadcrumbProvider` - Wraps the app and manages breadcrumb state
- `useBreadcrumbs()` - Hook to set dynamic breadcrumbs from any component
- `useBreadcrumbsOptional()` - Safe hook that returns undefined if not in provider

**Features:**
- Dynamic breadcrumbs automatically override route-based breadcrumbs when set
- Breadcrumbs auto-clear on route change (no stale breadcrumbs)
- Supports clickable links via optional `path` property on breadcrumb items

**Usage:**
```tsx
import { useBreadcrumbs } from '@/components/layout';

function MyDetailPage() {
  const { setBreadcrumbs } = useBreadcrumbs();
  const { data } = useMyData();

  useEffect(() => {
    if (data) {
      setBreadcrumbs([
        { label: 'Items', path: '/items' },
        { label: data.name }
      ]);
    }
  }, [data, setBreadcrumbs]);
}
```

**Files Added:**
- `frontend/src/components/layout/BreadcrumbContext.tsx`

## v1.14.4

**Admin Routes Registration Fix**

Fixed a bug where admin routes (PATCH, navigation, toolbar, users) were not being mounted due to duplicate path detection in the route registry.

**Problem:**
- Admin feature registered two routers on `/admin`: one for public GET (settings) and one for admin-only operations
- Route registry's duplicate detection skipped the second registration
- Result: Only `GET /api/v1/admin/settings` worked; all write operations returned 404

**Fix:**
- Merged both routers into a single router
- Applied `requireAdmin` middleware per-route instead of per-router
- `GET /settings` remains accessible to all authenticated users (for sidebar branding)
- All other admin routes properly require admin role

**Affected endpoints now working:**
- `PATCH /api/v1/admin/settings`
- `GET/PATCH /api/v1/admin/navigation`
- `GET/PATCH /api/v1/admin/toolbar`
- `GET /api/v1/admin/users`
- `PATCH /api/v1/admin/users/:userId/role`
- `DELETE /api/v1/admin/users/:userId`

## Versioning Infrastructure

Version bumps happen automatically for infrastructure changes. See [LLM Rules](/getting-started/llm-rules/) for the complete policy.

**Files requiring version bump:**
- `frontend/src/config/*.ts` - Registry files
- `frontend/src/lib/*.ts` - Utility libraries
- `backend/src/config/*.ts` - Registry files
- `backend/src/lib/*.ts` - Utility libraries
- `backend/src/middleware/*.ts` - Middleware
- `scripts/*.ps1`, `scripts/*.sh` - CLI scripts

**Bump commands:**
```bash
# Patch (bug fixes)
npm run template:version:patch

# Minor (new features)
npm run template:version:minor

# Major (breaking changes)
npm run template:version:major
```
