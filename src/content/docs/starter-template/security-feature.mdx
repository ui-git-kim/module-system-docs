The **security feature** (`features/security/`, frontend + backend) is a foundation feature ‚Äî a sibling to `auth`, **not a module**. It is too pivotal to be installable, uninstallable, replaceable, or downgradable by a module.

It is **not where security is *built* ‚Äî it's where security is *wired*.** Security is delegated to expert **managed providers** (Neon, Cloudflare, a managed KMS) and **standard, audited libraries**; the feature is the **thin, correct integration** of them ‚Äî done once, inherited by every app ‚Äî plus a **declaration layer** (the `securityRegistry`) so each app tells the providers what to enforce.

<Aside type="tip" title="We don't roll our own security">
The starter does **not** hand-build auth, sessions, encryption, or isolation. Those belong to providers whose job is to get them right. The template's value is doing the *integration* correctly once, so no app does DIY security. Bespoke code is a short, **named allow-list** (below) ‚Äî only where no provider exists.
</Aside>

<Aside type="note">
v1.28.0 shipped the **shell**: the tier config + the `securityRegistry`. The provider wiring (Neon RLS isolation, app-layer field encryption, the deletion cascade, the Cloudflare edge) lands in subsequent releases ‚Äî see the [roadmap](/starter-template/roadmap/#security-floor-v200). This page grows with each phase.
</Aside>

## Principles

- **Delegate; don't build.** Auth, sessions, MFA, encryption-at-rest, key storage, isolation, edge/WAF, and crypto are owned by **expert providers**. If a provider offers it, we use the provider ‚Äî we never hand-roll it.
- **The starter is correct wiring, done once.** A template's value is that the integration is written once, reviewed, and inherited by every app ‚Äî the opposite of "my system doing security".
- **Honest about the residue.** Two things can't be outsourced: **declaring** your app's data/access rules (so the provider can enforce them) and **wiring** the providers correctly. The first is configuration; the second is exactly what the starter centralises.
- **Tiers are additive configuration, not forks.** `standard ‚äÇ hardened ‚äÇ strict` ‚Äî a higher tier only ever *adds* controls. One `securityProfile` setting toggles the expensive ones; no per-tier branches.
- **Secure by default.** The floor ships on with zero configuration. The foundation default tier is **`hardened`**.

## Who owns what (the delegation map)

The providers and libraries security is delegated to:

| Layer | Provider / library | Manages |
|------|--------------------|---------|
| Identity, sessions, MFA, OAuth, reset | **Neon Auth** (managed Better Auth) | The auth lifecycle |
| Cross-user isolation | **Neon RLS** / **database-per-tenant** | Row- or instance-level isolation |
| Encryption at rest + key storage | **Neon** (AES-256, KMS-backed) | Disk encryption + key custody |
| Field-level encryption | **Node `crypto` (AES-256-GCM)** + managed KMS | App-layer encryption; key never sent to the DB |
| Edge: DDoS, WAF, bot, Zero Trust | **Cloudflare** | The perimeter |
| Errors / audit storage | **Sentry/Datadog** + the Audit module | Capture + retention |
| Input validation | **Zod** | Schema validation |
| Rate limiting, headers, TLS | **Cloudflare** + `express-rate-limit` + `helmet` | Throttling + secure transport |

For each concern: the provider owns the mechanism, the starter does the thin wiring, and the app **declares** its surface via the `securityRegistry`.

**The named bespoke allow-list** ‚Äî the *only* app-side security code, because no provider covers it:

1. **Field encryption** ‚Äî a thin `encryptField`/`decryptField` over Node's standard **AES-256-GCM** (`crypto`), with the key from env/KMS and **never sent to the database** (app-layer, *not* `pgcrypto`, so Neon only ever holds ciphertext).
2. **The deletion cascade wiring** ‚Äî a small orchestrator that calls each provider's delete API (Neon Auth identity, R2, Stripe, ‚Ä¶) in order; no provider deletes across *all* your stores.
3. **A thin `owned()` helper** as defence-in-depth behind Neon RLS.

Anything not on this list is provider- or library-backed, never hand-built.

## Security tier (`securityProfile`)

The headline knob is `SECURITY_PROFILE` (backend env) / `VITE_SECURITY_PROFILE` (frontend), one of `standard | hardened | strict`. It resolves to a full `SecurityConfig` via per-tier defaults plus optional per-knob env overrides ‚Äî an app changes posture through configuration, never by editing code.

```typescript
import { securityConfig } from '@/features/security';

securityConfig.profile;          // 'hardened'
securityConfig.fieldEncryption;  // tier default, overridable via SECURITY_FIELD_ENCRYPTION
securityConfig.auditMode;        // 'activity' | 'compliance'
```

| Knob | standard | hardened | strict |
|------|:--:|:--:|:--:|
| `mfaRequired` | false | false¬π | true |
| `fieldEncryption` | false | true | true |
| `auditMode` | activity | activity | compliance |
| `keyProvider` | env | kms¬≤ | kms¬≤ |
| `sessionStepUp` | false | true | true |

¬π At hardened, MFA is *available* and enforced for admins ‚Äî admin enforcement is handled in the auth layer, not this all-users boolean. ¬≤ Only the `env` key adapter ships working today; the KMS adapter is an interface wired per-deployment (the crypto phase resolves it and falls back to `env` with a warning at non-strict tiers). Override with `SECURITY_KEY_PROVIDER=env`.

See [Configuration](/starter-template/configuration/#security-tier) for every env var.

## The security registry

Modules declare a **security manifest** into `securityRegistry`; the foundation reads it to apply RLS policies + ownership scoping, field encryption, hashed-secret handling, audit subscriptions, and deletion cascades. A module with **no** manifest still inherits the floor (RLS + ownership) but cannot use field encryption, hashed secrets, audit, or deletion hooks until it declares them.

```typescript
// features/<module>/security.registration.ts  (side-effect import from index.ts)
import { securityRegistry } from '@/features/security';

securityRegistry.register({
  moduleId: 'billing',
  ownedModels: [{ model: 'Invoice', ownerColumn: 'tenantId' }],
  hashedSecrets: [{ model: 'ApiKey', field: 'token' }],
  encryptedFields: [{ model: 'Invoice', field: 'taxId' }],
  dataCategories: [{ name: 'invoices', classification: 'owned' }],
  auditEvents: ['invoice.created', 'invoice.voided'],
  // minimumTier: 'strict',  // refuse to run below this tier
});
```

See the [Registry Reference](/starter-template/registries/#securityregistry-backend) for the full manifest contract and query API.

<Aside type="caution" title="Isolation is delegated to Neon; the ownerColumn is internal plumbing">
Cross-user isolation is **enforced by Neon**, not by hand-rolled checks: **Neon RLS** policies (the configurable floor) and, for apps needing hard isolation, **database-per-tenant** (a Neon project per tenant ‚Äî cross-tenant leakage becomes *structurally impossible*). The registry's `ownerColumn` is just the **isolation key** those policies scope on. In the common **solo-owner** deployment it is the user's invisible personal space (key on `userId`; the deployment's own database is the wall); multi-user "shared spaces" (e.g. a family) add a shared key ‚Äî an opt-in capability, **not** enterprise team tenancy. The exact default key and the database-per-tenant control-plane are finalised in the isolation phase ‚Äî see the [roadmap](/starter-template/roadmap/#security-floor-v200).
</Aside>

## Field encryption

Modules declare `encryptedFields` / `hashedSecrets` in their manifest and call the foundation primitive ‚Äî they never hand-roll crypto. Added in v1.31.0.

- **`encryptField(plaintext, { tenantId? })` / `decryptField(envelope, { tenantId? })`** ‚Äî **AES-256-GCM** via Node's `crypto` (OpenSSL): a fresh random IV per call and the GCM auth tag verified on decrypt (tamper ‚Üí throws). The **key comes from env/KMS and is never sent to the database**, so Neon only ever holds ciphertext (a dump / console / leaked connection string is useless).
- **`hashSecret(token)` / `verifySecret(token, hash)`** ‚Äî SHA-256 + constant-time compare, for **high-entropy tokens / license keys** (not passwords ‚Äî those are Neon Auth's).

```typescript
import { encryptField, decryptField, hashSecret, verifySecret } from '@/features/security';

const sealed = await encryptField(taxId, { tenantId: req.user!.tenantId });
const value  = await decryptField(sealed, { tenantId: req.user!.tenantId });

const stored = hashSecret(apiToken);          // store this
const ok     = verifySecret(presented, stored);
```

**Keys (`keyProvider`):** the **`env`** adapter ships working ‚Äî `SECURITY_ENCRYPTION_KEY`, base64 32 bytes (`openssl rand -base64 32`). A **`kms`** adapter is an interface: wire a managed KMS per-deployment via `setKeyProvider()` (required at `strict`, where per-tenant keys enable crypto-shred). Field encryption is off at `standard`, on at `hardened`+ (the `fieldEncryption` tier knob).

<Aside type="caution" title="Why app-layer, not pgcrypto">
`pgcrypto` runs inside Postgres, so the key would travel to Neon on every query ‚Äî defeating "the database can't read it." App-layer AES-256-GCM keeps the key in the app; Neon only ever sees ciphertext. The stronger **zero-knowledge** guarantee (where even the app can't decrypt ‚Äî client-side E2EE) has no provider and stays on the [roadmap](/starter-template/roadmap/#security-floor-v200).
</Aside>
