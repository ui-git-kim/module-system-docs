The **security feature** (`features/security/`, frontend + backend) is a foundation feature — a sibling to `auth`, **not a module**. It is too pivotal to be installable, uninstallable, replaceable, or downgradable by a module.

It is **not where security is *built* — it's where security is *wired*.** Security is delegated to expert **managed providers** (Neon, Cloudflare, a managed KMS) and **standard, audited libraries**; the feature is the **thin, correct integration** of them — done once, inherited by every app — plus a **declaration layer** (the `securityRegistry`) so each app tells the providers what to enforce.

<Aside type="tip" title="We don't roll our own security">
The starter does **not** hand-build auth, sessions, encryption, or isolation. Those belong to providers whose job is to get them right. The template's value is doing the *integration* correctly once, so no app does DIY security. Bespoke code is a short, **named allow-list** (below) — only where no provider exists.
</Aside>

<Aside type="note">
v1.28.0 shipped the **shell** (tier config + the `securityRegistry`); subsequent releases wired the providers — Neon RLS isolation (v1.30.0), app-layer field encryption (v1.31.0), the deletion cascade (v1.32.0), AWS KMS key custody (v1.33.0), the **Cloudflare edge + startup posture guardrail** (v1.36.0), and the **admin Security page** (posture dashboard + env-knob editor, v1.37.0). See the [roadmap](/starter-template/roadmap/#security-floor-v200) for the phase tracker.
</Aside>

## Principles

- **Delegate; don't build.** Auth, sessions, MFA, encryption-at-rest, key storage, isolation, edge/WAF, and crypto are owned by **expert providers**. If a provider offers it, we use the provider — we never hand-roll it.
- **The starter is correct wiring, done once.** A template's value is that the integration is written once, reviewed, and inherited by every app — the opposite of "my system doing security".
- **Honest about the residue.** Two things can't be outsourced: **declaring** your app's data/access rules (so the provider can enforce them) and **wiring** the providers correctly. The first is configuration; the second is exactly what the starter centralises.
- **Tiers are additive configuration, not forks.** `standard ⊂ hardened ⊂ strict` — a higher tier only ever *adds* controls. One `securityProfile` setting toggles the expensive ones; no per-tier branches.
- **Secure by default.** The floor ships on with zero configuration. The foundation default tier is **`hardened`**.

## Who owns what (the delegation map)

The providers and libraries security is delegated to:

| Layer | Provider / library | Manages |
|------|--------------------|---------|
| Identity, sessions, MFA, OAuth, reset | **Neon Auth** (managed Better Auth) | The auth lifecycle |
| Cross-user isolation | **Neon RLS** / **database-per-tenant** | Row- or instance-level isolation |
| Encryption at rest + key storage | **Neon** (AES-256, KMS-backed) | Disk encryption + key custody |
| Field-level encryption | **Node `crypto` (AES-256-GCM)** + managed KMS | App-layer encryption; key never sent to the DB |
| Edge: DDoS, WAF, bot, Zero Trust | **Cloudflare** | The perimeter |
| Errors / audit storage | **Sentry/Datadog** + the Audit module | Capture + retention |
| Input validation | **Zod** | Schema validation |
| Rate limiting, headers, TLS | **Cloudflare** + `express-rate-limit` + `helmet` | Throttling + secure transport |

For each concern: the provider owns the mechanism, the starter does the thin wiring, and the app **declares** its surface via the `securityRegistry`.

**The named bespoke allow-list** — the *only* app-side security code, because no provider covers it:

1. **Field encryption** — a thin `encryptField`/`decryptField` over Node's standard **AES-256-GCM** (`crypto`), with the key from env/KMS and **never sent to the database** (app-layer, *not* `pgcrypto`, so Neon only ever holds ciphertext).
2. **The deletion cascade wiring** — a small orchestrator that calls each provider's delete API (Neon Auth identity, R2, Stripe, …) in order; no provider deletes across *all* your stores.
3. **A thin `owned()` helper** as defence-in-depth behind Neon RLS.

Anything not on this list is provider- or library-backed, never hand-built.

## Security tier (`securityProfile`)

The headline knob is `SECURITY_PROFILE` (backend env) / `VITE_SECURITY_PROFILE` (frontend), one of `standard | hardened | strict`. It resolves to a full `SecurityConfig` via per-tier defaults plus optional per-knob env overrides — an app changes posture through configuration, never by editing code.

```typescript
import { securityConfig } from '@/features/security';

securityConfig.profile;          // 'hardened'
securityConfig.fieldEncryption;  // tier default, overridable via SECURITY_FIELD_ENCRYPTION
securityConfig.auditMode;        // 'activity' | 'compliance'
```

| Knob | standard | hardened | strict |
|------|:--:|:--:|:--:|
| `mfaRequired` | false | false¹ | true |
| `fieldEncryption` | false | true | true |
| `auditMode` | activity | activity | compliance |
| `keyProvider` | env | kms² | kms² |
| `sessionStepUp` | false | true | true |

¹ At hardened, MFA is *available* and enforced for admins — admin enforcement is handled in the auth layer, not this all-users boolean. ² The **AWS KMS** adapter ships (v1.33.0) and **auto-wires** from `SECURITY_KMS_*` env — `npm run security:setup-kms` writes the vars and the feature registers `createAwsKmsKeyProvider` with no `app.ts` edit. If `kms` is selected but unwired (env incomplete), resolution warns and falls back to `env` at non-strict tiers (throws at strict). Override with `SECURITY_KEY_PROVIDER=env`.

See [Configuration](/starter-template/configuration/#security-tier) for every env var.

## The security registry

Modules declare a **security manifest** into `securityRegistry`; the foundation reads it to apply RLS policies + ownership scoping, field encryption, hashed-secret handling, audit subscriptions, and deletion cascades. A module with **no** manifest still inherits the floor (RLS + ownership) but cannot use field encryption, hashed secrets, audit, or deletion hooks until it declares them.

```typescript
// features/<module>/security.registration.ts  (side-effect import from index.ts)
import { securityRegistry } from '@/features/security';

securityRegistry.register({
  moduleId: 'billing',
  ownedModels: [{ model: 'Invoice', ownerColumn: 'tenantId' }],
  hashedSecrets: [{ model: 'ApiKey', field: 'token' }],
  encryptedFields: [{ model: 'Invoice', field: 'taxId' }],
  dataCategories: [{ name: 'invoices', classification: 'owned' }],
  auditEvents: ['invoice.created', 'invoice.voided'],
  // minimumTier: 'strict',  // refuse to run below this tier
});
```

See the [Registry Reference](/starter-template/registries/#securityregistry-backend) for the full manifest contract and query API.

<Aside type="caution" title="Isolation is delegated to Neon; the ownerColumn is internal plumbing">
Cross-user isolation is **enforced by Neon**, not by hand-rolled checks: **Neon RLS** policies (the configurable floor) and, for apps needing hard isolation, **database-per-tenant** (a Neon project per tenant — cross-tenant leakage becomes *structurally impossible*). The registry's `ownerColumn` is just the **isolation key** those policies scope on. In the common **solo-owner** deployment it is the user's invisible personal space (key on `userId`; the deployment's own database is the wall); multi-user "shared spaces" (e.g. a family) add a shared key — an opt-in capability, **not** enterprise team tenancy. The exact default key and the database-per-tenant control-plane are finalised in the isolation phase — see the [roadmap](/starter-template/roadmap/#security-floor-v200).
</Aside>

## Field encryption

Modules declare `encryptedFields` / `hashedSecrets` in their manifest and call the foundation primitive — they never hand-roll crypto. Added in v1.31.0.

- **`encryptField(plaintext, { tenantId? })` / `decryptField(envelope, { tenantId? })`** — **AES-256-GCM** via Node's `crypto` (OpenSSL): a fresh random IV per call and the GCM auth tag verified on decrypt (tamper → throws). The **key comes from env/KMS and is never sent to the database**, so Neon only ever holds ciphertext (a dump / console / leaked connection string is useless).
- **`hashSecret(token)` / `verifySecret(token, hash)`** — SHA-256 + constant-time compare, for **high-entropy tokens / license keys** (not passwords — those are Neon Auth's).

```typescript
import { encryptField, decryptField, hashSecret, verifySecret } from '@/features/security';

const sealed = await encryptField(taxId, { tenantId: req.user!.tenantId });
const value  = await decryptField(sealed, { tenantId: req.user!.tenantId });

const stored = hashSecret(apiToken);          // store this
const ok     = verifySecret(presented, stored);
```

**Keys (`keyProvider`):** the **`env`** adapter ships working — `SECURITY_ENCRYPTION_KEY`, base64 32 bytes (`openssl rand -base64 32`), with the key living in your env. For **provider-held key custody**, run `npm run security:setup-kms` — it provisions the wrapped key and sets `SECURITY_KEY_PROVIDER=kms` + the `SECURITY_KMS_*` env, and the feature **auto-wires** the shipped **AWS KMS** adapter from that env (no `app.ts` edit, survives `template:update`). The master key then lives in AWS, never in your env or the database, with CloudTrail logging and instant revocation. Walkthrough: [Field encryption with AWS KMS](/getting-started/aws-kms-setup/). Field encryption is off at `standard`, on at `hardened`+ (the `fieldEncryption` tier knob).

<Aside type="caution" title="Why app-layer, not pgcrypto">
`pgcrypto` runs inside Postgres, so the key would travel to Neon on every query — defeating "the database can't read it." App-layer AES-256-GCM keeps the key in the app; Neon only ever sees ciphertext.

This protects against a **database breach / Neon-console access / stolen dump** — the ciphertext is useless without the key. It does **not** make the *operator* unable to read the data: the app holds the key, and any app with server-side AI/search over content must be able to decrypt it. True "even the app can't read it" (client-side E2EE) is deliberately **not** a foundation goal — it forecloses server-side AI. Instead the foundation targets an **operator-accountability ceiling** — per-tenant keys, no operator-facing data surfaces, and access/decrypt auditing so a read is deliberate, single-tenant, and provable. See [The operator-accountability ceiling](/starter-template/roadmap/#the-operator-accountability-ceiling-why-not-zero-knowledge).
</Aside>

## Webhook signature verification

Third-party webhooks (Stripe, GitHub, ...) authenticate with an **HMAC over the raw request bytes** — not a session. Added in v1.54.0, the starter provides the full host-level chain so modules never hand-roll it:

- **`req.rawBody`** — the global JSON parser captures the exact raw bytes on every parsed request (a `verify` hook; a reference, not a copy). A re-serialised `req.body` is *not* byte-identical and will fail verification, so always verify against `req.rawBody`.
- **`verifyTimestampedWebhookSignature({ payload, header, secret, toleranceSeconds? })`** — the Stripe-style `t=...,v1=...` scheme: HMAC-SHA256 over `` `${t}.${rawBody}` `` with a **replay window** (default 300&nbsp;s), rotation candidates accepted, constant-time comparison.
- **`verifyWebhookSignature({ payload, signature, secret, prefix? })`** — the plain scheme (header = HMAC of body, e.g. GitHub's `X-Hub-Signature-256` with `prefix: 'sha256='`).
- **`computeHmac` / `timingSafeEqualStrings`** — the underlying keyed-MAC primitives for anything bespoke.

```ts
import { verifyTimestampedWebhookSignature } from '@/features/security';

const result = verifyTimestampedWebhookSignature({
  payload: req.rawBody,
  header: req.headers['stripe-signature'] as string,
  secret: env.STRIPE_WEBHOOK_SECRET,
});
if (!result.valid) {
  req.log.warn({ reason: result.reason }, 'Webhook signature rejected');
  return res.status(400).json({ error: 'Invalid signature' });
}
```

<Aside type="caution" title="Webhooks need the CORS carve-out in production">
Webhook deliveries are server-to-server and carry **no `Origin` header**, which production CORS rejects by default. Register the webhook route with `allowNoOrigin: true` (route registry) so signed deliveries reach the handler — the HMAC, not the Origin, authenticates the caller. See [Route Registry](/starter-template/registries/#route-registry).
</Aside>

## Deletion cascade

No provider deletes across **all** your stores, so account/tenant erasure is the second item on the [named bespoke allow-list](#who-owns-what-the-delegation-map) — a thin orchestrator that calls each provider's delete API in a safe order. Added in v1.32.0.

`deleteAccount(subject, opts)` runs every registered deletion handler grouped by the **data classification** the module declared, then removes the auth identity **last**, and returns a `DeletionReport` (it never throws — every outcome is captured for the caller to persist and, if needed, retry).

```typescript
import { deleteAccount } from '@/features/security';

const report = await deleteAccount(
  { tenantId: user.id, userId: user.id, email: user.email },
  { trigger: 'user-request', requestMetadata },
);
// report.status: 'completed' | 'partial' | 'failed'
// report.authIdentity.status, report.bundle, report.results, report.totals
```

**The order is fail-safe** — if the process dies partway, the least-bad state remains, and handlers are idempotent so the cascade can retry:

1. **Compliance bundle** — captured *before* any destruction (you can't export after you delete): the classification manifest + consent/decision record + legal-hold deferrals.
2. **`credential`** → **crypto-shred** (destroy keys first; even a later failure leaves encrypted data unreadable).
3. **`owned`** → **hard-delete** app rows. The core `User` row is a registered handler that runs *late* in this phase, so module-owned child tables clear first.
4. **`referenced`** → **unlink** the pointer only; never touch the external source.
5. **`preserved`** → recorded as a **deferred legal hold**, not deleted (a retention job erases it when the hold lifts — app-level).
6. **Auth identity** → removed **last**, and only if no `owned`/`credential` handler failed (a partial failure keeps the login alive for an idempotent retry rather than orphaning data).

There is deliberately **no wrapping DB transaction** — the cascade spans external providers (R2/Stripe/KMS) that can't enrol in a Postgres transaction. It's best-effort and fully reported.

### Modules register deletion handlers

```typescript
// features/<module>/security.registration.ts  (side-effect import from index.ts)
import { securityRegistry } from '@/features/security';
import { owned } from '@/lib/owned';
import { prisma } from '@/lib/prisma';

securityRegistry.registerDeletionHandler({
  id: 'billing.invoices',
  moduleId: 'billing',
  classification: 'owned',
  description: "Hard-delete the tenant's invoices",
  async run(ctx) {
    const { count } = await prisma.invoice.deleteMany({ where: owned(ctx.subject.tenantId) });
    return { action: 'hard-delete', recordsAffected: count };
  },
});
```

Handlers must be **idempotent** and must throw — or return `{ action: 'failed' }` — if they did not complete, since the orchestrator treats any non-failed return as success when deciding whether to remove the auth identity.

### The Neon Auth identity (closing the gap)

Historically `deleteUser` removed app rows but left the **Neon Auth login identity** behind. Deleting the identity is a provider action, so the feature ships a **seam**, not a hand-rolled auth store — exactly like the [key provider](#field-encryption). Neon Auth is now powered by **Better Auth**, whose supported server-side deletion is the Neon Auth server SDK (`@neondatabase/auth`, admin `removeUser`), so the recommended path is to implement the deleter against that SDK:

```typescript
// bootstrap, e.g. backend/src/app.ts
import { setAuthIdentityDeleter } from '@/features/security';

setAuthIdentityDeleter({
  name: 'neon-auth',
  async deleteIdentity(subject) {
    const userId = subject.userId || subject.tenantId;
    if (!userId) return { deleted: false, detail: 'no user id' };
    await serverAuth.api.removeUser({ body: { userId } }); // your Neon Auth server SDK
    return { deleted: true };
  },
});
```

If your provider exposes a plain REST admin endpoint instead, `createRestAuthIdentityDeleter({ buildUrl, headers })` is a thin provider-agnostic helper — you supply the URL and headers (verify them against your provider).

The **default** deleter is `unconfigured`: it does **not** fake success — it logs a WARN and the report carries `authIdentity.status: 'unconfigured'`, so the gap is visible rather than silent. App data is still erased; the identity simply remains until you wire a deleter (or remove it from the Neon dashboard).

<Aside type="note" title="Both deletion paths run the cascade">
Self-service deletion (`user.service.deleteUser`) and admin deletion (`admin.service.deleteUser`) both route through `deleteAccount`. The admin path keeps its atomic last-admin guard — the row is deleted inside the Serializable transaction, and the idempotent `user.account` handler then no-ops. The legacy `user.deleted` action hook still fires (after the cascade) for back-compat; modules migrating cleanup into a deletion handler should drop their `user.deleted` subscriber in the same step to avoid double cleanup.
</Aside>

See the [Registry Reference](/starter-template/registries/#securityregistry-backend) for the deletion-handler API and the full `DeletionReport` shape.

## Cloudflare edge & the startup posture guardrail

The perimeter — DDoS, WAF, bot management, edge rate limiting, TLS/HSTS, the frontend XSS-CSP, optional Zero Trust — is delegated to **Cloudflare**, the last entry on the [delegation map](#who-owns-what-the-delegation-map). It is **additive**: the app-layer floor (RLS, auth, validation, rate limiting, field encryption) runs and protects you with or without it. Added in v1.36.0. Full setup + the per-tier checklist: [Cloudflare edge](/getting-started/cloudflare-edge/).

There is almost no app code for the edge itself (it's dashboard configuration), so the code half of this phase is a **startup posture guardrail** — `runStartupPostureCheck()` (`features/security/posture.ts`), called at the end of `app.ts` bootstrap once every feature and module has registered. It evaluates the resolved posture and surfaces misconfiguration you'd otherwise hit at runtime:

| Check | Severity | Fatal when enforced? |
|------|----------|----------------------|
| No `SECURITY_EDGE` at `hardened`+ in production | warn | No (edge is additive) |
| `SECURITY_EDGE=cloudflare` but `TRUST_PROXY_HOPS=0` | warn | No |
| Field encryption on but key/KMS not ready | warn / **error** at `strict` | **Yes** |
| No auth-identity deleter wired | info / warn | No |
| A module's `minimumTier` exceeds the active tier | **error** | **Yes** |
| `REQUEST_TIMEOUT=0` in production | warn | No |

Findings log at their severity. **Enforcement** — halting startup on an `error` finding — is on by default **only at `strict` in production**; `SECURITY_POSTURE_ENFORCE` overrides it (`true` everywhere, `false` never — which at `strict` defeats the guard). The check is also exposed as a pure function for tests / a future health surface:

```typescript
import { evaluateStartupPosture } from '@/features/security';

const report = evaluateStartupPosture(); // no logging, no exit
// report.worst → 'ok' | 'info' | 'warn' | 'error'
// report.findings → [{ id, title, severity, detail, remediation? }]
```

<Aside type="tip" title="The edge is additive; the floor stands alone">
The guardrail never *requires* Cloudflare — it can't see Cloudflare's dashboard, and the floor doesn't need it. What it halts on (at `strict`) is genuine **app-side** misconfiguration: an unwired key at the tier that mandates encryption, or a module running below the tier it declared it needs. A separate one-shot middleware (`edgePostureMiddleware`) watches the first real request for Cloudflare headers and warns if traffic is reaching the origin directly — the one edge fault a startup check can't see, only observed traffic can.
</Aside>

New env knobs (see [Configuration → Security tier](/starter-template/configuration/#security-tier)): `SECURITY_EDGE` (`cloudflare` \| `none`) and `SECURITY_POSTURE_ENFORCE` (`true` \| `false`).

## Admin Security page

The admin panel ships a **Security page** (v1.37.0) — an admin-only posture dashboard plus an env-knob editor — backed by `GET /api/v1/admin/security/overview` and `PUT /api/v1/admin/security/env`. The reads are admin-gated (the posture reveals where the app is weak, so it must not leak to non-admins) and the dashboard surfaces the **live, resolved** posture: it delegates to the security feature's `evaluateStartupPosture()`, `securityConfig`, `getKeyProviderStatus()`, and `securityRegistry` rather than re-deriving anything.

The editor writes only the **non-secret**, env-backed knobs (a strict allow-list — secrets are never touched) by upserting `backend/.env`. Security config is resolved once at bootstrap, so an edit is flagged **restart-required**, and every edit is audit-logged.

Because production deploys usually read managed env vars (PM2/Docker/systemd) rather than `backend/.env`, the writer is **refused in production unless** `SECURITY_ADMIN_ALLOW_ENV_WRITE=true`. The editor is always enabled in development. Set the knob only if your production deploy genuinely loads `backend/.env`.

New env knob (see [Configuration → Security tier](/starter-template/configuration/#security-tier)): `SECURITY_ADMIN_ALLOW_ENV_WRITE` (`true` \| `false`, defaults off — only needed when a production deploy genuinely loads `backend/.env`).
