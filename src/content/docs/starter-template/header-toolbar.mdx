The header toolbar is the horizontal bar at the top of the app, to the right of the breadcrumbs. Modules add items to it via the `headerToolbarRegistry` â€” buttons, toggles, dropdowns, search inputs, and fully custom components.

---

## Registering Items

```typescript
import { headerToolbarRegistry } from '@/config/header-toolbar.registry';
import { Search, Bell, Moon } from 'lucide-react';

// Action button
headerToolbarRegistry.register({
  id: 'global-search',
  type: 'action',
  position: 'left',
  order: 10,
  icon: Search,
  label: 'Search (Cmd+K)',
  onClick: () => openSearchModal(),
});

// Dropdown with badge
headerToolbarRegistry.register({
  id: 'notifications',
  type: 'dropdown',
  position: 'right',
  order: 50,
  icon: Bell,
  label: 'Notifications',
  badge: 3,
  component: NotificationDropdown,
});

// Custom component (full control over rendering)
headerToolbarRegistry.register({
  id: 'theme-toggle',
  type: 'custom',
  position: 'right',
  order: 100,
  label: 'Toggle theme',
  component: ThemeToggle,
});

// Labelled button â€” visible text comes from `buttonLabel`, not `label`
headerToolbarRegistry.register({
  id: 'publish',
  type: 'button',
  position: 'right',
  order: 30,
  buttonLabel: 'Publish',
  buttonVariant: 'default',  // 'default' | 'secondary' | 'destructive' | 'outline' | 'ghost'
  buttonSize: 'sm',          // 'default' | 'sm' | 'lg'
  separatorBefore: true,     // draw a vertical separator before this item
  isVisible: () => canPublish(),  // contextual visibility predicate, re-evaluated each render
  onClick: () => publish(),
});
```

<Aside type="caution">
For `button` items, the visible text comes from `buttonLabel`, and the button style from `buttonVariant` / `buttonSize`. The `label` prop is the tooltip / aria-label only â€” setting `label` alone produces a button with no visible text.
</Aside>

To update an item after registration (for example, a live badge count), call `headerToolbarRegistry.update(id, updates)`:

```typescript
// Update the notifications badge without re-registering
headerToolbarRegistry.update('notifications', { badge: unreadCount });
```

## Item Types

| Type | Description | Conventional Props |
|------|-------------|--------------------|
| `action` | Button that calls a function on click | `onClick`, `icon`, `label` |
| `toggle` | Button with active/inactive state indicator | `onClick`, `icon`, `isActive`, `label` |
| `dropdown` | Opens a dropdown panel | `icon`, `component`, `label` |
| `button` | Labelled button (e.g., "Save Changes") | `buttonLabel`, `onClick`, `buttonVariant`, `buttonSize` |
| `search` | Compact search input | `onSearch`, `placeholder`, `searchValue` |
| `custom` | Fully custom component | `component` |

<Aside type="note">
Only `id`, `type`, and `position` are required by the `ToolbarItem` interface â€” every other prop is optional. The "Conventional Props" column lists what each type needs to render meaningfully, not props enforced by the type system.
</Aside>

<Aside type="tip">
For toggles that need different icons for active/inactive states (like a theme toggle), use `type: 'custom'` with a component that handles the icon switching logic internally.
</Aside>

## Position and Order

Items are split into `left` and `right` groups in the header, then sorted by `order` within each group.

```typescript
// Left side â€” primary actions
{ position: 'left', order: 10 }   // Search
{ position: 'left', order: 20 }   // Quick actions

// Right side â€” utilities
{ position: 'right', order: 50 }  // Notifications
{ position: 'right', order: 100 } // Theme toggle
```

### Order Convention (0-100 Scale)

| Range | Label | Examples |
|-------|-------|---------|
| 0-25 | Core | Search, primary actions |
| 26-50 | Feature | Notifications, module-specific actions |
| 51-75 | Secondary | Status indicators, utilities |
| 76-100 | Utility | Theme toggle, help |

## Dynamic Toolbar Actions

Admin pages (and any page) can dynamically inject toolbar items that appear only while the page is mounted. Use the `useAdminToolbarActions` hook:

```tsx
import { useAdminToolbarActions } from '@/features/admin';

function NavigationManagementPage() {
  const [hasChanges, setHasChanges] = useState(false);

  // Inject a "Save" button into the header while this page is mounted
  useAdminToolbarActions([
    {
      id: 'nav-save',
      type: 'button',
      position: 'right',
      order: 90,
      buttonLabel: 'Save Changes',
      buttonVariant: 'default',
      disabled: !hasChanges,
      onClick: handleSave,
    },
  ]);

  return <div>{/* Page content */}</div>;
}
```

The toolbar items are automatically removed when the component unmounts.

## Admin Overrides

Admins can hide toolbar items from the admin panel (Admin > Header Toolbar). Hidden items are stored in the database as toolbar overrides and respected by the header toolbar rendering logic (the `HeaderToolbar` component, via `applyToolbarOverrides`).

Items hidden by admins are not rendered regardless of registry state.

## Item Interface

```typescript
interface ToolbarItemComponentProps {
  item: ToolbarItem;
}

interface ToolbarItem {
  id: string;
  type: 'action' | 'toggle' | 'dropdown' | 'button' | 'search' | 'custom';
  position: 'left' | 'right';
  order?: number;              // Default: 50
  icon?: LucideIcon;           // For action / toggle / dropdown types
  label?: string;              // Tooltip text / aria-label only
  badge?: string | number;     // Badge count/text
  separatorBefore?: boolean;   // Draw a vertical separator before this item
  onClick?: () => void;        // For action; toggles state for toggle type
  isActive?: boolean | (() => boolean);   // For toggle type
  isVisible?: () => boolean;   // Contextual visibility predicate (re-run each render)
  disabled?: boolean | (() => boolean);
  component?: React.ComponentType<ToolbarItemComponentProps>;  // For dropdown/custom; receives { item }

  // Button type
  buttonLabel?: string;        // Visible button text
  buttonVariant?: 'default' | 'secondary' | 'destructive' | 'outline' | 'ghost';  // Default: 'default'
  buttonSize?: 'default' | 'sm' | 'lg';  // Default: 'sm'
  isLoading?: boolean | (() => boolean);  // Spinner + disabled while true (button/action)

  // Search type
  placeholder?: string;        // Default: 'Search...'
  onSearch?: (value: string) => void;
  searchValue?: string | (() => string);  // Controlled value
}
```

The registry also exposes an `update(id, updates)` method for mutating a registered item in place (for example, refreshing a badge count) without re-registering it.
