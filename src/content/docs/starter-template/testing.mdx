The template ships a minimum test suite ([Vitest](https://vitest.dev)) and a GitHub Actions pipeline so regressions are caught before they reach production — the coverage the foundation itself relies on, and a starting point for your app.

## What's covered

| Test | File | Needs a DB? |
|------|------|-------------|
| **`owned()` scoping** — the isolation key can't be overridden by a client-supplied `tenantId` | `backend/src/lib/owned.test.ts` | No |
| **Cross-user isolation** — user A can't read/update/delete user B's rows through any path (Neon RLS) | `backend/src/__tests__/isolation.test.ts` | **Yes** |
| **Registry smoke** — features register their security manifest + deletion handlers | `backend/src/__tests__/registries.test.ts` | No |
| **BaseRegistry lifecycle** — register / unregister / order / admin overrides | `frontend/src/config/__tests__/base-registry.test.ts` | No |

<Aside type="note" title="On “module install/uninstall”">
The roadmap's *module install/uninstall integration test* is exercised here as the **registry register→unregister lifecycle** (BaseRegistry), which is the mechanism modules plug into. A full scaffold-a-throwaway-module test belongs to the **module starter**, not the app template.
</Aside>

## Running tests

```bash
npm test                 # both packages (root)
cd backend && npm test   # backend only
cd frontend && npm test  # frontend only
```

The **cross-user isolation** test is DB-backed and **skips unless `TEST_USE_DB=1`** and a real Postgres is configured — so `npm test` is green locally with no database. It provisions a sample owned table with `FORCE ROW LEVEL SECURITY` + a `current_setting('app.tenant_id')` policy — the exact pattern owned models use (see [User-data isolation](/starter-template/database/#user-data-isolation-neon-rls)) — and drives it through the real `withTenantContext` helper.

## Continuous integration

`.github/workflows/ci.yml` runs on every push + PR with three jobs:

- **backend** — `npm install` → `prisma generate` → typecheck → unit tests.
- **frontend** — `npm install` → **test** (always). Typecheck + build (and lint) run only in a **complete app**: the bare template ships without the shadcn `components/ui/*` (generated per app) and without an eslint config, so those steps are gated on their presence — the template's own CI stays green while a cloned app gets the full pipeline.
- **isolation** — the DB-backed test against a **throwaway Neon branch**. **Opt-in** (see below).

### Enabling the Neon-branch isolation job

<Steps>

1. In the repo's **Settings → Secrets and variables → Actions**, add secrets `NEON_API_KEY` (Neon account settings → API keys) and `NEON_PROJECT_ID` (your Neon project).
2. Add a **repository variable** `RUN_DB_TESTS` = `true`.

</Steps>

CI then creates a fresh Neon branch per run, points the isolation test at it (`TEST_USE_DB=1`), and **deletes the branch afterwards** (even on failure). Without the variable set, the job is skipped and the rest of the pipeline still runs.

<Aside type="tip" title="Why a real branch, not a mock">
Isolation is enforced by **Postgres RLS**, not app code — so the test only means something against a real database. A throwaway Neon branch gives production parity (the real adapter + policies) and costs nothing to spin up and tear down.
</Aside>

## Enabling tests on an existing app (after `template:update`)

The test **files** and the **CI workflow** are template-owned and force-sync into your app on update; the root `test` scripts are reconciled by `sync-package-scripts`. Two one-time steps to activate:

<Steps>

1. Install the runner in each package: `cd backend && npm i -D vitest` and `cd frontend && npm i -D vitest`.
2. Make sure each package's `package.json` has `"test": "vitest run"` (new apps get this from the template; older apps can copy it).

</Steps>
