## How It Works

Apps created with the create script share git history with the template. This enables seamless updates using standard git merge.

The `template-upstream` remote points to your local copy of the template (pull-only, push disabled). When the template is updated, you can merge those changes into your app.

---

## Git Remotes

| Remote | Purpose | Push? |
|--------|---------|-------|
| `template-upstream` | Template repository for updates | No (disabled) |
| `origin` | Your app's repository | Yes |

The create script sets up `template-upstream` automatically. Push is disabled to prevent accidentally pushing your app code back to the template.

## Checking for Updates

<Tabs>
  <TabItem label="npm Script">
    ```bash
    npm run template:check
    ```
    This fetches the latest template and shows available commits.
  </TabItem>
  <TabItem label="Manual">
    ```bash
    git fetch template-upstream
    git log HEAD..template-upstream/main --oneline
    ```
  </TabItem>
</Tabs>

If no commits are shown, you're already up to date.

## Applying Updates

### Using the Update Script

The easiest way to update is using the built-in script:

```bash
npm run template:update
```

The script will:
1. Check for uncommitted changes (must be clean)
2. Fetch the latest template
3. Show available commits and version difference
4. Ask to merge or preview
5. Handle the merge automatically
6. Clean up template-only files

### Options

| Option | When to Use |
|--------|-------------|
| **Merge** (recommended) | Creates a merge commit. Preserves all history. Safe. |
| **Preview** | Shows what would change without merging. |

### Manual Update

<Steps>
1. **Ensure clean working directory**
   ```bash
   git status
   ```

2. **Fetch latest template**
   ```bash
   git fetch template-upstream
   ```

3. **Merge updates**
   ```bash
   git merge template-upstream/main
   ```

4. **Resolve any conflicts**

5. **Test your app**
   ```bash
   npm run type-check
   npm run dev
   ```
</Steps>

## What Gets Updated vs Preserved

### Updated (template infrastructure)

These files come from the template and should generally accept updates:

| Path | Contents |
|------|----------|
| `frontend/src/config/*.ts` | Registry definitions |
| `frontend/src/lib/*.ts` | Utility functions |
| `backend/src/config/*.ts` | Registry definitions |
| `backend/src/lib/*.ts` | Utilities (prisma, errors) |
| `backend/src/middleware/*.ts` | Auth middleware |
| `CLAUDE.md` | AI assistant instructions |
| `update-template.*` | Update scripts |

### Preserved (your code)

These files are yours and won't be touched:

| Path | Contents |
|------|----------|
| `frontend/src/features/*` | Your feature modules |
| `backend/src/features/*` | Your feature modules |
| `.env` files | Your credentials |
| `backend/prisma/schema.prisma` | Your database schema |
| `package.json` | Your app name/version |

### User-owned allow-list (force-sync opt-out)

Since v1.23.1 the update script force-syncs every template-changed file to the template's current version after the git merge, so stale lines from withdrawn template releases can't linger. A small set of **template-provided but user-customised** files opts out of this force-sync — the script preserves your local version and only warns you if the template also changed them:

| Path | Why it's user-owned |
|------|---------------------|
| `frontend/components.json` | Rewritten by `shadcn init`; mutated by `npm run components:sync` |
| `frontend/src/styles/globals.css` | Rewritten by `shadcn init` with the full light + dark token set (added in v1.23.2 — earlier versions force-synced it, which wiped shadcn's dark palette) |
| `backend/prisma/schema.prisma`, `backend/prisma/schema/*.prisma` | Your database schema |
| `backend/prisma/migrations/*` | Always user-owned — generated from your schema changes |
| `package.json`, `frontend/package.json`, `backend/package.json` | App-specific name, version, dependencies |
| `*-lock.json` | Regenerated by npm |
| `.env`, `.env.local` | Secrets |
| `README.md`, `CLAUDE.md` | Per-app readmes and Claude Code instructions |

<Aside type="note" title="Need to opt out of more files?">
If your app has legitimately customised a template-owned file and you want the update script to leave it alone, the [`.template-ignore` escape hatch](/starter-template/roadmap/#template-ignore-for-per-app-merge-customisation) is roadmapped as the long-term answer. Until it ships, flag the path you need — if it's a genuine user-owned category we missed, it should join the allow-list in a patch release.
</Aside>

### Potential Conflicts

These files exist in both but you may have modified:

| Path | Resolution |
|------|------------|
| `frontend/src/routes.tsx` | Merge carefully - keep your routes |
| `frontend/src/components/layout/*` | Usually accept template changes |
| Root config files | Review changes |

## Handling Conflicts

When git can't automatically merge a file, you'll see conflict markers:

```text
<<<<<<< HEAD
// Your version
=======
// Template version
>>>>>>> template-upstream/main
```

### Resolution Commands

```bash
# Keep your version entirely
git checkout --ours path/to/file

# Take template version entirely
git checkout --theirs path/to/file

# After resolving, mark as resolved
git add path/to/file

# Continue with merge
git commit

# Or abort if something went wrong
git merge --abort
```

### Common Conflict Patterns

**routes.tsx** - You added routes, template structure changed:
- Keep your route imports and spreads
- Accept any structural changes to the router setup

**Layout components** - You customised, template updated:
- Usually accept template changes
- Re-apply your customisations after

## After Updating

<Aside type="tip" title="Most of this is automated since v1.22.1">
`update-template.{sh,ps1}` now runs `npm run install:all` (which finishes with `prisma generate`) and, when new Prisma models were merged in, `npx prisma migrate dev --name template_update_v<version>` for you. The manual steps below are the fallback path if the script's auto-run fails or you ran the merge by hand.
</Aside>

<Steps>
1. **Install any new dependencies** (automatic — run manually only if `install:all` failed)
   ```bash
   npm install
   cd frontend && npm install
   cd ../backend && npm install
   ```

2. **Run database migrations** (automatic when schema changed — run manually only if the auto-migrate failed or you skipped it)
   ```bash
   cd backend && npx prisma migrate dev
   ```

3. **Generate Prisma client** (`install:all` calls this as its last step — run manually only if that step failed)
   ```bash
   cd backend && npx prisma generate
   ```

4. **Test your app**
   ```bash
   npm run dev
   ```

5. **Push to your repository**
   ```bash
   git push
   ```
</Steps>

## Troubleshooting

### "Not a git repository"

Run the update script from your app's root folder.

### "You have uncommitted changes"

Commit or stash your changes first:
```bash
git stash
# Run update
git stash pop
```

### "No common ancestor"

This happens on first merge if histories are unrelated. The script handles this with `--allow-unrelated-histories`.

### Template remote missing

If `template-upstream` wasn't set up:
```bash
# Point to your local template copy
git remote add template-upstream ../unique-react-fullstack-starter
git remote set-url --push template-upstream no_push
```

## Technical Details

### How the Update System Works

Apps are created via `git clone` (not copy), which preserves git history. This enables:

- `git fetch template-upstream` to get updates
- `git merge` to intelligently merge changes
- Conflicts only where both sides modified the same file

### Template-Only Files

These files exist in the template but should never exist in apps. The update script automatically removes them during merge, both in the post-success cleanup and in the conflict-resolution branch (since v1.23.3):

- `create-app.ps1`, `create-app.sh`, `create-app.bat` — one-time scaffolding scripts
- `docs/` — the module-system docs submodule, only meaningful in the template repo
- `.claude/` — user-specific Claude Code config
- `.template-archive/` — reserved for the upcoming archive-on-removal feature

When git detects any of these as "modify/delete" conflicts (most commonly `docs` when its submodule pointer bumps on the template side), the script resolves them automatically by removing the path — including stripping the submodule registration from `.gitmodules` for `docs`.

### Documentation Submodule

The template's `docs/` folder is a git submodule pointing to the shared module system documentation. This is intentional for the template itself, but apps don't need it.

<Aside type="tip">
  Apps should either remove the `docs/` submodule entirely, or replace it with their own documentation. See [App Documentation](/starter-template/roadmap/#app-documentation) in the roadmap.
</Aside>

To remove the docs submodule from your app:

```bash
# Remove submodule entry from .gitmodules
git config -f .gitmodules --remove-section submodule.docs 2>/dev/null || true

# Remove submodule entry from .git/config
git config --remove-section submodule.docs 2>/dev/null || true

# Remove the submodule directory
rm -rf docs .git/modules/docs

# Stage the removal
git add .gitmodules
git rm --cached docs 2>/dev/null || true

# Commit
git commit -m "Remove docs submodule"
```

### First Merge (Unrelated Histories)

The first merge requires the `--allow-unrelated-histories` flag. The script detects this by checking if `git merge-base` returns an error, then handles it automatically.

### Self-Updating Scripts

The update scripts can update themselves. When this happens:

1. New script version is fetched from template
2. Script commits its own update
3. Script pushes the commit
4. Waits briefly (prevents race conditions)
5. Re-executes with original arguments

This ensures you always run the latest update logic.

### Upstream Tracking

**Important**: The app branch should NOT track `template-upstream`. If it does, VS Code may try to push to it (which fails) and show files as "locked". The scripts run `git branch --unset-upstream` to prevent this.
