The **System** page (`/admin/system`) is a read-only diagnostics window into the running backend: database health, the resolved service/connection posture, and a live **errors + activity console**. It is admin-only and mirrors the [Security page](/starter-template/security-feature/) pattern.

It is the first phase of bringing observability into the starter **core** — audit and error logging are a built-in feature, not a separate module. See the [roadmap entry](/starter-template/roadmap/#observability-core-system-admin-page--auditerrors) for the full design and later phases.

## What it shows

<Tabs>
<TabItem label="Database">
Connectivity plus cheap structural signals via `checkDatabaseHealth()` and a couple of best-effort queries:

- Connection state + `SELECT 1` latency
- Active adapter (`neon` / `pg`)
- Last applied migration (from `_prisma_migrations`)
- Row counts for the starter's own models
</TabItem>
<TabItem label="Services & connections">
The startup security posture, reused — not reimplemented — from the security feature:

- `evaluateStartupPosture()` findings (Neon Auth, edge, trust-proxy, hardening, …)
- `getKeyProviderStatus()` field-encryption key readiness
- Tier, environment, edge, and DB adapter
</TabItem>
<TabItem label="Console">
The live **Errors** and **Activity** feeds from the in-memory audit sink (newest first). Manual refresh only — there is no polling, so a left-open admin tab never adds steady API load.
</TabItem>
</Tabs>

<Aside type="tip" title="In-memory by default; durable + compliance are opt-in">
The console is backed by an **in-memory** sink by default: buffers are bounded (about 200 errors / 500 activity), per-process, and reset on restart — intentional for a zero-config live console. Set `SECURITY_AUDIT_SINK=database` (or the strict / compliance tier) to persist events durably — see [Durable & compliance storage](#durable--compliance-storage-p2p3) below. Production observability still belongs in Sentry / a log aggregator.
</Aside>

## Where the console lives (v1.59.0)

The errors / activity / history console is its own admin page — **Console** (`/admin/console`, Platform section) — since v1.59.0; the System page keeps database health and service status. Everything below about the sink still applies; the console page is its UI.

## The audit/error sink

The capture pipeline is a small, swappable sink owned by the admin feature — `backend/src/features/admin/audit-sink.ts` (moved there from `backend/src/lib/` in v1.55.0; the old path is a deprecated shim until v1.56.0) — modelled on the key-provider pattern. The seam is exported from the admin barrel:

```ts
import { auditSink, setAuditSink } from '../admin/index.js';
// (or deep: '../admin/audit-sink.js' — MemoryAuditSink lives there)

// The default is an ephemeral in-memory ring buffer. Swap it for a durable or
// compliance sink at bootstrap:
//   setAuditSink(new MyPrismaAuditSink());

auditSink.recordActivity({ hook: 'thing.happened', actorId, summary: { id } });
const recent = auditSink.queryErrors(50); // newest first
```

| Export | Purpose |
|--------|---------|
| `auditSink` | Stable facade — always delegates to the active sink |
| `MemoryAuditSink` | Default ephemeral ring-buffer implementation |
| `setAuditSink(sink)` / `getAuditSink()` | Swap / read the active sink |
| `AuditSink` | The interface a custom sink implements |

### Capture is additive

Nothing in the emitting services changed — capture is wired by listeners:

- **Errors** — the global error middleware fires an `error:unhandled` action; `features/admin/audit.registration.ts` records it. Stack traces are kept server-side outside development.
- **Activity** — the same registration subscribes to the lifecycle hooks the starter already emits (`user.*` / `admin.*`), plus `security.dataAccessed` (v1.64.0) — audited reads of encrypted content via `auditedDecrypt`, recorded under the data **owner's** `tenantId` so cross-tenant reads are queryable per owner. Payloads are summarised and sensitive keys masked.

To record your own activity from a module, add a listener to an existing hook or call the sink directly:

```ts
import { actionRegistry } from '@/config/action.registry.js';
import { auditSink } from '@/features/admin';

actionRegistry.addAction('billing.invoice.paid', 'audit', (ctx) => {
  auditSink.recordActivity({ hook: 'billing.invoice.paid', summary: { invoiceId: ctx.invoiceId } });
});
```

## Durable & compliance storage (P2/P3)

By default the console is **ephemeral** (in-memory). Two opt-in env knobs upgrade it to durable storage. The sink is **best-effort**: it self-provisions its table at runtime and degrades back to memory-only on any DB error, so turning it on can never break the app.

| Knob | Effect |
|------|--------|
| `SECURITY_AUDIT_SINK=database` | **Durable (P2).** Persists errors + activity to a self-provisioned `"AuditLog"` table while the live console still serves recent events from memory. Prunable. |
| `SECURITY_AUDIT_MODE=compliance` (or the `strict` tier) | **Compliance (P3).** An immutable, **hash-chained**, append-only sink — never pruned. |
| `SECURITY_AUDIT_RETENTION_DAYS=30` | Activity-row TTL in days (default 30; `0` = keep forever). Compliance rows are never pruned. |

<Aside type="note" title="No schema migration required">
The `"AuditLog"` table is created at runtime via `CREATE TABLE IF NOT EXISTS` — it is **not** a Prisma model, so there is no schema change, migration, or `create-app` / `template:update` propagation to do. The sink uses raw SQL, so the code compiles in every app whether or not durable audit is enabled. Apps that prefer Prisma-managed schema + Row-Level Security can add an equivalent model themselves.
</Aside>

**Retention & erasure.** Activity rows age out after `SECURITY_AUDIT_RETENTION_DAYS`. They are also **owned data**: the [deletion cascade](/starter-template/security-feature/) erases a tenant's activity rows on account deletion (errors from authed requests carry the tenant; unauthenticated errors are operational telemetry that age out via retention). Compliance rows are **preserved** (legal hold) — the cascade never deletes them.

**Tamper-evidence (compliance).** Each compliance row stores the exact canonical serialisation it was hashed over plus `hash = SHA-256(prevHash + payload)`, chained across rows and serialised with a Postgres advisory lock. `GET /api/v1/admin/system/audit/verify` recomputes the chain from the stored payloads and reports the first break; `GET /api/v1/admin/system/audit/export` returns it for offline archival. The **History** tab on the System page surfaces durable rows and a "Verify integrity" action.

## API

All endpoints are admin-only (reads included — diagnostics reveal where the app is weak).

| Method | Path | Returns |
|--------|------|---------|
| `GET` | `/api/v1/admin/system/overview` | Database + services + recent errors/activity |
| `GET` | `/api/v1/admin/system/errors?limit=` | Newest-first errors (stack redacted outside dev) |
| `GET` | `/api/v1/admin/system/activity?limit=` | Newest-first activity events |
| `GET` | `/api/v1/admin/system/history?kind=&limit=` | Durable DB history (P2) — empty when in-memory only |
| `GET` | `/api/v1/admin/system/audit/verify` | Recompute the compliance hash chain (P3) |
| `GET` | `/api/v1/admin/system/audit/export?limit=` | Export compliance rows for offline archival (P3) |

## Security

- **Admin-only reads** — the posture and error detail are recon-relevant, so reads are gated like the Security page.
- **Redaction** — stack traces are stripped outside development; activity summaries mask sensitive keys and reduce nested objects to a shape hint.
- The rate limiter is unchanged — the console simply makes 429s and unhandled errors **visible** instead of silent.
