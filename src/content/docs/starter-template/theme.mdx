The starter template exposes two low-level helpers in `frontend/src/lib/theme.ts` for applying CSS custom-property overrides to the document root. These are primitives — direct DOM writers with module hook points.

<Aside type="caution" title="Prefer the Style Registry for module contributions">
If you're writing a module that contributes styles, use the **[`styleRegistry`](/starter-template/registries/#style-registry)**, not `applyThemeVariables()` directly. The registry handles merge semantics, exclusivity enforcement, admin overrides, and — importantly — clears tokens that fall out of the active merge (which `applyThemeVariables` on its own does not).

See the [Style System](/starter-template/style-system/) overview for the full picture of when to use which layer.
</Aside>

<Aside type="note">
The theme utility manages CSS variable overrides applied on top of `globals.css`. It does **not** control the dark/light mode toggle — that is handled separately by the Tailwind `dark` class in `main.tsx`.
</Aside>

---

## When to use this directly

Use `applyThemeVariables` / `resetThemeVariables` directly if you're building **app-level** theming behaviour that isn't a module — for example, a one-off colour scheme switcher inside a single feature. For anything cross-cutting (shipped as a module, composable with other modules, subject to admin enable/disable), the `styleRegistry` is the right surface.

---

## How It Works

The utility uses the [filter and action registries](/starter-template/hooks/) to create a pipeline:

```
applyThemeVariables(variables)
        │
        ▼
1. Run `theme.cssVariables` filter
   → Modules can inject or modify variables
        │
        ▼
2. Apply to document.documentElement.style
   → CSS custom properties set on :root
        │
        ▼
3. Fire `theme.changed` action
   → Modules can react (re-render, sync state)
```

Note this pipeline only **sets** CSS variables — stale values from a previous call stay on `:root` until explicitly cleared. The `styleRegistry` pipeline handles stale clearing for you (see [style-pipeline.ts](/starter-template/architecture/#frontend-structure)); if you call `applyThemeVariables` directly, you're responsible for tracking and clearing stale properties yourself.

## Applying Theme Variables

```typescript
import { applyThemeVariables } from '@/lib/theme';

applyThemeVariables({
  '--color-primary': 'hsl(220 70% 50%)',
  '--color-primary-foreground': 'hsl(0 0% 98%)',
  '--radius': '0.75rem',
});
```

These override the values defined in `globals.css`. Any CSS custom property can be set — the utility is not limited to a predefined list.

---

## Resetting to Defaults

```typescript
import { resetThemeVariables } from '@/lib/theme';

resetThemeVariables();
```

Removes all inline overrides, reverting to the values in `globals.css`. Also fires the `theme.changed` action with an empty variables object.

---

## Injecting Variables via Filter

Modules can hook into the `theme.cssVariables` filter to inject additional CSS variables or modify existing ones before they are applied to the DOM. This filter is the same one the `styleRegistry` pipeline runs through, so any module using either surface participates in the same transform pipeline.

```typescript
import { filterRegistry } from '@/config/filter.registry';

// Charts module adds chart-specific colour variables
filterRegistry.register('theme.cssVariables', 'charts-module', (vars) => ({
  ...vars,
  '--chart-primary': vars['--color-primary'] ?? 'hsl(220 70% 50%)',
  '--chart-secondary': vars['--color-secondary'] ?? 'hsl(160 60% 45%)',
}));
```

Filters run in order (lower number = earlier). The output of each filter is passed to the next.

---

## Reacting to Theme Changes

Modules can hook into the `theme.changed` action to react after theme variables are applied:

```typescript
import { actionRegistry } from '@/config/action.registry';

// Canvas module redraws when theme changes
actionRegistry.addAction('theme.changed', 'canvas-module', (ctx) => {
  redrawCanvas(ctx.variables);
});
```

The context object contains `{ variables }` — the final CSS variable map after all filters have run. When `resetThemeVariables()` is called, `variables` is an empty object.

---

## Hook Reference

| Hook | Registry | Type | Context |
|------|----------|------|---------|
| `theme.cssVariables` | filterRegistry | Filter | The CSS variable map (modules return modified map) |
| `theme.changed` | actionRegistry | Action | `{ variables }` — the applied variable map |

---

## API Reference

### Functions

| Function | Description |
|----------|-------------|
| `applyThemeVariables(variables)` | Apply CSS variable overrides to the document root. Does **not** clear previously-applied variables. |
| `resetThemeVariables()` | Remove **all** inline overrides from `:root` and fire `theme.changed` with an empty map. |

### Types

```typescript
/** CSS variable overrides — key is the custom property name, value is the CSS value */
type ThemeVariables = Record<string, string>;
```
