**Colour knowledge** is the per-colour meaning layer: what a colour *evokes* (its psychology) and *who already uses it* (well-known brands). Together with the cultural [Colour Semantics](/colour-palette/colour-semantics/) layer it forms the **Colour Knowledge** area of the [Thesaurus](/colour-palette/thesaurus/) — the colour-side reference data, surfaced for the working colour on the profile **Analyse** tab, not a per-palette property.

<Aside type="note" title="Why it lives with the Thesaurus">
Colour knowledge is colour-side reference data, a companion to the Thesaurus' names and design-system tokens — so it lives under the Thesaurus (Colour Names / Design Systems / **Colour Knowledge**), and the palette Method *consumes* it rather than owning it. The same data drives the per-colour card and, in a later phase, biases generation: the Method's **Impact axis** values are exactly the psychology dimensions used here, so picking Impact = *Trustworthy* maps onto the colours whose psychology is trustworthy.
</Aside>

<Aside type="caution" title="Tendencies, confidence-tagged — not citation-backed">
Psychology associations are conventions/tendencies, not effects a colour produces — the copy mirrors the Impact axis ("tends to feel…"), never "makes users feel". Every row carries a **confidence** (`high` / `medium` / `low`) so weak rows can be down-weighted, and **`source` is intentionally left empty** in the seed — never fabricated. Brand colours are factual references (a brand's known colour), **not endorsements**.
</Aside>

## The record

Each row is a `ColourKnowledgeRecord`, keyed to one of the 14 Berlin–Kay anchors and discriminated by `kind`:

```ts
{
  key: string;                    // deterministic identity, e.g. 'psychology:blue:trustworthy'
  colourAnchor: BasicColour;      // 'red' | 'blue' | … | 'gold' | 'turquoise'
  kind: 'psychology' | 'brand' | 'cultural';
  // kind = 'psychology'
  dimension?: string;             // an Impact-axis dimension: calm | trustworthy | energetic |
                                  //   bold | joyful | serious | luxurious | friendly (extensible)
  strength?: 'high' | 'medium' | 'low';
  valence?: 'positive' | 'negative' | 'mixed' | 'neutral' | 'context-dependent';
  // kind = 'brand'
  brandName?: string;             // 'Coca-Cola'
  brandHex?: string;              // the brand's reference colour
  industry?: string;              // 'Beverage'
  // kind = 'cultural' (the merged-in colour_semantics layer)
  scope?: 'cross-cultural' | 'regional' | 'religious' | 'domain';
  context?: string;               // 'China / East Asia', 'Western / Anglophone', …
  association?: string;
  culturalCaution?: boolean;      // a localisation landmine
  hueLo?: number; hueHi?: number; // approx OKLCH hue band (for the soft region lean)
  note?: string;
  confidence: 'high' | 'medium' | 'low';
  source?: string;                // EMPTY until a real citation is attached
}
```

Psychology dimensions deliberately reuse the [Method panel](/colour-palette/method-panel/)'s **Impact axis** vocabulary so the two stay in lockstep — one source of truth for "what blue feels like".

## Storage & API

The `colour_knowledge` table is **global, admin-curated reference data** — no `userId`, no relations, a sibling to the Thesaurus tables. Reads are open; writes are admin-gated. Identity is a deterministic `key` slug (e.g. `psychology:blue:trustworthy`, `brand:red:coca-cola`, `cultural:white:east-asia…`) so the seed upserts idempotently across all three kinds without nullable composite-unique pitfalls. **Cultural rows** carry the former `colour_semantics` fields; the cultural lookup helpers (`cautionsForPalette`, `cultureHueBias`, `availableCultures`) read them via `?kind=cultural`.

| Method | Path | Access |
|--------|------|--------|
| `GET` | `/api/v1/colour/knowledge` (`?anchor=` `?kind=`) | Authenticated |
| `GET` | `/api/v1/colour/knowledge/anchor/:anchor` | Authenticated |
| `POST` | `/api/v1/colour/knowledge` | Admin |
| `PATCH` | `/api/v1/colour/knowledge/:id` | Admin |
| `DELETE` | `/api/v1/colour/knowledge/:id` | Admin |
| `POST` | `/api/v1/colour/knowledge/seed` | Admin |

The seed ships per-anchor **psychology** rows (mapped to the eight Impact dimensions), a curated set of **brands** per colour family, and the **cultural** rows (the ~66 former colour_semantics associations), and upserts idempotently on `key` so re-seeding never clobbers admin additions.

## The per-colour card

On the colour profile **Analyse** tab (and, by reuse, the slide-in **Colour View** sheet) a **Colour Knowledge** card resolves the working colour to its nearest Berlin–Kay anchor (`nearestBasicColour`, the same classifier the semantics layer uses) and shows three columns for that anchor:

- **Psychology** — the Impact-axis dimensions the colour evokes, with strength + confidence.
- **Cultural semantics** — the cultural-kind rows (the merged-in [Colour Semantics](/colour-palette/colour-semantics/) layer: context + association + valence, cautions flagged).
- **Brands** — well-known brands that use the colour family, each with its reference swatch.

When nothing is recorded for an anchor the card points the admin to seed the data.

## Admin

The data is curated under **Admin → Colour Thesaurus → Knowledge**, a tab alongside the thesaurus' names and design-system sources. Three sub-tabs — **Psychology**, **Cultural**, **Brands** — each seed the built-in rows and browse them grouped by anchor. This **replaces the former standalone `/admin/colour-semantics` page**: cultural semantics now lives in this consolidated Colour Knowledge area.

<Aside type="note" title="Psychology moved out of the name layer">
Colour psychology used to live as free-text fields on the thesaurus **group** (`psychologySummary` / `psychologyPositive` / `psychologyNegative`, per named colour). Those fields have been **removed** from `colour_name_group`; psychology is now per-anchor structured data in `colour_knowledge`. The group keeps only name-scoped editorial metadata (`meaning` / `attributes` / `origin`).
</Aside>

## Seeding & authoring records

Two ways to populate the layer:

1. **Built-in defaults** — **Admin → Colour Thesaurus → Knowledge → "Seed reference data"** loads the shipped psychology + brand + cultural rows. It's **idempotent** (upserts on each row's `key`), so re-seeding never clobbers your edits or additions.
2. **Your own records** — `POST` them to `/api/v1/colour/knowledge` (admin-gated). Each is stored with `isSystem: false`, so a later re-seed leaves it untouched. There is no bulk-import endpoint yet — post one record per request (a small script loop for a batch).

Every record is one of three **kinds**. Only the fields for that kind apply; the rest are omitted. Use the colour's **Berlin–Kay anchor** as `colourAnchor` (`red` · `orange` · `yellow` · `green` · `turquoise` · `blue` · `purple` · `pink` · `brown` · `gold` · `silver` · `black` · `grey` · `white`).

```json
[
  {
    "colourAnchor": "blue",
    "kind": "psychology",
    "dimension": "trustworthy",
    "strength": "high",
    "valence": "positive",
    "confidence": "high",
    "note": "Blue–competence/trust (Labrecque & Milne 2012)"
  },
  {
    "colourAnchor": "red",
    "kind": "brand",
    "brandName": "Coca-Cola",
    "brandHex": "#F40000",
    "industry": "Beverage",
    "confidence": "high"
  },
  {
    "colourAnchor": "white",
    "kind": "cultural",
    "scope": "regional",
    "context": "East Asia and Hindu tradition",
    "association": "Mourning, death, funerals; white worn by Hindu widows",
    "valence": "negative",
    "culturalCaution": true,
    "hueLo": null,
    "hueHi": null,
    "confidence": "high",
    "note": "The single biggest East–West inversion"
  }
]
```

Post a single record like so (with an admin session):

```bash
curl -X POST https://your-app/api/v1/colour/knowledge \
  -H 'Content-Type: application/json' \
  --data '{ "colourAnchor": "green", "kind": "psychology", "dimension": "calm", "strength": "high", "valence": "positive", "confidence": "high" }'
```

### Field reference

| Field | Kinds | Required? | Values |
|-------|-------|-----------|--------|
| `colourAnchor` | all | ✅ | a Berlin–Kay anchor (see list above) |
| `kind` | all | ✅ | `psychology` · `brand` · `cultural` |
| `confidence` | all | ✅ | `high` · `medium` · `low` |
| `note` | all | optional | free text |
| `source` | all | optional | a citation (left empty in the built-ins — never fabricated) |
| `dimension` | psychology | ✅ | an Impact dimension: `calm` · `trustworthy` · `energetic` · `bold` · `joyful` · `serious` · `luxurious` · `friendly` (extensible) |
| `strength` | psychology | optional | `high` · `medium` · `low` |
| `valence` | psychology, cultural | optional | `positive` · `negative` · `mixed` · `neutral` · `context-dependent` |
| `brandName` | brand | ✅ | e.g. `Coca-Cola` |
| `brandHex` | brand | ✅ | 6-digit hex, e.g. `#F40000` |
| `industry` | brand | optional | e.g. `Beverage` |
| `scope` | cultural | ✅ | `cross-cultural` · `regional` · `religious` · `domain` |
| `context` | cultural | ✅ | the audience/context, e.g. `China / East Asia` |
| `association` | cultural | ✅ | the meaning, e.g. `Luck, prosperity, joy` |
| `culturalCaution` | cultural | optional | `true` flags a localisation landmine (surfaced as a non-blocking caution) |
| `hueLo` / `hueHi` | cultural | optional | approximate OKLCH hue band (drives the Culture axis's soft hue lean); `null` for L/C-defined anchors |

<Aside type="note" title="Identity is a deterministic key">
Each record's identity is a `key` slug computed server-side — `psychology:<anchor>:<dimension>`, `brand:<anchor>:<brand-slug>`, or `cultural:<anchor>:<context>-<association>`. It's `@unique`, so posting a record whose identity already exists (a built-in or one you already added) is rejected. The **built-in seed** is the canonical defaults; the seed array lives in code (`ColourKnowledgeSeed[]`) for contributors, and `POST` is for your own additions.
</Aside>

## Status

- **v2.31.0** — the data model (`colour_knowledge` table + admin-gated API + per-anchor psychology & curated brand seed), the per-colour Colour Knowledge card on the Analyse tab, the consolidated **Knowledge** tab under the Thesaurus admin (replacing the standalone Colour Semantics page), and the removal of the group-level psychology fields.
- **v2.32.0** — **merged the cultural layer in.** The `colour_semantics` table and `/semantics` endpoints were removed; their ~66 cultural associations are now `kind: 'cultural'` rows of `colour_knowledge` (seeded here, served from `/knowledge?kind=cultural`). The Method panel's culture selector / cautions / hue lean and the Knowledge tab's Cultural sub-tab read the cultural kind; behaviour is unchanged.

**Planned** (see the [Roadmap](/colour-palette/roadmap/)): wiring the axes *into* the criteria so the generator runs biased by each colour's psychology/cultural/brand match and a tolerance range; a configurable palette structure (e.g. a legal-type brand = 1 hero + 1 accent); locked colours analysed as an implicit axis; and treating the functional colour as the background/surface the rest of the palette must read against.
