The v1 thesaurus lives in client-side Zustand storage and couples two orthogonal concepts — **source provenance** and **editorial grouping** — into the same `sources[]` array. That coupling is why a source row with its own hex (BS381 Night at `#282b2f` vs XKCD night blue at `#040348`) doesn't surface cleanly as an alt colour, why design-system scales need special-cased hex matching, and why import dedupe has to walk tuples instead of doing a flat lookup.

v2 splits the concepts into four tables, moves the store to the backend as admin-global reference data, and derives every "alt names / alt colours" display list from relationships rather than storing them redundantly.

## Goals

1. **One place per concept.** A colour's *source provenance* (where it was imported from) lives on the entry. A colour's *editorial grouping* (what family it belongs to) lives on the group. A source is a namespace with its own config, not a tag squeezed into the entry row.
2. **Admin-global, read-only for users.** The thesaurus is reference data like a dictionary. Any user reads it; only admins write.
3. **Idempotent imports.** Re-running the same import produces no duplicates and no mutations to entries or groups.
4. **Ambiguous imports handled structurally.** When an incoming record's hex matches one group and its name matches a different group, the resolution is a schema-level many-to-many link rather than a separate chain table.

## The four tables

<Aside type="note">
  All tables follow the module-starter database conventions: `snake_case` names with a module prefix, `camelCase` fields, plain `String` foreign key fields with companion `@@index`, **no** `@relation` directives. Integrity between rows is enforced in the service layer.
</Aside>

### `colour_name_source`

Namespace rows — one per source the thesaurus knows about (`bs381`, `xkcd`, `tailwind`, `material`, `css`, `open-color`, brand-specific sources). Each carries a default `isDesignSystem` classification which entries inherit and may override.

```prisma
model colour_name_source {
  id             String   @id @default(cuid())
  name           String   @unique        // lowercased key
  displayName    String                  // original casing
  isDesignSystem Boolean  @default(false)
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt
}
```

### `colour_name_entry`

One row per `(sourceId, sourceCode, hex)` tuple — every imported colour record gets an entry. Hex is **indexed but not unique**: three sources can all describe `#f5f5f5` with their own names and their own source codes, giving three entries that later share a group.

```prisma
model colour_name_entry {
  id             String   @id @default(cuid())
  hex            String
  oklchL         Float?
  oklchC         Float?
  oklchH         Float?
  name           String
  nameNorm       String
  altNames       String[] @default([])
  sourceId       String
  sourceCode     String?
  isDesignSystem Boolean  @default(false)
  scaleKey       String?
  scaleStep      Int?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  @@index([hex])
  @@index([nameNorm])
  @@index([sourceId])
  @@index([scaleKey, scaleStep])
  @@unique([sourceId, sourceCode, hex])
}
```

**Why OKLCH fields.** Tailwind 4 and Open Color v2 define their palettes natively in OKLCH and hex is a derived projection. Preserving the native OKLCH keeps round-trips lossless — we never reverse-engineer OKLCH from hex when the source provided it directly. Fields stay null for sources that specified hex only; a later import from an OKLCH-native source backfills them on the same entry.

**Why `altNames[]` on the entry.** British Standards commonly lists two labels on a single row (`BS381 642 — "Night" / "Night Blue"`). Both belong on the one entry rather than forcing a synthetic sibling. These are source-provided aliases for the same hex, distinct from editorial alt names (which are derived from other entries in the group).

**Why `scaleKey` + `scaleStep`.** Design-system sources come in ladders. Tailwind `neutral-100` is a sibling of `neutral-50`, `neutral-200`, …, `neutral-950`. The `scaleKey` (e.g. `"tailwind:neutral"`) and `scaleStep` (`100`) give the Design Systems tab a trivial query:

```ts
await prisma.colour_name_entry.findMany({
  where: { scaleKey: 'tailwind:neutral' },
  orderBy: { scaleStep: 'asc' },
});
```

Both fields stay null for editorial sources (XKCD, BS381, CSS1).

### `colour_name_group`

Editorial grouping — what family a set of colours belongs to. The master name and master hex are **pointers to entries**, admin-selectable. They're usually the same entry but can diverge when the name canonical hex and the visual canonical hex come from different sources.

```prisma
model colour_name_group {
  id                 String   @id @default(cuid())
  masterNameEntryId  String?  @unique
  masterHexEntryId   String?  @unique
  deprecatedNames    String[] @default([])
  meaning            String[] @default([])
  attributes         String[] @default([])
  psychologySummary  String?
  psychologyPositive String[] @default([])
  psychologyNegative String[] @default([])
  origin             String?
  createdAt          DateTime @default(now())
  updatedAt          DateTime @updatedAt
}
```

**What isn't on the group.** Master name, alt names, alt colours — none of them are stored. They're all derived at read time from the group's primary members:

- `masterName` = `entries.find(e => e.id === group.masterNameEntryId).name`
- `masterHex` = `entries.find(e => e.id === group.masterHexEntryId).hex`
- `altNames` = union of every primary member's `name` and `altNames[]`, minus the master-name entry's contribution, de-duplicated case-insensitively
- `altColours` = unique hexes among primary members, minus the master hex

The single source of truth is always the entry rows. The group never has to keep a redundant cache in sync.

### `colour_name_entry_group`

Many-to-many membership with a reason. This replaces what would otherwise be a "chain table" between groups — because the ambiguity lives on a specific entry, not on the whole group.

```prisma
model colour_name_entry_group {
  id        String   @id @default(cuid())
  entryId   String
  groupId   String
  reason    String     // "primary" | "hex-match" | "name-match" | "manual"
  createdAt DateTime @default(now())

  @@unique([entryId, groupId])
  @@index([entryId])
  @@index([groupId])
  @@index([groupId, reason])
}
```

**Reason values:**

| Reason | When it's written | Effect on render |
|---|---|---|
| `primary` | Every entry has exactly one. Written on import rules 2, 3, 4. | The entry appears as a main member of this group — contributes to altNames / altColours / master pointer candidates. |
| `hex-match` | Ambiguous import: incoming hex matches this group but the primary link went elsewhere (to the name-matched group). | The entry appears in a secondary "Also referenced here" block on the group's profile. |
| `name-match` | Symmetric partner to `hex-match`. Written on the *other* group in the ambiguous case when primary went by hex. | Same as above — secondary block. |
| `manual` | Admin hand-linked the entry to an additional group via the UI. | Same as above. |

**Invariant:** every entry has exactly one row with `reason='primary'`. Enforced in the service layer (a guard on every write), not at the DB level, because Postgres doesn't allow partial unique indexes through Prisma's declarative schema alone. The invariant is documented and tested.

## Ingest rules

For each incoming record `{ hex, name, altNames?, sourceKey, sourceCode, oklch? }`:

### Rule 1 — exact row match

An entry with the same `(sourceId, sourceCode, hex)` already exists.
→ **Upsert:** update `name`, `altNames[]`, `oklch*` fields if changed. Membership rows untouched.

### Rule 2 — hex match only

No exact row match, but an entry at this hex exists (via any source).
→ **Create** a new entry.
→ Add `colour_name_entry_group { entryId: new.id, groupId: matchedEntry.primaryGroupId, reason: 'primary' }`.

### Rule 3 — name match only

No exact row match and no hex match, but an entry with this `nameNorm` exists.
→ **Create** a new entry.
→ Add `colour_name_entry_group { entryId: new.id, groupId: matchedEntry.primaryGroupId, reason: 'primary' }`.

### Rule 4 — no match

No matches anywhere.
→ **Create** a new group.
→ **Create** a new entry.
→ Add `colour_name_entry_group { entryId: new.id, groupId: new.groupId, reason: 'primary' }`.
→ Set `group.masterNameEntryId = new.id` and `group.masterHexEntryId = new.id`.

### Rule 5 — ambiguous

Incoming hex matches an entry in group `X`; incoming name matches an entry in group `Y`; `X ≠ Y`.
→ **Hex wins.** Create the entry and link it primary to `X`.
→ Also add `colour_name_entry_group { entryId: new.id, groupId: Y, reason: 'name-match' }` so the name collision is recorded without forcing a merge.
→ Surface both groups in the import report so the admin can resolve (merge, split, or accept).

### Normalisation

- **Hex:** lowercase, strip leading `#`, re-prefix. `#FF00aa` and `ff00aa` collide on lookup.
- **Name:** lowercase, trim, collapse whitespace runs to a single space. Punctuation is preserved (so `Grey 100` and `Grey-100` stay distinct — different design systems use different conventions).

### Transaction scope

Each record ingests inside a `withTransaction` block. Multi-record imports run the batch as a single transaction so a mid-import failure rolls back cleanly. Reports collect per-record outcomes (`created`, `merged-by-hex`, `merged-by-name`, `ambiguous`, `upserted`, `failed`) for the admin UI.

## Derived views

All the reading surfaces — the profile page's Names card, the toolbar autocomplete, the Design Systems tab — compute from the base tables rather than reading stored summaries.

### Names card (by working hex)

```ts
// 1. Find entries at this hex. Usually 1; may be several when multiple
//    sources describe the same hex.
const entriesAtHex = await prisma.colour_name_entry.findMany({
  where: { hex: workingHex },
});

// 2. For each, pull its primary group plus all entries in that group.
//    Pick the featured entry — the one whose group has it as master
//    name, else the one whose group has it as master hex, else the
//    first by createdAt.

// 3. Derived altNames = union of (primary member.name + primary
//    member.altNames[]), minus the master-name entry's contribution,
//    lowercased-dedupe.

// 4. Derived altColours = unique hex values among primary members at
//    a different hex to the master hex, with source/displayName hints
//    for the render.

// 5. Design system row = primary members whose source.isDesignSystem
//    (or entry.isDesignSystem override) is true and whose hex equals
//    the working hex. Rendered inline like "css whitesmoke, tailwind
//    neutral-100, material grey-100".
```

### Toolbar autocomplete (by text query)

```ts
// Simple prefix match on nameNorm:
await prisma.colour_name_entry.findMany({
  where: { nameNorm: { startsWith: queryNorm } },
  orderBy: [
    // Exact match first, then prefix, then length.
    { nameNorm: 'asc' },
  ],
  take: 16,
});
```

Trigram or tsvector indexes are a future upgrade if the seed grows past tens of thousands of entries; prefix match + btree is sufficient for v1.

### Design Systems tab (by group, by scale)

```ts
// The scale a given entry belongs to:
const siblings = entry.scaleKey
  ? await prisma.colour_name_entry.findMany({
      where: { scaleKey: entry.scaleKey },
      orderBy: { scaleStep: 'asc' },
    })
  : [];
```

## Auth model

- **Reads** — public within the app; no role gate. The thesaurus is reference data.
- **Writes** — admin role only, enforced at the route layer via the existing admin gate used by the current thesaurus admin page.
- **Workspace scope** — none. All four tables are module-global. There is no `workspaceId` column anywhere.

## Merge, split, rename

Editorial operations on groups that the admin UI surfaces:

- **Merge** `groupA` → `groupB`: update every `colour_name_entry_group` row with `groupId = groupA` to `groupId = groupB`, de-duping on `(entryId, groupId)` by keeping the stronger reason (`primary` > others). Union the editorial string arrays. Delete `groupA`.
- **Split**: move an entry's primary membership to a new group. Create the new group, flip one row's `groupId`, optionally seed the new group's master pointers from that entry.
- **Rename** the group's master name or master hex: repoint `masterNameEntryId` / `masterHexEntryId` to a different entry that is already a primary member of this group. The service validates membership before accepting.

All three run inside `withTransaction`.

## Migration from v1 (Zustand) to v2 (DB)

v1 persists to `localStorage` via `zustand/persist`. The migration path is:

1. **Export** the current Zustand store to a JSON payload (the existing admin export panel already does this).
2. **Run the reseed** against an empty v2 schema using the standard import endpoints.
3. The ingest rules above merge duplicates naturally — a Tailwind entry and a CSS entry at `#f5f5f5` will land as two primary members in one group automatically, no manual grouping required.
4. **Discard the localStorage state** once the server-side thesaurus is authoritative. The frontend admin store shrinks to ephemeral UI state only (selected entry, open tabs, dialog visibility).
