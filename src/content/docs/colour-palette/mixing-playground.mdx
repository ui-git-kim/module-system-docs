The Mixing Playground is a research tab on the Colour Profile's Mix page. It lets you visually compare multiple mixing algorithms side by side using the same input colours, so you can evaluate which approach produces the best output for different goals.

## Architecture

The playground is split across two template files:

- **`colour-mix-page.ts`** — the page shell: colour input section (base + mix colours + harmony quick-add), 4 tab buttons, save scheme, and colour wheel models.
- **`colour-mix-lab.ts`** — the `MixLab` component: all algorithm panels organised into 4 tabs with inline parameters and SVG visual diagrams.

Supporting utility:
- **`colour-ryb.ts`** — RYB cube interpolation (Gossett & Chen 2004).

## Tabs

### 2 Colour Mix

Compares algorithms that blend two colours (base + first mix colour). Each panel shows even-step and perceptual-step gradient strips with ΔE labels.

| Algorithm | What it does | Key params |
|-----------|-------------|------------|
| OKLCH Lerp (shorter hue) | Shortest path around hue wheel | Steps |
| OKLCH Lerp (longer hue) | Longer arc (rainbow-through) | — |
| K-M Spectral Mix | Kubelka-Munk reflectance mixing. 3 variants: raw, +Saunderson, +Munsell | Saunderson, Munsell |
| RYB Cube Interpolation | Trilinear interpolation in hand-tuned RYB cube | — |
| OKLab Cartesian | Rectangular a,b lerp (avoids polar discontinuities) | — |
| OKLab + chroma boost | L from OKLab (even spacing), H from OKLCH shorter arc, C from gamut % | Hue distribution bias |
| Harmony-mapped gradient | Chain-walk through every intermediate gamut triangle | — |
| pigmentMix | Bell-shaped chroma lift with 3 containment strategies | Strength, Position (t) |
| harmonizeMix | Brightness-weighted asymmetric blend | Median, Margin |
| neutralFromPair | Hue-cancel neutral between any two colours | Mix amount |

### Multi Colour Mix

Combines 3+ colours. Shows K-M weighted pile, OKLCH weighted pile, and Gurney key-colour hull sampling.

### Neutrals

Three approaches to neutral generation: K-M complementary neutral (5-step chroma reduction), OKLCH complementary neutral, and tinted black scale with adjustable L/C.

### Tonal Scales

Tint and shade ladders (K-M vs OKLCH), monochromatic painter strings, cusp-aware tinted endpoints, paint-mix tonal scale, and painter palette breakdown.

## Perceptual Steps — Open Research Problem

<Aside type="caution" title="Work in Progress">
The perceptual step spacing problem is not yet solved. The section below documents current findings and the open research question.
</Aside>

### The problem

When generating a gradient between two colours (e.g., blue H=262 → yellow H=95), even spacing in t produces steps that look uneven: the blue steps all look similar while the yellow steps look jumpy. Delta-E metrics say the steps ARE even — the problem is that delta-E doesn't capture what the eye actually sees.

### What we've tried

1. **Delta-E resampling** (equal ΔE_OK between steps) — produces near-identical results to even spacing because OKLCH is already designed to be perceptually uniform. The metric says it's even, the eye disagrees.

2. **Gamut-width-weighted zones** — distribute steps by inverse cusp chroma. The cusp chroma only varies ~1.4:1 between blue (0.252) and yellow (0.182), not enough to visibly shift the distribution.

3. **Gamut area weighting** — integrate maxChromaAtLH across L at each hue to get the gamut area, weight by inverse. Area varies ~2.5:1, slightly better but still insufficient.

4. **Cusp-following interpolation** — derive L from cusp position and C from gamut %. The cusp L landscape is inherently lopsided (blue/teal/green all at L≈0.50-0.55, yellow jumps to 0.88), which doesn't help.

5. **OKLab + chroma boost** — L from OKLab (even perceptual spacing), H from OKLCH shorter arc, C boosted to gamut %. Smoothest deltas so far. With a manual hue-distribution bias slider, values around 0.5-0.8 produce visually acceptable results for blue→yellow.

### The research question

> How do you distribute N colour samples along a hue arc between two arbitrary colours so that each adjacent pair appears equally distinct to a human observer, accounting for the varying gamut volume available at each intermediate hue?

### What the bias experiments tell us

For blue→yellow, a hue bias of ~0.5-0.8 (power curve on t) produces visually acceptable results. This value appears to correlate with the ratio of gamut areas between source and destination. If confirmed across more colour pairs, it could be computed automatically as:

```
hueBias = gamutArea(sourceH) / gamutArea(destH)
```

### Utility functions added for the playground

These live inside `colour-mix-lab.ts` as inline functions:

- `oklabMix(a, b, t)` / `oklabGradient` — OKLab cartesian interpolation
- `longerHueLerp` / `oklchLongerGradient` — OKLCH longer-hue arc
- `oklchWeightedPile` — weighted average in OKLab with circular mean for hue
- `sampleHull` — Gurney key-colour hull triangle/chain sampling
- `pigmentMixWithContainment` — pigmentMix with selectable gamut containment strategy
- `chromaThresholdFn` / `findThreshEdge` / `findThreshCurveEdge` — perceptual threshold (from harmony pipeline)
- `decomposeTR` / `reconstructFromTR` — gamut-relative decomposition/reconstruction
- `harmonyMapTo` — harmony cusp-line projection to arbitrary destination hue
- `harmonyGradient` — chain-walk through intermediate gamuts with delta-E resampling
- `cuspFollowGradient` — OKLab L + OKLCH H + gamut C% with hue distribution bias

### SVG diagram components

- `HueWheelDiagram` — two colours on a hue circle with shorter/longer arc
- `ABPlotDiagram` — OKLab rectangular a,b scatter with straight-line path
- `LCCrossDiagram` — gamut boundary at a hue with cusp and labelled points
- `BrightnessDiagram` — visual brightness comparison bars
