## 2.39.0 — 2026-07-03

### Criteria-driven hue targets (criteria redesign, Phase 2 — req 3, the heart)

- The **Impact** and **Culture** axes now derive their hue lean **live from the Colour Knowledge records** (default on): picking Impact = *Trustworthy* queries the psychology records for that dimension → the matching **positive** records' anchors → each anchor's representative hue, weighted by the record's strength (high 0.5 / medium 0.35 / low 0.2); picking a Culture uses its positive **cultural hue bands** (band midpoints), as the old localisation lean did — but now **live**, so editing the Colour Knowledge records changes generation directly.
- The dynamic targets **replace** those two axes' static hue centres (no double-count). **No matching records ⇒ no hue lean** — the criteria are the source of truth, so an unseeded host gets no lean rather than a silent static fallback. **L/C soft targets and hard bounds stay static**, and every dynamic centre is scaled by the axis's **influence** weight (v2.38.0), exactly like static ones.
- Achromatic anchors (black/white/grey) and silver carry no hue and contribute no target — their psychology influence remains the axis's L/C shape. **Gold** maps to the golden-yellow hue (~87.5°) the Culture axis already used.
- Each dimension's centre set is **normalised to a 0.5 aggregate budget** (the static Impact maximum), so a well-populated dimension shapes the lean's *distribution* without out-shouting the other axes' hue leans by sheer record count — adding records via bulk import changes the shape, not Impact's share of the pick. (Caught and fixed in the pre-ship adversarial review.)
- **Admin toggle** in `/admin/palette-methods → Axes & priority` (*Criteria-driven hue targets*, default **on**). Turning it off restores the static per-option hue leans (v2.33.0–v2.38.0 behaviour). While on, the static hue-lean editors are ignored for Impact and Culture; all other axes are unaffected.
- Engine: `MethodConfig.criteriaTargets?: { enabled }` (optional/lenient — missing means enabled); resolved hue centres now carry their **source axis** so the Playground's criteria layer can swap one axis's lean without touching the rest; new `psychologyHueBias(records, dimension)` beside the existing `cultureHueBias` in `palette-semantics.ts`. `resolveMethod` itself stays pure — the criteria merge is layered on its output in the Playground.
- **Note:** this partially reverses the v2.33.0 decision that froze the Culture lean into static admin config — with the toggle on, the records are live again (that was the point of the criteria redesign). The static path remains available via the toggle.

## 2.38.0 — 2026-06-30

### Per-axis influence weights (criteria redesign, Phase 2)

- New admin control (req 4a): an **influence** weight (0–1) per constraint axis, in **/admin/palette-methods → Axes & priority**. It scales how hard that axis's **soft** pull weighs in the merge — its L/C soft targets and hue bias centres are multiplied by the influence before the axes combine. **Hard min/max bounds are not affected.**
- This is the "let one axis dominate" control: keep everything at **1** for the previous behaviour, or turn an axis down (e.g. lower **Impact** so **Aesthetic** leads). An axis at **0** contributes no soft pull at all.
- Engine: `MethodConfig.influence?: Partial<Record<AxisKey, number>>` + `resolveMethod` multiplies each active axis's soft weight + hue-centre weights by `influenceFor(axis)` (missing/no-config ⇒ 1, clamped to [0,1]). **Optional + lenient** — a stored config without it behaves exactly as before, and `isValidOverride` still accepts it. Default config sets every axis to 1.

## 2.37.0 — 2026-06-30

### Locked colours as an implicit axis (criteria redesign, Phase 2)

- Locked colours now act like an extra constraint axis (req 6/7). When you **lock** colours in the Playground and roll, each locked colour is **analysed to its Berlin–Kay anchor** and the freshly generated colours are **leaned toward the nearest locked anchor's representative hue** — so the palette harmonises *toward* the locked vibe rather than spreading away from it.
- The lean is **gentle** (rolls still vary), **composes across multiple locked colours** (each generated colour follows whichever locked anchor it's nearest to, so multi-family palettes stay multi-family), and is **skipped under an active Aesthetic** (its per-role hue windows already define the look). **Neutral** locks (black/white/grey — e.g. a locked background surface) carry no hue and don't pull.
- New `ANCHOR_HUE` map + `anchorHue()` in `palette-bk-category.ts` (the canonical hue per chromatic anchor — the same midpoints the Culture axis uses); the Playground classifies each locked colour via `nearestBasicColour` and leans with `LOCKED_VIBE_LEAN` (default 0.3, tunable). Applied before the opt-in accessibility passes.
- **Note (judgement call):** the original plan merged the locked anchor hues into `resolvedMethod.soft.hueCenters`, but the generator short-circuits the locked path (it harmonises around the locked colour and ignores soft hue centres there), so that would have been a no-op. This ships an equivalent, contained **post-hoc lean** in the Playground roll instead — no change to the shared generators.

## 2.36.0 — 2026-06-29

### Functional = surface (criteria redesign, Phase 2)

- New opt-in **Readable on background** accessibility constraint, the "functional = surface" rule (req 8): when a functional colour is **locked as the background**, generated heroes and accents are nudged to clear a minimum **WCAG contrast** ratio against it — so the palette reads on that surface. A **dark** background lightens the rest; a **light** one darkens them.
- The background reference is the locked functional tagged role `background`, else the **darkest** locked functional. With no functional locked the pass is a **no-op** (lock one to use it).
- Method panel **Accessibility** section gains a **Readable on background** toggle + ratio slider; **default off**, with the default and ratio **admin-configurable** at `/admin/palette-methods → Constraints`.
- Engine: `enforceReadableOnBackground(colours, background, minRatio, range)` in `palette-generators.ts` (sweeps lightness, hue + chroma fixed, keeping the most-contrasting tone); applied in the Playground roll after dark-mode and before min-adjacent contrast. `MethodConstraintsConfig.readableOnBackground` added to the config + `isValidOverride`.
- **Note:** generated (unlocked) functionals still revert to hero on a roll, so structure-driven functional slots aren't auto-persisted yet — **lock** a functional to keep it as a surface. Auto-persisting functionals is a follow-up.

## 2.35.0 — 2026-06-29

### Palette structure from the Purpose axis (criteria redesign, Phase 2 · step 1)

- The **Purpose** axis now implies a palette **structure** — the hero / accent slot counts. Finance/Legal & Luxury resolve to **1 hero + 1 accent**; Data-viz to **6 heroes, no accents**; Children/Education to **4 heroes + 2 accents**, etc. (admin-editable per Purpose).
- The Method panel shows the resolved structure in the **Resolved constraints** readout, with an **Apply structure** button that restructures the current palette to those counts — adding/removing **unlocked** slots (locked colours are always kept, so the structure is a floor of the locked count) and then rolling to colour the new slots. It's **advisory until you click Apply** — generation never resizes your palette on its own.
- Admin: a **Palette structure** block on the Purpose options in **/admin/palette-methods → Axes & priority** (hero/accent count sliders). Also editable via Advanced JSON (`axes.purpose[i].contribution.structure`).
- The structure carries a **functional** count too, but it isn't applied yet (functionals are ephemeral until the functional-as-surface thread); Apply only prunes unlocked functionals so they don't revert into extra heroes.
- Engine: `ConstraintContribution.structure` + `ResolvedConstraints.structure`, resolved by `resolveMethod` (first active axis with a structure wins); `regenerate` now reads the store via `getState()` so a roll right after a restructure sees the new slots.

## 2.34.0 — 2026-06-29

### Colour Knowledge — bulk JSON import

- A new **Import** sub-tab under **Admin → Colour Thesaurus → Knowledge**: paste a JSON array of records (psychology / brand / cultural) and bulk-load them in one go. Backed by `POST /api/v1/colour/knowledge/import`.
- It validates each row, **upserts on the record's key** (creates new, updates existing in place), marks imported rows admin-owned (so a re-seed won't touch them), and reports **created / updated / skipped** counts plus a per-row error list. Idempotent — re-importing the same JSON is safe. Capped at 1000 records per request.
- The [Colour Knowledge docs](/colour-palette/colour-knowledge/) gained a "Seeding & authoring records" section with the sample JSON template and a field reference.

## 2.33.0 — 2026-06-29

### Culture becomes a first-class Method axis

- The Method panel's **Culture** is now a proper constraint axis, modelled exactly like the **Impact** (mood) axis — not a bolt-on dropdown. It has options with OKLCH `ConstraintContribution`s, is resolved by `resolveMethod` alongside Aesthetic / Style / Impact / Purpose, sits in the priority chain (last: Aesthetic > Purpose > Style > Impact > Culture), and renders as a **pill grid** in the panel like the other axes.
- **Editable in the Palette Methods admin** (`/admin/palette-methods` → Axes & priority) with the same per-option contribution editors (OKLCH bounds + soft target/weight + hue centres) used for Style / Impact / Purpose. Culture is **hue-only** by default — cultural colour meaning is about hue — and its defaults are **seeded from the Colour Knowledge cultural records** (the band midpoints of each culture's positive, hue-banded associations).
- The old separate culture hue-lean plumbing (a `regionBias` merged onto the resolved soft bias at roll time) is gone — the lean now flows through the axis like every other. The non-blocking localisation **cautions** stay (live from the cultural records), and now key off the Culture axis value.
- Cultures whose only association is a caution (e.g. Thailand — purple is mourning) contribute no hue lean by default; selecting them just surfaces the caution. Admins can add a lean.

## 2.32.0 — 2026-06-29

### Colour Knowledge — cultural semantics merged in

- The cultural **Colour Semantics** layer is now part of **Colour Knowledge**. The standalone `colour_semantics` table and its `/semantics` API endpoints have been **removed**; the ~66 cultural associations are now `kind: 'cultural'` rows of the unified `colour_knowledge` table, seeded alongside psychology and brands and served from `/api/v1/colour/knowledge?kind=cultural`.
- Everything that read cultural data — the Method panel's **Culture** selector, the non-blocking **cautions**, the soft per-region **hue lean**, the colour profile's **Colour Knowledge** card, and the Thesaurus admin's **Cultural** sub-tab — now reads the cultural kind. Behaviour and the lookup helpers (`cautionsForPalette`, `cultureHueBias`, `availableCultures`) are unchanged.
- One table, one API, one admin area, one seed for all per-colour knowledge (psychology + brands + cultural).

## 2.31.0 — 2026-06-29

### Colour Knowledge — per-colour psychology + brands (criteria redesign, Phase 1)

- A new **`colour_knowledge`** reference layer (sibling to `colour_semantics`) holds, per Berlin–Kay anchor, **colour psychology** — using the Method's eight Impact-axis dimensions (calm, trustworthy, energetic, bold, joyful, serious, luxurious, friendly) — and a curated set of **well-known brands** with their reference colours. Admin-gated writes, open reads, idempotent seed on a deterministic `key` slug. See [Colour Knowledge](/colour-palette/colour-knowledge/).
- The colour profile **Analyse** tab (and the slide-in Colour View sheet) gains a **Colour Knowledge** card: for the working colour it resolves the nearest anchor and shows its psychology, cultural semantics and brands together — finally putting the cultural/psychology data *on the colour profile* where it belongs.
- The admin moves under the **Thesaurus**: a new **Knowledge** tab (Psychology / Cultural / Brands) sits alongside Colour Names and Design Systems. This **replaces the standalone `/admin/colour-semantics` page** — cultural semantics is now the Cultural sub-tab.
- **Colour psychology moved out of the name layer:** the free-text `psychologySummary` / `psychologyPositive` / `psychologyNegative` fields are removed from the thesaurus **group**; psychology is now structured per-anchor data in `colour_knowledge`. The group keeps only name-scoped editorial metadata (`meaning` / `attributes` / `origin`).
- This is **Phase 1** of the criteria-driven Method redesign — the data foundation the axes will later plumb into (axis value → criteria match + tolerance range → generator biased by it).

## 2.29.0 — 2026-06-29

  - **Axis constraints** — pick a Style / Impact / Purpose option and tune its OKLCH lightness / chroma hard band (min–max) + soft target/weight + an optional hue lean, instead of editing raw JSON.

## 2.28.0 — 2026-06-29

### Playground + Admin: Min colour distance constraint

- A new opt-in **Min colour distance (ΔE)** accessibility constraint guarantees every palette colour is at least a set OKLab ΔE from every other — so no two swatches ever read as near-duplicates. Unlike the built-in distinctness pass (which the gradient generators skip), this runs on the final palette for **every** generator, including **Anchor blend** / Hue-shift / Cosine gradient. Toggle + ΔE slider (0.02–0.30) in the Method panel's **Accessibility** section; the default on/off and threshold are configurable globally in **/admin/palette-methods**. Default off; runs last (after dark-mode and min-adjacent contrast) so it has the final say on global distinctness.

## 2.27.7 — 2026-06-28

### Playground: Anchor blend — step dots sit on the eased path

- The wheel now draws the **actual eased path** and places the step dots on it, so the dots sit on the line (like the poline demo). Previously it drew straight chords between the anchors, but per-axis easing (X vs Y) curves the real path away from those chords, so the dots looked off the line. The path is densely sampled (smooth) and the dots are sampled from the same path.

## 2.27.6 — 2026-06-28

### Playground: Anchor blend — proper OKLCH colour wheel

- The Anchor blend wheel now renders the **actual OKLCH colour disc** (hue around the angle, lightness by radius, at max in-gamut chroma) on a canvas, instead of an approximate CSS gradient — so it looks like the real poline wheel and the anchor dots sit on their true hue. It redraws when you toggle invert-lightness, and the chord lines got a halo so they read on the vivid disc.

## 2.27.5 — 2026-06-28

### Playground: Anchor blend — fixed at 3 anchors

- Removed the anchor-count stepper. The Anchor blend is now always a **3-anchor** blend (re-rolled fresh on every Space), matching the typical poline setup — fewer knobs, more predictable gradients.

## 2.27.4 — 2026-06-28

### Playground: Anchor blend — Space is the only roll

- Removed the "Randomise anchors" button. **Every Space now rolls a fresh set of random anchors at random positions** and samples the blend into the playground (the wheel updates to show them) — so Space is the single roll action, consistent with the other generators. The anchor count is still set by the stepper.

## 2.27.3 — 2026-06-28

### Playground: Method panel scrolls; Anchor blend tidy-ups

- **The Method panel now scrolls internally** so the playground fits the viewport instead of growing the page. (The panel had a scroll area, but the height wasn't bounded to the viewport, so a tall panel stretched the whole layout.)
- **Anchor blend:** removed the colour-swatch preview strip from the settings (the wheel and the playground already show the result), and fixed a **duplicate-colour bug** — with **closed loop** on, the last slot used to repeat the first colour; the loop is now sampled so every slot is distinct.

## 2.27.2 — 2026-06-28

### Playground: Anchor blend — panel is settings, wheel is a visualisation

- Reframed the Anchor blend sub-panel as the **method settings** plus a **read-only wheel**. The wheel now just visualises the rolled anchors — their positions on the disc and the chords the palette is sampled along — instead of being hand-editable. The hand-editing that became a dead-end once Space re-rolls the anchors (dragging anchors, *From palette*, per-anchor chroma) is removed. Anchor count is a simple **2–6 stepper**, the per-axis easing selects are labelled (X·Hue/Light, Y·Hue/Light, Z·Chroma), and **Randomise anchors** rolls a fresh palette. Matches the real poline settings panel.

## 2.27.1 — 2026-06-28

### Playground: Anchor blend — Space re-rolls the anchors

- **Hitting Space (or Randomise) now re-rolls the anchor positions** within the resolved range and re-samples the blend with the current settings (per-axis easing, closed-loop, inverted-lightness) — so a roll is a fresh poline palette shaped by your settings, like the demo's "Randomize anchors". The wheel updates to the rolled anchors, so the panel and the playground stay in sync. (Previously Space re-used the fixed anchors, so it was deterministic.) The panel is the **method** (settings); Space is the roll.

## 2.27.0 — 2026-06-28

### Playground: Anchor blend rebuilt — real poline geometry + a draggable wheel

- The **Anchor blend** generator is rebuilt to faithfully reproduce [poline](https://github.com/meodai/poline)'s geometry, over OKLCH instead of HSL. Colours live on a **disc** (angle = hue, radius = lightness, depth = chroma) and the palette is sampled along the **chords between anchor points** — a straight chord sweeps hue *and* lightness, giving the signature curved gradients. (The old version blended OKLCH directly, which was flat and kinked at each anchor with 3+ anchors.)
- The full **position-function set** — linear, sinusoidal, asinusoidal, exponential, quadratic, cubic, quartic, arc, smoothStep — with **per-axis easing** (X / Y / Z), plus **closed-loop** and **inverted-lightness**. The ease flips on alternate segments so multi-anchor chains stay smooth.
- The sub-panel is now an interactive **draggable wheel** — drag anchor points on the disc, adjust each anchor's chroma, add / remove, **Randomise** or seed anchors **From palette**, choose the per-axis easing, toggle the options, and watch a live preview.

Adds a table, so installing this update needs a Prisma migration (`cd backend && npx prisma migrate dev`).

The accessibility-constraint defaults (and the dark-mode ratio) are seeded from the active config.

- The **Accessibility** section gains a **Dark-mode friendly** toggle (the second control from the mockup). When on, a generate nudges each swatch's lightness so it stays usable on **both** a light and a dark surface (trims near-white / near-black colours). **Off by default**; the exact dual-sided metric is a first cut to tune.

## 2.24.4 — 2026-06-26

### Playground: opt-in "Min adjacent contrast" accessibility constraint

- A new **Accessibility** section in the Method panel adds a **Min adjacent contrast** toggle + slider (1.5:1 – 7:1). When on, a generate nudges each swatch's lightness so it clears a minimum WCAG contrast ratio against its neighbour. **Off by default** — it never changes default generation. (The mockup's *Dark-mode friendly* constraint is the next step, pending visual tuning.)

## 2.24.3 — 2026-06-26

### Playground / Profile: separate working state (fixes the saved-palette leak)

- **The Playground and the Palette Profile now have fully independent working state.** Previously they shared one store, so generating (Space) in the Playground while a saved palette was open made the Profile / saved view show the *rolled* colours (the database was never changed — a browser refresh restored it). They no longer share anything: a roll or edit in one surface never touches the other.
- **A fresh Playground no longer keeps a stale saved-palette binding** — opening a new playground (or **New palette** from the library) clears any leftover saved-palette name / **Update** state, so a roll can't be accidentally saved over a different palette.
- **New palette / New playground always start fresh** and behave consistently.

## 2.24.2 — 2026-06-26

### Playground: batched colour-name lookups (fixes the 429 storm + Save), Generator dropdown, scrollable panel

- **One name request per roll, not one per swatch** — generating used to fire a nearest-name lookup for *every* swatch, so even a few rapid Space presses tripped the server's rate limit (HTTP 429). That storm also exhausted the shared rate budget and **broke Save palette**. The Playground now resolves all swatch names in a **single batched, debounced request** per roll (via a new name-only endpoint), and never retries on 429.
- **Generator is a dropdown** — the construction-strategy picker keeps its card look (label + one-line hint) but collapses into a single dropdown to save vertical space.
- **Scrollable Method panel** — the panel body now scrolls on its own, so a tall panel no longer pushes the whole page; the colour columns fill the viewport.

## 2.24.1 — 2026-06-26

### Playground: card-style Method panel + naming / interaction fixes

- **Card-style Method panel** — the Generator is a list of selectable **cards** (label + a one-line hint) and the constraint axes (Aesthetic / Style / Impact / Purpose) are **pill grids**; the Harmony scheme picker is a pill sub-panel and Culture is a full-width dropdown. The detailed "tends to feel…" copy and evidence tags now live in **hover tooltips** rather than tiny inline text.
- **Colour names no longer vanish under rapid rolls** — the thesaurus nearest-name lookup is now **debounced** per swatch and no longer **retries on HTTP 429**, so holding Space stops flooding the endpoint (the cause of the missing names).
- **Functional swatches show a colour name** — they now get the same nearest-name suggestion as hero / accent swatches, instead of the word "functional".
- **Space always generates** — it no longer opens the Generator control; choosing an option returns focus so the next Space rolls.
- **Edge "+" buttons** at the start / end of the row reveal on hover of that end column, matching the add-between buttons.
- **Names + hex sit higher** in each swatch, and the "Press Space" hint **auto-hides** after the first roll.

## 2.24.0 — 2026-06-26

### Method panel: Harmony becomes the Generator axis

Generation gains a first-class **Generator** dimension — *how* a palette is built — separate from the constraint axes that say *what* the colours should be.

- **Generator axis** — a new selector (**Auto**, **Harmony**, **Golden spread**, **Distinct**, **Anchor blend**, **Hue-shift**, **Cosine gradient**, **Scatter**) picks the construction strategy applied *within* the merged constraints. **Auto** keeps the previous varied roll. The old **Harmony** axis folds in as one generator, with a **contextual sub-panel** that swaps per selection (Harmony reveals a scheme select; Anchor blend will reveal its playground).
- **Restored modes + new generators** — the explicit modes removed in 2.22.0 are back (golden / distinct / the harmony schemes), plus three new native-OKLCH strategies: **Anchor blend** (poline-inspired path through anchor colours with an easing curve), **Hue-shift** (the pixel-art ramp) and **Cosine gradient** (Inigo Quilez per-channel cosine).
- **Constraint axes are now four** — Aesthetic × Style × Impact × Purpose-&-Industry. The priority chain drops Harmony: `Aesthetic > Purpose > Style > Impact`. An explicit Generator takes precedence over the Aesthetic's per-role band sampling (which now drives the roll only on **Auto**); the aesthetic still leans hue softly through the merge.
- **Harmony schemes keep their tonal spread** — Auto and the harmony schemes still get the deliberate light/dark lightness ramp most rolls; the other strategies build their own structure.
- **Config-as-data** — `MethodConfig` gains `generators` and `schemes`, extending the 2.23.2 admin-overridable seam. Backend config + the Palette Methods admin page follow in 2.25.0.

## 2.23.0 — 2026-06-26

### Method panel: the Aesthetic axis — colour regions, not fixed palettes

A fifth, optional Method axis that constrains generation to a **vibe**.

- **Aesthetic axis** — pick an aesthetic (**Cyberpunk**, **Retro-Futurism**, **Lunarpunk**, **Goth**, **Cottagecore**, **Minimalist / Scandinavian**, **Mid-Century Modern**, **Old Money / Ivy**) and generation is constrained to that look. The selector is **grouped into six families** (Neon / Futurist, Dark / Romantic, Natural / Cosy, Clean / Modern, Bold / Retro-graphic, Preppy).
- **Regions, not palettes** — each aesthetic is OKLCH **bands** (a hero/base band, an accent band, an optional functional band — L range + C range + hue windows) plus a **selection rule**, not a fixed set of hexes. Re-rolls (Space) sample *within* the bands, so each one looks visibly different but stays on-vibe. Versioned config data (`palette-aesthetics.ts`, `AESTHETIC_CONFIG_VERSION`).
- **Selection rules** — *pick-one* (Goth / Lunarpunk draw exactly one accent hue per roll), *warm-lead + one cool* (Retro-Futurism), and *avoid* (Lunarpunk rejects red / magenta).
- **Priority** — `Harmony > Aesthetic > Purpose > Style > Impact`: a strong driver that still yields to an explicit Harmony. It emits the same `ConstraintContribution` (a soft L/C lean + hue centres) and composes through the merge engine; the real per-role bands ride alongside and are applied at generation time, so a dark base + bright accent don't collapse into one region.
- **Functional colour type** — a third colour type now surfaces in the Playground *up with* hero and accent (backed by the existing functionals model), **absent by default**. Convert a swatch to **Functional** from its role menu; when the active aesthetic defines a functional band, a re-roll recolours those slots from it.
- **Honest by design** — the Aesthetic axis constrains *hue* more firmly than Style/Impact because it encodes a visual-style convention, not a mood effect; overall muted-vs-vibrant is left to the Style axis. Copy stays "tends to read…".
- First of a staged batch: 8 aesthetics across all six families; the remaining ~8 (Steampunk, Dark Academia, Studio Ghibli, Brutalist, Pop Art, Psychedelic, Postmodern/Memphis, Gen-Z Preppy) follow as config-data adds. See the [Roadmap](/colour-palette/roadmap/).

## 2.22.0 — 2026-06-22

### Playground redesign — full-bleed canvas, header controls, docked panels

## 2.17.0 — 2026-06-21

### Method panel (four-axis generation) + colour semantics layer

Two foundations for more intelligent — and honest — palette generation.

**Method panel — Harmony × Style × Impact × Purpose-&-Industry.** A new slide-in panel in the Playground with four optional axes, each defaulting to **Any**. Every axis value maps to an OKLCH `ConstraintContribution` held in versioned JSON config, tagged with its **evidence** (peer-reviewed vs convention). A merge engine composes the active axes: hard bounds **intersect**, soft targets **weight-average**, conflicts resolve by **priority** (Harmony > Purpose > Style > Impact), and a lower-priority bound **relaxes to a soft target** when the intersection would be empty. Style and Impact are driven mostly by lightness and chroma — the replication-proof levers (Valdez & Mehrabian 1994) — while hue biases stay soft. When any axis is set it drives generation; otherwise the existing mode + mood controls do. Copy is deliberately honest — "tends to feel…", never "makes users feel…" — and the panel surfaces each axis's evidence tag, the resolved constraints, and any relaxations. New `palette-method.ts` (engine + config); `palette-generators.ts` gains a soft-bias layer and a split-complementary scheme.

**Colour semantics — cultural & symbolic associations (data model).** A new global, admin-curated `colour_semantics` reference table keyed to the 14 Berlin–Kay + extended anchors (red … grey, gold, silver, turquoise). Self-contained (no `userId`, no relations); reads are open to any authenticated user, writes are admin-gated; ships an idempotent seed of 62 built-in records under `/api/v1/colour/semantics`. `source` is intentionally left **empty** (real citations are attached later — nothing is fabricated) and `confidence` is preserved so low-confidence/folkloric rows can be down-weighted. A new Berlin–Kay anchor classifier (`palette-bk-category.ts`) maps any OKLCH colour to its nearest anchor using **lightness and chroma, not hue alone**, so brown, pink and the neutrals are distinguished — the keystone the semantics lookup resolves against.

The frontend semantics consumer, the full per-colour attribute vector, the per-region localisation feed with non-blocking caution surfacing, and a native-reviewer pass on the regional claims are next — see [Method panel](/colour-palette/method-panel/), [Colour semantics](/colour-palette/colour-semantics/) and the [Roadmap](/colour-palette/roadmap/).

## 1.48.95 — 2026-04-29

### Refactor

- **Harmony pipeline rewritten as cusp-line projection.** The previous five-stage pipeline (H-K corrections, warm-lift, neon bell, multi-pass refinement) has been replaced by a single-pass geometric construction. `harmonyMap` decomposes the source colour as `(t, r)` inside a perceptual sub-triangle (vertices: `A` at the threshold-curve × gamut-boundary intersection, `B` at the constant-chroma threshold × gamut-boundary intersection, and `cusp` at the gamut peak). It then reconstructs the destination using the same `(t, r)` in the destination's sub-triangle. The mapping is mathematically self-inverse: `harmonyMap(harmonyMap(x, 180), 180) === x` in float OKLCH, with hex round-tripping limited only by 8-bit channel quantisation.
- **New helper `findThresholdCurveEdge(h, loL, hiL, rising)`.** Binary-searches for the L value where the gamut boundary crosses the chromatic-threshold *curve* (which itself varies with L). Used to place vertex A on the rising side where the gamut narrows fast.
- **Green/cyan neon suppression.** A Gaussian chroma cap centred at H=165° (σ=35°, peak strength 30%) pulls chroma back from the gamut boundary in the green-cyan band only, where sRGB's unusually wide gamut produced overwhelming neon results. Other hues are untouched.
- **Removed:** `CUSP_L_CEILING`, `warmBase` / `warmLiftL` / `warmLiftC`, `neonBell`, `neonRisk`, `coolnessSharp`, `hkDeficit`, the asymmetric Pass 5 reduction, and the dual-cusp pattern. The single-pass cusp-line projection subsumes all of these geometrically.
- **`harmonyMap` no longer calls `gamutMap3DShaded`.** The destination sits inside its own sub-triangle by construction, so containment is guaranteed without a final mapping step.

### Research section redesign

- The Harmonies tab's interactive **Research** view is now a hue slider (0–360° destination hue) instead of a relative angle offset, with three side-by-side gamut cross-sections — Source, Raw bary, and Harmony — at full width. The Harmony triangle shows the three construction lines (cusp→Q, B→edge-A→cusp, A→edge-cusp→B) converging at the destination point, and the values panel surfaces the three preserved ratios (`%cusp`, `%B`, `%A`).
