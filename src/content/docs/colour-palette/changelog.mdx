## 2.18.0 — 2026-06-21

### Colour semantics wired into the Method panel (Phase 2/3)

The colour-semantics layer now **drives generation and surfaces cautions**, not just stores data.

- **Frontend consumer & per-colour lookup.** A typed API client + TanStack Query hooks (`useColourSemantics`), and lookup helpers (`palette-semantics.ts`) that resolve any generated colour to its Berlin–Kay anchor and pull the matching records, derive the selectable audiences, and compute a palette's charged anchors.
- **Audience selector + localisation (Area 4).** The Method panel gains an **Audience** control (the regional/religious contexts present in the data). Selecting one folds a **soft hue lean** toward that audience's positive associations into generation.
- **Non-blocking cautions (Area 5).** For the selected audience, the panel surfaces quiet, dismissible notes for the palette's culturally-charged colours — e.g. *"White is commonly associated with mourning, death, funerals (negative, high confidence)"* — information, never a gate. With no audience set, a one-line hint lists any charged anchors. Copy reads "commonly associated with…".

Still to come: the full modelled per-colour attribute vector (PAD arousal/valence, nameability, CVD distances), the Method-panel benchmark, and the native-reviewer pass on the regional claims — see the [Roadmap](/colour-palette/roadmap/).

## 2.17.0 — 2026-06-21

### Method panel (four-axis generation) + colour semantics layer

Two foundations for more intelligent — and honest — palette generation.

**Method panel — Harmony × Style × Impact × Purpose-&-Industry.** A new slide-in panel in the Playground with four optional axes, each defaulting to **Any**. Every axis value maps to an OKLCH `ConstraintContribution` held in versioned JSON config, tagged with its **evidence** (peer-reviewed vs convention). A merge engine composes the active axes: hard bounds **intersect**, soft targets **weight-average**, conflicts resolve by **priority** (Harmony > Purpose > Style > Impact), and a lower-priority bound **relaxes to a soft target** when the intersection would be empty. Style and Impact are driven mostly by lightness and chroma — the replication-proof levers (Valdez & Mehrabian 1994) — while hue biases stay soft. When any axis is set it drives generation; otherwise the existing mode + mood controls do. Copy is deliberately honest — "tends to feel…", never "makes users feel…" — and the panel surfaces each axis's evidence tag, the resolved constraints, and any relaxations. New `palette-method.ts` (engine + config); `palette-generators.ts` gains a soft-bias layer and a split-complementary scheme.

**Colour semantics — cultural & symbolic associations (data model).** A new global, admin-curated `colour_semantics` reference table keyed to the 14 Berlin–Kay + extended anchors (red … grey, gold, silver, turquoise). Self-contained (no `userId`, no relations); reads are open to any authenticated user, writes are admin-gated; ships an idempotent seed of 62 built-in records under `/api/v1/colour/semantics`. `source` is intentionally left **empty** (real citations are attached later — nothing is fabricated) and `confidence` is preserved so low-confidence/folkloric rows can be down-weighted. A new Berlin–Kay anchor classifier (`palette-bk-category.ts`) maps any OKLCH colour to its nearest anchor using **lightness and chroma, not hue alone**, so brown, pink and the neutrals are distinguished — the keystone the semantics lookup resolves against.

The frontend semantics consumer, the full per-colour attribute vector, the per-region localisation feed with non-blocking caution surfacing, and a native-reviewer pass on the regional claims are next — see [Method panel](/colour-palette/method-panel/), [Colour semantics](/colour-palette/colour-semantics/) and the [Roadmap](/colour-palette/roadmap/).

## 2.16.0 — 2026-06-20

### Palette builder — roles, generators, View sheet, interactive Profile, favourites & history

The Palette Workspace becomes a real builder: colours flow into palettes by role, the Playground and Palette Profile both generate and edit in place, and colours can be favourited and revisited. Host-verified in a starter-template app.

- **Roles into palettes** — the Colour Profile holder is now a list of role-tagged entries (hero / accent / functional) plus a shortlist. The palette sheet groups pickers by role and offers **Open in Playground**, **Create draft**, and **add to an existing palette**. The legacy single-colour brand-save is removed.
- **Generator toolbox** (`palette-generators.ts`) — OKLCH generation primitives: golden-angle sequencing, distinct / best-candidate sampling, harmony inference + tone-line, `harmonySetColours` (analogous / complementary / triadic / tetradic / monochromatic), and a `generatePalette` orchestrator. Drives the Playground's spacebar regenerate.
- **Playground** — seven generation modes (Random, Distinct, Analogous, Complementary, Triadic, Tetradic, Monochromatic); editable hex per slot; a permanent role icon + role dropdown; seeds from the working colour; a **Save dialog** (name + type: brand / ui / game / illustration / web / marketing / data-viz / exploration); and a **History** panel that restores past generations.
- **Colour View sheet** — a wide right-hand sheet (Analytics / Harmonies / Scales) opened by the eye icon on any swatch; its "+" inserts the chosen harmony or scale colour next to the source.
- **Interactive Palette Profile** — drafts load into the workspace store and edit in place: role dropdown, lock, weight slider, shortlist swap, and View per colour; spacebar regenerate with undo; publishing keeps the published version read-only.
- **Favourites (database-backed)** — a new `colour_favourite` table with its own backend service and routes; a `useFavourites` hook; heart toggles on the Playground, Colour Profile, and View sheet; and a header button (via `headerToolbarRegistry`) scoped to the `/colour` and `/palette` routes.
- **Workspace store widened** — functionals, sub-palettes, reference palettes, context templates, changelog and a browsable history, with checkpoint / restore-snapshot and undo across all of them; legacy fractional weights are normalised on load.

The alts and changelog **data contracts** are in place (`PaletteAlts.custom[]`, extended changelog kinds, checkpoint/restore) but their dedicated UI tabs are not yet wired — see the [Roadmap](/colour-palette/roadmap/).

See [Palette](/colour-palette/palette/).

## 1.48.95 — 2026-04-29

### Refactor

- **Harmony pipeline rewritten as cusp-line projection.** The previous five-stage pipeline (H-K corrections, warm-lift, neon bell, multi-pass refinement) has been replaced by a single-pass geometric construction. `harmonyMap` decomposes the source colour as `(t, r)` inside a perceptual sub-triangle (vertices: `A` at the threshold-curve × gamut-boundary intersection, `B` at the constant-chroma threshold × gamut-boundary intersection, and `cusp` at the gamut peak). It then reconstructs the destination using the same `(t, r)` in the destination's sub-triangle. The mapping is mathematically self-inverse: `harmonyMap(harmonyMap(x, 180), 180) === x` in float OKLCH, with hex round-tripping limited only by 8-bit channel quantisation.
- **New helper `findThresholdCurveEdge(h, loL, hiL, rising)`.** Binary-searches for the L value where the gamut boundary crosses the chromatic-threshold *curve* (which itself varies with L). Used to place vertex A on the rising side where the gamut narrows fast.
- **Green/cyan neon suppression.** A Gaussian chroma cap centred at H=165° (σ=35°, peak strength 30%) pulls chroma back from the gamut boundary in the green-cyan band only, where sRGB's unusually wide gamut produced overwhelming neon results. Other hues are untouched.
- **Removed:** `CUSP_L_CEILING`, `warmBase` / `warmLiftL` / `warmLiftC`, `neonBell`, `neonRisk`, `coolnessSharp`, `hkDeficit`, the asymmetric Pass 5 reduction, and the dual-cusp pattern. The single-pass cusp-line projection subsumes all of these geometrically.
- **`harmonyMap` no longer calls `gamutMap3DShaded`.** The destination sits inside its own sub-triangle by construction, so containment is guaranteed without a final mapping step.

### Research section redesign

- The Harmonies tab's interactive **Research** view is now a hue slider (0–360° destination hue) instead of a relative angle offset, with three side-by-side gamut cross-sections — Source, Raw bary, and Harmony — at full width. The Harmony triangle shows the three construction lines (cusp→Q, B→edge-A→cusp, A→edge-cusp→B) converging at the destination point, and the values panel surfaces the three preserved ratios (`%cusp`, `%B`, `%A`).
