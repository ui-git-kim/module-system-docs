## 2.7.0 — 2026-05-01

Current release.

### Mixing Playground (new)

- **New Mix tab structure** — replaced the 7 mode tabs (Pair gradient, Weighted pile, etc.) with 4 goal-oriented tabs: 2 Colour Mix, Multi Colour Mix, Neutrals, Tonal Scales.
- **15 mixing algorithms** compared side by side with the same inputs: OKLCH shorter/longer hue, K-M spectral (3 variants), RYB cube interpolation (Gossett & Chen 2004), OKLab cartesian, OKLab + chroma boost, harmony-mapped gradient, pigmentMix (3 containment strategies), harmonizeMix, neutralFromPair, K-M/OKLCH piles, Gurney hull, K-M/OKLCH complementary neutrals, tinted black scale.
- **SVG visual diagrams** per algorithm: hue wheel arcs, OKLab a,b scatter, L-C gamut cross-sections with cusp/boundary/construction lines, brightness bars.
- **Inline parameters** per algorithm panel — each algorithm has its own controls right next to its output.
- **Perceptual steps research section** — gamut width chart, source/destination gamut triangles, even vs optimal gradient comparison.
- **Chain-walk harmony gradient** — walks through every 0.5-degree intermediate gamut using cusp-line projection, accumulates delta-E, resamples at equal perceptual intervals.
- **Hue distribution bias slider** on the OKLab + chroma boost algorithm for researching gamut-aware step spacing.
- **New utility**: `colour-ryb.ts` — RYB cube trilinear interpolation.
- **New component**: `MixLab.tsx` — the full playground component.
- **New docs page**: Mixing Playground — documents all algorithms, the open perceptual steps research problem, and key files.

## 1.48.95 — 2026-04-29

### Refactor

- **Harmony pipeline rewritten as cusp-line projection.** The previous five-stage pipeline (H-K corrections, warm-lift, neon bell, multi-pass refinement) has been replaced by a single-pass geometric construction. `harmonyMap` decomposes the source colour as `(t, r)` inside a perceptual sub-triangle (vertices: `A` at the threshold-curve × gamut-boundary intersection, `B` at the constant-chroma threshold × gamut-boundary intersection, and `cusp` at the gamut peak). It then reconstructs the destination using the same `(t, r)` in the destination's sub-triangle. The mapping is mathematically self-inverse: `harmonyMap(harmonyMap(x, 180), 180) === x` in float OKLCH, with hex round-tripping limited only by 8-bit channel quantisation.
- **New helper `findThresholdCurveEdge(h, loL, hiL, rising)`.** Binary-searches for the L value where the gamut boundary crosses the chromatic-threshold *curve* (which itself varies with L). Used to place vertex A on the rising side where the gamut narrows fast.
- **Green/cyan neon suppression.** A Gaussian chroma cap centred at H=165° (σ=35°, peak strength 30%) pulls chroma back from the gamut boundary in the green-cyan band only, where sRGB's unusually wide gamut produced overwhelming neon results. Other hues are untouched.
- **Removed:** `CUSP_L_CEILING`, `warmBase` / `warmLiftL` / `warmLiftC`, `neonBell`, `neonRisk`, `coolnessSharp`, `hkDeficit`, the asymmetric Pass 5 reduction, and the dual-cusp pattern. The single-pass cusp-line projection subsumes all of these geometrically.
- **`harmonyMap` no longer calls `gamutMap3DShaded`.** The destination sits inside its own sub-triangle by construction, so containment is guaranteed without a final mapping step.

### Research section redesign

- The Harmonies tab's interactive **Research** view is now a hue slider (0–360° destination hue) instead of a relative angle offset, with three side-by-side gamut cross-sections — Source, Raw bary, and Harmony — at full width. The Harmony triangle shows the three construction lines (cusp→Q, B→edge-A→cusp, A→edge-cusp→B) converging at the destination point, and the values panel surfaces the three preserved ratios (`%cusp`, `%B`, `%A`).

## 1.3.34 — 2026-04-08

### Features

- Import brand colours from Coolors URLs, Adobe Color URLs, hex input, and CSS variable blocks
- Auto-assign brand colours to semantic UI roles (action, structure, emphasis, highlight, background, foreground, neutral)
- Generate all 34 shadcn/ui CSS variables in OKLCH colour space for both light and dark modes
- Tinted neutral generation using brand hue for cohesive surfaces
- WCAG 2.1 AA contrast validation for every surface/foreground pair (with APCA support via `apca-w3`)
- Refinement sliders: neutral warmth, brand saturation, surface contrast, dark mode brightness
- Save, load, activate, and delete palettes per user
- Side-by-side light/dark mode comparison view
- Chart colour generation at 72° hue intervals
- Standalone **Colour Schemes** and **Colour Mix** showcase pages
- Embeds as a tab on Structure project nodes via `nodeTabRegistry`
- Self-contained Prisma schema (`colour_palette_saved`) with no cross-module relations
- Public API mounted at `/api/v1/colour-palette` with full CRUD plus `/active` and `/:id/activate`
- `PaletteOutput` exported as the canonical theme contract for consumer modules
