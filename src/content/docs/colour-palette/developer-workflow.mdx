This page documents how to work on the Colour Palette **module repository itself** — not how to use the installed module in an app. For that, see [Usage](/colour-palette/usage/).

## Repository Setup

<Steps>

1. Clone the module repo.

   ```bash
   git clone <module-repo-url> colour-palette
   cd colour-palette
   git submodule update --init --recursive   # pulls the docs submodule
   ```

2. Install dependencies and link for local testing.

   ```bash
   npm install
   npm link
   ```

3. Open it in your editor and read `CLAUDE.md` plus `src/module.ts` to get the lay of the land.

</Steps>

## Development Loop

The dev loop is: **edit a template → build → reinstall in a test app → verify**.

<Steps>

1. Edit the template file under `src/templates/` (or `src/module.ts` if you're changing config).

2. Compile.

   ```bash
   npm run build
   # or: npm run dev   (watch mode)
   ```

3. In a starter-template app you've cloned for testing, install or update.

   ```bash
   cd ../my-test-app
   npx colour-palette init                # first time
   npx colour-palette update               # subsequent edits
   npx colour-palette update --dry-run     # preview changes only
   ```

4. Run the app and verify your change.

5. Bump the version (see below) and commit.

</Steps>

### Useful CLI Commands

| Command | Purpose |
|---------|---------|
| `npx colour-palette init` | Install the module into a target app |
| `npx colour-palette init --force` | Overwrite existing files during install |
| `npx colour-palette update` | Update templates and schema in a previously installed app |
| `npx colour-palette update --skip-schema` | Update templates only — leave schema alone |
| `npx colour-palette update --dry-run` | Preview what would change without writing |
| `npx colour-palette uninstall` | Remove the module |
| `npx colour-palette info` | Print module metadata (id, version, templates, registrations) |

## Version Management

**Bump the version after ANY change to templates or `module.ts`.** The version is the only signal apps have to detect that an update is available, and the version header in each generated file drives hash comparison during `update`.

| Command | When to Use |
|---------|-------------|
| `npm run version:patch` | Bug fix in a template, typo, internal refactor |
| `npm run version:minor` | New template file, new feature, new registration |
| `npm run version:major` | Breaking change to `PaletteOutput`, removed export, schema migration |
| `npm run release:patch` (etc.) | Tagged release — pushes a git tag for the version |

After a release tag, push tags so consumers can find them:

```bash
git push --tags
```

<Aside type="caution">
If you skip the version bump, `npx colour-palette update` in target apps will compare hashes and may either miss the change entirely or report "no updates available" when there clearly are some. Always bump.
</Aside>

## Scaffold Updates

The scaffold infrastructure (`src/cli.ts`, `src/scaffold/*`, build scripts) lives upstream in the module starter template. Pull improvements from there with:

```bash
npm run scaffold:check    # preview upstream changes
npm run scaffold:update   # merge them in
```

| Updated by `scaffold:update` | NOT updated |
|------------------------------|-------------|
| `src/cli.ts` | `src/templates/**` (your templates) |
| `src/scaffold/**` | `src/module.ts` (your module config) |
| `package.json` scripts (merged) | `package.json` name/version/description |
| Scaffold sections in `CLAUDE.md` | Module-specific sections in `CLAUDE.md` |
| `tsconfig.json`, build scripts | `docs/` submodule |

If a merge conflict hits one of your customisations, resolve it manually — typically by keeping your version and re-applying any structural improvement from the upstream change.

## Common Development Scenarios

### Adding a new frontend page

<Steps>

1. Create the template string in `src/templates/frontend/pages.ts`.

2. Export it from `src/templates/frontend/index.ts`.

3. Add an entry to the `frontend.templates` map in `src/module.ts`:

   ```ts
   'pages/MyNewPage.tsx': frontend.MY_NEW_PAGE_TEMPLATE,
   ```

4. If it needs a route, add it to `frontend.routes.protectedRoutes`. If it needs a sidebar item, add it to the navigation registration template.

5. Bump the version: `npm run version:minor`.

</Steps>

### Adding a backend endpoint

<Steps>

1. Edit `src/templates/backend/routes.ts` and add a new route handler.

2. Add the corresponding service method in `src/templates/backend/services.ts`.

3. Add the type in `src/templates/backend/types.ts`.

4. **Make sure all relative imports use `.js`** — backend templates run as ESM in the target app.

   ```ts
   import { myFeatureService } from './my-feature.service.js';   // ✅
   import { myFeatureService } from './my-feature.service';      // ❌
   ```

5. Bump the version.

</Steps>

### Adding a database table

<Steps>

1. Edit `src/templates/schema.ts` and add the model. Use foreign key + `@@index`, **never** `@relation` to the User model.

2. Add the model name to `schema.modelNames` in `src/module.ts` so the scaffold can detect changes.

3. In a test app, run `npx colour-palette update`, then `npx prisma migrate dev --name colour_palette_<change>`.

4. Bump the version (`minor` for new tables, `major` if it removes or renames an existing column).

</Steps>

### Adding a new registry registration

<Steps>

1. Add a template export to `src/templates/frontend/registrations.ts`.

2. Either:
   - Add a `frontend.templates` entry if it's a custom path (e.g. `nodeTabRegistry` lives outside the standard `registrations[]` slot), **or**
   - Add an entry to `frontend.registrations[]` with the appropriate `type` (`preferences`, `settingsPage`, etc.) for scaffold-managed registries.

3. Bump the version.

</Steps>

### Adding an environment variable

<Steps>

1. Document it in this docs page and in `docs/.../configuration.mdx`.

2. Add it to the target app's `.env.example` via your install instructions (the scaffold does not currently auto-merge `.env.example` for this module).

3. Reference it in the relevant template via `process.env.MY_VAR`.

4. Bump the version (`minor`).

</Steps>

### Adding a prompt variable

<Steps>

1. Add an entry to `prompts: []` in `src/module.ts` with `name`, `message`, and `default`.

2. Reference it in templates as `{{variableName}}` or `{{variableName|default}}`.

3. Bump the version (`minor`).

</Steps>

### Module dependencies

If you make this module depend on another (e.g. require Structure to be installed first), declare it in `src/module.ts`:

```ts
dependencies: ['structure'],
```

The scaffold will refuse to install if the dependency is missing.

### Lifecycle hooks

`src/module.ts` supports `onInstall`, `onUpdate`, and `onUninstall` hooks. Use them for one-off setup tasks (seeding, file moves) that can't be expressed as templates. Currently this module sets `backend.hasInit: false` and uses no lifecycle hooks.

## Debugging & Troubleshooting

### Inspecting generated output

After `npx colour-palette init` runs in a test app, browse `frontend/src/features/colour-palette/` and `backend/src/features/colour-palette/` to see exactly what your templates produced. Diff against the previous state with `git diff`.

### Common template mistakes

| Symptom | Likely Cause |
|---------|--------------|
| `Cannot find module './foo'` in backend | Missing `.js` extension on relative import |
| `MyFeature` appears literally in generated code | Wrong placeholder casing — use `my-feature`, `myFeature`, `MyFeature`, `my_feature` |
| `Module not found: @/features/...` | The target app's path alias differs — check it's `@/` |
| Update doesn't pick up changes | Forgot to bump the version |
| Generated file has stale content | `--force` flag missing on init, or hash matches old version |

### Build errors

`npm run build` runs `tsc` over `src/`. The most common errors are:

- **TS2305 — module has no exported member** — you exported a template under one name in a file but imported it under another in `module.ts` or the index barrel.
- **TS1259 — esModuleInterop** — only an issue if you change `tsconfig.json`; don't.

### "Update not detecting changes"

Almost always a missing version bump. Check `package.json` and `src/module.ts` — both must show the new version (the `version:patch` script updates both).

### Verifying `.js` extensions

Quick grep before committing:

```bash
grep -rn "from '\.\./" src/templates/backend/ | grep -v "\.js'"
```

Should return nothing.

## Documentation Workflow

The `docs/` directory is a **git submodule** that lives independently. Push it separately, then update the parent reference.

<Steps>

1. Edit pages under `docs/src/content/docs/colour-palette/`.

2. Push the docs submodule.

   ```bash
   cd docs
   git add .
   git commit -m "Docs: describe the change"
   git push
   cd ..
   ```

3. Update the parent repo's submodule pointer.

   ```bash
   git add docs
   git commit -m "docs: update docs submodule"
   git push
   ```

4. The docs site at https://saas.uniqueicon.com.au deploys automatically via GitHub Actions.

</Steps>

<Aside type="caution">
If you skip step 2 (`cd docs && git push`), the parent repo will reference a commit that doesn't exist on the remote and CI will fail to clone the submodule.
</Aside>
