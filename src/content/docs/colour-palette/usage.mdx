After running `npx colour-palette init`, the feature lives at `frontend/src/features/colour-palette/`. Everything you need is exported from its `index.ts`.

## Embedding the Page

The main page component is exported and can be dropped into any layout — it does not require a route of its own.

```tsx
import { ColourPalettePage } from '@/features/colour-palette';

export default function ThemingPage() {
  return <ColourPalettePage />;
}
```

When the **Structure** module is installed, the page is also registered as a tab on project nodes via the `nodeTabRegistry`. No extra wiring required.

## Showcase Pages

Two standalone pages register themselves into the sidebar under the `workspace` section:

| Route | Component | Purpose |

| `/colour-schemes` | `ColourSchemesPage` | Browse and preview generated colour schemes |
| `/colour-mix` | `ColourMixPage` | Mix two colours and see the OKLCH gradient |

## The Store

State lives in a Zustand store at `store/palette-store.ts`. Import the hook anywhere:

```tsx
import { usePaletteStore } from '@/features/colour-palette';

function MyToolbar() {
  const { brandColours, generatedPalette, generate, importColours, savePalette } = usePaletteStore();
  // …
}
```

### Store Properties

| Property | Type | Purpose |
|----------|------|---------|
| `brandColours` | `BrandColour[]` | Current input colours with role tags and weights |
| `importSource` | `ImportSource` | Where the current colours came from |
| `generatedPalette` | `PaletteOutput \| null` | The generated light + dark palette |
| `currentPaletteId` | `string \| null` | ID of the loaded saved palette, if any |
| `currentPaletteName` | `string` | Name shown in the Save dialog |
| `savedPalettes` | `SavedPalette[]` | Cached list of the user's saved palettes |
| `isLoadingSaved` | `boolean` | Loading flag for the saved-palettes list |
| `isSaving` | `boolean` | Save-in-progress flag |
| `isDirty` | `boolean` | True when state has changed since the last save/load |
| `previewMode` | `'light' \| 'dark'` | Which mode the preview shows |
| `isGenerating` | `boolean` | Generation-in-progress flag |

### Store Actions

| Action | Signature | Purpose |
|--------|-----------|---------|
| `importColours` | `(input: string) => void` | Parse a Coolors / Adobe / hex / CSS string and replace brand colours |
| `addColour` | `(hex: string, roles?: ColourRole[]) => void` | Append a brand colour |
| `removeColour` | `(index: number) => void` | Drop a brand colour and rebalance weights |

| `toggleColourRole` | `(index: number, role: ColourRole) => void` | Add or remove a single role |
| `setColourWeight` | `(index: number, weight: number) => void` | Set a colour's weight (0–1) |
| `autoAssignRoles` | `() => void` | Run the assessment and assign suggested roles |

| `setCurrentPaletteName` | `(name: string) => void` | Set the name shown in the Save dialog |
| `loadSavedPalettes` | `() => Promise<void>` | Refresh the saved-palettes list from the API |
| `savePalette` | `(name: string) => Promise<SavedPalette \| null>` | Create or update the current palette |
| `loadPalette` | `(id: string) => Promise<void>` | Replace state with a saved palette |
| `deletePalette` | `(id: string) => Promise<void>` | Delete a saved palette |

## Backend API

All endpoints are mounted at `/api/v1/colour-palette` and require authentication. The current user is taken from `req.user` — there is no `userId` in the path.

| Method | Path | Body | Returns |
|--------|------|------|---------|
| `GET` | `/` | — | `SavedPalette[]` for the current user |
| `GET` | `/active` | — | The user's currently active `SavedPalette` (or null) |
| `GET` | `/:id` | — | A single `SavedPalette` |
| `POST` | `/` | `CreateSavedPaletteInput` | Created `SavedPalette` (201) |
| `PATCH` | `/:id` | `UpdateSavedPaletteInput` | Updated `SavedPalette` |
| `PATCH` | `/:id/activate` | — | Updated `SavedPalette` (sets `isActive: true`, clears others) |
| `DELETE` | `/:id` | — | 204 No Content |

## Admin — Colour Thesaurus

The module registers an admin page at `/admin/colour-thesaurus` for curating the colour thesaurus dataset. It's a single-operator editorial tool: data lives in `localStorage` between sessions and is committed back to the repo via JSON export.

**Four panels under one page:**

| Panel | Purpose |
|-------|---------|
| Browse | Searchable, sortable, filterable table of every entry. Row click opens the editor. |
| Import | File picker + drag-drop for one or many `.json` files. Preview before committing; conflict dialog for entries that match existing ones. |
| Add | Button that opens the editor on a fresh entry. |
| Export | Full export, per-hue-family export, or copy JSON to clipboard. |

### Import / export JSON format

Source files are either a bare array of entries or a `{ metadata?, entries }` wrapper. `id` fields are **not** part of the file format — the admin assigns an opaque `nanoid(8)` to every entry on import, and strips ids from every export. That means files round-trip through the admin cleanly without id collisions across machines.

### Dedupe pipeline — two passes

Each incoming entry runs through two passes in order:

**1. Exact source match** — any existing entry whose `sources[]` contains the incoming entry's `{ source, code }` tuple is an exact match. Merging in this case **overwrites** the existing entry with the incoming values and unions the `sources[]`. Purpose: idempotent re-import — importing `as2700.json` twice does not produce duplicates.

**2. Cross-source clustering** (only if step 1 found nothing) — match by either `masterName` (normalised: lowercase, trimmed, whitespace collapsed, compared against the existing entry's `masterName` and every `altNames` entry) or by canonical `hex` (exact string match on `#rrggbb`, no tolerance). If either hits, merge rules:

- existing `masterName` and `hex` are **sticky** — they do not change;
- the incoming `sources[]` items are appended (duplicates skipped);
- if the incoming `masterName` differs from the existing one and isn't already in `altNames`, it's added to `altNames`;
- editorial fields (`attributes`, `meaning`, `psychology`, `origin`, `deprecatedNames`) are **not** auto-merged — the user curates those manually.

Purpose: when AS 2700 and XKCD both contribute "Sapphire" with slightly different hex values, or "Charcoal" and "Dark Grey" with the same hex, they cluster into one entry rather than fragmenting.

If neither pass hits, a fresh entry is created with a new `nanoid(8)`.

Manually-added entries start with empty `sources[]` and never match anything on re-import — correct behaviour, since they're editorial, not sourced.

### SourceRef shape

Each item in `sources[]` is `{ source: string, code?: string, name?: string, hex?: string }`.

- `source` — namespace: `as2700`, `xkcd`, `css`, `tailwind`, `material-ui`, `open-color`, or any short slug you invent.
- `code` — stable identifier in that system: `B11`, `red-500`, `crimson`. Omitted when the source has no codes (XKCD).
- `name` — human-readable name when it differs from `code` (AS 2700's `B11` is named `Sapphire`). Usually omitted when `code` is already human-readable.
- `hex` — the source's own hex for this colour. Preserved so the editor and Analyse tab can show the cross-source spread, which may differ from the canonical entry hex.

The field pulls double duty: it's both the provenance trail (used for dedupe) and the cross-reference list (used for display). The Analyse tab renders a "Known as" row grouped by source, so a user looking at a colour can immediately see what to type in their own codebase — e.g. `CSS crimson`, `Tailwind red-500`.

### Authority toggle — hex vs OKLCH

Each staged file in the Import preview gets a radio: which field should be trusted as-shipped?

- **Hex-authoritative** (default) — the file's `hex` is trusted; `oklch` is recomputed from hex via culori. Use for AS 2700, XKCD, CSS named colours — sources that publish hex as their canonical form.
- **OKLCH-authoritative** — the file's `oklch` is trusted; `hex` is recomputed by running the OKLCH through `gamutMap3D` (perceptual hue-preserving gamut mapping) and formatting to sRGB. Use for Tailwind v4, which publishes wide-gamut OKLCH as the source of truth. A naive RGB clip would drift hue; `gamutMap3D` preserves it.

The toggle is per file, so multi-file imports can mix modes. A file's `metadata.notes` (if present) is rendered verbatim under the file preview as an authoring hint; it does not auto-select the toggle.

### Design-system sources are exact-only

A source is either an **editorial concept** (AS 2700, XKCD, manual curation) or a **design-system token** (CSS, Tailwind, Material, Open Color, Ant Design). The distinction matters because only concepts are valid as the masterName of a thesaurus entry — design-system tokens are references that attach to a concept via an exact hex match.

Design-system sources are recognised by namespace: `css`, `tailwind`, `material`, `material-ui`, `open-color`, `ant`, `ant-design`. When importing a file tagged with one of these namespaces — or any file with `metadata.exactOnly: true` — the admin will:

- Attach each incoming entry to an existing entry via exact hex match (adds the token to `sources[]`).
- **Skip any entry that has no existing hex match** — rather than create a standalone entry with the token as masterName. The preview footer surfaces the skipped count so you know how many tokens didn't attach.

Import editorial sources first (for the concept entries), then design-system sources (for the cross-references). If you import Tailwind before anything else, every token gets skipped — there's nothing to attach to.

At runtime, entries whose entire `sources[]` are design-system namespaces are excluded from the Analyse tab's "Closest named" lookup. They still appear under "Known as" when a concept entry references them.

### altNames cleanup on import

If any `altNames` entry exactly matches (after lowercase + trim + whitespace-collapse) a `sources[].name` or `sources[].code` on the same incoming entry, it's stripped before the entry enters admin state. The stripped string stays searchable via the extended admin search. Reason: `altNames` is reserved for genuine alternative display names (Sage ↔ Sage Green, Aqua ↔ Cyan), not source keywords like `red-500` that already live in `sources[]`. This runs on every import regardless of authority toggle.

### Shorthand import for design-system palettes

Bulk imports of design-system palettes (CSS named colours, Tailwind tokens, Material UI, Open Color) don't need pre-processing into full entry shape. Use file-level metadata:

```json
{
  "metadata": { "sourceName": "tailwind" },
  "entries": [
    { "name": "red-500", "hex": "#ef4444" },
    { "name": "blue-500", "hex": "#3b82f6" }
  ]
}
```

When `metadata.sourceName` is set, each `{ name, hex }` entry is promoted to a full entry: `name` becomes the masterName, and a `sources[]` row tagged with the declared namespace is synthesised. Cross-source clustering (by hex) will merge matching colours into existing thesaurus entries, adding the design-system token to the entry's sources list.

### Search

The admin browser searches across `masterName`, `altNames`, `deprecatedNames`, and every `sources[].source` / `sources[].code` / `sources[].name` — so `tailwind red-500`, `crimson`, `B11`, or `Sapphire` all find the relevant entry.

### hueFamilyOverride

Every entry has an auto-derived `hueFamily` (one of the nine `ThesaurusHueCategory` values, computed from OKLCH chroma + hue in `utils/thesaurus-classify.ts`). For the rare case where the classifier disagrees with editorial intent — e.g. Charcoal reads as grey but technically classifies as `blue` — the admin lets you set `hueFamilyOverride`. When set, the override takes precedence everywhere: display, filtering, and export grouping. The auto-derived value stays visible in the editor for comparison.

### Admin state

Admin state lives in a separate Zustand store (`store/thesaurus-admin-store.ts`) with the `persist` middleware wired to `localStorage` under the key `colour-thesaurus-admin-v1`. Nothing about the admin touches the runtime `data/thesaurus.ts` file that ships with the build — exports are the handoff point.
