This page tracks ongoing work for the Colour Palette module. Items move from **Planned** down to **Completed** as they ship. Each entry includes enough context that a future Claude Code session can pick it up without extra explanation.

There are two tracks: **hardening the current pipeline** (the shipped shadcn-theme flow — Known Issues and Planned Improvements below) and **building the new [Palette Workspace](/colour-palette/palette/)** (the multi-colour composition system that replaces the current palette flow — see [Palette Workspace Roadmap](#palette-workspace-roadmap)).

## Active Work

The colour engine's mixing pipeline is the active build. It feeds directly into the [Palette Workspace](/colour-palette/palette/).

### Mix step distribution

- **Where:** Mix tab (`colour-mix-lab.ts`) — the 2-colour gradient step-distribution stage.
- **Status:** In progress. The category-width approach works with a manual bias; it needs **auto-normalisation**, and a **cusp-peak** approach is being explored as a replacement.
- **Why it matters:** this is the algorithm the Playground's "add between" calls at N=1, so it unblocks both gradients and palette interpolation.
- **See:** [Mixing Playground](/colour-palette/mixing-playground/) for the full algorithm comparison and the open perceptual-step question.

### Chroma allocation

- **Where:** Mix 2-colour pipeline — the stage after step distribution.
- **Status:** Next up.
- **Symptom:** yellow looks washed out in intermediate steps because OKLCH lerps chroma linearly rather than using cusp-relative values.
- **Fix direction:** allocate chroma as a fraction of each intermediate hue's gamut cusp rather than interpolating the raw C value. Runs as a separate pipeline stage after step distribution lands.

### Example/demo views need functionals connected

- **Where:** the showcase/demo palette views.
- **Status:** To do alongside the Palette Profile build.
- **What:** the example views render heroes/accents but don't yet wire up the [functionals](/colour-palette/palette/#functionals-section) (background/text/surface). Connect them once the functionals model lands.

## Known Issues

### No input validation on backend POST/PATCH routes

- **Where:** `backend/src/features/colour-palette/colour-palette.routes.ts` — POST and PATCH handlers.
- **Severity:** Critical (blocking)
- **Symptom:** Raw `req.body` is passed directly to Prisma. Malformed payloads cause 500 errors with Prisma stack traces instead of clean 400 responses. No zod schema validates field presence, types, or size.
- **Fix:** Add a zod schema for `CreateSavedPaletteInput` and `UpdateSavedPaletteInput`, validate in the route handler before calling the service.

### `setActive` race condition — no transaction

- **Where:** `backend/src/features/colour-palette/colour-palette.service.ts` — `setActive` method.
- **Severity:** Critical (blocking)
- **Symptom:** `updateMany` (deactivate all) and `update` (activate one) are separate queries. Concurrent requests can leave multiple palettes active or none active.
- **Fix:** Wrap both operations in `prisma.$transaction()`.

### All API errors silently swallowed in store

- **Where:** `store/palette-store.ts` — `loadSavedPalettes`, `savePalette`, `loadPalette`, `deletePalette` all catch and discard errors.
- **Severity:** Critical (blocking)
- **Symptom:** Failed API calls leave the UI stale with no user feedback. No `error` state variable exists in the store.
- **Fix:** Add `error: string | null` to store state. Set it in catch blocks. Surface in UI via toast or alert.

### No React error boundaries on pages

- **Where:** All page templates (`pages/MyFeaturePage.tsx`, `pages/ColourSchemesPage.tsx`, etc.).
- **Severity:** Critical (blocking)
- **Symptom:** A render error (bad colour data, NaN in style prop) crashes the entire app to a white screen.
- **Fix:** Wrap each page in the starter template's `ErrorBoundary` component, or add a local fallback.

### Generate runs synchronously inside `requestAnimationFrame`

- **Where:** `store/palette-store.ts` — the `generate` action wraps `generatePalette` in a single `requestAnimationFrame` callback.
- **Severity:** Medium
- **Symptom:** With large brand inputs (8+ colours) the generation can briefly block the main thread, dropping a frame on slow devices.
- **Workaround:** Reduce the number of brand colours, or wait for the planned web-worker offload.

### No version history on saved palettes

- **Where:** Schema — `colour_palette_saved` stores only the latest snapshot.
- **Severity:** Low
- **Symptom:** Users cannot revert to a previous version of a saved palette after editing.
- **Workaround:** Save under a new name before making destructive edits.

### Colour Palette tab does not respond to `theme.changed` yet

- **Where:** `registrations/action.registration.ts` — the `theme.changed` handler is registered but the body is empty.
- **Severity:** Low
- **Symptom:** Switching themes elsewhere in the app does not update the palette preview live.
- **Workaround:** Reload the page.

## Active Research

### Perceptual step spacing for gradients

- **Where:** Mixing Playground (`colour-mix-lab.ts`) — Perceptual Steps section and OKLab + chroma boost algorithm.
- **Status:** Open research problem. Multiple approaches tried, none fully solve it.
- **Problem:** Even hue spacing produces gradients where some regions (blue) look uniform while others (yellow) look jumpy. Delta-E says the steps are already even, but human perception disagrees. The core issue is that the number of distinguishable colours varies by hue (gamut volume), and no current metric captures this for step distribution.
- **Current best approach:** OKLab L (even perceptual lightness) + OKLCH shorter-hue H + gamut-relative C%, with a manual hue-distribution bias slider. Values around 0.5-0.8 produce acceptable results for blue→yellow, but the bias needs to be computed automatically from gamut properties.
- **Research question:** How to distribute N colour samples along a hue arc so each adjacent pair appears equally distinct, accounting for varying gamut volume at each intermediate hue.
- **Key finding:** The bias value correlates with `gamutArea(sourceH) / gamutArea(destH)`. Needs validation across more colour pairs.
- **See:** [Mixing Playground docs](/colour-palette/mixing-playground/) for full details, attempted approaches, and search terms.

## Planned Improvements

### Short-term (next version — production blockers)

- **Add zod input validation to backend routes.** Create schemas for `CreateSavedPaletteInput` and `UpdateSavedPaletteInput`, validate `req.body` before calling service methods, return 400 with field-level errors.
- **Wrap `setActive` in `prisma.$transaction()`.** Prevent concurrent requests from corrupting the active-palette state.
- **Add error state to store and surface in UI.** Add `error: string | null` to Zustand store, set it in all catch blocks, display via toast or inline alert in BrandImport/PalettePreview.
- **Add React error boundaries to all pages.** Wrap each page component to catch render errors and show a fallback instead of crashing.
- **Fire `colour-palette.generated` and `colour-palette.activated` actions.** Add `actionRegistry.doAction(...)` calls in the store's `generate` and `setActive` paths so consumer modules can react. Bump minor version, document under Integration → Action Hooks Fired.
- **Wire up the `theme.changed` listener** so the preview re-renders when the active theme changes.

### Short-term (high-impact improvements)

- **Show save failure feedback.** `savePalette` returns null on error but PalettePreview doesn't check — add toast or inline error.
- **Fix `isDirty` flag during concurrent edits.** Track a save-started timestamp so `isDirty` only resets if no edits occurred since save began.
- **Add pagination to `getAll`.** Backend `getAll` returns all palettes; add `limit`/`offset` params to prevent massive responses.
- **Add request deduplication.** Use AbortController or a request-in-flight flag to prevent duplicate saves/deletes on rapid clicks.
- **Make Coolors URL parser case-insensitive.** Add `/i` flag to the Coolors regex so `COOLORS.CO` and mixed case work.
- **Support 3-digit hex shorthand in import fallback.** Extend the fallback regex from `{6}` to `{3,6}`.
- **Return error types from `importPalette`.** Distinguish "no colours found" from "invalid URL format" so the UI can guide the user.

### Medium-term (upcoming)

- **Web-worker offload for `generatePalette`.** Move palette generation off the main thread for large inputs.
- **Palette version history.** New `colour_palette_history` table that stores previous snapshots with `restoreFrom(historyId)` action.
- **Export to CSS / Tailwind config.** One-click export of the active palette as a `globals.css` block, a `tailwind.config.ts` theme extend, or a JSON registry blob.
- **APCA-first contrast mode.** Optional toggle to validate against APCA targets instead of WCAG 2.1 AA, with UI to explain the difference.
- **Filter hooks.** Add `colourPalette.beforeGenerate` and `colourPalette.afterGenerate` filters so consumer modules can mutate input or output. Deferred — will land with the dedicated style module.
- **styleRegistry contribution.** Push the generated 34-variable palette into `styleRegistry` so the runtime theme tracks the active saved palette. Deferred — will land with the dedicated style module.
- **Active-palette dashboard widget.** Register a dashboard widget that shows the currently-active palette at a glance. Deferred — will land with the dedicated style module.

### Long-term / Ideas

- **Closest Pantone colour.** Show the nearest PMS swatch alongside the OKLCH / hex / Munsell readouts on the Analyse tab, similar to the existing Munsell approximation. **Parked on licensing** — Pantone's developer API lives behind the X-Rite Developer Portal and requires a commercial licence (the consumer `connect.pantone.com` signup does not grant API access). Unofficial PMS → RGB tables exist on GitHub but shipping them is grey-area Pantone IP. Pick up once sanctioned credentials exist, then wire a backend proxy call (Pantone API terms generally prohibit surfacing raw PMS values in the frontend).
- **Image-driven palette extraction.** Drop an image, extract dominant colours via k-means in OKLCH space, feed straight into the brand input.
- **AI-assisted role assignment.** Use a small model to suggest functional roles based on brand description text.
- **Palette diff view.** Visual diff between two saved palettes, side-by-side, with per-token deltas.
- **Multi-brand support.** A single saved record holding multiple sub-palettes for sub-brands or campaign themes.
- **Figma plugin.** Push the active palette into a Figma library as colour styles.

## Palette Workspace Roadmap

The build plan for the new [Palette Workspace](/colour-palette/palette/) — the multi-colour composition system that replaces the current palette save/generate flow. Items are written so a future session can pick any one up cold.

<Aside type="tip" title="Shipped in v2.8.0">
The short-term set below — palette data model + API, the weight system, the Playground, the Palette Profile, and the Library — shipped as a first pass in module **v2.8.0** (pending install verification in a host app). The medium- and long-term items remain planned.
</Aside>

### Short-term (shipped v2.8.0 — first pass)

- **Palette Playground core.** Slot strips with lock/unlock, regenerate-all of unlocked slots, reorder, remove (to a recently-removed tray), add-between (calls step-distribution at N=1), and the per-slot consideration shelf. Lock state is UI-only React state; considerations persist in the palette JSON. See [Playground](/colour-palette/palette/#palette-playground-level-2).
- **Palette Profile core.** The live palette strip with swap-in-place, the alts section (light/dark/muted from the Scales engine), and the functionals section. See [Palette Profile](/colour-palette/palette/#palette-profile-level-3).
- **Palette Library.** Grid of cards (colour strip, name, status badge), eye-icon quick preview (expand in place), and search/filter/sort. See [Library](/colour-palette/palette/#palette-library).
- **Palette data model.** The `palette` table (`structureNodeIds`, `versionChainId`, heroes/accents/functionals/subPalettes/referencePalettes JSON) plus CRUD API endpoints. See [Palette Structure](/colour-palette/palette/#palette-record).
- **Weight system.** The auto-weight algorithm (`accentPool = min(n × 5, 25)`), per-group slider redistribution, and per-colour weight lock (`weightLocked`). See [Auto-weight algorithm](/colour-palette/palette/#auto-weight-algorithm).

### Medium-term

- **Sub-palettes** with their own playground-style generation (lock/regenerate; derivation from palette colours, distinct, or manual). See [Sub-palettes](/colour-palette/palette/#sub-palettes-section).
- **Context Preview (Visualize)** with user/admin-defined weight templates, starting with the proportional-blocks preview. See [Context Preview](/colour-palette/palette/#context-preview-visualize).
- **Reference palettes** — auto-detection of nearest Tailwind/Material/Open Color/Radix/shadcn matches by ΔE_OK, plus manual assignment.
- **Published palette exports** — CSS custom properties, Tailwind config, JSON, palette strip image.
- **Brand identity print export** — a published palette as a printable brand colour specification page.
- **Version chain model** — immutable publish (draft → published vN), edit-creates-new-draft, version navigation and forking.
- **Structure module integration** — palette tabs on client/project nodes via multi-assignment (`structureNodeIds`), dashboard widget, count badge.
- **Style module integration** — "import from palette" into the theme builder; the publish step that turns a published palette into the runtime theme.

### Long-term / Deferred

- **UI role mapping.** Map published-palette colours to shadcn CSS variables. Only for published palettes; build after publishing is solid. This is the current pipeline's headline feature, repositioned as a post-publish step.
- **Image extraction.** Drop an image, extract dominant colours via k-means in OKLCH, feed into brand input.
- **Print outputs.** CMYK values and nearest Pantone match (Pantone parked on licensing — see Long-term / Ideas above).
- **Advanced colour theory for the generator.** Richer generation algorithms beyond the current random / harmony-aware / distinct / mood-preset set. Needs more research.
- **Sub-palette derivation presets.** DataVis/Scales-based presets, finalised once the colour engine work settles.
- **Stock SVG mockup templates** for context preview (layout mockups beyond proportional blocks / geometric patterns).
- **CVD (colour vision deficiency) simulation** — a separate accessibility pass.
- **AI-assisted palette generation** — after the manual tools are solid.
- **Palette sharing / collaboration** — single-user first.
- **Public palette marketplace** — post-launch community feature.

## Open Questions

Decisions to resolve as the Palette Workspace is built (from the design specification). Each is a real fork, not an ambiguity in shipped behaviour.

1. **Context preview visual format** — start with proportional colour blocks (simplest), geometric patterns (more interesting), or layout mockups with stock SVG imagery (most realistic)? Leaning blocks-first.
2. **System template mapping** — when a system context template is applied to a palette it hasn't seen, how does it map generic weight hints to specific palette colours? By weight rank, or manual assignment?
3. **Neutral scale access** — every colour has its Scales tab for shade/tint ramps. Should the Palette Profile show a combined neutral scale derived from all heroes (via K-M mix), or is that fully handled by functionals now?
4. **Reference palette auto-detection** — automatically scan for nearest ΔE_OK matches in reference systems, or only show references when manually assigned?
5. **Multi-structure-node UX** — when a palette links to both a client and a project under that client, how does it appear in the Library? Nested, or shown once with both tags?

## Completed

### Colour engine (built)

- **Colour Profile tabs** — Analyse, Gamut, Harmonies, Data Vis, Scales, and Adjust are built; Mix is in progress (see [Active Work](#active-work)).
- **K-M spectral engine** — Kubelka-Munk reflectance mixing (`colour-kubelka-munk.ts`).
- **Gamut-triangle barycentric harmony matching** — the cusp-line projection behind `harmonyMap`. See [Harmony Map](/colour-palette/harmony-map/).
- **Custom `gamutMap3D`** — 3D neighbourhood-search gamut mapping (never `clampChroma`). See [Gamut Mapping](/colour-palette/gamut-mapping/).
- **Chromatic threshold with CAM16 eccentricity** — the achromatic-threshold curve used by the harmony construction.
- **Paint-mix tonal scales with ΔE labels** — `generateTonalScalePaintMix`. See [Tonal Scale](/colour-palette/tonal-scale/).
- **Mix tab basic structure** — 2-colour, multi-colour, neutrals, and tonal-scale sub-tabs (step distribution and chroma allocation are [Active Work](#active-work)).
- **Colour Thesaurus v2** — the DB-backed four-table model, idempotent ingest, and import pipeline. See [Thesaurus](/colour-palette/thesaurus/).

### Releases

- **1.48.95 (2026-04-29)** — Harmony pipeline rewritten as cusp-line projection. Replaces the five-stage H-K/warm-lift/neon-bell pipeline with a single-pass geometric construction inside a perceptual sub-triangle (vertices: curve-edge A, constant-chroma B, cusp). Decomposes source as `(t, r)` and reconstructs destination using the same parameters in its own sub-triangle. Mathematically self-inverse — complementary round-trips are exact in float OKLCH. Adds `findThresholdCurveEdge` helper, green/cyan Gaussian neon cap (centre H=165°, σ=35°, 30% pull-back), and a redesigned Research section with a destination-hue slider, three full-width cross-sections (Source / Raw bary / Harmony), and live readouts for the three preserved ratios `%cusp`, `%B`, `%A`. Removes `CUSP_L_CEILING`, the dual-cusp pattern, all H-K and warm-lift terms, the neon bell and the asymmetric Pass 5 reduction; `harmonyMap` no longer calls `gamutMap3DShaded` since the destination is in-gamut by construction.
