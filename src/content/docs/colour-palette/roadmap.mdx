This page tracks ongoing work for the Colour Palette module. Items move from **Planned** down to **Completed** as they ship. Each entry includes enough context that a future Claude Code session can pick it up without extra explanation.

There are two tracks: **hardening the current pipeline** (the shipped shadcn-theme flow — Known Issues and Planned Improvements below) and **building the new [Palette Workspace](/colour-palette/palette/)** (the multi-colour composition system that replaces the current palette flow — see [Palette Workspace Roadmap](#palette-workspace-roadmap)).

## Active Work

The colour engine's mixing pipeline is the active build. It feeds directly into the [Palette Workspace](/colour-palette/palette/).

### Mix step distribution

- **Where:** Mix tab (`colour-mix-lab.ts`) — the 2-colour gradient step-distribution stage.
- **Status:** In progress. The category-width approach works with a manual bias; it needs **auto-normalisation**, and a **cusp-peak** approach is being explored as a replacement.
- **Why it matters:** this is the algorithm the Playground's "add between" calls at N=1, so it unblocks both gradients and palette interpolation.
- **See:** [Mixing Playground](/colour-palette/mixing-playground/) for the full algorithm comparison and the open perceptual-step question.

### Chroma allocation

- **Where:** Mix 2-colour pipeline — the stage after step distribution.
- **Status:** Next up.
- **Symptom:** yellow looks washed out in intermediate steps because OKLCH lerps chroma linearly rather than using cusp-relative values.
- **Fix direction:** allocate chroma as a fraction of each intermediate hue's gamut cusp rather than interpolating the raw C value. Runs as a separate pipeline stage after step distribution lands.

### Example/demo views need functionals connected

- **Where:** the showcase/demo palette views.
- **Status:** To do alongside the Palette Profile build.
- **What:** the example views render heroes/accents but don't yet wire up the [functionals](/colour-palette/palette/#functionals-section) (background/text/surface). Connect them once the functionals model lands.

## Method Panel & Colour Semantics

The intelligent-generation track — a five-axis [Method panel](/colour-palette/method-panel/) and the cultural [colour semantics](/colour-palette/colour-semantics/) layer that plugs into it. Staged so a future session can pick up any phase cold.

### Shipped (v2.17.0)

- **Method panel skeleton (Stage 1).** Four optional axes (Harmony × Style × Impact × Purpose-&-Industry), each mapping to an OKLCH `ConstraintContribution` in versioned, evidence-tagged JSON config, composed by the merge engine (intersect hard bounds, weight-average soft targets, priority Harmony > Purpose > Style > Impact, relax-to-soft on empty intersection). Drives `regenerate()` when active; honest "tends to feel…" copy with evidence tags. `palette-method.ts` + a soft-bias layer in `palette-generators.ts`.
- **Colour semantics data model (Stage 2, part).** Global admin-curated `colour_semantics` table + admin-gated API + a 62-record idempotent seed (`source` empty, `confidence` preserved), and the Berlin–Kay anchor classifier (`palette-bk-category.ts`) using L and C — the Area 5 `bk_category` keystone.

### Shipped (v2.18.0)

- **Frontend consumer & per-colour lookup (Stage 2).** API client + TanStack Query hooks (`useColourSemantics`) and lookup helpers (`palette-semantics.ts`): generated colour → `bk_category` anchor → records, plus audience derivation and a palette's charged anchors.
- **Cultural localisation & cautions (Phase 3).** The Method panel's **Audience** selector folds a soft per-region hue lean (positive, hue-banded associations) into the Area 4 hue layer, and surfaces non-blocking `culturalCaution` notes for the palette's charged anchors (information, not a gate).

### Shipped (v2.23.0)

- **Aesthetic axis (Stage 4).** A fifth optional Method axis: 8 colour-region aesthetics across six families, each defined as OKLCH **bands** (hero / accent / optional functional) + a selection rule in versioned config (`palette-aesthetics.ts`, `AESTHETIC_CONFIG_VERSION`), sampled per-role by `aestheticColours` / `generateVaried` so re-rolls stay on-vibe but vary. Emits the standard `ConstraintContribution` (soft L/C lean + hue centres) and composes through the merge engine at priority `Harmony > Aesthetic > Purpose > Style > Impact`, with the real per-role bands carried alongside in `aesthetic`. A **functional** colour type now surfaces in the Playground (up with hero / accent, absent by default), backed by the existing functionals model. See [Method panel → Aesthetic](/colour-palette/method-panel/#aesthetic).

### Planned

- **Method panel benchmark (Stage 3).** Generate 50 palettes per (Style, Impact) preset; if > 20% come back empty/over-constrained, convert more hard bounds to soft targets. Validate the synthesised numbers — the Area 3 (L,C) boxes, warmth poles (60°/240°), weight weights (0.7/0.3), and the Valdez–Mehrabian OKLCH proxy scaling — in the Playground.
- **Full Area 5 attribute vector.** Consolidate the existing per-colour attributes (OKLCH, WCAG luminance/contrast, gamut flags, cusp proximity) into one aggregator and add the modelled ones (arousal/valence/dominance, nameability, visual weight, CVD confusion distances, explicit C/C_cusp) behind the same interface.
- **Structured culture/region import (thesaurus-style).** The Culture selector currently curates a short list in the frontend by keyword-matching the free-text `context` field. Replace that with a proper structured `culture`/`region` field on `colour_semantics`, populated through an import pipeline that carries **source links** — the way the [Thesaurus](/colour-palette/thesaurus/) ingests its data — so cultures and their granular contexts are first-class, deduped and citation-backed rather than matched in code. This is also where the empty `source` field gets filled.
- **Native-reviewer pass.** Before any granular regional cultural claim is relied on, have native reviewers sanity-check it — cultural-meaning errors are the kind that get noticed and can offend. Treat `low`-confidence/folkloric rows as provisional until verified.
- **poline-in-OKLCH for Scales/Mix (Stage 4 — separable track).** Port poline's anchor + per-axis position-function mechanic natively into OKLCH for the Scales/Mix playground (a different part of the app from the Method panel); keep gradstop as a lightweight fallback and Kubelka–Munk as the distinct pigment-mix mode.

### Planned — Aesthetic axis follow-ups

- **Remaining aesthetics (Batch 2).** Add the rest of the ~16-entry list as pure config-data in `palette-aesthetics.ts` (no engine changes): Steampunk, Dark Academia, Studio Ghibli, Brutalist, Pop Art, Psychedelic (absorbs 60s), Postmodern / Memphis, Gen-Z Preppy. Bands + rules are in the Aesthetic-axis colour-region spec.
- **Aesthetic × Saturation interaction.** Muted-vs-vibrant is intentionally a *separate* concern — today the **Style** axis is the saturation proxy and an aesthetic's bands describe hue/L/C tendencies only (never a locked saturation). When a dedicated saturation/vibrancy axis lands, decide how it scales chroma *within* an aesthetic's bands — it should modulate the region, not override it.
- **Aesthetic × Harmony — richer blend.** An explicit **Generator now takes precedence** over the Aesthetic's per-role band sampling (which drives the roll only on **Auto**): pick the Harmony generator alongside an aesthetic and the scheme drives hue while the aesthetic contributes only its soft hue lean. The richer behaviour — place harmony-template hues but snap/filter them to the aesthetic's hue windows so you get *both* the scheme and the vibe — is the next step, once validated in the Playground.
- **Accent soft-lean tuning.** The merge envelope's soft L/C target is the *hero* band mid, so it pulls accent slots slightly toward the hero tone (kept within the accent band). Validate in the host app; if neon accents read too muted, derive the accent soft target from the accent band instead of the hero band.
- **"70s Earthy" + Solarpunk — re-evaluate on demand.** Both were cut/deferred from the launch list (70s Earthy overlaps Mid-Century Modern; Solarpunk overlaps Cottagecore / MCM greens). Add a single "70s Earthy" entry and/or restore Solarpunk only if usage shows demand.
- **Aesthetic benchmark.** As with the Style × Impact benchmark: roll N palettes per aesthetic and confirm re-rolls stay in-band *and* visibly differ; widen/narrow bands where rolls cluster or drift off-vibe. Pay attention to all-hero palettes (no accent slots) — a dark-base aesthetic shows only base colours until the user adds/demotes an accent; consider whether the default seed should include an accent.

## Known Issues

### No input validation on backend POST/PATCH routes

- **Where:** `backend/src/features/colour/colour.routes.ts` — POST and PATCH handlers.
- **Severity:** Critical (blocking)
- **Symptom:** Raw `req.body` is passed directly to Prisma. Malformed payloads cause 500 errors with Prisma stack traces instead of clean 400 responses. No zod schema validates field presence, types, or size.
- **Fix:** Add a zod schema for `CreateSavedPaletteInput` and `UpdateSavedPaletteInput`, validate in the route handler before calling the service.

### `setActive` race condition — no transaction

- **Where:** `backend/src/features/colour/colour.service.ts` — `setActive` method.
- **Severity:** Critical (blocking)
- **Symptom:** `updateMany` (deactivate all) and `update` (activate one) are separate queries. Concurrent requests can leave multiple palettes active or none active.
- **Fix:** Wrap both operations in `prisma.$transaction()`.

### All API errors silently swallowed in store

- **Where:** `store/palette-store.ts` — `loadSavedPalettes`, `savePalette`, `loadPalette`, `deletePalette` all catch and discard errors.
- **Severity:** Critical (blocking)
- **Symptom:** Failed API calls leave the UI stale with no user feedback. No `error` state variable exists in the store.
- **Fix:** Add `error: string | null` to store state. Set it in catch blocks. Surface in UI via toast or alert.

### No React error boundaries on pages

- **Where:** All page templates (`pages/MyFeaturePage.tsx`, `pages/ColourSchemesPage.tsx`, etc.).
- **Severity:** Critical (blocking)
- **Symptom:** A render error (bad colour data, NaN in style prop) crashes the entire app to a white screen.
- **Fix:** Wrap each page in the starter template's `ErrorBoundary` component, or add a local fallback.

### Generate runs synchronously inside `requestAnimationFrame`

- **Where:** `store/palette-store.ts` — the `generate` action wraps `generatePalette` in a single `requestAnimationFrame` callback.
- **Severity:** Medium
- **Symptom:** With large brand inputs (8+ colours) the generation can briefly block the main thread, dropping a frame on slow devices.
- **Workaround:** Reduce the number of brand colours, or wait for the planned web-worker offload.

### No version history on saved palettes

- **Where:** Schema — `colour_palette_saved` stores only the latest snapshot.
- **Severity:** Low
- **Symptom:** Users cannot revert to a previous version of a saved palette after editing.
- **Workaround:** Save under a new name before making destructive edits.

### Colour Palette tab does not respond to `theme.changed` yet

- **Where:** `registrations/action.registration.ts` — the `theme.changed` handler is registered but the body is empty.
- **Severity:** Low
- **Symptom:** Switching themes elsewhere in the app does not update the palette preview live.
- **Workaround:** Reload the page.

### Standards-alignment gaps (2026-07-14 audit)

<Aside type="caution" title="Found by the 2026-07-14 module-standards audit">
Audited against module-starter scaffold **v1.18.0** / host starter **v1.52.1**. The module is healthy overall — self-contained schema (no `@relation`, fully indexed), correct registries, `.js` backend imports, standard headers, structured logging, action hooks. These are the gaps found; the fixes are tracked under [Planned Improvements → Standards alignment](#standards-alignment-2026-07-14-audit).
</Aside>

- **No account-deletion handler for per-user data.** _Severity: High._ The module ships **no `security.registration.ts`** template, yet stores per-user rows in `colour_palette_saved`, `palette`, and `colour_favourite` (each carries a `userId`). On account erasure the starter's deletion cascade has nothing to call, so those rows survive. Fix: generate a `security.registration.ts` with a `securityRegistry.register()` manifest plus a `registerDeletionHandler()` that hard-deletes the three tables by `userId`.
- **Module is invisible to the host module registry.** _Severity: Medium._ There is no runtime **`module.registration.ts`** — the starter v1.48+ modules-foundation feature diffs these against its `InstalledModule` table at startup and lists them on `/admin/modules`, so without it the colour module never appears as installed. Fix: hand-port the `MODULE_REGISTRATION` template (a no-op on pre-v1.48 hosts, so always safe).
- **`palette` table is not namespaced.** _Severity: Low._ Every other model is module-prefixed (`colour_palette_saved`, `colour_favourite`, `colour_knowledge`, …) but the Palette Workspace table is named just **`palette`** (`src/templates/schema.ts`), risking a Prisma merge collision with the host or another module in the shared database. Fix (deferred, phased): a safe rename to `colour_palette` must go expand → backfill → contract across releases — see [Planned Improvements → Standards alignment](#standards-alignment-2026-07-14-audit). A one-shot rename is unsafe because the schema step runs before the migration runner.
- **`colour_semantics` → `colour_knowledge` merge shipped without a data migration.** _Severity: Low._ The v2.32.0 merge dropped the standalone `colour_semantics` table; a host that populated it before v2.32.0 loses those rows on `update`. Now that the migration runner exists (scaffold v1.18.0), an opt-in backfill migration could move surviving rows into `colour_knowledge (kind = 'cultural')` before the drop.

## Active Research

### Perceptual step spacing for gradients

- **Where:** Mixing Playground (`colour-mix-lab.ts`) — Perceptual Steps section and OKLab + chroma boost algorithm.
- **Status:** Open research problem. Multiple approaches tried, none fully solve it.
- **Problem:** Even hue spacing produces gradients where some regions (blue) look uniform while others (yellow) look jumpy. Delta-E says the steps are already even, but human perception disagrees. The core issue is that the number of distinguishable colours varies by hue (gamut volume), and no current metric captures this for step distribution.
- **Current best approach:** OKLab L (even perceptual lightness) + OKLCH shorter-hue H + gamut-relative C%, with a manual hue-distribution bias slider. Values around 0.5-0.8 produce acceptable results for blue→yellow, but the bias needs to be computed automatically from gamut properties.
- **Research question:** How to distribute N colour samples along a hue arc so each adjacent pair appears equally distinct, accounting for varying gamut volume at each intermediate hue.
- **Key finding:** The bias value correlates with `gamutArea(sourceH) / gamutArea(destH)`. Needs validation across more colour pairs.
- **See:** [Mixing Playground docs](/colour-palette/mixing-playground/) for full details, attempted approaches, and search terms.

## Open Questions

Decisions to resolve as the Palette Workspace is built (from the design specification). Each is a real fork, not an ambiguity in shipped behaviour.

1. **Context preview visual format** — start with proportional colour blocks (simplest), geometric patterns (more interesting), or layout mockups with stock SVG imagery (most realistic)? Leaning blocks-first.
2. **System template mapping** — when a system context template is applied to a palette it hasn't seen, how does it map generic weight hints to specific palette colours? By weight rank, or manual assignment?
3. **Neutral scale access** — every colour has its Scales tab for shade/tint ramps. Should the Palette Profile show a combined neutral scale derived from all heroes (via K-M mix), or is that fully handled by functionals now?
4. **Reference palette auto-detection** — automatically scan for nearest ΔE_OK matches in reference systems, or only show references when manually assigned?
5. **Multi-structure-node UX** — when a palette links to both a client and a project under that client, how does it appear in the Library? Nested, or shown once with both tags?

- **1.48.95 (2026-04-29)** — Harmony pipeline rewritten as cusp-line projection. Replaces the five-stage H-K/warm-lift/neon-bell pipeline with a single-pass geometric construction inside a perceptual sub-triangle (vertices: curve-edge A, constant-chroma B, cusp). Decomposes source as `(t, r)` and reconstructs destination using the same parameters in its own sub-triangle. Mathematically self-inverse — complementary round-trips are exact in float OKLCH. Adds `findThresholdCurveEdge` helper, green/cyan Gaussian neon cap (centre H=165°, σ=35°, 30% pull-back), and a redesigned Research section with a destination-hue slider, three full-width cross-sections (Source / Raw bary / Harmony), and live readouts for the three preserved ratios `%cusp`, `%B`, `%A`.
