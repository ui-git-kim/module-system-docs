The **Colour Palette** module is a deep, OKLCH-native colour module. It takes you from single-colour exploration, through composing a multi-colour palette, to publishing a complete design theme — all on the same colour engine.

It is intentionally self-contained: it depends on nothing, registers no cross-module relations in the database, and exposes its `PaletteOutput` type as the canonical contract for any other module that wants to consume a published theme.

<Aside type="note">
The page component does not register a sidebar entry by default — instead it registers as a tab on project nodes via the Structure module's `nodeTabRegistry`. The standalone **Colour Schemes** and **Colour Mix** showcase pages are registered as their own sidebar items.
</Aside>

## The Three Levels

The module is built in three layers. Each is an independent stage with its own purpose, and colours flow downward from the engine, through the workspace, to a published output.

```
┌──────────────────────────────────────────────────────────────┐
│  1 · COLOUR ENGINE            the maths that make colours      │
│     Colour Profile (single-colour deep-dive) · Thesaurus       │
│     harmonyMap · gamutMap3D · paint-mix scales · Mixing Lab    │
│     ✅ Profile + Thesaurus built    🔧 mixing/step engine WIP  │
└───────────────────────────────┬──────────────────────────────┘
                                 │ colours feed into
                                 ▼
┌──────────────────────────────────────────────────────────────┐
│  2 · PALETTE WORKSPACE        compose colours into a system    │
│     Playground → Palette Profile → Library                     │
│     heroes/accents · weights · functionals · sub-palettes      │
│     🔲 in development (starting now) — replaces the old flow   │
└───────────────────────────────┬──────────────────────────────┘
                                 │ on publish
                                 ▼
┌──────────────────────────────────────────────────────────────┐
│  3 · PUBLISH                  turn a palette into outputs      │
│     shadcn/ui theme (34 vars, light/dark, WCAG) · exports      │
│     ✅ shadcn pipeline built — moving to a post-publish step   │
└──────────────────────────────────────────────────────────────┘
```

- **[Colour](/colour-palette/colour/) — the engine.** The single-colour Colour Profile (Analyse, Gamut, Harmonies, Data Vis, Scales, Adjust, Mix), the admin **Thesaurus**, and all the maths that create colours: `harmonyMap`, `gamutMap3D`, paint-mix tonal scales, and the Mixing Playground. This is where every colour the module produces comes from.
- **[Palette](/colour-palette/palette/) — the workspace.** The new multi-colour composition system: a Playground for generating and locking colours, a Palette Profile that turns a set of colours into a working system (alts, functionals, sub-palettes, weights, versions), and a Library to manage them.
- **Publish — the outputs.** When a palette is published it becomes a complete shadcn/ui theme (the 34-variable pipeline documented in [Usage](/colour-palette/usage/) and [Integration](/colour-palette/integration/)) plus exports.

<Aside type="caution" title="Direction of travel">
The current shipping product is the **shadcn pipeline** — brand colours in, a 34-variable theme out. That pipeline still works exactly as documented, but it is being repositioned as the **publish step** that runs once a palette is published. The new **Palette Workspace** (Level 2) is being built now and replaces the current palette save/generate flow. Throughout these docs, shipped behaviour is the source of truth for "how it works today"; planned behaviour is marked **Planned** or **In development**.
</Aside>

## Quick Install

```bash
npx colour init
```

The CLI creates the `colour_palette_saved` table, generates the frontend feature directory, installs the required shadcn components, and wires up the registrations.

## What's Built Today

The shipping module is the publish pipeline plus the colour engine. You feed it brand colours (from a Coolors URL, Adobe Color URL, hex list, or CSS variables), it analyses them in OKLCH, auto-assigns semantic UI roles, and generates all 34 shadcn/ui CSS variables — with tinted neutrals, chart colours, and WCAG-validated contrast — for both light and dark modes.

### Full Feature List

- Import brand colours from Coolors URLs, Adobe Color URLs, raw hex input, or CSS variable blocks
- Auto-assign brand colours to semantic UI roles (primary, secondary, accent, destructive, etc.)
- Generate all 34 shadcn/ui CSS variables in OKLCH colour space
- Tinted neutral generation using brand hue for cohesive surfaces
- WCAG 2.1 AA contrast validation for every surface/foreground pair
- Light and dark mode output with asymmetric lightness inversion
- Refinement sliders: neutral warmth, brand saturation, surface contrast, dark mode brightness
- Save, load, and activate palettes with full history per user
- Side-by-side light/dark mode comparison view
- Chart colour generation at 72° hue intervals
- Per-colour functional role tagging with multi-role support
- Standalone Colour Schemes and Colour Mix showcase pages
- A deep [colour engine](/colour-palette/colour/) — harmony, gamut mapping, tonal scales, and paint-simulation mixing — and an admin [Thesaurus](/colour-palette/thesaurus/)

<Aside type="note" title="In development">
The [Palette Workspace](/colour-palette/palette/) (Playground, Palette Profile, Library, context preview, weighted hero/accent system) is the next major build and replaces the current palette save/generate flow. See the [Roadmap](/colour-palette/roadmap/) for sequencing.
</Aside>

## Core Philosophy

- **Playground-first.** Algorithms are compared visually side by side before any are promoted to defaults. The designer's eye is trusted over a single mathematical metric — see the [Mixing Playground](/colour-palette/mixing-playground/).
- **Engines and consumers.** The Colour Profile tabs are independent engines. The Palette Workspace consumes those engines rather than re-implementing colour maths.
- **OKLCH-native.** All colour operations happen in OKLCH. The module's own `gamutMap3D` (3D neighbourhood search) is used for gamut safety — never culori's `clampChroma`.

### Pipeline Stages

These stages run when a palette is generated and (in the new model) when it is published into a theme.

| Stage | Module | Purpose |
|-------|--------|---------|
| Import | `palette-import.ts` | Parse Coolors / Adobe / hex / CSS into a list of brand colours |
| Convert | `colour-convert.ts` | OKLCH ⇄ hex ⇄ HSL conversions via `culori` |
| Analyse | `colour-analyse.ts` | Surface hue, chroma ceiling, harmony detection, perceptual ordering |
| Assess | `palette-assessment.ts` | Suggest functional roles for each input colour |
| Generate | `palette-generator.ts` | Build the full `InterfacePalette` for both light and dark modes |
| Validate | `contrast-validator.ts` | Check every surface/foreground pair against WCAG 2.1 AA |

## Prerequisites

- **Node.js 18+** and a starter-template-based app
- **Prisma + Neon (or PostgreSQL)** with `DATABASE_URL` configured
- **No module dependencies** — this module is standalone
- The **Structure** module is optional but recommended (the palette can be embedded as a project tab)
