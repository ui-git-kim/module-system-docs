The **Method panel** makes Playground generation intelligent without over-claiming. A **Generator** chooses *how* the palette is built; four optional constraint axes each contribute an OKLCH constraint set, and a merge engine composes them into one resolved constraint that biases generation. The Generator then runs its construction algorithm *within* those constraints.

<Aside type="caution" title="Associations are conventions, not guarantees">
Colour-emotion research is mostly about what people *associate* with colours, not what colours reliably *make* people feel, and the causal "red impairs performance" line has documented replication failures. So the panel **biases** generation toward conventions — it never promises a psychological effect. UI copy reads "tends to feel…", never "makes users feel…", and every mapping carries an **evidence** tag (peer-reviewed vs convention).
</Aside>

## The Generator and four axes

The panel has two parts: a **Generator** (how the palette is constructed) and **four optional constraint axes** (what the colours should be like). The axes default to **Any** and the Generator to **Auto**; with everything on Auto/Any it rolls a varied palette, and setting a Generator, an axis, or a Culture biases generation. The Method panel is the **only** generation control. In the Playground it docks on the right inside the canvas — toggle it from the **palette icon** in the global header.

### Generator

The **Generator** picks the construction strategy applied *within* the merged constraints — it replaces the old Harmony axis (which now folds in as one generator). Selecting a generator reveals a **contextual sub-panel**:

| Generator | What it does | Sub-panel |
|-----------|--------------|-----------|
| **Auto** | Rolls a varied structure each generate — a mix of schemes, golden spread, distinct sets and scatter (the default) | — |
| **Harmony** | Builds from a colour-wheel scheme | A **scheme** select — Monochromatic, Analogous, Complementary, Split-complementary, Triadic, Tetradic |
| **Golden spread** | Golden-angle hue sequence — hues stay maximally spread for any count | — |
| **Distinct** | Maximally distinguishable set (best-candidate sampling) — built to be told apart | — |
| **Anchor blend** | Samples the palette along the **chords between anchor points on an OKLCH colour disc** (angle = hue, radius = lightness) — a faithful [poline](https://github.com/meodai/poline) rebuild, so chords sweep hue + lightness for curved gradients. **Each Space re-rolls the anchors** (within the resolved range) and re-samples, so a roll is a fresh poline palette shaped by the settings | A **read-only wheel** (visualises the rolled anchors + the chords) + the **Poline settings**: per-axis easing (9 curves, labelled X·Hue/Light, Y·Hue/Light, Z·Chroma), **closed-loop** + **invert-lightness**, and anchor count (2–6). Every **Space** rolls a fresh random anchor set |
| **Hue-shift** | Hue drifts as lightness changes — the pixel-art ramp with life in highlights and shadows | — |
| **Cosine gradient** | Smooth cosine oscillation per channel (Inigo Quilez) — organic gradients | — |
| **Scatter** | Free random hues inside the constraints — anything goes | — |

Auto and the harmony schemes get a deliberate **light/dark tonal spread** most rolls; the other strategies build their own lightness structure.

### Constraint axes

| Axis | What it sets | Driven by |
|------|--------------|-----------|
| **Aesthetic** | The vibe — a colour *region* (Cyberpunk, Goth, Cottagecore, Old Money/Ivy, …), grouped into six families | Per-role OKLCH **bands** (hero / accent / functional) + a selection rule |
| **Style** | The look — Pastel, Soft, Bright, Vibrant, Neon, Muted, Moody, Dark, Earthy, Elegant, Retro, Warm, Cool, Playful | Mostly **L** and **C**, soft hue window |
| **Impact** | The feel — Calm, Trustworthy, Energetic, Bold, Joyful, Serious, Luxurious, Friendly | Mostly **L** and **C** (valence × arousal), soft hue lean |
| **Purpose & Industry** | The job — Finance/Tech, Healthcare, Food, Eco, Luxury, Children, Data-viz, Illustration, Generic UI | Hue prior + L/C lean + a palette-**size** prior |

Style and Impact lean on **lightness and chroma** because those carry most of the emotional signal and replicate across cultures (Valdez & Mehrabian 1994: Pleasure ≈ 0.69·B + 0.22·S, Arousal ≈ −0.31·B + 0.60·S). Hue is weaker and more culturally variable, so axes only ever apply a **soft** hue lean.

## How axes compose

Each axis value maps to a `ConstraintContribution`:

```ts
{
  L?: { min?, max?, target?, weight? },   // soft range + optional hard bound
  C?: { min?, max?, target?, weight?, relativeToCusp? },
  H?: { biasCenters?: { h, weight }[] },  // soft only
  harmonyTemplate?: SchemeKind,           // legacy: set by the scheme list, not read by the merge
  distinct?: boolean,                     // Purpose = Data-viz
  paletteSize?: { min?, max?, target? },  // advice, never forced
  rejection?: { minDeltaE?, cvdSafe? },
  aesthetic?: AestheticRegions            // Aesthetic axis only — per-role bands + rule
}
```

The **merge engine** (`resolveMethod`) composes the active axes:

- **Hard bounds intersect** across axes (tightest min/max), in priority order.
- **Soft targets weight-average** and clamp into the resolved range, fed to the generator as a pull toward the preferred L/C.
- **Conflicts resolve by priority: Aesthetic > Purpose > Style > Impact.** Aesthetic is the strongest visual-style driver; Purpose carries the hardest functional needs; Style and Impact are aesthetic biases that yield first. The Generator isn't in this chain — it's the construction algorithm, applied *after* the constraints merge.
- **Relax-to-soft:** if a hard intersection would be empty (e.g. Neon wants high chroma, Luxury wants low), the lower-priority bound is relaxed to a soft target rather than failing — and the relaxation is surfaced in the panel.
- **The Aesthetic axis rides alongside the merge.** Its per-role bands (hero / accent / functional) aren't squeezed into the single resolved region — they're carried in `aesthetic` and applied at generation time, so a dark base and a bright accent can coexist. See [Aesthetic](#aesthetic).

The result — including the resolved `strategy` — feeds `generateVaried` at the Playground's `regenerate()` seam, which dispatches to the chosen construction algorithm (`harmonySetColours`, golden-angle, distinct, anchor-blend, hue-shift, cosine, …). Locked colours are still respected; the soft bias only steers where inside the feasible region new colours land.

## Aesthetic

The **Aesthetic** axis constrains generation to a colour *region* that reads as a chosen vibe — **Cyberpunk**, **Retro-Futurism**, **Lunarpunk**, **Goth**, **Cottagecore**, **Minimalist / Scandinavian**, **Mid-Century Modern**, **Old Money / Ivy** — grouped in the selector into **six families** (Neon / Futurist, Dark / Romantic, Natural / Cosy, Clean / Modern, Bold / Retro-graphic, Preppy).

**Regions, not palettes.** Each aesthetic is defined as OKLCH **bands**, not fixed hexes: a **hero** (base) band, an **accent** band, and an optional **functional** band — each an L range, a C range, and one or more hue windows — plus a per-aesthetic **selection rule**. The generator samples *within* the bands, so each re-roll (Space) looks visibly different but stays on-vibe. The example hexes in the source spec are illustrative samples inside a band, never the output. The list and bands are versioned config data (`palette-aesthetics.ts`, `AESTHETIC_CONFIG_VERSION`).

**Selection rules** are part of what makes a re-roll read correctly:

- *pick-one* — choose exactly one accent hue window for the whole roll (Goth's single deep jewel; Lunarpunk's single glow).
- *warm-lead + one cool* — a warm hero with a single cool accent note (Retro-Futurism's sunset + one cyan).
- *avoid* — reject hues outside the vibe (Lunarpunk avoids red / magenta entirely).

**Where it sits.** Aesthetic emits the same `ConstraintContribution` as the other axes — a gentle L/C lean toward the dominant hero band plus soft hue centres — so it composes through the merge engine. Its real per-role bands ride **alongside** the merge (in `aesthetic`) and are applied at generation time; that split is what lets a dark base and a bright accent coexist instead of collapsing into one merged box. It now **leads the constraint-axis priority chain** (`Aesthetic > Purpose > Style > Impact`), outranking Purpose / Style / Impact. Its per-role band sampling drives the roll only when the Generator is **Auto** — any explicit Generator (a harmony scheme, Golden, Anchor blend, …) builds the palette its own way, while the aesthetic still leans hue softly through the merge.

<Aside type="caution" title="Aesthetic constrains hue — on purpose">
Unlike Style / Impact — where hue is kept *soft* because mood claims are weak — the Aesthetic axis constrains **hue** more firmly, because it encodes a *visual-style convention* ("what reads as cyberpunk"), not a mood effect. Overall muted-vs-vibrant scaling is intentionally left to the **Style** axis; the bands describe tendencies, never a locked saturation.
</Aside>

<Aside type="note" title="Functional colours">
An aesthetic may also define a **functional** band (e.g. Cyberpunk's cool-grey text, Old Money's cream ground). Functional colours surface as a third colour type in the Playground, *up with* hero and accent — but they're **absent by default**: they appear only once you convert a colour to functional (the swatch role menu) or load a palette that already has them. When the active aesthetic has a functional band, a re-roll recolours those functional slots from it; otherwise they're left untouched.
</Aside>

This first batch ships **8 aesthetics** spanning all six families; the rest of the ~16-entry list (Steampunk, Dark Academia, Studio Ghibli, Brutalist, Pop Art, Psychedelic, Postmodern / Memphis, Gen-Z Preppy) is staged as pure config-data follow-ups. See the [Roadmap](/colour-palette/roadmap/).

## Honesty & transparency

The panel shows, for each active axis, its "tends to feel…" description and evidence tag; the resolved L/C ranges, hue lean, generator and suggested size; and any relaxations. The palette-size prior is shown as **advice** — it never resizes the palette for you.

<Aside type="note" title="The numbers are config, not code">
Every semantic → OKLCH mapping lives in versioned JSON config (`METHOD_CONFIG_VERSION`), so ranges can be tuned, A/B-tested, or localised without code changes. The Area 3 (L,C) boxes, the warmth/weight formulas, the Valdez–Mehrabian OKLCH proxy scaling and the priority ordering are **engineering syntheses** — starting points to validate in the Playground, not established standards.
</Aside>

## Culture

Beyond the Generator and constraint axes, an optional **Culture** selector localises generation to a culture/region: it folds a *soft* hue lean toward that culture's positive associations into the hue layer, and surfaces non-blocking cultural **cautions** for the palette's charged colours. It draws on the [Colour Semantics](/colour-palette/colour-semantics/) reference data — information, never a gate. The list is a short curated set of cultures (China & East Asia, India & South Asia, Western, Islamic world, …) rather than every granular context in the data.
