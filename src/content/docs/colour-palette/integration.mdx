The Colour Palette module is designed to be **consumed**, not extended. Other modules should treat its `PaletteOutput` type as the canonical theme contract and read state from its store rather than going to the database.

## Public Exports

Everything available to other modules is re-exported from `@/features/colour-palette`:

```ts
// Components
export { ColourPalettePage, ColourPaletteTab } from '@/features/colour-palette';

// Store hook
export { usePaletteStore } from '@/features/colour-palette';

// Service (typed API client)
export { myFeatureService } from '@/features/colour-palette';

// Types — the contract
export type {
  // Core
  OklchColour,
  PaletteColour,
  ColourPair,
  // Brand input
  BrandColour,
  ColourRole,
  PaletteInput,
  ImportSource,
  ImportResult,
  // Output
  PaletteOutput,
  InterfacePalette,
  RoleEntry,
  FunctionalShade,
  FunctionalShades,
  TonalScale,
  ScaleStep,
  // Analysis
  PaletteCharacter,
  HarmonyType,
  AssessmentResult,
  SuggestedRoles,
  // Settings
  RefinementSettings,
  // Persistence
  SavedPalette,
  CreateSavedPaletteInput,
  UpdateSavedPaletteInput,
  // Validation
  ContrastCheckResult,
} from '@/features/colour-palette';
```

## The Canonical Type

The `PaletteOutput` type is the contract. Any module that wants to theme itself from a generated palette should accept this type rather than re-deriving colours from raw brand input.

```ts
export interface PaletteOutput {
  light: InterfacePalette;
  dark: InterfacePalette;
}
```

### `InterfacePalette` Shape

| Field | Type | Purpose |
|-------|------|---------|
| `mode` | `'light' \| 'dark'` | Which mode this variant describes |
| `character` | `PaletteCharacter` | Surface hue, chroma ceiling, harmony, etc. |
| `functional.structure` | `RoleEntry[]` | Navigation chrome, sidebars, dividers |
| `functional.action` | `RoleEntry[]` | Buttons, links, interactive affordances |
| `functional.emphasis` | `RoleEntry[]` | Hero blocks, marketing areas |
| `functional.highlight` | `RoleEntry[]` | Badges, dots, attention pops |
| `semantic.destructive` | `RoleEntry` | Destructive actions |
| `semantic.success` | `RoleEntry` | Success messaging |
| `semantic.warning` | `RoleEntry` | Warning messaging |
| `semantic.info` | `RoleEntry` | Informational messaging |
| `surface.tintHue` | `number` | Brand hue used to tint neutrals |
| `surface.tintChroma` | `number` | Tint strength |
| `surface.scale` | `TonalScale` | 11-stop tinted neutral scale (50–950) |
| `surface.backgrounds` | `RoleEntry[]` | User-tagged light surface candidates |
| `surface.foregrounds` | `RoleEntry[]` | User-tagged dark surface candidates |
| `text.primary` | `PaletteColour` | Primary text colour |
| `text.secondary` | `PaletteColour` | Secondary text colour |
| `text.tertiary` | `PaletteColour` | Tertiary text colour |
| `text.disabled` | `PaletteColour` | Disabled text colour |
| `text.inverse` | `PaletteColour` | Inverse text colour |
| `border.subtle` | `PaletteColour` | Subtle border |
| `border.default` | `PaletteColour` | Default border |
| `border.strong` | `PaletteColour` | Strong border |
| `chart.colours` | `PaletteColour[]` | Chart colours at 72° hue intervals |
| `states.hover` | `{ lightnessOffset }` | Hover state offset |
| `states.active` | `{ lightnessOffset }` | Active state offset |
| `states.disabled` | `{ chromaMultiplier, lightnessTarget }` | Disabled state transform |
| `states.focus` | `{ ringColourIndex }` | Index of the role used for focus rings |

### `RoleEntry` Shape

```ts
interface RoleEntry {
  base: PaletteColour;          // The base swatch
  shades: FunctionalShades;     // wash / soft / dark variants
}

interface FunctionalShades {
  wash: FunctionalShade;        // Very light tinted background
  soft: FunctionalShade;        // Subtle background or disabled state
  dark: FunctionalShade;        // Hover/active state
}

interface FunctionalShade {
  primary: PaletteColour;       // The shown swatch
  alternate?: PaletteColour;    // Generated alternative when primary came from a match
  warning: boolean;             // True when primary diverges noticeably from the generated value
}
```

## Registry Registrations

The module registers into the following registries on install. Other modules can read or extend these.

| Registry | ID | Purpose |
|----------|----|---------|
| `preferencesRegistry` | `my-feature` | Adds a "Colour Palette" section to user preferences |
| `settingsPagesRegistry` | `my-feature` | Adds a "Colour Palette" page to the settings shell |
| `navigationRegistry` (main) | `my-feature-colour-schemes` | Sidebar entry for `/colour-schemes` |
| `navigationRegistry` (main) | `my-feature-colour-mix` | Sidebar entry for `/colour-mix` |
| `nodeTabRegistry` (Structure) | _structure registration_ | Adds a Colour Palette tab to project nodes |
| `actionRegistry` | `my-feature-theme-sync` | Subscribes to `theme.changed` to keep the preview in sync |

### Action Hooks Subscribed

| Action | When Fired | Context |
|--------|-----------|---------|
| `theme.changed` | When the active theme changes anywhere in the app | Module-defined |

The Colour Palette module **does not currently fire its own actions or filters** — it is a consumer-friendly read-only data source. If you need a hook (e.g. `colour-palette.generated`), open an issue or add it locally and bump the minor version.

## Reading the Active Palette from Another Module

The most common integration: a consumer module wants the user's currently active palette so it can theme itself.

<Tabs>
<TabItem label="Frontend (store)">
```tsx
import { usePaletteStore } from '@/features/colour-palette';
import type { PaletteOutput } from '@/features/colour-palette';

function ThemedChart() {
  const palette = usePaletteStore((s) => s.generatedPalette);
  if (!palette) return null;

  const colours = palette.light.chart.colours.map((c) => c.hex);
  return <BarChart colours={colours} />;
}
```
</TabItem>
<TabItem label="Frontend (service)">
```ts
import { myFeatureService } from '@/features/colour-palette';
import type { SavedPalette } from '@/features/colour-palette';
```

## Example: a Consumer Module

A hypothetical "Themed Dashboard" module that uses the active Colour Palette to colour its widgets.

```ts
// features/themed-dashboard/services/theme.ts
import { myFeatureService } from '@/features/colour-palette';
import type { PaletteOutput, InterfacePalette } from '@/features/colour-palette';

export async function getThemeForMode(mode: 'light' | 'dark'): Promise<InterfacePalette | null> {
  const active = await myFeatureService.getActive();
  if (!active) return null;
  const output = active.generatedPalette as PaletteOutput;
  return output[mode];
}

export function paletteToCssVars(p: InterfacePalette): Record<string, string> {
  return {
    '--background': p.surface.scale[50].hex,
    '--foreground': p.text.primary.hex,
    '--primary': p.functional.action[0]?.base.hex ?? '#000',
    '--destructive': p.semantic.destructive.base.hex,
    // …all 34 shadcn variables
  };
}
```

## Hex vs OKLCH

Every `PaletteColour` is a `{ oklch, hex }` pair. Use `hex` when feeding values into CSS or third-party libraries; use `oklch` when doing any further colour maths so you stay in a perceptually uniform space.

```ts
import type { PaletteColour } from '@/features/colour-palette';

function styleFor(c: PaletteColour) {
  return { backgroundColor: c.hex };          // ✅ for CSS
  // return { lightness: c.oklch.l };          // ✅ for further maths
}
```

<Aside type="tip">
Never reconvert hex back to OKLCH for further calculations — the `oklch` field is already there and is the source of truth. Round-tripping through hex loses precision.
</Aside>
