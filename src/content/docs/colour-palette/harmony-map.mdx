`harmonyMap(colour, degrees)` rotates an OKLCH colour's hue by *N* degrees and reconstructs its lightness and chroma so the result *feels* like the same colour at a new hue rather than a mechanical coordinate translation. Every harmony generator in the Colour Schemes module — complementary, triadic, tetradic, split-complementary, analogous, square, hexagonal — calls `harmonyMap` to place each spoke of the wheel.

The current implementation uses a single-pass **cusp-line projection** through a perceptual sub-triangle. This page covers the geometry, the helper functions, and the one tuning constant that remains.

## Why a Naive Rotation Fails

OKLCH is perceptually uniform *along any single axis* — two colours with the same L look equally bright, two colours with the same C look equally saturated. But the **gamut shape changes dramatically with hue**. A saturated red at L≈0.63, C≈0.25 sits comfortably inside sRGB, but rotate it to yellow keeping coordinates and you get muddy olive — yellow's natural cusp lives near L≈0.96, so the same coordinates land far below the gamut's bright zone.

The whole purpose of `harmonyMap` is to reconstruct L and C at the new hue so the result sits at the *same perceptual position* within the new hue's sub-triangle.

## The Perceptual Sub-Triangle

For a given hue, we define a sub-triangle inside the full gamut whose vertices are perceptually meaningful:

- **A** — `findThresholdCurveEdge(h, 0, cusp.l, true)` — where the chromatic-threshold *curve* crosses the gamut boundary on the rising (low-L) side. Below this point colours appear achromatic at that lightness.
- **B** — `findThresholdEdge(threshC(1), h, cusp.l, 1, false)` — where the constant-chroma threshold (taken at L=1) crosses the gamut boundary on the falling (high-L) side.
- **cusp** — the gamut's brightest, most chromatic point at this hue.

The sub-triangle has three **straight** edges: A→cusp, cusp→B, and the base A→B. Any point inside the triangle can be parameterised by where it sits relative to those edges, and that parameterisation is what gets preserved across hues.

## The Three-Line Construction

For any source point inside the sub-triangle, three lines pass through it — one from each vertex to the opposite edge:

| Line | From | Crosses edge | Ratio measured |
|------|------|--------------|----------------|
| Green | cusp | base A→B | % distance from cusp |
| Blue  | B    | A→cusp    | % distance from B    |
| Red   | A    | cusp→B    | % distance from A    |

In a straight-edged triangle these three ratios are exactly consistent — preserving any one of them automatically preserves the other two. That's barycentric coordinates: the point is uniquely determined by its three ratios.

`harmonyMap` measures these ratios in the source's sub-triangle and applies the same ratios in the destination's sub-triangle. Because the destination triangle has different vertices (its own A, B, and cusp shaped by its own gamut), the absolute (L, C) coordinates change, but the *relative position* — same depth from cusp, same lean toward A or B — stays constant.

## The Cusp-Line Algorithm

Implementation uses the cusp→base line directly because it's the shortest line-line intersection and gives both `t` (along the base) and `r` (depth from cusp) in one calculation:

```ts
function harmonyMap(colour: OklchColour, degrees: number): OklchColour {
  if (degrees === 0) return colour;

  const newH       = wrapHue(colour.h + degrees);
  const oldCusp    = findCuspForHue(colour.h);
  const newCusp    = findCuspForHue(newH);

  // Source sub-triangle vertices
  const srcThreshC = chromaThreshold(1, colour.h);
  const srcA       = findThresholdCurveEdge(colour.h, 0, oldCusp.l, true);
  const srcB       = { l: findThresholdEdge(srcThreshC, colour.h, oldCusp.l, 1, false), c: srcThreshC };

  // Decompose source via 2D line-line intersection:
  // line from cusp through source, extended to A→B straight base
  const cdl = colour.l - oldCusp.l, cdc = colour.c - oldCusp.c;
  const bdl = srcB.l - srcA.l,      bdc = srcB.c - srcA.c;
  const odl = srcA.l - oldCusp.l,   odc = srcA.c - oldCusp.c;
  const cdet = bdl * cdc - cdl * bdc;
  const sHit = cdet !== 0 ? (bdl * odc - odl * bdc) / cdet : 1;
  const t    = cdet !== 0 ? (cdl * odc - cdc * odl) / cdet : 0.5;
  const r    = sHit > 0.001 ? 1 / sHit : 0;

  // Destination sub-triangle vertices
  const dstThreshC = chromaThreshold(1, newH);
  const dstA       = findThresholdCurveEdge(newH, 0, newCusp.l, true);
  const dstB       = { l: findThresholdEdge(dstThreshC, newH, newCusp.l, 1, false), c: dstThreshC };

  // Reconstruct: same t on dest base, same r from dest cusp toward base
  const dstBaseL = dstA.l + t * (dstB.l - dstA.l);
  const dstBaseC = dstA.c + t * (dstB.c - dstA.c);

  const destL = clamp(0.01, 0.99, r * dstBaseL + (1 - r) * newCusp.l);
  let   destC = Math.max(0,       r * dstBaseC + (1 - r) * newCusp.c);

  // Green/cyan neon suppression
  const neonDist = Math.abs(((newH - 165 + 180) % 360) - 180);
  const neonCap  = 1 - 0.30 * Math.exp(-0.5 * (neonDist / 35) ** 2);
  destC = Math.min(destC, maxChromaAtLH(destL, newH) * neonCap);

  return { l: destL, c: destC, h: newH };
}
```

### What `t` and `r` mean geometrically

- **`t`** is the parameter along the source's base edge A→B where the line from cusp through source intersects. `t=0` is at A, `t=1` is at B. It captures *which side of the cusp* the source leans toward.
- **`r`** is the depth from cusp toward base along that same line. `r=0` is at the cusp itself, `r=1` is at the base. It captures *how saturated relative to the cusp* the source is — pastels have high `r`, vivid near-cusp colours have low `r`.

Reconstruction expresses the destination as `r × basePoint + (1 − r) × cusp`, which is a straight-line interpolation between the cusp and the base point at parameter `t`. This is mathematically equivalent to barycentric coordinates with weights `(r·(1−t), r·t, 1−r)` for `(A, B, cusp)`.

### Self-Inverse Property

The mapping is mathematically self-inverse: `harmonyMap(harmonyMap(x, 180°), 180°) = x` exactly, in float OKLCH. The proof:

1. Reconstructing places the destination point at `cusp_d + r × (basePoint_d − cusp_d)` on the cusp_d→basePoint_d line.
2. Decomposing that destination point projects back through cusp_d to the same basePoint_d, recovering the same `t`.
3. The ratio along the cusp-line is preserved as `r`.

Round-tripping a complementary pair (e.g. yellow → blue → yellow) only loses precision through hex quantisation at the display boundary — never through the algorithm itself.

## Helper Functions

### `chromaThreshold(l, h)`

CAM16-based perceptual chromatic threshold — the chroma below which a colour at lightness L and hue H reads as achromatic.

```ts
function chromaThreshold(l: number, h: number): number {
  const hRad = h * Math.PI / 180;
  const e_t = 0.25 * (Math.cos(hRad + 2) + 3.8);
  let hueFactor = 0.708 / e_t;                    // 1.0 at yellow, ~0.59 at blue

  // Red-magenta secondary H-K peak (High-Green-Nussbaum 2023)
  const magentaBump = 0.15 * Math.exp(
    -0.5 * Math.pow((((h - 350 + 540) % 360) - 180) / 30, 2),
  );
  hueFactor *= (1 - magentaBump);

  const f2 = 2.5 - 2.5 * l;
  const lightnessFactor = 1 / Math.max(0.4, f2);  // dark colours appear chromatic at lower C

  return 0.035 * hueFactor * lightnessFactor;
}
```

The CAM16 eccentricity term (`e_t`) makes yellows reach chromatic appearance at lower chroma than blues — matching how the eye actually behaves.

### `findThresholdCurveEdge(h, loL, hiL, rising)`

Binary-search for the L value where the gamut boundary crosses the chromatic-threshold *curve* (which itself varies with L). Returns `{ l, c: chromaThreshold(l, h) }`.

```ts
function findThresholdCurveEdge(h: number, loL: number, hiL: number, rising: boolean) {
  let lo = loL; let hi = hiL;
  for (let i = 0; i < 16; i++) {
    const mid = (lo + hi) / 2;
    const above = maxChromaAtLH(mid, h) > chromaThreshold(mid, h);
    if (rising) { if (above) hi = mid; else lo = mid; }
    else        { if (above) lo = mid; else hi = mid; }
  }
  const l = (lo + hi) / 2;
  return { l, c: chromaThreshold(l, h) };
}
```

This finds vertex **A** — the lower-L corner where the perceptual threshold meets the actual gamut boundary.

### `findThresholdEdge(threshC, h, loL, hiL, rising)`

Binary-search for the L value where the gamut boundary equals a *constant* chroma value. Returns just the L value.

```ts
function findThresholdEdge(threshC: number, h: number, loL: number, hiL: number, rising: boolean) {
  let lo = loL; let hi = hiL;
  for (let i = 0; i < 12; i++) {
    const mid = (lo + hi) / 2;
    if (rising) { if (maxChromaAtLH(mid, h) > threshC) hi = mid; else lo = mid; }
    else        { if (maxChromaAtLH(mid, h) > threshC) lo = mid; else hi = mid; }
  }
  return (lo + hi) / 2;
}
```

This finds vertex **B** — the higher-L corner using the threshold value taken at L=1 as a flat horizontal line. The asymmetry between A (curve-edge) and B (constant-chroma) is intentional: A needs to track the threshold curve precisely on the rising side because the gamut narrows fast there, while B at L=1 lives in a region where the curve is nearly flat anyway.

## Green / Cyan Neon Suppression

The sRGB gamut is unusually wide in the green-cyan band (H≈140–200°). At high lightness the cusp can hold C≈0.20–0.25, so cusp-line reconstructions of source colours that sit near the cusp produce visually overwhelming neon results.

A Gaussian cap centred at H=165° pulls chroma back from the gamut boundary in this band only:

```ts
const neonDist = Math.abs(((newH - 165 + 180) % 360) - 180);
const neonCap  = 1 - 0.30 * Math.exp(-0.5 * (neonDist / 35) ** 2);
destC = Math.min(destC, maxChromaAtLH(destL, newH) * neonCap);
```

| Constant | Value | Purpose |
|----------|-------|---------|
| Centre   | 165°  | Mid-point of green-cyan neon range |
| Width σ  | 35°   | Falloff width — affects roughly H130–200° |
| Strength | 0.30  | Peak reduction — caps chroma at 70% of gamut boundary at H=165° |

The cap fades smoothly to zero outside the band, so other hues (yellow, red, orange, blue, magenta) are untouched.

## Tuning Constants Reference

| Constant | Value | Where | Purpose |
|----------|-------|-------|---------|
| `chromaThreshold` base | 0.035 | `chromaThreshold` | Overall threshold scale |
| Magenta bump amplitude | 0.15  | `chromaThreshold` | Secondary H-K peak at red-magenta (H≈350°) |
| Magenta bump σ         | 30°   | `chromaThreshold` | Bump width |
| `lightnessFactor` floor | 0.4  | `chromaThreshold` | Prevents division explosion as L→1 |
| Curve-edge iterations  | 16    | `findThresholdCurveEdge` | Binary-search precision (~1.5e-5 in L) |
| Edge iterations        | 12    | `findThresholdEdge` | Binary-search precision (~2.4e-4 in L) |
| Neon cap centre        | 165°  | Green/cyan suppression | Mid-point of neon band |
| Neon cap σ             | 35°   | Green/cyan suppression | Falloff width |
| Neon cap strength      | 0.30  | Green/cyan suppression | 30% chroma pull-back at peak |

<Aside type="tip">
The neon cap (centre / σ / strength) is the most visually-impactful knob if you need to retune. Everything else governs the sub-triangle shape, which the cusp-line algorithm preserves regardless.
</Aside>

## Harmony Generators

Every harmony function in `colour-schemes.ts` calls `harmonyMap` once per spoke:

| Function                | Angles (degrees from source)            |
|-------------------------|-----------------------------------------|
| `complementary`         | 0, 180                                  |
| `nearComplementary`     | 0, 180 ± offset (default 30)            |
| `analogous(count, spread)` | count colours evenly across ±spread |
| `splitComplementary(spread)` | 0, −spread, +spread, 180          |
| `triadic`               | 0, 120, 240                             |
| `balance`               | 0, 150, 270                             |
| `tetradicSquare`        | 0, 90, 180, 270                         |
| `tetradicRectangle`     | 0, 60, 180, 240                         |
| `hexadic`               | 0, 60, 120, 180, 240, 300               |
| `evenlySpaced(n)`       | n hues evenly spaced 0–360°             |

The generators only choose *which* angles to spoke; `harmonyMap` does all the geometric heavy lifting.

## Research Section

The Harmonies tab includes an interactive **Research** section that visualises the cusp-line construction. It renders three triangles side-by-side:

1. **Source** — the source colour's sub-triangle with all three construction lines drawn through the source point.
2. **Raw bary** — what plain barycentric reconstruction (without curve-edge A) produces, for comparison.
3. **Harmony** — the destination sub-triangle with the same construction lines, all converging at the harmony result.

A hue slider (0–360°) selects the destination hue. The values panel surfaces the three preserved ratios (`%cusp`, `%B`, `%A`) so you can verify they're identical between source and destination.

## Testing Visually

The Harmonies showcase page renders every harmony against the working colour. Dial in test hues if you're iterating on the algorithm:

- **Yellow (h≈90°)** — should come out bright and saturated, not olive.
- **Lime (h≈130°)** — should read as saturated lime, not white-ish neon. Cap window touches this hue.
- **Cyan (h≈190°)** — should be richer than pure white-ish cyan. Centre of the neon cap.
- **Blue (h≈260°)** — should come out deep and rich, not washed.
- **Red↔Yellow round-trip** — `harmonyMap(harmonyMap(red, 180), 180)` should match the original red within hex precision.
