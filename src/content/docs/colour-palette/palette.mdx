A **palette** is a complete colour system: a set of hero and accent colours with weights, light/dark/muted alternates, purpose-driven functional colours, sub-palettes for specific contexts, and a version history. The **Palette Workspace** is where you compose, refine, and publish one.

<Aside type="note" title="Host-verified through v2.16.0 — iterating">
The Palette Workspace shipped its scaffold in **v2.8.0** and was built into a working builder through **v2.16.0** (host-verified): the data model + API, the auto-weight system, the **Library**, **Playground** and **Palette Profile** pages, **roles into palettes**, the **generator toolbox**, the **Colour View sheet**, an **interactive Palette Profile**, and database-backed **favourites** and generation **history**. Sections still marked **Planned** below — context preview, sub-palette generation, reference-palette auto-detection, exports, and the publish-to-theme step — are next. The four-axis **[Method panel](/colour-palette/method-panel/)** for intelligent generation shipped its first stage in **v2.17.0**, with the [colour semantics](/colour-palette/colour-semantics/) layer that feeds it (see [Roadmap](/colour-palette/roadmap/)). The legacy Brand/UI save flow it supersedes is documented in [Usage](/colour-palette/usage/) and [Integration](/colour-palette/integration/).
</Aside>

The workspace sits on top of the [colour engine](/colour-palette/colour/) and has three levels you can enter from either end:

- **Start in the Playground** when you're exploring and don't know your hero colours yet — generate, lock, regenerate, iterate.
- **Start in the Palette Profile** when the heroes are already known (e.g. a client provides brand colours) — add them directly and build the system immediately.

## What a Palette Is Made Of

Before the workspace mechanics, here's the vocabulary — the parts that make up a palette.

| Part | What it is |
|------|------------|
| **Heroes** | The dominant brand colours. Full alt display, prominent labelling, full consideration shelf. |
| **Accents** | Secondary colours used sparingly. Compact display, but still have alts and considerations. |
| **Weights** | Each colour's emphasis as a percentage. Auto-distributed between heroes and accents, manually adjustable. Weights drive proportion in context previews — they do **not** change swatch sizes. |
| **Alts** | Per-colour **light / dark / muted** alternates, generated from the colour's Scales/Adjust engines or hand-picked from its scale. |
| **Functionals** | Purpose-driven working colours — background, surface, text, accent-text. Optional (an illustration palette doesn't need them). |
| **Sub-palettes** | Named colour groups for specific contexts (chart colours, presentation accents) with their own generation. |
| **Considerations** | Colours being auditioned for a slot but not committed — the heart of the Playground workflow. |
| **Reference palettes** | Tracked matches to design systems (Tailwind, Material, Open Color, Radix, shadcn) — informational. |

## Palette Playground (Level 2)

The multi-colour composition space. Colour slots show as large vertical strips; each slot is a palette colour position with a consideration shelf below it. Similar to Coolors' generator, but richer — colours pull from the [colour engine](/colour-palette/colour/), and each slot keeps the alternatives you're testing.

### Slot Actions

| Action | Trigger | Effect |
|--------|---------|--------|
| Lock / unlock | Click the lock icon | Locked colours persist through regeneration. Unlocking moves the colour to considerations. |
| Cycle colour | Spacebar | Cycles colour options for the selected unlocked slot. |
| Regenerate all | Button | All unlocked slots get new colours. |
| Reorder | Drag slot | Changes colour order. |
| Add between | Click "+" between two slots | Inserts a colour using the stepped-gradient algorithm at N=1 — the best intermediate between neighbours. |
| Remove | Click "×" on slot | Removes the slot (colour moves to a recently-removed tray). |
| Open profile | Click the slot swatch | Opens the [Colour Profile](/colour-palette/colour/) in a drawer. |
| View considerations | Expand the shelf below a slot | Shows alternative colours being tested for this position. |
| Swap consideration in | Click a consideration colour | Swaps it into the slot; the previous colour moves to considerations. |
| Undo | Ctrl/Cmd+Z | Recovers the previous colour on an unlocked slot that was regenerated. |

<Aside type="note">
**Lock vs regenerate behaviour:** unlocking a *locked* slot moves its colour to considerations (it was deliberately chosen, worth keeping). Regenerating an *unlocked* slot just changes the colour (it was temporary) — Undo recovers it if needed. Lock state itself is UI-only React state; it is not persisted in the palette data.
</Aside>

### Generation Modes

| Mode | Algorithm | Use case |
|------|-----------|----------|
| **Random** | Golden-angle OKLCH sampling within the chosen range | Quick exploration |
| **Distinct** | Best-candidate sampling for maximum perceptual separation | Chart / categorical sets |
| **Analogous / Complementary / Triadic / Tetradic / Monochromatic** | `harmonySetColours` rotates a base hue by the scheme's offsets, with tones inside the active range | Building a cohesive, harmonically-related palette |

When two or more slots are locked, generation infers the harmony from the locked colours instead. Mood presets are L/C/H range filters layered on top of any mode:

| Preset | L range | C range | H range |
|--------|---------|---------|---------|
| Vibrant | 0.5–0.75 | 0.15+ | Any |
| Pastel | 0.75–0.92 | 0.03–0.12 | Any |
| Muted | 0.45–0.65 | 0.03–0.10 | Any |
| Dark | 0.20–0.45 | 0.05–0.15 | Any |
| Earthy | 0.35–0.65 | 0.04–0.12 | 30°–120° |

### Consideration Shelf

Each slot has an expandable shelf of colours being auditioned for that position. The Playground is about auditioning, not committing.

- Colours can be added from the Harmonies, Data Vis, and Scales tabs of any colour's profile.
- Clicking a consideration swaps it into the slot.
- Considerations persist across sessions (stored in the palette data).
- Each consideration can carry a note ("from triadic harmony", "client preferred this one").

### Add Between

The "+" between two slots consumes the [Mixing Playground](/colour-palette/mixing-playground/) step-distribution work: it asks the equitable-distribution algorithm for a **single** intermediate step (N=1) between the two neighbours. The same algorithm that builds a 10-step gradient, asked for one.

### Weight Sliders

Weight sliders sit below each slot and are **linked within groups** — adjusting one hero slider proportionally adjusts the other heroes; the same for accents.

- New colours default to the [auto-weight algorithm](#auto-weight-algorithm).
- Swatch strips stay **equal-width** — weights never change visual swatch size.
- A colour can be promoted accent → hero or demoted hero → accent, which recalculates weights.

## Palette Profile (Level 3)

Where a palette goes from "colours I like together" to "a working colour system." A live palette strip lives at the top; sections for alts, functionals, sub-palettes, and context preview sit below.

### Palette Strip

The committed heroes and accents as equal-width swatches, heroes and accents visually grouped. Interactive:

- **Swap-in-place** — click an alt or consideration and it replaces the current colour in the strip; the displaced colour moves to the shelf. The whole palette updates instantly.
- **Weight adjustment** — sliders below the strip, same as the Playground. Individual weights can be locked (`weightLocked`).
- **Open profile** — click any colour to open its profile in the drawer.

### Alts Section

Each palette colour has generated alternates:

| Alt | Generation method |
|-----|-------------------|
| **Light** | From the colour's Scales tab — pick a lighter stop |
| **Dark** | From the colour's Scales tab — pick a darker stop |
| **Muted** | Reduce C to 40–60% of the original |

Alts can be auto-generated from the [Scales](/colour-palette/tonal-scale/) / Adjust engines, manually tuned with the Adjust tab as a dialog, selected from a specific scale stop, or swapped into the main palette position (the previous colour becomes an alt).

### Functionals Section

Purpose-driven working colours — backgrounds, text, surfaces. Optional. When enabled, the section shows the functional slots.

**Default functional roles** (admin-configurable; designers can add/remove/rename):

| Default role | Purpose |
|--------------|---------|
| `background` | Primary background — the dominant surface |
| `background-secondary` | Cards, surfaces, panels |
| `text-on-dark` | Body text when background is dark |
| `text-on-light` | Body text when background is light |
| `accent-text` | Links, emphasis text, interactive elements |

Controls: add/remove slots; choose each slot's source (a hero directly, a stop from a hero's shade scale, or manual); an auto-update toggle so non-manual functionals follow source changes; and APCA contrast validation of text functionals against their intended background functional.

<Aside type="note">
Semantic colours (warning, error, success, info) can be added as functional slots later, when UI role mapping is built. Note that the shipped `InterfacePalette` already exposes related `FunctionalShades`/`FunctionalShade` types — see [Integration](/colour-palette/integration/). The Functionals here are the richer, designer-defined evolution of that idea.
</Aside>

### Sub-palettes Section

Named colour groups for specific contexts, each with its own mini-playground (lock/regenerate). Derivation modes: from palette colours (consistent L shift, harmony variants), distinct generation (golden angle), or manual. Useful for data-visualisation sets, presentation accents, marketing variants, and UI component states.

### Context Preview (Visualize)

Shows the palette applied at different weight distributions for different output contexts. You define your own **context weight templates** — there are no prescribed categories, because every project's contexts differ.

A context weight template is a named set of weight overrides that re-distributes the palette's colours (including neutrals/whites/blacks) for a specific output. The same palette might have a "Dark Website" template (heavy near-black, purple accents) and an "Activation Proposal" template (mostly white, light grey, small accent) — same colours, completely different proportions.

| Template type | Created by | Scope |
|---------------|-----------|-------|
| **System templates** | App admin | Available to all users as starting points |
| **User templates** | Designer | Personal, reusable across palettes or palette-specific |

Each active template renders a preview at those proportions. The preview format is a separate, swappable decision — it can start as **proportional colour blocks** (simple rectangles at weighted sizes), then add **geometric patterns** or **layout mockups** later. The weight distribution drives the render, not a hardcoded layout. A **Shuffle** button re-assigns which colours fill which regions while holding proportions.

### Status & Versioning

| Status | Behaviour |
|--------|-----------|
| **Draft** | Freely editable. No version tracking. The exploration/ideation mode. |
| **Published** | Immutable. Frozen as a numbered version (v1, v2, v3…). |

Publishing workflow: work in draft → click **Publish** to freeze as v1 → click **Edit** on a published palette to create a new draft (same `versionChainId`, incremented `version`) → publish again to become v2. All versions are navigable and viewable; any version can be forked into a new draft.

<Aside type="note">
Once a palette is published it becomes eligible for the **publish outputs**: the shadcn/ui theme pipeline and exports. This is the repositioning described in the [Overview](/colour-palette/) — only committed, versioned palettes feed into theme generation. UI role mapping is deferred until publishing is solid (see [Roadmap](/colour-palette/roadmap/)).
</Aside>

## Palette Library

The grid view of all palettes, similar to a Coolors library.

**Card display:** colour-strip preview (hero + accent swatches), name, status badge (Draft / Published vX), and quick actions (eye icon, three-dot menu).

**Quick preview (eye icon):** expands the card in place to show the full palette strip with weights, functionals preview, sub-palette names, and alts per hero. No page navigation.

**Library features:** search by name or tag; filter by type (user-defined), status (draft/published), or scope (structure node / global); sort by recently edited, name, or date created; three-dot menu for duplicate, delete, export, assign to structure node, and fork version.

## Exports

At launch, exports are CSS/UI-focused plus a brand-identity print output. (Today the module already emits the 34-variable shadcn theme — see [Usage](/colour-palette/usage/); the targets below extend that once palettes can be published.)

| Export format | Contents |
|---------------|----------|
| **CSS custom properties** | `:root` and `.dark` blocks with OKLCH values for all palette colours, alts, and functionals |
| **Tailwind config** | Theme extension with palette colours mapped to Tailwind colour names |
| **JSON** | Full palette data structure for programmatic consumption |
| **Palette strip image** | PNG/SVG of the palette strip for documentation/sharing |
| **Brand identity print** | Published palettes printed/exported as a brand colour specification page (swatches, names, hex/OKLCH values, usage notes) |

Deferred exports (see [Roadmap](/colour-palette/roadmap/)): CMYK values, nearest Pantone match, Adobe ASE/ACO, Figma tokens, and UI-role CSS (after role mapping ships).

## Auto-weight Algorithm

Heroes and accents share a 100% total. The accent pool grows as accents are added; the hero pool distributes the remainder equally among heroes.

| Accents | Accent pool | Per-accent | Hero pool | Per-hero (3 heroes) |
|---------|-------------|------------|-----------|---------------------|
| 0 | 0% | — | 100% | 33.3% |
| 1 | 5% | 5% | 95% | 31.7% |
| 2 | 10% | 5% | 90% | 30% |
| 3 | 15% | 5% | 85% | 28.3% |
| 4 | 20% | 5% | 80% | 26.7% |
| 5+ | 25% cap | varies | 75% | varies |

Formula:

```text
accentPool = min(numAccents × 5, 25)
perAccent  = accentPool / numAccents
perHero    = (100 - accentPool) / numHeroes
```

These are **defaults**. Dragging one slider proportionally adjusts the others within the same group (heroes adjust heroes, accents adjust accents). Moving a colour between groups triggers a recalculation. A `weightLocked` colour stays fixed while the others redistribute around it.

<Aside type="note">
Neutral/white/black weights are set **within context templates**, separate from the hero/accent weight system, because different contexts use different neutral distributions.
</Aside>

## Palette Structure (developer reference)

This section is the data contract behind the workspace. It's here for developers building or consuming the palette system; designers can skip it. (Other modules read the **published** contract via [Integration](/colour-palette/integration/) — this is the authoring-side structure.)

### Palette record

Scoping uses structure-node associations rather than a single `projectId`, so a palette can belong to a client, a project, both, or be global.

```text
palette
├── id                 String @id
├── userId             String
├── structureNodeIds   Json (String[])   — links to structure nodes (clients, projects; nested/multiple)
├── name               String
├── description?       String?
├── tags               String[]          — for search/filtering
├── type               String            — user-defined; admin provides global defaults ("brand", "ui", "exploration")
├── status             Enum: 'draft' | 'published'
├── versionChainId     String            — shared across all versions of the same palette
├── version            Int  @default(1)
├── isGlobal           Boolean @default(false)
├── isSystem           Boolean @default(false)   — admin-created template palettes
├── heroes             Json (see below)
├── accents            Json (see below)
├── functionals        Json (see below)
├── subPalettes        Json (see below)
├── referencePalettes  Json (see below)
├── metadata           Json              — generation settings, notes
├── createdAt          DateTime
├── updatedAt          DateTime
└── changelog          Json[]
```

**Scoping rules:** `["client-123"]` makes a palette visible in that client's context; `["client-123","project-456"]` scopes it to that project under that client; an empty array with `isGlobal: true` makes it available everywhere; `isSystem: true` marks an admin template visible to all users.

**Palette types** are user-defined strings used for filtering in the Library. Admin can register global defaults; designers can create their own ("motion graphics", "print collateral").

**Versioning** is immutable-publish: all versions share `versionChainId`; drafts are freely editable with no version tracking; publishing freezes the next version number; editing a published palette creates a new draft row with `version` incremented.

### Heroes & Accents

Colours live in two groups — **heroes** and **accents** — with weight auto-distributed between the groups and manually adjustable. Every colour can open its full Scales tab to browse its ramp and assign specific stops to its light/dark/muted alts.

```json
{
  "heroes": [
    {
      "id": "hero-1",
      "name": "Brand Purple",
      "oklch": { "l": 0.50, "c": 0.20, "h": 300 },
      "hex": "#7B2D8E",
      "weight": 31.7,
      "weightLocked": false,
      "order": 0,
      "source": { "type": "manual", "ref": null },
      "alts": {
        "light":  { "oklch": {}, "hex": "", "name": "Soft Purple" },
        "dark":   { "oklch": {}, "hex": "", "name": "Deep Purple" },
        "muted":  { "oklch": {}, "hex": "", "name": "Dusty Purple" }
      },
      "considerations": [
        {
          "id": "con-1",
          "oklch": {},
          "hex": "",
          "note": "From triadic harmony of Brand Teal",
          "source": { "type": "harmony", "ref": "hero-2" }
        }
      ]
    }
  ],
  "accents": [
    {
      "id": "accent-1",
      "name": "Highlight Yellow",
      "oklch": { "l": 0.85, "c": 0.18, "h": 90 },
      "hex": "#E5C040",
      "weight": 5,
      "weightLocked": false,
      "order": 0,
      "source": { "type": "manual", "ref": null },
      "alts": {
        "light":  { "oklch": {}, "hex": "" },
        "dark":   { "oklch": {}, "hex": "" },
        "muted":  { "oklch": {}, "hex": "" }
      },
      "considerations": []
    }
  ]
}
```

Notes on the colour object:

- **No `locked` field** — lock/unlock for the regeneration cycle is UI-only React state, not persisted.
- **`weightLocked`** — when true, this colour's weight stays fixed while others redistribute around it.
- **Alts** can be auto-generated from the Scales/Adjust engines or set by selecting a specific scale stop.

Source tracking is informational — it records where a colour came from for easy reference back to the profile:

| `source.type` | Meaning |
|---------------|---------|
| `manual` | Entered directly (hex, picker) |
| `harmony` | Added from a colour's Harmonies tab |
| `scale` | Added from a colour's Scales tab |
| `datavis` | Added from a colour's Data Vis tab |
| `mix` | Added from the Mix tab |
| `generated` | Auto-generated by the Playground |

### Functionals

Purpose-driven working colours. Optional (`enabled` flag). Admin provides global default roles; designers add their own.

```json
{
  "functionals": {
    "enabled": true,
    "slots": [
      {
        "id": "fn-1",
        "role": "background",
        "name": "Primary Background",
        "oklch": { "l": 0.15, "c": 0.02, "h": 260 },
        "hex": "#1A1A2E",
        "sourceRef": { "type": "neutral-scale", "heroId": "hero-1", "stop": "950" },
        "manual": false
      },
      {
        "id": "fn-3",
        "role": "text-on-dark",
        "name": "Body Text (Dark BG)",
        "oklch": { "l": 0.92, "c": 0.005, "h": 260 },
        "hex": "#E8E8F0",
        "sourceRef": null,
        "manual": true
      },
      {
        "id": "fn-5",
        "role": "accent-text",
        "name": "Accent Text / Links",
        "oklch": {},
        "hex": "",
        "sourceRef": { "type": "hero", "heroId": "hero-1" },
        "manual": false
      }
    ]
  }
}
```

Each slot tracks its source (`type: "hero" | "neutral-scale" | null`); non-manual functionals can auto-update when source colours change. Functionals make context templates practical — a "Dark Website" template can reference `fn-1` (background) and `fn-3` (text-on-dark) by role rather than knowing specific neutral scale stops.

### Sub-palettes

Named colour groups for specific contexts (charts, presentations, diagrams), each with its own playground-style generation.

```json
{
  "subPalettes": [
    {
      "id": "sub-1",
      "name": "Chart Colours",
      "purpose": "Data visualisation, mind maps, diagrams",
      "colours": [
        { "id": "sc-1", "oklch": {}, "hex": "", "sourceHero": "hero-1" },
        { "id": "sc-2", "oklch": {}, "hex": "", "sourceHero": "hero-2" }
      ],
      "generationConfig": { "mode": "harmony-derived", "params": {} }
    }
  ]
}
```

Sub-palettes can be derived from palette colours, independently generated (own lock/regenerate cycle), or manually composed. Derivation presets are deferred until the colour engine work settles (see [Roadmap](/colour-palette/roadmap/)).

### Reference palettes

If a palette colour matches (or is sourced from) a design system, the match is tracked for reference. Informational — the palette's own OKLCH values are always authoritative.

```json
{
  "referencePalettes": [
    {
      "system": "tailwind",
      "version": "4.0",
      "mappings": [
        { "paletteColourId": "hero-1", "systemColour": "violet-700" },
        { "paletteColourId": "accent-1", "systemColour": "amber-400" }
      ]
    }
  ]
}
```

Supported reference systems (expandable): Tailwind, Open Color, Material Design, Radix, shadcn base colours. Mappings can be auto-detected (nearest ΔE_OK match) or manually assigned.

### Context weight templates

Stored per-palette as overrides that reference palette colours by ID and neutral scale stops by name. `backgroundStop` indicates the dominant background, informing the preview render.

```json
{
  "contextTemplates": [
    {
      "id": "ctx-1",
      "name": "Dark Website",
      "weights": {
        "hero-1": 25, "hero-2": 15, "accent-1": 5,
        "neutral-950": 40, "neutral-100": 10, "neutral-500": 5
      },
      "backgroundStop": "neutral-950",
      "notes": "For the main site — heavy dark bg, purple accents"
    }
  ]
}
```

**System templates** live in a separate admin-managed table and use *relative* weight hints (rather than absolute colour references) so they apply across any palette:

```text
context_template
├── id              String @id
├── name            String
├── description?    String?
├── defaultWeights  Json    — generic weight distribution, applied relative to palette colours
├── backgroundHint  Enum: 'light' | 'dark' | 'neutral'
├── isSystem        Boolean @default(true)
├── createdBy       String
├── createdAt       DateTime
└── updatedAt       DateTime
```

When a system template is applied to a palette, the system maps its generic distribution to the palette's actual colours by role (highest-weight colour gets the largest allocation, neutral scale provides background). The exact mapping rule is an open question — see [Roadmap](/colour-palette/roadmap/).
