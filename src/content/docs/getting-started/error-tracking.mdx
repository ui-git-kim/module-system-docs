The template ships the **thin, gated wiring** for [Sentry](https://sentry.io) error tracking + release health, and delegates the rest to the managed provider. It is **off by default and fully optional** — the SDK is only loaded when you set a DSN, so apps that don't use Sentry never carry it.

<Aside type="note" title="Complements the in-app console">
Sentry does not replace the built-in [System page](/starter-template/system/) audit/error console — it complements it. The System page is your always-on, in-app view; Sentry adds external alerting, aggregation, and release health across deploys.
</Aside>

## How it works

`@sentry/node` (backend) and `@sentry/react` (frontend) are **optional, app-installed dependencies** — the same pattern as the [AWS KMS](/getting-started/aws-kms-setup/) adapter. The module specifier is held in a variable so it is never statically resolved, and the SDK is imported **only** when a DSN is set. If the DSN is set but the package isn't installed, the app logs a one-line hint and continues **without** Sentry — error tracking never blocks startup or breaks the app.

## Backend

<Steps>

1. **Install the SDK** in your app (it is not a template dependency):

   ```bash
   cd backend && npm install @sentry/node
   ```

2. **Set the DSN** in `backend/.env` (the key is pre-seeded, commented):

   ```bash
   SENTRY_DSN="https://<key>@<org>.ingest.sentry.io/<project>"
   # SENTRY_ENVIRONMENT=          # defaults to NODE_ENV
   # SENTRY_RELEASE=              # e.g. app version or git SHA (release health)
   # SENTRY_TRACES_SAMPLE_RATE=0  # 0 = errors only; raise to sample tracing
   ```

3. **That's it.** `backend/src/instrument.ts` initialises Sentry as the first import in `app.ts` (as early as possible), and `Sentry.setupExpressErrorHandler(app)` is registered after routes and before the app's own error handler — so Sentry captures the error, then `errorHandler` shapes the client response.

</Steps>

## Frontend

<Steps>

1. **Install the SDK:**

   ```bash
   cd frontend && npm install @sentry/react
   ```

2. **Set the DSN** in `frontend/.env`:

   ```bash
   VITE_SENTRY_DSN="https://<key>@<org>.ingest.sentry.io/<project>"
   # VITE_SENTRY_RELEASE=
   ```

3. `main.tsx` calls `initSentry()` before render, and the root `ErrorBoundary` forwards React render errors (with the component stack) via `reportError()`. Global errors + unhandled rejections are captured automatically by the browser SDK.

</Steps>

## Release health

Set `SENTRY_RELEASE` (backend) and `VITE_SENTRY_RELEASE` (frontend) — typically the app version or the deploy's git SHA — in CI/deploy so Sentry groups errors by release and reports crash-free rates and regressions across deploys.

## Performance tracing

Tracing is **off by default** (`SENTRY_TRACES_SAMPLE_RATE=0`) — error capture and release health don't need it. To enable tracing, raise the sample rate **and preload the instrument module** before the app graph, so Sentry's auto-instrumentation can patch Express/HTTP:

```bash
# production (after build)
node --import ./dist/instrument.js dist/app.js
```

Without the preload, `Sentry.init()` runs *after* Express is already imported (init uses a dynamic import to keep the SDK optional), so spans won't be captured — but **errors + release health still work out of the box**.

<Aside type="tip" title="Data hygiene">
Sentry receives error payloads. Keep the [structure-only](/getting-started/llm-rules/) discipline and avoid putting secrets or raw personal data into error messages / breadcrumbs. Use Sentry's server-side scrubbing + `beforeSend` for defence-in-depth if you handle sensitive data.
</Aside>
