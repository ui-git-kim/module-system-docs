## The Registry Pattern (Core Concept)

**The registry pattern is the foundation of this entire system.** Everything extends through registries - never by modifying core files.

### How It Works

```
┌─────────────────────────────────────────────────────────────┐
│                     Your Application                         │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────────┐  │
│  │ Billing │   │  Teams  │   │ Uploads │   │ Your Code  │  │
│  │ Module  │   │ Module  │   │ Module  │   │  Feature   │  │
│  └────┬────┘   └────┬────┘   └────┬────┘   └─────┬──────┘  │
│       │             │             │              │          │
│       └─────────────┴─────────────┴──────────────┘          │
│                          │                                   │
│                          ▼                                   │
│       ┌─────────────────────────────────────────────┐       │
│       │         REGISTRIES (extension points)       │       │
│       │   navigation · dashboard · settings · routes │       │
│       └─────────────────────────────────────────────┘       │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                    Starter Template                          │
└─────────────────────────────────────────────────────────────┘
```

### The Golden Rule

<Aside type="danger">
  **NEVER modify core files to add features.** Always use registries.
</Aside>

```typescript
// ✅ CORRECT - Create a registration file
// features/billing/navigation.registration.ts
navigationRegistry.registerMain({
  id: 'billing',
  title: 'Billing',
  url: '/billing',
  icon: CreditCard,
  order: 40,
});

// ❌ WRONG - Editing routes.tsx, app.ts, or any core file
```

| Registry | Purpose | Location |
|----------|---------|----------|
| `navigationRegistry` | Sidebar menu items | Frontend |

| `headerToolbarRegistry` | Header actions/toggles | Frontend |
| `adminPagesRegistry` | Admin panel pages | Frontend |
| `preferencesRegistry` | User preference options | Frontend |
| `routeRegistry` | API routes (versioned) | Backend |
| `serviceRegistry` | Cross-feature services | Backend |
| `personalInfoFieldsRegistry` | Profile form fields | Frontend |
| `notificationPreferencesRegistry` | Notification toggles | Frontend |
| `languageOptionsRegistry` | Language options | Frontend |
| `integrationsRegistry` | Third-party integrations | Frontend |
| `styleRegistry` | Runtime CSS + shadcn components.json contributions | Frontend |
| `filterRegistry` | Data transformation hooks | Both |
| `actionRegistry` | Side effect hooks | Both |

### Self-Registration Pattern

Features register via **side-effect imports** in their `index.ts`:

```typescript
// features/billing/index.ts

// Side-effect imports trigger registration at import time
import './navigation.registration';
import './settings.registration';
import './route.registration';

// Named exports for public API
export { BillingPage } from './pages/BillingPage';
export { useBilling } from './hooks/useBilling';
```

When the app imports the feature, registrations happen automatically. No manual wiring.

## Critical Rules

These rules are **mandatory** across all parts of the system.

### 1. Use Registries, Not Core Files

Already covered above - this is the most important rule.

### 2. Version Bumping

**When you modify infrastructure files, you MUST bump the version.**

#### Which Version Level?

| Level | When to Use | Changelog? | Examples |
|-------|-------------|------------|----------|
| **Patch** | Bug fixes, typos, minor adjustments | No | Fix typo, correct CSS, fix broken import |
| **Minor** | New features, enhancements, refactors | Yes | Add registry method, new component, restructure |
| **Major** | Breaking changes requiring app updates | Yes | Remove/rename APIs, change type signatures |

<Aside type="tip">
  **Default to patch.** Most changes are patch-level. Only use minor for genuinely new capabilities, and major for changes that would break existing apps. Changelog entries are only needed for minor and major releases.
</Aside>

#### Files Requiring Version Bump

**Bump version for ANY code change** so downstream apps know updates are available.

<Tabs>
  <TabItem label="Starter Template">
```
frontend/src/**/*.ts(x)      # All frontend code
backend/src/**/*.ts          # All backend code
scripts/*                    # Scripts
*.ps1, *.sh, *.bat (root)    # Root scripts
package.json                 # Dependencies
```

**NOT requiring bump:** `docs/*` (push submodule instead), `CLAUDE.md`, `README.md`

**Rule of thumb:** If git will merge changes into downstream apps, bump the version.

Commands:
```bash
npm run template:version:patch   # Bug fixes (most common)
npm run template:version:minor   # New features
npm run template:version:major   # Breaking changes
```
  </TabItem>
  <TabItem label="Module Starter">
```
src/templates/**/*           # Code templates
src/scaffold/**/*            # Scaffold infrastructure
src/config/*.ts              # Configuration
```

Commands:
```bash
npm run release:patch   # Bug fixes (most common)
npm run release:minor   # New features
npm run release:major   # Breaking changes
```
  </TabItem>
  <TabItem label="Individual Modules">
```
src/templates/**/*           # Module templates
src/module.ts                # Module definition
Any file affecting generated output
```

Commands:
```bash
npm run release:patch   # Bug fixes (most common)
npm run release:minor   # New features
npm run release:major   # Breaking changes
```
  </TabItem>
</Tabs>

<Aside type="danger">
  Never skip version bumping. Apps track versions to receive updates. Missing bumps break the update system.
</Aside>

### 3. Push Documentation Changes

When editing ANY file in `docs/`, you MUST push:

```bash
cd docs && git add . && git commit -m "Docs: description"
git push                    # Triggers deploy to live site
cd .. && git add docs && git commit -m "Update docs submodule"
```

### 4. Backend Import Extensions

Always use `.js` for relative imports in backend TypeScript:

```typescript
// ✅ Correct
import { prisma } from '../../lib/prisma.js';

// ❌ Wrong - fails at runtime
import { prisma } from '../../lib/prisma';
```

### 5. File Headers

Every new file needs a header:

```typescript
// ========================================================================
// FILE: [path/from/repo/root]
// PURPOSE: [one-line description]
// ========================================================================
```

### 6. No Backwards Compatibility

**Do not add backwards compatibility layers or migration code.**

This system prioritises forward momentum over legacy support:

- **No compatibility shims** — update all code to use new patterns
- **No migration helpers** — apps update through the template update system
- **No deprecated re-exports** — remove old code completely
- **Clean breaks** — use major version bumps to signal breaking changes

When refactoring:
```typescript
// ✅ CORRECT - Clean removal
// Old code deleted, new pattern implemented

// ❌ WRONG - Compatibility layer
function oldFunction() {
  console.warn('Deprecated: use newFunction');
  return newFunction();
}
```

<Aside type="tip">
  If a change is breaking, bump the major version and document the migration path in the changelog. Apps will adopt changes when they update.
</Aside>

## Working in Different Contexts

### In the Starter Template

You're modifying the **foundation** that all apps and modules build on.

**Key files:**
- `frontend/src/config/*.registry.ts` — Registry definitions
- `frontend/src/lib/theme.ts` — Theme CSS variable utilities
- `frontend/src/components/settings-blocks/` — Shared settings/form components
- `frontend/src/components/page-blocks/` — Shared page-level components
- `backend/src/config/*.registry.ts` — Backend registries
- `backend/src/middleware/*.ts` — Auth and request ID middleware
- `backend/src/lib/logger.ts` — Pino structured logging

**Shared component libraries:**
- `@/components/settings-blocks` — Row layouts, toggles, dropdowns, tables (for settings and admin pages)
- `@/components/page-blocks` — PageHeader, LoadingState, ErrorAlert, EmptyState

**Rules:**
- Changes here affect ALL apps using the template
- Always bump version after changes
- Test that existing features still work

### In the Module Starter

You're modifying the **scaffolding system** that generates modules. Like the starter template, it has an upstream update system.

**Key files:**
- `src/templates/` - Code templates for generated modules
- `src/scaffold/` - Scaffolding logic

**Rules:**
- Changes affect ALL modules (existing modules can pull updates via `template-upstream`)
- Test by generating a new module
- Bump version after template changes

### In Individual Modules

You're building a **feature that plugs into the starter template**.

**Key files:**
- `src/templates/` - Code that gets copied to apps
- `src/module.ts` - Module definition and dependencies

**Rules:**
- Use registries to extend the app (never modify core files)
- Follow the exact feature structure from starter template
- Bump version when templates change
- Backend services must use `createModuleLogger()` for structured logging
- Backend services must fire `actionRegistry.doAction()` after CRUD operations
- No `console.log` in any template — use descriptive comments as placeholders

## Registry Details

### Type Names

Use the **exact type names** from registry files:

| Registry | Correct Type | NOT |
|----------|-------------|-----|
| `navigationRegistry` | `NavigationItem` | ~~NavItem~~ |
| `dashboardRegistry` | `DashboardWidget` | ~~Widget~~ |
| `settingsPagesRegistry` | `SettingsPage` | ~~SettingPage~~ |
| `headerToolbarRegistry` | `ToolbarItem` | ~~HeaderToolbarItem~~ |
| `adminPagesRegistry` | `AdminPage` | ~~AdminPageItem~~ |
| `preferencesRegistry` | `PreferenceSection` | ~~PreferencesItem~~ |

### Order Values

All registries share a unified **0-100 scale**:

| Range | Label | Examples |
|-------|-------|---------|
| 0-25 | Core | Dashboard, Home, Admin |
| 26-50 | Feature | Projects, Billing, Analytics |
| 51-75 | Secondary | Support, Docs, Integrations |
| 76-100 | Utility | Settings, Help, External links |

- **Default order is 50** if not specified (middle of the Feature range)
- **Lower numbers** = appears first / runs first
- Admin overrides (drag-and-drop) take precedence over developer defaults

### Creating a Registration File

```typescript
// features/my-feature/navigation.registration.ts
import { navigationRegistry } from '@/config/navigation.registry';
import { MyIcon } from 'lucide-react';

navigationRegistry.registerMain({
  id: 'my-feature',           // Unique ID
  title: 'My Feature',        // Display text
  url: '/my-feature',         // Route path
  icon: MyIcon,               // Lucide icon
  order: 35,                  // Position in list
  // Optional:
  // roles: ['admin'],        // Role-based visibility
  // badge: 3,                // Notification badge
});
```

## System Knowledge

→ See [Configuration](/starter-template/configuration/) for complete environment setup.

### Auth URLs Must Match

Frontend and backend must use **identical** auth URLs:

```env
VITE_NEON_AUTH_URL=https://xxx.neonauth.region.aws.neon.tech/neondb/auth
NEON_AUTH_URL=https://xxx.neonauth.region.aws.neon.tech/neondb/auth
```

### Database URLs

| Variable | Pattern | Use For |
|----------|---------|---------|
| `DATABASE_URL` | has `-pooler` | App queries |
| `DIRECT_DATABASE_URL` | no `-pooler` | Migrations only |

### Migrations

Always use `migrate dev`, never `db push`:

```bash
cd backend && npx prisma migrate dev --name describe_change
```

### Structured Logging

The backend uses [Pino](https://github.com/pinojs/pino) for structured logging. Never use `console.*` in backend code:

```typescript
// ✅ Correct
import { createModuleLogger } from '../../lib/logger.js';
const log = createModuleLogger('billing');
log.info({ userId, audit: true }, 'Subscription created');

// ❌ Wrong
console.log('Subscription created');
```

→ See [Structured Logging](/starter-template/logging/) for full details.

### Theme Utilities

Use `applyThemeVariables()` from `@/lib/theme` to apply CSS variable overrides. The utility runs the `theme.cssVariables` filter then fires the `theme.changed` action:

```typescript
import { applyThemeVariables, resetThemeVariables } from '@/lib/theme';
applyThemeVariables({ '--color-primary': 'hsl(220 70% 50%)' });
resetThemeVariables(); // Reverts to globals.css defaults
```

→ See [Theme Utilities](/starter-template/theme/) for full details.

### Style System & styleRegistry

Modules contribute styling through the unified `styleRegistry` — both runtime CSS (tokens + raw CSS) and build-time shadcn `components.json` values:

```typescript
import { styleRegistry } from '@/config/style.registry';

styleRegistry.register({
  id: 'my-feature.theme',
  order: 50,
  tokens: { '--primary': 'oklch(0.7 0.15 240)' },
  // componentsJson: { aliases: {...}, registries: {...} },
});
```

**Exclusivity:** Only **ONE** module per app may claim `componentsJson.style`, `componentsJson.tailwind.baseColor`, or `componentsJson.tailwind.cssVariables`. A second claim throws `StyleRegistryError`.

**After changing any `componentsJson` fields:** run `npm run components:sync` in the host app so shadcn components are reinstalled with the merged configuration.

→ See [Style System](/starter-template/style-system/) for the full pipeline and [Registry Reference](/starter-template/registries/#style-registry) for the complete API.

## Data Fetching

Use **TanStack Query** (React Query) for all server state. The starter template configures a single `QueryClient` at the app root — modules import hooks from `@tanstack/react-query` directly. Never construct another `QueryClient`.

### Server state vs client state

| State type | Tool | Examples |
|------------|------|----------|
| Server state (from the API) | TanStack Query | User profile, billing status, entity records, settings from DB |
| Client state (UI only) | Zustand or `useState` | Sidebar open/closed, selected tab, form draft before save, theme toggle |

Never store server data in Zustand. Never use TanStack Query for pure UI state.

### Query key convention

Module query keys **MUST** be namespaced with the module ID as the first array element:

```typescript
// ✅ Correct — module-scoped
useQuery({ queryKey: ['billing', 'subscription'], ... });
useQuery({ queryKey: ['structure', 'entity', entityId], ... });

// ❌ Wrong — leaks into the global cache namespace
useQuery({ queryKey: ['subscription'], ... });
useQuery({ queryKey: ['entity', entityId], ... });
```

This enables scoped cache invalidation without affecting other modules:

```typescript
// Invalidate everything owned by billing — nothing else
queryClient.invalidateQueries({ queryKey: ['billing'] });
```

App-level keys (not owned by any module) use a single top-level string: `['app-settings']`, `['session']`.

### Do not wrap TanStack Query

Do not build a custom hook factory, registry, or service layer around TanStack Query. Modules import `useQuery`, `useMutation`, and `useQueryClient` from `@tanstack/react-query` directly. Adding indirection provides no benefit and makes debugging harder.

### Opinionated defaults

The starter's `QueryClient` sets `refetchOnWindowFocus: false`, `staleTime: 60s`, `retry: 1` for queries, and `retry: 0` for mutations. If a query needs different behaviour, override per-hook — do not change the global defaults.

### Reference implementation

See `frontend/src/features/admin/context/AppSettingsContext.tsx` in the starter template for the canonical pattern: a `useQuery` wrapped in a Context to preserve a stable consumer API while gaining cache, deduplication, and devtools support.

→ See [Data Fetching](/starter-template/data-fetching/) for the full guide, including mutation patterns and a worked module example.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Editing core files for features | Use registration files and registries |
| Missing `.js` in backend imports | Always: `import { x } from './file.js'` |
| Using `NavItem` instead of `NavigationItem` | Use exact type names from registry |
| Editing `components/ui/` | Never edit shadcn - regenerate instead |
| Using `db push` | Use `prisma migrate dev` |
| Forgetting version bump | Bump after any infrastructure change |
| Forgetting to push docs | Always push docs submodule |
| Adding backwards compatibility | Clean removal, bump major version instead |
| Building custom settings/admin UI | Use `@/components/settings-blocks` and `@/components/page-blocks` |
| Importing from old `SettingsComponents` | Import from `@/components/settings-blocks` instead |
| Using `console.*` in backend | Use `createModuleLogger()` from `lib/logger.js` |
| Using old order ranges (10, 100, etc.) | Use unified 0-100 scale, default is 50 |
| Unescaped backticks/`${}` in module templates | Escape as `\`` and `\${}` — see [escaping rules](/module-starter/building-modules/#escaping-backticks-and-template-expressions) |
| Missing action hooks after CRUD operations | Fire `actionRegistry.doAction()` after create/update/delete in backend services |
| Using `console.log` as placeholder in module templates | Remove all `console.log` — use descriptive comments or API call patterns instead |
| Changing a `styleRegistry` entry's `componentsJson` and forgetting to run sync | After any module change that affects `componentsJson` (style, baseColor, cssVariables, registries, aliases), run `npm run components:sync` so shadcn components are reinstalled with the new config. Runtime CSS (tokens, raw CSS) applies immediately; cold-side changes do not. |
| Two modules claiming the same exclusive `componentsJson` field | Only one entry per app may carry `componentsJson.style`, `componentsJson.tailwind.baseColor`, or `componentsJson.tailwind.cssVariables`. The registry throws `StyleRegistryError(EXCLUSIVE_FIELD_CLAIMED)` at registration. Coordinate between modules — at most one should claim these. |
| Fetching server data with `useState` + `useEffect` + `fetch` | Use `useQuery` from `@tanstack/react-query`. Raw `fetch` loses deduplication, caching, and devtools. |
| Unnamespaced query keys (e.g. `['subscription']` inside the billing module) | Prefix every module query key with the module ID: `['billing', 'subscription']`. |
| Constructing a second `QueryClient` in a feature or module | There is exactly one `QueryClient`, mounted at the app root in `main.tsx`. Import `useQuery`/`useQueryClient` — never `new QueryClient()`. |
| Storing server data in Zustand / a custom store | Server state belongs in TanStack Query. Use Zustand only for client-only UI state. |

## Do and Don't

### DO

- ✅ Use registries for all extensibility
- ✅ Follow feature-based architecture
- ✅ Add file headers to all files
- ✅ Use `.js` extensions in backend
- ✅ Bump version after infrastructure changes
- ✅ Push docs after documentation changes
- ✅ Make clean breaks with major version bumps

### DON'T

- ❌ Modify core files for features
- ❌ Edit `components/ui/` (shadcn)
- ❌ Put business logic in route handlers
- ❌ Use `db push` instead of `migrate dev`
- ❌ Skip version bumping
- ❌ Leave docs changes unpushed
- ❌ Add backwards compatibility layers
