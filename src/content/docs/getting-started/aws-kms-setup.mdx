This guide sets up **AWS KMS** so the master key for [field encryption](/starter-template/security-feature/#field-encryption) lives in AWS, never in your app. It's **optional** — only needed if a module uses field encryption (`hardened`+ tiers) and you want provider-held key custody instead of an env key.

## What this gets you

Field encryption scrambles your most sensitive columns so that a leaked database password, a Neon console session, or a stolen backup yields only ciphertext. The key that unscrambles it has to live somewhere:

- **An env key** (`SECURITY_ENCRYPTION_KEY`) — works, but the key sits on your server, so the risk is yours.
- **AWS KMS** (this guide) — the key lives in AWS, every use is logged in CloudTrail, and you can revoke it instantly. The risk moves to the provider.

The pattern is **envelope encryption**: KMS holds the root key, the app unwraps a data key **once** at startup and caches it, and AES-256-GCM runs locally. KMS isn't called per field.

## Prerequisites

- An [AWS account](https://aws.amazon.com)
- Your app's dependencies installed — the setup command (Step 3) runs from the app

You create the key + IAM in the AWS console (like you create a project in the Neon console); the `npm run security:setup-kms` command then prompts for the values, generates the wrapped data key, and writes everything into `.env`.

## Step 1: Create a KMS key

1. AWS Console → **KMS** → **Customer managed keys** → **Create key**
2. Key type **Symmetric**, usage **Encrypt and decrypt**
3. Give it an alias, e.g. `alias/myapp-fields`, and add yourself as a key administrator
4. After creation, open the key → **Key rotation** → enable **automatic rotation** (yearly)
5. Copy the key **ARN** — `arn:aws:kms:REGION:ACCOUNT:key/KEY_ID`

:::tip
Use the **same region as your Neon database** (e.g. `ap-southeast-2`) to keep the unwrap call fast.
:::

## Step 2: Create an IAM principal

Attach this policy (scoped to **your** key ARN) to an IAM user, and create an access key for it (Console → **IAM** → Users → Security credentials → Create access key). For production, attach the policy to a **role** instead of using static keys.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["kms:GenerateDataKey", "kms:Decrypt"],
      "Resource": "arn:aws:kms:REGION:ACCOUNT:key/KEY_ID"
    }
  ]
}
```

:::tip
`GenerateDataKey` is used **once** at setup to mint the wrapped data key; `Decrypt` is the only permission the running app needs. For strict production you can split them — set up with a key that has both, then give the runtime role `Decrypt` only.
:::

## Step 3: Run the setup command

From your app root, run the interactive setup. It **prompts for the values above, generates the wrapped data key for you, installs the SDK, and writes everything into `backend/.env`** — no manual CLI:

```bash
npm run security:setup-kms
```

It asks for the **region**, the **KMS key id/ARN**, and your **access key + secret** (leave the credentials blank to use an IAM role / `aws configure`). It writes `SECURITY_KEY_PROVIDER=kms`, `AWS_REGION`, `SECURITY_KMS_KEY_ID`, `SECURITY_KMS_WRAPPED_DEK` (and the AWS keys if you pasted them). The plaintext data key is generated in a throwaway process and never stored.

:::note
`create-app` offers this same step during cloning, so a fresh app can be set up in one pass.
:::

:::caution[Keep the secret server-side]
The wrapped data key is ciphertext (safe-ish), but `AWS_SECRET_ACCESS_KEY` is a live credential. Never commit `.env`, and prefer an IAM **role** in production so there are no static keys to leak.
:::

## Step 4: Wire the adapter

The setup command prints this — add it once at bootstrap (`backend/src/app.ts`), before any field encryption runs:

```ts
import { setKeyProvider } from '@/features/security';
import { createAwsKmsKeyProvider } from '@/features/security/key-providers/aws-kms';

const kms = createAwsKmsKeyProvider({
  region: process.env.AWS_REGION!,
  keyId: process.env.SECURITY_KMS_KEY_ID!,
  wrappedDek: process.env.SECURITY_KMS_WRAPPED_DEK!,
});
setKeyProvider(kms);

// Optional: fail fast at startup instead of on the first encrypted field.
await kms.getKey();
```

## Step 5: Verify

Encrypt and decrypt a value round-trips through the unwrapped key, and AWS **CloudTrail** shows exactly one `Decrypt` event for the process:

```ts
import { encryptField, decryptField } from '@/features/security';

const sealed = await encryptField('hello');
console.log(await decryptField(sealed)); // 'hello'
```

If that prints `hello`, KMS key custody is working.

## Troubleshooting

### `AccessDeniedException`
- The IAM principal lacks `kms:GenerateDataKey` (setup) or `kms:Decrypt` (runtime), or the policy's key ARN / region doesn't match.
- Confirm the credentials you entered are for the principal you attached the policy to.

### `InvalidCiphertextException`
- The `SECURITY_KMS_WRAPPED_DEK` was generated under a **different key or region** than `SECURITY_KMS_KEY_ID` / `AWS_REGION`. Regenerate it (Step 3) against the right key.

### "needs the AWS SDK. Install it..."
- Run `cd backend && npm install @aws-sdk/client-kms`.

### `wrappedDek does not look like a KMS CiphertextBlob`
- The value isn't the base64 ciphertext from Step 3 (often a placeholder or truncated). Regenerate and paste the full `--output text` value.

## Production considerations

- **Use an IAM role, not static keys.** On ECS/EKS/Lambda/EC2, attach the policy to the task/instance role and drop `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` entirely.
- **Keep rotation on.** Automatic yearly CMK rotation is transparent to envelope encryption.
- **Audit.** Watch the `Decrypt` calls in CloudTrail.
- **Crypto-shred.** Disabling or deleting the KMS key makes every field encrypted under this data key **permanently unreadable** — that's the global crypto-shred. Per-tenant crypto-shred (shred one tenant, not all) is the documented growth path.
- **Managed HSM.** Step up to AWS KMS **Managed HSM** only if a compliance regime requires a dedicated, single-tenant HSM (FIPS 140-2 Level 3). Standard KMS is sufficient for most apps.
