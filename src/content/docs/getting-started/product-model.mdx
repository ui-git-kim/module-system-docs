This page states the **product model** the whole Module System is built for. It is the single most important thing to internalise before adding features, because it decides what you should — and should **not** — build.

<Aside type="caution" title="Read this before building account/tenant features">
Apps built on this template are for **individuals and solo businesses**: **one user ≈ one account**. There are **no teams** — no organisations, no seats, no membership hierarchy, no "invite your colleagues" flows. Build the solo path. Do **not** add enterprise-with-teams scaffolding.
</Aside>

## Who this is for

The template targets **solo operators** — an individual running their own business — who want **enterprise-grade capabilities** (isolation, auditing, compliance tiers, billing, security posture) **without the team machinery** that enterprise SaaS usually drags along.

Think "serious tools for a company of one", not "a scaled-down Salesforce". Each account is a single person.

## One user, one space

Every account is a single user, and each user has exactly one **space** — their private data boundary. The space is created **automatically** the first time the user signs in; the user never sets up an "account" or a "tenant" as a separate step. Their user *is* their space.

- **space** — the generic, user-facing word for a user's isolated area. Use this term in product copy, UI, and module design.
- **workspace** — do **not** use this word. It implies multi-user collaboration we don't have.
- **tenant** / `tenantId` — the *internal* name for the isolation key only (database plumbing). It is not user-facing, and normally not module-author-facing either.

## Isolation is tenant-keyed

Although the product is single-user, **isolation is keyed on `tenantId`, never on `userId` directly**. This keeps one uniform rule across every table — no branching between "solo" and "shared" code paths — and means growing capability later is a *feature addition*, not a schema-wide rewrite.

- In solo mode the isolation key **equals** the user id: `Space.id === User.id === tenantId`. See [`auth.middleware.ts`](https://github.com/ui-git-kim) (`tenantId: userId`), [`owned()`](/starter-template/database/#user-data-isolation), and Neon RLS.
- Cross-account isolation is therefore a **user-boundary** property: *user A must never read or write user B's rows through any access path*. Those are the mandatory isolation tests (see the [Roadmap](/starter-template/roadmap/)).

## The Space record

A `Space` row is the **canonical account entity** — the thing `tenantId` points at, the seam a future delegate feature hangs off, and the home for the foundation's account **lifecycle state**.

```prisma
model Space {
  id String @id // == the owning user's id (the tenantId value) in solo mode

  // Foundation-owned account lifecycle (the deletion cascade records intent here).
  deletionState       String    @default("active") // "active" | "pending_deletion" | "deleted"
  deletionRequestedAt DateTime?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([deletionState])
}
```

<Aside type="note" title="Foundation vs module data on the space">
`Space` holds only **foundation-owned, provider-agnostic** account state. **Module** data does not go here. A billing provider's customer id (Stripe, Paddle, …), plan, or subscription belong in the **billing module's** own table keyed on `tenantId` — the foundation never learns about a specific payment provider.
</Aside>

The space is provisioned in [`getOrCreateProfile`](/starter-template/user-feature/) (created in the same transaction as a new user; self-healed for pre-existing accounts) and removed by the `space.account` handler in the [deletion cascade](/starter-template/security-feature/).

## Delegate access is a *future* feature

There is a realistic future case — a solo operator inviting a **delegate** (a VA, a social-media manager) with scoped access to their space. That is deliberately modelled as an **additive feature on the same `tenantId` key** (a membership seam), **not** as a teams system, and **not** something the base template ships. Per-seat billing, organisations, and team management modules are **out of scope** and not on the roadmap.

## For contributors & LLMs — the guardrail

When you build a feature, model, or migration:

- ✅ Build the **solo path**. One user, one space, auto-provisioned.
- ✅ Key owned data on **`tenantId`** (via `owned()` / RLS), never on `userId` directly.
- ✅ Put provider-specific or feature-specific account data in the **owning module**, keyed on `tenantId` — not on foundation tables.
- ❌ Do **not** add organisations, teams, seats, membership tables, or "invite a teammate" UI to the foundation.
- ❌ Do **not** call the space a "workspace", or surface "tenant" as a user-facing concept.

If a requirement seems to need teams, stop and confirm — it is almost certainly either a delegate feature (a thin membership seam) or out of scope.
