Each repository includes the docs as a **git submodule**, pointing to the same documentation repository. This means:

- One source of truth for all documentation
- Changes to docs are immediately available to all projects
- LLMs can access docs via web without needing to read local files
- Consistency is enforced across the entire system

---

## Why a Shared Documentation Site?

### The Problem with Scattered Docs

| Issue | Impact |
|-------|--------|
| Each repo has its own README | Information gets duplicated and outdated |
| No single source of truth | Conflicting instructions across projects |
| LLMs read local files | Different context depending on which repo |
| Hard to cross-reference | "See the module starter docs" - where? |

### The Solution

| Benefit | How It Works |
|---------|--------------|
| Single source of truth | One docs site covers everything |
| Always current | Push to docs repo â†’ instant deploy |
| LLM accessible | Fetch from URL, same content everywhere |
| Cross-referenced | Link between sections naturally |

## Editing Documentation

### Local Editing

The `docs/` folder is a git submodule in each repository:

```bash
# Navigate to docs
cd docs

# Make changes to .mdx files
# ...

# Commit and push (triggers deploy)
git add .
git commit -m "Docs: describe your change"
git push

# Update the submodule reference in parent repo
cd ..
git add docs
git commit -m "Update docs submodule"
```

<Aside type="danger">
  Always push the docs submodule. Unpushed docs changes won't appear on the live site and won't be available to LLMs.
</Aside>

### Adding New Pages

1. Create the `.mdx` file in the appropriate folder
2. Add frontmatter with title and description
3. Add to sidebar in `astro.config.mjs`
4. Commit, push docs, update submodule

## Submodule Setup

When you clone a repository, the docs submodule needs initialising:

```bash
# After cloning
git submodule update --init --recursive
```

The `create-app` script handles this automatically for new apps.

### Updating to Latest Docs

To pull the latest documentation:

```bash
cd docs
git pull origin main
cd ..
git add docs
git commit -m "Update docs submodule"
```

## Content Guidelines

### Where to Put Documentation

| Content Type | Location |
|--------------|----------|
| Setup guides, LLM rules | `getting-started/` |
| Core architecture, registries | `starter-template/` |
| Module creation guides | `module-starter/` |
| Specific module docs | `modules/` |
| API reference | `reference/` |

### Writing Style

- **Australian spelling** in prose (colour, behaviour, organisation)
- **American spelling** in code (backgroundColor, customizeSettings)
- Keep explanations concise
- Use tables for comparisons
- Include code examples

### Frontmatter

Every page needs:

```mdx
---
title: Page Title
description: Brief description for SEO and previews
---
```

## Why This Matters for Consistency

The Module System is designed so that:

1. **Starter Template** provides the foundation
2. **Module Starter** creates modules that plug into it
3. **Individual Modules** extend apps built from the template

All three must follow the **same patterns and conventions**. A single documentation site ensures:

- LLMs receive identical instructions everywhere
- Developers find consistent guidance
- Updates propagate to all projects automatically
- Cross-references work naturally

<Aside type="tip">
  When in doubt about conventions, check the LLM Rules page â€” it's the authoritative source for patterns that apply across the entire system.
</Aside>
