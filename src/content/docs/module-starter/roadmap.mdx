## In Progress

Features currently being developed or designed.

### Template and Registry Alignment

**Status:** In Progress

Ensuring all module starter templates stay aligned with the starter template's refactored registry system. This includes ongoing work to keep integration categories, icon types, barrel file patterns, and registration APIs consistent across both repositories.

## Planned

Features planned for upcoming releases.

### Module Marketplace/Registry

**Priority:** High

A central registry for discovering and installing community modules:

- Browse available modules with descriptions
- One-command installation: `npx module-registry install billing`
- Version compatibility checking with starter template
- Module ratings and reviews
- Dependency resolution between modules

### Conditional Template Generation (File-Level)

**Priority:** High

Block-level conditionals (`{{#if}}`) are already implemented (see Scaffold v1.3.10 in the [Changelog](/module-starter/changelog/)). This item covers **file-level** conditionals — include or exclude entire template files based on prompt answers:

```typescript
frontend: {
  templates: {
    'index.ts': INDEX_TEMPLATE,
    // Only include if 'includeApi' is true
    'services/api.ts': { content: API_SERVICE, when: 'includeApi' },
  },
}
```

### Module Dependencies (Enhanced)

**Priority:** Medium

Basic module dependencies are already supported via `requiredModules`:

```typescript
export const moduleDefinition: ModuleDefinition = {
  id: 'billing-reports',
  requiredModules: ['billing'],  // Checked before install
  // ...
};
```

Planned enhancements:
- Automatic installation of missing dependencies
- Version compatibility checking between modules
- Dependency graph resolution

### Template Inheritance

**Priority:** Medium

Allow modules to extend base templates:

```typescript
import { BASE_PAGE_TEMPLATE } from '@module-starter/templates';

export const PAGE_TEMPLATE = BASE_PAGE_TEMPLATE.extend({
  imports: ['import { BillingContext } from "./context"'],
  beforeContent: '<BillingProvider>',
  afterContent: '</BillingProvider>',
});
```

### Dry Run Improvements

**Priority:** Medium

Enhanced `--dry-run` mode for init command:

- Show complete file tree that would be created
- Diff view for files that would be modified
- Estimated database changes
- Dependency installation preview

### Prompt Sections/Grouping

**Priority:** Medium

For modules with many prompts, support grouping with section headers:

```typescript
prompts: [
  { section: 'Branding' },
  { id: 'featureName', type: 'input', question: 'Feature name' },
  { section: 'Growth Settings' },
  { id: 'threshold', type: 'input', question: 'Growth threshold' },
]
```

### Default/Built-in Prompts

**Priority:** Medium

Common prompts that every module could use, reducing boilerplate:

```typescript
defaultPrompts: {
  featureName: true,   // Auto-adds "Feature display name" prompt
  navTitle: true,      // Auto-adds "Navigation menu title" prompt
  pageTitle: true,     // Auto-adds "Page title" prompt
}
```

### Module Testing Utilities

**Priority:** Low

Helpers for testing module installation:

```typescript
import { testModuleInstall } from '@module-starter/testing';

test('module installs correctly', async () => {
  const result = await testModuleInstall({
    starterPath: './test-app',
    answers: { provider: 'stripe' },
  });

  expect(result.files).toContain('frontend/src/features/billing/index.ts');
  expect(result.errors).toHaveLength(0);
});
```

## Under Consideration

Ideas being evaluated for feasibility and demand.

### Visual Template Editor

Web-based UI for creating and editing templates:

- Drag-and-drop component placement
- Live preview of generated code
- Import/export template packages
- Syntax highlighting and validation

### Module Composition

Combine multiple modules into a single installation:

```typescript
// meta-module that installs several related modules
export const moduleDefinition: ModuleDefinition = {
  id: 'ecommerce-suite',
  type: 'composite',
  modules: ['billing', 'inventory', 'orders', 'shipping'],
};
```

### Rollback Support

Automatic backup and rollback for failed installations:

- Snapshot files before modification
- Automatic rollback on error
- Manual rollback command
- Rollback history

### Module Analytics

Optional telemetry for module usage:

- Installation counts
- Error reporting (opt-in)
- Popular configurations
- Compatibility reports

## Known Issues

Technical debt and minor issues to be addressed in future releases.

| Issue | Status | Notes |
|-------|--------|-------|
| **No validation for duplicate registrations** | Planned | Two modules can register the same navigation ID, causing conflicts |
| **No TypeScript validation in templates** | Under consideration | Template strings aren't type-checked until installation |
| **Windows path handling edge cases** | Investigating | Some paths with spaces may fail on Windows |

Found an issue? Please report it on [GitHub Issues](https://github.com/ui-git-kim/unique-react-fullstack-module-starter/issues).
</Aside>

## Contributing Ideas

Have a feature request or improvement idea?

1. **Check existing items** - Review this roadmap and [GitHub issues](https://github.com/ui-git-kim/unique-react-fullstack-module-starter/issues)
2. **Open an issue** - Describe the feature and use cases
3. **Discuss** - Engage with feedback to refine the proposal
4. **Contribute** - Pull requests welcome for approved features

<Aside type="tip">
When proposing features, consider:
- Does this benefit most module developers?
- Can it be implemented without breaking existing modules?
- Does it align with the registry-based architecture?
</Aside>
