This page documents notable changes to the module starter template. The module starter uses semantic versioning: **major.minor.patch**.

- **Major** (1.0.0 → 2.0.0): Breaking changes requiring migration
- **Minor** (1.0.0 → 1.1.0): New features, backwards compatible
- **Patch** (1.0.0 → 1.0.1): Bug fixes, documentation updates

<Aside type="tip">
The module starter has **two version numbers**:
- **Scaffold version** (`VERSION` file) — Infrastructure version (currently 1.18.0)
- **Module version** (`package.json`) — Your module's version
</Aside>

## Scaffold v1.18.0

**Per-module database migration runner (SQL + TS backfills)**

Modules can now ship ordered database migrations that the runner applies at CLI time — for renames, data backfills, and anything `prisma migrate dev` can't express. See [Database Migrations](/module-starter/migrations/).

- **Declare migrations** in `module.ts` via a new `migrations` array (`{ name, content }`), embedded like every other template. Filenames are `<version>__<slug>.{sql,ts}`; a `.ts` migration exports `async up(sql)` and receives a tagged-template SQL handle. The example is scaffolded (commented) into `module.ts`.
- **Applied at `init`/`update`** after `prisma migrate dev` and before the lifecycle hooks (new numbered step in both) — the runner first materialises newly-shipped files to `backend/prisma/module-migrations/<id>/` (append-only), then applies pending ones. A failure **stops the run** before the hooks (stop-and-report); re-run to resume.
- **New `migrate-db` command** — `npx <module> migrate-db [--dry-run]` runs or previews migrations on their own (resume after a failure, or preview an update).
- **Tracking** in a runner-owned `_module_migrations` table (raw SQL, not a Prisma model), on `DIRECT_DATABASE_URL`. Each file applies in its own transaction *with* its bookkeeping row (atomic); applied files are checksummed (editing one fails loudly — append-only); forward-only.
- **TS backfills** are transpiled + loaded in-process via **esbuild**; the raw connection uses **postgres**. Both are new runtime dependencies (externalised from the bundle).
- Contract synced to **v1.2.0** (the `_module_migrations` shape + filename parse/order), so the host starter's admin migration-status view and this runner never disagree.

Requires host starter **v1.52.0+** for the admin Modules page to show migration status (older hosts still get migrations applied — they just don't display status). Non-breaking; `migrations` is optional.

## Scaffold v1.16.0

<Badge text="Foundation" variant="tip" />

**The module contract moves to the starter — `ModuleDefinition extends ModuleContract`**

The system-level half of `ModuleDefinition` — identity (`id`/`name`/`description`/`version`), dependencies and the lifecycle hook surface — is now **owned by the starter template** ([Modules feature](/starter-template/modules-feature/), starter v1.48.0). The scaffold consumes a verbatim, hash-stamped copy and extends it. One source of truth, two repos; **zero breaking changes** — every existing `module.ts`, import path and lifecycle hook keeps compiling unchanged.

- **`src/scaffold/module-contract.ts`** — the synced copy (see its `@synced-from` banner). Reaches every module automatically via `scaffold:update`.
- **`ModuleDefinition extends ModuleContract`** in `types.ts`; lifecycle types are re-exported as full-shape aliases (hook contexts still see the complete definition, including codegen fields).
- **New `depends` field** — module id → semver range (e.g. `{ billing: '^1.0.0' }`). Presence-checked at init (unioned with the deprecated-but-honoured `requiredModules`); **ranges are recorded, not yet validated** — enforcement and topological ordering land with the starter's dependency-resolution effort.
- **New generated `module.registration.ts`** — registers `{ id, name, version, depends }` with the host's `moduleRegistry` at import time, feeding the starter's `InstalledModule` state sync and its `module.installed`/`module.updated`/`module.missing` audit events. Uses a guarded dynamic import (top-level await), so on starters **older than v1.48.0 it degrades to a no-op instead of crashing the boot**. Existing modules adopt it by hand-port (the template files are module-owned) — new modules get it automatically.
- **Reserved template variables** — `{{moduleVersion}}`, `{{moduleName}}` and `{{moduleDepends}}` substitute from `module.ts` in any template, and files embedding them are **force-refreshed on version bumps** by `update` (the raw-template hash alone can't see the stale substitution).
- **`contract:sync` / `contract:check`** (module starter repo only; excluded from modules as scaffold-only) — refresh/verify the synced copy against a sibling starter checkout. The scaffold release bump runs the check automatically, so a release cannot ship a stale contract. `check-parity.cjs` now guards multiple `.ps1`/`.sh` pairs, including the new release gate.
- **Fixed (pre-existing):** escaped/unescaped backtick mangling in `NOTIFICATION_PREFERENCES_REGISTRATION`, `LANGUAGE_OPTIONS_REGISTRATION` (frontend registrations) and the `SCHEMA` template comment — the repo's `type-check`/`build` had been broken at head; both are green again.

## Scaffold v1.15.0

<Badge text="Tooling" variant="note" />

**`check:templates` — catch the #1 template footgun (unescaped `${}`)**

Every template is itself a JS backtick string, so a literal backtick or `${...}` in the *generated* code must be escaped (`` \` `` and `\${`). An unescaped **backtick** breaks the build with a clear "Unterminated template literal" error — but an unescaped **`${expr}`** that references an in-scope variable interpolates **silently** at build time and ships wrong output, which the build does *not* catch.

- New **`npm run check:templates`** (`scripts/check-templates.cjs`) scans `src/templates/**` and reports any unescaped `${` with its file and line. It's pure `fs` + regex — it **works in the template repo without `node_modules`** — and is injected into cloned modules' `package.json` on `scaffold:update`.
- The escaping rule is now surfaced in the AI instructions (`CLAUDE.MODULE.md` / `CLAUDE.md` critical rules + checklists), the [LLM Rules](/getting-started/llm-rules/), and the [Building Modules — Escaping](/module-starter/building-modules/#escaping-backticks-and-template-expressions) guide — whose "a clean build means correct escaping" advice was corrected (it isn't, for the silent `${}` case).

Add it to your pre-publish checklist and CI.

## Scaffold v1.14.0

<Badge text="New Feature" variant="note" />

**File-level conditional template generation**

Block-level conditionals (`{{#if}}`) have been available since v1.3.10; this adds **file-level** conditionals — include or exclude an *entire* generated file based on the user's prompt answers. A `templates` map value can now be either a plain content string (always generated, as before) or a `{ content, when }` object:

```typescript
prompts: [
  { id: 'includeApi', type: 'confirm', question: 'Include the API service?', default: true },
],
frontend: {
  templates: {
    'index.ts': INDEX_TEMPLATE,
    // Generated only when the user answers yes to 'includeApi'
    'services/api.ts': { content: API_SERVICE, when: 'includeApi' },
  },
},
```

- **`when`** accepts a `{{#if}}`-style **string** (`'includeApi'`, `"provider === 'stripe'"`, `"mode !== 'lite'"`) **or a function** `(answers) => boolean` — same evaluation as `{{#if}}` blocks and `EnvVariable.when`.
- Also supported on **`indexFiles`** and **`registrations`** entries (e.g. register a dashboard widget only when a prompt opts in).
- On **`update`**, a file whose `when` no longer holds is dropped from the expected set — if it still exists it surfaces as an **orphan** offered for removal, so flipping the answer cleanly removes the file.
- Fully **backward-compatible**: plain string templates behave exactly as before. New types `TemplateEntry` / `ConditionalTemplate` / `TemplateCondition` are exported from `./scaffold` for typed `module.ts` authoring.

<Aside type="note">
In a non-interactive run with no collected answers, a `when` referencing an unanswered variable evaluates falsy (the file is excluded), mirroring `{{#if}}` behaviour. Drive conditional files from prompts that the user actually answers (or a function with sensible fallbacks).
</Aside>

## Scaffold v1.13.0

<Badge text="Hardening" variant="note" />

**CI-safe `scaffold:update`, opt-in self-update push, and propagating CLAUDE.md scaffold sections — an audit + hardening pass across the scaffold tooling**

A "tighten everything" pass over the updater scripts, bump/runner scripts, and the non-interactive code paths. Every change is backward-safe: modules in the field keep running their installed updater until they pull this version, and a human running in a terminal sees identical interactive behaviour. The `.ps1` and `.sh` updaters were changed in lockstep (now guarded by `npm run parity:check`).

**CI / non-interactive safety**

- **The updater never hangs on a prompt again.** `update-scaffold.{ps1,sh}` now detect a non-interactive context (`$env:CI`, no TTY, or the new `-Yes`/`--yes` flag) and, instead of blocking on the `[u] Update` menu, either proceed (with `-Yes`/CI) or cancel cleanly. Run unattended with `npm run scaffold:update -- --yes` (or set `CI=1`).
- **`npx <module> init --no-interactive` and `update --force` no longer block on prompts.** Env-variable prompts during `init` are skipped (declared defaults used) when non-interactive, and `update --force` no longer stops to ask about UI/dependency/env sync. A shared `isNonInteractive()` guard makes every scaffold prompt fall back to its default when there is no TTY.
- **A non-interactive merge conflict now auto-aborts** (`git merge --abort`) instead of leaving a half-merged `MERGING` tree that would block every future run.

**Self-update is no longer a surprise push** <Badge text="Behaviour change" variant="caution" />

- The self-update step **no longer pushes to your `origin` by default.** Previously a `--merge` (i.e. `scaffold:update`) would `git commit` **and `git push`** the refreshed updater to `origin/main` as a side effect — unsafe in CI, pre-push hooks, and protected branches. The commit now stays **local**; pass `-PushSelfUpdate` / `--push-self-update` if you want the old auto-push. Flags (including `-Yes`/`-PushSelfUpdate`) are now correctly propagated through the self-update re-run.

**Version bump is now intentional, not automatic** <Badge text="Behaviour change" variant="caution" />

- `scaffold:update` **only bumps your module's published version when the update actually changed `src/templates/**` or `src/module.ts`** (e.g. a brand-new template was auto-added). An infra-only pull no longer silently increments the version downstream apps compare for `npx <module> update`.

**CLAUDE.md scaffold sections finally propagate**

- The `<!-- SCAFFOLD:START/END -->` smart-merge now sources its scaffold sections from the upstream **`CLAUDE.MODULE.md`** (which carries the markers) rather than the scaffold repo's own marker-less `CLAUDE.md` — so on update your CLAUDE.md's scaffold sections are refreshed while your module-specific block is preserved. (Previously the merge always bailed and kept your copy, so scaffold guidance never reached modules.) A CLAUDE.md with no markers is still kept as-is.

**Parity, portability & robustness**

- **`run.cjs` now tries `pwsh` → `powershell.exe` → `bash`**, probing availability up-front — so `scaffold:update` works on Windows boxes with only Windows PowerShell 5.1, and the injected npm scripts now use the cross-platform `node scripts/run.cjs …` dispatcher (work without PowerShell 7) instead of a hard-coded `pwsh -File …`.
- **New shared helpers eliminate `.ps1`/`.sh` drift:** `scripts/add-scaffold-scripts.cjs` (npm-script injection) and `scripts/bump-version-files.cjs` (package.json + module.ts version writes). Both write `package.json` with `JSON.stringify(…, null, 2)` — no more PowerShell `ConvertTo-Json` reformat/escape churn — and the `module.ts` version replace now targets the **first** occurrence only.
- **`create-module.sh` portability + parity:** it now sanitises any input to kebab-case (so `My Feature` / `stripe_billing` work, matching `create-module.ps1` and its own examples) and uses portable `sed`/`awk` (no GNU-only `sed -i` / `\U`, which broke on macOS).
- **Failed `git commit --amend` of the module-owned restore is now surfaced** instead of silently reporting success over a partially-restored tree.
- **New `npm run parity:check`** (`scripts/check-parity.cjs`) asserts the two updater variants expose the same flags and behaviours, guarding against future drift.

**No action needed** beyond running `npm run scaffold:update` once (it self-updates first). The reference templates remain `@scaffold-reference`, module-owned, and auto-add-only — that already prevents dead-file accumulation, so they're kept rich as a starting point rather than slimmed.

## Scaffold v1.12.1

<Badge text="Fixes" variant="success" />

**Three rough edges in `scaffold:update`, found testing v1.12.0 on a real customized module**

- **Module-owned guarantee now holds on the manual-resolution path (correctness).** Previously, if a merge hit a genuine conflict (e.g. CLAUDE.md) and exited to "Manual Resolution Required", the thorough module-owned restore never ran — and a module-owned file that upstream changed but the module hadn't could silently take the upstream version. The conflict path now forces `src/templates/**` + `src/module.ts` to your version (and the new/deleted policy) **before** any manual-resolution exit, so a user who just resolves CLAUDE.md and commits still keeps their templates.
- **CLAUDE.md is module-owned when it has no `<!-- SCAFFOLD -->` markers (friction).** Modules cloned before the marker convention used to conflict on CLAUDE.md *every* update (the smart-merge bailed with no markers). Now an unmarked CLAUDE.md is kept as yours (its scaffold-section updates are advisory), in both the clean and conflict paths — no more recurring conflict.
- **`scaffold:check` (preview) no longer pushes (surprise).** The script's self-update step used to `git commit` **and `git push`** the new `update-scaffold` to `origin/main` even on the read-only `--preview` path, then re-run. The push is now gated to real updates only — `scaffold:check` commits the self-update locally at most and never pushes (no surprise outward push / CI trigger during a check).

Reviewed by a 14-agent adversarial pass; the two HIGH git bugs it had flagged in v1.12.0 (fast-forward `HEAD~1` baseline, atomic batched-checkout) were already fixed in the v1.12.0 ship. As always, run `scaffold:update` on a real module to confirm.

## Scaffold v1.12.0

<Badge text="Update mechanism" variant="note" />

**`scaffold:update` now treats `src/templates/**` + `src/module.ts` as module-owned**

The scaffold ships concrete reference template files populated with `my-feature` placeholder content. The moment a module customises them — consolidating into `index.ts`, renaming, or splitting for scale — those files diverge, and the old `scaffold:update` would re-introduce or conflict on them, leave stale `my-feature` files behind, or produce duplicate exports. Root cause: `src/templates/**` and `src/module.ts` are **module-owned**, but were being merged like shared infrastructure.

Now `update-scaffold.{ps1,sh}` honours that boundary:

- **Infrastructure is merged** as before — `src/scaffold/**`, `src/cli.ts`, `scripts/**`, root config.
- **`src/templates/**` and `src/module.ts` are kept exactly as your module has them.** Existing files are never overwritten, files you deleted are never re-introduced, and only **genuinely-new** template files (absent at the merge base) are auto-added. The merge-base check distinguishes a brand-new upstream file from one you deliberately deleted.
- **Upstream template/pattern changes are advisory.** `npm run scaffold:check` now prints an "infrastructure (will merge)" section and a separate **"module-owned changes (ADVISORY — hand-port)"** section with the exact `git diff` command to see the full patch.
- Conflict handling for module-owned files is robust to **modify/delete** (consolidated/renamed files no longer wedge the merge).
- Every `src/templates/**/*.ts` now carries a `// @scaffold-reference` header — a signal it's your starting point to replace, split, or delete.

**No action needed for existing modules** beyond running `npm run scaffold:update` once (it self-updates the script first). New patterns (like v1.11.0's `security.registration.ts`) now arrive as a new file auto-add or an advisory diff, not a merge conflict.

## Scaffold v1.10.1

<Badge text="Performance" variant="success" />

**Template performance fixes from audit**

Four targeted fixes from a full performance audit of all generated template code.

**Backend service template (`src/templates/backend/services.ts`):**
- `update()` method replaced two-round-trip pattern (`updateMany` + `findUnique`) with a single `$transaction` — ownership check and update in one atomic operation
- Removed `await` from `actionRegistry.doAction()` calls in `create()`, `update()`, and `delete()` — action hooks no longer block the HTTP response. Downstream modules that register slow async actions on `my-feature.item.created` etc. will no longer add latency to every write operation

**Schema template (`src/templates/schema.ts`):**
- Removed redundant `@@index([userId])` on `my_feature_settings` — the `@unique` constraint on `userId` already creates an index, so the explicit `@@index` was wasted storage and write overhead

**Frontend page templates (`src/templates/frontend/pages.ts`):**
- Admin page `handleSave` wrapped in `useCallback` with `[globalEnabled, debugMode]` deps
- Billing page `handleSelectPlan` extracted to a `useCallback` — replaces the inline `() => {}` no-op in `.map()` that created a new function reference per plan card per render

**Migration notes for existing modules:**
- `update()` return type changes from `item!` (non-null assertion) to `item` (guaranteed by the transaction) — no impact on consumers
- Action hooks are now fire-and-forget — if your module relied on `doAction` completing before the response, add explicit `await` back in your customised service

## Scaffold v1.10.0

<Badge text="New Feature" variant="note" />

**Shell execution rewrite, linked dependency preservation, and individual shadcn installation**

Three related fixes to the scaffold's shell execution and dependency management, addressing DEP0190 warnings, broken `npm link` across updates, and fragile shadcn batch installs.

**Unified `execCommand()` helper (`utils.ts`):**
- New `execCommand(command, cwd)` replaces all `execSync(cmd, { shell: true })` calls across `init.ts` and `update.ts`
- Windows: runs via `execFileSync('powershell.exe', ['-NoProfile', '-Command', command])` — no `shell: true` flag
- Unix/Mac: runs via `execFileSync('/bin/sh', ['-c', command])` — explicit shell executable, no flag
- Eliminates Node.js DEP0190 deprecation warning completely (the v1.9.1 `cwd` fix reduced but didn't eliminate it)
- Removed duplicate `shellOpts()` functions from both `init.ts` and `update.ts`
- Removed `execPwsh()` — its PowerShell-on-Windows logic is now part of `execCommand`

**Linked dependency preservation (`utils.ts`):**
- New `detectLinkedPackages(projectDir)` — scans `node_modules/` for symlinks (including scoped `@org/pkg` packages)
- New `restoreLinkedPackages(projectDir, packages)` — re-links packages after `npm install`
- Both `installShadcnComponents()` and `installDependencies()` now detect links before running npm, and restore them after
- Covers frontend, backend, and provider-specific dependency installation
- shadcn component installation also preserves links (shadcn runs `npm install` internally)

**Individual shadcn component installation (`init.ts`):**
- Components are now installed one at a time instead of a single `npx shadcn add card button label ... -y` batch
- If one component fails, the rest still install — partial success is reported with the specific failures listed
- Manual fix command shown only for failed components, not the entire set

**Propagation:**
- Run `npm run scaffold:update` in cloned modules to pull these fixes
- After scaffold update, rebuild with `npm run build`

## Scaffold v1.8.0

**Settings page: debounced live-save pattern**

`SETTINGS_PAGE_COMPONENT_TEMPLATE` now demonstrates a live-save pattern rather than a manual "Save" button — the expected UX for modern settings pages.

**Changes to the generated settings page:**
- Local state updates immediately on control change (optimistic UI)
- A debounced effect (default 500ms) persists the current state to the backend after the last change
- In-flight saves are cancelled via `AbortController` when the user changes another setting before the previous save completes
- A status indicator shows `Saving…` / `Saved` / `Save failed` in the top-right
- The first render is skipped so initial mount does not trigger a no-op save
- Cleanup on unmount clears the pending timer and aborts any in-flight request

**Why this matters:**
Naive live-save implementations fire a request on every keystroke/toggle and race their own responses. The debounce collapses rapid changes into a single request; the `AbortController` prevents stale responses from overwriting newer state. This is the pattern that was previously called out in the roadmap as a follow-up from the v1.7.0 performance pass.

**Known issue carried over:**
Modules still lack a shared query cache for read operations — blocked on the starter template adding a `QueryClientProvider` (TanStack Query or equivalent) at the shell layer. See [Roadmap — Known Issues](/module-starter/roadmap/#known-issues). Once that ships, the module starter will scaffold matching `hooks/useMyFeature.ts` files alongside the service.

## Scaffold v1.7.0

**Template performance pass: pagination, atomic writes, AbortController, render hygiene**

Audit of every generated template for performance. Backend list endpoints are now paginated, Prisma writes are atomic with ownership enforced in-query, the frontend service supports request cancellation, and several React render pitfalls in the generated components have been fixed.

**Backend — generated API shape changes (breaking for existing modules):**
- `getAll(userId, { limit, cursor })` now returns `{ items, nextCursor, hasMore }` instead of a raw array. Default page size 50, max 200, cursor-based pagination using the compound index.
- `getAll` uses a narrow `select` clause — list payloads exclude `description` and Json columns. Full objects only come back from `getById`.
- `update()` and `delete()` use a single `updateMany`/`deleteMany` with ownership in the `where` clause — one DB round trip instead of two (previously `findFirst` then `update`/`delete`), and ownership is checked atomically with the write (no TOCTOU race).
- `GET /api/v1/my-feature` accepts `?limit=` and `?cursor=` query params and returns the paginated envelope.

**Schema — new compound index:**
- `my_feature_item` now has `@@index([userId, createdAt])` (was `@@index([userId])`). The default "items per user, newest first" list query no longer does an in-memory sort.

**Frontend service template:**
- Every method accepts an optional `AbortSignal` for cancellation. Usage example in the template comments shows wiring to `AbortController` in a `useEffect` cleanup.
- `getAll(options)` signature updated to match the new backend shape: `{ limit?, cursor?, signal? }` → `MyFeatureListResponse`.
- Header comment warns against calling `service.getAll()` directly inside `useEffect` and points at TanStack Query / `createQueryHook` for caching and dedup.

**Frontend types template:**
- New `MyFeatureListItem` (narrow list shape), `MyFeatureListOptions`, `MyFeatureListResponse` interfaces. `MyFeatureItem` remains the full detail shape.

**Frontend render fixes:**
- `BILLING_PAGE_COMPONENT_TEMPLATE` — `plans` array hoisted to module scope (was reallocated every render).
- `WIDGET_COMPONENT_TEMPLATE` — wrapped in `React.memo`, avoids re-renders when the dashboard list changes but the widget's own props don't.
- `INTEGRATION_COMPONENT_TEMPLATE` — `handleConnect`/`handleDisconnect` wrapped in `useCallback`.
- `DETAIL_PAGE_TEMPLATE` — breadcrumb effect now depends on `item?.name` instead of the `item` object reference. Comment explains why object-identity dep arrays are a trap.

**Filter/action registration templates:**
- Added header performance notes to both frontend and backend filter/action registration templates, explaining the cost model (sync runs on hot paths, I/O belongs in async).

**Migration notes for existing modules:**
- Consumers of `service.getAll()` must read `.items` off the response: `const { items } = await service.getAll()`.
- Backend services that externally called `myFeatureService.getAll(userId)` now pass an options object and must handle the paginated response.
- Schema change requires a Prisma migration — `npx prisma migrate dev --name add_userid_createdat_index`.
- Run `npm run scaffold:update` in cloned modules to pull these template changes.

## Scaffold v1.5.0

**Update command: automatic dependency and component installation**

The `update` command now detects and installs new npm dependencies, shadcn components, UI libraries, and environment variables that were added between module versions.

**Delta detection via module state:**
- `ModuleState` interface extended with asset tracking fields: `installedFrontendDeps`, `installedBackendDeps`, `installedProviderDeps`, `installedShadcnComponents`, `installedUILibraries`, `installedEnvVars`
- `init` command now saves installed assets to `.module-state.json`
- `update` command compares current module definition against stored state to find what's new
- Only new assets are installed — existing dependencies are not re-installed

**Update command now has 8 steps** (was 6):
1. Detect project structure
2. Scan installed files
3. Analyse template changes
4. Schema update
5. **New:** Install new dependencies and UI components
6. **New:** Add new environment variables
7. Run lifecycle hooks
8. Apply template updates

**Legacy install handling:**
- Modules installed before v1.5.0 have no asset tracking fields in their state
- First update after upgrading to v1.5.0 catalogues existing assets without trying to reinstall them
- Subsequent updates use delta detection normally

**Shared functions:**
- `installShadcnComponents()`, `installDependencies()`, `collectEnvVariables()`, and `writeEnvVariables()` are now exported from `init.ts` for reuse by `update.ts`

## Scaffold v1.4.15

**Space-separated placeholder replacement for human-readable strings**

Templates extensively use `My Feature` (Title Case) and `my feature` (lowercase) for UI strings — page titles, labels, descriptions, breadcrumbs, and comments. Previously, only code-oriented placeholders (`my-feature`, `myFeature`, `MyFeature`, `my_feature`) were replaced by `processTemplate()`, leaving literal "My Feature" text in generated output.

**Changes:**
- Added `titleCase()` helper — converts `ai-chat` → `Ai Chat`
- Added `lowerSpaceCase()` helper — converts `ai-chat` → `ai chat`
- `processTemplate()` now replaces `My Feature` → Title Case and `my feature` → lowercase space-separated
- All six placeholder patterns are now documented in CLAUDE.md and the docs site

**Full placeholder table (all six patterns):**

| Pattern | Case | Example (`ai-chat`) |
|---------|------|---------------------|
| `My Feature` | Title Case | `Ai Chat` |
| `my feature` | lowercase | `ai chat` |
| `my-feature` | kebab-case | `ai-chat` |
| `myFeature` | camelCase | `aiChat` |
| `MyFeature` | PascalCase | `AiChat` |
| `my_feature` | snake_case | `ai_chat` |

**Propagation:** Run `npm run scaffold:update` in cloned modules, then rebuild with `npm run build`.

## Scaffold v1.4.0

**Unified update command with schema integration, rich lifecycle context, and state preservation**

Schema updates are now part of the main `update` command instead of a separate `update-schema` command. The update flow detects Prisma schema changes, prompts for confirmation, and applies them alongside template updates. The `update-schema` command is deprecated.

**Schema integration:**
- `update` command now detects schema template changes (both multi-file and single-file Prisma setups)
- Interactive confirmation before applying schema updates
- Uses stored `moduleAnswers` for variable substitution in schema templates
- `--skip-schema` flag to skip schema updates
- Proper generator/datasource block stripping using brace-depth counting (replaces fragile line filter)

**Rich UpdateContext for lifecycle hooks:**
- `schemaChanged` — whether the schema template changed
- `moduleAnswers` — stored prompt answers from initial installation
- `changes` — detailed object with `updatedFiles`, `addedFiles`, `removedFiles`, and `schemaUpdated`
- `versionChange` — object with `major`, `minor`, `patch` booleans

**State preservation:**
- `moduleAnswers` from prompts are now saved to `.module-state.json` during `init`
- All state-saving operations (`update`, `enable`, `disable`) preserve `moduleAnswers`
- Answers are loaded and used for variable substitution during schema updates

**Bug fixes:**
- Fixed `removeSchemaBlock` false positives — now uses start-of-line regex instead of `indexOf`
- Fixed generator/datasource stripping — new `stripGeneratorDatasourceBlocks()` with brace-depth counting
- Extracted shared `findModuleSchema()` utility to eliminate code duplication

## Scaffold v1.3.10

**Conditional blocks in template processing**

Added Handlebars-style `{{#if}}` / `{{else}}` / `{{/if}}` conditionals to `processTemplate()`. Zero new dependencies — lightweight custom implementation.

**Supported syntax:**
- `{{#if variableName}}...{{/if}}` — truthy check
- `{{#if variableName}}...{{else}}...{{/if}}` — if/else
- `{{#if provider === 'stripe'}}...{{/if}}` — equality check
- `{{#if provider !== 'stripe'}}...{{/if}}` — inequality check
- Nesting supported

Conditionals are evaluated before variable substitution, so `{{variable}}` placeholders inside conditional blocks work correctly.
