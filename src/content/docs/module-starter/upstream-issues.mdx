Issues reported from downstream repos (individual modules and apps) that need to be fixed **here** in the module starter. When an issue is fixed and propagated, move it from **Open Issues** to **Completed**.

## How Upstream Issues Are Tracked

When Claude Code (or a developer) discovers a bug or limitation while working in a **module** or **app** that actually belongs in an **upstream** repo, it logs the issue to a dedicated page in the docs submodule — **one page per upstream repo**:

| Issue belongs in… | Log it to… | Visible at… |
|-------------------|-----------|-------------|
| **Module Starter Template** | `docs/src/content/docs/module-starter/upstream-issues.mdx` | [/module-starter/upstream-issues/](/module-starter/upstream-issues/) |
| **Starter Template** | `docs/src/content/docs/starter-template/upstream-issues.mdx` | `/starter-template/upstream-issues/` |

Because the `docs/` folder is a **shared submodule** across all repos, issues reported from *any* module or app are visible in one place per upstream repo. When working on the module starter, check this page first; when working on the starter template, check its equivalent.

<Aside type="tip">
This page is the canonical log of downstream-surfaced issues. The [Roadmap](/module-starter/roadmap/) **Known Issues** table mirrors the open items here alongside broader technical debt and planned features.
</Aside>

**Entry format:**

```text
Open:      **[YYYY-MM-DD] [module/app name]** — Description. Manifests in `path/to/file`. Severity: High/Medium/Low.
Completed: **[YYYY-MM-DD, vX.X.X] [module/app name]** — Description — fixed in vX.X.X, propagated.
```

## Open Issues

- **[colour module] `addVersionHeader` writes a `//` comment to *every* generated file** — `src/scaffold/update.ts` → `addVersionHeader` unconditionally prepends a `// @generated-by:` line. Legal in TS/JS but breaks non-code output (JSON/YAML/TOML/`.env`/`.md` front-matter). Surfaced in the colour module's `v1.19.0 → v1.19.1` cycle — a `.json` dataset broke `JSON.parse` on the leading `//`. Fix: pick comment syntax by extension (or a per-template `headerStyle`), or skip non-code files; update `stripVersionHeader` + `VERSION_HEADER_PATTERN` to match. Manifests in `src/scaffold/update.ts`. Severity: Medium.
- **[colour module] `MyFeature` placeholder collides with literal PascalCase file names** — `src/scaffold/utils.ts` → `processTemplate` replaces `MyFeature` → `pascalCase(moduleId)` in both paths and contents. A module whose id shares a PascalCase stem with a literal template file name collides (module id `colour` scaffolded `MyFeaturePage.tsx` → `ColourPage.tsx` on top of an existing literal `ColourPage.tsx`). Fix: detect collisions and error at install time, or add a per-template opt-out from placeholder substitution. Manifests in `src/scaffold/utils.ts`. Severity: Low.
- **[2026-07-15, re-scoped 2026-07-17] Modules without `SCAFFOLD:START` markers are permanently cut off from CLAUDE.md propagation** — and are still teaching a settings-components import path that has not compiled since host v1.52. The **source is fixed**: the scaffold's `CLAUDE.MODULE.md` no longer inlines a component table, it links to [Building Modules → Starter Components](/module-starter/building-modules/#starter-components), which is current. Propagation demonstrably works — `structure` has the markers and carries **zero** stale mentions.

  **But `Merge-ClaudeWithMarkers` (`update-scaffold.ps1`/`.sh`) returns `null` when either side lacks `<!-- SCAFFOLD:START -->`, silently skipping the CLAUDE.md merge entirely.** Modules whose CLAUDE.md predates the marker system therefore never receive scaffold-section updates and drift forever, with no warning. Current state (2026-07-17): **billing — 16 stale mentions** (including a literal `import { SettingsToggleRow } from '@/features/user/components/settings/SettingsComponents';` example, plus a hand-written note apologising for the surrounding text being wrong) and **cog-ingest — 15**. This is actively harmful now that CLAUDE.md is the first thing an LLM reads before editing a module: it steers the agent to an import that cannot resolve.

  Fix, in order: (1) make the skip **loud** — `Merge-ClaudeWithMarkers` should warn ("CLAUDE.md has no SCAFFOLD markers — scaffold sections not updated; re-add markers to resume") instead of returning `null` silently; (2) hand-add the marker pair to billing and cog-ingest around their scaffold sections, then run `scaffold:update` to pull current content; (3) consider a `scaffold:check` warning for missing markers. Manifests in `update-scaffold.ps1` / `update-scaffold.sh` → `Merge-ClaudeWithMarkers`. Severity: **High** (was Medium — raised because it now misdirects LLM-driven module work, and because the silent skip means the drift is invisible).

## Completed

- **[2026-06-24, v1.13.0] structure module** — `scaffold:update` self-update **auto-pushed** the refreshed `update-scaffold` to the module's `origin/main` as a side effect (unsafe in CI / pre-push hooks), and the self-update re-run / flagless invocation could **hang on the interactive menu** in non-interactive contexts. Surfaced while integrating the security feature into the structure module. Fixed in v1.13.0: push is now opt-in (`--push-self-update`, commit stays local), and a `-Yes`/`--yes` + `$CI`/no-TTY non-interactive mode never blocks (with `git merge --abort` on non-interactive conflicts). Propagated via `scaffold:update`.
- **[2026-06-24, v1.13.0] structure module** — `npx <module> init --no-interactive` and `update --force` could **hang on environment-variable / UI / dependency prompts**. Fixed in v1.13.0: a shared `isNonInteractive()` guard returns declared defaults instead of blocking.
