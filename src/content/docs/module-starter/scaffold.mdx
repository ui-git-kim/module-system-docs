The scaffold is the shared infrastructure that powers every module. It handles installing templates into apps, tracking what was installed, detecting changes, and updating modules. You don't edit these files ‚Äî but understanding how they work helps you build better modules and maintain the module starter.

---

## What Is the Scaffold?

The scaffold is the `src/scaffold/` directory in the module starter. When you clone the module starter to create a new module, you get this infrastructure automatically. It provides the CLI commands (`init`, `update`, `uninstall`, etc.) that users run to manage your module in their app.

```
src/scaffold/
‚îú‚îÄ‚îÄ init.ts              # Installation pipeline
‚îú‚îÄ‚îÄ update.ts            # Hash-based change detection and updates
‚îú‚îÄ‚îÄ uninstall.ts         # Manifest-based cleanup
‚îú‚îÄ‚îÄ enable.ts            # Re-enable a disabled module
‚îú‚îÄ‚îÄ disable.ts           # Disable without removing
‚îú‚îÄ‚îÄ update-schema.ts     # Deprecated ‚Äî schema updates now handled by update.ts
‚îú‚îÄ‚îÄ lifecycle-utils.ts   # Lifecycle hook infrastructure
‚îú‚îÄ‚îÄ module-contract.ts   # SYNCED COPY of the starter's foundation ModuleContract
‚îú‚îÄ‚îÄ types.ts             # TypeScript interfaces (ModuleDefinition extends ModuleContract)
‚îú‚îÄ‚îÄ utils.ts             # Shared utilities
‚îî‚îÄ‚îÄ index.ts             # Public exports
```

<Aside type="caution">
These files are marked "DO NOT EDIT". Changes to the scaffold propagate to all modules via `scaffold:update`. If you find a bug, report it upstream to the module starter repository.
</Aside>

### The foundation module contract (`module-contract.ts`)

Since scaffold v1.16.0 the system-level half of `ModuleDefinition` ‚Äî identity (`id`/`name`/`description`/`version`), dependencies (`depends`, deprecated `requiredModules`) and the lifecycle hook surface ‚Äî is **owned by the starter template** ([Modules feature](/starter-template/modules-feature/), v1.48.0+). The scaffold carries a verbatim, hash-stamped copy at `src/scaffold/module-contract.ts` (see its `@synced-from` banner) and `types.ts` defines `ModuleDefinition extends ModuleContract`, adding only the scaffold's codegen fields. One source of truth, two repos.

- The copy reaches every downstream module automatically via `scaffold:update` (it is scaffold infrastructure).
- In the **module starter repo only**, `npm run contract:sync` refreshes the copy from a sibling starter checkout and `npm run contract:check` verifies it ‚Äî the scaffold release bump runs the check automatically, so a release cannot ship a stale copy.
- Nothing changes for module authors: `import type { ModuleDefinition } from './scaffold/index.js'` keeps working, and all lifecycle types keep their names.

## Installation Pipeline

When a user runs `npx my-module init`, the scaffold runs a 10-step installation pipeline:

<Steps>
1. **Check dependencies** ‚Äî If your module declares `depends` (module id ‚Üí semver range) or the deprecated `requiredModules`, verifies each is installed by checking for its feature directory (the check unions both fields; ranges are recorded, not yet validated)

2. **Interactive setup** ‚Äî Runs module-specific prompts (from `module.ts`), detects frontend/backend paths, asks about auto-registration, shadcn components, and dependency installation

3. **Copy database schema** ‚Äî Detects Prisma structure (multi-file or single-file), processes template placeholders, appends models. For multi-file setups, creates `prisma/my-module.prisma`. For single-file, appends to `schema.prisma`

4. **Generate frontend files** ‚Äî Creates directory structure, processes all frontend templates (replacing placeholders), adds version headers, writes files. Processes `templates`, `indexFiles`, and `registrations` from your module config

5. **Generate backend files** ‚Äî Same process for backend templates. Uses your `indexTemplate` if provided, otherwise generates one with side-effect registration imports

6. **Auto-register** ‚Äî Wires the module into the app (see [Auto-Registration](#auto-registration) below)

7. **Environment variables** ‚Äî Collects values for env variables (supporting `secret` input and `when` conditionals), writes to frontend `.env` (with `VITE_` prefix) and backend `.env` separately. In `--no-interactive`/CI runs (v1.13.0+) it does not prompt ‚Äî it uses each variable's declared default and warns about any required variable left unset

8. **Install UI components** ‚Äî Runs `npx shadcn add <components> -y` (batched, with per-component fallback). Also handles third-party UI libraries (`uiLibraries` config) with their registries

9. **Install dependencies** ‚Äî Runs `npm install` in frontend and backend directories. Handles provider-specific dependencies based on prompt answers

10. **Save manifest and run lifecycle hooks** ‚Äî Saves `.module-manifests.json` tracking all installed files and registrations, then runs `onInstall` and `onActivate` lifecycle hooks
</Steps>

## Template Processing

The `processTemplate()` function handles all placeholder replacement. Every template string passes through this before being written to disk.

### Name Placeholders

| Pattern | Replaced With | Example (module id: `ai-chat`) |
|---------|--------------|-------------------------------|
| `My Feature` | Title Case (space-separated) | `Ai Chat` |
| `my feature` | lowercase (space-separated) | `ai chat` |
| `my-feature` | Module ID (kebab-case) | `ai-chat` |
| `myFeature` | camelCase | `aiChat` |
| `MyFeature` | PascalCase | `AiChat` |
| `my_feature` | snake_case | `ai_chat` |

<Aside type="tip">
Use `My Feature` for human-readable UI strings ‚Äî page titles, labels, descriptions, breadcrumbs. Use `my feature` (lowercase) for sentence-embedded text like "Turn on/off my feature functionality". The code-oriented patterns (`my-feature`, `myFeature`, `MyFeature`, `my_feature`) are for identifiers, file names, and code references.
</Aside>

### Conditional Blocks

Templates support Handlebars-style `{{#if}}` / `{{else}}` / `{{/if}}` conditionals, evaluated against prompt answers:

```typescript
// Simple truthy check
{{#if includeApi}}
import { api } from '@/lib/api';
{{/if}}

// Equality check with else
{{#if provider === 'stripe'}}
import Stripe from 'stripe';
const client = new Stripe(process.env.STRIPE_SECRET_KEY);
{{else}}
import { Paddle } from '@paddle/paddle-node-sdk';
const client = new Paddle(process.env.PADDLE_API_KEY);
{{/if}}

// Inequality check
{{#if mode !== 'manual'}}
startAutoProcess();
{{/if}}
```

**Truthiness rules:**
- Truthy: `true`, non-empty strings, non-zero numbers
- Falsy: `false`, `null`, `undefined`, `''`, `0`, variable not provided
- If no variables are passed to `processTemplate()`, conditional blocks are left unchanged

<Aside type="tip">
Conditionals are evaluated **before** variable substitution. This means you can use `{{variable}}` placeholders inside conditional blocks and they'll be substituted after the block is resolved.
</Aside>

### Variable Substitution

Variables from prompt answers use double-brace syntax with optional defaults:

```typescript
const provider = '{{provider}}';
const timeout = {{timeout|5000}};       // Default: 5000
const model = '{{model|gpt-4}}';        // Default: 'gpt-4'
```

### Processing Order

The `processTemplate()` function runs three steps in order:

1. **Name placeholders** ‚Äî `My Feature` ‚Üí `Ai Chat`, `my-feature` ‚Üí `ai-chat`, etc.
2. **Conditional blocks** ‚Äî `{{#if}}` / `{{else}}` / `{{/if}}` evaluated and resolved
3. **Variable substitution** ‚Äî `{{variable}}` and `{{variable|default}}` replaced

### File Path Processing

File paths in `templates` keys are also processed:

```typescript
templates: {
  'my-feature.routes.ts': backend.ROUTES_TEMPLATE,
}
// Becomes: ai-chat.routes.ts (for module id "ai-chat")
```

<Aside type="caution">
Templates are TypeScript template literal strings. If the generated code itself uses backticks or `${}` expressions, they must be escaped (`` \` ``, `\${...}`) ‚Äî otherwise the outer template string closes early or evaluates the expression. This is the **#1 source of template bugs**. See [Building Your Module ‚Äî Escaping Backticks and Template Expressions](/module-starter/building-modules/#escaping-backticks-and-template-expressions) for the rules and examples.
</Aside>

## Auto-Registration

The scaffold automatically wires modules into three integration points during installation.

### Frontend Routes (`routes.tsx`)

```typescript
// Before installation:
...userRoutes,
...adminRoutes,
// ...featureRoutes,

// After installation:
...userRoutes,
...adminRoutes,

// --- AI Chat Module routes ---
...aiChatRoutes,
// ...featureRoutes,
```

The scaffold finds the last spread route pattern and inserts after it. The `// ...featureRoutes` comment serves as a fallback insertion point.

### Frontend Navigation

If your module has a `navigation.registration.ts` template, the scaffold detects it uses the **registry pattern** and skips any config file editing. The registration runs automatically as a side-effect import in your module's barrel file.

### Backend Routes (`app.ts`)

```typescript
// Feature registrations
import './features/user/index.js';
import './features/admin/index.js';
import './features/ai-chat/index.js';    // ‚Üê Added by scaffold
```

The scaffold finds the feature registrations section or the last feature import and appends yours.

## Enable / Disable

Modules can be temporarily disabled without removing files:

```bash
npx my-module disable    # Runs onDeactivate hook, marks disabled
npx my-module enable     # Runs onActivate hook, marks enabled
```

Module state is tracked in `.module-state.json`. The scaffold stores:
- `enabled` flag ‚Äî whether the module is active
- `moduleAnswers` ‚Äî prompt answers from init (used by update for variable substitution)
- **Asset tracking** ‚Äî lists of installed dependencies, shadcn components, UI libraries, and env variables. The `update` command uses these to detect what's **new** between versions and only install the delta.

The scaffold does **not** comment out imports or modify generated code when disabling. What "disabled" means for your module is defined by your `onDeactivate` and `onActivate` lifecycle hooks (pausing jobs, unregistering webhooks, hiding UI, etc.).

For lifecycle hook details, see [Lifecycle Hooks](/module-starter/lifecycle-hooks/).

## CLI Commands Reference

| Command | Purpose | Key Options |
|---------|---------|-------------|
| `init` | Install module into app | `--force` (overwrite), `--no-interactive` |
| `update` | Update templates and schema to latest | `--dry-run`, `--force`, `--skip-schema` |
| `migrate` | Add version headers to old files | `--dry-run` |
| `uninstall` | Remove module completely | `--force`, `--keep-data`, `--export <path>` |
| `disable` | Disable without removing | `--force` |
| `enable` | Re-enable disabled module | `--force` |
| `info` | Show module information | ‚Äî |
| `update-schema` | *(Deprecated)* ‚Äî Use `update` instead | ‚Äî |

<Aside type="note">
When the `update` command detects schema changes and you confirm the update, it automatically runs `prisma generate` and `prisma migrate dev` afterwards ‚Äî no manual step needed. Schema comparison also normalises Windows line endings (CRLF ‚Üí LF) so that line-ending differences on Windows don't trigger false positives.
</Aside>

For detailed documentation of each command including step-by-step behaviour, output examples, and common errors, see [CLI Reference](/module-starter/cli-reference/).
