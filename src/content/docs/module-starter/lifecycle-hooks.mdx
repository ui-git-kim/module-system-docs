This page covers two types of hooks:

1. **Module lifecycle hooks** ā€” Code that runs during install, update, enable, disable, and uninstall
2. **Filter and action registries** ā€” Pipeline hooks that let modules extend app behaviour at runtime

---

## Module Lifecycle Hooks

Lifecycle hooks let your module run code at key moments. Define them in your `module.ts`:

```typescript
lifecycle: {
  onInstall: async (context) => {
    // Seed initial data, create indexes
  },
  onUpdate: async (context) => {
    // Run data migrations between versions
  },
  onActivate: async (context) => {
    // Start background jobs, register webhooks
  },
  onDeactivate: async (context) => {
    // Pause jobs, unregister webhooks
  },
  onUninstall: async (context) => {
    // Offer data export, clean up external resources
  },
},
```

| Hook | When It Runs | Use Cases |
|------|-------------|-----------|
| `onInstall` | After `npx my-module init` completes | Seed initial data, create database indexes, validate config |
| `onUpdate` | During `update`, **only on an upgrade** (installed version older than the module version). Skipped on `--dry-run`; if it throws, the update aborts unless `--force` | Data migrations, deprecation handling |
| `onActivate` | After install or `enable` command | Start background jobs, register webhooks |
| `onDeactivate` | During `disable`, **and at the start of `uninstall`** (if the module is still enabled, before `onUninstall`) | Pause jobs, unregister webhooks |
| `onUninstall` | During `uninstall` command (after `onDeactivate`) | Offer data export, clean up external resources |

<Aside type="tip">
Lifecycle hooks are optional. Most simple modules don't need them. Add them when your module needs setup beyond just generating files.
</Aside>

### Lifecycle Context

All hooks receive a context object with utilities:

```typescript
{
  cwd: string,                    // Working directory
  module: ModuleDefinition,       // Full module config (context types live in the synced foundation contract, generic over the definition; the scaffold aliases them to the full ModuleDefinition)
  interactive: boolean,           // Is running interactively
  utils: {
    log: {
      info: (message: string) => void,
      success: (message: string) => void,
      warning: (message: string) => void,
      error: (message: string) => void,
    },
    fileExists: (path: string) => boolean,
    readFile: (path: string) => string,
    writeFile: (path: string, content: string) => void,
    ensureDir: (path: string) => void,
    runCommand: (cmd: string) => Promise<{ stdout: string; stderr: string }>,
    prompt: (question: string, defaultValue?: string) => Promise<string>,
    confirmPrompt: (question: string, defaultValue?: boolean) => Promise<boolean>,
  },
}
```

<Aside type="note">
Path-taking utils (`fileExists` / `readFile` / `writeFile` / `ensureDir`) resolve **relative** paths against `context.cwd`. `runCommand` **never throws** ā€” on a non-zero exit it resolves with the captured `stderr` (or the error message), so check `stderr` rather than wrapping the call in try/catch. In CI/non-interactive runs, `prompt`/`confirmPrompt` return their default without blocking.
</Aside>

**Install-specific context:**
- `promptAnswers` ā€” User's responses to module prompts
- `envVariables` ā€” Collected environment variable values

**Update-specific context:**
- `fromVersion` ā€” Previous installed version
- `toVersion` ā€” New version being installed
- `versionChange` ā€” Object with `major`/`minor`/`patch` booleans; **exactly one is true** ā€” the highest-level segment that changed (e.g. `1.2.3 ā†’ 2.0.0` sets only `major`)
- `schemaChanged` ā€” Whether the Prisma schema template has changed
- `moduleAnswers` ā€” Stored prompt answers from initial installation (preserved across updates)
- `changes` ā€” Detailed change tracking:
  - `updatedFiles` ā€” Files that were updated
  - `addedFiles` ā€” New files added
  - `removedFiles` ā€” orphaned generated files (their template no longer exists), backed up to `.module-backups/` before deletion
  - `schemaUpdated` ā€” Whether the schema was updated in this run

```typescript
onUpdate: async (context) => {
  // React to specific types of changes
  if (context.versionChange.major) {
    context.utils.log.warning('Major version update ā€” running data migration');
    await runMigration(context.fromVersion, context.toVersion);
  }

  if (context.schemaChanged) {
    context.utils.log.info('Schema updated ā€” you may need to run prisma migrate');
  }

  // Access original prompt answers
  const provider = context.moduleAnswers.provider || 'default';
},
```

## Filter & Action Registries

Modules can hook into **runtime** application events using **filters** (transform data) and **actions** (execute side effects). These are pipeline registries ā€” available on both frontend and backend.

| Registry | Purpose | Returns Value? |
|----------|---------|----------------|
| **Filter** | Transform data at hook points | Yes ā€” modified data |
| **Action** | Execute side effects at hook points | No ā€” fire and forget |

<Aside type="tip">
Use **filters** when you need to modify data (e.g., format a display name).
Use **actions** when you need side effects (e.g., log an event, send a notification).
</Aside>

<Aside type="caution" title="Performance: sync callbacks run on hot paths">
Every sync filter and sync action you register runs in the caller's execution path. A slow callback on a frequent hook (e.g. `theme.cssVariables`, `user.displayName`) will directly degrade UI responsiveness or API latency across the whole app.

- **Sync filters** run in the response pipeline ā€” keep O(1), no I/O
- **Sync actions** run on the triggering thread ā€” no blocking work, no large DOM mutations
- For I/O, use `registerAsync` / `addActionAsync`. The starter awaits registered async work at `doActionAll`; for fire-and-forget dispatch use `void doActionParallel` (since v1.54.0, `doAction` dev-warns when async handlers would be skipped ā€” it runs the sync pool only)
- For deferrable work, use `queueMicrotask` / `requestIdleCallback` inside the action
</Aside>

<Aside type="note">
For the complete API reference, see the [Filter Registry](/starter-template/registries/#filter-registry) and [Action Registry](/starter-template/registries/#action-registry) in the starter template docs.
</Aside>

## Filter Registry

Filters transform data as it passes through hook points. Multiple filters can chain together ā€” each receives the output of the previous filter.

### Basic Usage

<Tabs>
  <TabItem label="Frontend">
```typescript
// filter.registration.ts
import { filterRegistry } from '@/config/filter.registry';

filterRegistry.register(
  'user.displayName',        // Hook name
  'my-feature-titles',       // Unique ID (prefix with module name)
  (name, context) => {
    if (context?.user?.title) {
      return `${context.user.title} ${name}`;
    }
    return name;
  },
  10                         // Order (lower = runs first)
);
```
  </TabItem>
  <TabItem label="Backend">
```typescript
// filter.registration.ts
import { filterRegistry } from '../../config/filter.registry.js';

// Sync filter
filterRegistry.register(
  'api.user.profile.response',
  'my-feature-mask-email',
  (profile) => ({
    ...profile,
    email: maskEmail(profile.email),
  }),
  20
);

// Async filter (for database lookups, external APIs)
filterRegistry.registerAsync(
  'api.user.profile.response',
  'my-feature-enrich',
  async (profile, ctx) => {
    const stats = await getStats(profile.id);
    return { ...profile, stats };
  },
  5
);
```
  </TabItem>
</Tabs>

### Applying Filters

```typescript
// Sync
const displayName = filterRegistry.apply('user.displayName', user.name, { user });

// Async (backend)
const response = await filterRegistry.applyAsync('api.user.profile.response', profile, ctx);

// Apply all ā€” sync then async (backend)
const result = await filterRegistry.applyAll('hook.name', data, ctx);
```

### Filter Methods

| Method | Description |
|--------|-------------|
| `register(hook, id, fn, order?)` | Register sync filter |
| `registerAsync(hook, id, fn, order?)` | Register async filter (backend only) |
| `unregister(hook, id)` | Remove a sync filter |
| `unregisterAsync(hook, id)` | Remove an async filter (backend only) |
| `apply(hook, value, ctx?)` | Run sync filters |
| `applyAsync(hook, value, ctx?)` | Run async filters (backend only) |
| `applyAll(hook, value, ctx?)` | Run sync then async (backend only) |
| `hasFilters(hook)` | Check if any filters exist |
| `getHooks()` | List all hook names |
| `getFilterCount(hook)` | Get count of filters for a hook |
| `getFilterIds(hook)` | List filter IDs |

## Action Registry

Actions execute side effects at hook points. Unlike filters, they don't return values and can't modify data.

### Basic Usage

<Tabs>
  <TabItem label="Frontend">
```typescript
// action.registration.ts
import { actionRegistry } from '@/config/action.registry';

actionRegistry.addAction(
  'user.loggedIn',           // Hook name
  'my-feature-analytics',    // Unique ID
  (context) => {
    trackEvent('login', { userId: context.user.id });
  },
  10                         // Order
);

actionRegistry.addActionAsync(
  'settings.saved',
  'my-feature-sync',
  async (context) => {
    await syncToExternalService(context.settings);
  },
  20
);
```
  </TabItem>
  <TabItem label="Backend">
```typescript
// action.registration.ts
import { actionRegistry } from '../../config/action.registry.js';
import { createModuleLogger } from '../../lib/logger.js';

const log = createModuleLogger('my-feature');

actionRegistry.addAction(
  'user.created',
  'my-feature-welcome',
  (context) => {
    log.info({ userId: context.userId }, 'New user ā€” queue welcome email');
  }
);

actionRegistry.addActionAsync(
  'user.deleted',
  'my-feature-cleanup',
  async (context) => {
    await cleanupModuleData(context.userId);
  },
  30
);
```
  </TabItem>
</Tabs>

### Triggering Actions

```typescript
// Sync only
actionRegistry.doAction('user.loggedIn', { user, method: 'email' });

// Async sequentially
await actionRegistry.doActionAsync('settings.saved', { settings });

// All ā€” sync first, then async (backend)
await actionRegistry.doActionAll('user.created', { user, req });

// Async in parallel (backend)
await actionRegistry.doActionParallel('notifications.send', { notifications });
```

### Action Methods

| Method | Description |
|--------|-------------|
| `addAction(hook, id, fn, order?)` | Register sync action |
| `addActionAsync(hook, id, fn, order?)` | Register async action |
| `removeAction(hook, id)` | Remove an action |
| `doAction(hook, context)` | Execute sync actions |
| `doActionAsync(hook, context)` | Execute async actions sequentially |
| `doActionAll(hook, context)` | Execute sync then async (backend) |
| `doActionParallel(hook, context)` | Execute async in parallel (backend) |
| `doActionWithResults(hook, context)` | Execute all and collect per-action success/error results |
| `hasActions(hook)` | Check if actions exist |
| `getHooks()` | Get all hook names |

<Aside type="caution" title="doAction does not run async handlers">
`doAction` executes **only** the handlers registered with `addAction`. Handlers registered with `addActionAsync` are **silently skipped** ā€” nothing errors, they just never run. If an event may have async consumers (true for most cross-module events), emit it with `await doActionAll(...)` (sync then async, in order) or `await doActionParallel(...)` (async consumers in parallel). Firing a bare `doAction` for an event you expect other modules to consume asynchronously is a common footgun.
</Aside>

### Cross-module events (a reusable pattern)

Actions aren't only for the built-in hooks ā€” a module can **define its own hook** and use the action registry as a lightweight, decoupled event bus that *other* modules subscribe to. This is how one module lets others react to its events without either side importing the other.

**Producer** ā€” the module that owns the event emits a well-defined hook, awaited as an async fan-out, carrying a small, documented payload (a **reference**, not large blobs):

```typescript
// In the producing module's service, right after the thing happens:
await actionRegistry.doActionParallel('documents.document.uploaded', {
  userId, providerId, remotePath, fileName, mimeType, size, contentHash,
});
// Depends on no consumer. When nobody subscribes, this is a no-op.
```

**Consumer** ā€” any other module subscribes with `addActionAsync` and keeps the callback **O(1)**: validate, then hand off to a [job](/starter-template/jobs-feature/). Do the slow or expensive work in the job handler, not in the callback ā€” the producer *awaits* the fan-out, so a slow callback slows the producer:

```typescript
// consuming module's action.registration.ts (side-effect import from index.ts)
actionRegistry.addActionAsync('documents.document.uploaded', 'my-feature.on-upload', async (ctx) => {
  const c = ctx as { userId: string; providerId: string; remotePath: string };
  await enqueueJob('my-feature.process-upload', c, { tenantId: c.userId });
});
```

Why this shape works:

- **Loose coupling.** The producer depends on nothing; the consumer depends only on the hook name and payload shape. Either module installs without the other.
- **Error isolation.** `doActionAll` / `doActionParallel` wrap each handler in try/catch ā€” a throwing consumer is logged, never propagated, so one bad subscriber can't break the producer or its peers.
- **Back-pressure and durability** live in the [jobs feature](/starter-template/jobs-feature/), not the hook: the callback only enqueues; the worker does the heavy work (and, under `JOBS_RUNNER=pg-boss`, survives restarts).

**Contract tips:** namespace hook names as `feature.entity.verb` (e.g. `documents.document.uploaded`); make the payload a **stable, documented reference** (ids and paths a consumer resolves on demand, not megabytes of data); and treat the payload shape like a public API ā€” additive changes only.

<Aside type="tip">
Worked example: the Document Management **upload ā†’ ingest pipeline** uses exactly this. Document Management (the producer) fires `documents.document.uploaded`; a consumer (cog-ingest) subscribes, enqueues a job, and the job pulls the bytes back on demand and ingests them. See the [Document Management roadmap](/document-management/roadmap/) and the [cog-ingest roadmap](/modules/cog-ingest/roadmap/).
</Aside>

## Common Hook Points

### Frontend Hooks

The `theme.changed` action and `theme.cssVariables` filter are wired in the starter template's [Theme Utilities](/starter-template/theme/). Other frontend hooks are planned and will be wired as concrete consumers are identified.

**Wired hooks:**

| Hook | Type | When | Context |
|------|------|------|---------|
| `theme.cssVariables` | Filter | Before CSS variables are applied to the DOM | The CSS variable map (`Record<string, string>`) |
| `theme.changed` | Action | After CSS variables are applied to the DOM | `{ variables }` ā€” the final applied map |
| `settings.saved` | Action | After settings save | `{ settings, page }` |
| `route.changed` | Action | After route navigation | `{ from, to }` |

Modules can also fire custom hooks from the frontend: `actionRegistry.doAction('my-feature.event', { data })`.

**Planned hooks:**

| Hook | Type | When | Context |
|------|------|------|---------|
| `user.displayName` | Filter | Format display name | `{ user }` |
| `user.avatarUrl` | Filter | Modify avatar URL | `{ user }` |
| `dashboard.widgets` | Filter | Modify widget list | `{ widgets }` |
| `nav.main.items` | Filter | Modify navigation | `{ items }` |
| `user.loggedIn` | Action | After login | `{ user, method }` |
| `user.loggedOut` | Action | After logout | `{ userId }` |

### Backend Hooks ā€” User Service

Fired from the user-service code paths (user acting on their own account).

| Hook | Type | Execution | Context |
|------|------|-----------|---------|
| `user.created` | Action | `await doActionAll` | `{ userId, email, role, isFirstUser }` |
| `user.roleUpdated` | Action | `doAction` | `{ userId, previousRole, newRole }` |
| `user.profileUpdated` | Action | `doAction` | `{ userId, fields }` |
| `user.preferencesUpdated` | Action | `doAction` | `{ userId }` |
| `user.deleted` | Action | `await doActionAll` | `{ userId, email, deletionRecord }` |

### Backend Hooks ā€” Admin Service

Fired from the admin-service code paths (admin acting on another user or settings). Admin-fired `user.roleUpdated` and `user.deleted` include the acting admin as `updatedBy` / `deletedBy`; user-fired versions from the section above do not ā€” listeners that care about the actor must check for the field and fall back gracefully.

| Hook | Type | Execution | Context |
|------|------|-----------|---------|
| `admin.settingsUpdated` | Action | `doAction` | `{ settings }` |
| `admin.navigationUpdated` | Action | `doAction` | `{ overrides }` |
| `admin.toolbarUpdated` | Action | `doAction` | `{ overrides }` |
| `user.roleUpdated` | Action | `doAction` | `{ userId, previousRole, newRole, updatedBy }` |
| `user.deleted` | Action | `await doActionAll` | `{ userId, deletedBy }` |

### Backend Hooks ā€” Filters (Not Yet Wired)

| Hook | Type | When | Context |
|------|------|------|---------|
| `api.user.profile.response` | Filter | Before sending profile | `{ userId }` |
| `api.*.response` | Filter | Before any API response | Varies |

<Aside type="note">
Filter hooks need concrete consumers ā€” code that calls `filterRegistry.apply()` with real data. Action hooks are wired first because they are fire-and-forget side effects. Filter hooks will be wired as the template adds data transformation points.
</Aside>

## Hook Naming Convention

Starter-template-owned hooks follow the patterns in the table below. **Module-fired custom hooks must be prefixed with the module ID** to avoid collisions ā€” e.g., a `billing` module firing `billing.subscription.created`, or a `my-feature` module firing `my-feature.item.created`. This matches the pattern used by the generated backend service template (`actionRegistry.doAction('my-feature.item.created', { ... })`).

| Pattern | Location | Examples |
|---------|----------|----------|
| `user.{property}` | Frontend | `user.displayName`, `user.avatarUrl` |
| `dashboard.{element}` | Frontend | `dashboard.widgets`, `dashboard.layout` |
| `nav.{section}.items` | Frontend | `nav.main.items`, `nav.secondary.items` |
| `api.{feature}.{action}.response` | Backend | `api.user.profile.response` |
| `service.{feature}.{method}` | Backend | `service.user.getProfile` |
| `auth.{event}` | Both | `auth.login`, `auth.failed` |
| `{module-id}.{entity}.{event}` | Module-owned | `billing.subscription.created`, `my-feature.item.updated` |

## Creating Custom Hooks

Modules can define their own hooks for other modules to use:

```typescript
import { filterRegistry } from '@/config/filter.registry';
import { actionRegistry } from '@/config/action.registry';

// Apply filter before returning data
export async function getFormattedData() {
  const rawData = await fetchData();
  const data = filterRegistry.apply('myModule.data.format', rawData, {
    source: 'api',
  });
  return data;
}

// Trigger action when something happens
export async function processItem(item: Item) {
  await doProcessing(item);
  await actionRegistry.doActionAsync('myModule.item.processed', {
    itemId: item.id,
    result: 'success',
  });
}
```

Document your hooks in your module's README so other developers can integrate:

```markdown
## Available Hooks

### Filters
| Hook | Purpose | Context |
|------|---------|---------|
| `myModule.data.format` | Transform data before display | `{ source }` |

### Actions
| Hook | When | Context |
|------|------|---------|
| `myModule.item.processed` | After item processing | `{ itemId, result }` |
```

## Best Practices

**Do:**
- Prefix IDs with your module name ā€” `my-feature-transform` not just `transform`
- Handle errors gracefully ā€” don't let filter/action errors break the app
- Use appropriate order values ā€” lower = runs first, default is 50 (see [Order Convention](/starter-template/hooks/#order-convention))
- Document your hooks ā€” help other developers integrate with your module
- Clean up on uninstall ā€” remove filters/actions if module provides uninstall logic

**Don't:**
- Throw unhandled errors in hooks ā€” catch and handle gracefully
- Use hooks for required logic ā€” hooks are optional extensions
- Block async actions ā€” keep them fast or use background jobs
- Modify the context object ā€” create new objects instead
