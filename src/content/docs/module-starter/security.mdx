The starter template ships a foundation **security feature** — a `securityRegistry`, app-layer field encryption, a deletion cascade, and tier-based posture. A module doesn't build any of that; it **declares** what it needs and the foundation enforces it.

Every generated **backend** module includes a **`security.registration.ts`** (a side-effect import from `index.ts`, alongside the route and service registrations) that declares the module's security surface and registers an account-deletion handler.

<Aside type="tip" title="Generated by default — delete if unused">
The scaffold writes `security.registration.ts` and wires its import for you. The default file declares the module's data category and a working deletion handler. If your module is frontend-only, or stores no per-user data, delete the file and its `index.ts` import.
</Aside>

## What the scaffold gives you

The generated file declares a manifest and a deletion handler:

```typescript
// backend/src/features/my-feature/security.registration.ts
import { securityRegistry } from '../security/index.js';
import { prisma } from '../../lib/prisma.js';
import { createModuleLogger } from '../../lib/logger.js';

const log = createModuleLogger('my-feature:security');

securityRegistry.register({
  moduleId: 'my-feature',
  ownedModels: [],                                           // RLS — opt-in (see below)
  dataCategories: [{ name: 'my_feature_items', classification: 'owned' }],
  // encryptedFields, hashedSecrets, auditEvents, minimumTier — opt-in below
});

securityRegistry.registerDeletionHandler({
  id: 'my-feature.items',
  moduleId: 'my-feature',
  classification: 'owned',
  description: "Hard-delete the subject's my-feature items",
  async run(ctx) {
    const userId = ctx.subject.userId ?? ctx.subject.tenantId;
    const { count } = await prisma.my_feature_item.deleteMany({ where: { userId } });
    log.info({ audit: true, userId, count }, 'my-feature items deleted (account erasure)');
    return { action: 'hard-delete', recordsAffected: count };
  },
});
```

<Aside type="caution" title="Backend imports are relative, not aliased">
Backend module code imports the security feature with **relative `.js` paths** — `../security/index.js`, `../../lib/owned.js`, `../../lib/prisma.js`. The `@/` alias is **frontend-only**; using it in backend code fails at runtime.
</Aside>

## The manifest

`securityRegistry.register(manifest)` declares the surface the foundation enforces:

| Field | Required | Purpose |
|-------|:--:|---------|
| `moduleId` | ✅ | Your module id (kebab-case). |
| `ownedModels` | ✅ (may be `[]`) | Models put behind **Neon RLS**, keyed on `tenantId`. |
| `dataCategories` | ✅ | Drive the deletion cascade + compliance bundle. |
| `encryptedFields` | — | Columns encrypted at rest (hardened+). |
| `hashedSecrets` | — | Columns stored as a SHA-256 hash, never plaintext. |
| `auditEvents` | — | Event names this module emits. |
| `credentialTypes` | — | Third-party secret types this module stores. |
| `minimumTier` | — | Refuse to run below `standard`/`hardened`/`strict`. |

## Field encryption

For sensitive columns (API responses, tax ids, tokens), encrypt at rest. The column stays a `String` holding an **AES-256-GCM** envelope; the key lives app-side and **never reaches the database**.

<Steps>
1. **Declare it** in the manifest:
   ```typescript
   encryptedFields: [{ model: 'my_feature_item', field: 'secretValue' }],
   ```
2. **Encrypt on write / decrypt on read** in your service:
   ```typescript
   import { encryptField, decryptField } from '../security/index.js';

   // create
   const sealed = await encryptField(input.secretValue, { tenantId: userId });
   await prisma.my_feature_item.create({ data: { ...input, secretValue: sealed, userId } });

   // read
   item.secretValue = await decryptField(item.secretValue, { tenantId: userId });
   ```
</Steps>

Field encryption is on at `hardened`+ tiers. `decryptField` throws if the envelope is tampered with or the key is wrong. The `{ tenantId }` option is forward-compatible with per-tenant keys (the shipped env adapter ignores it).

## Hashed secrets

For high-entropy tokens / licence keys you only ever need to *verify* (never read back) — **not passwords** (those belong to Neon Auth):

```typescript
import { hashSecret, verifySecret } from '../security/index.js';

const stored = hashSecret(apiToken);              // store this column
const ok = verifySecret(presented, stored);        // constant-time compare
```

Declare them so the foundation knows: `hashedSecrets: [{ model: 'my_feature_item', field: 'apiToken' }]`.

## Deletion handlers

No provider erases data across **all** your stores, so each module registers a handler that the account-deletion **cascade** runs. Handlers run grouped by `classification`:

| `classification` | What the handler does |
|---|---|
| `credential` | **crypto-shred** — destroy the decryption key first |
| `owned` | **hard-delete** the module's rows |
| `referenced` | **unlink** a pointer to an external resource (never touch the source) |
| `preserved` | **defer** — legal hold, not deleted |

<Aside type="caution" title="Handlers must be idempotent">
The cascade may retry. Use `deleteMany` (not `delete`) so a re-run finds 0 rows and succeeds. A handler must **throw** (or return `{ action: 'failed' }`) if it did not complete — the orchestrator treats any non-failed return as success and removes the auth identity last.
</Aside>

Scope the handler to the subject. The scaffold schema keys on `userId` (the actor), so it resolves `ctx.subject.userId ?? ctx.subject.tenantId` (a tenant-offboard may set only `tenantId`) and deletes `where: { userId }`. If you adopt `tenantId` isolation (below), switch to `owned(ctx.subject.tenantId)`.

## RLS & ownership (`tenantId`)

`ownedModels` puts a model behind **Neon Row-Level Security**, keyed on the isolation column **`tenantId`** (which equals `userId` in solo mode). The scaffold's default schema is `userId`-keyed, so RLS is an **explicit upgrade**:

<Steps>
1. Add the isolation column to your model in `schema.prisma`:
   ```prisma
   model my_feature_item {
     // ...
     tenantId String
     @@index([tenantId])
   }
   ```
2. Populate it on write (`tenantId: userId` in solo) and scope queries with `owned()`:
   ```typescript
   import { owned } from '../../lib/owned.js';
   await prisma.my_feature_item.findMany({ where: owned(tenantId) });
   ```
3. Declare the model:
   ```typescript
   ownedModels: [{ model: 'my_feature_item', ownerColumn: 'tenantId' }],
   ```
</Steps>

The foundation pairs this with the `FORCE ROW LEVEL SECURITY` + `current_setting('app.tenant_id')` policy template — see the starter's [Database & Prisma](/starter-template/database/) guide.

## Minimum tier

A security-sensitive module can refuse to run below a tier. At bootstrap the startup posture guardrail halts (at `strict` in production) if the active tier is lower:

```typescript
minimumTier: 'strict',   // requires field encryption + compliance audit
```

## Checklist

- [ ] Keep `security.registration.ts` if your module stores per-user data (delete it if not).
- [ ] List each data store in `dataCategories` with the right `classification`.
- [ ] Register a **deletion handler** per owned data category (idempotent, subject-scoped).
- [ ] Declare `encryptedFields` and call `encryptField`/`decryptField` for sensitive columns.
- [ ] Declare `hashedSecrets` for tokens; never store them plaintext.
- [ ] Adopt `tenantId` + `ownedModels` if you want Neon RLS.
- [ ] Set `minimumTier` if the module needs encryption / compliance audit.

See the starter's [Security feature](/starter-template/security-feature/) for the full foundation reference.
